# 1. 历史与缘由

Spring 是一个为 Java 虚拟机提供依赖注入特性的应用框架——这些特性为应用程序开发者带来了可测试性、可靠性和灵活性。它改变了 Java 的开发方式，以下是其如何以及为何发生改变的原因。

## 为更好的开发奠定基础

根据维基百科的说法，Spring 是四个温带季节之一，紧随冬季之后……不，不，这不是一本关于天气的书。让我们再试一次。

弹簧是一种储存动能的机械装置，当储存的张力……不，这听起来也不对。这应该是一本关于编程的书。

再试一次：

从上下文来看，Spring 是一个用于 JVM 的应用框架。它使用一种称为“依赖注入”（我们保证本章后面会介绍）的概念作为通用开发模型，并且它改变了 Java 程序的编写方式——即使所讨论的程序本身避免使用 Spring。

Spring 也是一个*生态系统*——一个以名为 spring-core 的库为中心的扩展和模块星系。这些扩展和模块增加了功能和特性，以覆盖许多许多可能的用例。通常，人们不会过多提及 spring-core；相反，他们只使用“Spring”，并期望其他人理解所指的是整个生态系统。

如前所述，这些模块相当广泛。如果你需要用 Java 处理某些事情，很可能有一个 Spring 模块适用于它……在某处。

当然，Spring 最初来自某个地方；那个“某个地方”实际上是“某个人”，Rod Johnson。

2002 年，Rod 写了一本书，名为《Expert One-on-One J2EE Design and Development》。^(¹) 这本书是对 J2EE 某些方面的回应，J2EE 是当时 Java 企业应用设计的主流框架，因此它代表了 Java 编程的一场革命——或者如果你愿意，可以说是快乐与便捷的重生。在书中，Rod 包含了一些示例代码，Juergen Hoeller 和其他人请他将其开源，这些代码最终进入了一个名为“Spring”的项目，该项目后来又出现在另一本书《Expert One-on-One J2EE Development without EJB》^(²) 中。*正是*这本书真正推动了事情的发展。^(³)

要理解 J2EE^(⁴)（以及实际上 Spring）的重要性，我们需要思考当时编程是如何演变的。“真正的应用程序”往往仍在小型机或大型机上运行；托管资源仍然占据主导地位，而那些能在个人计算机上运行并保持相关性的应用程序则令人惊讶。当然，每个人都知道 Lotus 1-2-3，^(⁵) dBase 是一个功能出奇强大的数据库应用程序，当然，学生和其他类似出身低微的人用个人计算机写文档也没问题……但要真正运行*真正的应用程序*，意味着要在企业大型机上运行。

对于今天的开发者来说，大型机开发会极其受限；安排作业运行（即使是编译或测试）的概念是陌生的。例如，我们习惯于能够快速启动一个 Docker^(⁷) 中的 PostgreSQL^(⁶) 实例，然后将我们的应用连接上去进行快速集成测试。今天，我们觉得运行一个应用程序两秒钟只是为了看看函数是否返回正确的值，这没什么大不了的……但在大型机时代，这代表着相当可观的时间、金钱和磁盘空间投入。你通常有一个生产数据库和一个测试数据库，两者都比黄金更珍贵，仅仅是因为它们很难替换。

在这种环境中，系统管理员是王者。系统管理员告诉你他或她把数据库放在哪里，以及如何连接它们，更重要的是，有多少系统资源分配给了你的程序。超出这些资源，你的程序就会被终止，以防止对其他所有程序产生负面影响。


Java 应运而生，它可以在性能平平的本地台式机上编译。突然之间，为大型机运行而开发的应用程序成本降低了，因为除了运行应用程序之外，大型机不再那么重要。更重要的是，Java 足够强大，能够将大型机的理念（“真正的程序在这里运行”）带入小型机甚至个人电脑等更小的机器中。

J2EE 被创建为一个应用程序框架，为企业级开发提供了一套标准。它包含表示层和计算层等，并规定了这些层之间通信的正式标准。它还采用了“容器”的概念，可以将 J2EE 模块（如 Web 模块和其他此类组件）部署到其中，而容器则代表了系统管理员的一个控制点。例如，管理员可以控制一个给定容器可以连接到特定数据库的连接数。

上一段中最重要的词是“正式”。次重要的词是“控制”，即“容器管理员所拥有的控制”。

Java 有一个称为“Java Bean”的概念，它旨在成为一个可部署的功能单元；你可以有一个遵循特定标准并代表某种相关操作的“Bean”；例如，你可能有一个 Bean，用于在提供样本数据时计算曲线上的点，或者一个执行矩阵计算的 Bean。

同样，J2EE 也有“企业级 Java Bean”（简称“EJB”）的概念，它是一个具有额外限制和功能的 Java Bean，主要集中在事务和远程调用能力上。一个 EJB 在被调用时，代表了一次进程间通信，即使它与调用者位于同一位置。

在实践中，这意味着每个 J2EE 程序员从一开始就能够设计出可工作的分布式架构。这绝非易事；分布式架构很难做对，而 J2EE 使得在独立机器上调用进程变得轻而易举。

当然，这一切都是有代价的。

EJB 的本质，尤其是在 J2EE 生命周期的早期，意味着当每次调用都可能成为远程调用时，每次调用都必须被视为远程调用来处理，并遵循特定的部署和开发周期（早期，你需要一个名为 `ejbc` 的独立打包编译器来构建 EJB 运行所需的支持类）。并且，由于要求即使 EJB 不是远程的也要被视为远程的，EJB 运行缓慢。

EJB 开发慢、测试慢、部署慢、调用慢。而且它们无处不在。

《无 EJB 的 J2EE 开发》一书正是对这种根深蒂固思维模式的逻辑性回应。Rod Johnson 和 Juergen Hoeller 审视了 EJB 的使用方式（以及滥用方式），以及它们对开发者生产力的实际成本——就其提供的价值而言，这个成本确实非常高。要使用一个 EJB，你不仅需要经历一个特殊的开发和部署周期，还必须在运行时查找它们，并在它们不可用时处理异常。

在清单 1-1 中，你不仅需要编写代码来从其他地方获取一个可以创建 EJB 的组件^(⁸)，还必须*配置*该引用，然后还需要经历一套语义来获得可以实际调用你的 EJB 的东西。但是，当这些远程语义很少被需要或期望时，为什么还要经历开发和远程语义的麻烦呢？尽管这些语义对你的代码产生了灾难性的影响，但它们很少是重要的；在 2000 年代初期的一次 Java 研讨会上，与会者被问及有多少人使用 EJB，大多数人都举手了；下一个问题是，这些 EJB 中有多少是实际被远程调用的。答案是：根据传闻，大约只有百分之二。^(⁹)

正是*这种*思维模式驱动了《无 EJB 的 J2EE 开发》一书的诞生——渴望将自由归还给编码者，为必要的复杂性编程提供支持。如果你需要调用一个远程服务，好吧，没有什么能阻止你这样做，但实际编码的经验表明，这很少是必要的，反而成为了完成工作的障碍。

Rod 的主要见解是，EJB 可能很强大，但它们有点像把 V6 发动机绑在儿童自行车上。大多数开发者不需要它们，也不想要它们；EJB 只是恰好成为了*当时流行*的语言特性，结果导致开发速度变慢，部署更加困难^(¹⁰)，测试困难，配置也晦涩难懂。大多数开发者没有使用“正确的技术”来查找 EJB，因为他们不想花时间配置 EJB 容器。配置容器也意味着重现条件更加困难。

这真是一团糟。有时，它也算是一种富有成效的混乱；毕竟，这项技术是为了响应实际用户需求而创建的。但是，即使开发者能够利用它很好地完成实际工作，它也给这些开发者带来了沉重的负担，而且现有的实践并没有改善这种情况。这项技术甚至根本没有*允许*更好的实践。



## Rod 与 Juergen 改变（Java）世界

在《无 EJB 的 J2EE 开发》一书中，Rod（还有 Juergen，不过为了简洁我简称为 Rod，希望 Juergen 不要介意！）勾勒出了一套 J2EE 开发流程，该流程聚焦于六大主题：

*   **简单性**
*   **生产力**
*   **面向对象的根本重要性**
*   **业务需求的首要地位**
*   **经验过程的重要性**
*   **可测试性的重要性**^(¹¹)

除了这些主题显然相当重要之外，认识到这些主题在典型的 J2EE 开发中属于*次要*关注点也同样重要。

例如，EJB 旨在提供对远程服务的*通用*访问，因此开发一个 EJB 意味着要创建能通过复杂协议调用的东西，同时还要处理相当复杂的事务问题。如果这*确实是你需要的*，那么 EJB 没问题……但大多数开发者*并不*需要担心 CORBA，^(¹²) 而且他们的事务需求通常非常简单。EJB 代表了*应用程序*本身并不需要的复杂性。

如果应用程序不需要这种复杂性，但程序员却必须了解它，那么这种复杂性就代表了开发中直接且不可避免的沉没成本。

此外，EJB 的工作方式往往不像 Java。它们代表的是功能端点，而非自包含的对象。设计 EJB 意味着要跳出最初使 Java 变得有用的那些范式。

Spring 的设计初衷是改变企业级开发，使其优先考虑——嗯，所有这六大主题。它被设计得简单（因此高效，并且理所当然地专注于实际业务需求）、面向对象，并且**可测试**，后者的价值难以估量。^(¹³)

随后，《无 EJB 的 J2EE 开发》一书大放异彩，展示了一个自包含的应用上下文，尽管使用了 XML，但开发者使用起来远比 J2EE 那种分层的开发者/部署者模型要容易得多。它将权力交还给了开发者，并在此过程中提供了简单的方法来创建仅用于测试的对象（这是 EJB 提供但实现起来非常困难的功能），以及改变应用对象获取资源方式的方法，使其对大多数开发者来说都合情合理。

> **注意**
>
> XML 是一个出色的规范，但它也是一个*正式*的规范，有时很难用好。像 JSON 和 YAML 这样的其他标记语言已在日常使用中很大程度上取代了它，因为它们编写起来容易得多，而且在可靠性方面的权衡通常被认为是值得的。

毕竟，如果你在 `FooFram` 中需要一个 `Widget`，通常不会认为在普通 Java 中 `FooFram` 应该以某种方式请求远程服务来提供一个 `Widget`；通常的做法是在 `FooFram` 对象中有一个 `setWidget(Widget widget)` 方法。

Spring 在让 Java 重新看起来像 Java 方面功不可没。客观的采用率尚不清楚，但可以肯定地说，Java 开发者对使用 Spring 进行开发的热情远高于传统的 J2EE。然而，Spring 并非 J2EE 开发格局的正式组成部分。

Spring 采用了一种近乎颠覆性的方法；你使用 J2EE，同时将 Spring 与 J2EE 结合使用来实际完成工作。这是一种奇特但有效的模式。Spring 逐渐成为许多人的标准依赖；它是一种标准习惯用法，却并非任何标准的实际组成部分。

通过创建社区流程，使得 Java 应用设计不再由圣何塞那些与世隔绝的架构师决定，更好的方法才成为可能。

## 杠杆：依赖注入

Spring 引入到更广泛 Java 世界的核心设计模式现在被称为“依赖注入”。^(¹⁴) 在“传统 J2EE 方法”中，需要资源的对象会以确定性的方式获取该资源，并指定一个名称。（我想，你可以*查找*你想要的名称，但随后你必须查找*特定的名称*，这也好不到哪去。）提供哪个资源的控制权被交给了 J2EE 应用服务器的管理员。

对于使用 J2EE 的开发者来说，他或她必须扮演管理员的角色，并以某种方式配置名称服务。借用芭比娃娃的一句话，名称服务很难；你必须确保拥有符合条件的类型（说实话，门槛不高，因为大多数你想用名称服务的东西，按设计都是符合条件的），你必须知道如何确保对象被正确构造，你必须知道如何*为你的特定应用服务器*设置名称和资源，然后你还必须知道如何设置特定于应用的、重定向后的名称——再次强调，是针对每个应用服务器，因为虽然大多数服务器相似，但它们并不完全相同。

这是一个非常强大的想法，但它超出了大多数开发者想要做的范围——而且许多开发者只做了最低限度的工作，甚至懒得为资源设置本地名称。这也使得测试变得非常困难，因为你不能仅仅在测试中使用一个 `Widget`，你必须**部署**一个 `Widget`，为它设置一个名称，然后让你的对象在测试中*查找*这个 `Widget`。这当然是可行的——程序员通常富有创造力和干劲——但这也很痛苦。

然而，通过依赖注入，想要 `Widget` 的对象不再需要*查找* `Widget`。现在，当需要一个 `Widget` 的对象被构造时，依赖注入框架（在本书中，就是 Spring！）会立即*提供*一个 `Widget`。^(¹⁵) 依赖注入框架通过几种不同的方式收集关于 `Widget` 的信息：最简单的方式（也是最早的方式）就是创建一个 XML 文件，其中包含对 `Widget` 的引用以及需要它的类。在 Java 和 Spring 的后续版本中，可以*搜索*应用代码以查找可注入资源的可能实例，这使得简单应用的配置变得非常轻量。

当程序员为 J2EE 设计代码时，他们受制于 J2EE 的思维模式；用 J2EE 做的每一件事都涉及大量的 J2EE 工作。你无法在不涉及 JNDI 配置的情况下测试使用了 EJB 的代码——一个简单的测试可能必须启动一个 JNDI 容器（可能还有 EJB 容器），同时填充 JNDI 数据集并确保 EJB 能够被解析。这些实际上都不是在**测试服务**——它们只是为了让你能够*开始*测试服务而必须做的苦力活。这意味着要忍受容器的启动时间（即使在最好的情况下这也是浪费时间），以及花在测试配置上的时间，而且许多容器的启动时间很长。

但如果*目标*是测试某些东西，实用主义会认为容器配置和启动时间*全部*都是浪费。

通过依赖注入，你只需创建一个反映所需行为的 Java 类，并将其提供给你正在测试的类。你的测试甚至不必依赖于 Spring 或其他任何东西。举个例子，让我们考虑两种形式的“Hello, World”^(¹⁶)——EJB 版本（大约存在于 2005 年^(¹⁷)）和一种代表已准备好进行依赖注入的类结构的版本。



### J2EE Hello World，2005 年版本

在 Spring 诞生的初期，一个 EJB 至少需要三个源文件：一个描述 EJB 契约的接口，一个提供 EJB 实际可执行代码的类，以及一个用于管理 EJB 生命周期（例如 EJB 创建或销毁等生命周期阶段的操作）的“home 对象”。我们甚至不打算深入探讨实际部署的开发过程——这*仅仅*是*运行时*所需的 Java 类结构。部署需要**至少**一个 XML 描述符，并且每个容器通常还需要另一个特定于容器的 XML 文件，用于连接容器 JNDI 树内的资源。即使在*最理想*的情况下，这也令人抓狂……而最理想的情况相当罕见。^(¹⁸)

```
HelloWorld 接口
```

请注意这个接口是多么简洁：它所做的并不比我们的 EJB 接口多（实际上 `HelloWorldEJB` 可以扩展这个接口，只添加 `EJBObject` 接口作为装饰）。这个类*没有任何地方*存在技术债务——既不是对 Spring 的，也不是对 J2EE 的。

```
package com.bsg6.hello;
public class HelloWorldImplementation implements HelloWorld {
public String sayHello(String name)
{
if(name==null) {
name="world";
}
return "Hello, "+name+".";
}
}
清单 1-7
HelloWorld 实现
```

我们在 `HelloWorldImplementation` 中看到了与 `HelloWorld` 相同的模式。当然，它依赖于 `HelloWorld` 接口，但也仅此而已。我们的类中没有任何内容不是专注于实际履行 `HelloWorld` 接口的契约。`HelloWorldEJBImplementation` **接近**这个想法，但必须实现 `SessionBean`——并且*不应该*实现代表其契约的接口。我们只展示了两个示例类，但技术债务却少得多，而且我们的类符合惯用的 Java 风格——甚至符合 2004 年左右（当时 J2EE 称王，恐龙统治地球）的惯用风格。

```
package com.bsg6.client;
public class Greeter
{
HelloWorld generator;
void setGenerator(HelloWorld generator)
{
this.generator=generator;
}
public void displayHello(String name)
{
System.out.println(generator.sayHello(name));
}
}
清单 1-8
HelloWorld 客户端
```

在这里，您看到 `generator` 引用指向一个 `HelloWorld` 实现。`Greeter` 并没有实例化一个 `HelloWorld`（尽管如果需要，它可以这样做）；然而，这里的想法是，`Greeter` *外部*的某个东西提供了一个 `HelloWorld`，而 `Greeter` 只是使用它。它不需要有任何获取代码，没有查找代码，它只是普通 Java 类上的一个属性。

`HelloBean` 太简单了，实际上并不*需要*其他实现……但假设我们需要将一些其他功能包装到一个实现中用于测试或其他诊断呢？^(²⁰) 我们可以创建该功能，然后直接使用新的实现进行测试；我们的 `Greeter` 类不必更改，并且除了使用新实现之外，我们无需做任何特殊的事情来切换到它。

*这*就是依赖注入；我们在需要时注入 `HelloWorld` 依赖。

Spring 不仅认识到了这种模式的力量，而且 Spring 框架还提供了使用它的工具；您可以定义一个 XML 文件，其中命名一个 `helloworld` bean（类型为 `com.bsg6.hello.HelloWorldImplementation`）以及一个 `Greeter` 实例，并自动将 `helloworld` 引用注入到 `Greeter` 中，而无需任何对 Spring 的源代码级依赖。

如果您使用框架的某些方面，Spring 会增加一些技术债务：例如，如果您愿意，您可以在 bean 中拥有生命周期钩子。（您仍然可以通过 XML 配置生命周期阶段，并避免源文件中的技术债务。）但 Spring 过去是，现在也基本上是务实的；作为开发者，您有权选择想要承担多少债务，而不是像 J2EE 那样强制施加且常常令人难以承受的技术债务。



## Spring 挣脱束缚，修复 Java EE

Java 最初的创建者和维护者 Sun 公司于 1998 年创建了 Java 社区流程，但几年后才开放成员资格。作为社区中*具有*影响力的成员，Spring 参与了 Java EE 6 的创建，当时 J2EE 已被重命名为一个不那么令人困惑的名称。^(²¹) 在 Spring 的参与下，Java EE 6 获得了一项新规范：“上下文和依赖注入”。添加了标准注解以支持 CDI 及其相关特性，整个行业都为之公开或私下地欢呼。

有趣的是，随着 Spring 对 Java EE 的影响力越来越大，Java EE 本身的影响力却*越来越小*。Spring 从未停止创新；现在，你可以使用 Spring 部署微容器，利用 Jakarta EE API 规范，而完全无需理会 Jakarta EE 容器本身。当然，你仍然可以使用 Jakarta EE 应用服务器，但这样做的必要性已经降低了。

这代表着“保守派”（那些仍然认为 2005 年是“美好旧时光”^(²²) 的系统管理员）失去了控制权——但这意味着今天的程序员拥有了比以往任何时候都更强大的能力和灵活性，并且公平地说，在 Java 领域，他们应该为此感谢 Spring 框架。

## Spring 6 与现代 Java

在过去的几年里，Java 增加了很多非常棒的功能。在 Oracle 将 Java EE 剥离并交给 Eclipse 基金会（从而产生了“Jakarta EE”而非“Java EE”）之后，JVM 本身在很大程度上实现了民主化。它仍然主要由 Oracle 控制，但更广泛的社区对 Java 的投入程度比自 21 世纪初以来任何时候都要高，Java 也因此得以快速发展。我们有了新的 `record` 类型（旨在成为不可变数据结构）和模块，并且很可能很快会看到新的多线程模型。

Spring 并没有在这些变化中停滞不前。Spring 6 认可并直接支持模块基础设施，并且 Spring 生态系统的大部分也已迁移到 Jakarta EE 的命名结构（使用 `jakarta` 作为包名，而非 `javax`）。在阅读本书的过程中，我们将探讨其中的一些内容。

警告

从 `javax` 到 `jakarta` 的迁移对于 Jakarta EE 来说并非无缝的。毕竟 Java 生态系统非常庞大，项目之间存在大量相互依赖关系；有些项目可能仍然使用旧的 `javax` 包，而人们更希望它们使用 `jakarta`。不幸的是，唯一的解决方案是时间和努力。我们会在后续内容中尽量小心并指出此类问题，但请注意，在现实中，很少有事情能像我们期望的那样完美。
