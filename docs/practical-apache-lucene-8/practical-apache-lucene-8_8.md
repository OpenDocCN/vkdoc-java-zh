# 8. 你的第一个 Lucene 补丁

本章涵盖了为 Apache Lucene 做贡献的一些最佳实践。本指南详细介绍了 Apache 软件基金会、Apache 之道，以及如何编写你的第一个补丁并与社区互动。



## 为什么要为开源项目做贡献？

为开源项目做贡献可能会让人望而生畏。每个人都很忙。也许你还有无数其他事情要做。为什么你要为开源做贡献，而不是去做别的事情呢？

这是一个棘手的问题，每个人为开源做贡献的动机都不同。请继续阅读一些常见的动机。其中一些可能会鼓励你做出贡献。

### 基于实际经验和经过验证的可行方案改进项目

这是显而易见的。我们使用一个开源软件，并意识到可以根据我们的用例和实际使用经验来改进它。积极的实际用户反馈能够促进产品的改进。为你喜爱的产品贡献代码，会让你感到温暖而充实。

### 发现并分享项目内部机制

你对开源项目的贡献可以帮助用户更多地了解产品/项目以及所使用系统的内部机制。你的贡献甚至可能带来更深层次的使用模式，这种使用基于对系统操作的增强理解。

你或许能让用户以最佳方式利用系统，同时避免复杂/代价高昂的陷阱。开源贡献也是你（以及其他用户）进入工程领域、一探究竟的绝佳机会。你可以从中学习，因为大多数流行的开源项目都经过精心设计，达到完美，并且对提交内容有很高的质量标准。

### 提升声誉，助推职业发展

为开源项目做贡献有助于建立/提升你作为工程师和开发者的声誉与可信度。你有机会展示自己的能力，而且由于互联网是“用墨水书写”而非“铅笔书写”，你的贡献是永恒且随时可见的。你的贡献可以在需要时被随时引用。

### 降低未来风险

与闭源项目不同，开源项目会清晰地公布其路线图，并明确说明诸如生命周期终止、弃用和组件移除等风险。成为你所关心的项目的开源社区一员，有助于缓解这些风险，同时确保你能够规划未来，了解即将发生的变化，并留意未来版本中值得关注的要点。

### 获得个人满足感

你可能仅仅出于贡献本身以及构建可能影响许多人的东西的愿望，而想为开源项目做贡献。还有什么其他方式能让你如此大规模地影响世界呢？

## 如何为开源项目做贡献

虽然为开源项目做贡献没有严格的流程，但在寻求贡献一个功能时，以下步骤通常效果不错：

1.  确定你解决方案的哪一部分值得贡献。
2.  研究目标项目的贡献指南。
3.  获取并构建项目。
4.  提取相关的代码片段。
5.  调整代码并整合所需的更改。
6.  提供所需级别的自动化测试用例和文档。
7.  提交一个 issue。
8.  提交更改。

## Apache 软件基金会

摘自维基百科：

*Apache 软件基金会 /əˈpætʃi/ (ASF) 是一家美国非营利公司（在美国归类为 501(c)(3) 组织），旨在支持 Apache 软件项目，包括 Apache HTTP 服务器。ASF 由 Apache 小组组建，并于 1999 年 3 月 25 日注册成立。[2][3]*

*Apache 软件基金会是一个由开发者组成的去中心化开源社区。他们生产的软件根据 Apache 许可证条款分发，是免费和开源软件 (FOSS)。Apache 项目的特点是协作、基于共识的开发过程以及开放、实用的软件许可证。每个项目由一个自我选择的技术专家团队管理，这些专家是项目的活跃贡献者。ASF 是一个精英管理组织，这意味着基金会的成员资格仅授予那些积极为 Apache 项目做出贡献的志愿者。ASF 被认为是第二代[4]开源组织，因为它提供商业支持，而没有平台锁定的风险。*

请注意，该基金会基于去中心化、自由软件的原则，并专注于精英管理和基于共识的软件开发方法。这些代表了 Apache Lucene/Solr 社区所依据的核心原则。潜在的贡献者需要了解这些原则，并尊重其所有标准。毕竟，Apache 软件基金会的核心理念是“社区重于代码”。

## 使用 Git

为了与社区互动并创建补丁，潜在的贡献者必须具备 Git 的基本知识。Git 是 Lucene/Solr 社区使用的版本控制系统，通过发起拉取请求可以轻松地进行贡献。

### 基本 Git 术语

#### Blob

Blob 代表二进制大对象。Blob 是在 Git 中存储和表示文件的一种通用方式。在 Git 控制的仓库中，任何被更改的文件都会在增量中表示为 blob。

#### 树

树是一个表示目录的对象。它包含 blob 和其他子目录。树是一个二进制文件，存储着对 blob 和子树的引用。（这些引用也称为树对象的 SHA-1 哈希值。）

#### 提交

提交保存了仓库的当前状态。每个提交都像是链表中的一个节点，每个提交都有一个指向其父提交的指针（从而允许构建一个层级链）。每个提交都以其内容的 SHA-1 值命名。如果一个提交有多个父提交，那么该特定提交是通过合并两个分支创建的。

#### 分支

分支用于创建另一条开发线。默认情况下，Git 有一个主分支，它类似于 Subversion 中的主干。通常，创建分支是为了开发一个新功能。一旦功能完成，它会被合并回主分支，然后我们删除该分支。每个分支都由 `HEAD` 引用，它指向该分支中的最新提交。每当你进行一次提交时，`HEAD` 都会更新为最新的提交。

#### 标签

标签用于标记仓库中的检查点。可以将标签视为给定分支的快照的只读分支。通常，标签用于标记发布版本，但它们也可以用于其他目的。

#### 克隆

克隆用于创建整个仓库的副本。克隆不仅仅是镜像；它是主代码库的一个功能完备的副本。通常，这些用于保留代码的本地副本。

#### 拉取

拉取操作将更改从远程仓库实例复制到本地仓库。拉取同步两个仓库，并允许将主仓库的更改同步到复刻中。

#### 推送

推送操作允许将本地更改推送到远程仓库。

## JIRA

JIRA 是 Apache Lucene/Solr 社区使用的问题跟踪器。JIRA 使用简单，是默认的社区互动机制。

当发现问题时，与社区互动的最简单方式是提交一个 JIRA。在提交 JIRA 之前，请进行适当的尽职调查，并在你的 JIRA 中包含以下内容：问题、重现步骤、建议的修复（如果有的话）等等。



### 编写补丁

如果你已经确定了想要在 Lucene/Solr 中修改的内容，并且对如何修改有了清晰的想法，那么编写一个补丁是合理的做法。补丁就是变更的增量。可以把它想象成你想要对代码库进行的“更改”。

具体做法是：首先在 Github 上 Fork Lucene/Solr 的代码库。然后，使用 `git clone` 克隆该仓库。接着，使用 `git checkout -b <foobar>` 创建一个分支。请记住，你提交到 Lucene/Solr 代码库的提交必须标记上你正在处理的 JIRA 的编号。因此，请确保你事先已经设置好了 JIRA 编号。

在你的仓库中本地完成更改后，执行 `git diff` 以确保代码是按照你的预期编写的，并且没有发生非预期的更改。

执行 `git commit` 来创建一个提交，并确保你已经将其推送（push）到你的 Fork 中。

### 测试

这里有一个重要的提醒：测试每一个变更，并验证你的补丁所影响的所有功能。你还需要确认所有现有的测试都能通过；不能有任何回归问题。

最好在项目上运行 `ant test` 以确保所有测试都能通过。如果某个特定的测试出现问题，请使用以下命令：

```
ant beast -Dtestcase= -Dtests.method= -Dbeast.iters=
```

### 编写良好的提交信息

虽然并非硬性规定，但在编写提交信息时，请遵循以下指南：

*   用一个空行将主题与正文分开。

*   将主题行限制在 50 个字符以内。

*   主题行首字母大写。

*   主题行末尾不要加句号。

*   在主题行中使用祈使语气。

*   正文每行限制在 72 个字符以内。

*   使用正文来解释“是什么”和“为什么”，而不是“怎么做”。

#### 编写文档

任何非琐碎的更改都需要附上适当的文档（包括代码、配置和参考文档）。如果做不到这一点，社区反馈将会不太积极。

#### 发起拉取请求

现在你已经完成了一个可供审查的补丁，可以发起拉取请求了。前往 Github 的用户界面，从你的仓库发起一个拉取请求。确保你关注上面的评论，积极迭代，尊重反馈，并持续改进。希望你的更改能被采纳。

#### 与社区互动

最好给 dev@lucene.apache.org 发送一封邮件，说明你的更改并征求反馈。通常，人们会给出建设性的反馈。请记住，最重要的目标是改进项目；因此，即使你的工作可能受到批评，也请将其视为建设性的批评。

做一个好的社区成员很重要，所以你要遵守礼仪，尊重既定的规范和标准。
