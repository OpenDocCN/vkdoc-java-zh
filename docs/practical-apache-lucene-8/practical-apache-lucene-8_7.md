# 7. 提升 Lucene 性能

本章重点介绍如何调优 Lucene 以获得最佳性能。你可以通过多种方式使用 Lucene，但核心性能理想情况下应聚焦于两个方面：索引和查询（本章将对此进行讨论）。

本章篇幅较短，旨在提供解决问题的指导，因此理论概念较少。本章内容建立在你之前章节所学知识的基础上。这里我们只讨论最佳实践。本章并未真正引入任何新内容。（因为你已经仔细阅读了本书至此的内容，所以你已经理解了这些实践背后的大部分原理。）




## 提高索引速度

我们先从速度提升说起：

*   **减少并发写入器的数量**。打开一个写入器，并在整个会话期间使用它。

*   过多的并发写入器会导致负载过高，从而普遍降低写入性能。你应该在搜索应用的整个生命周期中使用单一的写入器。

## 重用字段实例

要添加新文档，请创建 `Field` 实例，为其赋值，然后将它们分配给一个文档。在将文档添加到索引后，重新为字段赋值，将它们分配给一个新文档，然后添加这个新文档。

## 大型提交的特殊情况

`IndexWriter` 实例在合并大文件时可能会产生过多的开销。在这种情况下，切换到 `autoCommit=false` 可以确保最佳性能。然而，这种切换也会导致打开的搜索器在 `IndexWriter` 关闭之前无法看到更改。理想的方法是将数据分块成更小的块（批量处理你的文档），并保持 `autoCommit` 为 true。

## 在分析器中重用令牌

分析器会为序列中的每个词条创建一个新的令牌。

重用令牌是一个明智的策略。

```
StoredFieldsWriter.MergeVisitor.tokenStream(Analyzer analyzer, TokenStream reuse)
```

## 调整刷新间隔

Lucene 允许你根据不同的标准（如文档数量或 RAM 使用量）来定义刷新间隔。基于 RAM 使用量的刷新间隔可以更好地利用 RAM，并且在处理小/平均文档长度时，可以减少因频繁刷新而导致的磁盘抖动。请参见 `LiveIndexWriterConfig` 类。

## 增加 mergeFactor

小幅增加 `mergeFactor` 应该能提高索引速度。

## 选择正确的分析器

分析器在索引速度中扮演着重要角色。慢速的分析器会导致整体索引速度下降。

## 使用单个 IndexWriter 的多个线程

在 `IndexWriter` 中使用并发可以提高索引速度并更好地利用 CPU 核心能力。然而，存在一个临界点，所以要注意不要过度。该设置不应超过机器的核心数。请参见你的 `MergeScheduler` 实例的 `setMaxThreadCount` 方法。

## 分别索引到多个索引，然后合并

如果你有非常大量的内容需要索引，你可以将内容分成 N 个“筒仓”，在单独的机器上索引每个筒仓，然后使用 `writer.addIndexes` 将它们全部合并到一个最终的索引中。

## 提高搜索性能

本节和前一节并非互斥，并且可能取决于你在索引时关注的重点。请记住，索引和搜索通常发生在同一台机器上，资源重叠是可能发生的。

## 使用最新版本的 Lucene

如果你没有使用最新版本的 Lucene，你可能会错过升级以及 Lucene 提供的最新功能。

## 使用 readOnly 属性为 True 的 IndexReader

使用 `readOnly=true` 的 `IndexReader` 可以在一定程度上减少线程争用，从而提高性能。

## 使用 MMapDirectory/NIOFSDirectory

`MMapDirectory` 和 `NIOFSDirectory` 都能提供显著更好的 I/O 性能，从而提升整体性能。请注意，`NIOFSDirectory` 和 `MMapDirectory` 实现在 Windows 中会遇到文件通道问题。最好使用 `FSDirectory.open()`，Lucene 会为当前环境提供合适的 Directory 实现。

## 减少 mergeFactor

减少 `mergeFactor` 会导致段数量减少，从而减少扇出，进而提高查询性能。然而，这可能会对索引性能产生不利影响，因此需要谨慎操作。

## 忽略第一个查询的性能

第一个查询通常会遇到冷缓存，所以忽略它的性能。你需要一个更大的数据集来获得关于性能特征的整体答案。

## 避免重新打开 IndexSearcher 实例

重新打开 `IndexSearcher` 实例代价高昂。除非绝对必要，否则不要这样做。

## 共享 IndexSearcher 实例

延续上一点，共享 `IndexSearcher` 实例是一个好主意，这样可以减少争用。

## 谨慎使用存储字段和词向量

存储字段和词向量通常占系统整体内存使用量的很大一部分。不加区分地使用它们会导致更高的内存占用。

## 使用过滤器

过滤器可以减少 Lucene 为获取结果集所需命中项而进行的整体磁盘寻道。过滤器禁用分数计算并减少需要查看的数据集。将查询包装在 `ConstantScoreQuery` 中或使用过滤器。

## 关于最佳实践的最终思考

虽然这不是一个结论性的列表，但本章确定了一些提高 Lucene 索引和搜索整体性能的方法。请记住，过早优化很容易，然后会看到性能下降。更好的策略是首先使用默认配置构建你的搜索应用并测量性能。只有在识别出瓶颈之后，才应该专注于改进性能。
