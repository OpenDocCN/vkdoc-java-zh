# 11. 结论

我们的 API Marketplace 实施现已进入第五个年头。值得注意的是，如今平台的发展节奏与活力比以往任何时候都更加蓬勃、更加强劲。我真诚地相信，这个平台如今已经是一个有生命力、会呼吸、*几乎*能够自我维持的实体。当我查看自己的待办清单时，仍然有若干我已识别出的可优化领域。其中一个原因可归因于过去尚不可用的技术选项。我们始终在持续探索如何精简性能与执行——随着平台每次迭代都会出现新的路径和方法，这很可能是一场永无止境的追求。另一个重要因素是，后端产品团队如今非常热衷于将其服务发布给外部使用，而我们也有一份不断增长、已排期交付的 API 清单。

坦率地说，在写这本书的过程中，我曾质疑过：这些时间本可以用来继续培育平台。让我坚持写下去的原因，是希望通过分享过去四年的经验，尽力提炼出我们的成功公式——以帮助其他实施从零起步。正如我在多个场合提到的，每一个 Marketplace 实施都会是独特的，都会根据所在组织的需求与身份进行定制。你将要构建或正在构建的 Marketplace 极不可能采纳文中从平台架构到运营所列出的每一个概念或方法。不过，我由衷希望你在阅读本书时，书中提出的方案能够为你的平台创新提供基础、催化剂或触发点。

基于事后复盘的视角，以及对许多反复出现主题的细致观察，我坚信我们的公式中有一些关键要素，它们对实施成功至关重要。尽管其中许多并非技术层面，但它们对这一倡议的工程精神不可或缺。

## 团队动态

在第 1 章中，我强调了驾驶 Marketplace 星舰驶向全新且未知领域的关键角色。尽管这些角色可以由有才华的个人担任，但更关键的是这些角色必须作为一个统一体协同运作。在任何长期项目中，人力资源变动都是不可避免的。我们的情况也不例外，团队经历了多次变化，除了我自己和另外一两名成员之外，其他人都发生过更替。

这让我获得了一个独特的观察视角，见证了多种团队组合，有些成功，有些则不然。我们当前配置之所以最成功，简单来说可归因于出色的团队动态与文化。各种不同的个性，就像《星际迷航》的船员一样，彼此互补，更重要的是彼此平衡。很多时候，意见分歧会多于普遍共识。奇怪的是，这恰恰可能是成功团队文化的原因。通过积极、健康的讨论与辩论来审视多个视角，我们通常能找到最平衡的解决方案。反过来说，如果每个人都很相似，最终结果就会在某个特定方向上严重失衡。

在每一次讨论中，所有观点都会被听见并被纳入考量。这要归功于舰长出色的引导能力。如果无法达成共识，舰长会在综合所有输入后作出裁决。即使某位成员的方案没有被采纳，他们也会感到被安抚，因为他们有机会表达自己的看法。一旦确定了解决方案或目标，团队所有成员都会朝着同一方向发力。

> *整体大于部分之和（亚里士多德）。*

## 组织支持

我同样坚信，如果没有强大的企业级服务基础，我们不可能维持完整的运营能力，更不用说建立一个 API Marketplace。这并非我一直以来的看法。在构建最小可行产品（MVP）的早期阶段，我们曾将企业内其他部门视为阻碍成功的因素。第三方开发者利用组织能力这一理念，再加上我们激进的技术选型，远远超前于当时的标准实践。回过头看并加以反思，几年前我们项目团队那种任性与急躁的心态，其实相对不成熟。鉴于我们当时试图为组织构建一个全新的数字渠道，本应采取更审慎、更稳妥的推进方式。API Marketplace 位于传统企业边缘、用于连接第三方访问的这一位置特性，很可能既促成了项目团队选择前沿技术，也引发了组织层面的担忧。

项目团队从“烦躁”到“感激”的心态转变，以及组织其他部分从“怀疑”到“信任”的变化，发生在平台运营化的过程中。通过利用企业能力，我们继承了资深专家的经验与专业知识，这帮助平台从受控实验环境下运行，快速走向能够可靠承载真实世界工作负载的成熟状态。通过展现更多耐心与韧性，我们也得以更好地传播 API Marketplace 能为整个组织带来的能力与覆盖范围，并从企业各个领域争取到支持。

我个人和职业层面学到的一条关键经验是：试图绕开流程与治理，往往需要付出更多时间和精力。与其如此，不如给予组织应有的尊重，学习如何在现有流程中推进，更重要的是理解其功能与目的。很大概率上——你会发现这些机制是在保护组织及其用户。一旦你真正掌握了流程及其作用，再尝试围绕敏捷性进行优化。

> *首先要寻求理解。*



## 灵活转向的能力

在我们搭建平台使命推进的过程中，曾有几次遭遇重大挑战，有时甚至威胁到整个项目的存续。最典型的例子包括：我们首个 API 产品的关键后端提供方因监管要求在最后时刻退出；以及一家知名第三方使用方因接入新 API 遇到困难，极有可能错过一个关键年度目标。

第 1 章讨论过的快艇式运营模型，使项目能够迅速转向并快速调整方向。有意思的是，为了缓解失去关键后端的影响，我们选择了组织内一个重要性和可见度都低得多的系统作为替代，因为它提供了类似但使用频率更低的功能。起初，项目团队对未能借助那个更高曝光度的后端高调上线感到失望。随着时间推移，这种情绪很快转为庆幸，因为我们观察到了支撑在线 API 的各种挑战，也看清了自己在从原型过渡到落地执行方面准备不足。坦白说，如果当时上线的是那个更高关注度的 API，我们将没有足够时间稳定运营平台。

除了认识到我们的 API 设计应当更简洁、以及需要积极推动第三方开发者参与之外，这次险些错过关键里程碑日期的“重磅风险”还催生了一种新的集成模型。该模型使平台能够在可能的地方吸收复杂性，从而加速 API 采用。此后，这一模式已成为我们 Marketplace 的基石，并被积极用于未来的产品设计。这次经历带来的积极启示是：API Marketplace 的实施并非绝对的非黑即白。秉持真正的创业思维，团队和平台应尽一切可能促成并维持第三方采用。如果这意味着在开发者线下活动中发放 T 恤和马克杯来激发兴趣，或是构建一个用户界面来支持 API，

> *做必须做的事。*

## 以开发者为中心

正如在商业化章节中讨论的那样，驱动 Marketplace 的关键要素是第三方开发者。我可以毫无保留地说，一切都应为第三方消费而存在。即便你的 Marketplace 拥有最广泛的 API 产品，并建立在卓越且突破性的技术平台之上——如果没有*活跃的*第三方参与，这项计划就注定失败。

API 在构思与打造时，应把第三方的需求、兴趣和能力放在核心位置。技术团队可能会沉迷于“炫技感”和“可夸耀性”——例如通过由 Unix 操作系统创始人开发的摩尔斯电码协议去集成遗留主机系统。但如果它暴露的功能对组织外部消费者毫无兴趣或价值有限，这种投入就是浪费。产品负责人肩负着一项艰巨任务：在“半自我陶醉、坚信其遗留系统具备颠覆性能力”的后端提供方，与“急于打造下一个爆款应用”的多变第三方消费者之间弥合数字鸿沟。

前文已详细讨论的 Sandbox 概念，是一个能够且必须用于促进这种互动的工具。它不仅仅是支撑第三方测试的机制。它还可用于衡量第三方兴趣与需求、塑造 API 定义，更重要的是，作为超敏捷外部消费者与更稳健、治理完善、传统企业生态之间的缓冲层。

最后，也可能是最重要的一点：开发者门户是你 Marketplace 的门面，代表着第三方与平台的首次交互。下次当你站在实体店门前，或访问一个新的电商门户时，不妨思考一下促使你进一步参与的决策过程。带着这种视角上线你的开发者门户，并从第三方消费者角度去审视它。这应由团队所有人持续且坦诚地进行，甚至可邀请朋友、家人及其他团队同事参与，以尽可能获得更多反馈与评审。

尽快与你的第三方社区建立互动，以收集反馈并理解你的市场。你所在区域的开发者或第三方可能需要帮助与支持才能使用你的平台。请务必牢记这一点。

> *开发者是你宇宙的中心。*

## 从哪里开始

到现在为止，构建 API Marketplace 这项任务可能看起来极其困难。毕竟，你是在为组织建立一个全新的数字渠道，而这种能力需要由行星级规模的平台架构作为底座。更不用说 API 本身还必须设计精良、遵循敏捷原则开发，并具备企业级支持能力。

此时，我想与你分享我的第一编码原则：

> *保持简单，迭代构建。*

本书中分享的原则、平台架构、设计方法、敏捷开发与运维流程，都是经过*多年*演进而来，并且仍在持续进化。我们的实现始于一个 API 产品，用于展示 MVP。如果让我重来一次，我会把这个原本就不大的范围再大幅缩减。完全可以，甚至应该，从少量 API 产品起步来构建可持续的 API Marketplace。这些产品可以服务于细分功能，平台也完全可以运行在遗留架构上——只要它稳定。请限制初始 API 产品数量——我建议最多从两个开始。抵制把开发者门户（门面）堆满产品的诱惑。以一名开发者的朴素观点来看，一个完整可用的 API，远比十个长期“即将上线”的 API 更令人信服。

产品少一些，也更容易进行后期改造——包括性能优化、监控增强和运维可观测性提升。贯穿本书的一条关键经验，也是我多年集成实践中反复验证的一点：接口是为更大的解决方案服务的。只要它稳定、可预测、可靠，消费者其实并不在意它是用坠毁地球的外星飞船高级动力核心驱动，还是用建造金字塔时代的古老工艺实现。让你的平台架构随时间演进——与 Marketplace 的需求和要求保持一致。再次提醒，请克制，因为这往往是一个“上最新最炫技术”的诱惑点。

设定一个时间框架，尽可能短到三个月，最长不要超过六个月，发布你的第一个 API，并在此基础上持续迭代。

## 保持联系

在撰写本书时，我借鉴了多年的经验，主要是跨广泛项目与场景的集成实践。技术工作的奇妙之处之一在于：解决同一个问题往往有很多种，甚至无限多种方法。这也是我每天都迫不及待打开笔记本的原因——因为我会得到一个新问题，并以*我的*方式去解决它。我尤其希望了解你的 API Marketplace 之旅，以及你和团队为应对挑战所采用的方法——也许和我们一样，也许（而且很可能）完全不同。欢迎随时保持联系，告诉我你对本书任何内容的想法、建议、反馈，甚至反驳意见或批评。

我相信，只有通过分享与协作，我们才能不断前进。期待你的来信。
