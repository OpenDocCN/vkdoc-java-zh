# 1. API 工程

API Marketplace 是一个让组织与第三方开发者共享内部服务的平台。应用程序编程接口（API）是实现这种访问的机制。能力会被打包成不同的产品——例如，访问客户数据或在 GSM 网络上发送消息。它很像现实世界中的 Marketplace：这是组织（卖方）与第三方（买方）之间的汇合点，产品在这里被展示并被消费。

有些 API Marketplace 项目出于必要而启动——为了满足监管合规；有些是为了追赶竞争对手；还有一些则纯粹是演进驱动。我们的项目始于一个愿景。目标是赋能第三方提供商，利用我们企业的能力，更好地服务客户需求。从利他的角度看，这会为所有参与者带来积极结果。组织获胜——通过消费增长带来收入。客户获胜——通过创新获得更优服务。同样重要的是，第三方也获胜——能够利用成熟的组织能力来构建新的产品或服务。

作为我们市场乃至国家中的先行者，我们并未被项目截止日期或监管时限的枷锁束缚，也没有竞争对手在身后步步紧逼。从一开始，我们就清晰地意识到，API Marketplace 是一个巨大而稀有的机会：它可以构建一个与外部伙伴协作的全新数字渠道；但这项举措也必须能够自我持续，并创造收入。我们很幸运，获得了一张白纸和足够的空间，去做出一些超出组织常规标准的技术决策。我们的关键经验是：Marketplace 的基础要素是技术、流程，以及最重要的——人。

在本章中，我们先从赋能性的技术要素——API——讲起，讨论它能带来的能力，以及大型组织成功使用它的方式。接着我们讨论 API Marketplace 的流程、其目的，以及它在整个组织中的运作方式。然后我们转向“人”的要素，讨论支撑 API Marketplace 落地实施的关键角色。

## API 的实践应用

每天晚上睡前，我都会走到家中散落的众多 Amazon Echo 设备中的一个旁边，轻声说一句“*Alexa，晚安*”。电视、音视频（AV）设备和灯光都会关闭，Alexa 还会轻声说一句温馨的话向我道晚安。希望有一天我也能拥有一把智能门锁，把它也纳入这套流程。对外行人来说，这看起来似乎很容易实现——但这其实是集成工程的高光时刻：简单、可靠、易用。而这一切都由 API 提供底层支撑。

我对 Alexa 说的话会被发送到语音识别服务，由它把语音转换为文本。随后，自然语言理解（NLU）服务会把文本转换为 Alexa 意图。该意图会触发智能家居技能，通过 API 与我的万能遥控器和灯光自动化系统交互。看似一个简单请求，却触发了一连串可能跨越多个大洲的活动。下面我们聚焦到这个场景中的灯光自动化部分。

在灯光自动化中，有一个硬件继电器会响应无线电信号，在“开”或“关”状态之间切换。为此，制造商设计了发射器和接收器。这样一来，我就不必使用传统那种嵌在墙里的物理开关，而是可以通过移动发射器发送请求。制造商还提供了通过移动设备发起请求的能力。坦白说，这正是我选择这款继电器的关键需求之一——因为能远程开关灯，尤其是在朋友来家里聚会时，是每个技术宅的梦想。不过，真正让我拍板的，是产品文档角落里一个不起眼链接中对 API 的随意提及。

这个 API 允许通过 HTTP 直接以软件方式集成继电器，而无需使用制造商的专有硬件接口。通过开放 API，继电器的潜力显著提升。触发事件不再局限于遥控发射器，甚至也不局限于移动应用。它可以连接传感器，用于判断光照强度、运动或占用情况；也可以连接 Amazon Alexa 或 Google Assistant 等外部服务——并通过语音命令触发。可能性是无限的，而那个隐藏在电气管道里的普通继电器，如今已成为更大生态系统中有意义的一员。

而让这一生态系统成为可能的关键组件，就是 API。



## 什么让 API 如此特别？

尽管这个灯光场景看起来相当吸引人，很多人也可能会被“用软件关灯”的能力所启发，但读到这里的硬核工程师可能并不会轻易被打动。毕竟，这种“接口对接”能力也许多年前就已经存在了；如果你拆开外壳，露出底层电路板，板子上可能有跳线，只要给 PIN#1 加上电流就能实现这个能力。API？没什么了不起。

虽然灯光场景是可行的，而且也可以说它在*很多年*前就可行了，但真正具备技术敏锐度和实现能力的人其实只占很小一部分。而在这部分人里，愿意去给电路板引脚加电流的人更是少数。我至今仍能痛苦地回忆起，多年前我曾想在一次自动化构建结束时点亮一个熔岩灯来表示构建失败，但当时我做不到，因为那时我还无法轻松地从软件跨到硬件。

和任何推动普及的技术一样，API 让大众都能获得访问能力。API 之所以能做到这一点，是因为它易于使用、技术无关，而且文档完善。API 被有意识地基于 HTTP 等构造进行建模，而 HTTP 正是互联网通信的基础。使用中立平台可以让接口两端采用不同技术。虽然这看起来可能微不足道，但系统间集成要么一直是严格专有的，要么所谓中立化的尝试都异常复杂。凡是做过通用对象请求代理体系结构（CORBA）的人，都会认同：搭建客户端与服务端通信绝非易事。

JavaScript 对象表示法（JSON）也是这一变革中的关键组成部分。它提供了一种轻量的数据交换机制。它的前辈，比如简单对象访问协议（SOAP），多年来一直试图通过可扩展标记语言（XML）来实现这一点。然而，JSON 的可塑性和容错性都要高得多。关键在于它对人类来说是可读的（*human*）。当然，角落里那位端着黑咖啡的硬核工程师大概会低声嘟囔：XML 也是*人类*可读的。区别在于*简洁性*——根据我的亲身经验，浏览一串键值对，远比追踪带嵌套值的 XML 起止标签要容易得多。

我最喜欢的故事之一是巴别塔。最吸引我的部分是：人们说着共同的语言，并且彼此理解。这带来了*巨大的*潜力。尽管几个世纪以来，牛顿、爱因斯坦等人的个人贡献推动了惊人的进步，但通过协作和信息共享，我们可以实现更多。再次强调，底层前提是沟通的*便捷*。沟通应该像说话一样自然，而 API 正是朝这个方向迈出的坚实一步。

网站和移动应用已经彻底改变了我们获取信息的方式。用户界面（UI）和用户体验（UX）定义了界面的导航与流程。API 是前端的重要赋能者，并使 UI 与逻辑之间能够清晰分离。

## 连接、共享与生存

毫无疑问，互联网实现了前所未有的连接水平。我难以想象前几代人是如何在伴侣因商场排队而晚归时，毫无更新地耐心等待的。也难以想象他们在没有 GPS 应用的情况下，只靠纸质地图开车到处转，努力辨认街道名称，或者不得不向陌生人问路。在互联网走出学术圈的第一波浪潮中，组织通过建立网站来确立存在。今天，几乎每个组织都需要一个网站；没有网站，就难以建立身份与信任。

这随后从一个“联系我们”表单，逐步演变为完整的电子商务能力，使终端用户能够直接与供应商互动。智能手机也催生了应用，让最终消费者与服务提供方直接交互。幸运的是，由于设备价格低廉以及移动数据的广泛可用，全球人口中很大比例如今都已接入网络。这也催生了一股共享浪潮。

像 Airbnb 这样的服务让人们可以出租空余房间甚至整套房屋。像 Uber 和 Lyft 这样的网约车服务使用起来极其简单，并持续在低价出行、包裹配送等选项上不断创新。这促成了一种范式转变，因为它可能让人们不必再对房产和机动车进行长期投资。未来几代人甚至可能不再需要驾照，因为相比拥有汽车，使用网约车也许更经济、更实用。

从行业角度看，组织之间的边界正在变得模糊。东非最成功的移动支付服务之一，是由一家电信公司运营的。医疗服务提供方也已进入银行业务竞争，开始提供金融服务。银行也并未固步自封，如今正以移动虚拟网络运营商（MVNO）的身份提供电信服务。

尽管进军新行业是可能的，但对已成型的组织来说，要做到这一点困难得多。不过，协作与伙伴关系是另一条路径，能够帮助成熟组织完成转型。虽然这看起来可能有些“反生存”，但与初创公司和潜在颠覆者合作，反而可能成为组织得以生存的机制。“*打不过就加入*”这句箴言确实有道理，许多组织如今都在与初创公司合作，并参与训练营，以孵化全新而鲜活的视角。就像今天我们已难以想象一个组织没有网站或移动应用一样，未来没有可供使用的 API，也会被以同样的眼光看待。



## API 宣言

API 并不是什么新鲜事物。这个概念多年来一直存在，并已在企业内部用于实现应用程序之间的集成。2002 年，杰夫·贝索斯发布了一项 API 强制令，有时被称为“API 宣言”，内容如下：

今后，所有团队都必须通过服务接口公开其数据和功能。

*   团队之间必须通过这些接口进行通信。

*   不允许使用任何其他形式的进程间通信：不得直接链接，不得直接读取其他团队的数据存储，不得使用共享内存模型，也绝不允许后门。唯一允许的通信方式是通过网络上的服务接口调用。

*   你使用什么技术并不重要。

*   所有服务接口，无一例外，都必须从一开始就按可对外化的方式进行设计。也就是说，团队必须进行规划和设计，以便能够将接口开放给外部世界的开发者。没有例外。

*   该强制令最后写道：凡是不这样做的人将被解雇。谢谢，祝你今天愉快！

任何拥有体面集成战略的组织读到这些内容时，可能都不会太过担心。毕竟，中间件本就是为了实现上述所有目标——抽象、清晰接口。然而，宣言中的“*外部世界的开发者*”这几个字才是改变游戏规则的关键。

将你的组织向外部消费者开放，很可能会在各个层面引发不小的紧张情绪。架构师可能并不愿意与外部开发者共享他们神圣的通用数据模型。开发团队也许会强作镇定，但很可能在想着那些被他们以“技术债”名义藏在柜子里的“骷髅”。也请体谅一下运维团队，他们当时可能还在四处张望，怀疑这是不是整蛊节目。别忘了，那是在 2002 年，扩容额外云容量这件事大概还只是白板上的几张简略草图。挑战已经抛出，团队必须遵从。

亚马逊 API 宣言 i) 是企业范围内 API 思维范式转变的一个例子；ii) 也是这种转变能够带来显著成果的证据——看看今天亚马逊云服务（Amazon Web Services）的成功便可知。

## 快艇与航空母舰

尽管合作与协同的意愿可能很强，传统组织在实现这一点时却可能犹豫不决。这种犹豫可能源于冗长的项目周期，或与变更和发布管理相关的治理流程。通常也可能是因为组织内部深层功能不易向第三方开放。从长期来看，这可以通过跨企业的敏捷工作方式得到改善。然而，短期内也可以通过在组织边缘建立一种能力来取得成果。

我将这种能力称为“API 市场（API Marketplace）”，它是成熟企业与第三方提供商之间的交汇点。它能够以敏捷方式借助企业的强大能力来满足高强度需求。Martin Fowler 将其称为“双杠杆（two-lever）”方法。这种能力以人员、流程和技术原则为基础支撑。我们将在本书后文更详细地讨论这一点。

第一个杠杆代表传统企业，像航空母舰一样，运行平稳且坚定。第二个杠杆则像快艇一样，灵活敏捷，并且由于不承担同等的组织责任，能够更快改变方向。我们的 Marketplace 计划曾多次转向。尽管障碍会干扰交付，但我们的灵活性使我们能够迅速调整方向并持续前进。在最小可行产品（MVP）阶段，我们原计划集成的一个后端平台因组织层面对 API 访问的担忧而被突然取消，因为这可能会影响现有访问渠道的收入流。在通常情况下，这足以对该计划造成致命打击。凭借良好的领导力和决心，我们找到了一个类似但知名度较低的后端，并以更少的挑战完成了集成。这也要求我们围绕这个新后端塑造一个新的 API 产品。

最关键的经验是：从一个较低关注度的 API 产品起步，来建立平台基础，可能是明智的——同时避免引发组织层面的担忧（例如蚕食现有收入流）。较低关注度也可能意味着较低的调用量。这正好提供了一个绝佳机会，用于定义并完善围绕 Marketplace 的运营流程与支持体系。在这个快速变化的领域中，尽快发布并被消费一个 API 产品非常重要。

## 集成模式

集成一直都可以在不同层级上实现。组织之间通过定义良好的接口进行连接，这些接口可能是为实现特定功能而构建的。此类通信通常也会通过专用网络通道（如专线）进行高度安全防护。这通常被称为 *企业对企业*（B2B）集成。由于这类集成具有高度敏感性，这些接口通常只对少量合作伙伴开放。

然而，*企业对消费者*（B2C）集成模式允许组织将客户数据开放给众多*第三方*提供商，但*仅*在获得客户许可的前提下。这一模式彻底改变了集成格局，因为作为数据真正所有者的终端客户可以决定谁有权访问数据。使这一点成为可能的安全模型称为 *OAuth*，我们将在本书后文更详细地讨论。这也促成了接口标准化，因为若为每个提供商分别维护独立服务契约，将是不可持续的。



## 企业级影响

在传统的企业应用集成（EAI）中，消费者和提供者都处于组织内部相对安全的边界之内。尽管仍然建议采用内部安全机制，但“内部网络”这张安全网有助于缓解这样的担忧：集成仅对已知消费者开放。而通过构建 API 市场，集成将*主要*面向*外部*消费。

这是一个重大的变革。尽管应用团队可能会满足于通过 API 网关来封装接口，但这会产生企业范围的影响，并且需要多个团队参与——从信息安全、网络到取证团队，仅举几例。信息安全团队需要确认客户或组织数据只会在特定安全授权框架（如*OAuth*）下被释放。需要特别强调的是，信息安全团队是 API 市场的关键利益相关方，且这种协作应持续进行。信息安全团队应审核每一个 API 产品，以确保在正确的安全级别下，将恰当级别的信息提供给正确的对象。从技术或开发角度看，暴露或更新一个 API 产品以提供额外数据似乎相对容易。然而，信息安全团队对数据敏感性有更全面的视角，且作为企业与客户信息的守护者，必须始终被咨询。

网络团队也必须决定如今源自互联网的请求如何穿越组织边界并路由到内部服务。在这个节点上，有必要暂停一下，从网络管理员的视角思考建立 API 市场这件事的分量。

假设你是特勤局负责人，负责美国总统的安全。如果总统一直待在白宫范围内，你的工作很简单——白宫的边界参数是明确的：入口、出口、工作人员、访客等等。然而，一旦总统离开白宫，你的工作本质上就会变得更复杂，因为需要应对大量重要变量。

网络安全团队也是如此。自组织建立以来，网络边界参数一直定义明确。防火墙将所有不受欢迎的用户和流量拒之门外，保障内部网络安全。外部用户来自定义清晰的 IP 源，可能通过专线接入。而现在，API 市场计划出现了，它通过*Internet*向外部开发者开放公司资源！因此，让网络安全团队保持参与非常重要，最好从一开始就参与，以确保必要的思维转变到位、顾虑得到缓解，并且可能需要的技术变更能够被及时规划和落地，从而避免延误。

同样重要的是让取证团队参与进来，因为他们会利用每笔交易的审计数据来还原经由第三方向企业发起的客户请求。这在客户咨询或争议发生时至关重要，并可能为组织节省大量资金。对追踪信息进行实时分析，还可用于识别并阻止欺诈请求。取证团队还可以利用其经验，为攻击可能如何发起提供洞见。

组织内还有其他应当咨询并告知的团队，例如产品管理。关键结论是——尽管 API 市场可能运行在组织的*边缘*，但它依赖的是企业*内部*能力的支撑。利益相关方管理至关重要，也是我们平台的使能能力之一。教育是关键——不要想当然。在我们所有协作的开始阶段，都会创建并反复播放一段 YouTube 视频，介绍 API 的定义、使用方式及其收益。通过这样做，我们的受众获得了知识和上下文，也更愿意参与协作。

## 新数字渠道的分量

作为开发者，我很确定此时开发团队会发出不满的抱怨。信息安全、网络和取证听起来都像繁文缛节——在技术团队眼里，这就像超人看待氪石一样。可以理解——氪星是他的出生地，但如今它会削弱他的力量。繁文缛节也是同理——必要，但消耗精力。

根据我们的经验，建立 API 市场这项令人兴奋的工作很快滚雪球般演变成一项以技术为主导的计划。我们的唯一目标是暴露一个 API，在客户认证与授权的前提下提供企业数据。最初项目授权也助推了这一点：向组织管理层展示一个 MVP。

在新成立的数字创新实验室这个“高氧泡泡”中，技术交付团队争分夺秒地完成任务。过程并非一帆风顺——当我们遇到那些*必须*使用的组织级平台与系统时，时常会撞上“泡泡墙”。例如企业 API 网关，在那之前它仅用于内部消费。然而我们在每个检查点都亮出“MVP”和“概念验证（PoC）”徽章，再加上一些快速沟通和对后续迭代遵循企业标准的承诺（虽然多少有点“背后交叉手指”的意味），团队最终还是绕过了这些挑战。

就像几乎所有处于“泡泡”中的技术交付团队一样——我们交付了。

我们以破纪录时间完成了看似不可逾越的任务而激动不已，这大概也得益于吸入了高浓度氧气。接下来我们问的是：这项能力*何时*可以对外部开发者上线。交付团队*预期*的时间线是几周，或许一两个月；而*实际*时间线更接近九到十二个月。如此漫长的周期很可能会浇灭任何开发者热情。我们搬离数字创新实验室、为新的 MVP“新秀”腾位置，这也让空气中的氧含量回归正常。很快我们意识到：在 API 市场运营化过程中，技术交付只是需要考虑的一个维度。我们实际上是在为组织构建一个新的*渠道*！

开放访问是一场渐进式转型。举例来说，看看多年来银行机构访问方式的演进。最初它们依赖高度安全的实体网点，且仅在特定营业时间开放。随着自动取款机（ATM）的引入，用户可以全天候取现——但要建立这项能力，设备本身必须高度安全且管理完善。如何把现金输送*到*ATM 里也必须被考虑，这不仅是安全问题，还涉及机器地理分布带来的物流问题。在“实验室”环境中这概念看似简单——在真实世界中却完全不同。

其演进旅程的另一步是通过网页与移动应用渠道提供访问。这里面挑战无数——我只强调一个在“实验室”环境中很可能从未出现的问题——网络钓鱼。只需假装是金融机构员工给终端用户打电话，就可能轻易通过社工手段骗取客户登录凭据。如今双因素认证（2FA）等机制已经很普遍——但在该渠道最初建立时，这些机制并不 readily 可用。



借助于建立其他渠道所获得的经验，组织对潜在攻击向量有了更多洞察。对银行实体建筑和 ATM 的攻击也许更容易防御，因为威胁是可感知、可触达的。对于 API Marketplace 这样的数字渠道，攻击者可能身处世界另一端。ATM 最多只会损失现金；而 API 可能泄露客户数据，在当今《通用数据保护条例》（GDPR）的环境下，这可能给组织带来更高昂的声誉损失。

建立一个数字渠道并非易事。需要考量的不仅是开发或技术视角，还包括广泛的多元视角。一次安全事件就可能造成声誉受损，而对平台失去信任则可能进一步影响 API 产品路线图，甚至危及其存续本身。

在所有渠道中，API Marketplace 可能是上市速度最快的——它不受 ATM 那样必须具备实体场所的约束，也不像 Web 一样受限于用户界面。然而，速度始终必须以责任为前提加以平衡。

## 开发者生态系统

API Marketplace 的存在是为了服务第三方开发者。没有第三方使用，就不会有流量；没有流量，就没有收入。收入维持着 Marketplace 的运转，而 Marketplace 的存在是为了实现愿景。可用的 API 产品应始终保持可靠且可用，以维持现有使用者。定义清晰的路线图将有助于吸引新开发者加入平台。

API Marketplace 的开发者门户本质上是平台的门面，因此应在此投入大量精力并保持高度关注。尽管其主要目的是支持技术集成，门户也应面向广泛受众进行设计。一个考虑使用该平台和 API 产品的初创公司，可能希望先获得一个宏观层面的整体认知。视频和客户案例研究可以轻松传达这一概念。同样重要的是要覆盖不同层级的开发者技能水平。不要想当然，像*OAuth*这类内部交付团队可能轻松理解的概念，也需要去神秘化并加以说明，以确保理解与采用。

API Marketplace 的一个内在优势是沙箱环境。关于如何实现与管理该环境的技术，将在后续章节讨论。从外部开发者视角看，它允许在开发生命周期早期就开展开发和集成测试。开发者访问与注册该环境必须通过自助方式并实现全自动化。有些开发者很有韧性，但多数可能并非如此——访问便利和易用性是刚需。门户应有完善文档，提供用于展示流程流转的时序图、分步指南以及代码示例。此内容也会在后续章节讨论。

## 角色

在我们最初启动、并最终促成本书诞生的项目时，我们从一个技术导向很强的团队起步。成员包括负责门户的前端开发者、整合后端接口的集成开发者，以及构建本地容器平台的 DevOps 工程师。事后回看，我如今理解了敏捷教练当时的挫败感：他不断尝试让团队关注更大的目标，而不只是完成下一个迭代冲刺目标。我们的项目团队成员来自多个国家，横跨不同大洲与时区，有些日子我们花在网络会议上的时间甚至比陪伴家人的时间还多。

随着 Marketplace 的演进，我们的人员配置也在变化。受益于时间积累，我们有幸尝试了多种团队组合——从纯粹初创形态到与传统团队角色协同。由于产品需求增长，我们需要建立产品小队（squad）——每个小队负责一个特定领域。随着 Marketplace 从初创阶段成长为运营平台，以下核心角色构成了团队支柱：

*   项目执行负责人

*   产品负责人

*   交付负责人

*   运营负责人

*   工程负责人

正如《星际迷航》的粉丝会认同的那样，这些角色对应着“企业号”星舰上的分工。每个角色都能力卓越并承担关键职能，整体协作带来任务成功。其运行极为出色之处在于：每位成员都拥有充分自主权去执行工作，同时始终得到其他船员支持。可用人员会被调配来协助交付目标——例如，在调查争议时，产品负责人可能会查看审计追踪记录，帮助运营团队拼接客户旅程。团队共同要求是：每个人都应卷起袖子，深入细节。

作为平台的船长，**项目执行负责人**根据组织数字化领导层设定的目标来掌舵。尽管 API Marketplace 像一艘快艇那样灵活迅速，它仍是更大舰队的一部分；该角色必须谨慎平衡成熟企业偏稳健的视角与敏捷但有时仓促的交付团队之间的关系。执行负责人也是组织内部 API Marketplace 的倡导者，并承担一项吃力不讨好的任务：为项目建设和早期阶段争取资金——直到平台能够自我维持。确保 API Marketplace 被视为企业严肃的数字渠道也是一项重大挑战。必须在内部有效“售卖”其概念与能力，因为如此具有变革性的理念很容易被归类为研发尝试，从而得不到足够重视。随着采用率和收入提升，这会变得更容易。

**产品负责人（PO）**负责塑造 API 产品。技术交付团队可能会低估第三方使用 API 产品所需面对的复杂度。PO 理解第三方开发者需求、企业内部能力，并识别如何将能力映射以满足需求。同时也要兼顾商业价值，因为 Marketplace 需要实现自我造血并创造收入。尽管许多 API 产品可以基于监管要求预先定义，但仍有充足空间去定义并发布更多产品。PO 与项目执行负责人紧密协作，共同定义 API 产品路线图。组织方向是一个输入，第三方需求/要求是另一个输入。该角色的另一项职责是在企业内部向各团队推广 API Marketplace 的潜力，并共同塑造产品供给。这并非易事，因为往往需要该内部团队提供部分资金来建设相关产品。



**交付负责人（Delivery Lead）**的职责是确保产品路线图的目标和任务得以实现。其职责范围包括基于可用开发产能，对来自产品负责人的新增需求进行排期。此人还需要与后端平台团队紧密合作，以确定可供使用的接口何时可用。这有时是一项微妙的平衡工作，因为后端团队的任何延迟都可能导致开发时间损失。在与离岸交付团队协作时，交付负责人还肩负着一项并不轻松的任务：在远程与本地人员之间进行协调。基于对后端可用性和交付能力的洞察，有时也有必要调整 API 产品的发布日期与更新节奏。目标是让平台始终保持推进。在开发过程中，严格执行双周生产发布（production drops）会带来巨大帮助。这些发布可以有意安排在交付冲刺结束时进行。发布不仅能最大化开发团队的潜力——团队也清楚他们的努力在持续进入生产环境，任何由此产生的问题或更新都可以快速处理。

**工程负责人（Engineering Lead，EL）**横跨平台的多个技术领域开展工作。对技术栈中每一层都了然于胸是一项优势，因为为了解除阻塞，往往需要深入细节并亲自介入。API Marketplace 也为我们提供了一个独特机会，可以在企业内部测试新方法与新技术。EL 的职责是对技术版图有深入理解，并且常常需要与组织内的支持团队分享这一视角。另一个职责——就我个人而言也是一种荣幸——是在团队其他成员及后端平台团队的共同输入下，定义满足新需求的解决方案。从治理角度看，EL 还需要与企业架构师、领域架构师和解决方案架构师协作，以获得平台与方案设计的审批。EL 必须具备端到端解决方案的全局视角，并能够将其拆解为团队可交付的工作项。理想的 EL 是能够并且经常深入细节的人——从编写代码快速实现概念验证，到对接新的后端平台接口，再到在集成测试期间协调开发人员之间的协作。EL 还负责界定交付工作量与时间线，并在工程团队的协助和输入下推进。经验在其中起着重要作用，因为它有助于支持 API 产品路线图的制定。

最后，**运营负责人（Operations Lead，OL）**本质上是团队的枢纽。尽管其主要关注点是 Marketplace 执行平台的可靠性与稳定性，但该角色还对组织内支撑服务与后端平台的可用性承担责任。Marketplace 支持团队向运营负责人汇报。稍后会讨论的报表与分析仪表盘，是该角色的核心支撑。对于平台的日常运行而言，故障恢复、集成以及与外部团队的协同是关键要求。理解复杂性并深入细节的能力同样是必需的，因为这使其能够知道在*何时*需要拉通*哪些人*。从创收角度看，这可能是*最*重要的角色，因为停机就意味着机会流失。

最后，必须强调以下几点：这些角色是相互联结的。产品负责人在判断潜在 API 产品是否可行时，会得到工程负责人的支持。同样，工程负责人也依赖产品负责人将交付延迟传达给项目高管。团队层级也被压缩到最低限度，所有成员都可以自由地与上述任一角色协作以实现目标。再次强调，将团队凝聚在一起的是相互信任与尊重，更重要的是，我们共享同一个愿景。我们所做的一切，都是朝着实现该目标旅程中的坚实一步。
