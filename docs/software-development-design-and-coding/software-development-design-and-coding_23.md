# 20. 总结

> *所有程序员都是乐观主义者。或许正是这种现代魔法，特别吸引那些相信美好结局和仙女教母的人。或许正是那数百个令人沮丧的琐碎问题，驱走了所有不习惯专注于最终目标的人。或许仅仅是因为计算机还年轻，程序员更年轻，而年轻人总是乐观的。*
> 
> ——小弗雷德里克·布鲁克斯^(³⁰⁸)

> *这是我能想到的唯一一份让我同时成为工程师和艺术家的职业。它有着令人难以置信的、严谨的技术元素，我喜欢这一点，因为你必须进行非常精确的思考。另一方面，它又有着狂野的创造性，在那里，想象力的边界是唯一的真正限制。*
> 
> ——安迪·赫兹菲尔德

阅读亚历克斯·E·贝尔^(³⁰⁹)和马克·古兹迪尔^(³¹⁰)在 2008 年 8 月《ACM 通讯》上发表的“观点”专栏时，我们被这两篇文章的协同效应所震撼。一篇是关于专业软件开发中应使用工具的警示故事，另一篇则至少部分是关于编程教学中语言和语法使用的警示故事。这让我们思考了在开发和教育中尝试过的所有“银弹”，以及为什么它们中的大多数对真正的软件开发无关紧要。这似乎是总结这场关于软件开发的深入讨论的恰当方式。

## 你学到了什么？

正如我们在本书中一直坚持的那样，软件开发是困难的。它需要大量的奉献和实践才能完成，而要做得更好则需要更多。当然，这正是它的吸引力所在。简单的问题很无聊，没有神秘感，没有追逐的刺激，也没有成就的回报。挑战在于处理你从未做过的事情，甚至是你不知道能否解决的问题。这正是让你一次又一次地回到软件创作谜题中的原因。

软件开发是人类能做的最具创造力和最强大的事情之一。从无到有，你接受一个问题，与之搏斗，探索它，戳弄它，将其拆解，以不同的形式重新组合，迸发出解决问题的灵感，然后将其全部转化为他人可以轻松使用的成品。用你的软件满足现实世界的需求，这简直是最酷的事情。

编写软件也是一种令人谦卑的经历。把软件做对是如此困难，而做错却如此容易。在编写软件的过程中，你学会了拥抱失败。失败是过程中令人兴奋又沮丧的一部分。失败教会你关于问题的知识，也教会你关于你自己：你学会了如何解决问题，你容易犯哪种类型的错误，以及如何规避它们。失败教会你坚持不懈，因为你不太可能第一次就做对所有事情，所以你只能不断努力，直到程序能正常工作。

大多数软件是由小团队构建的，而他们构建的也是最好的软件。规模小、积极性高且被充分授权的团队效率最高。小团队也倾向于使用精简的开发流程。除非你为一家迫切希望达到 SEI 能力成熟度模型 5 级^(³¹¹)的大公司工作，否则你的流程可以非常精简。详细的问题描述（最近以用户故事的形式出现）、头脑风暴设计会议、简单的配置管理、同行代码审查以及独立的测试团队，就能处理创建几乎无缺陷代码所需的一切。流程的灵活性、有效的沟通和共同的产品所有权是项目成功的关键。

每年都有大量真正优秀的软件被编写、测试并发布；其数量远超所谓的“失败”数字所让人相信的程度。^(³¹²,³¹³,³¹⁴) 计划驱动开发和敏捷开发的主要驱动力，是认识到需求的持续变化，以及最终目标始终是可工作的软件这一理念。敏捷开发最棒的一点是，它承认了这些事实，并将重构融入其简单的流程中。

简单的工具是最有效的。简单的工具能让你直达问题的核心，毫无阻碍地仔细审视它。它们让你能够把它拿出来，捧在手里，翻来覆去，快速轻松地戳弄它。简单的工具还能让你将它们组合起来，完成更复杂的事情。我们推荐你阅读斯蒂芬·詹金斯关于“老派”编程的文章，^(³¹⁵) 他在其中说得比我们好得多。

编码、调试和单元测试至少与设计同等重要。经验能让优秀的程序员对设计有深刻的理解，并拥有丰富的模式可供借鉴；经验能让卓越的程序员对其所使用的编程语言有深入而透彻的了解。正是这种深入而透彻的知识，才能产生优美的代码。

调试一个冗长复杂的程序的过程，是一项回报极其丰厚的努力。隔离问题、发现错误、搭建调试脚手架、假设解决方案、重新设计、最终确定错误，然后创建正确的修复方案，这一切会带来一种近乎令人难以承受的狂喜和满足感。



## 接下来该做什么？

既然你已经阅读了关于软件开发的所有内容，或许还尝试了一些示例，那么接下来该做什么？如何继续成为一名更优秀的软件开发者？以下是一些建议。

*编写代码，大量编写代码。* 经验大有裨益。编程是一门需要练习和不断强化的技艺。你几乎每天都需要学习新东西，而且很可能每隔几年就需要学习一整套全新的工具和编程语言。编写大量代码会让这项任务变得更容易、更有趣。这就像升级打怪：需要付出努力，但能让你为下一场对抗更有趣问题的重大战役做好准备。

*学习简单的工具。* 简单的工具能给你灵活性。它们还能帮助你掌握基础技能，之后你可以将这些技能应用到更复杂的集成开发环境（IDE）中。当这些 IDE 被取代时（它们迟早会被取代），你可以退回到简单的工具，直到学会新的 IDE。

*阅读关于问题解决和设计的书籍。* 人类解决问题和设计解决方案已有数千年历史。不要忽视计算机科学文献中的经典著作，比如迪杰斯特拉的《结构化编程》一书^(³¹⁶)、布鲁克斯的经典之作《人月神话》^(³¹⁷)、本特利的《编程珠玑》^(³¹⁸)、麦康奈尔的《快速软件开发》^(³¹⁹)以及贝克的《解析极限编程》^(³²⁰)。其他领域的研究和发展也能提供通用的解决问题的策略，这些策略同样适用于软件开发。例如，波利亚的《怎样解题》一书是为解决数学问题而写的，但它非常、非常适用于软件领域^(³²¹)。

*阅读关于编程的书籍，也阅读关于程序员的书籍。* 关于编程的文献浩如烟海。前几章已经提到了不少书籍，但有两本值得再次强调：亨特和托马斯的《程序员修炼之道》^(³²²)以及麦康奈尔的《代码大全 2》^(³²³)。阅读关于伟大程序员如何思考、工作以及编写优秀代码的书籍也是一个绝佳的主意。两本值得注意的书是拉默的《编程大师访谈录》^(³²⁴)以及奥拉姆和威尔逊的《代码之美》^(³²⁵)。

*与其他程序员交流。* 书籍和文章是获取信息的好方法，但不要低估与同行交流的价值。结对编程的一个附带好处是，你能看到别人是如何工作的，他们如何解决问题，如何编码、调试和编写测试。代码评审会议也是了解他人工作方式的好途径。此外，代码评审强化了杰拉尔德·温伯格提出的*无我编程*理念^(³²⁶)。一旦你克服了“拥有”软件产品中代码的想法（你的雇主拥有它；读读你入职第一天必须签署的那些文件，并查看关于知识产权的第 5 章），你就能更客观地审视自己以及同事的代码，从而从代码的优点和缺点中学习。

*加入 ACM 和 IEEE-CS：* 美国计算机协会（ACM；[`www.acm.org`](https://www.acm.org/)）和 IEEE 计算机学会（IEEE-CS；[`www.computer.org`](https://www.computer.org/)）是计算机科学家的两大主要专业组织。它们的期刊包含大量与计算机和计算相关的信息，其会议值得参加，并且它们为会员提供免费的在线书籍和课程。加入其中一个或两个组织，你不会后悔。

*保持谦逊：* 迪杰斯特拉的以下引述道出了一切。

> *称职的程序员完全清楚自己头脑的容量极其有限；因此他以完全的谦逊态度来对待编程任务……*
> 
> ——艾兹格·迪杰斯特拉^(³²⁷)

软件开发是困难的。任何规模的程序都可能极其难以完全理解。计算机软件不仅是人类做过的最具创造性的事情之一，也是最复杂的事情之一。保持谦逊。努力工作。享受乐趣！最后，我们忍不住引用一句同时包含“魔法”和“计算机”这两个词的引述……

> *神话和传说中的魔法在我们这个时代变成了现实。人们在键盘上敲出正确的咒语，显示屏便活了起来，展现出从未有过、也不可能存在的事物。编程之所以有趣，是因为它满足了深植于我们内心的创造渴望，并愉悦了我们与所有[人]共有的感受。……在这方面，计算机也与传说中的魔法相似。如果咒语中有一个字符、一个停顿没有严格遵循正确的形式，魔法就不会奏效。人类不习惯完美，也很少有人类活动要求完美。我认为，适应对完美的要求，是学习编程中最困难的部分。*
> 
> ——弗雷德里克·布鲁克斯^(³²⁸)
