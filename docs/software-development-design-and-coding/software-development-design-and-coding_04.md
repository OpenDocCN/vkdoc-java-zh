# 3. 项目管理基础

> *“任何项目的第一步，都是严重低估其复杂性和难度。”*
> 
> ——尼科尔·亨特

项目一旦启动，就需要进行管理。你需要管理进度、人员、流程，而最重要的是管理好你自己（无论你在团队中扮演什么角色）。管理开发没有唯一正确的方法，但有一些方法可以做得很好，而更多的方法则会导致糟糕的结果。多年来，我们确实发现了一些成功与失败的方法，所以让我们努力让你的旅程少一些风险。外面的世界很危险，请收下这一章！

从事个人开发项目以外的任何工作都意味着在团队中工作；而在团队中工作则意味着被管理，无论是来自上级还是来自内部。因此，从双方角度学习项目管理是学习软件开发的重要组成部分。项目管理可以被视为管理工作和管理人员，因此我们将本章分为两个主要部分：

1.  *项目规划*：日常任务的管理方式在很大程度上取决于所选的软件开发模型，你在前一章已经了解过。剩下的部分是在项目开始之前需要确立的项目要素（即项目规划任务）：
    1.  组织架构

    2.  风险分析

    3.  资源需求

    4.  任务估算与进度安排

    5.  缺陷管理

2.  *软技能意识开发*：在任何项目中，必须管理的最重要且最复杂的方面是人。开发人员和管理者不仅是一种宝贵的资源，更是开发生态系统的核心。维护和支持这个具有复杂动态的生态系统，不仅会影响当前项目的成功，还会影响后续所有项目。在本章的后半部分，我们将通过讨论以下内容，回顾我们为何以及如何在开发过程中努力成为“软技能意识开发者”：
    1.  团队协作的阴暗面

    2.  支持个体

    3.  支持文化

    4.  将冲突视为机遇

    5.  团队协作的产物

## 项目规划

项目计划是一个很好的工具，用于明确你认为自己正在做的事情、如何完成的概要，以及你计划如何执行该概要。项目计划的问题在于，一旦它被编写并签署，高层管理人员就会认为项目将完全按照计划中所述的方式运行。但项目的现实往往会挫败计划。

项目规划是永恒的；它贯穿项目的整个生命周期。“计划”从来都不是一成不变的，因为典型的软件项目通常处于不断变化之中。在使用计划驱动流程模型的项目中，项目计划是一份由项目经理编写、开发团队和高层管理人员批准并签署的实际文档。实际上，它是一份*合同*，尽管是滚动式的，规定了团队将要做什么以及如何做。它说明了项目将如何管理，在最极端的计划驱动项目中，甚至规定了文档本身如何以及何时进行修改。

### 项目规划：组织架构

在工作开始之前，你需要决定如何组织开发。如果你与经验丰富的团队合作，可以重用之前的组织文档，但绝不应跳过这一步。项目组织的两个主要方面（通常相互交织）是开发工作和人员：

*   你打算如何*组织产品*？（瀑布式还是迭代式开发？版本控制工具？项目范围与进度？）

*   你打算如何*组织制造产品的人员*？（团队组织/层级结构是什么？每个人何时以及如何沟通与协调？）

在所有利益相关者之间就进度安排和期望达成明确共识并拥有精确的共同理解至关重要。选择开发流程模型（见前一章）将为你提供许多关于如何组织项目的答案。例如，任何敏捷项目都将围绕功能展开，并需要迭代进度安排，每次迭代都以规划开始，以最小可行产品结束。敏捷团队在层级上是扁平且自组织的，由团队（而非项目经理）拥有敏捷项目计划，并要求在多种开放渠道上进行频繁沟通。敏捷团队还共同拥有所有代码，因此要求每个人都遵循共享的代码管理实践，例如用于源代码控制的 git-flow 分支策略。将流程模型从敏捷缩小到具体实现（如 XP、Scrum、Crystal、功能驱动开发等）将进一步明确组织产品和人员的细节。



### 项目规划：风险分析

接下来我们需要思考风险分析。一个项目可能出现什么问题？当问题出现时我们又该如何应对？让我们看看一些可能的风险^(³⁴)：

*   **进度延误**：你预估需要三天完成的任务，结果花了三周。在计划驱动的项目中，如果你没有定期召开状态会议，这可能会成为一个问题。等了三周才告诉老板你进度落后，总比一知道要延期就立刻告知她要糟糕。不要拖延传递坏消息。在敏捷项目中，这种情况不太可能发生，因为大多数敏捷项目都有每日状态会议（或站会）。这样进度延误几乎能立即被发现，并可以迅速采取纠正措施。

*   **过高的缺陷率**：你的测试发现了大量错误。你该怎么办：继续添加新功能，还是停下来修复错误？同样，在那些按固定计划（比如每周一次）进行集成构建的项目中，这可能是一个真正的问题。在每天或每次添加新功能时都进行集成的项目中，你可以更容易地跟上缺陷的处理。无论哪种情况，如果你正经历高缺陷率，最好的做法是停下来，审视全局，在添加更多功能之前找到缺陷的根本原因。从项目管理的角度来看，这很难做到，但最终你会感谢自己这么做。

*   **需求被误解**：你所做的并非客户想要的。这个经典问题源于客户和开发者生活在两个不同的世界。客户生活在应用领域，他们从用户的角度理解自己想要产品做什么。开发者则从技术角度理解产品将如何工作。有时，这两个世界会交汇，这很好；但通常它们不会，这就是需求被误解的根源。最好的方法是尽可能让客户在场，并尽可能频繁地交付可交付的产品。

*   **需求变更**：新功能、功能修改、功能删除……无休止的需求变更很可能是导致交付日期延误、缺陷率高和项目失败的最大单一原因。当客户（或你自己的市场人员，或开发团队本身）在开发过程中持续更改需求时，就会发生需求变更。这会导致大量的代码返工、基线重新测试，以及一次又一次的延误。管理需求是项目经理最重要的工作。在计划驱动的流程中，变更控制委员会（CCB）会审查每个新需求，并决定是否将其添加到待实现的功能列表中。CCB 中可能有一名开发团队成员，但这并非强制要求，因此风险在于 CCB 可能在未完全理解所有进度和工作量影响的情况下添加新功能。在敏捷流程中，开发团队控制着优先级排序的需求列表（在 Scrum 中称为产品待办列表），并且只在项目的设定时间点——在 XP 中是在迭代之后，在 Scrum 中是在每个冲刺之后——调整该列表。

*   **人员流失**：你最资深的开发人员在产品交付前三周决定加入一家初创公司。减少人员流失的最佳方法是：(1) 给开发人员有趣的工作，(2) 让他们在愉快的环境中工作，(3) 让他们掌控自己的进度安排。奇怪的是，金钱并不是软件开发人员的首要激励因素之一。这并不意味着他们不想获得高薪，但确实意味着为了让他们更努力工作或留住他们而给他们更多钱，通常并不奏效。尽管如此，即使你尽了最大努力，一些开发人员还是会离开。减轻人员流失影响的最佳方法是在开发团队的所有成员之间分散项目知识。像**集体代码所有权**这样的原则和**结对编程**这样的技术，有助于让所有团队成员都投入到项目中，并将代码知识分散到整个团队。我们将在本章的“软体开发”部分进一步讨论如何支持开发人员并维持健康的工作场所文化。

一旦你列出了项目的风险清单，就需要逐一处理，并讨论两件事：**规避**和**缓解**。针对每个风险，思考如何规避它。在你的进度中预留缓冲时间，进行持续的代码审查，尽早冻结需求，频繁发布，要求结对编程以分散代码知识，等等。然后你需要思考，如果最坏的情况真的发生了，你该怎么办；这就是**缓解**。从发布中移除功能，停止新功能开发并进行错误排查，将新功能协商到未来的发布中，等等。如果一个风险变成了现实，你必须采取**某些**措施；提前计划好要做什么会更好。

一旦你处理了规避和缓解措施，你就会有一个如何处理可识别风险的方案。这并不能让你完全免责，因为总会有你遗漏的风险，但处理你确实想到的风险的经验，将使你能够更好地处理项目中那些让你措手不及的新风险。当使用迭代过程模型时，最好在每个迭代之后重新审视你的风险，看看哪些发生了变化，识别出任何新的风险，并移除那些不再可能发生的风险。

### 项目规划：资源需求

你的项目需要资源。项目需要多少人？他们是都需要同时开始，还是他们的项目开始日期可以随着阶段的启动而错开？你需要多少台计算机？你将使用什么软件进行开发？你需要什么样的开发环境？每个人都接受过该环境的培训吗？你需要哪些支持软件和硬件？是的，无论你使用什么过程模型，你都需要一个配置管理系统和一个独立的构建机器（通常是云服务器上的虚拟机）。拥有一台每天构建产品新版本的单一机器，可以让你控制编译和构建中的所有变量。这包括但不限于构建机器所使用的操作系统版本以及程序所使用的每个库的确切版本。这种一致的构建环境确保你每次都能获得正确版本的新产品构建。

你所针对的平台以及你工作的应用领域可以回答许多这些资源问题，所以这部分相对容易。关于团队规模、开始日期和项目阶段的问题，很可能在你对工作量估算和进度安排进行首次尝试之前无法回答，因此迭代方法在这里可能有用。此外，一些资源可能会带来权衡，或迫使项目范围发生变化。考虑替代方案可能需要所有直接利益相关者的参与。



### 项目规划：任务估算与进度安排

制定项目进度的第一步，是明确你要做什么，以及每个步骤需要多长时间。这是一个经典的“先有鸡还是先有蛋”的问题：除非你将功能或用户故事相当细致地分解为具体任务，否则你无法真正进行估算。但你的经理总是希望你在开始设计之前就提供工作量估算和进度数据。要抵制这种要求，优先在初步了解需求后进行设计。选择一小批高优先级的需求，为该功能集设计解决方案，然后仅对该功能集进行工作量估算。不必担心需求可能会变化；它们肯定会变，因此过早地进行分解和估算作用非常有限。你需要将功能详细分解为可执行的任务，才能进行工作量估算，因此这需要逐步进行。永远不要相信任何告诉你“那个功能需要六个月才能完成”的人。那纯粹是瞎猜，与现实几乎毫无关系。你根本无法估算如此庞大的东西。你最多只能说：“我曾经在一个团队里，花了六个月实现过一个类似的功能。”即便如此，这也只是略有帮助。

估算的口诀是：*先估算规模，再估算工作量，最后制定进度*。整体工作应分解为最多一周内能完成的任务，理想情况下是一到两天。将工作分解为定义明确的小任务，会迫使你更深入地思考如何完成它。小到可以一次性完成的任务，已经能大大降低任务切换带来的认知和技术成本。

*规模*始终需要优先考虑，因为只有当你对工作有多大有了概念，才能弄清楚它需要多长时间。一旦你有了一个合理的任务列表，就可以开始进行规模估算。规模可以指多种事物，具体取决于你的工作分解方式和开发模型：功能模块、类数量、方法数量、功能点数量、对象点数量或故事点数量。

接下来是*工作量*估算。任务应以*人时*为单位进行估算。有几种技术可用于获取工作量估算：COCOMO II^(³⁵)、功能点分析法和德尔菲法只是其中三种。然而，所有这些方法都依赖于能够统计设计中的各项内容。德尔菲法是一种快速且相对高效的估算技术。这里有一种简化的运作方式：找三位你最有经验的开发人员（那些经验最丰富，因此应该能给出较好估算的人），将任务分解列表交给他们（假设他们之前没有参与初始分解——这是理想情况）。为了估算每个任务的工作量，请他们给出三个以人时为单位的数字：最短可能耗时、最长可能耗时和典型可能耗时。计算平均最短时间、平均最长时间和平均典型时间，并使用这些平均值来指定预期的工时范围（最小-最大）以及该任务的典型预期工作量。请注意，你需要开发人员来进行这些估算。经理永远不应该做开发估算。即使经理过去曾是开发人员，除非他深度参与实际的开发工作，否则也不应参与开发估算。

为了提高任务估算的长期准确性，你需要追踪每个任务的*速率*，这是敏捷方法论中的一个术语，^(³⁶) 定义为任务的估算工作量除以实际工作量。速率高于 1.0 意味着任务实际耗时少于估算；速率低于 1.0 意味着任务实际耗时多于估算。任务速率有助于你了解自己悲观高估或乐观低估的倾向，最终帮助你做出更好的估算。理想情况下，随着开发人员经验的增长，他们每个任务的速率将趋近于 1.0，稳定在 0.85-1.15 的范围内，从而使整体进度更加准确。如果某个开发人员的速率波动剧烈（一个任务 0.6，另一个 1.8，第三个 1.2），那么可能有必要进行估算技术的速成培训。

一旦你确定了第一个发布版本或迭代中任务的规模以及每个任务的人时工作量估算，就可以着手创建*进度表*了。在你查看那张漂亮的甘特图，以及标记发布日期的那个醒目的黑色菱形之前，有几件事需要考虑：

*   让开发人员明确任务之间的*依赖关系*，因为依赖关系会推迟你的交付日期。虽然有些任务可以同时开始，但有些任务必须等其它任务完成后才能开始，或者可以在其它任务完成一半时开始。

*   弄清楚你的*工作周期*（系统处于活动状态的时间比例）。在每个八小时工作日中，你的开发人员实际用于*开发*的时间有多少小时？请记住，阅读邮件、参加会议、进行代码审查、休息等都会消耗时间。整体企业文化对工作时间利用率的影响很大。你不能假设一个八小时的任务能在一天内完成。实际上，在每个八小时工作日中，有 2-4 小时会被其它事情占用，因此你的开发工作周期可能低至 50%（四小时，即八小时工作日的一半）。

*   在制定进度表时，要考虑周末、假期、病假、培训和缓冲时间。如果你的高级开发人员有一个任务处于项目的关键路径上，你可能需要知道她五月份要参加那个为期一周的研讨会。

*   你不应该安排一个开发人员同时处理两个任务。大多数项目进度规划软件默认不允许这样做，但大多数也允许你覆盖此设置。请不要这样做。你可能会为了满足经理或市场团队要求的截止日期而忍不住这样做，但请尽力抵制诱惑。最终，当你无法按时完成时，你只会不得不再次修改进度表。

*   最后，你必须有一些方法来跟踪项目进度，以指定任务、截止日期、优先级、估算以及负责任务的开发人员。虽然完整的项目管理工具一开始可能显得大材小用，但它们通常足够灵活，能够适应不同的项目需求，而不会带来太多额外负担。使用像 Jira、Notion 或 Trello 这样的真实项目管理应用程序，它们提供了许多功能，使得保持进度表更新变得容易得多。



考虑自动化能为每个人了解项目状态带来的好处：自动提醒、日历集成、将任务分配给开发人员以及更新通知。项目管理工具还可以自动跟踪各种指标（例如任务速度），生成可视化图表以更好地理解这些指标（例如展示整个项目中平均速度的变化），并将项目信息组织成不同的便捷布局（例如按任务类型和开发人员对速度数据进行分组）。对于团队（即使是 10 人或以上的中等规模团队），自动跟踪项目依赖关系有助于管理谁在做什么以及何时做。这种认知对于具有短时间盒迭代的敏捷开发尤为关键。在规划每个冲刺时，你需要了解产品待办事项列表中每个用户故事的优先级以及它们之间的相互关系。你必须跟踪冲刺待办事项列表中任务的进度，而了解依赖关系可能是实现冲刺目标与未能实现之间的关键区别。

项目管理工具也有助于工作量估算，因为按小时估算对人来说可能非常困难。一些项目管理工具会要求你使用量级来创建估算。最简单的类型让你使用*小*、*中*或*大*进行估算，而其他工具则允许你为每个任务分配一定数量的点数，例如，每个任务 1、2、4 或 8 个点。这使得汇总团队在每个时间盒迭代中试图实施的总点数变得容易，并且随着时间的推移，可以得出团队每个迭代平均能实施的点数。

### 项目规划：缺陷管理

抱最好的希望；做最坏的打算。不可避免地，你会在程序中引入缺陷。项目计划应考虑到持续测试的需求，并且进度安排中应留出处理任何已发现缺陷的空间。作为开发人员，你的目标有两个：

*   在你编写的代码中尽可能少地引入缺陷。

*   在发布代码之前尽可能多地发现它们。

然而，尽管你尽了最大努力，你发布的代码中仍会包含缺陷。这是不可避免的。对于任何规模的程序来说，程序中可能的路径太多，引入错误数据的方式也太多，以至于不可能没有缺陷。你的目标是发布时尽可能少地带有缺陷，并且使这些缺陷不会真正影响产品或性能。为了实现这一目标，大多数开发组织都有一套缺陷等级来表征缺陷的严重程度，例如致命（程序崩溃）、严重（主要功能损坏）、较重（功能损坏但有变通方法）、轻微（小问题）和功能请求（未来希望拥有的功能）。这些缺陷等级通常被内置到缺陷跟踪系统中。

每当你在代码中发现缺陷时，就在缺陷跟踪系统中提交一份缺陷报告，让开发人员和管理者能够看到当前缺陷的数量和严重程度。在修复缺陷时，开发人员会从最严重的开始，然后依次向下处理。

## 软技能开发

> *“当（项目经理）启动任何项目时，他们肩负着艰巨的任务：弄清楚并管理众多不同利益相关者的期望。但是，当（项目经理）能够认识到这些期望的根源，扮演起夫妻顾问的角色，建立有意义的、个人的、信任的关系，并确保更广泛的团队专注于整体目标时，他们就能成功地管理每个人的期望，同时引导项目跨越那条令人垂涎的终点线。”*
>
> —艾米·肖恩塔尔^(³⁷)

> *“管理者的职能不是让人们工作，而是让人们有可能工作。”*
>
> —汤姆·德马科，蒂莫西·李斯特^(³⁸)

当我们说“团队项目”时，你首先想到的是什么？是压力？恼怒？担忧？还是对即将到来的轻松任务的解脱？想想我们大多数人是如何了解团队协作的：我们被要求在学校项目中合作，但通常没有被教导如何合作。由新手自我管理的临时短期小组项目通常会导致有缺陷且不可持续的长期应对策略，例如独自承担团队工作或搭便车。毕竟，我们很可能再也不用和这些人一起工作了！关于功劳和责任的归属问题也层出不穷。最终，学生们会认为团队合作是有害且不愉快的^(³⁹)。在通过多个压力重重的课堂项目反复强化这些做法和信念之后，学生们毕业并走向世界。当然，问题在于，现实世界是建立在人际关系网络之上的：与你的同事、老板、下属、客户、公众等建立并维持关系。那些相同的短期应对策略根本行不通，因为现实世界（如同北方一样）会记住一切！

在本章的剩余部分，我们将介绍什么是“软技能意识”（Soft-Aware），以及有缺陷的团队合作如何对开发人员和开发工作产生负面影响。然后，我们将探讨支持个人和创建安全的共享文化如何促进健康且可持续的协作。最后，我们将研究如何将这些理论付诸实践，通过一系列团队协作工件，形成一个共同学习和改进的敏捷过程。^(⁴⁰)

*软技能开发*是一种新的范式，它将个人、人际关系和工作场所文化视为健康、可持续的软件联合创作的核心。具备*软技能意识*包括构建软件、构建利益相关者关系，以及在此过程中培养每一位开发人员。*软技能开发*一词最早由卡扎科娃博士^(⁴¹)在向计算机科学本科生教授基于团队的软件开发时提出。其重点是培养开发人员的专业素养和人际交往能力，因为在尝试制作软件的过程中，*学习如何合作*比学习制作软件本身更为关键。随着技术的高速更迭、日益高级的编程实践以及不断演进的软件开发模型，不变的是社会情感学习的力量，它使我们能够与他人合作，创造出远超我们个人所能实现的创新。社会情感学习是指人们“*获取并运用知识、技能和态度，以培养健康的身份认同、管理情绪并实现个人和集体目标、感受并表达对他人的同理心、建立并维持支持性关系，以及做出负责任和充满关怀的决定*”。^(⁴²) 虽然开发通常以冲刺的形式进行，但团队合作是一场马拉松：它需要承诺、训练、可持续的节奏以及克服不适。公开重视并实践社会情感学习，正是我们为团队合作马拉松进行训练、支持共同的*软技能开发*文化以及相互支持的方式。



研究表明，“*生产力取决于具备社交和情感能力的劳动力*”。^(⁴³) *软意识*开发的核心在于人；它始终关注每一位团队成员，以及必须精心培育和维护的、涌现出的动态团队协作。没有唯一正确的协作方式，只有理想的团队协作成果。一个团队的能力和复杂性总是大于成员之和，因为我们涌现出的互动既能成就也能破坏共同的工作。在团队中工作，不仅意味着为项目现身并适应项目，也意味着为彼此现身并相互适应。

这部分内容可能看起来是为“掌权者”所写，但权力有多种形式，并非仅限于职位头衔中带有“管理”二字的人。我们都拥有某种形式的权力。例如，考虑一下，更直言不讳的人如何在会议中实际上拥有更多发言权；再考虑一下，利用这种直言不讳将每个人的想法都摆上台面，又如何能极大提升决策质量。考虑一下，多年的经验如何能催生许多关键想法，但也要考虑，缺乏经验的观点如何能让我们在看似无法通行的方向上实现创新。我们所说的*权力*，是指影响变革的能力。关注自己和身边的人，将让你能够建立一个能与你共同推动变革的团队。为了实现真正的合作，我们不应将自己划分为管理者和被管理者，而应只有协作者（其中一些人的任务是促进他人的工作）。这样一来，我们每个人都能贡献自己独特的“超能力”，而非依赖等级权力，从而组建最全面的超级开发团队^(⁴⁴)。找到你的权力，找到运用这种权力的声音，并帮助你的队友与你一起运用他们自己的权力。

为什么我们非要让每个人都和睦相处呢？虽然我们可以期望员工或队友*埋头专注于工作*（不幸的是，我们在现实世界中屡见不鲜的态度），但这将是非常短视的。与不喜欢的团队一起工作需要付出更多的情感努力，让我们感到疲惫和不快，从而阻碍软件开发。工作中的个人能动性和幸福感能提高开发者的生产力，^(⁴⁵,⁴⁶)而情商则有助于缓解压力、建立信任。^(⁴⁷)当工人在面对裁员、任务或项目不确定性、感觉不被重视、遭受排斥或歧视等无法控制的压力情境时感到无力，都可能导致各种负面的身心影响，包括“习得性无助”——一种缺乏个人能动性的感觉，从而导致工作投入度降低。^(⁴⁸)根据盖洛普公司发布的《2023 年全球职场状况报告》，全球 77%的员工认为自己的工作没有意义或缺乏投入，导致 59%的人“静默离职”（只做最低限度的工作或“磨洋工”），18%的人“高调离职”（积极从事破坏性行为），51%的在职员工正在考虑或寻找新工作。^(⁴⁹)盖洛普的报告还指出，70%的团队投入度归因于管理者。更健康的工作场所文化是解决员工不敬业和高离职率的良方，管理者、主管、SCRUM 大师、团队领导以及任何拥有权力和/或资历的人都必须带头冲锋。虽然大多数开发者和管理者没有明确的社会情感学习背景，但许多公司已经认识到转变职场文化的必要性，并开始提供培训研讨会，以培养员工的人际交往或*软*技能。

团队协作不仅仅发生在通往可交付成果的路上；它本身就是一个持续的关键任务。请记住，大多数合作关系会延续到当前项目结束之后，因此队友不应被视为可消耗的开发资源，而应被视为开发赖以繁荣或消亡的生态系统。事实上，软件开发可以被视为“*一系列资源有限、目标导向的发明与沟通游戏*”。^(⁵⁰)我们见过优秀的开发者在一起分崩离析。我们也见过挣扎的开发者共同崛起。成为独行侠式的明星开发者并非目标。如果你表现出色，你会被要求去帮助、指导、培训或监督他人。你会被要求向管理层和客户展示你的工作成果。你会被赋予更大的项目，届时你必须学会授权。你必须与他人良好合作。团队协作是关键，而且必须有意识地为之。任何联合头脑风暴、直接协作、团队范围更新以及团队建设活动的机会，都是所有共同工作的基石部分，即使是最紧张的开发日程也应欢迎它们。尽管我们都忙得不可开交，时间宝贵，但不花时间去培养*软意识*，最终付出的代价将远超我们的承受能力。重视并支持队友和团队协作是一项明智的投资（投资于他们的未来、团队的未来以及产品的未来），绝非浪费。



### 软·件开发：团队协作的阴暗面

并非所有共同工作都能达到真正*团队协作*的程度，即有意义地支持和提升团队的整体产出。相反，开发过程和开发者常常会遭受不良团队协作带来的负面影响。让我们来看一些常见且会引发问题的具体行为示例：

*   **基于社交舒适区做出工作决策**：只与同性别、同年龄段、同种族、同文化背景等的人沟通，可能会疏远他人，让他们觉得自己被排除在头脑风暴或决策之外。因为与某位队友没有私人交情就不主动与其合作，而不是努力去理解摩擦的根源，这只会让问题恶化。避免与那些不主动发言的开发者交谈，会加剧不必要的孤立。我们付出一点努力，就可能极大地提升他人感受到的舒适感、归属感和自主权。基于当前任务分工形成隐性的子团队是可以的，但重要的是要尽可能（例如在咖啡休息时间或每日站会时）将整个团队放在心上。

*   **让部分团队成员无法获取项目信息**（例如，在私人聊天或邮件中讨论开发问题）会使每个人都难以全面了解决策、挑战、需求等。要让工作讨论能够按需被整个团队访问，但不要强迫每个人在没时间或忙于其他事务时参与每一次讨论。例如，你可以在一个独立但可见的线程中回复在公共线程中提出的问题。同时，也要考虑使用 Slack 或 Discord 这类消息服务的优势，它们允许信息分组或置顶，可以添加回复和子线程，并且可以@团队成员，将相关讨论推送给他们。

*   **开发者直接动手开发**。即使在敏捷方法论中，这也是一个错误。花时间就设计达成一致，创建情绪板以确保每个人都共享相同的整体愿景，并清晰地划分任务和子任务（即使只是为了产品的下一个迭代，而不是最终交付物），对于避免混乱和简化开发过程中的决策至关重要。定义不清或理解有误的任务通常会导致工作重叠、合并冲突、混乱、沮丧以及团队内部的争执。

*   **排斥经验不足的队友**。因为不信任新开发者的技能，就不让他们承担项目中有意义且有趣的部分，这会打击学习积极性并降低创造力。将经验不足的团队成员排除在关于开发的关键讨论之外，会降低他们的自主权，并对其认同共同的项目愿景产生负面影响。

*   **认为完成自己的任务就足以实现协作**。如果开发“群岛”的其他部分都在挣扎，那么成为一个高效的“孤岛”是不够的。协作围绕着一个共同的目标，而关注团队就是在关注这个目标。不记录已找到的解决方案和变通方法，不利于共同学习和建立共享知识库。不花时间去理解他人的贡献，可能会导致大量冲突，包括成员认为只有自己在做有意义的贡献，或者没有注意到某人的工作未被正确合并到代码库中，从而被排除在演示之外，这对开发者来说可能非常打击士气。

*   **低估人际需求**。不尊重彼此的工作、时间、职责、需求和关切，就是不尊重彼此。不告知团队那些会影响你工作或可用性的个人情况，很可能会引起对某些贡献能否在截止日期前完成的担忧，从而导致重复工作、压力、加班和信任破裂。开会不准时，体现了缺乏规划、可靠性存疑以及对他人时间的不尊重。在做出可能影响队友的决策时不与他们沟通，必然会导致代码冲突和人际冲突。未能向主管和客户更新进展和困难，可能导致理解脱节、压力增加和业务损失。

*   **不照顾好自己**。不花时间和精力去清晰地分享你的想法或担忧，会对你产生直接的负面影响，并对团队产生间接的负面影响。不分享困难和挫折，会造成不切实际的期望，并减少团队的支持。坚持不切实际的时间表或贡献量是短视的，因为它会导致怨恨和倦怠。



### 软件关怀：支持个体开发者

众所周知，用户故事很重要，但我们必须记住，开发者故事也同样重要，因为双方都是开发过程中的利益相关者。如果我们期望开发人员为我们工作、与我们协作，就必须将他们作为个体进行投资：支持有意义的工作，关注开发者及其需求，支持他们的自主权，重视他们的个人与职业成长，并理解创造性时间。

*微观管理对双方都不可持续*：管理者无法无处不在，也无法始终掌握所有信息；而开发者的手脚也不能被时刻束缚，否则任何开发都无法进行。试想一下，当有人在你背后盯着时，思考会变得多么困难；或者拿着别人的购物清单去采购，却不知道哪些替代品可以接受；又或者必须用某种你并不理解的具体方式去解决问题。Ron Jeffries 描述了源于敏捷实践错误实施的*黑暗 Scrum*：“*但掌权者已经知道自己的职责。他的职责是掌握每个人的动向，确保他们做正确的事，并在他们偏离方向时予以纠正。每天都有一次强制会议让他可以这样做，多么方便！*^(⁵¹) *结果呢？团队不再围绕共同使命集结并梳理出当天的最佳方法，而是由别人从他们那里榨取信息，在脑中处理，然后告诉每个人该做什么。由于事情几乎从不会像我们昨天早上预期的那样发展，这种不当行为常常伴随着大量的指责和紧张气氛。黑暗 Scrum 每天都在压迫团队。自组织无法涌现。*”^(⁵²) 开发者需要被信任去完成工作，而管理者则需要通过满足需求、消除障碍、促成自下而上的自组织来为这项工作提供便利。

让我们把开发想象成*驾驶一辆车*，穿越未知的地形前往某个目的地。客户、管理者或产品负责人选择目的地，但不选择路径。开发者是司机，他们选择如何最好地到达目标。Scrum Master 或项目经理不应在后座指手画脚，但他们可以跟踪进度，让每个人专注于通往终点的总体方向。产品负责人可以呼叫车内，但他们不在车上，呼叫司机会分散其对路况的注意力，因此应谨慎为之。司机需要拥有自由，根据他们对所驾车辆的熟悉程度、行驶速度、驾驭各类地形的技能、天气、意外的车辆故障，以及他们从内部视角认为相关的任何其他因素，来决定应该走哪条路。不应强迫司机走他们不习惯的路或开他们不适应的速度；你可以向他们提供更多关于进度的信息，以帮助他们更好地评估需求，你也可以选择下次旅行不雇佣他们，但你需要让他们驾驶。即使是关键信息，也需要以不干扰的方式提供。试想一下，司机即将错过一个转弯，但可能已经来不及调整路线；有时，犯下错误并在事后纠正反而更安全，更不用说从错误中学习从长远来看更具指导意义。给开发者一些空间和计划内的时间去犯错，对他们的个人成长、编程和调试技能、解决问题的信心以及试错能力等至关重要，而这反过来又对软件开发本身至关重要。

*领导者必须关注*。领导者/主管/管理者只有*贴近团队*及其问题，才能真正有效。领导力必须具有亲和力、可及性和适应性。围绕人员及其需求建立关系并调整结构。工作场所文化中的价值观应在所有个体之间共享并得到支持，但结构应具有灵活性，以支持人们学习、沟通和工作的不同方式。关注情境（团队情境、个人情境和项目情境），并利用它做出更好的决策。情境也决定了一个事件是否真的是需要解决的问题，以及如何最好地解决它。但请注意，相同的客观情境对于不同的团队和不同的开发者可能会转化为不同的主观情境，因此领导者征求反馈，并让反馈感到受欢迎、被重视并在未来得到考虑，这一点至关重要。

一个富有成效和创造性的开发环境的关键方面是*有意义的工作*。所有团队成员（尤其是担任管理角色的成员）都需要帮助所有利益相关者找到工作的意义，因为这将激发最大的奉献精神和创造力。虽然雇佣关系或学术环境通常提供外在动机（例如薪水或分数），但这通常只能激发服从。要挖掘真正的动力，我们还应该通过将项目/公司目标与所有利益相关者的目标联系起来，来激发内在动机，^(⁵³,⁵⁴,⁵⁵) 例如对自主权、个人成长和成就感的渴望。有意义工作的一个关键起点是传达工作的*影响*，并帮助他人认同这种影响，有时可以通过灵活调整工作以适应所有利益相关者来实现。共同的愿景有助于每个人理解目标，并像北极星或冲刺目标一样聚焦于此。共同确定优先级，理解哪些要素至关重要及其原因，协商时间表，并公开地、以团队方式解决问题，都将有助于确保每个人都认同工作计划，并允许个人在各自的任务中沿途做出正确的开发决策。理解、认同并参与计划能增强每个人的个人自主权，这对开发者至关重要。^(⁵⁶)

*相互奉献*促进信任和支持。如果你希望你的开发者付出额外努力，你也必须同样通过投资于他们的成长和福祉来做到这一点。^(⁵⁷) 如果你需要开发者加班来处理不可预见的情况，那么你也需要投资于他们的个人时间和需求（额外的假期、缩短工作周以补偿延长的工作日等）。如果我们希望个人尊重公司时间，那么公司也必须尊重个人时间。促进*合理的工作量*，并倡导工作与生活的平衡，而不是鼓励加班。如果我们需要每个人都愿意偶尔投入更长时间来满足临时的紧急需求，那么管理层不仅要对额外的工作时间负责，还要对员工在标准工作时间之外为此做出调整的意愿负责。试想一下，今天多工作两小时，并不能被明天少工作两小时有意义地补偿。工作时间的任何变动，尤其是临时通知，都会给个人带来不可忽视的额外负担（包括疲劳工作、睡眠不足、取消计划、不去健身房、重新安排家庭和宠物照看等）。如果这些负担没有得到有意义的考虑，员工将失去关键的恢复时间，最终导致倦怠、隐性离职、工作质量下降以及人员流动率上升。



### 软技能开发：一种安全与成长的文化

团队协作并非凭空发生。无论我们如何精心关照每一位开发者，我们共同营造的协作环境才是软件开发的生态系统，它最终将决定我们能否共同繁荣。如果只重视产出，学习就可能变得不安全。如果所有冲突都被视为负面，多样性就可能变得不安全。一个可持续的工作场所文化需要关注学习与成长、承诺问责制，并高度重视人际交往技能以支持健康的关系。

#### 学习中的安全感

软技能开发需要一个安全的学习环境。我们所有的工作都必须基于其发生的*情境*，而每个新情境都应以好奇心和成长为导向来对待。团队协作中没有放之四海而皆准的解决方案：与新团队、新项目或新客户合作，都需要细心、问责和重新适应。它还需要有安全犯错的空间，以及在知识创造和新技能习得期间，开发工作可能停滞的时间。

学习协作的方式因团队构成而异，因此需要试错。团队成员必须欢迎学习和适应的过程，这也意味着要欢迎工作中技术与非技术方面的错误，甚至将其视为成长的机遇而加以庆祝。*成长*需要学习新事物，这始于未知，且往往在首次尝试时无法成功。*成长导向*将失败视为学习常态，让开发者能够安全地表达困难、寻求帮助或承认错误。开发者可以在此采用敏捷方法：尝试、获取反馈、调整、再尝试。为成长创造安全环境需要尽量减少内疚和指责，转而专注于在通常伴随学习经历的成长阵痛中相互支持。

当开发者拥有完成工作所需的必要自由时，问责制便分散在整个团队中。因此，团队成员必须以建设性的方式相互*问责但无责*。^(⁵⁸) 指责关注过去，而问责则关注为未来制定更好的计划。一个让承担责任感到安全的环境，将促成关于如何调整方向、继续前进的公开坦诚的讨论。成长型思维自然有助于管理恐惧、减少指责，并在事情进展不顺时培养问责制。试想，当一个错误发生，且在到达客户之前未被任何人发现时，团队中的每个人都负有责任。任何人都可能发现这个错误，但没有人做到。为最初的错误找替罪羊并不能解决问题。问题不在于错误本身。错误总会发生。真正的问题在于，很可能存在有缺陷的流程，未能设置保护性冗余机制，以确保在共享流程中更早地发现错误。团队中的每个人是如何被动或主动地导致了这个问题？每位团队成员可以做些什么来防止未来再次发生？面对任何不尽人意的结果，团队都应思考所有可用的信息、技术、实践、人员和环境是如何共同导致了不良结果，以及他们如何共同防范未来此类结果的发生。即使在最简单的有人忘记任务的情况下，团队也可以修改其实践，通过共享任务板、自动截止日期和提醒、日历集成、结对编程等方式来改进任务跟踪。

项目的共同所有权支持集体问责，并促进所有任务中的相互支持。虽然适度的良性竞争是好事，但团队和公司应同甘共苦，才能真正培养出同袍情谊和共同目标感。共同所有权孕育的是支持而非竞争。共同所有权还能确保工作和理解在团队中更均匀地分布，而学习文化则重视在此过程中新技能的习得。其结果是形成一个工作负载更均衡、更能应对不可预测情况和未来需求的团队（例如，帮助测试彼此的代码、为生病的队友代班等）。为促进更具协作性的环境，主管人员应关注、重视并示范协作、资源共享，以及保持支持性和鼓励性的态度。

在技术方面，安全的环境支持实验。大胆、创造力和创新需要探索性的试错。它们还需要相当长的时间，在此期间开发工作可能停滞，而知识正在被创造。有些尝试不会立即改进产品，但会创造新技能、加深理解并探索新想法。不正确的生产力指标和短视的激励措施会导致有缺陷的开发与团队协作实践。如果除了产出之外什么都不被重视，团队将不可避免地陷入不可持续且自我保全的行为，例如：

*   开发者只接手能快速完成的小任务，以显得效率更高
*   不信任和零和博弈行为（例如，通过指责来竞争和寻求个人安全）
*   风险规避，导致开发者回避任何不确定如何完成的任务，避开更具创造性和挑战性的机会，并在此过程中优化以减少学习
*   跳过创造性头脑风暴，直接跳入下一个可交付成果的编码

不要轻视过程、失败、学习和讨论，而只关注结果。为共同实验和学习创造空间，因为没有人会拥有所有答案（无论是主管、客户，还是最有经验的开发者），而且合适的答案会随着时间、团队和团队成员的变化而变化。此外，虽然关注问题行为作为主要处理事项是自然的，但请记住也要强调我们在团队成员身上所珍视的行为。



#### 清晰即安全

无论公司简介中列出了哪些价值观，我们最终创造的文化，是我们在日常运营中不断倡导的文化。我们共享的文化将反映出那些被所有人，尤其是负责人，持续关注、重视和效仿的行为。

正如阿德娜·弗里德曼所说：“*想法的好坏，取决于你传达它们的能力。*”要让“软意识”真正渗透到组织文化中，它必须被组织各层级的人员所践行。会议是领导者展示和促进预期建设性行为与反应的绝佳机会：鼓励并示范开放理性的对话，展现并奖励好奇心，提供支持，多提问而非妄下结论。表达需求或不同意见应得到奖励，以此表明直言不讳并非徒劳，而是非常值得付出努力、勇气和坦诚。例如，如果团队成员表达了需求或偏好，应做出合理努力来满足该需求。

一致性和清晰性是安全的。需求和约束可能会改变，但为之做好准备的方式可以保持一致。工作和团队协作的背景可能会改变，但期望可以保持明确和一致。给予清晰的反馈短期内可能让人感觉不安全，但最终对工作场所的可持续性至关重要。根据布琳·布朗的说法，“*在领导者们指出的阻碍勇气的十种行为和文化问题中，有一个问题被领导者列为最大的担忧：回避艰难的对话，包括给予诚实、富有成效的反馈。*”^(⁵⁹) 回避行为会削弱信任和参与度，增加问题行为，降低共同目标感，并对生产力产生负面影响。布朗将此总结为：“*清晰是友善。不清晰是不友善。*”

让我们考虑一些通过清晰表达来体现友善的其他例子：

*   在团队中明确所有假设和计划（经常比较理解，使用可视化工具，讨论步骤）；确保每个人都理解愿景以及他们在实现该愿景中的个人角色。

*   明确区分最低要求目标与锦上添花的延伸目标。基于猜测工作既危险又代价高昂，还会在团队中滋生焦虑。

*   清晰说明限制条件，例如紧急需求或人手不足。透明度有助于验证顾虑，并营造开放沟通、信任和共同解决问题的文化。

*   不要让别人猜测自己是否做得足够，因为这只会导致焦虑。回避告诉开发者如何改进或改变什么，从长远来看是不可持续的。明确具体地说明期望，这样每个人都能通过思考要求是什么、做了什么以及为什么这样做，来进行有意义的回顾、学习和调整。

*   不同的人对对话的理解和消化方式不同。为确保共识，明确总结每个人的要点和后续步骤。

*   对自己和团队保持开放、诚实、勇敢和坦诚。在每日站会中加入“氛围检查”，有助于提高对团队成员日常状况的共同了解。

*   所有批评都应指向支持和成长，而非羞辱或贬低。与其指出不良或低效的做法，不如引导彼此寻找更好的替代方案，并且始终附上关于为什么某种做法优于另一种以及如何实施的解释。

#### 多元即安全

虽然相似性最容易带来融洽，但异质性团队展现出更强的批判性思维和创新能力。^(⁶⁰) 建立并维护一种欢迎差异的文化，需要每个人都致力于一系列个人和人际实践。“*但要‘包容地’构建任何事物，我们必须首先清晰地理解我们在哪里排除了他人。*”^(⁶¹)

正如你在上一章所见，Scrum 明确定义了一套价值观，供开发者践行承诺、专注、开放、尊重和勇气。如有疑问，以同理心为先。同理心使我们能够更好地理解客户需求、开发者需求以及我们新兴协作中的挑战。同理心在于将他人视为独立的个体来重视，而不仅仅是共同工作的资源。这包括关注他们的需求和困难、沟通和工作风格以及个人情况。同理心需要坦诚、真实，并愿意承认自己可能犯错。要考虑到你可能看不到某人工作的全部复杂性和范围，每个人都有不同的文化和技术背景，我们的大脑处理信息的方式也大相径庭，而所有这些多样性正是我们共同的力量。

让团队中的每个人都感到安全、受尊重和被重视，需要致力于*多元、公平和包容*。包容性策略师维尔纳·迈尔斯有句名言：“*多元是被邀请参加派对；包容是被邀请跳舞。*”^(⁶²) 那么，公平至少要求确保受邀者能够到达派对现场，场地设施适合受邀者，并且所有受邀者在规划活动、音乐和茶点方面都有发言权。是的，这要复杂得多，但如果没有这些步骤，我们就无法声称每个人在派对上都是真正受欢迎的。这需要关注个人需求和偏好、历史和文化背景以及个人目标，而不仅仅是软件开发任务。

一个常被忽视的多元维度是团队成员的*工作类型*。保罗·格雷厄姆的《创造者日程 vs. 管理者日程》^(⁶³) 讨论了创造性工作与管理工作的根本区别。默认情况下，管理者的日程被分解成小的时间段（例如，按小时计），用于安排大量会议。创造者（例如，开发者）无法在短时间段内进行创造，他们需要更长的时间段，例如半天，因为创造性任务需要一些预热时间和较长的整块时间来进行深度专注和发明创造。会议会打乱可用的整块时间，如果安排不当，可能会毁掉创造者的一整天：“*一个会议就能毁掉整个下午，因为它把下午分成两段，每段都短到无法处理任何困难的事情。而且，你还得记得去开会。这对管理者日程上的人来说不是问题。下一个小时总有事情要发生；唯一的问题是什么事。但当创造者日程上的人要开会时，他们必须为此费心。对于创造者日程上的人来说，开会就像抛出一个异常。*”^(⁶⁴,) ^(⁶⁵) 为了将对开发者及团队中其他创造者的生产力损害降到最低，你可以将需要创造者参加的会议安排在工作日的开始或结束时段。



多样性的另一个重要维度是神经多样性，这在软件开发团队中很常见，并伴随着许多独特的优势和挑战。^(⁶⁶) 研究表明，许多神经多样性个体在生命后期才被诊断出来，并且即使他们对有意义的调整有具体想法（例如替代性绩效评估和更安静的工作环境），也不会透露自己的诊断结果或寻求便利措施。^(⁶⁷) 一种包容性的方法可能是提供多种选择，让团队成员根据自身需求自由选择：替代性工作空间、替代性沟通方式、替代性时间表等。

请记住，某些方法或行为即使客观上没有错误，也可能在特定情境下造成不利影响。所有社交互动都依赖于丰富的语境，而其中大部分语境对各方而言都是隐含且不可见的。例如，考虑一下为上午 9 点的会议给每个人买早餐这个看似周到的想法。是否考虑到了每个人的饮食限制？还是说有些团队成员反而因此感到比之前更不受重视？再举一个例子，人们在讨论中感到足够安全而敢于发言的阈值各不相同。仅仅询问是否有人有问题或评论，可能不足以真正打开局面，邀请每个人的意见。还要注意，意见越是反对，在群体中表达出来就越令人望而生畏。同样需要考虑的是，在某些文化中，不进行眼神接触被视为粗鲁，而在另一些文化中则被视为礼貌；以及，僵化地固守自己的社会规范和习俗会显得排外。最后，考虑一下在提建议时随口说“只要”（例如，“你为什么不直接……”）实际上预设了双方有广泛的背景相似性（例如特定的技术技能），最终非但没有提供预期的支持，反而会打击士气、造成疏远。^(⁶⁸) 谦逊、好奇心和同理心是以可持续、健康且富有成效的方式共同工作的基础。各方都可能犯错，但对成长的共同承诺将创造一个更安全的环境，让每个人都能尝试、观察、讨论、学习，并在下次做得更好，同时也允许他人学习和进步。

#### 冲突中的安全

安全的最大威胁是冲突，但冲突不能也不应被避免。冲突是评估不同想法、方法和人际需求的一种方式。当以建设性的方式处理时，冲突是让产品和团队变得更强大的宝贵机会。

冲突在任何团队中都会发生，但在那些沟通开放、开发者有自主权、代码共同拥有、并留有实验空间的、高度投入的自我组织型团队中尤其常见。意见分歧是团队多样性和创造力的标志。面对分歧时，不应忽视相互冲突的意见。任何一个人的意见，都代表了可能考虑使用你产品的用户、可能想与你签订开发合同的客户，以及可能决定开发替代方案的竞争对手中一定比例的观点。忽视冲突意见会造成紧张，危及共同愿景，并扰乱工作。尽早考虑不同观点是提升团队和产品整体质量与稳健性的机会，因此应被欢迎并谨慎处理。团队是你最早期的焦点小组；不要将他们至关重要的见解弃之不顾。

所有冲突都有可能产生积极或消极的结果。*建设性争议* “*存在于当一个人的想法、信息、结论、理论和观点与另一个人不相容，并且双方都试图达成一致时。*”^(⁶⁹) 请注意这与*寻求共识*的区别，后者“*发生在群体成员强调一致、抑制讨论以避免任何分歧或争论、并避免对替代想法和行动方案进行现实评估时。*”^(⁷⁰) 我们的目标不仅仅是达成一致，而是基于积极的合作学习，共同构建新的理解。以下是已被确定为建设性争议所必需的一些条件：^(⁷¹)

1.  合作性目标结构，因为竞争会阻碍有效沟通
2.  熟练地处理分歧，以有效驾驭意见差异
3.  理性论证，以在面对证据和逻辑时生成、提出并更新观点
4.  所有参与者的积极讨论，而非回避、支配或顺从

根据托马斯-基尔曼冲突模式量（TKI），人们会根据其合作性和果断性的程度（或定义为关心他人与关心自己）偏好特定的冲突解决风格。^(⁷²) 识别冲突管理风格对于团队管理至关重要：

*   *回避*风格：延迟或未能达成解决方案会滋生更多的不满、怨恨和沮丧；仅在收集信息、等待日程安排上的空档或处理完其他正在进行的情况时暂时有效。
*   *竞争*风格：对团队合作有害，因为它将某些贡献者的需求置于他人之上；解决方案很可能滋生怨恨、造成不平等，并引发未来的冲突，损害工作环境。
*   *迁就*风格：与竞争风格相反，但总体效果往往相似，因为它可能导致因自身需求未被满足而产生的倦怠。
*   *妥协*风格（双输）：每个人都最终为某个可接受但最终并非任何人真正想要的折中方案做出牺牲。
*   *协作*风格：真正理想的双赢；它最耗时，因为需要进行有意义的讨论，才能真正找到满足每个人需求的方法。



布琳·布朗建议我们都要准备好“开诚布公”：“*所谓开诚布公，是指一种讨论、对话或会议，其核心在于承诺坦然面对脆弱、保持好奇与慷慨、坚持在问题识别与解决的混乱中间地带工作、必要时暂停并折返、无畏地承担自身责任，并且，正如心理学家哈里特·勒纳所教导的，要以我们渴望被倾听的同等热情去倾听。*”^(⁷³)

开发团队不可避免地会遇到任务冲突和人际冲突。*任务冲突*是指理解上的差异、对采用何种方法的不同意见、对风险和成本的评估分歧等。解决任务冲突有助于通过集思广益、弥补弱点或澄清需求来改进产品。*人际冲突*则涉及沟通风格、文化背景、神经多样性、个性、边界、触发因素等方面的差异。人际冲突会极大地消耗开发者的精力。解决人际冲突能够改善团队合作、工作满意度和生产力。

冲突对成长而言价值连城，但绝不能让它们恶化。及时主动地解决是关键，这需要一种文化：欢迎艰难的对话，赞扬对话所需的勇气，并帮助每个人掌握必要的技能，以便沟通、倾听和协作，共同寻找新的共识。人际间的紧张关系需要被摆到台面上并及时解决，以防止关系恶化。团队可能需要外部资源、信息或公正的仲裁来化解冲突，防止怨恨像滚雪球一样扩大。那么，我们如何才能建设性地解决冲突呢？一系列*有技巧的分歧*技术已被提出^(⁷⁴)：

*   批评想法，而非针对个人。
*   将自我价值与对你想法的批评区分开。
*   专注于团队获胜，而非个人。
*   鼓励每个人都去了解情况并参与讨论。
*   倾听所有想法，无论你个人观点如何。
*   复述观点以确保共同理解。
*   清晰陈述所有论点，阐明差异，然后通过共识点进行整合。
*   尝试理解对立的观点。
*   面对证据时更新自己的观点。
*   强调理性和数据。
*   遵循*冲突的黄金法则*：己所不欲，勿施于人。

职场中的层级关系也必然会影响冲突。以下建议旨在帮助职位较高的人士进行有效的冲突管理^(⁷⁵)：

*   腾出时间解决人际冲突。要认识到冲突不会自行消失，拖延只会带来危害。
*   判断冲突是需要与个人、小组、整个团队，还是上述组合来共同解决。
*   为避免盲目附和并鼓励讨论，不要在对话初期就强加个人观点。
*   鼓励团队在小的担忧和摩擦演变成冲突之前就将其摆上台面。
*   寻找机会定下基调，并示范你希望从团队身上看到的行为（例如，感谢他人勇于发言、建设性地提出异议、承认错误等）。
*   通过讨论某些方法为何在特定情境下可能行不通（而非客观错误），帮助建设性地构建和聚焦分歧。
*   花时间解释你的决策，并尽可能让团队参与决策过程。

### 软技能开发：团队协作工件

具备“软技能意识”能改善团队协作和团队产出。但我们如何为所有这些实践腾出时间？如何持续关注团队协作和文化？如何确保冲突在腐蚀团队内部之前得到解决？为了保持足够近的距离以便关注，同时又保持足够远的距离以给予开发一些创造空间，可以在开发流程中添加一些工件。

让我们看看过去我们认为对开发者团队有帮助的一些工件和实践：^(⁷⁶)

*   从个人且保密的*评估*开始，以确定每个人的需求、实践和个人特质，从而帮助组建有效平衡的团队、建立有益的团队协议，并实施适当的培训课程。
*   让团队成员定期（例如，每个开发迭代一次）填写*自我和同伴评估*（个人、保密，仅用于帮助团队协作）：这是一个精简的表格，用于主观反思每个开发者自身的表现、态度和贡献，以及对同伴表现、态度和贡献的看法。为反思留出明确的空间，对于开始全面了解团队协作情况并及时干预至关重要。
*   *玩乐*是创造性思维的精髓。拥有分享表情包或非项目新闻的公共频道，有助于创造一个更人性化、更有活力的空间，而不是只有工作没有娱乐。创造大大小小的团队建设机会。即使是喝杯咖啡、聊聊天充充电的空间，也能将艰难而压力重重的一天转变为值得且充实的共同经历，从而拉近团队距离。大型活动能带来更大的兴奋感，并创造共同的回忆。
*   将错误和学习常态化，无论是在工作的技术方面还是非技术方面。不要搞突然袭击，也不要针对个人。将*例行检查*作为工作流程的一部分，以创造分享机会，并帮助每个人做好接受反馈的准备。如果情况敏感，请提前告知队友，你们需要带着同理心和勇气一起克服不适感。
*   建立*开放的多向沟通渠道*，例如简短的定期更新、面对面的团队工作会议、共享知识库，以及便捷的按需同步和异步沟通渠道。
*   填写包含最低要求的*团队契约*，团队成员可以在此基础上进行扩展，以最好地适应他们共同的舒适度。契约应涵盖团队会议频率（例如，每日站会的时间和地点）、检查和更新异步沟通渠道的频率（例如，每天检查一次 Slack 频道，每次工作结束后更新敏捷看板）、无法联系的及时通知（例如，迟到时告知团队）、所需技术（例如，git）、协议（例如，遵循 git flow）以及可交付成果（例如，站会报告和每两周一次的最小可行产品演示）。契约有助于为团队明确设定期望和责任，并让主管了解团队的实践情况。

将团队协作本身视为一个敏捷开发过程是很有帮助的：团队协作立即开始，频繁检查，并根据需要进行调整。事实上，如果你已经在使用敏捷方法论，敏捷工件可以直接用于监控团队协作：

*   状态*评审/演示/展示*是你审查产品状态的方式。在报告状态时，要清晰说明项目当前所处的位置以及在下次状态报告之前的发展方向：
    *   自上次评审/演示/迭代以来取得了哪些成果？
    *   本次迭代原计划的工作中，哪些尚未完成？
    *   下一步计划是什么？



请注意，评审会和每日站会（参见前一章）都是通过讨论成就、挫折和下一步计划来审视项目状态的。评审会覆盖的时间跨度更长（通常是一个开发迭代周期），并且会邀请产品直接开发人员之外的受众（如主管、产品负责人和客户）参与，以便他们了解进展并提出反馈。你不应粉饰、找借口或掩盖问题。要诚实地面对问题以及你在进度中的位置。虽然没人喜欢报告问题，也没人喜欢听别人报告问题，但这种矛盾必须被接纳并及时处理。缓解问题的最佳方式是让其他人参与进来，共同寻找解决方案，从而维护一种信任与安全的氛围。务必了解你的受众，以便根据他们的专业水平以及与你的工作关联度（管理者、客户和最终用户希望看到项目及其状态的不同方面）来调整内容。对已完成工作进行有代表性的评审，将让你能够从利益相关者那里收集到最相关、最有帮助的反馈。

*   *回顾会议*是团队用来审查、评估和修改其工作流程的方式。回顾会议通常在产品发布之后或产品的每个主要迭代周期之后（例如，在每个迭代或冲刺之后）举行，因此你可以将其作为内置的机会，来观察团队如何反思自身的内部运作，并提供支持、澄清或外部视角。回顾会议的目标是确定当前的流程和实践是否服务于整个团队以及每个团队成员的需求：
    *   我们的团队协作中哪些方面做得好？（我们是否按时完成了计划？我们是否有效沟通和协作？是否每个人都感到被支持、被尊重？）

*   我们的团队协作中哪些方面出了问题？（团队成员是否偏离了预期实践？我们是否在截止日期前才合并代码？我们是否因为任务分工不明确而做了重复的功能？我们是否不得不加班？）

*   展望未来，我们应该如何更新我们的流程、态度或工作环境？

这是软体开发中最重要的会议，开发者和主管都必须为其投入时间、关怀和尊重。想想看，我们常常被教导要抛开个人问题，专注于手头的工作。讨论手头任务之外的任何事情常常不被鼓励，甚至被认为是不专业的。但是，如果我们不能讨论个人需求、情况和情绪，我们又该如何讨论仓库中最后一刻的合并冲突，而不制造紧张和指责呢？为了有效地修改我们共同的流程，我们必须理解我们共同情境的全貌，而人正是这个情境的核心。假设我们在演示前夜很晚才合并代码。演示进行得很糟糕，由于仓促解决的合并冲突导致功能缺失，每个人都又累又沮丧，并且矛头指向了一位开发者。训斥那位开发者，并威胁说如果再发生这种情况会有严重后果，这看起来似乎合理且高效。但这不是正确的方式。这位开发者是否在挣扎，并且害怕寻求帮助？害怕显得准备不足或能力不够？或者他们一直感到孤立无援？又或者他们曾试图表达需要帮助，但没有人站出来？团队是否跳过了每日站会而错过了这个信号？或者这位开发者的功能在最终调整时出现了无法恢复的故障？我们的开发流程是否未能恰当地鼓励小步增量提交和留出足够时间来处理意外情况？有意义的解决方案需要时间和关怀，才能就产品本身以及产品创造者需要改进的领域进行富有成效的讨论。

*   *事后剖析*（团队层面和个人层面）是一种规模更大的*回顾会议*，它让开发者能够反思整个项目、挑战、经验教训和感悟。需要考虑的一些问题包括：
    *   项目的目标是什么，它们是否实现了？

*   项目中哪些方面进展顺利？哪些不顺利？

*   团队中哪些方面表现良好？哪些不理想？

*   团队在整个项目过程中学到的主要经验教训是什么？

*   如果有机会重做这个项目，团队会做出哪些不同的选择？

之后，团队成员应完成个人事后剖析，每个人都能借此机会进一步反思自己的挑战和经验教训：
    *   我为自己设定的项目目标是什么，它们是否实现了？

*   就项目而言，我哪些方面做得好？哪些不理想？

*   就团队而言，我哪些方面做得好？哪些不理想？

*   我在整个项目过程中学到的主要经验教训是什么？

*   如果有机会重新进行这次合作，我会做出哪些不同的选择？

开发者常常将事后剖析视为敷衍了事的工作，他们渴望跳入下一个创意冒险，而不是填写“文书工作”。慢下来；这不是无谓的忙碌。在项目之间抓住机会进行反思，是开发者成长过程中最重要的实践之一。研究表明，反思是将我们的经历转化为有意义学习的关键。^(⁷⁷) 所以，回想一下：收集所发生的事情（技术层面和人际层面），从你未来的视角审视它，并形成新的见解，了解是什么赋予了你力量，是什么阻碍了你，以及你计划如何运用这些新知识继续前进。恭喜你，现在你升级了！

## 结论

在本章中，我们讨论了如何通过关注产品本身和产品创造者的需求来管理项目。为了预测和支持产品的需求，我们考虑了组织方案、分析风险、评估资源需求、估算进度，并规划了不可避免的缺陷管理。为了预测和支持共同创造产品的人员的需求，我们首先考虑了一些可能阻碍团队合作的实践。然后，我们讨论了如何通过支持个体开发者、创建让我们能够共同蓬勃发展的安全文化，以及将冲突视为成长的机会，来成为“软体”。最后，我们介绍了一系列实践和工件，它们可以帮助我们监控团队中复杂涌现的动态。关于良好团队合作的原因的进一步讨论和示例，请考虑阅读 Scott Tannenbaum 和 Eduardo Salas 合著的《高效团队》。^(⁷⁸)
