# 1. 软件开发导论

> *“不仅目前看不到银弹，而且软件的本质决定了未来也不太可能出现——没有任何发明能像电子学、晶体管和大规模集成电路对计算机硬件那样，对软件的生产力、可靠性和简洁性产生同样的影响。我们无法指望每两年就能看到两倍的提升。”*
> 
> ——小弗雷德里克·J·布鲁克斯^(²)

那么，你可能会问自己，为什么这本书叫《软件开发、设计与编码》？为什么不叫《关于编程的一切》或《软件工程》？毕竟，软件开发不就是这些吗？嗯，并非如此。编程是软件开发的一部分，但肯定不是全部。同样，软件开发是软件工程的一部分，但也不是全部。

以下是本书中我们将使用的软件工程定义：软件开发是一个过程，它从用户那里获取一组需求（问题陈述），对其进行分析，设计问题的解决方案，然后在计算机上实现该解决方案。

但这不就是编程吗？嗯，不是。编程实际上只是软件开发的实现部分，或者可能包括设计和实现部分。编程是软件开发的核心，但它并非全部。

那么，它是不是软件工程呢？同样，也不是。软件工程也涉及一个过程，并包含软件开发，但它还包括创建供人们使用的计算机程序的整个管理方面。软件工程包括项目管理、配置管理、进度安排与估算、基线建立与调度、人员管理以及其他若干事项。软件开发是软件工程中有趣的部分。

因此，软件开发是将软件工程的焦点缩小到仅与创建实际软件相关的部分。同时，它也是将编程的焦点拓宽，以包含分析、设计和发布问题。

## 我们在做什么

事实证明，在使用了计算机大约 80 年之后，我们发现开发软件是困难的。学习如何有效、高效且可持续地开发软件同样困难。你并非生来就知道如何做，而且许多人，即使是那些参加过编程课程并在行业工作多年的人，也做得并不特别好。这是一项你需要大量学习和练习的技能。你无法通过阅读书籍——即使是这本——来学会编程和开发。你需要通过开发软件来学习它。当然，这正是其吸引力所在：致力于解决有趣且困难的问题。挑战在于处理你从未做过的事情，甚至是你不知道能否解决的问题。这正是让你一次又一次地回来创建新程序的原因。

学习软件开发可能有几种方法。但我们认为，所有这些方法都涉及阅读优秀的设计、阅读大量代码、编写大量代码，以及深入思考如何解决问题并为其设计解决方案。阅读大量代码，尤其是那些非常优美且高效的代码，能为你提供许多关于如何思考问题以及以特定风格解决它们的好例子。编写大量代码则让你能够尝试在阅读中看到的风格和例子。深入思考问题解决过程，让你能够审视自己如何工作、如何进行设计，并让你从自己的劳动中提炼出那些对你有用的模式；这会让你的编程更具目的性。



## 那么，如何开发软件？

嗯，你首先要做的就是阅读这本书。它当然不会告诉你所有事情，但会为你提供一个很好的入门介绍，让你了解软件开发究竟是怎么回事，以及你需要做什么才能编写出优秀的代码。它有其独特的视角，但这个视角是基于我们合计约 40 年的专业编码经验，以及另外 24 年试图弄清楚如何教会他人进行开发的经历。

尽管软件开发只是软件工程的一部分，但它是每个软件项目的核心。毕竟，*归根结底，你交付给用户的是能正常运行的代码*。一个协同工作的开发团队通常负责创建这些代码。因此，首先，我们或许应该从外部审视一个软件项目，并问一问：这个团队需要做什么才能让项目成功？

为了成功进行软件开发，你需要具备以下几点：

*   **要意识到在项目开始时，你并非无所不知**。软件开发项目并非如此运作。你总会发现新的需求；有些需求会被发现并不像客户想象的那么重要；还有一些原本计划在下一个版本中实现的需求，突然之间变成了头号需求。这被称为*需求变更*。在项目过程中管理需求变更，是软件开发人员可以拥有的最重要技能之一。如果你正在使用新的开发工具（比如一个新的 Web 开发框架），你会发现一些你未曾意识到的局限性，以及一些副作用，这些副作用会迫使你去学习，例如，另外三个工具才能理解它们（例如，你想用的那个 Web 开发工具是基于 Ruby 的，需要特定的关系型数据库系统才能运行，并且需要 Apache 的特定配置才能正常工作）。

*   **一个规模小、整合良好的团队**。小团队比大团队拥有更少的沟通线路。更容易了解团队成员的优点和缺点，理解他们的个性和偏好，并确定谁是特定问题或工具的专家。整合良好的团队通常一起合作过多个项目。让一个团队在多个项目中保持稳定，是团队经理和团队成员个人的一项重要工作。整合良好的团队效率更高，更善于遵守时间表，并且在发布时更有可能产生缺陷更少的代码。保持团队稳定的关键在于，给他们有趣的工作，给他们决定如何工作的自由，并通过消除障碍和帮助解决冲突来促进工作流程。

*   **团队成员之间良好的沟通**。团队成员之间持续、直接的沟通对于日常进展和成功完成项目至关重要。同地办公的团队通常比地理上分散（即使只是分布在建筑物的不同楼层或侧翼）或远程工作的团队沟通得更好，沟通也更频繁。^(³) 这对于那些软件开发站点遍布全球的大型公司来说是一个主要问题。

*   **团队与客户之间良好的沟通**。与客户的沟通对于在项目过程中控制需求和需求变更至关重要。现场或附近的客户允许与开发团队进行持续互动。客户可以对新版本提供即时反馈，并可以参与为产品创建系统测试和验收测试。敏捷开发方法论强烈鼓励客户成为开发团队的一部分，甚至更好的是，每天都能在现场。关于一些敏捷方法的快速介绍，请参见第 2 章。

*   **一个每个人都认同的流程**。每个项目，无论大小，都遵循一个流程。较大的项目需要更多的协调，以及对沟通和配置管理更严格的控制。因此，较大的团队往往更倾向于计划驱动，并遵循需要更多规则和文档的流程。如今，较小的项目和较小的团队往往倾向于遵循更敏捷的开发流程，具有更大的灵活性，并且需要的文档更少。这当然并不意味着敏捷项目*没有*流程；它只是意味着你只做对项目当前阶段有意义的事情，以便你能正确地发现并满足所有需求，按时完成计划，并交付高质量的产品。关于流程和软件生命周期的更多细节，请参见第 2 章。

*   **对该流程保持灵活的能力**。没有哪个项目会完全按照你第一天设想的那样进行。需求会变化，人员会流动，工具会失效或更新，等等。这一点完全关乎如何处理项目中的风险。如果你能识别风险，计划如何减轻风险，然后制定应急计划来处理风险实际发生的情况，你的处境就会好得多。第 4 章讨论了需求和风险。

*   **一个每个人都认同的计划**。你不会在没有算法的情况下就开始编写排序程序，所以你也不应该在没有计划的情况下启动一个软件开发项目。项目计划概括了你将如何实施你的项目。它涉及流程、风险、资源、工具、需求管理、估算、时间表、配置管理和交付。它不必很长，不需要包含项目日常生活的所有细微细节，甚至不需要写下来，但团队中的每个人都需要对其有投入，需要理解它，并且需要同意它。除非每个人都认同这个计划，否则你注定会失败。关于项目规划的更多细节，请参见第 3 章。

*   **要时刻了解项目所处的位置**。这又回到了沟通的问题上。大多数项目都有定期的状态会议，以便开发人员能够“同步”他们当前的状态，了解整个项目的状态，并在团队内部建立一种同志情谊。这对于较小的团队（比如，最多约 20 名开发人员）非常有效，其中许多团队每天都会举行“站立会议”来在每天开始时同步状态。不同的流程模型处理这种“站立会议”的方式不同。例如，计划驱动的模型不要求这些会议，而是依赖团队经理之间相互沟通。敏捷流程通常要求全员每日会议，以促进在高度动态的项目环境中进行持续的团队沟通。

*   **要勇敢地说：“嘿，我们落后了！”** 几乎所有的软件项目在开始时，时间表都过于乐观。这是客户想听到的，是公司想提供的，也是经理和开发人员想承诺的。“当然，我一周内就能搞定！”“我今天下班前给你。”“明天？没问题。”不，不，不，不，不。面对现实吧。在某个时候你会落后。对此最好的做法就是立即告诉你的经理。当然，他们可能会生气。但如果你最终落后了一个月而他们却毫不知情，他们会更生气。弗雷德·布鲁克斯对软件项目是如何变得如此落后的著名回答是“一天一天地落后”。不过，好消息是，你越早发现自己落后了，你拥有的选择就越多。这些选择包括延长工期（不太可能，但确实会发生），将一些需求移到未来的版本中，获得额外的帮助，等等。重要的是让你的经理随时了解情况。



*   *为该项目选择正确的工具和正确的实践*。软件开发最棒的一点在于每个项目都各不相同。即使你正在开发某个现有产品的 8.0 版本，情况也会发生变化。这意味着，对于每个项目，你都需要审视并选择一套合适的开发工具。选择不合适的工具就像试图用螺丝刀钉钉子；你最终或许能完成，但这肯定不轻松、不美观、也不有趣，而且用锤子能在更短时间内钉入更多钉子。即使你必须先获得一把锤子，然后第一次学习如何使用它，你也在过程中升级了你的工具包和技能，为未来更高效工作的能力进行了投资。选择工具时最重要的三个因素是：你正在编写的应用程序类型、目标平台以及开发平台。通常你对这三件事中的任何一件都无能为力，所以一旦你了解了它们，就可以选择能提高生产力的工具。工具选择的第四个且几乎同等重要的因素是开发团队的构成和经验。如果你的团队都是经验丰富、熟悉多种平台的开发者，那么工具选择就容易得多。另一方面，如果你有一群刚毕业的新人，并且目标平台对所有人来说都是全新的，那么你就需要谨慎选择工具，并预留时间进行新工具的培训和实践。

## 结论

软件开发是每个软件项目的核心，也是软件工程的核心。其目标是在不断变化的需求面前，按时、在预算内为用户交付卓越、无缺陷的代码。这使得开发工作变得尤为艰巨。但是，为难题找到解决方案并让你的代码正确运行，这几乎是世界上最酷的感觉。

> *“[编程是]我能想到的唯一一份能让我同时成为工程师和艺术家的职业。它有着令人难以置信的、严谨的技术元素，我喜欢这一点，因为你必须进行非常精确的思考。另一方面，它又有着极其创造性的一面，唯一的真正限制是想象力的边界。这两者的结合正是编程的独特之处。你可以同时成为艺术家和科学家。我喜欢这一点。我喜欢创造位于核心的、真正构成编写程序基础的魔法技巧。第一次看到那个魔法技巧，也就是你程序的精髓，能够正确运行，是编写程序过程中最激动人心的部分。”*
> 
> ——安迪·赫兹菲尔德（第一代 Mac OS 的设计者）^(⁴)
