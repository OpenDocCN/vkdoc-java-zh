# 16. 代码构建

> *大多数时候，当你看到程序员时，他们似乎什么都没做。程序员的一个迷人之处在于，你无法仅凭观察就判断他们是否在工作。他们常常坐在那里，看似在喝咖啡闲聊，或者只是凝视着虚空。程序员真正在做的，是试图理清脑海中那些四处奔窜、互不关联的零散想法。*
> 
> ——查尔斯·M·斯特劳斯^(²⁵⁷)

> *伟大的软件同样需要对美的狂热追求。如果你审视优秀软件的内部，你会发现那些本不该被任何人看到的部件也同样优美。我并非声称自己能写出伟大的软件，但我知道，在对待代码时，我的行为方式如果用在日常生活中，足以让我需要处方药来稳定情绪。看到缩进糟糕或使用丑陋变量名的代码，会让我抓狂。*
> 
> ——保罗·格雷厄姆^(²⁵⁸)

我们终于触及了软件开发的真正核心：编写代码。本章旨在提供一些编写*更优质*代码的技巧，因为我们都能写出更好的代码。在整个讨论中，我们假设你已经*确实*掌握至少一门编程语言。本章将用几种语言举例，每种语言都经过精心挑选，以最好地说明所阐述的观点。

对于计划驱动型流程的开发者（见第 2 章）而言，编码是那条“摇动开发流程这条狗”的尾巴。一旦你完成了详细的需求、架构和详细设计，代码就应该从最终设计中自然流淌出来，对吗？然而，凭借 20 年的行业软件开发经验，我们从未见过这种情况发生。编码是困难的；即使是一个优秀、详细的设计，将其转化为代码也需要大量的思考、经验和知识，即便是小程序也不例外。根据你使用的编程语言和目标系统，编程可能是一项非常耗时且艰巨的任务。另一方面，对于需要数十甚至数百名开发者的大型项目，拥有非常详细的设计对成功至关重要，所以先别急着否定计划驱动型流程。

对于敏捷开发流程而言，编码就是一切。^(²⁵⁹)《敏捷宣言》（[`https://agilemanifesto.org`](https://agilemanifesto.org/)）开篇即言：“可工作的软件胜于详尽的文档。”敏捷开发者倾向于尽早并频繁地创建代码；他们相信应频繁地向客户交付软件，并利用客户的反馈来改进产品。他们欢迎需求变更，并将其视为重构代码、使产品对客户更易用、对开发者更易维护的机会。这并不意味着使用敏捷流程编码会变得更容易；而是意味着你的关注点不同了。在敏捷流程中，你不是专注于尽早敲定需求和设计，而是专注于尽可能快速、频繁地向客户交付可工作的代码。你经常修改代码，整个团队拥有所有代码的所有权，因此只要合适，任何成员都有权修改任何内容。

你的代码有三个受众：

*   代码编译版本的目标机器，即实际执行代码的机器
*   包括你自己在内的、为了理解和修改而*阅读*代码的人
*   使用你创建的程序来完成某些目标的用户

为此，你的代码需要满足需求、实现设计，同时还要具备可读性和易于理解。我们将首先关注这些目标中的可读性和可理解性部分，然后探讨一些与性能和流程相关的问题。

本章并非面面俱到，也不会提供编写优秀代码的所有提示、技巧和技术；这方面有专门的书籍，本章后面及参考文献中会提到一些。关于编码的两本最佳著作是史蒂夫·麦康奈尔的《代码大全 2：软件构建实践手册》^(²⁶⁰)以及亨特和托马斯的《程序员修炼之道》^(²⁶¹)。

史蒂夫·麦康奈尔的《代码大全 2：软件构建实践手册》是一本厚达 960 页的巨著，带你深入了解优秀代码的构成要素。麦康奈尔讨论了从变量命名、函数组织、代码布局、防御性编程到循环控制等方方面面。“软件构建”这个比喻正是源自麦康奈尔的书，它暗示构建软件应用程序类似于建造建筑物。小型建筑（例如菲多的狗屋）更容易建造，需要的规划更少，出问题时也更容易更改（重构）。较大的建筑（你的房子）需要更多细节、更多规划和更多协调，很大程度上因为这不再是单人工作。真正的大型建筑（摩天大楼）需要多个详细的设计和规划层级、紧密的协调以及许多处理变更和错误的流程。尽管建筑构建模型并不完美（它不能很好地处理增量开发，麦康奈尔也讨论了一种“增生模型”，即一层软件被添加到现有层之上，就像珍珠在牡蛎中形成一样），但这个比喻清晰地传达了这样一个观点：软件越大，构建起来就越复杂、越困难。

第二本经典著作是亨特和托马斯的《程序员修炼之道》，全书由 46 个简短章节组成，包含 70 条技巧，清晰地阐述了程序员应有的行为方式。它提供了关于源代码控制、测试、断言、DRY（不要重复自己）原则等一系列主题的实用建议，其中一些我们将在本章后面讨论。亨特和托马斯本人对这本书以及实用主义编程的精髓做了最好的描述：

> *“编程是一门手艺。简单来说，就是让计算机做你想让它做的事（或者你的用户想让它做的事）。作为一名程序员，你既是倾听者，也是顾问，既是翻译者，也是决策者。你试图捕捉难以捉摸的需求，并找到一种方式来表达它们，以至于一台纯粹的机器也能恰当地实现它们。你试图记录你的工作，以便他人能够理解；你试图设计你的工作，以便他人能够在此基础上进行构建。更重要的是，你必须在项目时钟无情的滴答声中努力完成这一切。你每天都在创造小小的奇迹。这是一份艰难的工作。”*



## 一个编码示例

在《代码大全 2》中，史蒂夫·麦康奈尔给出了一个值得审视的糟糕代码示例^(²⁶²)，以便我们开始理解可读性、可用性和可理解性方面的问题。我们已将其从原始的 C++版本转换为 Java：

```
void HandleStuff(CORP_DATA inputRec, int crntQtr, EMP_DATA empRec, Double estimRevenue,
double ytdRevenue, int screenx, int screeny, Color newColor, Color prevColor, StatusType
status, int expenseType) {
int i;
for ( i = 0; i < 100; i++ )
{
inputRec.revenue[i] = 0;
inputRec.expense[i] = corpExpense[crntQtr][i];
}
UpdateCorpDatabase( empRec );
estimRevenue = ytdRevenue * 4.0 / (double) crntQtr;
newColor = prevColor;
status = SUCCESS;
if ( expenseType == 1 ) {
for ( i = 0; i < 12; i++ )
profit[i] = revenue[i] – expense.type1[i];
}
else if ( expenseType == 2 ) {
profit[i] = revenue[i] – expense.type2[i];
}
else if ( expenseType == 3 )
profit[i] = revenue[i] – expense.type3[i];
}
```

那么这段代码有什么问题呢？嗯，有什么没问题呢？让我们列个清单：

*   因为这是 Java，`HandleStuff`方法应该有一个*可见性修饰符*。不，这不是必须的，但你应该始终加上一个。你不是在为编译器编写代码；你是在为将要阅读代码的人编写。可见性修饰符为人类读者明确了信息。

*   方法名很糟糕。`HandleStuff`完全没有说明这个方法的功能。

*   这个方法做了太多事情。它似乎根据`expenseType`计算某个名为`profit`的值。但它似乎还改变了颜色并指示了成功状态。方法的行为应局限于只处理一件事。

*   注释在哪里？没有任何信息说明参数是什么，或者方法应该做什么。所有方法至少应该告诉你这些。

*   布局简直糟透了。它不一致，缩进错误，花括号有时是语句的一部分，有时又是分隔符。而且你确定最后一个右花括号真的结束了这个方法吗？

*   该方法没有保护自己免受错误数据的影响。如果`crntQtr`变量为零，那么第 8 行的除法将引发除零异常。

*   该方法使用了硬编码的*魔法数字*，包括 100、4.0、12、2 和 3。它们从何而来？它们是什么意思？魔法数字是不好的。

*   该方法有太多输入参数。如果我们知道这个方法应该做什么，也许我们可以改变这一点。

*   还有至少两个输入参数`screenx`和`screeny`完全没有被使用。这表明设计不佳：这个方法的接口可能被用于多个目的，因此它“臃肿”，意味着它必须适应所有可能的用途。

*   变量`corpExpense`和`profit`没有在方法内部声明，因此它们要么是实例变量，要么是类变量。这可能很危险：因为实例变量和类变量在类的每个方法中都是可见的，我们也可以在任何方法中更改它们的值，从而产生副作用。副作用是不好的。

*   最后，该方法没有始终如一地遵守 Java 命名约定。

所以这个示例因为一系列不同的原因而成为糟糕的代码。在本章的其余部分，我们将看看这里被违反的一般编码规则，并给出如何使你的代码更具可读性、更易于维护且更不容易出错的建议。

## 规模与聚焦

首要之事：你的类、函数和方法都应该*只做一件事*。这是*封装*背后的基本思想。让你的方法只做一件事可以隔离错误，并使它们更容易被发现。它鼓励重用，因为小的、单一特性的方法更容易在不同的类中使用。单一特性（以及单一抽象层）的类也更容易重用。

“单一特性”这个短语意味着小。你的方法/函数应该很小；20 行可执行代码是一个函数的上限。在任何情况下，你都不应该编写 300 行的函数。我们知道；我们都干过这种事。这并不美观。在第 9 章中，我们讨论了*逐步细化*和*模块分解*。获取一个初始的函数定义并重构它，使其只做一件小事，这将把你的函数分解成两个或更多更小、更容易理解和维护的函数。正如你将在第 18 章中看到的，更小的函数更容易测试，因为它们需要更少的单元测试（它们有更少的代码路径）。*小是好的，更小则更好*。

## 格式化、布局与风格

格式化、布局和风格都与代码在页面上的呈现方式有关，正如你上面所见，这也与其正确性相关。麦康奈尔的*格式化基本定理*指出：“良好的视觉布局展示了程序的逻辑结构，”^(²⁶³) 使其更具可读性，并有助于减少错误数量。因此，良好布局和格式化的目标应该是：

*   准确表示程序的逻辑结构
*   保持一致，这样你所选择的布局风格就很少有例外
*   提高人类读者的可读性
*   易于修改（因为事情总是在变化）



## 通用布局问题与技巧^(²⁶⁴)

大多数布局问题都与代码块的布局有关。块布局有多种类型，有些是语言内置的，有些则需要你自行选择。最常见的三种块布局是：内置块边界、begin-end 块边界，以及模拟内置块。

某些语言为其所有控制结构提供了*内置块边界*。在这种情况下，你无需选择；因为块边界元素是语言特性，你必须使用它。具有内置块边界的语言包括 Ada、PL/1、Lisp、Scheme 和 Visual Basic。例如，Visual Basic 中的 if-then 语句如下所示：

```
if income > 25000 then
statement1
statement2
else
statement3
...
end if
```

在 Visual Basic 中，你无法在不使用结束块元素的情况下编写控制结构，因此块更容易被找到和区分。

但大多数语言没有内置的块边界词法元素，而是使用 *begin-end 块边界*：每个块是由零个或多个语句（其中语句有特定定义）组成的序列，这些语句由 *begin* 和 *end* 词法元素分隔。最常见的 begin 和 end 元素是关键字 `begin` 和 `end`，或者左右花括号 `{` 和 `}`。例如：

**Pascal:**

```
if income > 25000 then
begin
statement1;
statement2
end
else
statement3;
```

**C/C++/Java:**

```
if (income > 25000) {
statement1;
statement2;
} else
statement3;
```

注意，在两个示例中，单个语句被视为一个块，不需要块分隔元素。还要注意，在 Pascal 中，分号是语句*分隔符*符号，因此它必须出现在语句之间，但因为 `else` 和 `end` 不是语句的结尾，所以不能在 `else` 或 `end` 之前使用分号。（感到困惑？大多数人都是如此。）在 C、C++ 和 Java 中，分号是语句*终止符*符号，必须放在每条语句的末尾。这更容易记忆和编写：你几乎可以在每一行的末尾都加上分号，除了花括号之后。一致性是好的。

最后，在格式化块时，你可以尝试在那些没有内置块边界的语言中模拟内置块边界，方法是要求每个块都使用块分隔词法元素。让我们看一个例子：

**C/C++/Java:**

```
if (income > 25000) {
statement1;
statement2;
} else {
statement3;
}
```

在这个例子中，你希望假装左右花括号是控制结构语法的一部分，因此无论块有多大，你都使用它们来分隔块（尽管单行块在这里不需要花括号）。为了强调块分隔符是控制结构的一部分，你将其放在控制语句起始行的同一行。然后，你可以将结束块边界元素与控制结构的起始位置对齐。这并非对内置块元素语言特性的完美模拟，但它非常接近，并且具有一个优点：你不太可能遇到像下面这样因错误缩进而导致的问题：

**C/C++/Java:**

```
if (income > 25000)
statement1;
statement2;
statement3;
```

在这个例子中，`statement2` 和 `statement3` 的错误缩进可能会让读者误以为它们是 `if` 语句的一部分。编译器可不会产生这种错觉；这段代码只有在条件表达式为真时才会执行 statement1。但它始终会执行 statement2 和 statement3。

总的来说，模拟块边界的风格效果非常好，可读性强，并能清晰地展示程序的逻辑结构。同时，为每个块（包括仅包含单个语句的块）都加上块边界也是一个好主意，这样可以消除上面看到的错误缩进的可能性。因此，如果你这样写：

```
if (income > 25000) {
statement1;
}
```

那么，在下面的代码中：

```
if (income > 25000) {
statement1;
}
statement2;
statement3;
```

很明显，`statement2` 和 `statement3` 不属于该块，无论它们的缩进如何（这里的缩进是故意错误的）。这也意味着你现在可以安全地向块中添加额外的语句，而无需担心它们是否在块内，因为新语句将位于花括号内，从而位于块内。

```
if (income > 25000) {
statement1;
statement2;
statement3;
statement4;
statement5;
}
```

## 空白

空白是你的朋友。你不会写一本单词之间没有空格、段落之间没有换行、或者没有章节划分的书，对吧？那么，你为什么要编写没有空白的代码呢？空白允许你在逻辑上分隔程序的各个部分，并对齐块分隔符和其他词法元素。它还能让你的眼睛在程序各部分之间得到休息。让眼睛休息是件好事。以下是一些关于使用空白的建议：

*   使用空行分隔语句组（就像段落一样）。

*   在块内，将所有语句对齐到相同的制表位（默认制表符宽度通常为四个空格）。

*   使用缩进来显示每个控制结构和块的逻辑结构。

*   在运算符周围使用空格。

*   同样，在数组引用和函数/方法参数周围使用空格。

*   不要对 begin-end 块边界使用双重缩进。

## 块与语句风格指南

如前所述，“模拟块边界”风格适用于大多数块结构语言。其他指南包括：

*   *使用比你认为需要的更多的括号*。特别是在所有算术表达式周围使用括号，主要是为了确保你没有搞错优先级规则。

    ```
    fx = ((a + b) * (c + d)) / e;
    ```

*   *一致地格式化单语句块*。使用“模拟块边界”技术：

    ```
    if (average > MIN_AVG) {
    avg = MIN_AVG;
    }
    ```

*   *对于复杂的条件表达式*，*将各个条件放在单独的行上*，并尽可能对齐它们以增加可读性。

    ```
    if (('0' <= inChar && inChar <= '9') ||
    ('a' <= inChar && inChar <= 'z') ||
    ('A' <= inChar && inChar <= 'Z')) {
    mytext.addString(inChar);
    mytext.length++;
    }
    ```

*   *将单个语句* *在大约第 70 列处换行*。这是从 80 列穿孔卡片时代遗留下来的习惯，但它也是让你的代码更具可读性的好方法。非常长的代码行会迫使读者水平滚动，或者让他们忘记行首到底是什么！你可能需要打印代码，而换行的行看起来非常凌乱；你可能想在屏幕上并排查看两段代码；你可能想在半屏上工作，另一半用于 Stack Overflow；等等。使用较短的代码行，你会有很多选择。

*   *非常谨慎地使用* *goto，无论唐纳德·克努特怎么说*。^(²⁶⁵) 有些语言，比如 Java，甚至没有 `goto` 语句。大多数语言都不需要它们（汇编语言除外）。领会克努特论文的精神，只在确实有意义且能使程序更具可读性和可理解性的地方使用 `goto`。

*   *每行只写一条语句*。（不要像参加年度国际混乱 C 代码大赛那样编写代码！^(²⁶⁶）以下写法是合法的，但看起来不好，而且很容易忽略中间的那条语句：

    ```
    g.setColor(Color.blue); g.fillOval(100, 100, 200, 200);
    mytext.addString(inChar);mytext.length++;System.out.println();
    ```

    下面这样看起来好得多：

    ```
    g.setColor(Color.blue);
    g.fillOval(100, 100, 200, 200);
    mytext.addString(inChar);
    mytext.length++;
    System.out.println();
    ```



## 声明风格指南

就像编写可执行代码一样，你的变量声明需要整洁且可读。

*   **在变量使用处附近声明**。大多数过程式和面向对象编程语言都有“先声明后使用”的规则，要求你在任何表达式中使用变量之前必须先声明它。在过去，比如在 Pascal 中，你必须在程序（或子程序）的顶部声明变量，并且不能在块内部声明变量。这有一个缺点，即你可能在真正使用变量之前的许多行代码处就声明了它。（尽管请记住，你的类和函数/方法不应该太长。）Python 是“先声明后使用”规则的一个例外。由于 Python 通常是解释执行（而非编译），解释器会在首次看到变量时猜测其类型，这使得先声明后使用变得不那么必要。

    如今，大多数语言允许你在程序的任何块中声明变量。该变量的作用域就是它被声明的那个块，以及该块内部的所有块（子块、孙块等）。一个好的做法是，在变量被使用的最内层块中声明它们。这样，你就可以*就在那里*看到变量的声明和使用。

*   **每行只声明一个变量**，或者至少每行只声明一种逻辑类型的变量。例如，虽然我们认为

    ```
    int max,min,top,left,right,average,bottom,mode;
    ```

    有点拥挤，但我们会将其重写为

    ```
    int max, min;
    int top, bottom;
    int left, right;
    int average, mode;
    ```

    虽然这不是每行一个，但逻辑上相关的变量被分组在一起。这对我们来说更有意义。

*   **合理地排列声明顺序**。按类型和用途对声明进行分组（参见前面的示例）。

*   **使用空白行分隔声明**。再次强调，空白行是你的朋友。这里的关键思想是让你的声明更显眼，将它们放在使用它们的代码附近，并使其用途更易于理解。

*   **不要嵌套头文件——永远不要！**（这是针对 C 和 C++ 程序员的。）头文件的设计初衷是让你只需定义一次常量、声明全局变量和函数原型，然后就可以在（可能大量的）源代码文件中重用该头文件。嵌套头文件会将其中一些声明隐藏在嵌套的头文件内部。这很糟糕——因为可见性是好的。它可能导致你错误地多次包含同一个头文件，从而引发变量和宏的重定义，导致错误。

    你可能在自己的头文件中嵌套的唯一头文件是系统头文件，例如 `stdio.h` 或 `stdlib.h`，即便如此我们也不喜欢这样做。

*   **不要把源代码放在头文件中——永远不要！**（再次强调，这是针对 C 和 C++ 程序员的。）头文件用于声明，而不是用于源代码。库才用于源代码。将函数放在头文件中意味着该函数会在你包含该头文件的每个地方被重新定义。这很容易导致多重定义——编译器可能直到链接阶段才会发现。唯一应该放在头文件中的源代码是 `#define` 预处理语句中的宏定义，即使这些也应该谨慎使用。

## 注释风格指南

就像空白行一样，注释是你的朋友。现有的每一本编程书都告诉你要在代码中添加注释——但没有一本（包括本指南）确切地告诉你应该在哪里添加注释以及注释应该是什么样子。这是因为如何编写好的、信息丰富的注释属于“视情况而定”的建议类别。一个好的、信息丰富的注释取决于你编写它的*上下文*，因此通用建议非常有限。尽管如此，这里还是有一些主要要点：

*   **先写注释**。按照计划编码是最简单的，这样你就不必一边推理大问题，一边还要细化当前行的语法。尽可能让你的思维保持在同一个层次，从宏观图景到细节。将注释作为你解决方案的伪代码，然后在注释行之间编写代码。最终，你将同时拥有代码和解释。这也有助于你在深入编码时不会忘记解决方案的任何部分（例如，关闭文件输入流、释放某些内存等）。

*   **简要解释推理过程和更大的计划**。注释重在质量而非数量。另一位程序员（或者第二天早上的你）通常能够理解你的代码在做什么，但*为什么*这样做往往更难以理解。因此，不要解释语法或基本的语言结构，但要解释你代码的效用。如果你正在做一些可能不是当前立即需要，但后续会被其他部分用到的事情，请留下注释解释这一点，以免以后被当作不必要的内容意外删除。如果你即将编写几行代码来解决一个复杂问题，通常最好为下一段代码的功能写一个总体规划注释。

*   **保持注释简短且切中要点**。注释必须集中且简洁。你不需要完整的句子，也不需要冠词。

*   **使注释的缩进与其对应的语句对齐**。这对可读性很重要，因为这样注释和相关代码就能对齐。

```
    /* 确保我们有正确数量的参数 */
    if (argc < 2) {
    fprintf(stderr, "用法: %s 文件名\n", argv[0]);
    exit(1);
    }
    ```

这会对可读性产生巨大的负面影响。要么让注释更短，要么将其改为语句上方的块注释。

*   **不要让注释换行。改用块注释**。如果你将冗长复杂的注释附加到源代码行的末尾，通常会发生这种情况，例如

```
    if (argc < 2) { // 确保我们从命令行获得了正确数量的参数
    ```

*   **用空行分隔块注释**，如果你在起始标记 `/*` 之后立即开始注释，并将结束标记 `*/` 放在一行的末尾，如下所示：

```
    /* 确保我们从命令行获得了正确数量的参数 */
    if (argc < 2) {
    fprintf(stderr, "用法: %s 文件名\n", argv[0]);
    exit(1);
    }
    ```

不过我们并不真正推荐这种风格。相反，如果你将起始 `/*` 和结束 `*/` 标记单独放在一行并对齐，那么你就不需要额外的空行了，因为这些行本身几乎就是空白的，已经创造了足够的间隔。

```
    /*
    * 确保我们从命令行获得了正确数量的参数
    */
    if (argc < 2) {
    fprintf(stderr, "用法: %s 文件名\n", argv[0]);
    exit(1);
    }
    ```

*   **所有函数/方法都应该有一个头部块注释**，以便读者知道该方法是做什么的。如果你为方法名和输入参数使用了良好的标识符名称，那么对详细头部注释的需求会有所减轻。尽管如此，至少你应该告诉用户该方法将要做什么以及返回值是什么（如果有的话）。在 C++ 中，你可以这样写：

```
    #include <string>
    /*
    * getSubString() - 从输入字符串中获取一个子串。
    *  子串从索引 start 开始，
    *  一直到但不包括索引 stop。
    *  返回结果子串。
    */
    string getSubString(string str, int start, int stop) { }
    ```



*在 Java 中，请为你的所有方法使用**JavaDoc 注释**。* JavaDoc 内置于 Java 环境中，所有 Java SDK 都附带用于生成 JavaDoc 网页的程序，因此你最好使用它！JavaDoc 能够以极低的成本提供关于类功能的良好概览。只需确保这些注释保持最新即可！

```
    /**
    * getSubString() - 从输入字符串中获取一个子字符串。
    *      该子字符串从索引 start 开始，
    *      一直到索引 stop 结束（但不包含 stop）。
    *  @param str 输入字符串
    *  @param start 起始索引（整数）
    *  @param stop 结束索引（整数）
    *  @return 结果子字符串。
    */
    public String getSubString(String str, int start, int stop) { }
    ```

*   *“**自文档化代码**”是一种理想*，是那些不愿花时间向读者解释其代码的懒惰程序员的圣杯。自文档化代码是一种柏拉图式的编码理想，它假设每个阅读你代码的人都能读懂你的心思。如果你的算法有任何复杂性，或者输入有任何晦涩之处，你都需要进行解释。不要指望读者能理解你代码中的每一个细微之处。解释它。去做就是了。话虽如此，一些编程语言和工具集允许你在代码的注释中嵌入文档标签。JavaDocs 就是这里的典型例子。这种特性属于半自文档化。

关于编写注释的主要建议是——去做。哦，还有，既然你会修改代码——那就再写一次。这是关于注释的第二大难题——让它们保持最新。当你完成某个特定函数的单元测试后，通过更新已有的注释来为该函数编写最终的注释集。这样，你就能在发布的代码中拥有一套相当新的注释。

## 标识符命名约定

正如 Rob Pike 在他那篇关于编程风格的精彩白皮书中精辟指出的那样：“长度并非名称的优点；表达的清晰度*才是*。”^(²⁶⁷) 遵循“金发姑娘”原则，你需要标识符名称既不能太长以至于笨拙烦人，也不能太短以至于信息不足或令人困惑，而是要恰到好处。与注释一样，这对不同的人、在不同的上下文中意味着不同的事情。常识和可读性应占主导地位。

*   *所有* *标识符* *都应是描述性的*。你可能现在理解你的代码做了什么以及如何做的，但最终你可能会忘记自己就是编写它的人。甚至可能第二天你自己都看不懂。它通常也需要对他人有意义，因为我们往往不是软件整个生命周期中的唯一开发者。描述性标识符能让你更容易阅读代码并弄清楚你在凌晨 3 点试图做什么。一个名为 *interestRate* 的变量比 *ir* 容易理解得多。当然，*ir* 更短，输入更快，但请相信我们，在你发布那个程序大约 10 分钟后，你就会忘记它代表什么。合理描述性的标识符可以在编码、调试和维护软件时为你节省大量时间和精力。

*   *OverlyLongVariableNamesAreHardToRead* *（且难以输入）*。另一方面，不要让你的标识符太长。首先，它们难以阅读；其次，它们并没有真正为你的程序上下文增加任何内容；它们占用了页面上太多空间；最后，输入它们非常耗时。

*   *Andtheyareevenharderwhenyoudontincludeworddivisions*。好吧，这太痛苦了……如果你需要多词标识符，请始终使用驼峰命名法来提高可读性：从第二个单词开始，将多词标识符中每个新单词的首字母大写。例如，`maxPhysAddr` 比 `maxphysaddr` 更容易阅读。额外的好处是，文本转语音软件能够正确读出这些标识符，所以#请在你的所有社交媒体标签中也使用它。

*   *单字母变量名难以理解，但有时很有用*。对诸如抵押贷款支付、窗口名称或图形对象之类的东西使用单字母变量名并不是可读性的好例子。`M`、`w` 和 `g` 即使在你的代码上下文中也毫无意义。`mortPmnt`、`gfxWindow` 和 `gfxObj` 则更有意义。这里的一个重大例外是用作索引值的变量——循环控制变量和数组索引变量。在这里，`i`、`j`、`k`、`l`、`m` 等是易于理解的，尽管我们不会反对你使用 `index` 或 `indx` 来代替。

```
    for (int i = 0; i < myArray.length; i++) {
    myArray[i] = 0;
    }
    ```

看起来更简洁，并且与以下代码同样易于理解：

```
    for (int arrayIndex = 0; arrayIndex < myArray.length; arrayIndex++) {
    myArray[arrayIndex] = 0;
    }
    ```

尽管如此，我们建议你在处理嵌套循环、多维数组等情况时，深入挖掘并追求更有意义的字母。例如，我们经常看到用 `i` 表示行，用 `j` 表示列，但使用 `r` 和 `c` 要清晰得多！这样，你在执行例如图像旋转等操作时，犯错误的可能性就会小得多。

*   *遵守编程语言的命名约定* *（如果存在的话）*。

在某个地方，某个时候，你会遇到一份名为“风格指南”或类似名称的文档。如果你在一个有多个开发者的团队中工作，风格指南是个好主意。它们使你的所有代码具有统一的外观和感觉，并使一个开发者更容易修改由其他人编写的代码。几乎所有具有一定规模的软件开发组织都有这样的指南。有时你可以违反指南，但最常见的是，在代码审查期间，你会因不遵循指南而受到批评，并被要求重写你的代码。

风格指南中常见的一套指南是关于命名约定的，它规定了不同类型标识符的名称应该是什么样子。

Java 有一套常见的命名约定：

*对于类和接口*：标识符名称应为名词，使用大写和小写字母数字字符，名称的首字符大写。

```
public class Automobile {}
public interface Shape {}
```

*对于方法*：标识符名称应为动词，使用大写和小写字母数字字符，名称的首字符小写。

```
private double computeAverage(int [] list)
```

*对于变量*：标识符名称可以使用大写和小写字母数字字符，名称的首字符小写。变量名可以包含但不能以 $ 或 _（下划线）开头。

```
double average;
String firstSentence;
```

*对于所有标识符（常量除外）*：应使用驼峰命名法，以便内部单词的首字母大写。

```
long myLongArray;
```

*对于常量*：所有字母应大写，组成单词之间用下划线分隔。

```
static final int MAX_WIDTH = 80;
```

## 重构

代码构建的一个重要部分是在编码时牢记设计，尤其是尽可能保持设计的简单性。当你修复错误或向现有代码添加新功能时，设计的简单性尤为重要。你应该思考你正在处理的代码是否尽可能简单，或者它是否变得过时、陈旧和复杂。如果是这样，请考虑*重构*，即更改代码以更新、简化和清理它。

Martin Fowler 将重构定义为“对软件内部结构进行的更改，旨在使其更易于理解，并在不改变其可观察行为的情况下降低修改成本。”^(²⁶⁸) 重构是所有敏捷方法论的关键，这些方法论强烈鼓励你在每次更改代码时都进行重构。



### 何时重构

重构代码的时机、原因和技巧有很多。在《代码大全 2》中，史蒂夫·麦康奈尔列举了一些。^(²⁶⁹) 马丁·福勒在他的网站上给出了更长的列表。^(²⁷⁰) 让我们看看重构代码的一些原因：

*   *如果存在重复代码*，请记住 DRY（不要重复自己）原则；创建一个封装该代码的新方法，然后根据需要多次调用它。

*   *如果函数或方法过长*，它可能做了太多事情。一个函数或方法应该只做一件事，这样才能保持简短。通常，一个方法不应超过一个屏幕的高度；这大约在 24 到 50 行之间。

*   *如果类的内聚性差（不够紧密）*，它很可能做了不止一件事（有多个职责）；你应该将其拆分为两个或更多类。

*   *如果类接口没有呈现一致的抽象层次*。随着时间的推移，当你对类进行修改后，其接口可能会变得复杂且难以理解。这时就应该通过将方法移到其他类或合并方法来简化接口。

*   *形式参数列表有太多输入参数*，这意味着该方法仅用于将数据传输到另一个方法，或者它做了太多事情。无论哪种情况，都应该简化参数列表。

*   *如果代码的修改需要并行修改多个类或模块*，那么你需要考虑重新组织数据和/或方法以简化结构。

*   *如果一起使用的相关数据位于不同位置*（例如，总是用两个或多个数据来计算第三个数据，但输入数据却位于两个不同的地方），那么它们应该被移到同一个类中。

*   *如果一个方法使用其他类的特性多于使用自身类的特性*，也许它应该被移到那个其他类中。

*   *如果使用方法调用链通过参数传递数据*（输入到某个方法的参数未被使用，只是传递给其他方法），你应该审视一下控制流，看看是否需要做出修改。

*   *如果中间人类本身不做任何事情*（即，一个主要工作只是调用另一个类中的方法的类），那么也许你可以消除这个中间人，直接调用那些方法。

*   *如果实例变量是公开的*，你就打破了面向对象编程的基本理念之一：*信息隐藏*。对象中的所有实例变量都应该是私有的，通过公开的 getter 和 setter 方法进行访问。^(²⁷¹)

*   *如果子类只使用了其父类继承方法中的一小部分*，你可能需要重新思考当前的层次结构。正如麦康奈尔所说，“这通常表明创建该子类是因为父类恰好包含了它所需的例程，而不是因为该子类在逻辑上是父类的后代。考虑通过将子类与其父类的关系从‘是一个’（is-a）关系转变为‘有一个’（has-a）关系，以实现更好的封装；将父类转换为原子类的成员数据，并且只暴露原子类中真正需要的例程。”^(²⁷²)

*   *如果使用注释来解释过于混乱的代码*。俗话说，“不要为糟糕的代码写文档——重写它。”^(²⁷³) 此外，在重构代码时，始终要重新审视你的注释。

*   *如果代码使用了全局变量*。避免使用全局变量（或者至少非常谨慎地使用它们）。如果你在重构的代码中发现了全局变量，请仔细考虑它们，并看看是否可以通过参数将数据传递给方法来消除它们。全局变量很危险，因为它们可能导致副作用，即两个或多个对象试图更改全局变量的值，从而可能引发竞态条件。

*   *如果一个方法在调用另一个方法时需要进行设置或收尾代码*，请重新考虑你的类接口。假设你要向一个`CarParts`库存中添加一个条目，你的代码如下所示：

```
    AddInventory transaction = new AddInventory();
    transaction.setPartID(partID);
    transaction.setPartName(partName);
    transaction.setPartCost(partCost);
    transaction.setDate(transactionDate);
    ```

然后你执行实际的添加操作：

```
    processAdd(transaction);
    ```

你可能会考虑，需要这种类型设置的方法调用是否使用了正确的抽象层次。为了消除设置代码并使代码更简单，你可以将`processAdd()`方法的形式参数列表改为如下形式：

```
    processAdd(partID, partName, partCost, transactionDate);
    ```



### 重构的类型

让我们回顾一些可以改进代码的变更类型：

*   *用命名常量替换魔法数字*。魔法数字之所以不好，有几个原因。首先，如果你多次使用某个魔法数字，并且需要修改它，那么你必须在每一行使用它的代码中修改这个数字。其次，如果后来有人接手你的代码，他们可能不知道这个数字的含义。命名常量允许你只在一个地方修改数字的值，并且能提示这个数字的含义。

*   *用更具信息量的名称重命名变量*。始终力求简洁但有意义（偶尔的基础循环迭代器，如 `i`、`j`、`k` 等除外）。

*   *用表达式替换变量*。检查那些你可能创建了不必要的中间变量，而这些变量仅用于得出最终答案的情况。看看是否可以跳过创建这些变量，而是直接在最终答案的表达式中使用它们的公式或表达式。

*   *用方法替换表达式*。如果某个表达式可能在代码的两个或多个部分重复出现，请将该表达式替换为单个方法，然后直接调用该方法。再次强调：不要重复自己（DRY）。

*   *将一个用于不同用途的变量转换为多个各自用于单一用途的变量*。重复使用单个变量用于两个或多个不同的目的会令人困惑。将所有重复的变量替换为具有恰当名称的新变量，以使代码更易于理解。

*   *创建局部变量，而不是过度使用输入参数*。在方法中重新利用输入参数同样令人困惑。请创建一个专用的局部变量来代替；这只是在系统栈上多增加一个条目而已。

*   *创建类，而不是使用数据基本类型*。如果你使用数据基本类型创建了一个变量（例如使用 *double* 创建了一个名为 money 的变量），并且该变量可能需要额外的行为，那么请用类声明替换这个基本类型。

*   *将一组命名常量（类型码）转换为类或枚举*。之前我们说过使用魔法数字不好，应该使用命名常量。然而，如果你有一组彼此相关的命名常量，应该考虑将它们放在一个单独的类或枚举类型中。这将使你的代码更易于维护和阅读。

*   *将布尔表达式分解为变量*。如果你有一个复杂的布尔表达式，可以考虑将各个子句分离到中间变量中，然后在最后组合结果。

*   *将复杂的布尔表达式移入一个返回布尔值的方法中*。如果你的布尔表达式很复杂，并且可能在程序的其他地方重复出现，请将其放入一个评估该表达式并返回布尔结果的方法中。

*   *使用 `break` 或 `return` 代替布尔循环控制变量*。很多时候你会看到类似这样的 `while` 循环：

```
    boolean done = false;
    while (!done) {
    // 在这里做一些操作
    if (some-expression)
    done = true;
    }
    ```

这会产生一些不必要的工作，这里的建议是用 `return` 或 `break` 语句替换 `done = true;`。

*   *一旦知道答案，就立即从方法中返回*。这也意味着不要使用布尔变量或其他变量来告诉自己已经找到了方法的返回值。一旦知道答案，就直接返回它。

*   *将简单例程中的代码移到其使用的地方*。如果你有一个方法，其包含的代码只在一个地方使用，那么直接将代码内联到使用它的地方，而不是放在一个方法里。

*   *将查询与计算分离*。通常，查询操作只会返回一个值。如果你有一个方法既进行计算又为查询返回值（例如像 `getAverage()` 或 `getTotal()` 这样的方法），考虑将其拆分为两个方法，一个用于计算，一个用于查询。这能让你遵循“方法应该只做一件事”的原则。

*   *合并相似的方法*。你可能有两个不同的方法，它们仅在计算中使用的常量上有所不同。考虑合并这些方法，并将该常量作为输入参数。

*   *将专用代码移入其自己的类或子类*。如果你有代码只被超类的某个子集实例使用，那么请将该代码移入其自己的类或子类中。

*   *将类中的相似代码移入超类*。另一方面，如果你在几个类中有相似的代码，那么创建一个新的超类，并将这些代码向上移入其中。

*   *将具有多重职责的类拆分为多个类*，遵循单一职责原则。

*   *删除一个类*。如果你有一个类最终没做太多事情（例如，它只有一个方法，或者只包含数据成员），那么考虑将该工作移入另一个类。

*   *封装公有实例变量*。如果你有一个声明为公有的实例变量，请将其转换为私有，并创建一个 getter 方法来访问它，遵循信息隐藏原则。

*   *仅在必要时使用* `get()` *和* `set()` *方法*。默认情况下，所有实例变量都应该是私有的，并通过公有的 setter 和 getter 方法访问。仅为那些需要在对象外部使用的实例变量创建 `get()` 方法，并且仅在需要能够更改实例变量值时创建 `set()` 方法。例如，如果你有一个用于零件编号的实例变量，你可能不需要 `set()` 方法，因为零件编号通常不会改变。

*   *隐藏公有方法*。仅当方法需要成为类接口的一部分时才公开它们。与实例变量一样，所有方法都应该是私有的，除非有充分的理由将其设为公有。

*   *合并相似的超类和子类*，如果它们几乎相同，则将其整合到一个单独的类中。



## 防御性编程

所谓防御性编程，是指你的代码应能自我保护，免受不良数据的侵害。这些不良数据可能来自命令行、图形文本框或表单的用户输入，也可能来自文件。此外，不良数据还可能源于程序中其他例程通过输入参数传入的数据，如上文第一个示例所示。

如何保护你的程序免受不良数据的侵害？验证！尽管听起来很繁琐，但你始终应该检查从程序外部接收到的数据的有效性。这意味着你应该做到以下几点：

*   检查命令行参数的数量和类型。

*   检查文件操作。
    *   文件是否成功打开？

    *   读取操作是否返回了内容？

    *   写入操作是否写入了内容？

    *   是否已到达文件末尾（EOF）？

*   检查函数/方法参数列表中的所有值，确保其类型和大小正确。

*   始终初始化变量，不要依赖系统为你完成初始化。

*   检查空指针（Java 中的引用）、不正确的类型、分母为零以及超出范围的值。

例如，下面是一个 C 程序，它从一个文件中读取房屋价格列表，并计算列表中的平均房价。输入文件通过命令行提供给程序。

```
/*
* 程序功能：计算一组房屋的平均售价。
* 输入：通过命令行传入的文件。
* 输出：所有房屋的总价和平均售价，
* 以及文件中的价格数量。
*
* jfdooley
*/
#include 
#include 
int main(int argc, char **argv){
FILE *fp;
double totalPrice, avgPrice;
double price;
int numPrices;
/* 检查用户是否输入了正确数量的参数 */
if (argc \n", argv[0]);
exit(1);
}
/* 尝试打开输入文件 */
fp = fopen(argv[1], "r");
if (fp == NULL) {
fprintf(stderr, "文件未找到: %s\n", argv[1]);
exit(1);
}
totalPrice = 0.0;
numPrices = 0;
/* 读取文件，累加价格并统计房屋数量 */
while (!feof(fp)) {
fscanf(fp, "%10lf\n", &price);274
if (price <= 0.0) {
fprintf(stderr, "错误数据 %10.5f\n", price);
exit(1);
}
totalPrice += price;
numPrices++;
}
avgPrice = totalPrice / numPrices;
printf("房屋数量为 %d\n", numPrices);
printf("所有房屋的总价为 $%10.2f\n", totalPrice);
printf("每栋房屋的平均售价为 $%10.2f\n", avgPrice);

```

### 防御性编程：断言很有用

如果你的编程语言支持断言，那么使用断言可以极大地辅助防御性编程。Java、C99、C11、C23 和 C++ 都支持断言。断言会测试你提供给它的表达式，如果表达式为假，则会抛出错误并通常中止程序。对于你认为可能发生的错误（例如，错误的用户输入），应使用错误处理代码；而对于*绝不应该*发生的错误（例如，循环中的差一错误），则应使用断言。断言非常适合测试你的程序，但由于在将程序交付给客户之前应移除它们（你肯定不希望程序在用户面前崩溃，对吧？），因此它们不适合在生产程序中用于验证输入数据。

### 防御性编程：异常^(²⁷⁵)

防御性编程的一部分是确保你的程序能够从容地从常见的运行时问题（例如不正确的输入）中恢复，从而使程序尽可能长时间地运行（使你的程序健壮）。虽然*断言*用于处理绝不应该发生的错误，但*异常*应用于处理可恢复的错误，这些错误是正常操作中预期会发生的（例如，FileNotFound 异常）。

你应该充分利用你所使用的编程语言中内置的*异常处理*机制。你的程序如何处理异常取决于语言的能力（我们稍后会讨论 Java 中的异常处理）。异常处理机制会向你的程序提供关于刚刚发生了什么错误的信息。然后由你来决定如何处理。通常，在异常处理机制中，你的程序有两个选择：1）自行处理异常，或者 2）将异常传递（即抛出）给调用它的代码，让那段代码来处理。

### 防御性编程：错误处理

与验证一样，你最有可能在输入数据中遇到错误，无论是命令行输入、文件处理，还是来自图形用户界面表单的输入。这里我们讨论的是运行时发生的错误。编译时和测试时的错误将在下一章关于调试和测试的内容中介绍。其他类型的错误可能包括：你的程序计算错误的数据、与你的程序交互的其他程序中的错误、操作系统（例如，竞态条件）以及交互错误（即你的程序在与另一个程序通信时，错误出在你的程序一方）。

错误处理的主要目的是让你的程序尽可能长时间地正确运行。当程序运行到无法继续的地步时，它需要尽可能好地报告问题所在，然后优雅地退出。退出是错误处理的最后手段。那么你应该怎么做呢？好吧，我们又回到了“视情况而定”这个答案。你应该怎么做取决于错误发生时程序的上下文以及它的目的。你不会以处理心脏起搏器错误的方式来处理电子游戏中的错误。在任何情况下，你的首要目标都应该是尝试恢复。

恢复意味着你的程序需要尝试忽略错误数据、修复它，或者用其他有效的数据来替代它。关于错误处理的进一步讨论，请参见 McConnell^(²⁷⁶)。以下是一些如何从错误中恢复的示例：

*   你可以*忽略错误数据并继续运行*，使用下一个有效的数据。假设你的程序是数字压力表中的嵌入式软件。你每秒采样返回压力的传感器 60 次。如果传感器有一次未能提供压力读数，你是否应该关闭压力表？可能不需要；一个合理的做法是跳过这次读数，并准备读取下一个到达的数据。但如果压力传感器连续多次跳过读数，那么传感器可能出了问题，你应该采取不同的措施（例如，发出警报或关闭程序所监控的机器）。

*   你可以*用最后一个有效数据来替代*缺失或错误的数据。再次以数字压力表为例，如果传感器错过了一次读数，由于每个时间间隔只有六十分之一秒，缺失的读数很可能与上一次读数非常接近。在这种情况下，你可以用最后一个有效数据来替代缺失的值。

*   有时你可能没有任何先前记录的有效数据。你的应用程序使用异步事件处理程序，因此你没有数据历史记录，但你的程序知道数据应该在一个特定的范围内。假设你提示用户输入一个工资金额，而返回的值是一个负数。显然没有人会得到负数的薪水，所以这个值是错误的。处理这个错误的一种方法（可能不是最好的）是*用范围内最接近的有效值来替代*，在这种情况下是零。虽然这并不理想，但至少你的程序可以继续运行，并且在该字段中拥有一个有效的数据值。



*   在 C 程序中，几乎所有系统调用和大多数标准库函数都会返回一个值。你应该测试这些返回值！大多数函数会返回表示成功（非负整数）或失败（负整数，通常为-1）的值。有些函数会返回一个表示其成功程度的数值。例如，`printf()` 系列函数返回打印的字符数，而 `scanf()` 系列函数返回读取的输入元素数量。大多数 C 函数还会设置一个名为 `errno` 的全局变量，其中包含一个整数值，即发生的错误编号。错误编号列表位于名为 `errno.h` 的头文件中。`errno` 变量中的零表示成功。任何其他正整数都是发生的错误编号。由于系统会告诉你（1）发生了错误以及（2）它认为的错误原因，你可以采取多种不同的方式来处理它，包括仅仅*报告错误*并退出。

例如，如果你尝试打开一个不存在的文件，程序

```
    #include 
    #include 
    #include 
    int main(int argc, char **argv) {
    FILE *fd;
    char *fname = “NotAFile.txt";
    if ((fd = fopen(fname, "r")) == NULL) {
    perror("File not opened");
    exit(1);
    }
    printf("File exists\n");
    return 0;
    }
    ```

将返回错误消息

```
    File not opened: No such file or directory
    ```

函数 `perror()` 会读取 `errno` 变量，并使用提供的字符串加上与错误编号对应的标准字符串，将错误消息写入控制台的标准错误输出。这个程序也可以提示用户输入另一个文件名，或者使用一个默认文件名。这两种方式都能让程序继续运行，而不是因错误而退出。

错误处理和恢复还有其他技术。这些示例应该能让你了解在程序中可以做什么。这里要记住的重要思想是，尽可能尝试恢复，但最重要的是，***不要静默失败！***

### 防御性编程：Java 中的异常

一些编程语言内置了错误报告系统，当错误发生时，它会通知你，并让你自行决定如何处理。这些通常会导致程序悲惨死亡的错误被称为*异常*。遇到错误的代码会*抛出*异常。一旦有东西被抛出，通常最好让程序的某个部分*捕获*它。因此，在编写代码时，你需要了解异常的两个方面：

*   当一段代码可能遇到错误时，你*抛出*一个异常。像 Java 这样的系统会为你抛出一些异常。这些异常在 Java API 文档的 `Exception` 类中列出（参见 [`https://docs.oracle.com/en/java/javase/20/docs/api/index.html`](https://docs.oracle.com/en/java/javase/20/docs/api/index.html)`)`）。你也可以编写自己的代码来抛出异常。我们将在本章后面给出一个示例。

*   一旦抛出异常，程序的另一部分必须*捕获*它。如果你在程序中什么都不做，这个*未捕获的异常*会沿着调用栈向上传播到 Java 虚拟机（JVM）并在那里被捕获。JVM 会终止你的程序，并提供一个堆栈回溯信息，该信息应能引导你回到代码中最初导致异常抛出的位置，并显示你是如何到达那里的。另一方面，你也可以编写代码来封装可能产生异常的函数调用，并使用 Java 的 `try-catch` 机制自行捕获它们。Java 要求某些异常必须被捕获。稍后你会看到一个示例。

Java 有三种不同类型的异常：错误、受检异常和运行时异常。

*错误*，另一方面，是通常与程序外部发生的事情相关的异常，除了优雅地失败之外，你对此无能为力。你可以尝试捕获错误异常并为用户提供一些输出，但通常你仍然必须退出。

*受检异常*是那些你应该使用异常处理器自行捕获和处理的异常；它们是你应该在设计和编写代码时预见并处理的异常。例如，如果你的代码要求用户输入文件名，你应该预见到他们会输错，并准备好捕获由此产生的 `FileNotFoundException`。受检异常必须被捕获。

第三种类型的异常是*运行时异常*。运行时异常都源于程序运行时内部出现的问题，并且几乎总是表明你的代码中存在错误。例如，`NullPointerException` 几乎总是表明你的代码中存在一个 bug，并以运行时异常的形式出现。^(²⁷⁷) 错误和运行时异常统称为*非受检异常*（这是因为你通常不会试图捕获它们，所以它们是不受检查的）。在这个程序中，你故意引发了一个运行时异常：

```
public class TestNull {
public static void main(String[] args) {
String str = null;
int len = str.length();
}
}
```

这个程序可以正常编译，但当你运行它时，你会得到如下输出：

```
Exception in thread "main" java.lang.NullPointerException
at TestNull.main(TestNull.java:4)
```

这是一个典型的运行时异常。没有必要捕获这个异常，因为唯一能安全做的事情就是退出。如果你确实捕获了它，程序可能看起来像这样：

```
public class TestNullCatch {
public static void main(String[] args) {
String str = null;
try {
int len = str.length();
} catch (NullPointerException e) {
System.out.println("Error. Found a pointer: " + e.getMessage());
System.exit(1);
}
}
}
```

这给了我们输出：

```
Error. Found a pointer: null
```

请注意，`getMessage()` 方法将返回一个包含 Java 认为合适的任何错误消息的字符串（如果有的话）。否则，它返回 `null`。这比上面的默认堆栈跟踪信息量要少一些。

让我们用 Java 重写上一节中的简短 C 程序，并说明如何捕获一个*受检异常*。

```
import java.io.*;
import java.util.*;
public class FileTest {
public static void main(String [] args) {
File fd = new File("NotAFile.txt");
System.out.println("File exists " + fd.exists());
try {
FileReader fr = new FileReader(fd);
} catch (FileNotFoundException e) {
System.out.println(e.getMessage());
}
}
}
```

当你执行 `FileTest` 时得到的输出是：

```
File exists false
NotAFile.txt (No such file or directory)
```

顺便说一下，如果你在上面的程序中没有使用 `try-catch` 块，它将无法编译。你会得到这个编译器错误消息：

```
FileTestWrong.java:11: unreported exception java.io.FileNotFoundException; must be caught or declared to be thrown
FileReader fr = new FileReader(fd);
^
1 error
```

记住，在 Java 中，受检异常**必须**被捕获。这种类型的错误不会出现在非受检异常中。这远非你应该了解的关于 Java 异常和异常处理的全部内容；开始深入研究 Java 教程和 Java API 吧！
