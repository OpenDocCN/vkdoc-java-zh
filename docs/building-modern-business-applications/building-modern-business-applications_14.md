# 11. 事件物化器

事件物化器负责维护一个有状态的视图模型，该模型将用于满足 GraphQL 查询和订阅。在本章中，我将讨论该组件如何消费来自我们系统事件日志的数据，并且也可以消费来自协作系统的事件流。我还将描述这种视图维护方法如何能够直接实现 GraphQL 订阅。我将讨论事件物化器可能如何失败，以及如何对其进行测试以最小化这些故障的发生。

## 定义视图模型

事件物化器维护的视图模型存在的目的是为了满足 GraphQL 请求。我们可以利用这种特异性来为我们在 GraphQL 模式中定义的查询定制模型。更具体地说，存储在视图模型中的状态形状应该能够非常直接地传递给你的 GraphQL 引擎，只需最少的转换。如果你的模式中为了调用者易用性而反规范化了数据，那么在存储模型中也应将其反规范化。

在第 5 章中，我讨论了业务规则，其中一种类型是可派生数据。在可能的情况下，优先预先计算这些派生数据，而不是为每个请求按需计算。一次性完成工作可以带来效率提升。系统将能够支持更多的查询，因为每个查询的计算成本更低。可以保留中间输出用于增量计算，也可以随着输入的变化存储不同版本的结果。后者对于调试很有价值。如果知道请求的时间，就有可能确定返回的数据而无需重新计算。

事件应包含来源信息、生成时间、提交产生该事件的命令的用户身份，以及在你的领域中可能有用的其他元数据。在物化视图模型时，可以将这些信息放置在适当的位置，以告知用户谁最后接触了某条数据。事件的序列号也可以类似地使用。在上一章中，我提到向变更传递一个 `asOfRevision` 值，以支持命令处理器中的乐观锁。考虑在你的领域中，更细粒度的乐观锁是否合适，并在你的模式中包含修订号。

在我的工作中，我喜欢创建一个可重用的 GraphQL 类型，其中既包含来源信息也包含修订号。不幸的是，GraphQL 中还没有一个正式的 `DateTime` 标量，但有一个开放的提案。

不要过度追求模型的未来适应性。将视图模型视为临时的。我们系统的权威状态存储在事件日志中。如有必要，可以重新创建视图模型，我将在本章后面更深入地讨论这个主题。

虽然我主要讨论的是用于 GraphQL 的模型，但如果你需要同时支持其他 RPC 方法，请考虑在你的服务中拥有多个事件物化器实例，每个实例维护一个定制的视图。

## 事件物化器行为

事件物化器的行为很直接。它从事件日志中消费新事件，并为每个事件调用一个函数来更新有状态的视图。你的视图可以完全是临时的，为每个应用程序实例维护在内存中。在实践中，我相信你最终会使用某种外部状态存储。在为一个事件更新视图后，它接着处理下一个事件。*至关重要的是*，事件必须按顺序处理，否则视图的状态将会损坏。

### 事务

我*强烈*建议使用事务性存储。在前一节中，我鼓励在视图模型中使用反规范化。单个事件可能涉及更新多条数据，而你系统的调用者应始终看到一致的状态。他们不应该能够读取部分物化的事件。

还可以提供额外的一致性保证，即视图相对于某个命令是一致的。在第 9 章中，当我描述命令处理器时，我引入了一个数据侧通道，即已处理命令日志。已处理命令日志包含每个命令的一个事件，该事件有一个指向处理该命令所创建的最新事件的指针。可以利用这个信息将事件日志窗口化为与产生每个批次的命令相对应的批次。然后，每个批次就是事件物化器的一个事务边界。这确保了系统的读取操作始终遇到在每个变更结束时一致的状态。



### 一致性

事件物化器维护着一个有状态的视图。如果系统需要重启，它必须存储关于最后物化事件的元数据，否则它会重新处理事件并损坏视图。此状态应作为视图内的一个附加结构进行存储。我使用术语*检查点*来指代此状态。检查点将包含最近物化的事件序列号，并在事件物化时以事务方式更新。当系统重启时，它将使用此序列号来确定从何处开始重新消费事件日志。

检查点还可用于为变更操作提供写后读一致性。在第 10 章中，我描述了成功的变更操作如何仅产生一个单一值，即最终事件的序列号。此序列号随后可作为 GraphQL 查询的可选参数，用于指示事件物化器必须至少已物化到此事件。

需要注意的是，这仅适用于已执行变更操作的调用方。只有他们知道系统状态已更改，因此也只有他们知道自己应等待系统执行操作。这种行为是最终一致性，这是我在第 6 章讨论时间时探讨过的主题。如果来自两个不同调用方的变更操作和查询请求先后到达，则查询不保证能看到变更操作的结果，因为该变更可能仍在命令处理器和事件物化器中处理。如果这两个调用方协同工作，则他们有责任将变更方的 `visibleAsOfRevision` 共享给查询方。这虽然是额外的工作，但同时也明确了这两个调用方之间的耦合关系。我相信从长远来看，组合系统中的依赖关系会得到更好的暴露，从而有助于可维护性。如果这个认识令人不安，可以考虑如何从架构上将其从调用方中消除。

### 多数据源

虽然事件物化器的主要输入是我们系统的事件日志，但在计算派生数据时，有时需要使用来自其他系统的数据。

在第 9 章描述命令处理引擎时，我提到我们事件日志的泛化形式是消息日志。事件物化器不必局限于消费我们系统的事件日志。它可以被泛化，以消费来自任何消息日志的数据。

有一类业务规则涉及显示逻辑，这是我在第 5 章中介绍过的。如果显示逻辑需要来自另一个系统的信息，并且该系统的数据可以表示为消息日志，那么将其与我们的事件物化器集成就会变得直接。这也符合响应式系统的一个特性，即消息驱动。

具有单一事件日志源的事件物化器遵循以下循环：

1.  从事件日志中获取下一个事件。
2.  调用物化函数，用该事件更新有状态视图。
3.  用该事件的序列号更新检查点。
4.  返回步骤一。

为了将其泛化以支持多个数据源，需要一种机制来知道事件源自哪个事件日志，以便为每个源维护一个检查点。为了保持事件物化器的顺序性，它正在消费的所有事件日志应该被合并。从事件物化器的角度来看，它将物化的事件流可以按源事件日志进行分区。在此上下文中，一个合理的命名是将每个事件日志视为事件物化器的一个**分区**。在进入事件物化器之前，每个事件都应被增强，以包含其来源的分区信息。然后，检查点组件将使用此信息来跟踪每个分区的检查点。当系统重启时，每个分区的检查点可以正确地恢复对每个事件日志的消费。

当合并多个事件日志时，事件将根据其到达时间交错排列。我们维护有状态视图的显示逻辑代码需要得出相同的结果，而与到达顺序无关。

与在请求时维护派生数据相比，在物化时维护派生数据也提高了我们系统的响应能力和弹性。我们的系统响应更快，因为数据派生所需的时间仅在事件物化时发生一次，而不是作为服务请求时间的一部分。通过在事件物化时消费外部数据，我们还从我们系统的响应时间中消除了外部系统响应时间的可变性。我们系统的弹性也得到了增强。与另一个系统通信失败不会影响我们系统服务读取请求的能力。它也不会影响事件物化过程，因为这将表现为接收新事件的一段静默期。

我发现这种从外部系统消费数据的模式非常强大。我的理想是拥有一组都像我们系统一样实现的系统。代表外部系统的消息日志可以是一个 GraphQL 订阅（我将在本章后面讨论的一种机制），用于宣布该系统的有状态视图的变更。它也可以是该系统事件日志的一个版本，理想情况下，该版本已将事件映射为一种在系统间作为契约而持久的格式。

也可以使用此范式与请求/响应式系统集成。外部事件日志的实现可以在后台向其他系统发起请求，并将其结果呈现为事件日志。同样，这是一种弹性的集成方式，因为通信延迟和故障不会影响事件物化器。



## 演进视图模型

在本章前面部分，我介绍了为不同消费者维护不同视图而使用多个事件物化器的概念。同样的思路也可以扩展到支持单个视图的演进。回顾一下，视图模型应被视为临时性的，因为状态的权威存储是事件日志。这意味着破坏性的模式变更并非我们系统的限制。事件物化器可以“回滚”到事件日志的起始位置，从而完全重建视图。

在介绍第 4 章的响应式原则时，关于弹性，我提到部署总会使系统暂时进入非预期的状态。要么是运行的实例数少于预期（如果在启动新实例前停止所有旧实例），要么是实例数多于预期（如果先启动新实例，待旧实例能处理请求后再停止它们）。

正是后一种状态（运行更多实例）类似于拥有多个事件物化器。新旧版本的事件物化器可以同时运行！包含向后不兼容视图模型变更的新版本代码，可以将其有状态视图存储到新位置。这带来了多重好处。首先，在新代码重建视图期间，旧版本代码可以继续处理请求。其次，如果新代码出现意外问题，由于先前代码使用的视图保持不变，部署可以安全回滚。

要安全使用此技术，关键在于将新增事件的变更与破坏性视图模型变更分开处理。如果新代码库处理命令时产生了旧系统无法处理的事件，那么回滚将无法实现。

在第 2 章描述的原地修改型系统中，向后不兼容的模式变更管理起来非常棘手。从表中删除列是不可逆的操作；必须从备份中恢复数据库本身。有经验的开发者会回避向后不兼容的变更，因为处理过此类问题的经历让他们深知其风险之高。即使是可逆的变更（例如更改列的数据类型）也可能难以管理。数据库模式管理通常与应用程序代码库集成。系统启动时，会执行任何必要的正向迁移。如果代码回滚，旧版本代码不知道如何撤销新版本所做的更改。这就需要单独运行回滚脚本来逆转迁移。单独运行脚本的风险在于，这种操作不太可能频繁执行，从而在需要回滚变更的压力下正确执行增加了额外风险。这最终成为另一个论据，说明为何要按我描述的方式将业务应用程序构建为响应式系统。我所描述的结构和实践引导开发者走向降低风险的做法。

## 向他人传达变更

业务应用程序并非孤立存在。虽然它们作为所管理信息的真实来源，但应用程序也需要与其他系统协作。这可能是通过应用程序用户界面与人类协作，或是与其他软件系统协作。无论哪种情况，系统都应能够*主动*传达变更。我所阐述的架构描述了一个能够感知数据何时发生变化的系统。这旨在有效向他人传达这些变更。这种沟通分为两大类。第一类是向可能受益的各方传达系统内部的状态变更。第二类是一类业务规则（副作用），明确涉及我们自身系统之外的状态变更。



### GraphQL 订阅

我们系统在第 7 章中自我施加的一个约束是支持实时用户界面的能力。之所以存在这个约束，是因为我认为将这种能力引入业务应用以满足用户对软件系统行为的期望至关重要。事件物化器为系统的 GraphQL 端点维护了一个有状态视图。每次物化事件后，都有机会传达发生了什么变化。

GraphQL 订阅是一种传输事件的通用机制。GraphQL 规范并未定义订阅流的任何语义，仅说明它是字段返回类型的一个序列。

我喜欢使用的模式被称为*实时查询*模式。GraphQL 社区内一直断断续续地讨论着如何通过使用查询上的 `@live` 指令来正式化这种模式。其结果将与我描述的一致。

我设计我的 GraphQL 模式时，使得订阅和查询之间的类型可以共享，如下所示：

```
type Customer {
id: ID!
name: String
}
type Query {
customerById(id: ID!): Customer
}
type Subscription {
customerById(id: ID!): Customer
}
```

这个简短的 GraphQL 模式片段定义了一个 `Customer` 类型和一个用于访问状态快照的查询 `customerById`。它还定义了一个同名的订阅字段，复用了相同的类型。

查询行为是直接的 GraphQL；请求查询字段将在单个响应中返回客户的当前状态。不幸的是，GraphQL 模式不包含定义订阅语义的信息（这也是 `@live` 指令存在的部分原因）。对于实现实时查询模式的订阅，订阅调用者将收到当前状态的快照。当请求的数据发生更改时，将发送额外的快照。订阅是无限的；如果查询结果存在，订阅将继续随着查询结果的变化而更新。

从概念上讲，我们系统中代表变化的事件流，由事件物化器物化为一个有状态视图。当这个有状态视图发生变化时，订阅允许以一系列状态的形式观察这些变化。

我发现这种通信范式与用户界面配合得很好。UI 通常显示完整状态，并在每次底层变化时传达完整状态，这简化了它们的实现。作为实现说明，每次订阅更新都传达完整状态的语义，并不意味着底层传输需要在每次更新时发送完整状态。订阅是有状态的，因此实现可以通过仅传达更新之间的差异来优化网络层。这种优化可以存在于传输层，为开发者提供每次都能传达完整状态的假象。

订阅的使用并不仅限于 UI。可能还有其他协作系统需要使用我们系统的状态。它们也可以使用 GraphQL 订阅来访问我们系统的变化。当协作系统只需要了解状态变化，而不关心*为什么*会有状态变化时，这种方式效果最佳。如果协作系统需要知道原因，那么这种交互可能更适合被视为我们系统的副作用。如果可能，为 UI 和协作系统使用相同的机制，有助于限制我们系统拥有的外部契约数量。这里需要找到一个平衡点。作为开发者，限制外部契约的数量可以减少进行更改时所需的工作量。与此相矛盾的是，在进行更改时需要与更多方进行协调。平衡点在哪里并没有正确答案。我认为重要的是要有意识地做出这个决定。

每次物化事件后，你可以利用事件中包含的信息来确定受影响的 GraphQL 订阅集合。这也可能是你为有状态视图选择的数据存储所提供的功能。当有状态视图更新时，它可以生成或促进变更事件的传播，我们的系统消费这些事件并用于驱动 GraphQL 订阅。

### 其他副作用

除了通知用户和协作系统关于我们系统状态的变化外，我们的系统可能还参与某些业务流程。在第 5 章中，介绍我如何思考业务规则时，我描述了一类我称之为*副作用*的规则。副作用可以进一步分为两类：依赖于视图状态的，以及不依赖于视图状态的。对于不直接依赖于视图状态的副作用，请考虑作为开发者的你是否存在间接依赖。如果系统中发生了意外情况需要调查，确保副作用在事件物化之后发生是否有助于未来潜在的调试工作？也许不会，但我总是鼓励在设计系统时进行这种思考。未来的你可能会感谢自己。

一个依赖于视图的副作用的例子是维护另一个衍生数据存储。我们用于存储视图的有状态存储可能对于通过标识符直接查找很高效，但可能无法高效地提供其他所需的产品功能，例如全文搜索或分面搜索。可能存在另一个能够更高效地响应这些查询的存储，并且为了操作简便，该存储会加载来自我们服务的 GraphQL 查询结果。是的，你可以有一个额外的事件物化器来维护此存储中的状态，但这需要在两个存储的代码重复与让一个存储成为另一个的衍生之间进行权衡。

这种类型的副作用可以通过跟踪系统的事件日志来运作，但需要延迟处理每个事件，直到事件物化器成功物化了该事件。当副作用处理每个事件时，它可以重用与事件物化器相同的检查点机制。其进度可以作为事件物化器检查点内的一个额外分区来存储。这里一个重要的细节是，如果视图被重建，副作用将再次发生，因为存储在视图中的状态将会丢失。对于手头的例子，即基于我们的视图更新衍生数据存储，这是期望的行为。对于另一类，即参与业务流程，这将是不期望的。

作为业务流程一部分的副作用，例如向银行发送已批准的付款，或向新客户发送欢迎邮件，在视图重建时不应重复执行。它们需要一个类似于事件物化器所使用的检查点机制，但必须是持久的。通过将执行操作的意图以及操作的处理结果都建模为命令和事件，可以复用命令处理器本身来进行这种状态保持。无论选择哪种路径，并且与本书中关于业务应用的想法无关，该操作都应该是幂等的。我们系统内的信息，例如事件日志序列号，可以帮助构建一个幂等令牌，业务流程的下一步可以使用该令牌来确定是否存在重复执行操作的尝试。故障会发生，操作会被重试。虽然让客户收到第二封欢迎邮件可能可以接受，但多次发送付款则不行。



## 管理故障

故障总会发生，无论是有计划的还是意外的。事件物化器维护着一个有状态视图，这通常意味着有一个外部系统在管理状态存储。这个外部系统不可能 100%可用，至少它很可能需要软件更新。意外故障也随时可能发生，例如事件物化器在处理特定事件时出现错误。因此，有必要明确地为故障制定计划，而不是自认为是不会写出 bug 的完美开发者，并认为意外故障的可能性足够低。我们的系统对业务至关重要，这正是我们首先编写它们的原因。

如果有状态视图存储本身不可用，就会影响从系统读取数据的能力。然而，值得注意的是，系统将继续接受写入操作。命令处理器与事件物化器是解耦的，事件物化器中的故障不会停止命令处理。不过，根据乐观锁的使用位置，写入操作可能会被有效禁用，因为如果无法观察最新的序列号，调用者将无法读取该值以将其提供给变更操作。

事件物化器按顺序从事件日志中消费。如果在处理某个事件时发生故障，事件物化器**必须**停止。如果事件物化器试图“跳过”一个事件，其视图将变得不一致。然而，整个事件物化器并不需要停止。如果事件物化器正在处理多个输入分区，处理一个事件时的故障只需停止该分区，而不是整个物化器（尽管你可能会发现，停止整个物化器的策略在初始实现时更简单）。

当解释这种行为时，某些人可能会觉得不可取。你可能会被鼓励将命令处理和事件物化耦合到一个要么全部成功、要么全部失败的工作单元中。我对此持谨慎态度，因为我认为这是试图让系统架构屈从于个人开发实践。其他人担心的风险与编程错误有关，而非视图存储故障，因为这种耦合在视图存储故障时并不会改变行为。我相信，并且在实践中也发现，测试可以充分缓解这种风险，这是下一节的主题。测试中仍然可能存在漏洞，我的后备方案则是本书范围之外的开发实践。通过拥有快速的构建流程并实践持续交付，理想情况下，你应该能够及时地将所做的更改投入生产。你可以将这些流程的持续时间作为部署任何代码更改所需的最短时间。这是与业务伙伴沟通以了解系统可用性目标的起点，这些目标可以指导实现决策。当事件物化器因编程错误而无法处理事件时，它不会影响系统服务读取请求的能力。能够处理此类部分故障本身就是一种弹性形式。

## 测试

事件物化器是确定性的。给定一个输入事件，它每次都应执行相同的操作。这是“应该”，而非“一定”，因为操作可能会有所不同：当事件物化器从多个输入分区消费时。确定性仍然存在，但可能操作的数量是分区数量的函数。在有两个分区且事件分别为 `A` 和 `B` 的简单情况下，有必要处理 `A` 后 `B`，以及 `B` 后 `A` 的情况。不过，这也取决于你的实现。处理 `A` 和 `B` 可能是独立的，并在处理完 `A` 或 `B` 后执行一个公共例程。在我的许多工作中，测试事件物化器如何处理事件并非作为事件物化器的测试来执行，而是作为整个系统的测试来执行。为了测试事件物化器如何处理事件，有必要查询有状态视图。这与针对该视图测试 GraphQL 查询和订阅有关。我发现测试整个系统更可取，原因我将在下一章深入解释。

有状态视图模型也是检查命令处理器结果的一个机会。如果有状态视图存储在允许声明式约束的系统中，例如关系型数据库，那么可以利用该存储的能力。我发现交叉检查唯一约束和外键关系非常有益。如果处理某个事件违反了约束，它将指向命令处理器中的错误，而非事件物化器。这是应该在测试中捕获的错误类型的一个例子，因为如果这种情况发生在已部署的系统中，补救措施将需要移除视图约束，然后按照第 6 章所述，将其作为业务规则变更来处理。

虽然测试整个系统对于事件物化器如何处理任何给定事件是有意义的，但事件物化器本身可以，也应该被隔离测试。在测试整个系统时，重要的是要最小化这些测试因事件物化器本身的错误而失败的可能性。事件物化器的隔离测试应针对一组与你的领域无关且极其简单的事件来编写。我使用简单的领域，并且像命令生成器测试一样，经常使用一些个性化的、甚至有点傻的测试，比如计算我的猫被喂了多少次。

隔离测试事件物化器也使模拟故障更加直接。可以编写测试来模拟有状态视图存储的故障，以及围绕软处理程序故障（例如在给定时间范围内未能完成）的防护措施。

## 总结

事件物化器维护着用于满足 GraphQL 查询和订阅的有状态视图模型。你已经了解了这个模型如何随时间演变，以及视图模型的演变如何类似于维护多个专用视图模型。我解释了视图最好以事务方式维护的原因，以及如何使用命令处理器的已处理命令日志来提供视图相对于命令的一致性。你已经了解了如何思考测试事件物化器，哪些测试适合物化器本身，以及哪些测试最好在整个系统上进行——这是下一章的主题。
