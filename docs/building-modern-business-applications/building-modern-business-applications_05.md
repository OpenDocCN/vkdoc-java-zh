# 3. 什么是响应式系统？

我已经解释了什么是业务应用以及它们目前是如何构建的。在本章中，我将探讨其架构中既必要又理想的属性和特性。我首先定义什么是“响应式”，然后介绍响应式系统的概念，它源于[响应式宣言](https://www.reactivemanifesto.org/)中提出的属性。^(⁸)

本章主要讨论宏观层面的响应式系统。下一章将探讨为什么要把业务应用构建成响应式系统。

## 什么是响应式？

你可能在不知不觉中使用过响应式系统：电子表格。当你在电子表格中更改某个单元格的值时，任何引用该单元格的公式都会自动重新计算。如果这些公式单元格又被其他单元格引用，这个过程就会重复。这是一个简单而强大的想法。

为了介绍响应式系统，我们将首先讨论响应式编程。毕竟，系统是编程概念在更广泛尺度上的投影。编程有两大范式：命令式和声明式。

命令式可能是你日常使用 Java 等语言时所做的。语言中的每个表达式都是一条关于“做什么”的指令。当程序执行时，它就是在遵循这些指令。

声明式则相反。程序不是表达“做什么”，而是对我们希望发生的事情的规范。如果你接触过函数式编程概念，那么你就已经接触过声明式编程的一种形式。

响应式编程的基础是数据流思维。我在脑海中使用的视觉隐喻是管道系统。不一定是现实世界中的那种，而是那种游戏类型，你面对各种谜题，目标是将指定的水龙头和排水口匹配起来。

这个隐喻可以延续到程序的表达上。水龙头，即输入，是程序接收到的请求。排水口，即输出，不仅是对这些请求的响应，还包括日志语句、捕获的指标，甚至是对数据库的写入。

响应式编程就是以这种形式来表达问题。输入和输出被连接起来，响应式运行时确保“数据”通过我们隐喻中的“管道”流动。

当然，任何真实的系统都不是输入到输出的直接连接。在数据流中会有分流、合并，甚至反馈循环。关键在于，我们关注的是“做什么”，而不是“怎么做”。

这些都不是新概念。数据流编程最早在 20 世纪 60 年代被构想出来，而随着 1979 年 VisiCalc 电子表格程序的推出，个人电脑上就有了响应式环境。

响应式编程关注的是单个进程。响应式系统则是以响应式方式组合而成的程序集合。同步远程过程调用可以被视为“命令式”的，而通过消息总线通信的两个程序则可以被视为“响应式”的。这是一个非常粗略的概括，但希望足以说明两者的区别。

在下一节中，我将讨论《响应式宣言》，这是一份为响应式系统辩护的文档。在讨论过程中，我将进一步解释响应式系统的构成。

## 响应式宣言

2014 年，Jonas Bonér、Dave Farley、Roland Kuhn 和 Martin Thompson 发布了[响应式宣言](https://www.reactivemanifesto.org/)。其底层概念在业界已经酝酿了数年。就像前一章讨论的《敏捷宣言》为一系列编程实践贴上了标签一样，《响应式宣言》为软件系统的架构特性贴上了标签。

我在前一章谈到了 Web 应用的兴起。对于流行的业务和消费服务来说，这些应用都在非常大规模地运行。撇开规模不谈，在亚马逊网络服务（AWS）于 2006 年推出之后，它们的运行方式也发生了转变。虚拟化环境的概念已经存在多年，VMWare 在 21 世纪初通过其 ESX 产品在企业中普及了这一概念。亚马逊的产品使虚拟机对任何人都可用，并开启了现代“云计算”时代。

这份宣言是对这个新世界中系统普遍采用的架构实践的总结。它围绕四个属性展开，我将逐一讨论。



### 响应性

《牛津词典》对“响应性”的定义是“快速且积极地做出反应”。因此，它作为反应式系统的一个属性非常贴切，因为这个词本身关乎“反应”。宣言将该属性定义为 *“[系统]在可能的情况下及时做出响应。”* 这可以被视为对请求做出响应或执行某个操作的合理最大时间上限。一个具有响应性的系统会优先保证一致的响应时间。在实践中，这意味着要采取诸如使用内部超时机制等措施，并在超时时返回响应，而不是表现出挂起状态。

用户*确实*非常看重响应性。关于人机界面响应时间的研究已经确定，250 毫秒是一个合理的目标。谷歌、亚马逊和阿卡迈科技都进行过研究，表明用户会放弃加载缓慢的网站。正如我们之前所确认的，商业应用程序的使用不一定是自愿的，但总体观点依然成立：一个缓慢的应用程序会导致用户不满。

其好处不仅惠及用户，也惠及你——系统的运维者。系统的响应时间可以纳入服务等级目标（SLO）。一个简单的例子是声明在给定时间范围内回复一定比例的请求。这可以通过在系统中设置超时机制来强制执行，如果响应将超过超时时间，则返回错误消息。这可以进一步与告警关联，当系统可能承受压力时提供反馈。

这还可以建立一个良性反馈循环。通过了解系统的响应性，可以对其进行持续跟踪和改进。这可以增加用户对系统的参与度和信心，这与企业投资应用程序的业务目标是一致的。

响应时间也可以成为开发人员的骄傲。你可以安心入睡，因为你知道你的应用程序不会让用户抱怨等待电脑响应。

### 韧性

故障总会发生。具备韧性意味着在故障发生时仍能保持响应性。你可能听说过这样一个观点：*“任何足够复杂的系统都始终处于部分故障状态。”* 在这种状态下运行，并表现得好像一切正常，这就是韧性。

打个比方，想想舱壁，尤其是在船上。舱壁有助于将船体分隔成多个隔间，这样一旦船体出现破损，只有那一个隔间会进水，而不是整个船体。理想情况下，这将防止，或至少减缓船只的下沉。

同样的理念可以应用于软件。你系统的每个逻辑组件都应该能够经历故障而不影响其他组件。另一种体现是能够重启数据库，而无需重启连接到该数据库的应用程序。无法连接到数据库的故障是暂时的，连接到它的应用程序会自动恢复。

这与响应性密切相关，因为如果某个必需的依赖项发生故障，系统会传播一个错误，而不是在等待恢复时挂起请求。

虽然一定程度的韧性可以“附加”到现有应用程序上，但我认为，要实现真正的韧性，它必须是架构中一个有意的方面。

### 弹性

弹性。就像这个词本身一样，它的适用性可以在多个维度上延伸。在宣言的语境中，它指的是保持响应性的能力。系统对目标响应时间有一定程度的自我感知，并能够动态增加其正在使用的资源以维持该目标。反之，它也能够释放不再需要的资源。

简单来说，这就是水平扩展的自动化。更复杂的形式可能是数据分片的自动化。

在所有属性中，这是最难利用的一个。你的系统完全有可能永远不需要实现这个属性。然而，就像韧性一样，这必须是架构中一个有意的方面。

我从两个方面来思考这个问题。第一个是运行时弹性，即随着系统工作负载变化而保持响应性的能力。第二个是设计弹性，即随着系统演进，能够实现运行时弹性的能力。

再次借用管道的比喻，在思考数据（或数据请求）如何流经你的系统时，要留意哪里可以插入一个三通来分流或合并流量。你的系统可能永远不需要插入这些三通。但考虑它们应该放在哪里，会在你的设计中建立逻辑边界，并为实现运行时弹性提供路径。即使它可能永远不会被用到，建立这些边界也会产生一个更易于沟通和推理的设计。



### 消息驱动

这是我最喜欢的特性，也是最重要的特性。我们隐喻中的管道并不包含连续的数据流，而是将数据流原子化为离散的消息。

消息已经存在于你的系统中，系统正是通过消息来接收输入的。互联网是一个分组交换网络，数据包就是消息。任何更高层次的抽象最终都由消息构成。

将一系列消息抽象为连续的数据流有其用途，例如大文件的交换。但我认为，直到真正需要这种抽象时再引入它才更有价值——对于我们的业务应用来说，这种需求可能永远不会出现。保留对离散消息流的控制，能让我们获得可显式管理的单元。

仅有消息本身还不够，关键在于消息的交换方式——**异步**。正是**异步**与**消息传递**的结合，构成了这一特性的核心。

在现代计算机上，代码的同步执行只是一种假象。我们将摒弃这种虚构，拥抱现实：我们的代码无需阻塞等待响应，从而能更好地控制故障处理方式。我们可以将故障也表示为更多的消息。

通过异步传递消息，我们引入了一个边界，实现细节可以在边界之后被抽象化。

想象一下，向一个由 DNS 名称标识的服务器发送 HTTP 请求。在发起请求时，你并不知道（坦白说，也不关心）接收服务器位于何处。最终唯一重要的是收到响应。

同样的原则也适用于系统内部及系统间的通信。这就是**位置透明性**。一个组件并不关心其消息的接收者位于何处。它们可以位于同一进程中、同一台机器的另一个进程中，或者完全不同的地方。这为系统架构提供了灵活性。组件可以自由移动，而不会破坏调用者与其交互的方式。

如果你有使用 Spring 框架或更广泛的依赖注入的经验，那么你已经接触过这个理念。依赖注入是指系统中的某个组件并不持有与其协作的其他组件的固定引用，而是由控制平面或框架（如 Spring）将其所需的组件注入进来。

依赖注入是一个更通用理念——[控制反转](https://web.archive.org/web/20040202120126/http:/www.betaversion.org/~stefano/linotype/news/38/)——的具体化。^(⁹) 组件不再控制自己的依赖，而是将控制权交给了控制平面。其目标是实现组件的隔离，而依赖注入是实现这一目标的手段。

异步消息传递也是一种控制反转的形式。它注入的不是所需的类或接口，而是通信通道——我们的管道。

通过这种机制在组件之间进行通信，我们为系统建立了用于弹性的隔板。这些隔板也是可以注入用于弹性的三通管。同时，它们也创造了观察系统的切入点。通过观察组件之间传递的消息，我们可以了解系统的行为。

如果某个组件接收消息的速度快于其处理速度，它可以施加背压。背压是一种反馈机制，消息消费者可以通过它向生产者表明自己无法接受更多消息。这允许生产者决定如何处理这种情况——这可以是实现弹性或为了保持韧性而卸载负载的信号。

## 组合响应式系统

我在响应式系统中发现的一个美妙之处在于，它可以“层层嵌套，直至底层”。通过采用统一的组件通信方式（即异步消息传递），这种位置透明性使得系统无需重大变更即可演进。一个组件最初可以与它协作的其他组件位于同一进程中。随着时间的推移，也许该组件需要迁移到另一台机器上。这对它的协作者来说并非破坏性变更，因为它们之间的通信方式保持不变。当然，没有免费的午餐——可能会产生新的延迟问题，或需要处理新类型的故障——但基本原则依然成立。

像流行的乐高™品牌那样的互锁积木的隐喻，长期以来一直被用于模块化软件架构中组件的组合。虽然我认为这个以积木为中心的隐喻有其价值，它描述了组件如何轻松地“拼接在一起”以实现目标，但这个隐喻却忽略了乐高™积木成为经久不衰的玩具的关键：它们连接方式的统一性。

为了进一步推进这个类比，在我成长的岁月里，当我用乐高™积木搭建东西消磨时间时，我们的官方零件中混入了一些泰科超级积木。泰科的积木名义上与乐高™兼容，但它们与正品零件的契合度并不相同。

从作为支撑我创作的积木的功能角度来看，它们是可以用的。但是，由于它们在**接口统一性**上存在缺陷，它们往往成为更大设计中的薄弱环节。

这与响应式系统有相似之处。通过为组件间的通信提供统一的接口，整个系统因此变得更强大、更具韧性。

系统中每个组件之间的互连都是一个扩展点，数据流可以在这些点上被操纵——无论是为了增加弹性还是提供额外的韧性。即使这个扩展点未被使用，它依然存在。由于扩展点源于统一的接口，任何与之相关的成本都已分摊到整个代码库中。因为它并非仅为代码中的这个特定位置而创建，所以它接近零成本特性的理想状态。

无论是乌龟还是积木，统一性在于每个组件在外部观察时的行为方式，它们都遵循并受这些响应式系统特性的指导。

## 总结

这四个特性定义了一个响应式系统。它依赖异步消息传递进行通信，这有助于实现弹性和韧性，所有这一切都是为了保持对用户的响应性。消息传递的行为是一种数据流的形式，这是编程中一项历史悠久的技巧。
