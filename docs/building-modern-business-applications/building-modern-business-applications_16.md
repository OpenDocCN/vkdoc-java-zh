# 13. 所需技术

本章是我对业务应用所提出设计的巅峰。它首先简要回顾了系统的设计，然后概述了我们的约束和原则，并讨论了它们是如何被整合的。

在脑海中清晰记住所有目标之后，我将接着介绍实现该设计所需的技术类型。这仍将保持在高层次，讨论所需的能力，而非具体的实现。

在本章结束时，你将理解所提出的设计如何满足目标，以及你需要哪些类型的构建块来构建实现。让我们开始吧。



## 设计评审

系统的主要接口通过 GraphQL 提供。它将为只读查询、实时订阅以及变更其包含的数据提供接口。

从数据变更开始，GraphQL 模式包含可调用的变更操作。每个变更操作都属于一个名为命令生成器的模块。命令生成器是调用者接口所依赖的请求/响应交互与消息驱动的系统内部机制之间的桥梁。每个变更操作都由一个函数支持，该函数将变更操作的输入转换为命令。这个函数可能是不纯的，需要时需咨询外部系统。命令是一种不可变的消息类型，描述了更改系统数据的意图。它们始终面向未来，并且必须包含处理所需的所有信息。创建后，新命令将被追加到命令日志中以供进一步处理。调用者现在拥有命令的序列号，并会阻塞后续执行。它将跟踪已处理的命令日志，根据命令的序列号进行过滤，以确定命令的处理结果。收到已处理的命令后，它会被映射为变更结果。如果命令失败，则通过 GraphQL 模式中适当描述性的类型向调用者说明。如果命令成功，则返回最终生成的事件序列号，使调用者能够知道更改何时已纳入查询，或者何时已观察到订阅的更新信号。

命令日志由命令处理器跟踪并消费。该模块负责将命令转换为事件。事件是另一种不可变的消息类型，描述了系统数据已发生的某些情况；它们始终回顾过去。命令处理器需要两个纯函数来管理其操作以及与不可变状态的交互。当此模块启动时，它将从事件日志中读取，并使用其中一个函数（应用器）来推导系统的当前状态。应用器接收一个事件和当前状态，并返回一个包含该事件的新状态实例。此状态包含执行一类业务规则（即静态数据规则）所需的信息。它将是系统内所有信息的一个子集。一旦推导出当前状态，它将使用变异器函数与命令结合，以生成描述该命令所隐含更改的事件集。然后，应用器将用于这些事件，以推导系统的新状态。如果两个函数都处理无误，则事件被提交到事件日志，并且一条包含命令序列号、最后一个事件的序列号以及成功标志的已处理命令消息被提交到已处理命令日志。这是一个事务性的全有或全无过程。如果任一函数失败，这可能表示编程错误或约束违规。如果应用器失败，则丢弃来自变异器的事件。一条已处理命令消息被提交到已处理命令日志，其中包含命令序列号、最后一个成功处理的事件的序列号以及有关失败性质的详细信息。

GraphQL 查询和订阅由视图模块处理。该模块的具体细节未作规定，因为实现将取决于您领域的细节。视图模块访问系统数据的物化视图。此物化视图由事件物化器模块维护。事件物化器消费事件日志，并构建视图模块将使用的必要结构。如果使用了多个视图，则可能存在多个事件物化器实例，尽管我们一直专注于单个实例，即用于支持 GraphQL 模式的实例。当事件物化器跟踪事件日志时，它可以使用已处理命令日志对流进行窗口化处理。这允许模块在匹配命令边界时批量物化事件，从而防止技术上有效但非明确预期的系统状态变得可见。在物化一个事件后，事件物化器可以生成一个即发即弃的事件，允许视图模块在任何活跃的 GraphQL 订阅上生成新的更新信号。

该系统的设计融合了事件溯源和 CQRS 的原则。事件日志是系统的权威数据存储，这借鉴了事件溯源。通过 GraphQL，命令和查询被分离，这借鉴了 CQRS。变更操作生成命令，并按照我描述的方式流经系统。查询（以及实时订阅）由视图模块处理。

您领域内的任何其他协作系统都可以通过添加事件日志的额外消费者来启用。您可能希望在事件日志中使用的事件模式与向协作者公开的模式之间设置一个转换层。我始终建议这样做，以防止与事件日志格式紧密耦合。这在做出更改时非常有益，因为它消除了一个协调点。系统本身可以修改，而向协作者公开的接口可以按照对双方都适用的不同节奏进行修改。

在您对系统设计有了清晰了解之后，下一节将回顾我们的约束和原则，重点介绍设计如何满足每一项。

## 约束与原则回顾

系统设计有两个自我施加的约束，以及八项指导原则。这些约束代表了战略决策，我认为这些决策与行业的发展方向一致，并且与这些原则相互协同。在列举系统的一系列原则时，这相当于声明了在审视实现时应如何思考该系统。决策应回溯到一个或多个原则。它们为将来可能从事该系统工作的人员提供了指导方针。从某种角度来看，它们与具体的架构决策相隔一步。它们是制定架构决策所依据的原则。

### GraphQL

GraphQL 是一种基于模式的系统接口机制。由于其类型系统以及客户端能够控制获取哪些数据，它在 UI 开发者中变得极为流行。它是一个开放标准，并且在整个行业中正获得积极的投资和采用。GraphQL 模式可以包含所包含类型和操作的内联文档，以及弃用字段的能力。我相信这些原语是系统和团队之间松散耦合的基础。通过使用 GraphQL 模式描述自身，需要与系统交互的开发者可以获取到基本信息。有许多可用于处理 GraphQL 的工具，使得在多个平台上都能享受到处理模式的好处。是的，存在一些机制可以通过 REST 风格的 API 实现许多相同的好处。然而，这样做的工具并不那么丰富和广泛可用。

GraphQL 的离散输入和输出类型与 CQRS 保持一致。变更操作映射到驱动事件溯源架构的命令。



### 实时更新

我认为，让用户界面能够实时响应数据变化，是业务应用早该具备的能力。其具体使用方式会因应用而异。实时更新旨在实现对系统变更的主动观察，无论其用于用户界面、推送通知还是电子邮件。通过将其作为系统的一项约束，就相当于宣告这些变更是*重要的*。许多消费类应用的界面都利用了实时更新，导致用户期望他们使用的软件也具备此能力。我认为，是时候让业务应用也拥有这种能力了。

由于系统完全由事件驱动，实时更新自然得以实现。GraphQL 的订阅功能是观察这些变更的自然机制。当成功的变更操作返回一个序列号，指示何时可以观察到结果时，调用者就会被引导使用订阅作为与系统交互的主要方式，从而有助于实现*默认实时*。

### 永不遗忘

作为开发者，我们使用源代码控制来记住源代码的先前状态。我们也应该为用户在他们系统之外操作的数据提供这种能力。数据存储成本持续下降，并且借助云环境，存储容量可以按需添加。过去那些原地修改数据、遗忘先前状态的技术原因已不再成立。遗忘数据是一个策略决策，而非技术决策。业务应用应默认保留所有数据变更。这使用户能够理解数据是如何形成的。观察数据随时间的变化，也是业务洞察的来源之一。

将数据以变更粒度存储为离散事件，使得系统能够永不遗忘。关于在 GraphQL API 中暴露哪些内容的决定可以在后续时间点做出。当对 GraphQL schema 中暴露内容的需求发生变化时，事件物化器可以重新物化事件。默认情况下，数据永远不会被丢弃，即使它最初可能不可见。

### 消息驱动

通过采用消息驱动，系统与我在第三章讨论的响应式宣言的基本原则保持一致。消息交换也是面向对象编程中一个有时会被忽视的核心概念。与 GraphQL API 的某些交互将是同步的，例如执行查询和变更操作。GraphQL 订阅则直接接入系统底层消息驱动的本质。一个变更操作会生成一个命令，而命令的处理过程就是一个消息驱动的流程。如果命令被成功处理，它将生成描述已发生变更的事件（即更多消息）。这些事件将被事件物化器用于更新用于查询和订阅的视图模型。与协作系统的交互通过将事件映射为要与之交换的消息来完成。

### 读写分离

GraphQL 对查询、订阅和变更操作的分离，为读写分离提供了基础。写入通过变更操作完成，变更操作生成命令并馈送给命令处理器。这些组件不用于系统的读取路径，读取路径只需要查询事件物化器的输出。虽然读取和写入可能通过同一个 GraphQL schema 暴露，但内部实现路径是离散的，遵循 CQRS 的思想。底层基础设施的分离程度随后成为一个可推迟的实现决策。系统的架构已经为这种分离做好了准备。

### 部分可用性

在高层面上，部分可用性与读写分离密切相关。由于读取和写入在系统内走不同的路径，它们可以独立地发生故障而不会危及整个系统（尽管在某些情况下，一个仅可用于写入的系统其价值可能存疑）。通过命令和事件日志进行消息传递，允许生产者或消费者独立地发生故障。日志会缓冲消息，使系统中受影响的部分能够在故障后恢复其活动。系统被设计为具备部分可用性。

### 设计灵活性

正如我最初讨论设计灵活性时提到的，这是一个模糊的话题。并没有一个具体的度量标准来精确衡量一个设计有多灵活。这需要随着时间的推移，通过观察系统能否逐步演进而非需要整体替换来体现。业务应用可能成为更大业务流程中的关键部分，从而变得根深蒂固。如果业务需求发生变化，而系统架构无法适应，那么业务就会陷入困境：是应该绕开软件工作，还是应该采取更困难的工作，要么强行将变更塞入系统，要么完全替换系统？

我相信所提出的设计是灵活的。系统的各个组件通过交换消息实现松耦合。由于基础数据存储是以事件为变更粒度，因此可以根据需要引入新事件。事件的物化视图并非永久性决策，可以根据需要调整。甚至选择 GraphQL 也提供了灵活性；可以调整 schema，或者完全移除对 GraphQL 的使用。

### 模块化

模块化与设计灵活性直接相关；它是创建灵活设计的手段之一。本节中的章节解释了系统的核心模块：命令处理器、命令生成器和事件物化器。你的实现可能需要额外的模块，通过接入命令和事件日志即可轻松集成这些模块。通过使用消息传递作为主要的交互模式，模块之间是松耦合的，从而使设计保持灵活。

### 可测试性

在介绍构成高层数据流的每个模块时，我都讨论了该模块如何被测试。我认为这证明了系统的可测试性。第 12 章讨论了使用基于属性的测试来测试组合系统。将并发变更操作序列化到命令日志中，使得命令处理器可以顺序执行，从而具有确定性。系统的设计消除了测试变更操作竞态条件的需要，因为它们在追加到命令日志时就已经被提前解决了。如何测试系统的模块以及整个系统本身，从一开始就是一个考虑因素。



### 可变更性

这仍然是我最喜欢的原则，也是我常挂在嘴边的一句话。具备可变更性的系统能够存活并蓬勃发展数十年之久。而那些不具备此特性的系统则会被淘汰，并有望被具备该特性的系统取代。当然，这取决于负责替换的开发者是否参与了原始系统的实现，并能理解重写系统的必要性。如果由一个新的开发团队接手现有系统，我常见的一种模式是，他们会在没有反思*为何*需要重写的情况下就着手重写。跳过这种反思，他们可能会重蹈被替换系统的覆辙。

具备可变更性是前述各项原则的集大成者。仅仅陈述这一点是不够的，必须列举出原因。一个经过充分测试的系统，可以在不担心引入意外行为变化的情况下进行修改。一个模块化的系统允许增量式扩展和替换，从而限制变更的影响范围。当某些环节不可避免地出现问题时，部分可用性提供了隔离舱。这里的各项原则相互支持和巩固，共同构建出一个我认为具备可变更性的弹性系统。

现在你理解了系统的设计是如何支撑我们希望系统拥有的约束和原则的。下一节将探讨实现具体方案所依赖的技术。

## 所需技术

实现所描述的系统需要少数几项技术。当我描述系统的每个模块时，你的大脑可能已经开始思考该模块如何实现。本书的这一部分聚焦于设计，而接下来对技术的讨论也保持在这个较高层面。这里描述了技术的广泛特性，但尚未将它们与具体实现相匹配，这是本书最后一部分的主题。

### 仅追加日志

仅追加日志是一种数据结构，它表示一系列条目，其唯一的写入操作是在日志末尾追加新条目。更简洁地说，这就是一个日志，但我发现在与他人讨论时，加上*仅追加*这个限定词会很有帮助。

命令、事件和已处理命令日志是系统中此类日志的实例。虽然每个日志存储不同类型的消息，但其基本行为是相同的。

当消息被追加到日志时，它们会被分配一个严格递增的序列号；这些序列号不必是连续的整数。日志必须支持将多条消息作为事务的一部分进行追加。

事务必须具有标准的 ACID 属性，即原子性、一致性、隔离性和持久性。在我们的系统上下文中，写入多条消息必须是原子性的。一致性唯一需要的不变量是为每条消息分配严格递增的序列号。只有命令日志会接收并发写入，因此会存在必须相互隔离的并发事务。命令被追加到日志的顺序定义了它们将被处理的顺序。当一批事件被提交到日志时，事务成功完成表明它们已被持久化存储。

对于读取操作，需要支持两种情况。第一种是读取整个日志，第二种是查找特定的序列号并从该位置开始向后读取。虽然第二种情况可以通过读取整个日志并过滤掉小于指定序列号的消息来模拟，但原生支持查找功能是可取的，因为它与系统启动时物化引擎与日志交互的方式一致。

### 发布/订阅消息

Pub/Sub 是发布/订阅的缩写。发布/订阅消息系统是指可以有多个发布者和多个订阅者的系统。当一条消息被发布时，它会被传递给所有订阅者。消息没有定义的顺序，每个订阅者*可能*看到不同的顺序。消息是短暂的。如果订阅者离开，那么任何未传递的消息都会丢失。订阅者不会收到在其到达之前发布的消息。所有订阅者都必须收到在其存在期间发布的消息。

仅追加消息日志在系统中表现为无限日志。读取操作永远不会结束，而是等待新消息并在消息到达时发出它们。由于这可能不是仅追加日志的原生特性，因此它被单独列为一项技术需求，因为它可以通过将仅追加日志与发布/订阅消息相结合来构建。在向消息日志提交追加操作后，写入者会发布一条消息，指示日志有新消息。这使得消息日志在应用层面的表现能够订阅发布/订阅消息，并将这些消息作为触发器，从最后观察到的序列号开始重新查询。

物化引擎也利用了发布/订阅消息。当物化引擎更新给定领域标识符的视图时，它可以发布一条消息，指示更新已发生。GraphQL 订阅由这些更新支持；每个 GraphQL 订阅都是对更新的临时订阅。当更新发生时，可能会导致在订阅上发送一条新消息。GraphQL 订阅上的消息并不保证送达，因为视图中实际更新的数据与 GraphQL 字段选择集包含的内容之间存在脱节。虽然视图更新可以做得更细粒度，但我发现这通常会导致系统复杂性的收益递减。必须有足够多的更新，才会产生实质性的工作量来确定是否有更新需要发送。意识到这个决策是好的，我倾向于监控并根据需要处理，而不是从一开始就承担更多工作。

### 分布式锁

系统的设计并不局限于单个应用实例。当部署到云环境时，为了弹性，它*应该*以多个实例的方式运行。部署时也会同时运行多个版本的应用，因为新实例会在销毁旧实例之前投入使用。虽然任何实例都能处理读取流量，但系统的设计假设命令处理器和物化引擎是单例的，运行在单个实例上。

为了防止多个实例运行这些模块，我们需要一种分布式锁机制。命令处理器和物化引擎将由各自的锁保护，允许它们各自在不同的实例上运行，同时只允许最多一个实例运行每个模块。每个实例都会有一个后台进程持续尝试获取锁，如果成功，就会启动受保护的模块。当实例终止时，锁会被释放，允许另一个实例接管该职责。

你的实现中可能还有其他需要作为单例进程运行的领域模块，同样的分布式锁实现也应可重用于这些目的。



### 数据库

物化引擎需要将视图物化到数据库中。具体选择哪种类型的数据库，取决于你的领域数据以及你擅长的运维方式。某些领域可能适合文档型数据库，其中实体粒度较粗，为每个实体维护单一文档符合读取特性。其他领域可能希望使用关系型数据库，因为其普及性使得许多开发者对其非常熟悉。你的领域和业务规则可能要求物化引擎执行聚合操作，而这正是关系模型所擅长的。

没有放之四海而皆准的解决方案，你在此处选择的技术将取决于你的具体情况。该系统设计的一个优势在于，选择哪种数据库并非一劳永逸的决定；这体现了我们拥抱变化的原则。数据的权威存储位于仅追加的事件日志中。如果你决定迁移到新的存储，多个物化引擎可以维护多个视图。

## 总结

现在你理解了所提议的设计是如何与系统约束和操作原则保持一致并得到支持的。实现该设计所需的技术数量很少：发布/订阅消息、分布式锁、仅追加日志，以及用于物化视图的数据库。我希望你已经开始思考，并考虑自己熟悉且能舒适使用的实现方案。本书关于系统设计的这一部分到此结束。下一部分和章节将深入探讨具体实现。
