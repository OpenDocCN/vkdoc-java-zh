# 10. 命令生成器

命令生成器是连接面向外部的 GraphQL 变更的请求/响应交互与应用程序内部消息传递世界的桥梁。在本章中，我将讨论 GraphQL 变更如何与命令生成器交互，以及在设计命令和变更时如何在具体性和通用性之间取得平衡。我还将介绍如何构建命令，以使命令处理器的修改器函数成为纯函数，命令生成器如何失败以及如何补偿，最后是如何思考测试包含在这一层中的逻辑。让我们开始吧。

## 与 GraphQL 的交互

命令生成器是使我们的 GraphQL 变更能够正常运行的逻辑组件。其行为可分解为以下三个步骤：

1.  将 GraphQL 变更的输入参数转换为命令。

2.  将命令追加到命令日志中。

3.  等待命令处理完成，并将处理结果作为变更的返回值返回。



### 从输入参数到命令

在系统中，一个变更（mutation）代表一个单一事务。这源于命令处理器的行为——它一次只处理一个命令。有意省略了可能包含多个命令的“复合命令”所带来的额外复杂性。这种复杂性涉及是否允许复合命令嵌套，以及复合命令中命令数量的限制。虽然在某些情况下这种复杂性可能是合理的，但我认为创建符合系统用户需求的离散命令是一种更简单的方法。

单个 GraphQL 请求可能会调用多个变更，因为 GraphQL 允许调用者在查询和变更请求中请求多个字段。如果调用者使用此功能，每个变更将按照请求中指定的顺序依次处理。GraphQL 规范不允许异步执行变更，因为这样很难推断变更的顺序。在实践中，我并未看到此功能被广泛用于变更，因为它在语义上等同于多个请求。GraphQL 规范对于单个请求中的多个变更应如何全部成功或全部失败没有规定。如果请求只调用一个变更，则更容易推断失败场景并在调用方进行管理。

在我们的系统中，一个变更最多生成一个命令。当你将变更映射到命令时，可能会发现一个命令在多个变更中被复用。我认为这是一件好事——在系统实现中保持通用性，同时暴露更具体的接口。命令的参数可能隐含在变更所描述的行为中。

在设计 GraphQL schema 时，考虑到 GraphQL 建模的限制，你应尽可能为输入指定尽可能多的非空约束。你可能会遇到的一个限制是输入类型缺乏联合类型。如果你有一个输入对象，希望调用者最多指定该类型的一个字段，则没有语言机制可以在 schema 中指示这一点。截至撰写本文时，GraphQL 规范有一个开放的 RFC，旨在为输入对象添加 `@oneOf` 指令，以支持输入对象的有限形式的多态性。

创建命令的第一步是对 schema 中无法表达的输入值执行额外的验证。这可能包括数字范围限制、字符串长度检查以及命令被视为“有效”所需的任何其他条件。在你的实现中，可以创建自定义指令，以便在 schema 中指定这些约束。这将使你的实现能够使用这些指令来驱动验证，同时在 schema 中起到描述作用。

从概念上讲，命令“完全成型”所需的所有验证都是此步骤的一部分。此步骤中的验证仅是结构性的，类似于填写表单的所有字段，而不考虑这些值在语义上是否正确。正如调用者在调用时填写了变更参数的所有字段一样，命令生成器需要在将命令追加到命令日志之前创建一个完整的命令。命令被添加到日志后，系统将没有机会调整命令所隐含的内容。

GraphQL 社区通常建议让变更尽可能具体。这条建议的关键部分是*尽可能*。一个极端是使用一个单一的变更来处理任何场景的 schema。我希望这听起来至少令人困惑，理想情况下是毫无意义的。这样的 schema 需要大量文档来说明如何针对特定场景调用它。社区指南的意图是让 schema 本身对变更具有自文档性。通过拥有具体的变更，变更名称本身就指示了它将执行的操作。

在创建 GraphQL schema 时，我首先让所有变更尽可能具体，并在生成的命令中反映这种具体性。然后，随着调用者开始了解 schema 如何融入他们的工作流程，可能会出现某些变更范围需要扩大的情况。

例如，考虑以下两个变更（暂时忽略它们的返回值）：

```
input CreateCustomerInput {
name: String!
}
input UpdateCustomerRegionInput {
customerId: ID!
regionId: ID!
}
type Mutation {
createCustomer(input: CreateCustomerInput!): Boolean
updateCustomerRegion(input: UpdateCustomerRegionInput!): Boolean
}
```

在这个假设的系统中，存在属于区域的客户。新创建的客户在分配区域之前没有关联区域。在实践中，创建客户的用户界面流程总是在创建客户后调用 `updateCustomerRegion`。因此，向 `CreateCustomerInput` 类型添加一个可选的 `regionId` 是合理的，从而为调用者提供更简单的交互模式。我所支持的就是这种基于实际用例驱动的通用化变更。任何通用化的目标都应是解决 schema 用户的实际痛点。这应与具体性的目标相平衡，以避免调用者与实现之间的耦合。理论上，具体的变更允许调用者按照他们认为合适的方式组合他们的流程。



### 追加到命令日志

命令创建后，需要追加到命令日志中，以便命令处理引擎对其执行操作。正是在这里确定了命令的顺序。如果系统同时收到两个变更请求，它们被追加到命令日志的顺序将决定它们被处理的顺序。

追加命令的操作也是执行另一项验证（即拒绝重复请求）的机会。设想一个场景：调用方发送了一个变更请求，但随后遇到网络中断，无法收到响应。调用方如何确保其请求已被处理？如果调用方重新发送请求，该请求可能带来的副作用将是不希望重复的。我们需要一种机制来允许客户端请求是*幂等*的。严格按部就班的幂等性对于变更操作是不可行的，因为变更的意图就是产生副作用。多次调用同一个变更以触发多次副作用，这是一个有效的场景。为了处理这两种情况（有意和无意地重复变更），我们将为每个变更引入一个新参数：客户端请求标识符。

客户端请求标识符，缩写为`requestId`，是一个对于每个逻辑请求都唯一的值。如果客户端打算调用一个变更五次，他们需要提供五个不同的 ID。然而，如果客户端调用了一个变更但遇到网络中断，不确定系统是否已接受该请求，他们应该使用*相同*的 ID 重新发送请求。

检查`requestId`是否之前已被观察到的理想位置，是在尝试将命令追加到命令日志时。通过让命令日志也跟踪已观察到的 ID 集合，可以检测到重复请求。

如果检测到重复请求，系统应如何表现？我倾向于将其视为重复请求的失败。调用方知道他们是否重试了请求。如果重试的请求因为`requestId`已被使用而失败，调用方有上下文来理解发生了什么。如果这个失败发生在调用方没有重复的请求上，那就说明有其他问题需要调查。可能是系统中存在逻辑错误，或者跨客户端生成的 ID 不够唯一。

如果观察到了预期的重复请求，调用方如何得知原始请求的处理结果？`requestId`是一个唯一标识符，可用于查找先前的处理结果。我偏好的机制是提供一个查询，可以调用该查询来返回任何先前`requestId`的处理结果。另一种方案是让重复请求返回成功情况的一个变体，即“此请求已被执行”。这种情况发生的频率可能会指导你的选择。对于通常由位置相对固定的用户使用的业务应用，与在移动设备上使用的业务应用相比，因网络中断导致的重复请求会更少见。

### 返回变更结果

变更流程的最后一步是将变更结果返回给调用方。成功将命令追加到命令日志后，该命令将对命令处理器可见。如前一章所述，还有一个额外的消息日志，即已处理命令日志，它直接将命令与其结果关联起来。将临时订阅此消息日志以观察变更结果。

这里有一个*关键*的时间点细节。你的代码需要在将命令追加到命令日志*之前*订阅已处理命令日志。如果顺序颠倒，你可能会在已处理命令日志订阅激活之前，因命令处理器已完成处理而错过该事件。

已处理命令日志包含所有命令的结果。我们需要一种机制来过滤出我们追加的命令的结果。与调用方的交互是一种请求/响应交互，这与命令处理引擎使用的模式相同。然而，对于命令处理引擎，它是间接的，因为我们是在一个消息日志上追加，而在另一个消息日志上观察结果。我的解决方案是使用信封模式。

信封模式是指将你的消息包裹在信封中，就像通过邮政服务寄信一样。命令日志不包含命令，它包含的是包裹在信封中的命令。当命令处理器处理一个命令并追加到已处理命令日志时，它会*重用*命令到达时所用的信封。由于命令和已处理命令之间存在一对一的关系，这是一个直接的操作。在事件日志中重用信封会更棘手，因为一个命令可以生成多个事件。那样我们就需要制定策略来安全地复制信封，这是我希望尽可能避免的复杂性。

有了信封模式，命令生成器可以在将命令追加到命令日志之前，在信封上放置一个唯一标识符（例如 UUID）。然后，已处理命令日志的结果可以通过此标识符进行过滤，以找到与我们的命令相对应的已处理命令。这是一个与`requestId`不同的值，它在系统内是全局唯一的。它标识一个特定的命令，即使该命令由于`requestId`的重复使用而成为逻辑上的重复命令。

一个已处理命令包含：与我们的命令对应的序列号、任何生成事件的最大序列号，以及命令的结果（成功或失败）。这些信息将用于构建变更结果。

失败将通过 GraphQL 模式中的类型传达给调用方，而不是使用 GraphQL 的错误机制。这是因为 GraphQL 中的错误是*无类型*的。GraphQL 规范定义了错误在响应中如何呈现，但没有提供描述其结构的工具。我将此视为一种引导机制。变更失败最终需要传达给用户。作为一个业务应用，有些变更会失败，因为该更改会违反系统的业务规则。当这种情况发生时，向用户传达尽可能多的违规细节至关重要，以便他们理解违规的性质并可以修正输入。

我倾向于将变更的返回类型结构化为成功和失败的联合类型：

```
union MutationResult = MutationSuccess | MutationFailed
```

`MutationFailed`类型是一个容器，用于保存额外的细节。就像你的数据在 GraphQL 模式中被建模一样，你的业务规则可能被违反的方式也应该在模式中被建模。

我偏好的机制是：使用一个枚举来指示错误类别，然后使用一个字段来实现一个包含更具体细节的接口。回想一下本章前面的例子：


```graphql
input CreateCustomerInput {
  name: String!
}
input UpdateCustomerRegionInput {
  customerId: ID!
  regionId: ID!
}
type Mutation {
  createCustomer(input: CreateCustomerInput!): Boolean
  updateCustomerRegion(input: UpdateCustomerRegionInput!): Boolean
}
```

与其返回一个 `Boolean`，不如将其结构化为如下形式：

```graphql
enum MutationFailureCode {
  DUPLICATE_REQUEST_ID
  BUSINESS_RULE_VIOLATION
  UNKNOWN
}
interface MutationFailureDetails {
  message: String
}
type GenericMutationFailureDetails implements MutationFailureDetails {
  message: String
}
type InvalidCustomerName implements MutationFailureDetails {
  message: String
}
type MutationFailed {
  code: MutationFailureCode!
  details: MutationFailureDetails
}
union MutationResult = MutationSuccess | MutationFailed
input CreateCustomerInput {
  name: String!
}
type Mutation {
  createCustomer(input: CreateCustomerInput!): MutationResult
}
```

在这个例子中，关于新客户名称存在一些业务规则。为了在违反这些规则时通知调用方，我们使用了特定的 `InvalidCustomerName` 类型。我还为 `MutationFailureCode` 枚举添加了一个值 `DUPLICATE_REQUEST_ID`。此代码可用于标识重复的 `requestId` 值，这是本章前面讨论过的一个主题。

当变更操作成功处理命令时，只需要返回一条信息：最终事件的序列号。这便产生了一个简单的 `MutationSuccess` 类型：

```graphql
type MutationSuccess {
  visibleAsOfRevision: Int!
}
```

这种模式与其他关于如何设计 GraphQL 变更的建议有所不同。这是有意为之，因为我们的系统是一个事件驱动系统。命令的成功处理*并不*意味着视图已经更新；那是事件物化器的职责，也是下一章的主题。

这*确实*意味着系统的最终一致性本质暴露在了外部。我相信这其中蕴含着力量。设计原则之一是实现实时的 UI 更新。使用 GraphQL 时，这可以通过 GraphQL 的订阅机制来实现。如果调用方有一个活跃的订阅，然后提交了一个变更，则无需在直接响应变更时反映任何数据更改。当视图更新后，已建立的订阅将传递这些更改。通过在视图模型中包含事件序列号，调用方可以将这些值与 `visibleAsOfRevision` 变更结果进行比较，从而了解变更何时已被纳入视图。

## 命令与其他必需数据

变更的输入可能并不总是包含构建命令所需的全部信息。由于命令处理器的变更函数必须是纯函数，任何额外的处理都需要在创建命令并将其提交处理之前进行。

提交命令之前执行的工作应该是可安全重试的。如果网络中断，客户端可能会重新提交变更，导致命令创建过程中执行的任何工作被重新执行。

这个阶段应被视为对接收到的数据进行丰富，以满足第一类业务规则（即针对静态数据的规则）。任何业务规则的副作用都应在命令成功处理之后发生。

例如，考虑一个接受表格数据的变更。一种方法是使用输入对象数组将数据分解为行和列。但是，如果数据是由用户在电子表格中编写的呢？让用户将电子表格上传到系统并由系统提取表格数据可能更有意义。该提取将作为命令生成的一部分发生。

另一个例子是文件上传，其中文件的内容与系统无关。命令生成器可以接收文件并将其内容存储到对象存储中，例如 Amazon 的 S3。然后，命令仅包含对对象存储的引用。

## 处理失败

命令生成和提交可能因多种方式失败。我在本章中已经提到了其中两种：对 GraphQL 模式中无法表达的约束进行输入验证，以及客户端提供的幂等令牌 `requestId`。

根据输入参数创建命令的失败机会将取决于过程的性质和复杂性。除了处理无效输入外，失败很可能源于实现中的软件错误，测试可以帮助缓解这些问题。

如果命令日志无法接受写入，则将新命令追加到命令日志可能会失败。这种失败应作为系统不可用的情况进行传达，以便客户端可以重试其请求。

还有两种情况我们可能希望主动引发失败：乐观锁的实现和超时。

### 乐观锁

在对数据进行更改时，用户知道当前状态。这在多用户系统中会失效。如果两个用户同时更改同一份数据会怎样？在我们的系统中，其命令第二个被追加到命令日志的用户将“获胜”。第一个更改并未“丢失”，因为事件会持久化在事件日志中。当这种情况发生时，对于进行第一个更改的用户来说将是意外的，因为他们如果知道第二个更改，可能会采取不同的操作（反之亦然）。幸运的是，在我们的系统中有一个直接的解决方案：乐观锁。

乐观锁的前提是通过跟踪数据是否同时被修改来避免对排他锁的需求。在我们基于请求/响应的系统中，我们需要一个调用方可以为其变更提供的值，以便系统能够确定数据是否已被更改。我们可以使用事件日志轻松地为我们的系统生成这样一个值。它已经是顺序的，这是其正确性的一个基本属性。每个事件都会被分配一个递增的数字，即序列号。然后，你提供给命令处理器的函数可以利用这个值。当应用函数将事件应用于状态以产生新状态时，它也可以将事件的序列号存储在状态中。你的事件物化器（将在下一章中描述）可以使此序列号在你的 GraphQL 模式中可见。这将允许调用方知道模式中某条数据的最后修改版本（内部是序列号）。然后，此版本号可以作为另一个输入参数传递给变更，以指示用户在做出更改时所查看的版本。从变更的角度来看，更改是“基于”特定版本进行的。它可以作为“基于版本”存储在命令的信封中。最后，命令处理器中的变更函数可以利用此值来检查相关状态是否具有比提供的基于版本更大的序列值。

此检查可以根据你系统的需要，可以是粗粒度的，也可以是细粒度的。你也可以强制客户端使用它，或者允许其检查是可选的。如果此检查失败，则是一个主动引发的失败。将这些失败与其他失败区分开来对调用方很重要，因为需要用户干预来确定下一步的行动方案。用户可能会决定使用更新的基于版本重新提交其变更，或者他们可能完全放弃更改。


### 超时

在像这样的最终一致性系统中，防止“最终”变成遥遥无期至关重要。系统的用户看重确定性的响应时间，即使响应是失败。虽然我们无法预防所有可能发生的故障，但可以通过使用超时来提供有界的等待时间。这确保了响应能力，这是我在第 3 章中讨论过的一个原则。

在命令生成器中有两个地方需要实施超时：提交命令和等待命令处理器对命令的处理结果。向调用者区分这两种失败是有益的，因为针对每种情况，他们需要执行的补偿操作是不同的。

如果在等待命令处理器时发生超时，我们知道命令已成功提交。如何处理这种失败取决于你观察到它的频率以及将其呈现给用户的重要性。这里简单的解决方案是不做任何补偿。是的，发生了超时，但系统最终能够恢复并处理该变更。如果调用者最终是系统的人类用户，他们可能已经继续他们的工作，而观察到变更就足以让他们决定是否手动重试。如果该变更是由另一个系统执行的，或者它是需要用户明确观察成功的工作流的一部分，那么就需要更复杂的方法。`requestId` 值将用于添加一个新的 GraphQL 查询，该查询返回请求的 `MutationResult`。我在本章前面提到过，放置命令的信封可以重复用于最终处理过的命令。这意味着 `requestId` 在处理过的命令上也是可用的。前面描述的快乐路径是使用命令信封上每个命令唯一的值来过滤已处理的命令。由于处理过的命令信封也将包含 `requestId`，因此已处理的命令也可以通过此值进行过滤。然后客户端可以轮询这个新查询来观察命令的处理结果。这个新查询也应该有一个超时，因此调用者需要准备好必要时进行重试。另一种方法是使用 GraphQL 订阅。订阅签名将与查询相同。使用订阅的好处是，当连接处于静默状态时，订阅的传输机制可能具有保活机制，允许调用者区分是正在等待响应还是系统无响应。

对于提交命令时的超时，调用者将不知道提交是否成功。本章前面讨论过这个问题的解决方案，即 `requestId`。调用者可以再次调用变更操作，并提供相同的 `requestId` 值。这次后续尝试的结果将提供关于第一次尝试的信息。如果尝试成功，则先前的命令未被成功提交。如果尝试因 `requestId` 重复而失败，则调用者知道先前的命令已成功提交。

命令提交超时后，由于 `requestId` 已被使用而导致失败，这意味着命令*确实*已成功提交。在这种情况下，调用者可以使用我描述的处理超时的相同机制来确定命令的处理结果。

## 测试命令生成器

命令生成器可以分为两部分来考虑。首先，是用于命令提交和等待命令处理器结果的通用基础设施；其次，是将变更输入转换为命令的领域逻辑，即命令的创建。

### 命令提交

提交命令并等待处理结果的行为是一种与领域逻辑分离的机械性操作。我喜欢这样的过程，因为它们实现了应用程序基础设施与业务规则之间的关注点分离。此过程的规则严格且有限。只有一条输入数据有规则，即 `requestId`。

在为此过程编写测试时，我喜欢使用来自一个与当前业务问题无关的虚构领域的命令。这确保了对领域类型的更改不会对我的代码库的这一部分产生任何影响，而且作为额外的好处，还提供了一个找点乐子的机会。我经常通过使用与我的宠物相关的例子来个性化我的测试。对于命令提交，测试中使用的命令可能与领养新宠物有关，通常使用我多年来养过的宠物的名字和类型。

在测试命令提交时，我会为提交命令和过滤已处理命令编写一个快乐路径测试。我不会为这些测试使用真实的命令处理引擎，而是使用我可以在测试中控制的桩实例。我重视创建代表系统这些组件之间 API 契约的接口。使用 API 接口的桩实现，既强化了系统这些组件之间的关注点分离，也加强了对它们之间耦合的严格控制。

测试此过程的失败模式也很重要。你创建的桩实现应该支持故障注入，以便你可以确保系统在发生故障时能按预期运行。快乐路径测试将在第 12 章描述的测试中得到充分演练。那些测试不太可能暴露在生产环境中可能观察到的失败模式，而确保调用者在面对失败时有良好的体验，对于提供一个响应迅速且有弹性的系统至关重要。



### 命令创建

每个 GraphQL 变更（mutation）*可能*都需要创建一个测试，以确保它能生成预期的命令。老实说，我倾向于不编写这些测试，而是让这种快乐路径验证作为我将在第 12 章描述的综合系统测试的一部分。在大多数情况下，这里执行的逻辑将是从变更的输入参数到命令的直接转换。

真正值得测试的是对无效输入的处理，以及任何其生成过程涉及访问可能导致自身故障的外部系统的命令。

GraphQL 模式中定义的每个特定错误类型，都应该有一个测试来确保它在预期情况下被生成。这些测试应模拟系统调用者的体验。这意味着测试可能不会直接调用命令创建过程。相反，它们通过向 GraphQL 端点输入一个将执行命令创建的输入，并断言预期的 GraphQL 响应，来间接调用此过程。我认为这是在确保代码行为符合预期、验证调用者体验符合预期以及不过度编写测试之间取得的一个务实平衡。如果用于将输入转换为命令的底层函数已经过测试，仍然需要确保命令创建失败能正确传达给调用者。我倾向于编写测试来获得对系统的必要信心，而从 GraphQL 层测试命令创建能给我这种信心。

一个例外是依赖于外部系统的命令创建，在这种情况下我需要在测试中注入故障。如果创建命令需要与另一个系统通信，那就存在发生故障的可能性。为了保持系统的响应能力，我们需要限制外部通信的执行时间。这意味着对任何调用设置超时，并决定是否应执行重试，或者将故障传递给我们的调用者，让他们决定何时重试。这些策略的具体细节将取决于你系统的需求。除了确保系统响应能力这一通用规则之外，并没有严格且快速的规则来规定具体该做什么。

## 总结

现在你明白了命令生成器是如何成为 GraphQL 变更和命令处理器之间的桥梁的。它负责收集必要的信息来创建一个完整的命令，这是一个关键步骤，它使得命令处理器的变更函数能够保持无状态和纯粹。你也了解了如何开始确定命令的适当粒度，以及对该组件进行何种级别的特定测试是合适的。由于命令生成器仅在成功变更时返回一个序列号，下一章关于事件物化器的内容将成为命令处理器事件与用于支持 GraphQL 查询和订阅的视图之间的桥梁，以便调用者能够看到他们变更的效果。
