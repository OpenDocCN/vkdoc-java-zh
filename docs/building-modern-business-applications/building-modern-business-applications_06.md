# 4. 为什么要把业务应用构建为响应式系统？

现在，我们对什么是业务应用以及什么是响应式系统有了共同的理解。在本章中，我们将探讨将业务应用构建为响应式系统的好处。在定义业务应用时，我们曾提及业务方对应用的期望。我将通过进一步阐述这些期望以及你作为开发者的期望来开始本章。然后，我将再次逐一介绍响应式系统的四个特性，并明确地将它们与业务和开发者的期望联系起来。之后，你将知道本章标题所提出的问题的答案：为什么要把业务应用构建为响应式系统？



## 业务期望

在第一章中，作为界定业务应用定义的一部分，我描述了业务如何将应用作为关键组件来使用。在思考业务对应用的*期望*时，这不仅仅是关于如何使用它的机制。

作为业务的记录系统，业务期望在需要存储或检索数据时，应用能够保持可用。如果业务在某个维度上增长，例如员工数量或订单量，应用应能继续满足其需求。如果业务工作流需要调整，通过修改应用来实施变更不应成为限制因素。

我大致将其描述为：业务期望应用在需要时可用，并能及时响应业务需求的变化。我希望这在一定程度上是不言自明的，因为这是任何服务提供商与其服务订阅方之间关系的核心。这是一个简单的期望。复杂性在于如何实现它。

我们的业务应用是由像你这样的开发者编写的，无论是作为员工还是顾问。业务对其应用的期望，间接地，也是对你如何构建应用并使其对业务可用的期望。

## 开发者期望

你也是你应用的用户！我认为明确说明这一点很重要。你可能不会使用与其他用户相同的界面，但你仍然是用户。你的主要界面是应用的源代码。其次是构建和部署流水线，以及用于监控应用健康和行为观察的工具。

你期望代码易于处理。我使用“易于处理”而非“简单”这个词，因为你的领域内会存在固有的复杂性。理想情况下，这应是复杂性的唯一来源。除此之外的任何复杂性都是不必要的，并且有机会将其最小化。这正是本书的目标之一：提供构建业务应用的指南，以减少不必要的复杂性。如果这里的方法看起来复杂，我推测那是因为它们可能对你而言比较陌生。

减少复杂性并不意味着对专业知识的需求也会减少。当我们构建应用时，实际上是在构建机器。本书中的方法是一种不同的，并且我相信是更好的构建这些机器的方式。我正在分享我的一些经验，以帮助你在构建专业知识的旅程中加深理解。

业务对其应用的期望会间接地施加于你，因为你将这些期望变为现实。作为其中的一部分，你将在约束条件下工作。最终，这些约束可以归结为两种形式：时间和金钱。

时间可以表现为金钱。总有一个价格能让施加时间约束的人允许更多时间，但这通常是不可行的解决方案。出于实际目的，尤其是在我们的语境中，将它们视为独立的变量更为简单。深入探讨这一思路本身就是一个话题，本书并不专门讨论它。我将其描述为一种系统思维。如果你接触过或进一步阅读过相关主题，你可能会看到我们的响应式系统在概念上是如何与之对齐的。

业务从应用中获取经济价值。如果应用造成亏损，运营成本高于其回报的价值，那么应用的可持续性将受到质疑。我认为开发者意识到这一点很重要。建立这种理解所需的与同事的关系，也有助于理解业务本身，这在实现业务规则时是很有用的领域知识。

金钱成本通常是一个直接的计算。它包括用于开发和运营的硬件和软件资源，以及员工薪资。虽然重要，但我认为运营成本的重要性较低。它们相对于规划周期而言，往往以更可预测的方式变动。给定一个两年、五年甚至十年的计划，有足够可靠的数据来做出有依据的预测。摩尔定律和计算成本的总体趋势对误差范围有抑制作用。

时间才是关键。这有两种变体。企业会尝试通过增加人员来“购买时间”。其负面结果已被《人月神话》和布鲁克斯法则很好地概括：“向一个已经延迟的软件项目增加人手，只会让它更延迟。” 这种认为增加人员能让项目更快完成的想法，其核心是：有效修改软件的能力是一个限制因素。如果企业想要改变其运营方式，而软件无法及时随之改变，就会产生机会成本。

我发现，应用*易于变更*的程度，直接关系到开发者与应用的界面——即其源代码。源代码中表达的是*应用的结构*。如果一个系统不易于变更，那么进行变更将花费更多时间。通过能够更快地变更，我们实际上是在购买时间。你可能以前接触过这个想法，它常被冠以“好代码”或“整洁代码”的名义。这些观点仍然适用。根据我的经验，它们往往抽象地描述概念，而不是提供具体的指导。这完全可以理解，因为它们是指导原则，而非具体的架构。在本书中，我将为该架构提供具体细节。响应式系统提供了一种整体架构，一种可组合的架构，因此适用于大局的也适用于细节，反之亦然。

## 属性对齐

与前一章相呼应，我将逐一介绍每个属性，说明它们如何与业务和开发者对系统的期望对齐。我还会从业务的角度，提及开发者期望中提到的两个约束：时间和金钱。虽然在这些约束下工作是开发者的期望，但对时间和金钱的管理对业务而言是有价值的。

### 响应性

如果你曾与某人通电话，而对方不得不为他们的电脑速度慢而道歉，那么你就亲身体验过响应性的价值。没错，“时间就是金钱”，让用户不必要地等待会成为一种拖累。这会产生乘数效应。受缓慢响应时间影响的不仅仅是应用的直接用户，用户所帮助的任何人也会受到影响。

作为开发者，这是一个难以估值的属性。如果直接问用户系统应该有多快，他们通常会给出一个模糊的答案，比如“要快”。用户通常很容易识别出什么是不可接受的，但反过来却非常主观。性能优化工作以开放性著称，这使得围绕它进行规划变得困难。

我对响应性的关键理解在于建立一个可接受的上限。这消除了在诊断故障时“是慢还是卡住了”的模糊性。响应性的上限也为讨论和规划响应时间改进提供了一个起点。

应用的用户欣赏响应性强的应用，而开发者会发现它们更易于运维。这是最面向用户的属性，做好这一点有助于与业务建立积极的反馈循环，让业务对应用及其开发者能够及时提供系统建立信任。



### 弹性

故障总会发生，无论是有计划的还是意外的。如果你不计划优雅地处理故障，那它就不会优雅。优雅地处理故障可以减少系统的运维负担。通过最小化故障的影响，可以缩短恢复时间，快速安全地恢复服务。业务方期望应用在需要时始终可用，而弹性正是实现这一目标的手段。在用户看来，系统是否“在线”即代表是否可用。

从更广阔的视角来看，弹性关乎处理*意外的系统状态*。如果某个组件发生故障，那便是意外情况，否则该故障本应已被处理。

还有一种开发者将系统置于的预期状态，可能导致系统不可用，即**部署**。

部署可以被视为两个状态之间的转换：系统运行旧代码的状态和运行新代码的状态。这种转换的频率取决于部署发生的频率。如果你正在实践持续交付，那么这种转换可能会持续发生。部署还有另一个维度，取决于其发生方式：一段时期内运行的代码*多于*平常，或*少于*平常。

如果你希望部署时没有可见的停机时间（这在实践持续部署时很常见），那么你将经历一段运行代码多于平常的时期。这段时期的持续时间会因新版本上线所采用的策略而异。即使是最简单的情况——新版本一运行就立即停用旧版本——也会存在一个两者同时活跃的时期，必须加以考虑。还有更复杂的策略用于增量激活新代码。在所有情况下，这些策略都会导致更长的重叠期。

另一种选择是停机部署：停止旧代码，启动新代码。在部署期间，系统会有一段时间不可用。从用户的角度，或从协作服务的角度来看，这与意外故障没有区别。虽然这并未为正在部署的代码提供弹性，但在一个由多个服务组成的系统中，能够单独部署一个服务，并让其协作者处理其可用性问题，这本身就是弹性在发挥作用。

无论是从处理意外故障入手，还是从优化部署流程入手，两者都是相辅相成、互相支持的。能够在整个系统中统一处理故障，有助于管理和修复。

### 可伸缩性

可伸缩性是最难利用的特性，也是最难明确其清晰收益的特性。有效利用它是一个高级话题。我认为关键部分是使其*成为可能*，因为为一个没有考虑这些扩展点的系统增加可伸缩性可能是一项挑战。

如果系统的使用量是恒定的，那么工作负载就没有可伸缩性。如果使用量存在波动，那么这种波动必须足够显著，才值得分配或释放系统资源。我们的业务应用更可能以每分钟请求数来衡量，请求量翻倍可能算不上显著变化。在云环境中，假设为了弹性而部署了一个三实例的集群，那么对这个规模的集群进行有效的自动伸缩是很棘手的。或许将其扩容到一个新的固定规模，比花时间微调自动伸缩策略更好。如果你有一个大型集群，自动伸缩会更容易。

然而，另一方面，如果波动性很大且自动伸缩有效，那么具备可伸缩性就能节省成本。不再需要过度配置资源来处理峰值负载，也无需在业务低谷期为未使用的容量付费。如果成本已经是一个考量因素，系统可能会故意相对于峰值负载进行低配，寄希望于运气，希望峰值不会导致太多不良后果。

### 消息驱动

这是最终的答案。在某种程度上，其他特性对于业务应用来说是“锦上添花”。构建业务应用时，响应式系统的主要好处在于其消息驱动的特性。

前三个特性——响应性、弹性和可伸缩性——可以被视为运维层面的特性。而要真正做到消息驱动，系统的架构需要从核心层面以此方式运作。这个特性从根本上影响了系统架构。正如我在前一章所讨论的，你的系统已经由消息构成。我们的目标是保留它们。

让我们思考一下系统中存在的消息类型、它们的目的，以及保留这种抽象的价值。

想象一个现状服务：一个通过 ORM 在关系数据库中存储数据的通俗 REST 端点。一个用于更改系统中某些数据的 HTTP 请求就是一个消息。应用消费这个请求，使用适当的逻辑更新对象模型并持久化更改。然后返回一个响应消息。

我们从一条消息（请求）开始，以一条消息（响应）结束。我们*转换*了请求，得到了响应。

进一步分解这个例子，HTTP 请求是一条代表*更改请求*的消息。从概念上讲，这是一个**命令**。通常会对传入请求进行一些验证，以确保其结构正确，例如检查必填字段。如果这些验证通过，我们可以认为该命令已被应用*接受*。

应用消费这个命令并更新对象模型。这很可能仍是预期性的，因为可能还需要满足数据库约束。直到事务提交时，应用才能确定更新是否成功。这就是命令的*处理*过程。事务的处理结果（无论是成功提交还是失败）就是命令的*结果*。

如果结果是失败，可以立即返回一个错误响应，这是对结果的一种转换。在成功的情况下，作为命令一部分被修改的对象模型可能被用来返回一个响应负载，这是另一种类型的转换。无论是成功还是失败的情况，都代表了另一条消息。

如果没有对命令进行显式表示，上述逻辑很容易变得混杂，使测试更加困难。这可能违背开发者对系统易于变更的期望。应用的内在结构需要持续警惕以防止这种情况发生。

如果我们把命令具体化为一条消息，我们就可以显式地对其进行操作。这个命令会有一个清晰的处理流水线。每一步都可以由一段离散的、可测试的代码来表示。归根结底，这就是本书的主题，而第 3 部分将深入探讨如何实现这一点。

通过使用显式的命令，系统可以成为消息驱动的。利用这种对系统正在做什么的表示，它为系统的逻辑组件如何交互提供了一个框架。这种一致（并且是的，有主见的）设计理念指导着应用的结构。

在组件之间交换消息，使我们能够应用与将组件引用连接到系统数据流时相同的控制反转原则。我们获得了同样的好处：提高复用性、降低耦合度以及独立演进。

在一个作为事实来源并管理工作流的业务应用中，能够安全且自信地改变数据流的方式至关重要。这就是将它们构建为响应式系统的原因。



## 总结

本章解释了*为何*将业务应用构建为响应式系统是有益的。业务对应用可靠性的期望，正是由响应式系统所提供的。其中最重要的是你期望能够有效地变更系统。通过将应用的数据流建模为消息，可以充满信心地实现这一点。第 3 部分将解释*如何*实现。在此之前，接下来的两章将深入探讨业务应用的两个重要特性：管理业务规则和时间。
