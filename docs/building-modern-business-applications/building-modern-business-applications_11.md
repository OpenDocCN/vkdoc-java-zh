# 8. 高级数据流

鉴于我们自我施加的使用 GraphQL 的约束，在本章中，我们将考虑一个利用 GraphQL 来支持 API 契约与用户界面及其他协作系统的系统的高级数据流。我将首先解释事件溯源（ES）和命令查询职责分离（CQRS），然后介绍如何以 GraphQL 的模式语言进行思考，最后将这三个概念整合在一起。



## 事件溯源与 CQRS

正如在讨论时间时提到的，我们将构建一个融合**事件溯源**（ES）和**命令查询职责分离**（CQRS）的系统。

事件溯源最初由马丁·福勒于 2005 年[提出](https://www.martinfowler.com/eaaDev/EventSourcing.html) ^(¹³)。虽然命名归功于他，但这一理念无疑早于他的相关著述。其基本主张十分简单：将应用程序状态的每一次变更都捕获为事件，并将事件集合作为应用程序的**事实来源**。事件根据发生顺序自然形成序列。这个有序的事件集合被称为事件日志。事件日志是应用程序的事实来源，描述了已发生的变更序列。虽然这是一种自然的记录变更方式，但并非查询系统当前状态的天然机制。为了获得"当前状态"的概念，需要重放事件日志来推导出状态值。

事件日志中的第一个事件是特殊的，因为它独立存在于时间中，不依赖任何其他事件。后续每个事件都建立在前一个事件的基础上。要推导出"状态"的概念，必须遍历事件日志，通过反复整合每个事件所代表的变更来构建状态。如果你熟悉函数式编程概念，这相当于一个折叠操作。在 Java 的 Stream 库中，这对应`reduce`操作。这样做的一个好处是，可以通过在所需时间点终止操作来重建任意时间点的状态。

虽然事件溯源对于捕获状态变更非常有用，但基于事件的表示形式并不适合查询。于是 CQRS 应运而生。

CQRS 由[乌迪·达汉](https://udidahan.com/2009/12/09/clarified-cqrs/) ^(¹⁴)和[格雷格·杨](http://codebetter.com/gregyoung/2010/02/16/cqrs-task-based-uis-event-sourcing-agh/) ^(¹⁵)在 2000 年代末期推广。这是一种非常直接的模式：为命令（写入）和查询（读取）使用独立的模型。CQRS 模式可以应用于任何场景，但与事件溯源结合时尤为流行。

两者结合时，系统中负责事件溯源的部分处理写入操作。当事件发生时，新事件被整合（物化）到读取模型中。事件是自然的通信方式，因为这是事件溯源的输出。这进一步符合我们构建响应式系统的目标，即采用消息驱动的交互方式。

事件溯源中缺失的一个细节是*什么导致了事件的创建*。可以说，这通常被当作"留给读者的练习"。别担心，本书会给出答案。在解释解决方案之前，我将更详细地说明使用 GraphQL 的决策。

## 以 GraphQL 模式思考

前一章简要介绍了 GraphQL，作为解释为何将其设为自我约束的一部分。为了理解它将如何与 ES 和 CQRS 交互，有必要进行更详细的说明。

GraphQL 模式代表了提供方与消费方之间的契约。GraphQL 社区建议采用"模式优先"的方法。模式应尽可能用该语言最真实地表示模型，而不是将现有模型投影到 GraphQL 中。因此，理解 GraphQL 的所有建模能力是必要的先决条件。

顾名思义，GraphQL 是一种图结构。我将从底层向上描述其结构和可用的建模原语，从图的叶子节点开始。GraphQL 模式中的所有顶级名称共享一个命名空间。单独来看这不是问题。但如果你的模式消费者会将其与其他模式组合，就需要意识到这一点，因为可能存在名称冲突。

GraphQL 要求调用者明确指定请求内容。没有"通配符"机制来返回数据。构建请求时，调用者必须指定希望返回的数据。

### 标量和枚举

在最底层，即图的叶子节点，有两种可能：标量值和枚举。

有一小部分内置的标量类型：字符串、整数、浮点数、布尔值和对象标识符（ID）。也可以定义自定义标量类型，因为内置的基本类型对大多数模型来说是不够的。自定义标量通常用于容易表示为单一值的值类型，例如各种日期时间组合、URL 和固定精度数字。我也喜欢为模型中特定的值类型使用自定义标量，而不是依赖 String 这样的通用类型。

没有内置的枚举类型。GraphQL 的枚举在建模必须来自固定集合的值时非常有用。与某些编程语言中枚举的实现不同，GraphQL 枚举没有对应的数值。枚举值不是任何东西的代理，它们本身就是值。

### 对象

模型中最简单的节点类型是对象。对象有名称，并由一个或多个字段组成。对象名称来自共享命名空间，但字段名称是对象定义本地的。字段返回标量、枚举或对象。后续小节将讨论其他几种可能性。

一个粗略的心智模型是将对象视为键值对的集合。这很准确，但还不够。对象的字段可以声明可选和必填参数，使其具有函数的行为。更贴切的心智模型是将对象视为命名函数的集合。

对象字段还可以指定可空性约束。默认允许任何值为可空。虽然你可能想用此来镜像模型中的现有约束，但我建议谨慎使用。

GraphQL 的执行模型将可空性作为错误处理的一部分；如果在执行过程中发生错误，响应将从错误点开始向查询根方向转换为 null，并在第一个可空字段处停止。大量使用非空会增加错误的"影响范围"。使用非空也会限制模式的未来演进，因为将字段从非空转换为可空被视为模式的破坏性变更。

### 列表

列表是对对象字段的增强。对象字段的任何返回值都可以用列表结构包装，表示可能返回多个值。字段的基数要么是零要么是一，使用列表则将其扩展为零到无穷。列表中的值是同质的。

可空性约束与列表有两种交互方式。列表中的条目可以声明为非空，列表本身也可以声明为非空。执行模型的错误处理在这里尤其重要。我建议将列表值声明为可空，这样在执行过程中发生错误时，其他列表值仍能被返回。

### 输入对象

输入对象是为对象字段参数提供结构的机制。参数必须是"原始"值（标量或枚举），或者可以组织成输入对象。

输入对象是键值对的集合。输入对象有字段，字段值可以是标量、枚举或其他输入对象类型。值可以声明为列表，也可以声明为非空。

在此上下文中，可空性更有用，因为它向模式使用者传达了哪些是必填参数。我建议适当地将项目标记为非空，因为从可空变为非空对调用者来说是破坏性变更。



### 接口与联合类型

接口和联合类型是介绍对象类型时我提及的两种组织方式。它们都可用于定义对象字段的值类型。

接口是一组命名字段的集合，这些字段在多个对象类型间共享。它们在语义上等同于许多编程语言中接口的行为。GraphQL 的执行模型允许调用者直接使用接口字段，同时也能根据具体类型实现条件化行为。在运行时，接口必须由具体类型实现。只有模式（schema）会将具体类型的确定延迟到执行阶段。接口只能由对象类型和其他接口实现。

联合类型与接口相反。联合类型是一组互不相交的对象类型集合。联合类型不能包含接口。它们在语义上等同于编程语言中的"选择"或"二者择一"类型。由于模式将其视为互不相交的集合，在运行时调用者必须指明如何处理每种可能性。

接口和联合类型**不**能用于输入对象。

### 查询与订阅

查询和订阅是定义数据读取方式的两种机制。它们分别由模式中的特殊类型表示：`Query` 类型和 `Subscription` 类型。通常直接使用这些名称，但也可以为它们自定义名称并映射到这两个概念。

它们的定义方式与对象类型相同，对象类型的规则同样适用。这些类型上的字段构成了模型的入口点。常见做法是在这里使用参数，因为这是访问特定类型实例的直观机制。

查询和订阅的执行模型不同。对于查询，调用者可以指定多个返回字段。GraphQL 规范声明实现可以"并行解析"（即执行）这些字段。对于订阅，则有不同的语义。请求订阅字段时，必须且只能选择一个字段。订阅字段的实现具有隐藏语义：它返回的是该字段值类型的序列。这是从 GraphQL 系统提供实时更新的机制。由于序列由字段的值类型定义，在此处使用联合类型非常强大，因为它允许在订阅的每个"节拍"中返回不同的内容。

### 变更

变更是写入数据的机制。我认为 GraphQL 的写入实现相当巧妙，因为它利用了每个字段都是概念性函数这一特性。与查询和订阅类似，变更由模式中的特殊类型表示：`Mutation` 类型。

由于变更类型是对象类型，它也遵循相同的规则。该类型的每个字段都是一个可用的变更操作，字段的参数是调用者提供值的机制。

按照惯例，此类型的字段应改变系统状态。您可能注意到我在前面的小节中并未禁止这样做，确实如此。就像实现修改状态的 HTTP `GET` 调用是不良实践一样，其他对象类型（包括查询和订阅类型）的字段也是如此。

执行模型在这方面确实提供了便利。与查询类似，调用者可以请求变更类型的多个字段，作为在单个请求中调用多个变更的方式。如果采用与查询相同的并行执行模型，将限制此功能的实用性，因为变更将没有确定的执行顺序。为解决此问题，变更类型的执行模型是*串行*解析每个字段。这简化了对变更行为的心智模型重建。

使用 GraphQL 读取数据时，调用者"掌控"一切。模式规定了可用数据，调用者必须指定他们想要接收的内容。对于变更，接收方"掌控"一切。模式现在规定了调用者必须提供的内容。对于读取操作，返回的具体内容存在一定的不确定性。每当出现接口或联合类型时，就代表一组可能性。由于这些构造不能用于变更输入，模式中声明的是*确切*结构，而非潜在结构。

我经常遇到 GraphQL 新手提出的一个问题是："如何将输出类型复用为变更的输入？"简单（但令他们不满意的）答案是：*不能*。GraphQL 有意将输入类型和输出类型分开。复制 REST 模式中 `GET`、`PUT` 和 `POST` 操作使用相同负载形状的做法并不容易。我认为这是 GraphQL 作者在引导实现者采用特定的交互模式。

GraphQL 社区建议变更应尽可能*具体*。这与 REST API 形成对比，后者中 `GET`/`PUT`/`POST` 三件套通常使用相同的负载形状协同工作。通过使用具体的变更，变更的范围和副作用得以清晰传达。当变更涉及工作流时，这可以揭示意图。

我在 REST API 中观察到的一种模式是基于字段变化触发副作用。考虑一个负责发票支付系统的 API。当收到新发票时，它们被录入系统。存在一个工作流：用户必须审批发票，审批后自动安排付款。REST API 实现有一个 `GET` 端点用于检索发票，一个 `POST` 端点用于修改发票。发票上有多个可变字段，本例中我们关注 `status` 字段。要审批发票，`status` 字段从"已接收"修改为"已审批"。收到此负载后，端点观察到变化，并触发付款工作流。该工作流是 `POST` 的副作用。从表面上看，API 定义并未表明会发生这种情况。是的，这可以记录在文档中，但在调用代码库中不会立即可见。可以设置单独的端点来明确这一点。然而在实践中，我观察到这并非总是如此。

在 GraphQL 中实现相同的交互模式并非最直接的方式，因为需要有一个描述 `Invoice` 的对象类型，然后为变更创建一个并行的 `InvoiceInput` 输入类型。这可以比 REST API 有所改进，因为输入类型只会列出可变字段。除了*具体*之外，变更还应该是*语义化*的。遵循这一建议，本例将有一个名为 `approveAndPayInvoice` 的变更，明确描述其功能。它有意模糊了与 `status` 字段的关系。审批本身就是一个动作。审批行为更新了状态字段，而不是更新状态字段触发动作。这可能颠倒了 API 的概念化方式。我认为这非常强大。

我发现输入类型和输出类型的分离是 API 设计中的宝贵工具。它们是状态的两个方面：改变状态的请求和观察状态。这与第 6 章介绍的命令与事件之间的状态关系类似。输入类型与命令、输出类型与事件之间的对称性，正是我们将 GraphQL 与事件溯源和 CQRS 结合的方式。



## 结合 GraphQL + ES + CQRS

正如我在前面关于事件溯源的章节中提到的，ES 模式并未规定事件的触发原因。我在第 6 章讨论时间时引入了*命令*的概念，这里将使用这一概念。我将解释各个部分如何组合在一起，然后更详细地描述它们。

GraphQL 变更（mutation）应该是*具体*且*语义化*的。变更可以清晰地映射到命令。命令可以应用于系统的当前状态，从而产生事件，推动状态随时间向前演进。这些事件可以物化为与 GraphQL 查询模式相对应的视图。由于我们确切知道哪些内容发生了变化，这些事件还可以用于驱动 GraphQL 订阅。

事件溯源驱动着变更。CQRS 则是变更及其模型（由 GraphQL 模式中的输入对象描述）与查询和订阅（使用对象类型）之间的分离。GraphQL 中输入和输出类型的分支很好地支持了这种分离，并鼓励我们恰当地对每一侧进行建模。

从变更开始，存在一个“命令生成器”，负责从变更中派生出命令。变更与命令之间可能并非一一对应。每个变更会生成一个命令，但多个变更可能生成相同的命令，或者一个变更可能根据其特定参数生成不同的命令。在 GraphQL 模式中，变更本质上是一个函数。函数的调用被封装在一条消息（即命令）中。第 10 章将详细探讨这个组件。

命令是对变更的请求。存在一个“命令处理器”接收这些请求并输出事件。命令处理器负责一类业务规则，即“静态数据”规则。下一章将重点介绍这个组件。

事件由“事件物化器”接收，它负责创建用于满足 GraphQL 查询和订阅的模型。第 11 章将探讨这个组件。

这就形成了一个简单的循环。客户端使用查询来观察系统状态。然后，他们发出变更以进行修改。命令生成器将变更转换为命令。这些消息随后被命令处理器接收并生成事件。事件被事件物化器接收，并用于更新用于满足客户端查询的视图。系统的外部是一个 GraphQL 接口。内部则由消息传递组成，无论是命令还是事件。

这种数据流使我们能够将系统内部实现为响应式系统，因为它从根本上由消息传递构成。

## 本章小结

在本章中，我们了解了 GraphQL 的模式语言如何优雅地适用于暴露一个基于事件溯源和 CQRS 概念构建的系统。我们学习了如何从 GraphQL 模式的角度进行思考，事件溯源和 CQRS 如何互补，以及如何将这三个概念结合起来。我们对数据如何流经这样一个系统有了初步的理解。
