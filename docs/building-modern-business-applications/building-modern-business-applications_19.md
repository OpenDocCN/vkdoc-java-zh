# 15. 扩展点与展望

在本书的开头，我分享了我对如何定义业务应用的观点。基于该定义，我接着描述了如何思考构建它们，以满足一组我认为对业务和你（开发者）都有价值的属性和特性。我所介绍的设计在解释时并未规定任何具体的实现。在本书第三部分充分解释该设计之后，前一章介绍了成功用于实现该设计的一组工具。在最后一章中，我分享了一些关于本书系统基础哲学的临别思考、扩展核心组件的想法，以及你可以将你自己的实现带向的方向。

## 设计的灵活性与易变更性

如果我回顾我参与过的各种软件系统，将它们分为我讨厌处理的和不讨厌处理的两类，其分界线在于系统进行变更的难易程度。这与我所被要求的变更范围无关。如果系统及其环境不易于变更，那么一个看似简单的业务逻辑修改可能会变得极其复杂。

这是系统内部一个深层次的属性。我相信，如果它从一开始就没有被纳入，那么事后将其纳入将极其困难。这就是为什么本书系统的每个关键组件都包含了如何测试的方法。这也是为什么尽可能首选不可变性。我发现偏好不可变性的代码更容易推理，使得修改更加直接。不可变性也与响应式宣言的基石属性——消息驱动——相关。消息一旦发送就不能修改；描述变更需要消息。

软件系统会有一定程度的固有复杂性。业务工作流通常混乱且充满边缘情况。作为开发者，我们有责任为业务最小化软件系统增加的复杂性。它们应该尽可能复杂到必要程度，但不能再多。

业务对其软件系统的需求*会*随时间变化。我们创建的软件系统应具有适应这些变化的灵活性，并且实施这些变化的过程应该直接明了。这是一项持续的活动，也是一项需要平衡的活动。一个**过于**灵活的系统（通常通过高层次的抽象体现）可能难以自信地进行变更。

本书描述的系统旨在实现这种平衡。通过思考第 5 章中的业务规则分类，系统为每种规则提供了特定的整合位置。这就像一个地图，为开发者提供指导，说明不同类型的变更应该在哪里进行。CQRS 模式的使用，特别是与 GraphQL 的模式语言结合时，允许系统数据的视图独立于其变更方式而演化。这看起来可能像是额外的复杂性，尤其是在开始一个新项目时。然而，如果一开始没有纳入，事后改造系统以添加这个扩展点可能会更加困难。

当我最初迭代最终成为本书所述设计的内容时，我希望最小化固定决策。理想情况下，系统中的每个组件都是可替换的，并且同样理想的是，对其协作组件的干扰最小。我牢记的平衡是思考它*可能*如何变化，但仅此而已。目标是既不排除未来的可能性，同时也不在今天做超出必要的事情。我无法预测系统的需求将如何演变。

所呈现的架构存在局限性。它体现了一组权衡。本章的各节是利用设计灵活性的练习。也许其中一些对你的工作足够重要，以至于你希望从一开始就将其纳入。没有一种方案能放之四海而皆准。



## 多命令处理器

本书所描述的系统中的一个关键简化之处在于，它只使用了一个命令处理器。但这不应被理解为整个业务系统只有一个命令处理器。在第一章 1 介绍本书对业务应用的定义时，这些应用既是领域特定的（相对于业务运营的领域而言），也是针对组织内特定部门的。部门本身也是领域，例如会计、销售或客户支持。

尽管事件溯源的基本范式已经使用了数千年，但它在软件系统中的广泛应用仍处于起步阶段。在研究事件溯源时，你可能会看到建议限制事件日志中事件数量的推荐。虽然这是一个基于他人经验的良好建议，但它也增加了复杂性。使用单一命令处理器和事件日志是一种有意为之的设计简化。命令处理器为执行业务规则而维护的状态，类似于开发者可能熟悉的在关系型数据库中维护的状态。执行规则维护的代码可以完全访问系统的所有数据。这最大限度地减少了从原地修改数据到命令处理器方法之间的概念差异。

在最初构思时，我就将集成多个命令处理器视为一种未来的可能性。我认为，根据你的目标，可以有多种思考方式。

可以存在一个单一的命令日志和一个单一的命令处理器。然而，事件可以被标记上一个可用作分区键的值。这将允许事件日志的消费者限制他们接收的事件。如果你的系统瓶颈与事件消费有关，这可能是一种可行的方法。

单一命令处理器是系统所能支持的写入吞吐量的瓶颈。如果变更请求是突发性的，那么根据处理积压任务所需的时间，单一命令处理器可能仍然足够。如果允许队列堆积会违反系统的响应性目标，或者变更速率会阻止单一命令处理器赶上进度，那么可以创建多个命令处理器。

如果使用了多个命令处理器，它们很可能会被串联起来。考虑一个管理客户的系统。除了维护新客户的数据外，还可以向系统添加新客户。该系统的业务规则对客户唯一性有一个约束。每个客户都有一个唯一的税号，添加一个税号已被其他客户使用的客户应该失败。所有其他业务规则都限定在单个客户范围内。为了使用本书的组件来强制执行这个唯一性约束，新客户的添加和税号的修改需要由一个命令处理器（即全局规则维护器）来维护。然后，每个客户可以拥有自己的命令处理器，用于维护客户特定的信息。

系统将需要使用来自全局规则维护器事件日志的事件来确定已知客户的集合，并管理客户特定的命令处理引擎。传入的变更请求将被分发到相应的命令日志。我提到的串联与新客户的创建如何实现有关。全局规则维护器负责已知客户集合及其税号。公开的创建客户变更请求允许指定税号以及其他客户元数据。来自全局规则维护器的事件日志只包含与客户集合相关的事件；所有客户详细信息都应存在于每个客户各自的事件日志中。为了确保这一点，命令生成器需要向全局规则维护器提交一个命令并等待响应，然后再向客户特定的命令日志提交另一个命令。

这增加了系统内部的复杂性。但这种复杂性也可以隐藏在系统公开的接口中。这种更改可以在系统生命周期的后期进行，无需系统调用方进行更改，也无需对第 12 章 [#511062_1_En_12_Chapter.xhtml] 中组合系统的测试进行任何修改。

转向多个命令处理器的另一个原因可能是为了解决启动速度问题。命令处理器在内存中维护执行业务规则所需的状态。这意味着每当系统启动时，所有事件都需要从存储中重放。多个命令处理器允许并行化这项工作，但还有另一种方法：在应用程序启动之间缓存此状态。



## 命令处理器状态缓存

命令处理器维护着执行业务规则（即静态数据规则）所需的状态。尽管该状态是事件日志所包含内容的一个子集（如第 9 章所述），但命令处理器在启动时仍会读取整个事件日志，并使用你提供的应用函数来重建此状态。

这样做是有益的，因为它允许你更改应用函数的行为——例如，当你需要开始从现有事件中追踪额外数据以执行新的业务规则时。

随着系统投入生产，事件日志持续增长，重建此状态所需的时间也会增加。最终，这一时长可能会增长到对你的用例而言不可接受的程度。这是一个值得在你的实现中追踪的优秀指标，即观察这一时长随时间的变化情况。你可以结合时长增长速率与事件日志的预期增长率，来预测未来这一过程可能需要多长时间。这将为你留出时间，以便适当地优先安排任何变更。

前一节描述了如何引入多个命令处理器，这可能是针对你所在领域的一种解决方案。但并非所有领域都适合以限制单个事件日志随时间增长的方式来划分问题空间。在某些情况下，务实的（或者我有时喜欢称之为“最不坏”的）解决方案可能是缓存命令处理器的状态。

我特意使用了*缓存*一词，而非*快照*。快照可能暗示信息是持久存储的，并可能被视为另一种真相来源。但这里**并非**如此。跨应用运行存储命令处理器状态仅是一种性能优化。缓存的状态可以随时丢弃，系统仍会按预期运行，只是重启命令处理器所需的时间会增加。

在缓存命令处理器状态时，应在将事件持久写入事件日志之后进行。如果存储状态失败，不应导致系统崩溃。当命令处理器启动时，它不会从事件日志的开头开始，使用应用函数将每个事件归约成新的状态实例；而是会首先加载缓存的状态，并利用其已合并的最后一个事件来确定从事件日志的何处开始消费。按时间间隔持久化缓存状态是可行的，因为启动时消费少量事件通常不会造成问题。

具备对缓存状态进行版本控制的能力非常重要。如果应用函数在你的代码版本之间发生了变化，则不应使用现有的缓存状态，以便使用更新的应用函数实现来重建状态。你需要权衡这是应该采用显式的、由开发者驱动的版本控制，还是隐式的、利用代码元数据来生成稳定版本号的方式。

缓存也可以与多个命令处理器协同使用，以改善单个处理器的启动时间。

需要注意的一个风险是故障模式：当缓存状态不可用，且命令处理器无法在足够时间内启动时的情况。应追踪命令处理器的启动时间。如果预计启动时间将达到不可接受的程度，则应着手改进系统，使得丢失缓存状态不会引发问题。在许多场景中，我预计解决方案是增加多个命令处理器，将事件日志分割成多个更小的事件日志。使用多个事件日志还能消除并行物化事件时的一些复杂性。

## 并行事件物化

事件物化器从事件日志中消费数据，以维护一个有状态的视图模型，该模型用于满足 GraphQL 请求或为其他目的维护额外的有状态模型。它可以将多个消息日志（其中一个是来自命令处理引擎的事件日志）进行多路复用，并逐个事件地物化合集中的事件。

对于系统的调用者来说，要观察到其变更操作带来的变化，他们需要等待事件物化器完成来自其命令的事件的物化。如果这一时长过长——无论是直接由于执行的工作量过大，还是间接由于其他调用者发起了自己的变更操作——都需要采取措施来缩短这一时长。

一种可能采用的方法是允许事件物化器并行物化多个事件。同样，由于系统设计的灵活性，有多种途径可以实现这一点。

如果你希望将变更完全局限在该组件内部，你可以利用对事件物化函数所执行工作的内部了解，来构建临时分区。这些分区的构建方式如下：事件物化器首先将传入的事件放入一个分区，然后由多个组件内部的进程并行地从这些分区中消费数据。这种方法的一个优点是分区完全是临时的。分区策略可以根据需要随时更改，而不会对持久化数据产生任何影响。这种方法的一个缺点是检查点操作变得更加复杂。消息日志中的当前位置不能再简单地存储为单个值（即最后物化的序列号）。相反，检查点需要维护已物化的序列号集合。在理想情况下，它应收敛到每个消息日志只维护一个值。实现时需要处理通过消息日志追踪非线性进度的问题。

或者，事件物化器的并行化可以利用多个命令处理器。前一节描述的所有方法都共享相同的实现。多个命令处理器也并非严格必需；从事件物化器的角度来看，多个事件日志与将其他数据源表示为用于物化的消息日志之间，并没有实质性的区别。主要的（或许也是唯一的）区别在于事件物化器本身的创建。其他数据源很可能在开发时就已经确定，实现可以静态地配置事件物化器，指定其需要消费的消息日志集合。如果使用了多个命令处理器，那么来自你服务的事件将被分散到多个事件日志中。如果命令处理器的数量是动态的，事件物化器还需要能够支持在运行时动态发现新的事件日志进行物化。

检查点操作本身并不会变得更复杂，因为在原始设计中就已经包含了跨多个消息日志追踪进度的功能。不过，这一变化可能会影响调用者。回想一下，一次成功的变更操作会返回一个单一值 `visibleAsOfRevision`，该值表示处理变更命令时最终事件的序列号。



当使用多个命令处理器时，你需要决定事件序列号空间是所有处理器共享（从而在所有事件间建立全局顺序），还是每个命令处理器拥有独立的序列号。两种方法各有优劣。如果所有命令处理器共享序列号空间，则命令处理器中的分区存在性可以被隐藏在系统的外部契约之后。`visibleAsOfRevision` 的语义保持不变。然而，共享序列号空间在生成序列号以及等待事件物化器时会产生协调成本。`visibleAsOfRevision` 的单一标量值无法指示事件来自哪个事件日志。当使用检查点来实现写后读一致性时，需要等待事件物化器完成*所有*事件日志中该版本的物化。这不太可能成为良好的长期解决方案，但如果是在现有生产系统中增加多个命令处理器和并行事件物化，这便是一个非常可行的选择。现有语义和性能特征都将得以保留。以修改变更响应为代价，可以将单一标量值 `visibleAsOfRevision` 与另一个指示版本来源事件日志的值配对。命令生成器通过读取已处理的命令日志来确定变更的处理结果，因此这里自然成为存储该信息的位置。调用方随后可以在查询时提供这对值，以读取自己的写入内容。如果序列号是按命令处理器分配的，则必须首先修改外部契约，要求调用方提供另一个值来指示其事件所在的事件日志。

在任何场景组合中，都存在一条迁移路径，用于在活跃系统中实现事件的并行物化。

本节与前两节重点描述了可被视为性能改进的系统变更。接下来的三节将介绍将 GraphQL 与系统结合使用的其他方式。

## 命令日志与事件日志作为 GraphQL 订阅

前一章介绍了一个描述消息日志的 Java 接口，该接口是对系统命令日志和事件日志（来自第 9 章）的泛化。

```
interface MessageLog {
Flux messages();
Flux messagesSinceSequence(long seq);
}
```

这个 Java 接口也可以用 GraphQL 表示。

```
type Message {
sequence: Int!
}
type Subscription {
messages(sinceSequence: Int): Message
}
```

GraphQL 没有泛型，因此我在示例中创建了 `Message` 类型来代替 `T`。由于源自浏览器环境，GraphQL 没有 `Long` 标量类型，其数值类型仅限于 32 位。可以定义自定义标量类型（我推荐这样做）来绕过此限制。

`MessageLog` 接口的实现可以由使用 GraphQL 订阅从其他主机消费消息的代码来支撑。无需修改接口即可实现这一点，这充分体现了该接口及其使用的响应式流抽象的简洁性和强大功能。

系统内部的所有消息日志都可以通过这种方式暴露。我在工作中发现这样做有若干好处。最初推动以这种方式扩展系统是为了观察生产系统。能够使用标准的 GraphQL 工具（例如基于浏览器的 GraphiQL IDE）非常有价值。它提供了一种安全的方式来观察传入的命令和生成的事件。这确保了对生产系统数据库凭证的严格控制。开发者不应直接修改系统的数据存储。系统对数据库中的数据有特定假设：对于命令日志和事件日志，假设它们一旦写入便不可变；对于事件物化器维护的有状态视图，假设它是唯一进行状态变更的组件。将消息日志作为 GraphQL 订阅提供，既可用于调试，又能确保其只读性。

这些 GraphQL 订阅也为回归测试提供了机会。考虑对命令处理器的变更器和应用器函数进行了大量修改的情况。虽然第 12 章介绍的基于属性的测试提供了很大信心，但有时了解生产环境中的行为是否符合预期也很有益。

在测试环境中（无论是部署环境还是开发者的本地环境），可以创建一个命令处理器，它从一个由 GraphQL 订阅支持的命令日志中消费数据。这是可行的，因为命令包含了命令处理器操作所需的所有信息。可以将测试环境的事件日志与生产环境的事件日志进行比较，以确保它们等价。

这种方法也可以用于事件物化器。虽然对输出没有同样直接的交叉检查，但它可以确保事件物化器能够处理系统已知的所有事件。

作为进一步扩展，可以创建一个生产系统的持久化测试替身。它将是只读的，不接受变更。相反，它跟随生产环境的命令日志，处理相同的命令，并使用自己的物化引擎。结合前两种技术，它可以为用户提供一个匹配生产数据但运行更新代码的系统。

将命令日志和事件日志暴露给*你*使用，并不意味着它们不再是系统的实现细节。它们仍然是实现细节，不应被视为与你协作者的契约。下一节将讨论如何将这些想法用于系统间的进程间通信（IPC）。

## 使用 GraphQL 进行进程间通信

基于上一节，我们可以使用相同的基于 GraphQL 订阅的方法来显式暴露数据。我在第 11 章介绍事件物化器时曾提及这个想法。

事件物化器可以从多个源消费数据。它在代码库中的实现契约是消费那些以 `MessageLog` 形式呈现的源。

本书中的设计针对的是管理领域数据的业务应用。通常，业务中不同领域之间的界限可能很模糊。可能存在一个管理客户的系统，而客户与区域相关联。也可能存在另一个管理供应商的系统，供应商也与区域相关联。客户和供应商使用的区域需要保持一致，因为区域可能是数据分析的连接键。让第三个系统负责管理区域，并由客户和供应商管理系统共同使用，可能是合理的。如果客户系统需要将区域数据纳入事件物化器维护的视图中，它可以使用 GraphQL 订阅来访问区域管理系统的事件流。对于此用例，事件流应该是底层事件日志的映射形式。区域管理系统在向其他系统提供服务时，应使用独立的契约。这降低了系统间的耦合度。区域系统内事件结构的变更无需与 GraphQL 订阅的用户同步变更。它们可以独立演进。

在这个例子中，客户、供应商和区域之间存在关系。如果每个系统都暴露自己的 GraphQL 端点，这些关系对系统用户来说并不直接可见。它们分散在三个模式中。下一节将介绍一种统一这些关系的技术。



## 联邦式 GraphQL

承接上一节的示例，假设某组织拥有多个独立的应用程序，分别用于管理客户、供应商以及它们所在的区域。利用更大领域内的自然边界，将其细分为更小、通常更易于管理的单元，这是领域驱动设计的核心原则，正如埃里克·埃文斯在其开创性著作《领域驱动设计：软件核心复杂性应对之道》^(¹⁸)中所描述的那样。埃文斯将这些单元称为*限界上下文*。本书所采用的业务应用部门视图，与埃文斯对限界上下文的定义高度一致。

我们的示例包含三个系统，它们对应三个具有相互关联概念的 GraphQL 模式。客户和供应商都与它们运营所在的区域存在关联。假设有一个用于管理客户的用户界面，该界面提供了将客户分配到区域的功能。这个用户界面需要区域信息来显示客户的当前状态，并在允许用户进行更改时提供支持。

一种简单的实现方式是让用户界面分别调用每个系统。虽然在这个仅有两个系统的示例中这似乎合理，但请设想一个关系更复杂的领域。用户界面所需的工作量将会增加。这促使人们将这部分工作从运行在用户机器上的用户界面代码中集中出来，转移到专为用户界面服务的后端服务中，这种模式被称为*面向用户界面的后端*（BFF）。BFF 是为单个用户界面量身定制的。在我们的示例中，最多可能有三个 BFF，分别对应每个领域的用户界面。每个 BFF 执行的工作将极其相似，因此又有了消除这种重复的机会。

这种重复现象被 Apollo^(¹⁹) 公司（一家为 GraphQL 提供开发者工具的公司）所认识。如果我们的每个领域服务（客户、供应商和区域）都已经暴露了 GraphQL 端点，那么是否存在一种方法，可以描述这些模式如何相互关联，然后提供一个能够接收 GraphQL 请求并恰当地将其委派给其他服务的服务呢？这就是 Apollo Federation 的精髓所在。

在 Apollo Federation 模型下，示例中的三个服务都将被视为子图。取代 BFF 的将是一个图路由器，它将这些子图组合成一个单一的超级图。超级图服务（也称为路由器）提供了一个 GraphQL 端点，该端点暴露了所有子图的组合。此服务会将传入的请求路由到相应的子图服务。

我们的示例客户系统可能会在其 GraphQL 模式中暴露以下内容：

```
type Customer {
id: ID
name: String
regionId: ID
}
```

它只提供了客户所在区域的标识符，因为这是该服务管理的全部信息。区域系统则暴露了一个包含区域名称的 `Region` 类型：

```
type Region {
id: ID
name: String
}
```

Apollo Federation 允许客户系统在其模式中暴露以下内容：

```
type Customer {
id: ID
name: String
region: Region
}
```

客户系统无需在模式中仅暴露区域 ID，而是可以暴露 `Region` 类型，即使它并非客户服务的一部分。客户系统暴露了以下查询来访问客户：

```
type Query {
customer(id: ID): Customer
}
```

使用 Apollo Federation，调用者将能够发送跨越子图边界的查询。以下查询请求了区域的名称，这些数据由区域系统管理：

```
query {
customer(id: "42") {
id
name
region {
id
name
}
}
}
```

路由器会首先将请求路由到客户系统，加载客户及其区域 ID。然后，它会向区域系统发出另一个请求来加载该 ID。对调用者的响应掩盖了查询了多个子图的事实，使得两个系统看起来像一个整体。这与 BFF 提供的体验类似。然而，超级图服务比特定于用户界面的 BFF 更加通用。

通过使用 GraphQL 作为系统的暴露 API，Apollo Federation 促进了统一图的创建，该图作为所有系统所有用户的单一入口点。连接每个领域服务的模式这一行为，为业务提供了关于其模式之间连接程度（或缺乏连接）的知识。这些知识是对组织内数据建模决策的宝贵反馈。

业务应用是其数据的权威来源，而业务本身则利用这些数据来辅助决策。通过确保数据得到恰当建模和适当连接来提高数据质量，可以使业务做出更明智的决策。

在探讨了如何在系统中利用 GraphQL 之后，接下来的几节将探索实现该设计的其他可能性。



## 外包分布式系统问题

为了实现本书所描述的架构，我在第 13 章列举了几项必需的技术：仅追加日志、分布式锁、发布/订阅系统以及数据库。第 14 章接着描述了如何通过单一解决方案使用 PostgreSQL 来满足这些需求。这正是我在工作中采用的方法，并且已被证明是成功的。

然而，这并非唯一途径。分布式系统中存在一些横切需求，可以将其抽象出来并实现可复用。我曾用 Temporal^(²⁰) 对本书的设计进行过原型验证，这是一个开源平台，能够可靠地执行工作流。

在 Temporal 的模型中，工作流类似于一个用命令式代码表达的状态机。Temporal 提供了一个软件开发工具包（SDK），帮助你确保工作流以确定性方式编写。工作流维护其自身的私有状态。任何与工作流外部状态交互的操作，例如查询数据库或发送电子邮件，都被称为活动。当工作流调用一个活动时，Temporal 通过处理超时和重试来管理工作流的安全执行。工作流可以被视为*故障无感知*的，因为它无需关心活动失败的问题。这使得业务工作流能够被简洁地表达。

所有工作流都有一个唯一标识符，你可以选择自行指定。对于给定的标识符，只能存在一个正在运行的工作流实例。这种行为使得 Temporal 工作流可以替代本书系统中分布式锁的用途。

工作流之间有两种通信方式。工作流可以使用 Temporal 称为信号的概念向另一个工作流发送消息。当工作流接收到信号时，它可以修改其私有状态作为响应。信号的发送方不会收到任何响应。如果工作流需要从另一个工作流获取信息，它可以使用称为查询的概念来询问另一个工作流的状态。查询是只读的，不能修改工作流的状态。

Temporal 提供的基本构建块——工作流、活动、信号和查询——足以实现本书系统的三个关键组件。命令生成器、命令处理器和事件物化器都可以表达为 Temporal 工作流。

Temporal 工作流可以永远运行。这使得它们非常适合同时运行命令处理器和事件物化器。Temporal 活动可用于实现从消息日志中读取、追加到事件日志以及更新有状态视图。对于命令处理器，由于应用器和转换器都是纯函数，因此在 Temporal 工作流内部使用它们是安全的。Temporal 本身可以用作缓存命令处理器状态的一种手段。如果需要清除缓存，可以将工作流重置到任意时间点。

对于命令生成器，每个变更的处理都可以是一个工作流。客户端请求标识符可以用作工作流标识符的一部分。GraphQL 层将为该变更启动一个工作流，并从请求中传入必要的信息。然后，它将等待工作流完成，以将处理结果返回给调用者。使用 Temporal 将增加额外的容错能力，这超出了我在第 10 章中描述的范围。如果命令的创建需要与其他系统交互，这些交互可以实现为活动。

Temporal 的内部实现与本书中的系统非常相似。工作流的状态存储为一系列事件。如果执行工作流的计算机中断，当 Temporal 重启工作流时，它会通过重放存储的事件来重建工作流的私有状态。Temporal 使用事件溯源来管理工作流的状态。本书描述的系统则使用事件溯源来管理业务数据。

Temporal 不是一个数据库。用它来存储用于满足 GraphQL 查询的有状态视图是不合适的。下一节将介绍一个具有独特能力的数据存储。

## 双时态物化视图

本书中的系统使用事件日志作为应用程序状态的权威来源。事件日志中的每个事件都描述了一次变更。这些变更通过事件物化器用于维护系统数据的有状态视图。有状态视图的确切模式取决于你的系统所负责的数据。由于事件包含了随时间变化的所有变更，因此针对这一维度进行查询将非常有用。

从两个维度来思考时间。首先，是数据库发生变更的时间。这可以是事件被追加到事件日志的时间戳，也可以是事件被事件物化器物化的时间戳。这个时间戳将被视为*事务时间*，即系统处理该事件的时间。第二个维度是*有效时间*。这个时间是一个*业务领域*概念。我在第 2 章描述的状态是许多业务应用在原地修改数据。当在原地修改数据时，无法在保留修改事实的同时追溯性地对数据进行更正。事件溯源以及使用两个时间戳（即双时态）使得捕获这一点成为可能。通常，变更的有效时间等于事务时间；信息在记录时是真实的。有时，记录今天的信息，但该信息在过去某个时间点或未来某个时间点才是真实的，这也会很有用。

一个能够按有效时间回答查询的数据库，或者一个能够提供存储实体变更历史的数据库，将是本书所描述系统的一个强大补充。XTDB 就是这样一个数据库。我被 XTDB 吸引，是因为他们的愿景^(²¹)与我的不谋而合：使用不可变性，并且磁盘上的持久化存储成本低廉。

XTDB 以状态为中心，而本书的系统以变更为中心。事件物化器是这两个世界之间的桥梁，它将事件日志中的变更折叠成一个有状态视图。它所维护的有状态视图可以存储在 XTDB 中。通过使用 XTDB 作为数据存储，GraphQL 模式可以轻松地暴露系统内数据的变更历史。它还可以允许查看数据存储在任意事务时间和有效时间组合下的状态。如果用户对系统在过去任何时间点的数据状态有疑问，都可以得到解答。如果你的系统为任何主要关注观察变更的基于审查的工作流提供支持，这一点将尤其强大。



## 结语

我很高兴能与您分享本书所蕴含的设计理念。希望我的设计原理、原则与目标能引起您的共鸣。尽管商业应用已存在数十年，但软件开发的未来依然充满朝气。正如本章开篇所述，我认为持久耐用的软件系统应围绕“易于变更”这一特性来设计。实现这一目标需要刻意为之——这未必是开发过程中始终优先考虑的事项。解决当下问题的重要性，有时会掩盖未来解决问题能力的价值。我认为商业系统应当具备更长远的视野。当企业委托您这样的开发者（无论是作为雇员还是承包商）定制软件时，这本身就是一项投资。通过以可持续的方式构建系统来最大化投资回报，才是明智之举。

我认为“可持续”的本质正在发生变化。持久化存储曾是一种稀缺且昂贵的资源。这种状况影响了应用的构建方式——开发者需要在系统运营成本与增加存储带来的价值之间寻求平衡。数十年前开发者围绕存储所做的权衡取舍如今已不再适用，但许多人仍沿用旧范式编写软件。开发者正在全新的环境中运用全新的能力工作，我们需要与之匹配的新思路。

我的目标之一，是让本书所述的设计能够经受住下一阶段计算演进的考验。我深知自己无法预知未来，但我确信今日所做的工作必须能够适应未来的变化。正因如此，本书某些部分可能显得较为抽象——尽量减少实现细节的阐述，正是我对持久性的尝试。从某种意义上说，这本身就是本书的一种设计灵活性。希望这种刻意的抽象性能让您有机会思考这些理念如何在您的环境中落地。如果本书能激发您以不同视角思考并开辟自己的道路，我将视之为成功。

愿您的工作能为他人创造价值，并长久服务于世，终成不朽遗产。
