# 9. 命令处理器

命令处理器是我们系统中的一个关键组件。它将负责确保一部分业务规则（即静态数据规则）得到执行。在本章中，我将解释这个组件背后的理念、构建它所需的条件、它的工作原理、失败方式以及如何测试它。让我们开始吧。

## 它是什么，以及不是什么

命令处理器是系统中事件溯源的部分。它负责协调构成系统状态的事件的创建。作为其中的一部分，它还强制执行一类业务规则，即静态数据规则。事件描述的是过去，即已经发生的事情。为了有效地创建事件，必须确保它们满足当时生效的业务规则。

在本书描述的系统中，有一个单一的命令处理器，它串行处理事件。这是一种权衡，其他事件溯源的实现可能不会选择这样做。我认为单一的命令处理器可以简化系统，尤其是在早期阶段。这是一个可以推迟的决策。如果单个命令处理器无法跟上传入命令的速度，那么存在一条通往并行化的路径。然而，在走这条路时必须小心，因为系统中的某些约束将更难执行。对于许多业务应用程序所运行的规模，我认为更简单的解决方案更可取。我倾向于在必要时才引入复杂性，但同时要提前考虑如何处理它，以免在设计上排除这种可能性。



## 需求

命令和事件各自是一种消息类型。在前一章中，我解释了按顺序排列的事件集合被称为事件日志。由此可以推论，对于任何类型的消息，其泛化形式可称为消息日志。

命令处理器接收命令作为输入，并为每条命令输出零个或多个事件。为简化该组件的操作，命令将从消息日志（即命令日志）中接收。关于写入命令日志的更多细节将在下一章中作为命令生成器的一部分进行说明。就命令处理器和本章而言，命令日志是等待处理的有序命令集合。

将命令存储到命令日志中的这一做法，毫不意外地被称为命令溯源。在本书的系统中，我将这两种实践结合起来。命令被溯源以生成事件，而事件被溯源以生成系统的官方状态。从命令到事件的转换，类似于从命令日志到事件日志的映射函数。下一节关于构建核心循环的内容将进一步阐述这一概念。

处理完命令后，系统并不需要特意保留该命令。虽然事件日志必须永久持久化，但已处理的命令可以被丢弃。然而，保留它们也有好处，本章后面的测试部分将对此进行探讨。

命令处理器强制执行了我在第 5 章中描述的业务规则分类之一，即针对静态数据的规则。这是命令处理器的一项关键职责。对于该组件的拟议运行方式而言，事件的生成必须*无副作用*，这一点至关重要。副作用是业务规则的另一个独立类别，必须发生在事件生成之后，而不是作为生成过程的一部分。副作用有两种形式。通常想到的常见形式是通知的情况：发生了某些事情，我们需要通知另一个系统。这种形式类似于返回 void 的函数；它是“即发即忘”。有时被忽视的副作用类别是那些查询其他状态源并从中返回某些内容的副作用。必须认识到，这些交互*同样*是副作用。命令代表改变系统状态的请求。为了准确表示，命令必须完整包含满足请求所需的*所有*信息。如果数据验证涉及命令处理器外部的状态，则该状态必须在命令生成阶段（而非命令处理阶段）被查询。

这为命令处理器列出了一份简短的需求清单，我将在下一节中将其作为构建其处理循环的一部分：

*   命令从命令日志中溯源。
*   处理命令是从命令日志到事件日志的映射函数。
*   映射函数必须无副作用。

## 构建核心循环

将命令处理器视为一个循环是很有用的。给定一个命令和系统的当前状态，调用一个函数，该函数将这两个数据点作为输入，并返回一个新事件列表作为输出。当新命令到达时，重复此过程。命令日志为系统提供新命令，但当前状态从何而来？在像这样的事件溯源系统中，不存在单一的“当前状态”。状态的变更分散在事件日志中。为了处理命令，必须推导出“当前状态”的视图。

由于命令处理器仅负责强制执行针对静态数据的业务规则，其对当前状态的视图只是事件中包含的所有信息的一个子集。其当前状态只需包含执行规则所必需的信息。

消息日志是无限的流。对于事件日志，每个事件都意味着“当前状态”的一个新版本。为了实现这一点，我将使用流处理中的一个概念：`scan` 运算符。这是一种折叠函数。折叠函数是一种高阶函数，用于递归分析数据结构。你可能通过它在许多标准库中常见的名称 `reduce` 熟悉这个概念。折叠函数需要两个输入：一个初始值，以及一个接受两个参数（数据结构中的一个元素和初始值）并生成下一个值的函数。该函数会为数据结构中的每个元素重复调用，一次调用的返回值成为下一次调用的初始值。当数据结构中的所有元素都处理完毕后，最终的返回值被用作折叠函数的返回值。

计算整数列表的总和是折叠函数的一个例子。给定一个数字列表：

```
[1,3,5,7,9]
```

要计算总和，我们将按如下方式调用 `reduce` 函数：

```
reduce([1,3,5,7,9], 0, (state, item) => state + item)
```

`reduce` 的返回值将是 `25`。第三个参数是我们的“归约”函数。第一个参数是归约的状态，在处理列表的第一个元素时，它将是初始值 `0`。第二个参数是列表中的当前元素。展开 `reduce` 执行的内部循环，类似于：

```
reducer = (state, item) => state + item)
result = 0
result = reducer(result, 1) # result 为 1
result = reducer(result, 3) # result 为 4
result = reducer(result, 5) # result 为 9
result = reducer(result, 7) # result 为 16
result = reducer(result, 9) # result 为 25
```

看到展开的形式，你可能会意识到 `scan` 所执行的技巧。由于 `scan` 是在流上操作，它可以基于 `result` 的增量值生成一个输出流。输出流以初始值开始，然后对于输入流中的每个元素，输出包含将归约函数应用于输出流的前一个元素加上输入流的下一个元素的结果。

当应用于事件日志时，`scan` 可用于在每个事件之后推导出一个状态值。初始值是添加任何事件之前的系统状态。然后，归约器会使用先前的状态和下一个事件被调用。通过这种方式，事件日志被映射到一个系统状态流。



核心循环现在可以用流处理的方式来表达。我们有两个流：命令日志和事件日志。事件日志可以通过 `scan` 操作映射成一个代表系统状态的流。这两个流，即状态流和命令流，然后可以被“拉链”在一起，形成一个事件流，也就是我们的事件日志！“拉链”是另一种流处理操作，其名称来源于你在衣物上找到的装置——拉链。物理拉链将它的两排齿“拉”在一起，从每一边各取一个齿，然后将它们连接起来。

流处理中的 `zip` 操作是两个流的成对组合，它调用一个函数，该函数接收每个流中的一个元素，然后创建一个由这些返回值组成的新流。

核心循环需要两个流处理原语，`zip` 和 `scan`，以及两个特定于我们领域的函数。从事件构建状态的函数被称为*应用器*，因为它将事件应用于当前状态。使用状态加命令来生成事件的函数被称为*变更器*，因为它通过命令来变更状态。

鉴于我们要求这些函数没有副作用，它们是纯函数。纯函数是指给定相同的输入，总是返回相同输出的函数。纯函数非常易于测试，这是本章后面一节的重点。

通过将核心循环表示为对消息流的操作，它符合我们的原则以及响应式系统的属性，即消息驱动。尽管核心逻辑很简单，但在处理故障场景时，还会增加一些额外的复杂性。

## 处理故障

命令处理器有两个方面。第一个方面由两个特定于领域的函数组成：变更器和应用器。第二个方面是其他所有内容，即通过我们隐喻性管道连接流的底层基础设施。

每个方面都是一个独立关注点。第一个方面是应用关注点，第二个方面是基础设施。通过分离这些关注点，每个方面都可以有适当的故障处理策略。

### 基础设施

基础设施方面的故障处理是一个简单的策略。如果故障是可重试的，则重试该操作。如果故障不可重试，或者重试次数有限，则“向上游报告失败”，让编排命令处理器执行的任何组件来处理该故障。

可能发生的故障类型很可能集中在 I/O 上——当读取作为处理器输入的命令和事件时，或者当向事件日志写入新事件时。如果用于生成事件的函数允许每个命令生成多个事件，则必须确保写入事件日志是事务性的。要么所有生成的事件都被写入，要么一个都不写入。如果事件日志不是事务性的，并且写入了多个事件，那么中途的故障会使系统处于不一致状态；该命令实际上被部分处理了，这是无效的。

执行两个特定于领域函数之一时发生的故障*不是*基础设施故障。由于这些函数是纯函数，它们可以根据基础设施故障导致的重试需要进行重试。否则，它们需要单独处理，因为其故障的性质在类别上是不同的。

### 应用

两个特定于领域函数之一的故障无法通过简单的重试来解决。由于这些函数是无状态的，故障表明输入存在问题或函数本身存在逻辑错误。这些函数产生的故障有两大类：与处理新命令相关的故障，或从现有事件构建状态时的故障。

较简单的情况是从现有事件构建状态时的故障。从类别上讲，这不应该发生。由于现有事件是系统的真相，它们需要被如此看待。是的，由于先前的逻辑错误，它们可能包含某种“不良”信息，但这应被视为异常情况。派生状态的函数需要适当地处理这些事件。第 12 章将讨论有助于最小化这种情况发生几率的测试技术。

应用故障的其余情况则集中在命令的处理上。这里的故障是不同的，故障是*预期的*且*正常的*。每当一个命令会使系统进入违反静态数据规则的状态时，它就需要被拒绝。请记住，命令最终是改变系统状态的请求。该请求并非预先注定会成功。（正是在这里，术语“命令”的用法开始变得牵强。虽然命令是权威性的指令，但系统说“不”也是可以的）。

命令将源自想要改变系统状态的人，很可能是用户。由于命令是请求，发起者想知道他们请求的处理结果。为了实现这一点，有必要将处理结果额外记录到另一个消息日志中，即已处理命令日志。已处理命令日志包含每个命令的条目，指示它是成功还是失败。提交命令后，可以监视此日志以确定命令是否成功。

已处理命令日志在核心循环中也扮演着角色。此日志中的条目包含三部分数据：

1.  指向命令日志中已处理命令的指针

2.  命令的处理结果，指示它是成功还是失败

3.  指向事件日志的指针，指示处理后最近的事件

这些数据点是相关的，但用于不同的目的。指向命令日志的指针用于记录哪些命令已被处理。命令的处理结果允许命令的发起者观察结果。事件日志指针允许命令生成多个事件。

我还没有明确提到命令生成多个事件。这是一个有用的实践，因为它允许事件比命令更细粒度，并且允许多个命令生成相同的事件。这样做的好处是针对事件日志的消费者。通过细粒度的事件，支持每个事件的逻辑可以更独立，事件之间的重叠最小化。命令生成多个事件给核心循环带来了一个难题。当我最初介绍核心循环时，我描述了事件日志将如何用于在每个事件之后生成一个状态，并且这些状态将与命令进行“拉链”处理。如果命令和事件之间存在一一对应关系，那么这种拉链处理过程是直接的；处理一个命令后，将生成一个新状态，该状态已准备好用于下一个命令。当命令生成多个事件时，将生成多个状态。循环如何知道哪个是正确的状态用于下一个命令？这就是事件日志指针的用途。派生的状态流与已处理命令日志连接，以对状态流进行采样，使其*仅*包含在命令生成的所有事件都被应用之后出现的状态。这允许在命令失败时重用状态；对于失败命令的已处理命令，将包含与先前已处理命令*相同的事件日志指针*，因为*没有新事件*。



### 执行业务规则

对静态数据执行业务规则存在失败的可能。当构建命令处理器状态时，事件会被反复应用于该状态。每次应用事件时，都需要确保下一个事件不会产生无效状态。在提供给命令处理器的两个函数中，应用器是确保所有生成状态均有效的合适位置。对于事件日志中的事件，此函数应始终成功执行。如果执行失败，则表明系统存在错误。可能是事件被篡改，也可能是业务规则已更改，导致旧事件不再满足新规则。

如果业务规则未更改，但根据现有事件生成状态失败，则需要操作员介入。如果业务规则*确实*发生了更改，则需要恰当记录该变更。我之前在第 6 章讨论过这一点。业务规则的变更发生在某个时间点。有必要将该变更*作为事件*记录下来。这样，根据事件生成状态的函数就能利用这个业务规则变更事件来调整其行为。

在处理新命令时，必须确保其生成的事件满足业务规则。为避免重复执行业务规则，在调用修改器函数后，命令处理器还会使用应用器来确保：当预期事件应用于当前状态时，能够成功创建新状态而不会失败。

在实践中，我发现大多数命令失败最终都发生在确保它们能够生成有效状态的过程中。如果命令是请求进行某项更改，那么由该命令生成的事件就是实现该更改所需的计划步骤。只有当这些事件成功应用于状态后，该计划才能被视为成功。

作为处理命令的一部分，将事件应用于状态是*前瞻性*的。新生成的状态会被丢弃，因为用于下一条命令的状态将来自事件日志。之所以可行，是因为该函数没有副作用。

与处理失败密切相关的是最小化意外场景。最好的方法是通过全面测试来实现。

## 如何进行测试

我发现命令处理器中的关注点分离非常有利于测试。处理器接收两个函数，以便处理特定的命令和事件类型。这使得测试可以拆分为两部分：命令处理器本身的测试，以及领域特定函数的测试。

### 测试命令处理器

命令处理器本身的机制应使用一个简单的领域进行测试，例如数字加法。命令可以表示为整数，即当前状态应等于的数字。然后，修改器可以生成一个事件，该事件是另一个整数，即必须加到当前状态上才能使其等于命令值的整数。应用器则是一个加法运算。可以添加一个简单的业务规则，例如状态值必须始终为奇数。

通过使用简单的领域，就可以对命令处理器进行压力测试，观察它在各种失败场景下的行为。由于其操作方式可以用流处理来表达，如果实现使用了具备必要原语的库，那么测试主要就是确保库被正确使用，并且其他协作组件按预期组合。

### 测试你的代码

需要测试两个函数：应用器和修改器。每个函数都可以独立测试，无需命令处理器。这是要求它们为纯函数所带来的好处。它们执行所需的所有信息都作为参数提供。

对于应用器，每个事件至少应有一个测试，以确保对于有效事件，状态能按预期更新。在实践中，每个事件可能需要进行少量测试，以覆盖该变更所影响的业务规则。值得注意的是，除了业务规则约束检查外，应用器的实现应基本不包含条件逻辑。应用器是将事件应用于状态。事件是回顾性的，描述已经发生的事情。条件判断应仅用于确保已发生的事情是有效的。如果你发现自己编写了与检查业务规则约束无关的条件逻辑，这暗示该逻辑应属于修改器函数。

修改器是包含分支决策逻辑的函数。修改器的输入是命令和当前状态，它返回一组事件来描述如何随时间推移推进状态。命令使用状态中包含的信息来指导其行为并更改返回的事件集，这是完全合理的。与应用器类似，每个命令至少应有一个测试。实际测试数量将根据给定命令在修改器内的分支逻辑而变化。穷举覆盖给定命令的所有可能场景是可行的。测试还可以确保：当向应用器提供事件以及与提供给修改器相同的状态时，命令返回的事件能够通过应用器的验证。可以不使用命令处理器本身来完成此操作，并且推荐这样做，因为命令处理器最终只是编排修改器和应用器的执行。

在命令处理器上下文中测试这些函数将在第 12 章中介绍。

还有一种额外的测试类型可以执行，即针对生产数据的回归测试。严格来说，命令日志中的命令在处理后无需保留。然而，保留命令允许在测试中使用它们。可以使用更新版本的修改器重新处理命令，以确保生成相同的事件。这是一种强大的技术，用于确保对系统逻辑进行大规模重构后，系统仍具有完全相同的行为。

## 总结

命令处理器位于我们系统的核心，负责执行一些最重要的业务规则，即针对静态数据的规则。我已经解释了其运行原理的本质以及运行所需的条件。你已学习了如何考虑测试该组件，以及如何结合它来测试你的命令和事件。下一章将以同样的方式介绍命令生成器，它负责创建命令处理器所操作的那些命令。
