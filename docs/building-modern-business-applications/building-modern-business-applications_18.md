# 14. 使用现代 Spring、Java 和 PostgreSQL 进行构建

在我们迄今为止的旅程中，我解释了本书对业务应用的定义、业务应用应具备的特性和功能，并介绍了一种满足这些特性和功能的设计。虽然本书的前面部分与具体实现无关，但最后这部分将专注于使用特定技术来实现该设计。

在本章中，我将介绍我用来实现该设计的一套工具。希望你能认可我的选择。

## 通过 PostgreSQL 实现所需技术

前一章提到了实现该设计所需的四项关键技术：

*   仅追加日志
*   发布/订阅消息
*   分布式锁
*   数据库

所有这些技术都可以通过单一解决方案 PostgreSQL 来实现。它是一个健壮的数据库系统，已经积极开发了 30 多年，在类似 BSD 或 MIT 许可的自由许可下作为开源软件提供。我早在 2000 年就开始使用它，当时它是唯一支持 ACID 事务和外键的开源关系型数据库系统。它对可靠性和数据完整性的高度重视吸引了我，它已成为我在需要数据库的项目中的首选系统。

在设计系统时，我喜欢最小化运维开销。如果每项技术都由独立的实现来解决，那就需要管理五个技术栈（四项技术加上应用本身）。在当前团队实践 DevOps 并同时负责开发和运维的趋势下，这种运维职责的最小化最终能让团队更加敏捷。

如果 PostgreSQL 成为其中某项技术的限制因素，始终可以选择替换其实现。这又回到了我们的两个原则：设计灵活性和模块化。

从另一个角度来看，其中一项技术*就是*数据库。如果该数据库还具备其他开箱即用的能力，为什么不同时利用它们呢？我曾经有队友主张使用独立的工具，他们认为一个具有多种功能的工具类似于“叉勺”，一种兼具勺子和叉子功能的混合餐具。叉勺常常被人看不起，因为大多数接触到的都是一次性叉勺，作为快餐或预包装餐食的一部分。在这些情况下，使用叉勺通常是为了节省成本和减少浪费，用单一餐具替代两种。选择叉勺是被迫的。我的反驳是，有时人们*可以主动选择*使用叉勺。如果你要去露营，想尽量减少携带的餐具数量，那么将功能整合到单一餐具中就变成了一个理想的特性，而且这是你自己做出的决定。

在接下来的章节中，我将详细阐述 PostgreSQL 如何能够满足所有这些需求。



### 建议锁

与许多数据库一样，PostgreSQL 在内部使用锁来保证其一致性。在正常情况下，这对用户通常是透明的。在调试问题时，例如由于等待锁而导致语句未按预期完成时，你才会意识到它们的存在。

PostgreSQL 提供了创建具有应用程序定义含义的锁的能力。这些锁被称为*建议锁*，因为从数据库服务器的角度来看，它们没有语义含义，并且服务器不会强制要求使用它们。

建议锁有两种类型：绑定到事务的锁和绑定到连接的锁（PostgreSQL 文档将其称为*会话*）。锁也可以是共享锁或排他锁；我们希望使用排他锁，因为目标是在应用程序集群中获得运行进程的独占权。

如果你认为锁可以通过表中的标志来模拟，那么你是对的。使用建议锁有两个优点。第一个优点是锁活动被保留在 PostgreSQL 用于存储表数据的结构之外。PostgreSQL 使用多版本并发控制（MVCC）模型，其中对行的更新不会就地修改数据。相反，它们会创建该行的新版本，该版本仅对新事务可见。随着旧事务的完成，将出现任何事务都不可见的行，这些行可以被清理。这就是清理（vacuuming）的本质，这是一个关键的后台维护过程，通常是透明的，但了解它很重要。第二个优点是一个关键点，我认为它使天平倾向于使用建议锁；当获取锁的事务或连接完成时（取决于获取的类型），它们会自动释放。

对于我们的系统，我们将使用绑定到连接的建议锁。当连接终止时（无论是有意还是无意），服务器都会保证已获取的锁将被释放。

在尝试获取锁时，可以使用两种基本模式。第一种是使用阻塞调用，即锁获取会阻塞，直到可以获取为止。我不喜欢这种方法，因为阻塞语句表现为一个长时间运行的 `SELECT` 语句；锁操作是通过使用 `SELECT` 调用的函数来完成的。我发现长时间运行的语句会扭曲对系统的监控，因为鉴于它们是预期行为，需要将它们过滤掉。长时间运行的语句也可能不会从调用方产生任何网络活动，这可能导致底层 TCP 连接在某些环境中超时。这尤其难以调试！

我偏好的模式是使用尝试获取锁并立即返回一个布尔结果（指示获取是否成功）的锁获取函数。这允许我的代码控制尝试获取锁的频率，通常是每 5 到 10 秒一次。我更喜欢这种方法，因为它避免了上一段中的两个问题；获取尝试是确定性地运行的，并且每次尝试都会使用连接，从而防止它显得异常空闲。

一旦获取了锁，就需要使用连接以确保其健康。为此，我每隔几秒在一个循环中执行一次 `SELECT 1` 调用。如果你不熟悉这个惯用法，它与连接池用于验证 PostgreSQL 连接健康状况的语句相同；它返回字面值 `1`。如果连接意外终止，该语句将失败，从而使我们的代码能够对终止做出反应。

### LISTEN 和 NOTIFY

PostgreSQL 包含一个发布/订阅（pub/sub）消息系统，其形式为 `LISTEN` 和 `NOTIFY` 命令。消息被组织到通道中，并且可以包含可选的有效载荷。`NOTIFY` 命令用于发送消息，`LISTEN` 用于监听消息。

#### NOTIFY

对于发布，PostgreSQL 文档建议的一种模式（也是我使用的模式）是附加到表上的触发器，用于生成消息。这可以与 PostgreSQL 的 JSON 函数结合使用，将更新后的行作为 JSON 传递回应用程序。以下是一个示例片段，我将用它来进一步解释：

```
CREATE FUNCTION send_notification()
RETURNS TRIGGER AS
$$
BEGIN
PERFORM pg_notify('my_channel_name', row_to_json(new)::TEXT);
RETURN NULL;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER notify_table_updated
AFTER INSERT OR UPDATE
ON your_table
FOR EACH ROW
EXECUTE FUNCTION send_notification();
```

从底部开始，一个触发器被附加到一个表上，在示例中是 `your_table`。你还需要命名触发器，而不是我使用的占位符名称 `notify_table_updated`。对于插入或更新的每一行，将调用 `send_notification` 函数。这是在表上创建触发器的样板代码；要执行的操作与在表上创建触发器是分开的，因为这允许该函数在多个表之间重用。

`send_notification` 函数就是该操作。它是一个定义为触发器的函数。它执行单个任务，即调用 `pg_notify` 函数。此函数等同于 `NOTIFY` 命令；之所以使用它而不是 `NOTIFY`，是因为它更容易与非恒定的通道名称或有效载荷一起使用，而示例正是这样做的。通道名称是 `my_channel_name`，有效载荷是修改后行的 JSON 表示。在触发器中，`new` 指的是数据的新状态，我们将其传递给 `row_to_json` 以将其转换为 JSON，然后强制转换回 `TEXT`，因为这是 `pg_notify` 所需的数据类型。

使用 `NOTIFY` 的另一个好处是它是事务性的。在事务中发送的消息直到事务提交后才会被传递。如果事务回滚，则永远不会传递任何消息。

从应用程序代码中显式调用 `NOTIFY` 也完全没有问题。在大多数情况下，我更喜欢触发器方法，因为事件通常与表中的数据更改相关，并且将其与数据库中的表耦合可确保始终生成这些事件。在某些情况下，我想发送与表更改无关的消息，例如在集群中传递心跳消息。

#### LISTEN

对于订阅，使用 `LISTEN` 命令。它接受一个参数，即要监听的通道名称。一个连接可以同时监听多个通道。

由于此功能不在 SQL 标准之内，因此对事件的访问是 PostgreSQL 驱动程序中的专有功能。它不会在许多平台提供的数据库无关 API 中公开。

正在 `LISTEN` 的连接应专用于此目的。返回的数据结构将根据所使用的客户端 API 略有不同，但所有 API 的共同点是可以访问通道名称和有效载荷。

我将在本章后面介绍 R2DBC API 时更深入地讨论 `LISTEN`。



### 模式（Schemas）

PostgreSQL 通过模式（schemas）在数据库内部提供了一层组织架构。表、类型和函数等数据库对象始终存在于某个模式中。如果你之前使用过 PostgreSQL，可能见过对 `public` 的引用，这是所有数据库都有的默认模式，也是表和其他对象默认被创建的位置。有一个名为 `search_path` 的运行时参数，用于控制搜索对象时要查找哪些模式。默认情况下，它包含 `public`，这就是为什么你可以直接写 `SELECT * FROM my_table` 而不是 `SELECT * FROM public.my_table`。

模式的首要好处在于授权。由于我们将使用 PostgreSQL 实现多种功能，我们可以利用模式来组织对象并授予访问权限。我们的系统将使用 PostgreSQL 来存储命令日志、事件日志以及用于查询的物化视图。这些将各自作为一个独立的模式，然后我们可以为这些模式创建读写角色，代码的不同部分在访问数据库时可以使用这些角色。严格来说，模式并非实现此功能的必要条件，但它们有助于维护，因为可以在模式级别授予权限，从而无需显式枚举模式内的每个对象。

模式的第二个好处是允许表名重用。数据库对象名只需在*一个模式内*保持唯一。将模式的内容视为一个接口。一个充当事件日志的模式内部会包含某些对象。如果需要另一个事件日志，可以创建一个包含相同内容的新模式。应用程序代码可以完全重用，唯一需要做的更改是修改连接的 `search_path` 运行时参数，使其包含所需的模式名称。

我发现当与物化引擎结合使用时，这一点甚至*更*加强大。设想一个场景：应用程序的新版本需要对用于满足系统读取请求的表进行不向后兼容的更改（如果可能做到向后兼容，应始终优先选择）。读取模型可以在一个新模式中重建，使用新的应用程序代码和数据库内部的新模式。如果需要回滚应用程序代码，操作也很直接，因为先前版本使用的模型仍然存在于先前的模式中。新模型可以在后台物化，旧代码从旧模型提供读取服务，直到新代码完成新模式的物化。此时，旧代码退出服务，新代码开始处理读取请求。

### 仅追加日志（Append-Only Log）

使用 PostgreSQL 实现仅追加日志非常直接：它就是一个表！仅追加特性可以通过使用 PostgreSQL 的授权系统并拒绝更新或删除行的能力来实现。我们期望从仅追加日志中获得的其他特性——ACID 事务以及查询部分或全部行的能力——已经是 PostgreSQL 的特性。

PostgreSQL 是一个功能丰富的数据库系统。我相信它是大多数项目的绝佳起点，并且对于更多项目的整个生命周期来说也足够使用。在本节中，我解释了我为何选择使用它来满足我们系统所需的全部四项技术。在下一节中，我将“向上移动技术栈”，讨论我对承载我们业务逻辑的应用层的选择。

## Java 与 JVM

Java 虚拟机（JVM）已被证明是一个极其强大的业务应用运行时平台。它拥有成熟的工具生态系统，用于软件打包、调试和性能监控。同时，还有一个充满活力的开源社区，提供各种库和框架，可协助处理你可能遇到的许多任务。

虽然 JVM 上还有其他语言可用，但我发现 Java 语言是我的首选。语言设计者在添加新特性时采取务实的态度，倾向于深入思考这些特性如何相互作用，并理解基于它们编写的代码将服务多年。我相信这种审慎的变更节奏是 Groovy、Scala 和 Kotlin 等其他语言兴起的原因之一。虽然这些语言中的每一种都可能带来短期或中期的生产力提升，但我认为它们并非那些生命周期可能长达十年或更久的软件系统的最佳选择。

人们常将 Java 与 COBOL 进行比较，甚至有一种说法是 *Java 是新的 COBOL*。尽管这两种语言截然不同，但*事实上的企业级语言*的角色已经从 COBOL 转移到了 Java。

Java 还非常注重向后兼容性。为早期语言版本编写和编译的代码，可以在新版本的 JVM 上继续编译和执行（前提是代码作者使用了公共 API）。我相信这对业务应用非常重要，因为这有助于最大限度地减少系统所需的维护工作。

没有哪种编程语言是完美的（尽管我怀疑一些 LISP 的支持者会对此提出异议）。Java 是一个务实的选择。如果你对其他语言或运行时更熟悉，那就用那种语言来实现这个系统。但我恳请你考虑一下你工作的预期寿命，要意识到如果它最终成功，其寿命很可能会比你想象的要长。你是在为未来的自己，或者最坏情况下，为一个完全陌生的人，设置一个从废弃语言或运行时迁移的难题吗？正是这种思路让我坚持使用 Java 和 JVM。它拥有超过二十年成功应用的可靠记录。

## GraphQL-Java

GraphQL 拥有一个健壮的规范。它定义了语言、类型系统、验证方式以及要使用的执行模型。规范中包含的详细程度对技术的使用者非常有益，因为它允许生产者和消费者理解预期的行为。

规范中拥有如此详细程度的另一面是，它不适合一次性实现。值得庆幸的是，有许多语言都有该规范的开源实现。2015 年，Andreas Marek 启动了 GraphQL-Java 项目，它已成为 Java 领域领先的实现。这是一个专注的库，反映了 GraphQL 规范的专注点。该库*仅*管理 GraphQL 文档的执行，将代码如何与其他系统通信的问题留给用户处理。这一点，加上代码库专注于最小化额外的第三方依赖，使其成为构建系统的可靠选择。

该库内部还有一个协同的实现决策。回想一下我们自我施加的约束：希望实现实时性，并通过使用 GraphQL 的订阅来实现。GraphQL-Java 的订阅实现使用了响应式流（Reactive Streams）规范，即 [`www.reactive-streams.org`](http://www.reactive-streams.org)。响应式流是一项努力，旨在为异步流处理提供一个标准，其理念源于响应式宣言以及我们正在构建的这类响应式系统。GraphQL-Java 依赖于一个 API，而非一个具体实现。在下一节中，我将更深入地解释响应式流。



## Spring Boot

自 2002 年起，Spring 框架便已成为 Java 生态系统的一部分。当时，它是 Sun Microsystems（Java 的创始者与管理者）所推崇的企业级 JavaBean 编程模型的一种更简单的替代方案。Spring 不断适应 Java 应用程序在架构和打包方式上的变化。在 21 世纪末微服务架构兴起之前，开发者将代码打包并部署到正在运行的应用服务器中，例如 Apache Tomcat、JBoss 或 IBM 的 WebSphere（仅举几例）。这些通常是多租户系统，由中央团队管理。与微服务兴起同时出现的是 DevOps 实践，即开发者负责其系统的完整生命周期，包括运维。DevOps 与本书第二章中提到的敏捷和 CI/CD 实践密切相关。

在 21 世纪初，Spring Boot 诞生了。它颠覆了 Spring 框架与应用服务器之间的关系。Spring Boot 不再将代码部署到应用服务器中，而是将应用服务器的必要部分（通常仅限 HTTP 服务器）嵌入到 Spring Boot 应用程序内部，从而将进程生命周期的控制权交到开发者手中。在过去十年中，Spring Boot 已成为使用 Spring 框架的主要方式。它提供了易于使用的模式，使开发者能够快速组装和维护应用程序。

与 Java 和 JVM 一样，Spring 的维护者重视向后兼容性，并努力在主要版本中保持这一点。我的观点是，Spring 正在策划一套经过深思熟虑的抽象。通过针对这些抽象进行构建，我们的代码得以免受实现细节变化的影响。我在自己的工作中重视这种做法，因为我相信这是构建易于变更的系统的重要组成部分，而这也是本书所描述系统的指导原则之一。Spring 的演进速度确实比 Java 快，相应地，随着抽象的完善，某些领域的弃用速度也更快。本书的指导原则在此再次发挥作用。通过确保我们的系统是模块化的、可测试的且易于变更，适应系统所基于框架的变化的过程，与适应业务变化的过程是相似的。Spring Boot 背后的原则与我们的原则是一致的，我发现它是我工作的绝佳起点。

## Project Reactor

我在第 3 章和第 4 章中讨论了响应式宣言的四个属性，以及如何组合它们来创建响应式系统。这些思想催生了一套供实现者使用的抽象，即响应式流。回顾第 3 章，我曾说过消息驱动是宣言中最重要的属性。该规范所启用的正是这个属性。流由离散的消息组成，该规范通过引入`Publisher`和`Subscriber`的概念，形式化了低层级的交互模式。在第 3 章中，我还讨论了宣言的属性是如何可组合的。我将响应式流规范视为组合层级中的最小单元，使得围绕此规范构建的系统能够彻底地展现这些属性。

响应式流有时会被误解。响应式编程模型的使用有时被认为仅仅是为了让那些在高负载下运行、并希望使用异步消息传递来减少总线程数以优化可用硬件利用率的系统受益。虽然这确实是使用响应式流的一个原因，但这并非我们业务应用的理由。我们看重的是消息传递范式在系统所有层级的组合。有效利用计算资源是一个有益的副作用，而非主要目标。我重视的是构建一个完全响应式的系统，并最小化因范式不匹配而产生的阻抗失配。

响应式流规范处于一个比我们应用程序设计所需更低的层级。幸运的是，我们不需要自己实现该规范。Project Reactor 是 Java 中该规范的一个实现，也是 Spring 框架使用并集成到 Spring Boot 中的实现。Reactor 提供了许多流操作，例如`map`和`filter`，使我们能够专注于编写使用响应式流的代码，而无需实现流处理原语。

Project Reactor 提供了响应式流`Publisher`的实现，并通过静态方法（用于启动流）和实例方法（用于应用流操作）对其进行了增强。由于`Publisher`规范将其定义为长度未指定的流，Reactor 将其 API 表面分为两类。已知最多包含单个元素的流表示为`Mono`，而所有其他流则表示为`Flux`。这很有益处，因为`Mono`类型类似于 Java 标准库中的`CompletableFuture` API，并且 Reactor 提供了两者之间互操作的方法。

在第 3 章中，我引入了数据流的管道隐喻。Project Reactor 就是这些管道的实现。响应式流定义了管道的各个段如何交互。它形式化了发布和订阅、取消以及流量控制的语义。

能够以数据流的方式思考，正是吸引我使用响应式流和 Project Reactor 的原因。需要特别注意的是，该抽象在线程模型方面存在泄漏。在撰写本书时，JVM 将线程直接映射到操作系统线程。应谨慎限制并行执行的 JVM 线程数量，因为如果此数量超过底层硬件的能力，将导致体验下降。以数据流的方式思考*应该*能让开发者不必担心这些实现细节，在一般情况下也确实如此。但你需要留意可能执行阻塞操作（例如执行 I/O）的代码，因为这可能导致停顿。作为 Project Loom 的一部分，向 JVM 添加虚拟线程的工作正在进行中。完成后，虚拟线程将使 JVM 能够高效管理大量线程，并修复这个有泄漏的抽象。有些人认为这将消除对 Project Reactor 这类库的需求，但我不同意。正如我在此描述的，通过响应式流和 Project Reactor 进行数据流编程，无论底层实现如何，都将继续具有价值。JVM 中的这项工作*应该会*增加采用率，因为当前实现面临的许多批评都将得到解决。

我提到这个重要的细节，是因为有一个非常流行的阻塞 API 被广泛应用于业务应用程序，即 Java 数据库连接 API（JDBC）。为了实现将响应式流范式扩展到数据库的价值，人们创建了一个新的 API，即响应式关系数据库连接（R2DBC）API。



## R2DBC

R2DBC API 的创建旨在将响应式编程扩展至与关系型数据库的交互。JDBC 需要每个连接对应一个线程，而在响应式系统中高效管理连接可能相当棘手。通过创建一个专门的 API，响应式流的流量控制可以传播到数据库服务器。

我认为使用 R2DBC 的一个关键价值主张在于其响应式 API。这消除了包装 JDBC 以及为确保其阻塞 API 的安全使用而必须采取的相关注意事项。我过去曾成功做到过这一点，但现在很高兴不再需要借助 R2DBC 来这样做了。

R2DBC 的 PostgreSQL 驱动程序有一个让我非常满意的特性——能够将 `LISTEN` 命令的结果作为 Project Reactor 的 `Flux` 获取。这使得 PostgreSQL 的发布/订阅消息能够作为我们应用程序中响应式流内的消息被消费。

## jqwik

Java 中有一个用于基于属性测试的优秀库，即 jqwik ^(¹⁷)。它是广泛使用的 JUnit 5 平台的另一个引擎，允许你将其与同样使用该平台的其他基于示例的测试一起使用。jqwik 对生成一系列要执行的操作提供了一流支持，这使其成为驱动我在第 12 章中描述的那种测试的理想选择。每个变更都会有一个关联的操作。当被调用时，操作会获得一个你定义的状态对象。这个状态对象充当预言机。操作将调用其变更，并在成功后也更新预言机。然后，该操作可以执行一个 GraphQL 查询，利用预言机中的信息来询问系统的当前状态，以了解其应该是什么状态。为了控制操作被调用的顺序，它们可以有选择地设置前置条件。使用第 12 章中的示例，发票在提交审批之前必须存在。调用变更以提交审批的操作将有一个前置条件，即至少有一张可以提交的发票。这将防止 jqwik 创建一系列对你的领域无效的操作。

对于测试的详尽程度，你可以控制两个参数：要执行的操作数量以及执行这些操作的次数。如果你想运行 100 个操作，你可以进行一次包含 100 个操作的尝试，或者进行 4 次每次包含 25 个操作的尝试。适当的平衡取决于你领域的性质，但我倾向于进行少于 10 次尝试，每次包含几十个操作。我会根据构建系统所花费的时间来平衡每次测试运行中的尝试次数和操作数量。如果你正在实践持续交付并且每天构建多次，那么较短的构建时间可能对你很有价值。

倾向于少量操作的另一个原因是当你不可避免地重现故障时。当 jqwik 测试失败时，输出包含一个种子值，可用于重现相同的随机值。这就是确定性至关重要的原因。测试运行中的所有随机性*必须*与种子值相关联，才能使种子在重现故障时有用。

随机生成的一系列操作不一定会调用你定义的所有操作。为了确保测试能够按照你的意愿进行，jqwik 允许你收集每次尝试中操作的统计信息。然后，你可以为这些统计信息定义成功标准，如果未达到标准，测试将失败。我使用此功能来确保每个操作在每次尝试中至少被调用一次。

## 整合所有内容

我在本章中描述的工具可以组合起来形成一个响应式系统，正如我在第 3 章中所描述的那样。此外，GraphQL 外观背后的实现也将遵循响应式流规范，从而允许形成一种连贯的内部编程范式。

该系统将在 PostgreSQL 数据库中使用三种模式：一种用于存储命令，另一种用于存储事件，第三种用于物化视图以处理 GraphQL 读取请求。就像代码被组织成包来分组和隔离关注点一样，数据库中的对象也采用相同的方法。

Spring 框架包含一个将 GraphQL-Java 与 Spring 集成的库，即 Spring for GraphQL，它可以在 Spring Boot 中使用。系统 GraphQL 模式中定义的每个字段都由提供该字段数据的代码支持。大多数字段将使用默认实现，该实现能够在常见的关联结构（例如 `Map`、类的字段或遵循 JavaBean 约定的方法）中查找字段的值。这些代码片段被称为 `DataFetcher`，这是它们必须实现的函数式接口的名称。Spring for GraphQL 的一个特性是 `DataFetcher` 可以返回 Project Reactor 类型，例如 `Mono` 或 `Flux`，从而允许将响应式流代码与 GraphQL 运行时直接集成。这使得诸如请求截止时间之类的功能以及内部组件之间隐喻性管道的直接连接能够得以直接实现。

我们的系统可以被视为一个循环，这是我在第 8 章中介绍的一个概念。通过使用 GraphQL 模式的读取能力，可以查看系统的当前状态。可以通过查询获取某个时间点的快照。可以使用订阅来获取某个时间点的快照，然后随着数据的变化获取后续快照。要更改系统的状态，可以发出一个变更。该变更被系统接收，命令生成器将变更的负载转换为一个命令。然后，此命令被追加到命令日志中。命令处理器从命令日志中消费，并使用变更器和应用器函数，结合必要的状态来处理命令，以执行业务规则。此过程的一个输出是处理后的命令，用于记录命令的处理结果。此结果会反馈给变更的发起者。成功的命令还会产生一组事件，这些事件被追加到事件日志中。物化引擎正在消费事件日志，并逐步更新我们数据库中支持 GraphQL 读取 API 的模式。本书第 3 部分详细描述了这些步骤中的每一步。

从消息日志开始，对每个日志的读取可以表示为一个包含所有当前和未来消息的无限 `Flux`。对此方法可以进行一种简单的优化。大多数消息日志消费者不会消费整个日志；他们希望从他们消费的最后一条消息之后的消息开始。这为消费者带来了一个简单的 API 契约：

```
interface MessageLog {
Flux messages();
Flux messagesSinceSequence(long seq);
}
```

由于使用了 `Flux`，这个接口既可以满足想要访问消息子集的消费者，也可以满足那些想要在新消息可用时订阅它们的消费者。



该接口的实现结合了其存储表中的过去和未来消息。过去的数据很简单：它是一个带有 `WHERE` 子句的 `SELECT` 语句，仅选择大于给定序列号的消息。未来消息通过调用 `LISTEN` 来复用相同的代码路径，该调用会接收包含新序列号的通知。当新序列号可用时，如果它大于最后已知的值，则重新执行 `SELECT` 并返回新消息。由于对 `LISTEN` 的调用返回的是单个值，如果值的生成速度快于消费者的处理速度，则应用背压很简单，因为只需要保留最新的消息，即包含新最大序列号的那条消息。

物化引擎使用包含事件的消息日志（即事件日志）来维护数据库中的模式，该模式用于满足 GraphQL 查询。当物化引擎更新表时，更新可以在适当的位置为你的领域发出 `NOTIFY`。这些通知将由驱动 GraphQL 订阅的代码监听，以了解何时在订阅流上生成新的数据点。这与消息日志使用的模式相同；`LISTEN/NOTIFY` 用于在新数据可用时通知系统的各个部分。

我们的系统将有几个 `DataFetcher` 实现。至少，它们会存在于 `Query` 和 `Subscription` 类型的每个字段上，因为这些字段代表了模式的入口点。这些实现将针对包含物化视图的数据库模式发出查询，该视图旨在处理这些请求。这样做的一个好处是，两个模式之间应该几乎没有阻抗不匹配。

在设计模式时，我喜欢重用 `Query` 和 `Subscription` 类型的字段返回的类型：

```
type Invoice {
id: ID!
customer: Customer
}
type Customer {
id: ID!
name: String
invoices: [Invoice]
}
type Query {
customer(id: ID!): Customer
}
type Subscription {
customer(id: ID!): Customer
}
```

`Query.customer` 和 `Subscription.customer` 都返回相同的数据。唯一（且重要）的区别在于，订阅字段将返回当前状态，随后返回任何更新后的状态。这遵循了我上面提到的实现 `MessageLog` 的相同模式；有一个可以加载数据的查询，并且通过 `LISTEN` 接收到的数据库通知会信号通知应用程序重新执行查询。通过重用模式，对系统某一部分的理解可以应用于其他部分，同时也为共享实现代码提供了可能性。
