# 2. 现状（及其成因）

既然我们已经确立了业务应用的定义，接下来我将讨论我对现状的看法以及现状是如何形成的。这基于我过去二十年（从 20 世纪 90 年代中期到 2021 年的今天）对行业的观察。

在本章中，我将描述我在职业生涯中观察到的实践，以及我认为它们可以如何改进。这将是一个以 Java 虚拟机（JVM）为中心的视角，因为我的大部分职业生涯都投入于此。根据你职业生涯所处的阶段，这可能也适用于你，或者可能适用。让我们开始吧！

## 业务应用架构

尽管业务应用架构在过去 20 年中发生了变化，但它们在很大程度上也保持不变。作为开发者，我们的系统重复着相同的基本交互模式，只不过是用当下的技术来实现。理解过去对于把握未来的机遇至关重要。



### 现状

一个“现代”商业应用很可能以基于 Web 的用户界面呈现。如果它足够新，可能是一个单页应用，即数据在后台加载，页面本身被动态重写。但这通常只是表面功夫。如果仔细观察，页面很可能正在向后端发起 HTTP 请求，来回传递 JSON 负载。HTTP 动词被粗略地用来描述预期的操作：GET 用于检索，DELETE 用于删除，PUT 用于创建，POST 用于更新。URL 模式可能经过一些设计，比如包含版本号，但很可能没有。它只是一个扁平的端点命名空间，就像一堆杂乱无章的函数。

如果你问开发者他们创建了哪种类型的 API，他们会称之为“REST API”，尽管事实远非如此。用 HTTP 动词来回传递未充分定义的 JSON，并非罗伊·菲尔丁提出该术语时的本意。我喜欢将这些 API 描述为“口语化 REST”。

有许多框架，例如 Rails、Django 和 Spring，它们帮助开发者处理暴露 HTTP 请求的机制，使得开发者只需指定 URL、HTTP 方法和负载的形状即可。

一旦收到请求，它将经过适当的验证，然后使用对象关系映射（ORM）框架来最终存储或从关系数据库中检索数据。这个 ORM 框架很可能是 Hibernate，因为它自 2001 年首次亮相以来就获得了极大的关注。

关系数据库是商业应用的主力军，而且理由充分。如果数据库不能可靠地存储和检索数据，那么存在于数据库和用户之间的软件层的质量就毫无意义。关系数据库有一个定义明确且标准化的交互机制：结构化查询语言（SQL）。尽管在业务中引入任何新技术都很困难，但引入一个新的主数据存储系统则更加困难。由于我们的商业应用对业务至关重要，这导致了对数据存储选择的规避风险心态。这就是为什么我*不*建议在数据存储选择上进行创新。在我的职业生涯中，大部分时间都在使用 PostgreSQL，它一直运行良好，并且还在不断改进。这也是我主张在本书中使用 PostgreSQL 的原因。如果你有偏好的其他数据库，请继续使用它。你拥有来之不易且值得保留的运维知识。

数据库中使用的模式可能大体上是合理的，这取决于业务在其生命周期内要求对软件进行的更改的性质，以及参与开发的开发者数量。一个非常可能的情况是，应用会原地修改数据。随着数据的变化，它过去的样子就消失了。有时，会有明确的业务需求要求“取消删除”操作或跟踪特定字段的变更。由于这需要额外的开发工作，除非明确要求，否则通常会被省略。

简而言之，一个“现代”的 Java 商业后端就是通过一个口语化 REST API 交换未充分定义的 JSON，并通过 Hibernate 在关系数据库中原地修改数据。

### 绿屏

我接触到的第一个商业应用是“绿屏”应用。这个术语是对连接到中央计算机的终端的回溯，这些终端拥有多个显示器和键盘（终端），而不是联网的计算机。这些终端通常是绿色的 CRT 屏幕（因此得名），功能有限。它们可以显示来自中央计算机的文本，转发输入，或许还有打印功能。如果你在 Linux 中配置过终端，你可能会在 TERM 环境变量的潜在值（如 vt100）中看到对这些设备的引用。

随着个人计算机变得普及，硬件终端模拟器被运行在 Windows（当时常见的桌面操作系统）上的软件终端模拟器所取代。这就是我接触到它们的地方，这些业务依赖的应用比它们最初为之开发的硬件的生命周期更重要。硬件终端已被桌面软件取代，中央计算机也被局域网取代。

其中一些系统至今仍在使用。它们是用现在被视为遗留的语言编写的，例如 COBOL 或 RPG。用另一种语言重写这些系统是一项巨大的工程，并且会给业务带来风险。作为一个经济决策，通过继续使用旧系统并管理维护这些系统的团队的人员风险，可以减轻迁移系统中因错误导致的业务风险。如果你足够仔细地审视，可以将这些系统在终端上渲染输出的方式视为一种巴洛克式的基于文本的协议。有一些产品采用这种方法，并允许为它们创建基于 Web 的用户界面。

我提到这些应用是因为它们是真正的“瘦客户端”架构。客户端设备（终端）几乎只能进行输入和输出。没有在终端上运行“客户端”逻辑的设施。

### 富客户端

由 Microsoft Access 和 Visual Basic (VB) 推广开来的富客户端，是一种安装在用户机器上并通过局域网连接到中央应用的应用。在许多情况下，中央应用就是一个关系数据库。局域网当时尚未标准化为 TCP/IP，创建自定义协议也并非易事。一个足够复杂的系统可能使用了 CORBA，这是一种跨语言和跨机器执行方法调用的技术。创建多用户应用的最简单方法是依赖数据库服务器的线路协议，并将其用作应用的服务器部分。

Access 和 VB 为构建用户界面提供了可视化体验。这使得那些几乎没有传统编程经验的人也能构建简单的应用来存储和检索数据。Microsoft 提供了一套强大的内置 UI 控件，同时还有销售具有附加功能的控件的第三方供应商。构建多用户商业应用变得简单直接，其结果的行为方式与其他桌面软件类似。作为用户机器上的一个独立应用，也意味着能够直接与其他工具（如 Excel 或 Word）进行交互。

然而，对于开发者来说，分发他们的软件是一个巨大的难题。应用并非一个可安装的单一文件。共享库需要被复制到机器上的特定位置。如果发生冲突，调试结果可能会极其困难。“DLL 地狱”这个术语就是用来指代这种痛苦的。

富客户端为用户提供了出色的体验。但这以运维困难为代价。如果软件无法在你的机器上运行，再好的用户体验也毫无价值。

模型-视图-控制器（MVC）架构模式揭示了如何在图形用户界面中分离职责。Access 和 VB 并未严格遵循此模式。关系数据库可以被视为模型，而每个显示窗口（用他们的术语来说是“窗体”）类似于组合的视图和控制器。



### Web 应用程序

与富客户端同时期，万维网诞生了。随着 1990 年代接近尾声，TCP/IP 成为局域网的事实标准，因为它除了解决互联网接入问题外，还解决了局域网通信问题。这开启了在局域网上运行 Web 服务器、让用户访问“内网”的可能性。

虽然与富客户端相比，最初的用户体验较差，但它简化了开发人员的工作，因为无需在用户机器上安装软件。解决“DLL 地狱”的运维难题对 IT 部门来说很有价值。

最初的 Web 应用程序是基于页面的，这让人回想起万维网起步时以文档为中心的视角。我认为有两个发展导致了 Web 应用程序的井喷。

第一个是 HTML 规范修订的暂停。在整个 1990 年代，W3C 发布了 HTML 2、3，最终是 HTML 4。HTML 4 最初于 1997 年底发布，最终修订版 4.01 于 1999 年底发布。直到 HTML 5 的制定工作开始，中间相隔了将近九年。

虽然这是规范上的停滞期，但也是一个稳定期。我相信这种稳定性为构建应用程序提供了稳定的基础。

第二个重要的发展是现在所谓的 XMLHTTPRequest，即 XHR。这是一种无需完全重新加载页面即可与 Web 服务器通信的技术，使得 Web 应用程序能够拥有开始模仿富客户端的交互模式。微软 Internet Explorer 浏览器的一项专有功能允许浏览器中运行的 JavaScript 通过一种名为 ActiveX 的技术访问安装在系统上的组件。这是 Visual Basic 用于构建其界面的同一技术 COM 的后代。正是通过这条路径，微软为 Outlook Web Access（允许基于 Web 访问企业邮件服务器）开发的一个组件可以在任何页面上使用。一旦这项技术被发现，它便成为了事实标准，并被其他浏览器引擎所实现。

有了稳定的规范和在后台与服务器通信的能力，就有可能构建出“足够好”以用于日常使用的业务应用程序。一些业务应用程序仍会需要富客户端，因为基于 Web 的应用程序无法满足其需求。然而，如果 Web 浏览器的能力足以满足某个应用程序的需求，那么 Web 应用程序在运维上的简便性使其成为更简单的长期选择。

那些帮助开发人员编写支持这些基于 Web 的应用程序的应用服务器的框架，都趋向于采用 MVC 模式来组织自身。模型不再是数据库本身；它是应用服务器中的一个对象模型，通过 ORM 映射到数据库。代码被组织成名为控制器的类，它们的响应负载是视图。这就是当前现状的起源。

我认为，基于 Web 的业务应用程序正是在这里陷入了停滞。虽然 Web 领域持续有新的发展（我将在下一节讨论），但总的来说，许多业务应用程序的需求已经得到了满足。应用程序的浏览器端部分的开发方式发生了变化，但它们所使用的底层浏览器原语并未改变。

### 消费类应用程序的兴起

在整个 2000 年代，如果你想使用一个网站，你必须使用台式机（或笔记本电脑）上的 Web 浏览器。这十年间，家庭宽带互联网接入迅速普及。随之而来的是“Web 2.0”运动，它带来了网站交互性的增强，而这得益于浏览器中 XHR 的可用性。面向用户的网站蓬勃发展，用户在线活动也扩展到了类似于业务应用程序的事务性交互之外。

2007 年，苹果公司发布了第一代 iPhone，一股冲击波席卷了整个科技行业。它允许人们用手持设备浏览网页，页面渲染效果与在台式机上相同。苹果最初的计划是让所有用户可安装的应用程序都是网站，但他们后来放弃了这一计划，并允许开发者编写通过 App Store 交付的原生应用程序。应用程序有了一个新名称：“App”。谷歌的 Android 移动操作系统迅速效仿了苹果的做法，推出了自己的应用商店。

这使得面向消费者的 UI 开发人员的生活变得复杂。以前，他们只需要交付一个在主流桌面浏览器引擎上都能正常运行的网站。他们可以对可用的输入设备（键盘和鼠标）、大致屏幕尺寸、像素密度，甚至互联网连接的性质（稳定的延迟和丢包率，尽管带宽因宽带可用性而异）做出假设。随着移动设备的爆发，出现了无数种屏幕尺寸，鼠标被触摸交互取代，网络具有可变的延迟和丢包率。不过，至少有一个通用的浏览器引擎：苹果的 WebKit。一个可交付物——桌面网站——演变成了四个：桌面网站、移动网站、iOS 应用和 Android 应用。为了限制目标平台的数量，一些原生移动应用嵌入了浏览器视图来提供部分功能。

为了让移动网站拥有类似应用的能力，浏览器创新再次启动，增加了许多新功能。Web Workers 允许后台执行，SSE 用于服务器推送数据，WebSockets 用于持久的双向连接。

JavaScript 生态系统中出现了利用这些新 API 的框架的“寒武纪大爆发”，并全面简化了开发体验。一种新的数据交付模式 GraphQL 应运而生。

GraphQL 提供了一种机制，允许 API 调用者精确指定他们希望返回哪些数据。在通过移动网络传输数据时，这是一个非常有价值的功能。这与之前的 MVC 风格 API 是截然不同的范式。为了在旧的 Web API 和移动客户端所需的 API 之间架起桥梁，出现了“服务于前端的后端”（BFF）模式。这是一种服务，充当现有 API 与客户端所需体验之间的适配器。

用户体验方面的创新也在发生。在 iPhone 和 Android 发布之后，移动设备的采用率加速增长，应用的易用性成为市场上一个有价值的差异化因素。应用中发明了新的范式，并渗透到整个生态系统中（例如，下拉刷新、*无需*刷新、通知、无限滚动和社交协作）。

以前，通过 Web 交付的业务应用程序与面向消费者的网站之间的区别主要在于规模和受众。随着移动设备的出现，面向消费者的网站不仅仅是网站，它们还必须提供应用体验。从事消费者体验工作所需的技能集已经扩大。

因此，我得出关于现状的论断。在这次移动设备爆发之前，业务应用程序和面向消费者的网站基本上可以用类似的方式构建。为什么业务应用程序没有跟上步伐？我有一些理论。



首先，对于新的业务应用而言，那些在各类应用中普遍存在的功能如今已被视为理所当然，因此产品经理在撰写需求时不会像对待业务特有功能那样明确提及。刷新/重新加载按钮是我最喜欢的例子。自 XHR 发明以来，能够自动更新数据的网站就已存在。作为一种用户体验范式，直到移动端爆发式增长，它才变得普遍。如果使用刷新按钮，浏览器除了重新请求已更新的数据外，还可能重新请求那些未发生变化的资源。为了优化体验，刷新功能被移入应用内部，并最终消失。取而代之的是，内容会自动保持最新状态。如果一个功能既不可见又无处不在，那么在明确列举需求时就更容易被遗漏。

其次，是开发者经验的缺乏。随着处理消费端体验所需的技能集不断扩展，技能集之间出现了分化。开发者可能需要接触过那些利用了新浏览器功能的系统，才能提出将其整合进来的建议。

最后，是架构上的阻抗。为了拥有实时更新的页面或为用户触发通知，要“硬性附加”一个变更流可能并不简单。这触及了本书剩余部分的核心。我将在本书的第三部分介绍一种构建系统的不同方式，以解决这种阻抗问题，并有望为下一阶段的计算设定标准。MVC 模式一直为我们服务得很好，但我们还可以向前迈出一步。在为此做准备的同时，还有其他一些有价值的实践可以融入我们的工作中。

## 并行实践

交付软件所涉及的实践也在不断演进。软件架构不仅仅关乎各个部分如何组合在一起，还关乎这些部分如何协同工作、如何随时间变更、如何被验证，以及其生产环境性能如何评估。在本节中，我将介绍一些我认为对应用长期成功有价值的实践。

### 敏捷

当敏捷宣言^(⁵)在 2001 年发布时，它极大地改变了人们对软件交付方式的广泛认知。我记得第一次读到它时，感觉就像脑中亮起了一盏灯。它简洁地描述了软件团队面临的一些压力，并提供了一个极其宽松的框架来思考这些问题。

构建软件的行为不受构建物理对象时相同的物理约束限制。我认为，充分利用这种差异并以此来创建更好的软件至关重要。软件归根结底是比特的排列。我们构建了奇妙的系统来简化精确排列它们的过程。

宣言中让我感触最深的一点是“拥抱变化”。改变软件需要花费时间；但在这个过程中没有材料损失。我坚信，构建系统的一个关键原则就是它必须易于变更，这一点将贯穿我们在本书剩余部分构建的内容。

宣言也谈到了围绕变更的流程。正如第一章所述，我们的业务应用是为业务量身定制的。开发者将自己视为业务的协作者而非承包商（即使实际关系是通过咨询公司建立的）非常重要。

业务应用的*目标*是帮助业务，而敏捷宣言描述了如何处理这些交互。

### 测试驱动开发

测试驱动开发（TDD）源于敏捷实践中的极限编程（XP）。肯·贝克在 20 世纪 90 年代“重新发现^(⁶)”了这项技术，并在其 1999 年出版的《解析极限编程》^(⁷)一书中进行了阐述。

其核心思想是在实现功能本身之前，先为功能编写测试。起初，测试应该运行并失败，因为功能尚未实现。此后，目标就是让测试通过。

我发现这个实践非常有价值，因为它让我在编写代码之前就处于代码使用者的位置。编写测试的行为有助于从 API 的角度聚焦于什么才是重要的，以及代码应如何与代码库的其他部分通信。我觉得它也有助于创建更易于维护的代码。在实现之前编写测试的行为，不知何故会导向更周到的设计。这种设计更易于未来的变更。我确信，代码有测试有助于可维护性，因为可以放心地进行更改。然而，我发现先编写测试再实现代码，与先实现代码再编写测试，两者之间存在差异。根据经验，我亲眼目睹了它的有效性，并已成为这一实践的坚定信徒。无论如何都需要编写测试来验证代码是否工作，那么为什么不先写呢？

### 持续集成与持续交付

持续集成与持续交付（CI/CD）是另外两项 XP 实践。它们既相互区别又相互关联。当结合使用时，它们将软件的交付流程正式化。CI 已成为一项标准实践。如果你的项目使用构建服务器，在代码推送后编译项目并运行测试，那么你就在实践 CI。要进入 CD 阶段，构建服务器还应该将软件打包以供部署，甚至可能自动化部署本身。

当两者结合实践时，CI/CD 允许你将更改推送到仓库，无需任何额外的人工干预，就能让它在测试环境中运行。这也可以扩展到自动将构建提升到生产环境，尽管我工作过的大多数业务环境都倾向于在该环节保留人工参与，即使只是简单地点击一个按钮。

我发现这些实践很有价值，因为它们完善了反馈循环。存在于构建服务器上的测试环境与部署后的环境并不相同。这种交付软件本身的“流水线”方法，将与我们构建现代业务应用内部结构的方式相呼应。这里也存在一个哲学思想实验的元素，类似于“如果一棵树在森林里倒下，而周围没有人听到，它发出声音了吗？”——“如果软件被更改了但从未被部署，它真的被更改过吗？”我将反复强调的一个观点是，应用应该易于变更。让这些变更轻松地交付到用户手中，对于实现这一目标至关重要。



### 可观测性

可观测性是一个时髦的词汇。它源于控制理论，衡量的是从系统外部输出推断其内部状态的能力。在 2010 年代末，它开始被应用于通常被称为“监控”的领域。

作为软件开发人员，我们一直在实践这一点。我们利用系统的输出（例如日志消息）来构建对系统内部情况的理解。从最简单的意义上说，“实践可观测性”意味着确保你的系统能够发出足够的信息，以便你可以重建它们正在执行的操作。这始终是一项最佳实践，但我发现其实现方式往往是临时且针对具体情况的。

随着我们的应用程序越来越不孤立运行，将这一实践正式化的重要性日益增加。它们是支撑业务的大型应用生态系统的一部分。让应用程序以统一的方式发出这些输出是有价值的，这样就能跨应用程序调试问题。

我从这一实践中发现的价值，改变了我处理调试问题的心态。理想情况下，我不希望必须使用调试器，也不希望添加打印语句来找出问题的原因。我试图将构建系统作为目标，使其能够自然地发出足够的信息，以便在脑海中推演可能的原因，然后通过测试来验证假设。这是另一种实践，它将渗透到你的软件结构方式，以及我们构建现代业务应用的方式中。

## 总结

业务应用的架构方式并没有发生实质性改变。在 iPhone 问世之前，业务应用和消费类应用是相似的。iPhone 带来了消费类应用的许多变革，但这些变革并未在业务应用中得到广泛采用。其他开发实践可能已经融入了你的工作流程，因为它们与应用程序的架构在一定程度上是正交的。

我认为，有机会将消费类应用的技术和敏捷实践完全整合到现代业务应用的设计中。
