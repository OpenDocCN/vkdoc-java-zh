# 6. 管理时间

时间能被“管理”吗？在我们的应用程序中，我们控制着它们的通信，并由此管理它们如何了解、通信和处理数据。正是通过控制信息流，时间本身才得以管理。当我们创建业务应用程序时，我们正在建立一个业务本身的虚拟缩影。作为事实来源，应用程序创建的时间线可以成为回答“发生了什么？”这一问题的官方时间线。

## 回答“发生了什么？”

在我参与过的业务应用程序中，用户经常问我一个问题：“为什么系统会有这个值？”这个值可能是计算的结果、从其他系统接收的数据，或者由用户提供。在所有这些情况下，系统都需要*记住*它的输入，才能回答这个问题。

最简单的例子是我在上一章讨论过的关于推导数据的业务规则。规则的输入存储在系统中，推导出的值作为展示逻辑的一部分被使用。那么，增强一个计算过程，使其输出也包含其输入的副本，就变得很简单。这将允许理解该规则的观察者独立验证计算结果。

除了这个简单的例子，事情就变得复杂了。如果一个计算结果被持久化了，我们如何知道计算时它的输入是什么？理想情况下，它们应该被记录下来，但这是明确的产品需求吗？如果不是，很可能就没有做。如果该值的持久化包含了时间戳，你可以尝试重建当时的数据是什么。

为了回答这个问题，我们需要知道系统在*过去某个时间点*的数据是什么。如果系统在事件发生时没有追踪这些数据，那么信息最终将会丢失。这就是现状架构中的“原地修改”。



## 追踪变更

在当前的架构中，你可能熟悉两种追踪变更的模式。每种模式都有不同的复杂程度。

如果追踪变更主要是为了支持应用程序的运维人员，那么可能会使用历史表。假设有一张用于追踪客户的表：

```
CREATE TABLE customers (
customer_id   BIGINT PRIMARY KEY,
customer_name TEXT
);
```

可能会使用以下历史表：

```
CREATE TABLE customer_history (
-- 0/1/2，分别表示插入/更新/删除
revision_type      INT,
revision_timestamp TIMESTAMP,
customer_id        BIGINT,
customer_name      TEXT
);
```

这里，`revision_type` 是一个枚举类型，表示变更的类型（插入、更新或删除），而 `revision_timestamp` 是记录变更的时间。设计时需要考虑，这条记录是 `customers` 表中变更前的行还是变更后的行。

如果变更追踪是应用程序的一个功能，那么有条件地查询历史表的复杂性可能并不理想。相反，可以将所有时间点的值都记录在同一张表中：

```
CREATE TABLE customers (
valid_from    TIMESTAMP,
valid_to      TIMESTAMP,
customer_id   BIGINT,
customer_name TEXT
);
```

在这个例子中，`valid_from` 和 `valid_to` 捕获了两个时间点，用于指示该行数据在哪个时间段内有效。这种模型可以轻松支持“在某个特定时间点，客户的名称是什么”这类时间点查询。但在写入时会增加复杂性，因为需要插入新值，并调整前一条记录的 `valid_to` 时间戳。这种模型如果使用 `customer_id` 作为主键也容易出问题，因为会有多行数据包含相同的值。需要一种更复杂的约束，以确保在给定的时间戳下，主键只有一个唯一的值。

这两种方法都存在一个额外的陷阱。数据库模式是*跨时间统一的*。唯一安全的变更是放宽约束，例如将 `NON NULL` 列改为允许 `NULL`，或者添加一个可为空的新列。任何其他类型的变更要么有风险改变现有数据，要么需要修改数据。

这带来了一个两难困境。是应该原样保留过去，还是应用程序需要以其他方式进行调整？任何解决方案都会有缺点，并导致“技术债务”。这是一个不易变更的系统。

即使采用了上述两种方法之一，或者另一种替代方法，仍然存在一个问题：捕获*原因*。

## 为什么会发生这种情况？

通过存储先前的值，我们可以看到过去是什么样子。通过比较过去的状态，我们可以看到值是如何变化的。这足以回答*是什么*。然而，我发现用户真正想问的是*为什么*。

数据不会凭空发生变化。存在因果关系。我提到的变更追踪机制只捕获了结果，即“是什么”。数据模型本身并没有被设计用来捕获原因，即“为什么”。

“为什么”或许可以推断出来。如果你知道在变更发生时应用程序中哪些操作是可能的，你就可以利用这些知识对数据进行模式匹配，以判断该变更是否是由某个特定操作引起的。这依赖于为每种变更类型设置特定的代码路径，但如果应用程序使用的是第 2 章中提到的“通俗 REST”，那么这些路径可能并不存在。如果用户界面通过发送包含更新状态的有效负载进行交互，那么上下文信息就会丢失。

我希望你对这个问题的本质有所了解。这是我们作为开发者曾经遇到并为自己解决的问题。我们在源代码中遇到过这个问题，并通过版本控制系统（VCS）解决了它。这是一个最佳实践，而且理由充分。VCS 系统的基本功能是追踪代码库的变更，即“是什么”。当进行变更时，你可以选择提供一条消息，即“为什么”。并没有要求必须提供合理的消息。我曾与一些开发者共事过，他们将其视为一个烦人的守门人，只提供一条象征性的消息来满足要求。我向他们强调了一个观念：一条好的提交信息*解释了为什么*要进行变更。“是什么”是显而易见的，就是变更本身。虽然我明天可能还记得为什么，但一年后可能就不记得了。我曾将提交信息描述为“写给未来自己的笔记”。即使是独自工作时，我也会让它们变得有意义。

如果追踪系统代码的所有变更是最佳实践，那么为什么追踪系统数据的所有变更不是最佳实践呢？我认为答案是**它应该是**。我相信在过去这是不可行的。追踪所有数据变更需要不断增长的存储容量。如今，现代存储系统的容量，尤其是结合按需云存储，使得这完全可行。那么，一个存储变更的系统应该如何构思呢？

## 事件

当我们考虑存储变更时，它们就是事件：系统中某个操作产生的结果或输出。从这个角度看，每一次变更都是一个事件。通过存储事件，我们可以构建一个系统变更的规范时间线。一个按时间排序的事件集合就是一个日志或账本。这就是 VCS 系统在概念上的运作方式，因为它们向我们展示了变更的历史。

在业务应用程序中，一个事件应该包含数据的变更以及变更的*意图*。这同时捕获了“是什么”和“为什么”。事件日志对于描述过去很有用，但对于理解当前状态效率不高。遍历不断增长的事件日志来查找数据片段是不可扩展的。相反，我们可以保留所有事件所隐含的持续运行结果，并用它来回答关于当前状态的问题。在 VCS 中，这相当于将仓库“检出”到文件系统。利用事件在特定时间点构建“世界状态”的行为称为*物化*。

这种技术被称为*事件溯源*；将系统的变更作为事件存储，并构建这些事件的物化视图来回答问题。通过将变更表示为事件，它们就变成了消息。这让我们回到了响应式宣言；将系统数据的变更表示为事件，使得变更本身成为消息。

我们也可以将同样的技术应用于对系统数据的*变更请求*。

## 命令

命令和事件是一枚硬币的两面。它们的区别在于与系统中数据的关系。对所有系统事件进行持续更新的物化视图就是系统的*当前状态*。

事件是向后看的。它们描述已经发生的事情。命令是向前看的。它们描述对系统状态的*变更请求*。关键词是*请求*。想要进行变更的意愿并不意味着它会被允许。是否允许，取决于针对静态数据的业务规则。

这是来自一种称为命令查询职责分离（CQRS）模式的概念。在这种模式中，有一个模型用于处理状态变更，另一个独立的模型用于查询。这与事件溯源自然互补。事件可以以多种方式物化，一种用于服务查询，另一种用于强制执行针对静态数据的业务规则。我将后者称为*命令处理*。它是指接收一个命令，判断其是否被允许，如果允许，则确定它会产生哪些事件。

我想指出的是，这*并非新颖*。这些想法和模式已经存在并实践了几十年。但我相信，在最近几年，它们的*实用性*或*适用性*已经发生了变化。这些是业务应用程序的宝贵实践，并且适用于面向未来的现代架构。



## 理解时间

我从历史视角开启本章，探讨如何回答“发生了什么？”这个问题。由此引出了将事件作为一种细粒度方式来记录*发生了什么*以及*为什么发生*。命令是事件的前身，而事件可用于构建状态的概念。这些概念清晰地映射到时间上。命令代表未来，事件代表过去，而状态则是当下。

这使我们能够在系统内部构建一条局部时间线。通过仅物化部分事件，就可以实现时间旅行回到过去。不过，这只是一条*局部*时间线。让我们暂且偏离主题，探讨一下*时间代表什么*。

时间是沟通*何时*发生的一种手段。沟通需要一个共同的参照系。在地球上，我们使用协调世界时（UTC）。

当接收到关于某事或某个事件*何时*发生的通信时，有两个维度值得关注：

1.  事件*实际发生*的时间。

2.  你*意识到*该事件已经发生的时间。

这两个时间点都是离散的时间点。**它们总是不同的！** 有时，差异微乎其微，比如你执行的操作，但并非总是如此。

如果你触摸炉子，在你意识到炉子很烫之前，你的手指就已经被烫伤了。这是一个关于信息传播延迟的直观例子。时间点 1 和 2 之间的间隔虽然短暂，却足以烫伤你的手。你需要*意识到*某个事件并对其采取行动。只有当你意识到炉子的温度时，你才能移开手。

在许多系统中，这两个维度被混为一谈。系统接收到信息的时间最终被当作事件发生的标准时间。理解这一点很重要：这是一种虚构，如果系统有多个事实来源，就会导致差异。

## 时间的序列化

最终，一个应用程序会生成一条关于“发生了什么”的单一时间线。在许多应用程序中，这并没有被显式管理，而是委托给了存储数据的关联数据库。然后由事务隔离级别来控制时间如何流动。只有一种模式能提供完全一致性，即`SERIALIZABLE`（可序列化）。它并非任何主流关系数据库的默认设置。我推测部分原因是性能，部分原因是它给应用程序开发者带来的负担。

大多数数据库的默认设置允许开发者以一种通常不会产生错误的方式操作数据库。错误会足够罕见，以至于错误可以沿着调用栈向上传播很远，或许最终导致用户需要重试某个操作。

`SERIALIZABLE`则不同。它要求所有交互都严格线性化。如果某个修改会违反这一点，就会返回一个错误，并且必须重试该事务。这就要求应用程序开发者将与之交互的数据库代码结构化，以支持多次调用。如果业务规则的类型相互混杂，例如执行副作用操作，就存在多次执行副作用的风险。

归根结底，数据库的事务日志是一个已执行更改的有序列表。它是一个*事件日志*，隐藏在数据库内部。

通过将变更请求表示为命令并将其存储为事件，我们将这个事件日志提升到应用程序层面，使系统能够将其作为一等公民来使用。这让我们能够*控制*时间的序列化。在决定处理命令的顺序时，我们就是在构建一条时间线。

## 最终一致性

时间的序列化提供了局部一致性。然而，我们的应用程序并非在真空中运行。它们是业务的一个*事实来源*，但并非所有事实的*唯一*来源。最终一致性指的是信息随时间传播的过程。当系统 B 意识到系统 A 中的变化时，B 与 A 就*最终一致*了。

不存在唯一真实的时间线。在一个由众多应用程序组成的大型系统中，每个应用程序对事件发生的顺序都有自己的视角，因为每个应用程序都会独立地意识到事件的发生。

通常，这可能是可以接受的。通过使用事件来沟通发生了什么，每个应用程序可以记录自己何时意识到信息，从而在局部回答“发生了什么”的问题。然而，在商业应用中，有时事件*何时*发生也很重要，这也必须被追踪。

## 双时态

双时态是指追踪*两个*时间维度：事件发生的时间，以及系统意识到该事件的时间。在业务领域，追溯性政策就是一个例子。

想象一个忠诚度计划，购物可以获得积分。积分是一种货币形式，可以在商家处消费。未使用的积分在一段时间后过期。在我们的例子中，积分在获得一年后过期。

积分追踪应用程序是一个独立的应用程序。它从业务的其他地方消费交易事件，以追踪其积分余额的状态，并直接管理用于消费积分的事务。因此，它追踪两个时间点：它意识到交易的时间，以及交易实际发生的时间。它是双时态的。

我“今天”的积分余额取决于该应用程序与交易事件流同步的最新程度。积分是根据交易发生的日期授予的，而不是根据积分追踪应用程序看到它的时间。

对于过去的任何一个时间点，积分余额都有两种视图。基于*当时*已知信息的积分余额是多少，以及基于*现在*已知信息的积分余额是多少。

一个保留两个时间戳的事件日志使得回答这些问题变得简单直接：一个交易时间，一个接收时间。要回答基于当时已知信息的余额是多少，只需查阅接收时间早于所查询时间的事件。要回答基于所有可用信息的过去余额是多少，则会使用交易时间。

在这个假设的系统中，交易时间可能是主要使用的时间戳。然而，记录接收时间仍然有价值。如果忠诚度计划的客户对他们的积分余额有疑问，拥有接收时间的信息对于解答问题可能是无价的。数据是有价值的，最好在可能的时候捕获它。

除了这个人为的例子，还有一种普遍适用的双时态形式：不断变化的业务规则。



## 业务规则变更

我曾将*可变更性*视为业务应用的重要属性。业务会发生变化，而软件适应变化的速度可能成为制约因素。

事件流可被视为时间流。随着事件发生，时间流逝。事件不必局限于*数据*的变化，它们也可以代表*行为*的变更。

以忠诚度计划为例，设想其在航空公司的应用场景。可能会出现某种情况（例如全球疫情），航空公司希望将所有积分的有效期延长一年。也就是说，如果你有在特定时间段内即将过期的积分，其有效期将延长一年。而在接下来一年中累积的任何积分，则应拥有两年的有效期窗口。应该如何对此建模？

一种方法是使用*事件*来表示规则变更。当应用正在物化事件时，它会将规则变更事件作为标记来调整自身行为。通过将业务规则变更纳入事件日志，可以丰富事件日志作为事实来源的价值。

这一想法的实现方式可以多种多样：事件可以非常具体，在物化过程中调用目标代码；如果该规则需要频繁调整（例如用于短期促销活动），事件也可以包含参数。

通过*掌控时间*，我们有机会在应用内部实现更丰富的功能。它还能让我们更深入地回答“发生了什么”和“为什么发生”的问题，因为数据和规则都可以被检视。
