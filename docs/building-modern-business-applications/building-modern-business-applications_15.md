# 12. 测试、监控与可观测性

前三章介绍了我们系统的关键组件：命令处理器、命令生成器和事件物化器。每一章都包含一个关于如何测试该组件的部分。隔离测试每个组件可以建立对该组件本身的信心。但这并不能转化为对组合后系统的信心。有必要端到端地测试所有组件协同工作。在本章中，你将学习我如何通过测试来建立这种信心。我将讨论从系统输出中理解其内部状态的价值，以及如何为我们的系统添加检测以实现这一点。你将了解为什么在设计阶段就必须考虑系统在生产中如何运行。让我们开始吧。

## 关键组件

在讨论如何端到端地测试系统的主要组件之前，我想先回顾一下这些组件以及它们是如何被隔离测试的。



### 命令处理器

命令处理器负责维护一类业务规则，即针对静态数据的规则。它接收一个命令作为输入，该命令是改变系统状态的请求。您提供的函数（称为变更器）会接收命令和当前系统状态，并负责生成一组描述该命令希望执行变更的事件。您提供的另一个函数（称为应用器）会接收事件和系统状态，并返回一个包含事件所描述变更的新状态实例。应用器和变更器都是纯函数，即没有副作用的函数。状态实例是一个不可变的数据结构，包含执行业务规则所需的状态。

这三项内容——应用器函数、变更器函数以及状态——是命令处理器中特定于应用程序的部分。命令处理器实现的其余部分则与这些函数内部的逻辑无关。

在第 9 章中，我描述了如何单独测试应用器、变更器和状态。它们也可以在不使用命令处理器本身的情况下，在测试中进行编排。

### 命令生成器

命令生成器负责根据变更请求（例如系统的 GraphQL 变更操作）创建命令。它通过将命令追加到命令日志中来提交命令进行处理。然后，它负责等待命令的处理结果，并将结果传达给调用方。

第 10 章中描述的测试主要围绕该组件的机械基础设施，而非创建命令的领域特定工作。我将领域特定逻辑的测试推迟到本章进行。

### 事件物化器

事件物化器维护用于响应 GraphQL 查询的有状态视图。

与命令生成器类似，我在第 11 章中描述了如何单独测试事件物化器本身。检查点基础设施和视图维护的编排可以使用与您的实际领域无关的简单事件进行测试。这让我们有信心在将物化函数用于领域事件之前，事件物化器本身能够正常工作。与命令生成器一样，我认为事件物化器中特定于领域的部分最好通过测试组合后的数据流来进行验证。

### 组合数据流

从写入到读取，变更请求以 GraphQL 变更操作的形式传达给系统。命令生成器将变更操作输入转换为命令，并执行与外部系统必要的通信。它将命令提交到命令日志，从而确定执行顺序。命令处理引擎一次从命令日志中消费一个命令。对于每个命令，它会调用您的变更器函数来创建该命令所隐含的一组事件。然后，它会调用您的应用器函数来推进命令处理器的状态。如果这两项操作都成功，事件将被追加到事件日志中，并且已处理的命令会连同其处理结果一起被记录。

命令生成器正在等待这个处理结果。如果命令失败，则返回详细的错误信息。否则，返回该命令生成的最后一个事件的序列号。

事件物化器正在从事件日志中消费。一旦事件被追加，它就会开始更新有状态视图。这与向调用方传达命令处理结果是并行进行的——当然，仅针对成功的命令，因为失败的命令不会产生任何事件。

为了观察其变更，调用方可以执行 GraphQL 查询，并将变更操作响应中的序列号作为所需最小视图版本的指示器。这为进行变更的调用方提供了写后读一致性。调用方还可以使用主动的 GraphQL 订阅来观察变更。在事件物化器更新视图后，他们将收到一个订阅通知。

为了以能够确信系统能够处理调用方请求的方式来测试系统，有必要执行这个组合后的数据流。



## 测试组合系统

我认为针对系统（或服务）的测试并未得到充分重视。过去几年中，“测试金字塔”的概念逐渐普及，这一概念源自迈克·科恩的著作《敏捷成功之道》。^(¹⁶) 金字塔的底部由相对更独立、速度更快的单元测试构成；顶部则是集成度更高、速度更慢的 UI 测试；中间层则由服务（或系统）测试组成。如果执行得当，系统测试能够为系统提供高水平的信心，并且性能良好。通过结合使用容器、在云端机器上轻松运行测试，甚至借助远程开发环境，速度上的劣势正在消失。我还认为，有时人们会认为任何比单元测试更复杂的测试都是缓慢且昂贵的。

我对系统测试的方法是：测试有价值的内容。我重视系统的外部 API 契约（在我们的案例中是 GraphQL 模式），以及用于满足 API 契约的任何基础设施。事件物化器维护着一个有状态的视图。如果使用另一个系统来存储该视图（例如数据库），那么测试应该使用该系统的实际实例。容器在这方面提供了巨大帮助，因为它们为管理这类资源的生命周期提供了标准化的契约。

GraphQL 通常通过 HTTP 交付。在进行系统测试时，我避免使用 HTTP，而是倾向于直接与我的 GraphQL 实现进行交互。GraphQL 实现与暴露它的 HTTP 端点之间的接口是可以独立测试的。这同样适用于事件物化器测试所采用的理念。该组件提供的功能最终与其执行的具体操作无关。事件物化器并不关心任何给定事件如何影响有状态的视图。HTTP 端点与 GraphQL 实现之间的接口是类似的，并且由于它位于系统的边缘，很容易被绕过。

我相信，组合系统——即满足 GraphQL API 所需的一切——可以作为一个整体进行测试；如果你需要用这种术语来表述的话，可以称之为一个单一的*单元*。我们的系统需要维护其外部契约，从类似于系统用户的角度进行测试，能让我们更有信心地认为测试结果将反映系统在生产环境中的行为方式。

从外部契约的角度测试我们的系统，其推论是：同样的方法也可以应用于任何需要与我们系统协作的其他系统。如果命令生成器需要查询另一个系统才能创建命令，那么只要该存根实现满足另一个系统的契约，我们就可以在测试中使用该系统的存根实现。我发现这种视角还能避免创建分布式单体架构——即由于服务间的相互依赖而需要同步发布多个服务的情况。如果系统间的契约定义不明确，甚至可能根本不存在，开发者可能会倾向于在测试中使用协作系统的真实实例。而一个定义良好的契约则允许创建替代实现，这是避免测试中出现此类耦合的一个促成因素。

对于我们的系统，一个实现应该允许实例化 GraphQL 接口、命令生成器、命令处理器、事件物化器以及任何其他必要的服务。这就是整个系统的精髓，在部署时即可通过 HTTP 暴露出来。

这个组合系统的实例足以测试组合后的数据流。这类测试也属于一种未被充分重视的范式：基于属性的测试。

## 介绍基于属性的测试

在向同事解释基于属性的测试时，我首先会让他们从已知的概念入手。大多数开发者都熟悉以下形式的测试：

```
@Test
void shouldCalculateScore() {
var customer = new TestCustomer();
var calculator = new ScoreCalculator();
assertEquals(10, calculator.scoreFor(customer));
}
```

这种类型的测试是基于示例的测试。这个名称的理由很简单：该测试是针对一个特定创建的客户来断言其分数。想想看，我刚才就为你提供了一个熟悉测试的示例。

要针对分数计算器的行为创建通用的断言可能很困难。那个单一的测试客户究竟能覆盖多少实现代码？

开发者通常会利用他们对实现的了解，构造多个示例来覆盖实现中的不同代码路径。这些示例共同建立起信心，让我们相信这些测试能够为系统部署后的行为提供保障。

这种方法存在一个局限性：随着被测单元输入复杂度的增加，所需示例的数量也必须增加。我发现这与大多数开发者希望尽量减少需要编写的代码量（尤其是在代码高度重复的情况下）的愿望并不契合。测试框架提供了一些技术来帮助减少重复，但最终，如果你有 100 个示例，代码库中至少会有 100 个*某种东西*。

正是这一点促使我转向了基于属性的测试。其前提很简单：不是测试特定的输入输出示例，而是针对所有可能的输入来测试单元的属性。在实践中，测试所有可能的输入通常是不可行的。支持基于属性的测试的框架会从一个随机种子值开始，并用它来生成一组输入。如果某个输入导致测试失败，可以将该随机种子值提供给框架，以便重现该场景供开发者修复。

关于基于属性的测试的文献通常会在示例中使用对称函数。对于任何对象 `A`，如果它被序列化为外部表示形式，然后再反序列化回来，它应该等于原始值。用数学方式表达就是 `A = deserialize(serialize(A))`。

我发现一个强大的方法是使用*测试预言*。测试预言是一个组件，它能够在不知道非测试预言实现方式的情况下，为给定的输入提供预期的输出。如果你的第一反应是这需要编写单元逻辑的第二个实现，那么你是对的！

当使用基于属性的测试并结合一个能为给定输入提供预期输出的测试预言时，你实际上就是在编写单元逻辑的第二个实现。我认为这是一件好事。要成为一个有效的预言，它绝不能依赖于被测试的实现。它是一个完全独立的实现。由于它只是用于测试的代码，因此不需要像生产环境中运行的代码那样健壮。它可以针对手头的任务进行简化。

可以将其视为开发者版本的复式记账法。就像会计师使用两本分类账来确保账目正确一样，作为开发者，我们可以拥有业务逻辑的两个实现。通过将业务逻辑实现两次，且彼此独立，它们可以相互交叉验证。这些实现将帮助你确保另一个实现正在按预期工作。



一旦创建了测试预言机，其复用成本极低。维护数十个（更不用说数百个）基于示例的测试用例将变得繁琐。基于属性的测试可以使用数千个随机生成的输入（或者，对于来自可枚举有限集的输入，可以使用所有可能的输入）来运行。其限制因素在于测试运行时间与通过额外输入所增加的信心之间的平衡——你可以根据自身环境和风险承受能力来调整这一平衡。

当基于属性的测试框架生成输入时，它们通常会提供用于生成特殊值的钩子。特殊值是指那些可能引发边界情况的值，例如零、空值，或有符号数的最小值和最大值。这些特殊值钩子确保它们以更高的概率被纳入随机生成的输入中。

能够用比我手动管理更多的场景来测试业务逻辑，对于建立对系统的信心极为强大。下一节将解释如何为我们的系统实现这一点。

## 使用基于属性的测试

我们将使用基于属性的测试来验证系统相对于暴露的 GraphQL 接口的行为。在最基本的层面上，在执行 GraphQL 变更操作后，验证查询和订阅是否包含预期的更改。一次只测试一个变更操作不太可能建立起所需的信心。用户工作流可能跨越多个变更操作。考虑一个管理发票的系统。可能存在以下变更操作：

*   创建发票
*   提交发票以供审批
*   批准发票
*   拒绝发票
*   归档发票

创建后，发票可以被批准或拒绝。被拒绝的发票可以重新批准。只有被拒绝的发票才能被归档。

我们真正想要的是让基于属性的测试不仅生成创建发票的输入，还要生成要执行的操作序列。

这项技术变得非常强大。如果存在一个实现边界情况，即发票在多次提交审批后归档失败，该怎么办？这个场景是你可能已经编写为基于示例的测试的吗？我猜想我可能会编写一个测试来验证被拒绝发票的归档，而没有考虑到拒绝次数可能是触发失败的因素。

通过让基于属性的测试框架生成一系列要执行的操作，我们可以模拟用户可能采取的系统工作流中的各种路径。在我自己的工作中，这揭示了在意外序列中出现的微妙边界情况，而这些情况如果仅通过用户的支持请求来拼凑，将会困难得多。

为了支持测试一系列操作，我们系统的基于属性的测试需要由一个**有状态的测试预言机**来支撑。当执行一个操作时，变更操作会被发送到系统以及测试预言机。该变更操作所隐含的变化将在 GraphQL 查询和订阅中可见，并且预言机将被用来预测这些值。

在编写基于属性的测试及其关联的预言机时，至关重要的是所有操作都必须是**确定性的**。这允许在预期失败时重放测试场景。例如，我遇到的一个陷阱是让测试预言机使用 Java 的 `HashMap` 来存储状态。Java 的 `HashMap` 不保证其内容的迭代顺序，而你可能在测试中依赖它来从映射中获取“第 n 个”条目。修复方法很简单，改用 `LinkedHashMap`，因为它确实提供了所需的保证。

系统本身也需要是确定性的，但我们已经在设计本身中很大程度上保证了这一点。变更操作由命令生成器处理，它最终会将一个命令追加到命令日志中。命令日志中命令的顺序定义了它们将被处理的顺序。在面对并发变更操作时，这就是建立顺序的地方。在命令生成器之后，所有处理都是有序的，无论是在命令处理器中消费命令日志、追加到事件日志，还是在事件物化器中消费事件日志。测试可以顺序执行变更操作，知道它们将生成由于并发请求而在部署时可能出现的各种顺序。

设计本身是确定性的。但命令生成器内部为了创建命令而咨询外部系统所执行的任何工作可能不是确定性的。你需要为你的系统解决这个问题，解决方案将根据具体工作内容而有所不同。一种可能性是将来自外部系统的响应视为需要它的操作的另一个输入。当测试执行该操作时，它可以使用一个侧信道与桩实现通信，告诉它下一个响应应该是什么。

在实践中，我使用基于属性的测试来验证我*认为*是系统快乐路径的内容。生成的要执行的操作序列*预期*总是导致成功的变更操作。我的理由有两个。首先，我不关心所有无效值的完整宇宙。如果创建发票有一个约束，即发票日期不能是过去的日期，那么测试所有可能的过去日期是否有价值？就个人而言，这样做并不会让我对系统获得额外的信心。我很乐意依赖我对检查如何实现的知识。其次，更实际地说，它简化了测试和预言机的实现。生成的输入被约束以匹配系统的规则，并且预言机不需要能够预测错误。如果来自系统基于属性测试的变更操作失败，诊断是直接的。输入值的生成没有被正确约束（这应该从错误消息中直接可见），或者测试发现了系统中的真正问题。

在这个级别进行测试的另一个好处是能够演进系统的内部结构。由于测试锚定在系统的**外部契约**上，因此有更多自由来演进系统的内部结构。你可以通过交叉检查你的更改与测试没有变化，来确保那些应该没有外部影响的内部系统更改确实没有产生影响。

基于属性的测试很强大，但它并不是满足所有测试需求的万能药。



## 使用示例测试

在上一节中，我描述了如何约束基于属性的测试，使其生成预期能成功的输入和操作，即所谓的“快乐路径”。这些测试通过 GraphQL 查询和订阅来验证变更操作所导致的状态变化。

我们系统的调用者不会只走快乐路径。系统必然会收到一些违反其业务规则的输入。当这种情况不可避免地发生时，应该将违反规则的原因传达给调用者，以便他们能够决定下一步的行动。

这正是我发现示例测试价值的地方。示例测试可用于确保系统拒绝无效的输入或操作。示例测试所执行的验证不仅应包括变更操作被拒绝这一事实，还应包括关于*为何*失败的返回信息。为了让调用者能够决定下一步行动，需要向他们清晰地传达失败的原因。示例测试中的断言应确保错误消息具有足够的描述性。

在系统生命周期的早期，示例测试也很有用。当添加第一个变更操作，并对照视图检查其结果时，使用示例测试是验证此项工作的最直接方法。即使对于像发票审批示例这样的多步骤工作流，最初将其编写为示例测试也有助于我尽早获得关于实现的反馈。一旦示例测试通过，我就会开始或增强系统的基于属性的测试。

当测试期望失败时，你需要理解其原因。可能是输入无效，也可能是系统内部存在逻辑错误。在开发过程中尽早遇到逻辑错误是理想的，但这并非总是可能。一旦系统在部署环境中运行，调试就不会像开发时那样直接。我通常会尝试将导致错误的场景重现为一个示例测试。为了有效地做到这一点，我可能需要了解系统的一些内部状态。

## 可观测性

可观测性是一个来自控制理论的术语，它描述了从系统输出推断系统内部状态的准确程度。推断越准确，系统的可观测性就越高。

对于软件开发人员来说，我认为这是通过观察系统已接收和返回的请求与响应、日志输出以及系统指标（如内存消耗或 CPU 使用率）来推断系统内部状态的能力。

请求、响应、日志和指标都是可以关联的。在最简单的情况下，可以使用时间进行关联。当系统存在并发请求时，这就开始变得困难，需要在每条日志语句上包含额外的元数据，以指明它对应哪个请求。系统指标是聚合数据，无法精确分解出某个特定请求消耗了多少内存或 CPU。

对此的一个解决方案是链路追踪。你可能在分布式追踪的语境中听说过链路追踪，它用于跟踪请求在分布式系统中的流转。在常用的术语中，单个请求就是一个*追踪*。一个追踪由一个唯一 ID 和多个跨度（span）组成，跨度是用于跟踪持续时间的容器。每个跨度都有一个开始时间和一个持续时间，即一个*时间跨度*。跨度被组织成一棵树，因此可以可视化依赖关系。如果服务 A 在处理请求时调用了服务 B，那么将会有一个代表调用服务 B 的跨度，它是代表调用服务 A 的跨度的子节点。

打个比方，链路追踪类似于包裹追踪。想象一个朋友通过邮政系统寄给你的包裹。你的朋友在寄出包裹时会收到一个追踪号。这类似于追踪的唯一 ID。当包裹在运往你处的途中经过各个设施时，每个设施都会有带时间戳的记录。最终，包裹会到达你的家门口。

链路追踪提供了在计算机系统中实现这种可见性的模型，我们可以将其应用于系统的内部。即使系统不是一个更大的分布式系统的一部分，链路追踪模型也极其有用。进入我们系统的每个请求至少是一个跨度，它代表了该请求的外部视图，包括持续时间。我们系统中的每个关键组件都是一个离散的步骤。链路追踪模型允许为每个组件执行的工作创建一个跨度，并将它们链接在一起。对于一个变更操作，追踪将包含一个代表整个请求的跨度，然后有代表命令生成器、命令处理器和事件物化器的子跨度。即使调用者在命令处理器完成后就会收到结果，事件物化器的工作也应该是追踪的一部分，因为它是相关的。

调用者将知道追踪的唯一 ID。当使用分布式追踪时，有机制可以将此上下文信息随请求传播；或者，如果系统独立使用追踪，则该值可以作为响应的一部分返回。如果请求导致错误或异常缓慢，可以查询追踪以获得更多洞察。这有助于系统变得可观测，因为追踪是系统的另一个输出。我们可以利用追踪来构建更完整的系统内部状态心智模型。我在第 10 章中提到的与命令生成器一起使用的信封模式，也可以用于跟踪与信封内容关联的追踪。



在实现系统组件时，请考虑在何处添加额外的跨度（span）来捕获有价值的工作片段。编写代码时应**大胆**添加跨度。我将其视为在封墙前为在建建筑铺设电缆——在建造初期完成这项工作，比后期改造既简单又经济高效。

链路追踪已实现商品化，针对您所选语言已有现成解决方案可供构建。一个常见功能是能够为每个跨度添加任意键值对（通常称为标签）。我同样鼓励您在此处**大胆**添加，尽可能多地附上当时可用的上下文信息。未来什么信息有用，很难提前预判。

您可以将链路数据发送给供应商，以便对其执行分析。每个跨度相当于一个计时器，因为它包含开始时间和持续时间。通过用链路对代码进行埋点，就无需单独捕获计时信息作为指标。链路追踪也更强大。指标是预聚合的，因此围绕一段代码的计时器最多只能记录其观测持续时间的直方图。如果查看计时器值时发现一个缓慢的异常值，您无法将其追溯到具体的慢请求。而使用链路作为输入，可以通过聚合链路的跨度来推导出计时器。如果观察到缓慢的异常值，可以查询输入链路来精确了解异常值在做什么。这也是预先添加标签变得有价值的地方，因为工具还可以将标签作为额外的查询维度。

链路已成为我观察已部署系统的首选方法。如果出现与意外行为相关的支持请求，并且我获得了与该交互相关的链路 ID，我就能准确了解输入是什么以及系统如何响应。

CPU 和内存使用率等指标确实有其用武之地。描述系统及其主机环境整体行为的系统级指标很有价值，仍应收集。了解系统的聚合行为是有益的，但并非所有必要数据都能从链路中构建。

链路在基于属性的测试中也很有用。每个任意选择的动作都可以启动一个新的链路，使得变异执行和用于验证的任何查询都能表示为各自的跨度。如果您能自信地使用链路调试测试失败，那么在系统部署后，您将更有信心使用链路诊断行为。使用链路调试测试失败也是早期了解领域中有价值的上下文标签的机会。

虽然链路提供了监控系统的机制，但我更倾向于将其称为可观测性工具，用于理解系统正在做什么。监控是一项独立的活动，可以使用链路来实现其他目标。

## 监控生产系统

我认为监控是确保系统按预期运行的行为。监控通常会伴随某种形式的告警，因为系统运维人员（也就是您）希望在系统出现异常时得到通知。

本书描述的系统适合通过单一进程来确保整体健康。我发现这很有益，因为有时系统会有多个监控进程生成告警。当系统有大量告警时，误报的可能性会增加，或者当出现问题时您会收到大量告警，需要您判断哪个指向真正的问题。

对于业务应用程序，理想的监控机制是执行与系统调用者相同的操作。不幸的是，对于业务应用程序，让监控系统更改系统数据是业务方不希望看到的。

我们的系统可以采用另一种方法。我们最终想知道的是命令是否正在处理、事件是否正在物化、查询是否正在被响应。这可以通过使用不影响系统业务信息的变更来实现。可以添加一个变更来递增计数器。计数器将生成一个命令，处理该命令将产生一个事件。事件物化器将物化该事件，更新视图中存储的值。可以使用 GraphQL 查询和/或订阅来观察更新后的值。

如果此过程及时完成，那么对系统的其他变更和查询也应该能完成（如果没有逻辑错误）。此过程完成的时间也是系统响应能力的健康指标。如果时间开始变慢，可以生成告警，作为潜在性能问题的早期预警。

这种类型的流程称为**合成监控**，因为我们正在模拟用户可能采取的操作。我们利用所有变更都流经同一流程这一事实，无需更改任何业务信息，同时确信这些流程能够处理任何更改业务信息的实际请求。

我发现合成监控与基于属性的测试配合得非常好。基于属性的测试确保系统内的逻辑按预期运行。合成监控则验证数据是否流经系统的“隐喻管道”。这再次使用了测试事件物化器时的相同范式：将基础设施测试与业务逻辑分离。如果基础设施本身可以得到验证，那么就有信心它能为依赖该基础设施的任何业务逻辑正常工作。

当系统监控器触发告警时，首要行动将是利用您可用的工具来诊断系统，例如可以查询链路或查看系统指标的系统。



## 练习使用工具

编写测试有助于确保系统按预期运行。监控则有助于确保系统正常运行并能响应请求。当监控最终触发警报，或调用方报告他们收到的错误时，你将需要依靠工具来诊断问题。

在*试图诊断问题*的过程中学习使用现有工具来诊断问题，是一种令人沮丧的体验，更不用说这会延长诊断时间或导致无法找到问题根源。

理想情况下，你应当知道如何使用这些工具并解读其数据。这需要学习，并且应该明确地进行。

在我目前的职位上，我们每周轮换一次，由谁担任系统问题的第一响应者。通常，一周内可能不会出现任何问题，从而形成一个非常直接的交接流程。如果出现过问题，则会进行讨论，将已了解到的背景信息分享给接任者。

为了学习我们的工具以及如何解读其数据，上一周负责的人会带领团队了解我们的工具，学习如何使用它们。这解决了多个目标。

首先，它建立了我所说的对系统行为的*环境理解*。通过查看系统的高级指标，如 CPU 或内存使用率，你将建立起对系统正常状态的认知。也许 CPU 使用率存在逐日重复的模式。你可能会了解到，在特定时间出现的小峰值并非异常。那么，当排查系统问题时，你就会知道同样的小峰值不太可能是问题的因素，因为它是一种常规现象。如果你没有建立这种对正常状态的环境理解，你可能会花时间调查一些最终被证明是误导性线索的问题。

其次，这也是深入探究潜在误导性线索的机会！以同样的 CPU 小峰值为例，第一次观察到时，深入探究其原因！采用与排查实际问题时相同的诊断方式，但没有实际问题带来的压力。这正是你和你的团队学习如何使用工具来回答有关系统问题的时候。也许 CPU 峰值与延迟峰值相关。然后你可以查看追踪信息，了解系统当时正在接收哪些请求。如果追踪信息没有提供足够的数据，你就有机会增强所捕获的数据。

在事故处理过程中学习如何使用工具会令人沮丧。意识到你的系统没有发出足够的信息来让你理解其行为，会延迟你诊断问题的能力。为了提供一个有弹性的系统，其操作者——也就是你——学习如何操作它至关重要。实践是必需的，而实践最好在受控环境中进行，而不是在事故这种压力更大的环境中。

测试将有助于最大限度地减少事故，但不可能消除意外情况。

## 总结

虽然系统的各个组件可以单独测试，并且这样做有其价值，但测试组合后的系统同样有价值。你已经了解了基于属性的测试以及如何将其应用于我们的系统。你还了解到示例测试也有其用武之地，尤其是在开发初期以及测试系统错误消息时。我解释了使用追踪信息来理解系统内部行为的价值，以及如何使用合成监控来检查系统在部署后是否按预期运行。最后，我介绍了一种有价值的实践：练习调查系统，以建立对其行为的环境理解，并确保系统发出足够的信息来诊断问题。下一章将根据我们的目标回顾系统，并展望实现所需的技术。
