# 7. 约束与原则

设计时间到了！本书的前两部分是背景铺垫。我需要确保我们对业务应用有共同的理解，并了解当前常用的模式。随后，我通过响应式宣言介绍了响应式系统，并阐述了为何要将业务应用构建为响应式系统。最后，我描述了对业务规则的分类，以及时间为何是一个重要（且有时被低估）的维度。

本书的这一部分将致力于我提出的业务应用设计方案。这一路径的第一步是引入一些对设计的自我约束，以及将指导架构的自主选择原则。让我们开始吧。

## 约束

我想对设计施加两个约束，它们都围绕应用如何向用户界面呈现自身。我将简要提及它们，然后进行更详细的解释。

第一个约束是支持并鼓励以支持数据实时更新的方式构建用户界面。第二个约束是使用 GraphQL 来驱动用户界面体验。

### 从 REST 到图

我在第 2 章简要提到了 GraphQL，它值得全面描述。让我们退一步思考它的发展历程，以此解释为何我认为它是一项重要技术。

可以将系统内部的数据视为一个巨大的图。URL 的层级路径部分被用作索引到这个概念图中的方式。这是构建数据 API 的基础。例如，端点 `/api/v3/customers` 会列出所有客户。要访问特定客户，其键值会作为另一个路径段附加，如 `/api/v3/customers/42`。客户有多个地址，这些地址可以在 `/api/v3/customers/42/addresses` 下列举。然后可以通过 `/api/v3/customers/42/addresses/67` 访问特定地址。

这种对数据图的层级索引有两个缺点：

1.  遍历需要多个请求。
2.  没有直接的机制来指示每个请求中需要哪些数据。

在我们业务应用的规模下，这些缺点通常不是障碍。然而，对于两家硅谷公司——Facebook 和 Netflix——来说，这些缺点足以促使它们同时开发替代方案。它们都基于使用路径层级和每个路径上的数据来形成系统数据的统一图这一理念。我将首先讨论 Netflix 的解决方案。

### Falcor

Netflix 开发了一项名为 Falcor 的技术。^(¹⁰) 它将统一数据图视为一个单一的 JSON 文档，并允许 JSON 的层级特性包含循环，从而对图进行建模。调用者随后提供一个“路径”（语法类似于但不完全等同于 JSONPath），以指示要加载图的哪些部分。这使得用户界面能够*精确地*在单个请求中获取所需内容。Netflix 的模型是从返回 JSON 片段的层级路径的明显演进，因为它将所有内容整合到一个端点中，允许用户界面指定超出 URL 路径范围、深入到 JSON 负载中的路径。

### GraphQL

Facebook 采用了略有不同的方法，即 GraphQL。它也基于系统是一个巨大图的概念。GraphQL 有一个用于描述图的模式。图节点是对象类型，每种类型由字段组成。一个关键的区别在于，GraphQL 中的字段可以接受参数。GraphQL 不应被视为纯数据图，而应被视为*函数*图。就像可以向 REST 端点传递查询参数一样，也可以对 GraphQL 类型的字段执行相同操作。

与 Falcor 不同，GraphQL 并不强加图是单一概念 JSON 文档的观点。Falcor 使用文档抽象作为机制，为其基于路径的访问提供图的“根”，而 GraphQL 在图中有一个特殊类型——`Query` 类型，其字段定义了图的入口点。

函数图的抽象还允许 GraphQL 拥有数据修改的解决方案，这是它保留的另一种 REST 行为。修改（用其术语称为变更）是通过请求另一种特殊类型——`Mutation` 类型的字段来执行的。

尽管 Falcor 和 GraphQL 都在 2015 年相隔一个月内发布，但 GraphQL 取得了更大的成功。我认为这是因为它在替代 REST 风格访问模式方面提供了完整的解决方案。



### 为什么选择 GraphQL

GraphQL 的地位与 SQL 类似。它定义了数据模型以及与之交互的方式，但其核心并未规定其他内容。其[规范](https://spec.graphql.org/June2018/) ^(¹¹) 侧重于如何定义模式以及为满足查询而应使用的执行模型。虽然有约定俗成的做法和库支持通过 HTTP 暴露服务，但规范底层的抽象性也允许采用其他传输方式。

通过定义模式，并鼓励 GraphQL 用户从定义模式开始，GraphQL 服务实现了自文档化。GraphQL 要求客户端*精确*指定他们希望检索的数据，没有对类型字段进行通配的机制。这使得每个字段的使用情况都可被追踪。当与内置的弃用机制结合使用时，就具备了支持模式演进的必要元素。

对于构建用户界面（这也是业务应用的目标之一），有许多工具可用于与 GraphQL 服务交互。在我撰写本书时，React 是创建 Web UI 的主流技术，其生态系统深受 GraphQL 影响。即使趋势发生变化，GraphQL 仍然适用。

### 实现实时性

我认为业务应用应具备与消费级应用相同的实时能力。我在第 2 章提到现状时曾提及这一点。`XMLHTTPRequest` 使得无需刷新页面即可加载数据，但它仍保留了请求/响应模式。还有两种额外的 Web 技术允许将数据流式传输到浏览器：服务器发送事件（SSE），这是一种仅推送的机制；以及 WebSocket，它允许建立基于双向推送的通道。这两种技术现已广泛部署并可供使用。然而，有效使用它们需要一个架构上适合此类实时交互的系统。

这是使用 GraphQL 的另一个原因。GraphQL 拥有第三种特殊类型，即 `Subscription` 类型。订阅字段返回一个值的*流*。当与 SSE 或 WebSocket 等流式传输结合使用时，GraphQL 可以作为查询、变更和实时更新的单一解决方案。

## 原则

在开始一个项目之前，我喜欢制定一套清晰可表述的开发原则。明确一套原则有助于为你的团队成员以及任何可能负责系统未来维护的人提供指导。作为原则，它们是努力追求的价值。与约束不同，它们是指导方针，而非硬性要求。我将首先列举这套原则，然后逐一讨论：

*   永不遗忘

*   消息驱动

*   读写分离

*   部分可用性

*   设计灵活性

*   模块化

*   可测试性

*   易于变更

### 永不遗忘

这个措辞源自一家我成长时期存在的软盘公司——大象内存系统（Elephant Memory Systems）。他们的标志是一张大象的脸，下方写着“永不遗忘”。这是对谚语“大象永不遗忘”的致敬，这个说法一直留在我心中。

正如我在上一章关于时间的讨论中所提到的，有一些技术可用于让系统保留先前状态。对于开发者来说，将所有代码变更记录在版本控制系统中是最佳实践。同样，对业务应用进行类似的架构设计也应当成为最佳实践。业务应用中包含的数据可能对业务成功至关重要，默认策略应该是追踪所有变更。

“永不遗忘”更进一步，因为它不仅仅是记录新状态的简单机制，而是记录变更背后的意图，记住“是什么”和“为什么”。

作为一项原则，我认为应该在项目开始时就将这一点纳入。除非在变更发生时就被捕获，否则这些信息将永远丢失。

### 消息驱动

这一原则直接取自响应式宣言（Reactive Manifesto）。我在第三章介绍响应式系统时讨论了其基础。同样，这一原则最好在项目开始时就被遵循。为未设计此类通信方式的系统改造消息传递机制可能颇具挑战。通过从一开始就采用消息驱动，系统便具备了利用宣言中其他特性（弹性、韧性和响应性）的基础。

这里有一个关键的控制方面。通过将系统中的内部通信具体化为消息，我们获得了对它们的控制权。消息成为可由代码操作的离散项。消息传递是“面向对象编程”（OOP）背后的原始思想之一。开发者通常并未意识到这一点，因为对象提供的封装特性往往更为突出。

我们希望在我们的系统中复用封装和消息传递的思想。交换消息的组件将比面向对象的领域模型粒度更粗。我们需要在实现语言提供的基础之上，建立消息传递的设施。并非所有语言都允许在对象之间动态处理消息，Ruby 和 Objective-C 是两个显著的例外。即使我们使用其中一种语言来实现系统，我们仍然希望在语言设施之上建立一种机制来操纵消息流。



### 读写分离

系统的读写访问模式通常并不对称。对于用户规模较小的年轻系统而言，这通常不会带来问题。用户数量少且运行时间短，意味着数据量和变更率都较低。

随着系统逐渐成熟，其中包含的数据量通常会随服役时间而增长。在我们提议的系统中，由于还需要存储历史状态，这一问题将更加突出。

我们的业务应用将为业务提供两大功能：数据录入和决策支持。数据录入的访问模式通常要求具备“读取自身写入”的能力。对系统做出更改的用户非常清楚应用程序现在应该显示什么内容，他们期望读取结果与自己执行的写入操作保持一致。这类活动在经典定义中被称为在线事务处理（OLTP）。

决策支持则有所不同。在此用例中，系统的用户通常不会修改数据。他们运行报表，并向应用程序提出复杂的聚合查询问题。由于不进行修改，用户不会注意到数据是否略微过时。所提出的问题也可能完全容忍过时数据，因为他们总是在回顾一个固定的时间段。想想每日运行的报表，只要数据在报表执行时是最新的，就满足了报表的要求。这类活动被称为在线分析处理（OLAP）。

在大型企业中，OLAP 活动通常跨业务部门或其下属分支所使用的系统进行。一个系统的数据会被复制到一个中央存储库，该存储库汇总来自其他系统的信息。

你的系统用户可能也有类似的需求，我发现这是一个常见的请求：应用内分析和报表。

当使用关系型数据库时，一个自然的解决方案是添加数据库的只读副本，并用它来满足这类请求。然而，一个在设计之初并未考虑此点的系统，在引入另一个数据库主机并恰当使用它时，可能会面临挑战。

因此，这一原则旨在通过早期解决这些问题来规避未来的麻烦。如果系统在设计时就为读取和写入规划了独立的路径，那么系统在此维度上就易于变更。初始实现很可能为所有功能使用单个数据库实例。通过提前引入逻辑分离，当未来需要物理分离时，对系统进行修改将更加容易。

### 部分可用性

我们的业务应用对业务至关重要，这也是业务对其进行投资的原因！它们需要在*业务需要时*保持可用。历史上，在互联网广泛普及之前，商业软件的使用通常与工作时间挂钩。虽然某些领域已经全天候运行多年，但其他领域则相对奢侈地仅在本地工作时间使用。

系统有规律的闲置期，为执行维护活动创造了一个天然的窗口。有了这样的机会，就有可能将系统完全离线以进行工作。随着应用程序开始拥有面向外部的组件，例如面向业务客户的用户界面，或与客户/供应商系统的 API 集成，将应用程序完全停止服务变得越来越困难。

作为一项原则，部分可用性是指能够仅将系统的*一部分*下线进行维护或升级。它也是指仅让*一部分*发生*故障*的能力。这是一种弹性形式，是响应式宣言所期望的特性之一。

部分可用性的粒度将取决于你的业务领域和应用程序的性质。存在一些横切性的粗粒度方面，例如对读取可用但对写入不可用。特定领域的方面则需要与业务协调，以确定故障模式的“爆炸半径”可能是什么。

部分可用性更常规的用途是围绕定期维护和系统升级。其最大价值体现在故障发生时。通过在系统中实现概念上的“隔舱”，只有系统的一部分会失效。这有助于在故障修复期间减轻运维人员的压力。处理关键系统的中断总是充满压力。遵循这一原则，可以将中断限制在受影响的子系统内，从而仅影响依赖该子系统的用户。这并非万能药，但在缓解处理中断人员压力方面可能价值连城。

对部分可用性的简洁描述是：能够在降级状态下运行，而不是处于非此即彼的“运行或停止”状态。

### 设计灵活性

软件中的灵活性是一个模糊的话题，因为对于什么是灵活的软件，并没有明确的定义。我对设计灵活性的定义是：系统设计的任何方面都应有多种实现选项。在实践中，我发现这意味着要考虑软件未来可能的发展路径，并尽可能*不排除*这些可能性。

未来，尤其是在软件项目中，是出了名的难以预测。与敏捷宣言一致，这是响应变化而非遵循固定计划。我们的方法是响应变化，而设计灵活性则关乎为未来保留选择余地。

随着系统成熟，有时做出会排除某些未来演进路径的决策是合理的。这是对设计灵活性的削减。此时，项目上积累的智慧将有助于最小化这些决策的影响。这不是一个可以轻易做出的决定，因为逆转这类决策可能代价高昂。然而，在某些情况下，这可能是恰当的行动方针。

在启动新项目时，尤其是在项目早期，为未来的可能性保留空间是有价值的。这就是这一原则的精髓。



### 模块化

“模块”是一个被过度使用的术语。除了软件领域，它还用于许多其他领域，例如货架、沙发，甚至建筑。在这些领域中，模块的粒度或特异性各不相同。然而，模块与整体的关系是明确的；模块是粗粒度的细分，可以通过包含或排除来重新组合，以产生新的变体。

在软件领域，微服务就是新的模块。它们是离散的功能单元，被打包成可独立部署的单元。不过，最后这一点并非必需。如今备受诟病的“单体”架构（即只有一个可部署单元）并不意味着系统的内部结构像一盘意大利面。盲目追随微服务趋势而不关注整体架构，会导致*分布式*意大利面。软件单元如何组合仍然至关重要。

解决之道是创建一个模块化系统，其中模块以指定的方式交互。模块的部署是一个独立的问题；一个极端是每个可部署单元对应一个模块，即微服务架构。另一个极端是单体架构，或者正如某些人所创造的“[模块化单体](https://twitter.com/gernotstarke/status/901496976612655106?lang=en) ^(¹²) 。”

模块化的原则是避免意大利面式的系统。独立于部署单元，一个系统必须被良好地分解为对代码和业务都有意义的模块。模块可以作为部署和开发人员工作并行化的一个维度。

通过将系统视为模块，然后让模块通过异步消息传递进行通信，将模块迁移到新的部署单元不会改变其与系统其余部分的交互模式。模块化与响应式系统是一致的。

### 可测试性

软件总会有缺陷。我继续在我编写的代码中犯错误，你也会如此。为了弥补这一点，我们测试我们的工作。测试软件是一项机械性的任务，非常适合自动化。可测试性是指编写软件，使得编写自动化测试变得简单直接，从而降低编写测试的门槛。

这个原则在其他方面有些模糊。可测试性*究竟*是什么？我发现测试驱动开发（TDD）是一种有助于我工作的机制。从测试开始，你就是你将要编写的代码的消费者。这种 API 的消费者优先视角有助于确定测试将覆盖的边界。这与模块化（理想情况下，测试应尽可能隔离到单个模块）以及模块本身的内部结构有着重要的相互作用。

这个原则也与响应式系统高度一致。发生消息传递的边界也是单个测试将覆盖的良好边界。总体目标是鼓励为我们的系统编写测试。

### 易于变更

我把最喜欢的这个原则留到了最后。这种“易于变更”的表述，是本书中反复出现的主题。我曾读到过一句俏皮话，可以转述为：

> *愿你的系统服役足够久，以至于被视为遗留系统。*

当“遗留”一词应用于软件系统时，往往带有贬义。从事遗留系统的工作不受欢迎的情况并不少见。这是为什么？我认为这与这个原则有关。在一个不易变更的系统上工作是令人沮丧的。导致这种情况的因素有很多，例如缺乏测试、业务规则交织、没有或很少进行模块划分。这样的系统不会遵循本章前面阐述的原则。

创建业务应用程序是业务方的一项投资。如果一个使用不到一二十年的应用程序需要重写，我认为这通常是系统架构的失败。但这并非必然。

除了为系统的长期成功奠定基础之外，这也符合敏捷开发的精神。通过强调与系统用户合作以满足他们的需求，并承认变化会发生，易于变更成为一个隐含的要求。

这种理念既是前述原则的一个方面，本身也是一项原则。我们希望我们的工作受到重视，并为业务的整体生产力做出贡献。这个原则就是为了确保在我们的系统预期生命周期内实现这一点。这个原则的阴暗面是过度工程化。*易于*这个词是经过精心选择的。这并不意味着每个变更都必须简单甚至直接。而是说变更的可能性已被考虑，并且实现变更的步骤不会与程序的基本约束相悖。

## 总结

本章为我们的系统设计奠定了基础。我声明了一些自我施加的约束，然后是一套指导设计本身的原则：

*   GraphQL API

*   实时 UI 更新

*   永不遗忘

*   消息驱动

*   读写分离

*   部分可用性

*   设计灵活性

*   模块化

*   可测试性

*   易于变更

牢记这些原则，本书本部分的剩余内容将逐步推进到我们系统的设计。下一章将通过介绍高级数据流并建立一些核心模块，更接近实现。
