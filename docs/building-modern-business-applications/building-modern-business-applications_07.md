# 5. 什么是业务规则？

业务应用是业务的真实数据源。为了有效完成这一任务，数据需要符合某种结构。这将允许围绕它构建流程和工作流。从分类上讲，数据结构的规范、数据如何修改以及数据变更时会发生什么，都被视为**业务规则**。在特定时间点，这些规则是具体的。随着业务的变化，其规则也会随之改变。本章探讨了我对规则的思考方式以及如何对其进行分类。通过理解它们与系统的关系，我们将知道在架构中应将其置于何处。

## 思考业务规则

拥有正确的规则至关重要。错误的规则会导致用户在系统之外、在其约束范围之外工作。由于业务的主要目标之一是让用户*实际*使用系统，因此规则需要准确反映业务的运作方式，包括各种例外情况。

那么，什么是正确的规则呢？必须与潜在用户沟通才能理解。这可能是你职责的一部分，或者在较大的组织中，这可能是产品经理职责的一部分。理解业务规则是收集需求的关键环节。

从正确的受众那里收集需求非常重要。如果只采访管理层，他们想象的工作方式与业务*实际*的运作方式之间可能存在脱节。意识到这种脱节很重要。人因工程学领域将这种现象描述为“想象的工作”与“实际的工作”。在创建应用时，我们正在编码第三种形式：“规定的工作”。如果你的组织进行事件分析和回顾，你可能已经接触过这些概念。新兴的站点可靠性工程学科（一种将软件工程原则应用于运维的实践）在进行无责故障事后分析时，一直在使用这些概念。

作为应用开发者，我们控制着应用所能实现的功能。我们正在规定工作必须在系统上下文中如何执行。收集需求的行为，就是理解工作应如何完成（如想象的那样），并将这些流程在系统中形式化的过程。开发人员对精确性的执着在这一过程中可能是有益的，可以提出必要的问题来理解可能的边界情况。

在确定系统规则时，务必记住，形式化过程中存在风险因素。添加一个必须为真的约束，就排除了违反该约束的数据存在的可能性。这代表了一种风险。如果业务需要改变该决策，变更能够多快安全地完成？能够适应这种类型的变更正是我们的目标之一。

“必须始终为真”的事物与“可能应该为真”的事物之间存在差异。前者是规则，后者是指导方针。在我们讨论如何对规则进行分类时，请牢记这一点。

## 分类

在我的职业生涯中，我观察到所谓的“业务规则”中存在一些共同主题，并利用这些主题构建了以下分类。

### 静态数据

最终，数据存储在*某个地方*。这些数据必须满足一些要求。

如果你在关系数据库中创建过表，你就遇到过静态数据的规则。任何在数据库模式中表达的约束都属于此类。这些约束包括可空性、唯一性和外键。某些数据库允许对列表达更复杂的约束，例如为数值定义固定范围，甚至跨表列进行约束，例如“开始日期”必须在“结束日期”之前。在 SQL 中，这些是`CHECK`约束。

这些声明式约束仅限于单行，这限制了它们所能强制的内容。想象一个跟踪小部件的系统。如果有一个表示小部件可用库存的表，可能存在所有行的总和不能低于零的要求。如果每一行代表一个添加或移除小部件的操作，该表可能有一个约束，例如`CHECK (count <> 0)`，以确保每一行的数量非零，因为有效的操作必须移动一定数量的小部件。为了表达库存永远不能为负，编写`CHECK (SUM(count) >= 0)`似乎很直接。然而，据我所知，没有任何广泛使用的关系数据库允许以声明方式指定这种类型的约束。约束仅在单行上下文中检查，而不是整个表。表级或跨表约束需要使用触发器来调用存储在数据库中的过程代码。

这种从声明式约束到过程式约束的转变意味着这些约束通常是在数据库外部实现的。随着测试驱动开发，以及更广泛的单元测试和持续集成变得越来越流行，自动化测试的能力对开发者来说更有价值。数据库内的过程逻辑更难测试，因为它需要运行中的数据库软件实例。在 2010 年代中后期 Docker 普及之前，对于大多数团队来说，实现这一目标的努力通常是不可行的，时间最好花在其他地方。对于 JVM，有一些嵌入式数据库引擎可用于测试，但它们与生产环境中必须使用的引擎确实存在差异。这促使开发者在代码中依赖数据库功能的最低公共标准。

如果约束仅在应用代码中强制实施，那么从哲学和实践角度来看，它是否真的始终为真就存在疑问。为了保证静态数据的约束得到强制实施，所有对数据的修改都必须通过强制机制。如果该机制不在数据存储本身内部，则必须仔细保护对数据存储的访问，以防止绕过应用直接与存储交互的修改。

在应用代码中强制实施约束确实可以更轻松地管理随时间变化的约束。如果约束被放宽，现有数据无需更改，因为它已经满足更宽松的约束。如果约束被收紧，例如将可选字段变为必填字段，则必须对违反新约束的现有数据做出决策。这可能是选择在应用代码而非数据库中强制实施约束的一个考虑因素。下一章将更深入地探讨时间及其与业务规则的关系。



*静态数据*规则的另一种形式是带有渐进状态的工作流。考虑一个正在经历工作流的实体，其数据可能需要满足不同要求才能进入下一步。想象一个允许定制自行车订单的系统。开始下单时，不需要任何必填信息。你启动了流程，做了一些选择，但随后想保存以便稍后继续。应用程序可能会将其作为工作流的一部分提供，保存你的进度。为了稍后保存订单，需要额外的必填信息——你的电子邮件地址。定制过程中有五个决策点，你可以任意选择数量，同时反复保存订单以便稍后处理。然而，要进入“准备下单”状态，这五个决策点中的每一个都必须做出选择。随着自行车订单实体在工作流中移动，数据的完整性要求是渐进的。

在工作流示例中有一个细微的区别：即达到下一个“准备就绪”状态所需的要求，与完成工作流下一个状态可能代表的内容之间的分离。我们的自行车订单工作流可能会区分“准备提交”和“已提交”。在工作流中，通常存在与外部系统的协调点，即副作用。

### 副作用

应用程序通常不会孤立地运行。我们的业务应用程序几乎总是会与业务本身交互。在充当事实来源和自动化工作流时，系统内数据的更改会触发其他活动。我将这类活动统称为*副作用*。

这个术语借用于计算机科学，指的是修改其局部环境之外状态的函数。考虑以下函数：

```
private int multiply(int a, int b) {
var result = a * b;
System.out.println(a + " * " + b + " = " + result);
return result;
}
```

向控制台打印输出就是一种副作用。将这个示例映射到更大的应用程序中，我们的系统相当于这个函数，而与其他系统的交互就是副作用。

这些是发生在数据*更改之后*的任务。你会在应用程序需求中识别出它们，因为它们通常采用以下形式：

```
如果/当 那么 
```

具体来说，它们可能会执行以下操作：

*   用户注册后发送欢迎邮件

*   订单创建后，将信息发送到另一个系统进行履行

*   付款批准后，将其发送给银行

在更改系统数据*之前*与其他系统交互不是副作用，而是工作流的一部分。这种区别很重要，因为执行该交互不一定是你系统的责任。从实际实现的角度来看，它可能由系统执行，但如果发生在其他地方，工作流的整体行为也不会改变。一个工作流可能（而且很可能）会跨越多个应用程序。在考虑*你的应用程序*的规则时，这里只考虑它需要发起的副作用。

在实现系统时，将副作用与使前置条件成立的数据修改分开非常重要。这看起来可能是一个直接的实现，但故障模式是不可取的。如果在确保更新后的状态已成功写入数据库之前，你将订单的付款状态更新为“已批准”并将其发送给银行，那么如果数据库更新失败会发生什么？

我对副作用的理想实现是将其建模为状态机。你的应用程序可能是一个更大的业务工作流的一部分，理想情况下，该工作流也被建模为跨越多个应用程序的状态机。就像响应式系统组合一样，状态机也是如此。每个副作用都可以使用相同的基础模式，其价值在于如何管理故障。通过使用状态机，应用程序可以跟踪其成功执行副作用的尝试，从而允许进行适当的错误处理。

状态机应明确跟踪执行副作用的尝试的开始。如果在执行过程中发生故障，下一步很可能不仅仅是重复初始操作。如果我们开始发送付款但在过程中遇到故障，重试可能不是合适的操作。再次以订单批准后发送付款为例，如果发送付款失败，下一步操作将取决于我们发送付款的系统的行为。如果接收系统是幂等的，则可以重试原始操作。我们可能不会收到“成功”响应，而是收到一个表示“已处理”的“失败”响应，这表明我们的初始尝试实际上是成功的，尽管我们将其视为失败。在第一次尝试时收到“已处理”将是意外的，并且是一种错误状态。如果接收系统不是幂等的，则需要使用不同的机制来确定下一步应该做什么。

作为副作用执行的操作应尽可能具有幂等性。在实践中，可以通过要求调用方每次调用时提供唯一标识符来实现这一点，从而使接收方能够识别重复请求。



### 可推导数据

应用程序中的所有数据并非同等重要。延续上一节的例子，小部件的总数必须始终大于或等于零。业务规则所检查的值通常也会以某种方式展示给用户。在这个例子中，它就是“可用小部件”。

一种直接的实现方式是：在必要时重新计算总数，将其持久化，并在用户界面中使用这个持久化的值。这是一种能够满足业务需求的有效方法。

这个例子中隐含了一些相互交织的关注点，即使实现方式将它们绑定在一起，我们也应该独立地思考它们。这里涉及两条业务规则：

1.  对*操作*的约束，即其聚合结果必须满足的条件。
2.  在用户界面中使用该聚合结果进行展示。

这种区别可能很微妙，但我认为它很重要。对总数的约束，并非是对作为独立数据项的总数的约束。当添加一个新操作，并需要确保满足第一条规则时，你如何看待这个操作？是：

*   添加操作，重新计算/更新总数，然后检查它是否满足约束？

还是：

*   获取先前的总数，检查添加新操作是否会导致违反约束，然后再添加操作？

在前一种方式中，操作被添加时并未考虑约束检查，而是将推导出的总数视为约束所施加的对象。在后一种方式中，检查是针对新操作的，使用先前的总数作为一种优化手段，而不是每次都重新计算。

虽然第一条规则是对聚合值的约束，但这并不意味着必须存储该聚合值。这就是区别的本质所在。聚合值是*推导值*的一个例子。

在为系统构建数据模型时，区分构成模型基础的最基本事实，以及在此基础上构建的衍生事实，这一点至关重要。

关于对静态数据的约束，提及推导值可以将其视为一个“提取出的变量”，是一种无需每次都重复其推导来源就能讨论该推导过程的方式。如果推导值*仅*用于对静态数据的约束，那么它在系统中实际上并不作为一个数据点存在。

如果推导值*也*用于展示目的，那么推导它的行为就是我所说的*展示逻辑*。这是业务规则的第三种分类。展示逻辑是对系统数据进行的一种**非破坏性**转换。非破坏性这一点至关重要。用数学术语来说，该逻辑应该是一个*纯函数*，对于相同的输入返回相同的结果。

如果展示逻辑遵循这一特性，即作为数据的非破坏性转换，那么存储其结果就不是必需的。给定输入和支持该转换的代码版本，可以得到完全相同的结果。存储推导值是一种优化，而非必需。

展示逻辑的另一种变体是使用来自其他系统的信息。有时这会非常直接，即原样展示来自其他系统的数据。其他时候，则是对你系统中的数据进行转换，并将来自其他系统的数据作为其输入之一。在金融系统中，货币兑换就是一个很好的例子。可能存在另一个提供汇率的系统，而使用来自该系统的汇率将你系统中的数据以另一种货币展示出来，就是展示逻辑的另一种形式。

将某物命名为“展示逻辑”，并不意味着该逻辑存在于我们应用程序的用户界面代码中。出于性能原因，应用程序可能会持久化展示逻辑的输出。这应被视为对结果的缓存，而非系统的基本事实。如果在你的用例中这个类比不成立，我会质疑它是否真的是展示逻辑，或者是否应被视为一个基本事实。

## 总结

这些对业务规则的分类，有助于我们思考应用程序模型的属性。本书并不描述如何定义你的模型。相反，我关注的是如何着手实现它。实现过程会为设计提供一些反馈。我希望，根据这些类别来思考你的业务规则，将有助于澄清你的设计。

业务规则可以归入以下类别之一：

*   对静态数据的约束
*   数据变更时的副作用，例如启动流程
*   展示逻辑，包括推导和聚合

这些类别描述了规则与系统内数据的关系。这将在后续内容中发挥作用，因为它可以指导我们在设计的哪个位置放置实现。我们将在本书的第三部分讨论这一点。在此之前，还有一个先决条件：时间管理。时间与业务规则有着关键的交互作用，因为规则会随时间变化，它们与数据的关系也需要反映这一点。
