# 2. 企业级 Java、微服务与云

多年来，软件应用开发已发生显著变化。编写、测试、交付和部署代码的整个流程经历了快速变革，最终迎来了当前的云原生软件工程时代。Java 平台也逐年演进，成为现代应用开发的强大平台。

本章将带你了解云原生应用开发，通过回顾 Jakarta EE 应用部署的历史、加速云采用率上升的一些因素、通过 12 要素应用介绍云原生软件开发，最后探讨 Jakarta EE 在云原生软件应用开发世界中的角色。

## Java 2 企业版（J2EE）的诞生

Java 编程语言最初名为 Oak，于 1995 年 5 月由当时的 Sun Microsystems 公开发布。“一次编写，到处运行”的承诺立即吸引了开发者，加之当时 Netscape 宣布在其同名浏览器中支持这种新语言，为 Java 开启了通往应用开发主导地位的征程提供了必要推力。那时，Web 浏览器已开始让更多人接入互联网，企业也开始寻求为网站用户提供更多交互性的方法。

Java 编程语言最初是为创建手持娱乐控制器等家用设备应用而开发的。然而，这并未达到预期效果。借助当时 Netscape 浏览器的内置支持，该语言逐渐转向通过 applet 和 JavaBeans 结构成为客户端开发语言。随着时间的推移，数据库连接 API——Java 数据库连接（JDBC）被加入，以满足应用中数据访问的需求。

易用性、支持 Web 以及自动垃圾回收是推动 Java 在企业间普及的关键因素。意识到该语言可用于服务器端应用开发后，Sun 决定利用这一趋势以及新兴的应用服务器市场，通过引入 Servlet 规范来抢占先机。Servlet 规范的理念是在服务器上运行一个 Java 应用，客户端可向其发起调用，并接收处理后的数据作为响应。这些数据可以是 HTML 或浏览器能处理的任何内容。

与此同时，IBM^(⁵¹) 发布了一个独立项目——企业级 JavaBeans 规范，旨在通过自动提供大多数企业应用通用的服务（如安全性、事务、可扩展性和远程调用能力）来简化 Java 服务器端开发。EJB 规范还支持通过远程接口对业务方法进行远程调用。凭借这些多样化的能力，Java 成为开发 n 层架构应用的绝佳语言，并被许多大型企业采用。

1999 年，Sun 最终推出了一个全新的、统一的平台，完全专注于开发大规模、分布式的企业应用。该平台将所有作为 Java 向服务器端编程语言过渡过程中开发出的分散规范整合到一个“伞式”规范中。最初由 IBM 发布的 EJB 规范也作为 EJB 1.0 被纳入这个新平台。这个新平台被称为 Java 2 平台企业版，即 J2EE。其初始版本为 1.2。

J2EE 构建于 Java 平台之上，旨在运行于实现该规范的应用服务器上。这意味着理论上你可以一次编写应用，然后在任何实现该规范的服务器上运行。该平台也借鉴了分布式企业系统领域的先前工作，如公共对象请求代理体系结构（CORBA^(⁵²)）。为确保应用服务器遵守规范并能自称是经过认证的 J2EE 运行时，Sun 引入了一项许可计划，其中部分要求应用服务器必须通过兼容性测试套件（CTS）。



正如我们之前提到的，当时的应用服务器市场尚处于萌芽阶段，分析师预测它将发展成为一个价值数十亿美元的产业。这促使众多供应商涌入市场。随着 J2EE 的发布，其中许多供应商都去申请了 J2EE 认证。这些供应商包括 BEA Systems、Nokia、Compaq、IBM、Oracle、SAP、Sybase、Macromedia、Hitachi 和 TIBCO Software。所有这些公司都提供了可以部署 J2EE 应用的应用服务器。这些服务器也都是专有的、本地部署的。值得注意的是，Sun 公司本身也有一款应用服务器，最初名为 iPlanet，后来更名为 Sun ONE。然而，这个市场上的两大主导供应商是 IBM 的 WebSphere 和 BEA 的 WebLogic。

还有一个新兴的开源项目，名为 Enterprise JavaBeans Open Source Software (EJB-OSS)，它实现了 J2EE 规范中的 EJB 应用程序编程接口（API）。该项目逐步实现了该规范，但并未获得 J2EE 认证。该项目后来也改名为 JBoss。这款开源应用服务器在当时的企业环境中取得了重大进展。

## 单体架构

当时主流的应用开发模型是单体架构。这种开发范式将应用构建成一个巨大的“捆绑包”，所有层和组件形成一个单一的、内聚的部署产物。这种范式有其优点和缺点。修改应用的任何部分都意味着需要重新部署整个应用。

一个典型的 J2EE 应用会以 Web 应用归档（WAR）或企业应用归档（EAR）的形式部署。WAR 文件格式用于部署使用规范中 Web 概要文件开发的 J2EE 应用。WAR 归档可以部署到 Servlet 容器或任何实现了更大 J2EE 规范中 Web 概要文件规范的运行时中。EAR 归档用于部署使用 J2EE 规范全系列功能开发的完整 J2EE 应用。

J2EE 平台非常适合开发单体应用。这主要是因为在该平台开发和发布之时，以及之后的多年里，单体架构是当时最主流、最流行的应用开发范式。因此，该平台的发展是为了满足使用这种范式开发软件的企业需求。

单体应用开发架构的优点如下。

### 开发速度

鉴于应用的每个组件都在一个单一的代码库中，让多个开发团队同时工作，实现不同的特性和功能是很简单的。加上像 Git 这样的版本控制系统，单体应用开发变得简单直接。

### 更易上手

让新开发者加入一个使用单体架构的项目是很容易的。因为所有内容都统一在一个内聚的、单一的代码库中，新开发者很容易尽快上手并提高生产力。

### 简化测试

由于所有内容都在一个单一的部署产物中，为任何给定单体应用的不同组件编写单元测试和集成测试通常更容易。

### 简化部署

由于运行单体应用所需的一切都打包在一个单一的归档文件（WAR 或 EAR）中，部署相对简单：将生成的归档文件放入任何给定运行时的部署文件夹即可。

### 简化扩展

由于部署模型更简单，所有内容都捆绑在一起，通过在不同服务器上运行多个副本，并使用负载均衡器将流量路由到这些副本，来扩展给定的单体应用相对简单。

这些优点，以及其他一些因素，使得单体架构成为大多数企业的直接选择。可以以更可控的速度添加功能，可以扩展开发团队，并且应用本身可以轻松地通过新变更进行部署，所需工作量很小。

## 不断变化的开发格局

然而，随着互联网连接设备的激增、软件消费从本地安装向云托管的普遍转变、像 Gmail 这样的 Web 应用的流行、第三方托管部署基础设施（例如 AWS）的兴起、REST 架构的流行、API 生态系统的兴起、外包部署基础设施的成本日益降低、提供竞争性产品和服务的公司数量激增，以及世界日益互联，这些都意味着软件开发不能再像以前那样了。让我们粗略地看一下这些关键因素。

### 互联网连接设备的兴起

从 21 世纪初开始，互联网的普遍流行以及互联网可访问性的普遍扩展，随之而来的是互联网连接设备的增加。像 BlackBerry 和 Nokia 这样的品牌开创了在手机上使用互联网的先河，而 iPhone 的推出则极大地普及了“口袋里的互联网”这一概念。

### 软件消费的普遍转变

互联网连接设备的兴起也标志着用户消费软件方式的改变。在计算的早期，软件被内置到计算机的硬件中。这种范式不断演变，直到连接设备兴起。互联网的天然特性意味着软件不一定非要安装在消费设备上才能使用。

### Web 应用的流行

随着连接设备的兴起和软件消费模式的普遍转变，Web 应用也随之流行起来。像 Gmail 这样广受欢迎的应用开启了一个时代，Web 应用成为消费软件的默认方式。这一转变意义重大，因为它意味着企业需要转变其将软件交付给最终用户的方式。

### 第三方托管基础设施的兴起

大约在上述几点发生的同时，第三方托管基础设施的兴起标志着成熟企业和初创企业应用开发的新时代。亚马逊云服务（AWS）的出现，以其提供的大规模云基础设施，意味着企业现在可以将创建、测试和持续部署应用所需的一切完全外包给外部服务提供商。因此，这必然导致了应用开发方式的转变。

### REST 架构的流行

由 Roy Fielding 博士在其博士论文中首创的表述性状态转移架构^(⁵³)大约在这个时候开始流行起来。这种架构提出了一种方式，让应用可以通过可预测的接口（以端点的形式）来暴露和消费信息。

### API 生态系统的兴起

API 生态系统也大约在这个时候开始蓬勃发展。在这个生态系统中，其他公司通过互联网以 API 的形式提供专业化服务，例如，一个用于管理货币汇率的 API 服务。企业现在可以将正交功能外包给专门从事该特定领域的第三方公司。所有这些都通过 RESTful 端点来消费。

### 外包基础设施成本降低

专门从事托管基础设施的公司的兴起意味着，将开发到部署所需的整个技术栈外包出去，通常比内部托管更便宜。



### 竞争性企业的激增

上述因素以各种方式共同作用，导致了任何特定市场中竞争者数量的增加。更大的潜在用户池、通过互联网交付软件的近乎公平的竞争环境，以及相对更便宜的基础设施托管服务，所有这些因素共同为更多企业进入任何特定市场创造了平等的基础。

## 微服务的兴起

上述几点都以某种方式成为促使软件开发行业重新审视软件开发、测试和部署方式的因素。企业将应用程序部署到他们主要拥有和管理的服务器上。大多数企业拥有完全专用的内部团队，负责管理整个应用程序开发生命周期所需的全部服务器堆栈。这意味着企业需要操心为其应用程序采购、安装和管理服务器堆栈。对于较小的公司和初创企业来说，这被证明是相当昂贵的。

当某个应用程序的规模和复杂性增长时，单体架构的优势也带来了其他随之而来的挑战。一个应用程序可能会变得非常庞大和复杂，以至于添加最微小的更改都需要进行大量的自动化和手动测试，以确保没有负面的、非预期的副作用。即使是最简单的更改也意味着需要重新运行整个应用程序部署管道。因为所有东西都捆绑在一起，最简单的错误也可能导致整个应用程序崩溃。

由于上述所有因素的不同组合，软件开发行业开始关注另一种软件开发范式。这种被称为微服务^(⁵⁴)的范式，提出了一种将前述所有因素都考虑在内的软件开发方式。这个想法源于 Unix 哲学，即通过将完整的应用程序分解为独立的单元来编写，所有这些单元都通过一个完善且可预测的接口进行通信。

微服务范式建议通过将软件分解为更小的服务来编写，其中每个服务负责整个应用程序的单一功能或特性，每个服务彼此独立，所有服务通过 RESTful 端点上的 HTTP 协议进行通信。这些服务可以由编排服务在给定的服务器集群上独立部署和管理。微服务架构中还隐含着一个事实，即应用程序和整个开发过程都在“云”中执行。

微服务架构一经推出便大受欢迎，因为它解决了许多超大型单体应用中的痛点。其优势如下。

### 通过分解管理复杂性

由于应用程序的每个单元都是作为独立的子应用程序开发的，因此随着其复杂性的增加，管理整个应用程序开发相对更容易。

### 更易于维护

由于整个应用程序由独立的单元组成，因此可以轻松添加新功能和修复错误，而不会破坏整个应用程序。给定服务中的任何问题都仅限于该服务。

### 更快的开发速度

由于开发人员不需要了解整个应用程序，因此使用微服务架构进行开发相对更快，因为每个开发团队大多只需要了解他们正在开发的服务。因此，应用程序可以更快地发布。

### 可扩展性

通过微服务架构构建的应用程序可以相对容易地进行扩展，因为可以将资源定向到仅扩展那些需要扩展的部分或服务。

### 弹性

在生产环境中，作为一套微服务开发的应用程序不会因为一个服务宕机而整体崩溃。为对整个应用程序至关重要的服务构建有针对性的弹性也更容易。

### 更容易采用新技术

微服务架构的核心特性——服务通过 RESTful 接口暴露其功能——意味着这些功能的实现与面向客户端的接口完全分离。这意味着单个服务可以更快地采用新技术，只需处理实现细节，而不必影响面向客户端的接口。

## 云原生开发

微服务架构中隐含了云原生应用程序开发的概念。云原生计算基金会（CNCF）将其定义为：“云原生技术使组织能够在现代、动态的环境（如公有云、私有云和混合云）中构建和运行可扩展的应用程序。容器、服务网格、微服务、不可变基础设施和声明式 API 体现了这种方法。这些技术使系统具有松散耦合、弹性、可管理和可观察的特性。结合强大的自动化，它们允许工程师以最小的辛劳频繁且可预测地进行高影响力的更改。”

CNCF 的定义概述了现代软件开发的方式。它旨在优化整个开发过程，使其尽可能高效，并能够为客户交付尽可能多的价值。云原生原则旨在帮助企业管理现代应用程序开发的复杂性。在一个任何应用程序都可能被无数设备和方式消费的时代，需要一套可适应的模式来提高成功开发流程的几率。

云原生开发背后的理念是能够更快、更便宜地向客户交付弹性的应用程序。随着用户对应用程序提出更多要求，企业必须找到一种有效的方式，以更低的成本向用户交付价值。云原生开发是企业实现敏捷开发流程的一种方式，从而为用户提供弹性、健康、安全和可靠的应用程序。

云原生应用程序依赖于云服务提供商。正如 CNCF 对云原生的定义中所述，企业可以利用三种云服务。

### 公有云

公有云是由第三方提供商管理的一组计算服务，这些服务可通过互联网供不同的客户访问。一个典型的例子是来自亚马逊云服务（AWS）的通用计算服务套件。

### 私有云

私有云是指由单个企业拥有的一组计算服务，可以由该企业在其自己的数据中心管理，也可以由第三方服务提供商收费管理。在云托管计算服务普及之前，这是应用程序部署的主要模式。企业会建立自己的计算基础设施，由内部员工管理。现在有许多第三方私有云提供商，例如 AWS 虚拟私有云。

### 混合云

混合云是公有云和私有云的结合。它通常提供更大的灵活性和更多选择。

## 12-Factor 应用^(⁵⁷)

12-Factor 应用是一套被广泛接受的、用于开发云原生应用的指导方针。它提出了一些因素，可以帮助开发人员创建高度优化以适应云环境的应用程序。这些因素如下。



### 代码库

这一要素要求应用程序的每个服务或组件拥有独立的代码库。通用功能或共享服务应放入库中，通过依赖管理系统（例如 Maven）引入到各个代码库。给定的代码库可以部署到部署管道的不同阶段。

### 依赖关系

每个微服务应声明、打包并使其依赖关系与其他服务相隔离。任何两个服务都不应拥有共同的依赖关系，以免一个服务的依赖变更影响到另一个服务。所有服务在依赖关系方面应完全独立。

### 配置

微服务的配置应存储在外部环境中，以便这些配置可以在不同的部署阶段之间进行更改，而无需修改代码。

### 后端服务

后端服务是指某个微服务在其正常运行过程中，通过网络需要并消费的任何服务。12-Factor 应用应将后端服务视为可通过可寻址 URL 访问的资源。目标是使应用程序与后端服务解耦，以便后端服务可以被另一个服务替换，而不会对微服务产生任何影响。

### 构建、发布、运行

构建、发布和运行阶段应完全分离。每个阶段都应使用唯一标识符进行标记，以便能够回滚到该特定点。像 Jenkins^(⁵⁸) 这样的现代发布工具有助于实现这一点。

### 进程

12-Factor 应用应作为独立的、无状态的进程运行，彼此不共享任何内容。任何服务都不应基于运行时自身生成数据之外存在某些数据的假设来运行。

### 端口绑定

每个微服务应绑定到特定的端口，并在此端口上监听请求。任何两个服务不应绑定到同一个端口。

### 并发

当需要扩展时，每个微服务应进行水平扩展而非垂直扩展。当每个服务作为无状态进程运行时，水平扩展很容易实现。

### 可处置性

每个微服务应是可处置的。给定的服务应能在不丢失数据的情况下随时被处置。将微服务创建为无状态进程意味着它们可以随时被销毁而不会产生任何副作用。

### 开发/生产环境一致性

开发、预发布和生产环境应尽可能相似，但不共享任何数据。由于 12-Factor 应用设计为持续部署，因此尽可能减少各个阶段之间的摩擦至关重要。

### 日志

日志可以洞察每个运行中微服务的内部情况。因此，每个服务只需关注将日志写入可用的默认输出流。然后可以使用日志聚合工具收集和分析这些日志，以便在必要时采取纠正措施。

### 管理进程

一次性管理任务（如数据库迁移）应作为一次性进程运行，最好在与应用程序运行环境相同的环境中执行。

## 容器与云

云原生应用的核心在于能够将应用程序打包成镜像并以容器形式运行。像 Docker 这样的容器化工具和像 Kubernetes 这样的编排工具，使得将应用程序及其运行时打包并部署到服务器堆栈变得容易，这些服务器堆栈会自动配置以服务请求。这种抽象使得任何给定的 Java 应用程序都可以在能够将打包镜像作为容器运行的任何地方运行。

正如本章前面所讨论的，企业级 Java 应用程序被部署到实现企业级 Java 规范的运行时中。容器化使得企业级 Java 应用程序越来越有可能作为容器化应用在公有云、私有云和混合云中运行。应用服务器供应商现在必须在如何使其运行时易于打包并作为云原生工件运行方面展开竞争。

## 企业级 Java 与云原生开发

J2EE 及其后继者 Java EE，尽管最初是为开发多层、大型、单体应用而设计的，但也可以用于开发微服务。正如前一章所讨论的，在微服务开始流行之际，该平台开发速度的放缓意味着它在采用一些必要的变革方面落后了，这些变革对于将其定位为适合云原生开发的平台至关重要。

然而，这一发展并未阻止 Java 社区启动项目来填补这一空白。其中值得注意的是 Eclipse MicroProfile 项目。还有其他同样致力于构建云原生企业级 Java 应用程序的项目，例如 Dropwizard、^(⁵⁹) Helidon、^(⁶⁰) Micronaut^(⁶¹) 和 Quarkus。^(⁶²) 所有这些框架都提供了用于开发具有弹性、容错性、可配置性、无状态、可容器化且可在云环境中移植的应用程序的 API。
