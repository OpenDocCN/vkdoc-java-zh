# 13. Jakarta EE 应用程序部署注意事项

本书涵盖了 Jakarta EE 平台上用于构建应用程序的核心 API。我们从代码角度广泛探讨了如何使用不同的 API 来开发应用程序的各个方面。作为一个通用平台，Jakarta EE 规范允许你构建各种类型的应用程序。本章涵盖了让应用程序为生产环境做好准备的一般理论。

本章首先讨论了在生产准备阶段需要注意的事项、部署 Jakarta EE 应用程序的选项和注意事项，以及如何构建应用程序以便于添加功能和修复 Bug，并给出了一般性建议。到本章结束时，你应该对如何让应用程序为生产部署和后续维护做好准备有了坚实的理论基础。

## 生产就绪

判断应用程序是否已准备好部署，需要综合考虑技术和业务因素。以下是让应用程序为生产环境做好准备时需要考虑的一些因素。

### 是否满足业务需求？

应用程序是否满足了为其开发的业务需求？这主要应由业务分析师来决定。需要将一份全面的需求清单与应用程序进行对比来确定。根据这份清单，将衍生出测试需求。

### 测试需求

需要决定为应用程序采用何种测试组合。自动化测试和手动测试应各占多少比例？当然，在应用程序开发过程中，通常会创建单元测试和集成测试的组合。但是，需要一个测试策略来测试应用程序的发布版本。

这个测试套件独立于应用程序本身的测试。它们主要从业务角度出发，涵盖应用程序功能需求的不同方面。这组测试应包含手动测试和自动化测试的不同组合，其中自动化测试应由集成测试和性能测试的健康组合构成。

### CI/CD 流水线

需要考虑用于将 Bug 修复和新功能从开发环境交付到生产环境的持续集成/持续部署流水线。当然，理想情况下，应该从一开始就将其作为开发过程的一部分建立起来。



### 进一步发展战略

部署应用程序只是持续优化和修复缺陷这一过程的开始。只要应用程序还在使用中，就会不断有新的功能需求和缺陷修复。与开发阶段可以随意改动不同，已部署的应用程序需要谨慎考虑，以免导致生产实例宕机。应采用诸如数据库迁移之类的缓解策略，为扩展现有数据提供清晰的路径。同时，还应考虑并标准化新版本部署到生产环境的频率。

### 安全性

对于即将部署的应用程序而言，安全性是唯一最重要的考量因素。无论部署何种类型的应用程序，都需要考虑端到端的安全性，这涵盖用户创建、身份验证、授权、访问控制、权限撤销以及其他安全流程。

更重要的是，要决定安全性是由内部处理，还是外包给安全服务。作为非安全专家，我们推荐使用 Keycloak IAM 框架来管理应用程序安全。除非公司内部有专门的安全专家，否则我们不建议自行开发自定义的安全实现。

以上绝非判断应用程序是否就绪所需考虑的全部因素。每个业务领域都会有自己的一套就绪检查清单。最终，一个应用程序永远不会真正完成。它只是在被持续完善的过程中投入使用。完成就绪检查后，就需要考虑部署选项了。

## Jakarta EE 部署

Jakarta EE 应用程序通常被打包为 Web 归档（WAR）文件。这些文件大多被放入兼容运行时的部署文件夹中，然后运行时便会部署该应用程序。因此，Jakarta EE 应用程序部署的基本单元是运行时。运行时可以像到目前为止看到的示例代码那样，打包成 Docker 容器，也可以独立运行。

我们建议在整个应用程序开发过程中采用容器化技术，因为它能在团队之间以及团队成员之间提供一致性和可预测性。“但在我的机器上明明能运行”这类问题，在很大程度上可以通过使用容器来消除。

为应用程序开发所选择的运行时，几乎总是会用于生产部署。无论选择哪种运行时，选择运行时都是部署 Jakarta EE 应用程序的第一步。

### 选择运行时

选择与 Jakarta EE 兼容的运行时，需要组织进行一些考量。其中一些考量因素如下。

#### 运行时的流行度

运行时的流行度是一个重要的考量因素。流行的运行时意味着拥有更庞大的用户群，这反过来也意味着该运行时（无论是闭源还是开源）的发展势头更强劲。

#### 许可证考量

运行时是开源、闭源还是混合许可证？组织需要确定运行时的许可证是否符合其整体应用程序开发策略。

#### 支持

运行时的支持模式是什么？如果是开源的，是否有足够强大的社区可以提供社区支持？如果是闭源的，其支持结构和成本相对于组织的预算如何？

#### 附加功能

Jakarta EE 的一个卖点是它是一个开放平台，实现供应商可以通过在基础平台之上添加附加功能来进行竞争。选择实现方案时，应考虑运行时除了平台基础功能之外，还具备哪些其他功能。

#### 现有知识

选择运行时还应考虑将要使用它的人员的现有知识。团队对该运行时的熟悉程度如何？这将有助于缩短团队熟练掌握该运行时所需的时间。

#### 其他考量

在选择应用程序运行时，总会有一些特定于业务领域的考量因素。Jakarta EE 拥有一个健康的实现运行时和供应商生态系统，为组织提供了丰富的选择。

### 容器化

容器的兴起使得应用程序的打包和部署变得更加容易。例如，本书的示例代码已使用 Docker 打包，并可在任何由 Docker/Kubernetes 管理的基础设施上运行。容器易于使用、普遍可用且与平台无关的特性，意味着每个应用程序开发过程默认都应采用容器化。使用容器来运行和打包应用程序有多种不同的方式。

在将容器纳入应用程序开发时，创建可预测的镜像非常重要。可预测的镜像意味着，在任何时候、任何机器上，当给定相同的输入数据时，容器通常应表现出相同的行为。这意味着在从镜像仓库拉取镜像时应使用标签。使用标签可以确保始终拉取到相同的镜像。

### 云部署

将应用程序部署到云端需要选择一种云方案。应用程序是部署到内部自建的服务器集群，还是采用混合云或纯公有云？这些决策将基于组织现有的基础设施、知识、经验和预算来做出。通过使用容器，无论选择哪种方案，最终都将归结为基于上述因素的组织偏好。

在选择云提供商时，有大量的选择。几乎所有的云提供商都支持 Docker 和 Kubernetes。这意味着一个现代的 Jakarta EE 应用程序可以轻松地部署到任何云提供商。这就是我们建议在 Jakarta EE 应用程序开发周期中采用容器化的原因。

### 数据库

每个应用程序都需要持久化的数据存储。数据库的选择将基于业务和技术考量。最终，需要在以下两者之间做出选择：使用云提供商提供的托管数据库服务，或者将数据库作为容器集合的一部分与应用程序捆绑在一起。我们建议使用所选运行时的所有数据源管理功能。Jakarta EE 运行时传统上提供了定义数据源的方法，这些数据源随后可以传递给应用程序。

### 安全性

这里再次提及安全性，因为部署选项（无论是本地部署、混合云还是纯公有云）都会带来安全影响。在权衡所有可用的部署选项时，牢记安全性至关重要。

### 部署单体应用

从理论上讲，部署单体应用比部署一套微服务更容易。这主要是因为单体应用由单个单元组成。当这个单一单元被打包成 WAR 文件时，它就是整个应用程序。就实际的部署过程而言，没有什么特别之处。前面讨论的所有要点都应适用于部署任何 Java 应用程序。而部署 Jakarta EE 应用程序，就是在部署一个纯 Java 应用程序。

## 应用程序维护

一旦应用程序部署完毕，维护它就成为确保其长期运行的重要方面。然而，应用程序的维护实际上从应用程序开发开始时就开始了。无论应用程序领域如何，将最佳实践融入应用程序开发过程，都将使应用程序的维护变得更加容易，并提高应用程序及其代码库的寿命。以下是开发长寿应用程序并使应用程序代码开发工作富有成效的一般性指导原则。



### 使用平台

Jakarta EE 平台结合 MicroProfile 项目，为构建通用型企业应用提供了相当丰富的 API 集合。几乎所有核心功能需求都能在该平台中找到对应的 API。例如，平台提供了用于依赖注入、数据持久化、REST Web 服务、WebSocket、配置和容错等功能的 API。

你的首选 API 应始终是平台中已有的。使用第三方库需要仔细权衡，并且仅当平台中没有针对该功能的 API 时才应采纳。

### 制定包命名规范

拥有标准化的包结构意味着组织中所有应用、模块和库都将具有相同的结构，从而便于新加入者快速上手。每个组织都不可避免地会有人事变动。拥有标准化的应用结构意味着无论由谁进行开发，应用的结构都保持不变。

### 谨慎选择运行时

选择应用运行时需要谨慎，因为运行时会影响开发者的生产力。一个易于作为容器启动、快速、易于学习和采用的运行时，远比不具备这些特性的运行时更受欢迎。由于开发时选择的运行时最终将用于部署应用，因此选择一个开发者熟悉且能提升其生产力的运行时至关重要。

### 避免过度使用第三方库

Java 生态系统拥有海量的第三方库，几乎可以解决所有应用开发任务。这些库可能看起来很吸引人，让人忍不住想要采用。但务必牢记，引入的每一个额外库都会成为应用维护负担的一部分。因此，应尽量减少第三方库的使用，仅将其作为最后的手段。

### 保持简单

应用开发在很大程度上是一门艺术。在复杂性与简洁性之间找到恰当的平衡需要反复尝试。然而，简洁性应成为贯穿整个代码库的主题。保持简单有助于相对轻松地进行扩展、添加功能和修复错误。它还能帮助其他开发者更快地熟悉代码库。这些都是长期积累下来的内在优势。正如那句格言所说，简洁是终极的复杂。

### 建立约定

拥有应用开发流程的约定对于维护一个支持生产力的系统至关重要。如何引导新加入者、创建新功能的标准操作流程是什么、如何处理生产环境中的错误，这些都是应用开发流程中应标准化的常规部分。SCRUM 和 Kanban 等系统有助于创建一个为参与者提供例行流程的系统。无论选择哪种系统，都应支持尽可能自动化工作流程的结构。

### 充分记录文档

从代码级别（内联注释、Javadoc）到 API 级别，再到业务级别，文档都应是应用开发流程中不可或缺的一部分。文档首先充当应用开发者的扩展记忆。覆盖尽可能多代码库的文档有助于不同的开发者处理应用的不同方面。它还能帮助新开发者更快地熟悉代码库。无论应用是带有用户界面的消费型应用，还是 API 应用，为客户提供的文档都应是整个流程中的一个重要方面。

### 构建与依赖管理

每个企业级 Jakarta EE 应用很可能都会使用两种主流构建和依赖管理工具之一——Maven 或 Gradle。这两种工具可以自动化许多常见的应用开发任务，例如管理传递性依赖和构建应用制品。无论选择哪种工具，都应将其标准化。所有服务都应使用所选定的构建工具。由于许多 Jakarta EE 应用都是使用 Maven 构建的，我们推荐使用 Maven 而非 Gradle，这仅仅是因为该领域内用户的采用率。然而，Gradle 同样是一个功能强大的构建工具，可以完成 Maven 能做的所有事情。选择哪种工具应基于哪种工具能带来最大的业务回报。

有多种方法可以自动化部分文档流程。例如，MicroProfile OpenAPI 规范^(¹¹⁹) 提供了声明式的方法来记录 Jakarta REST 资源，这与 Swagger API 非常相似。^(¹²⁰) 文档可以发布在中心位置，方便最终用户访问。无论选择哪种形式，都应鼓励和培养充分代码文档的文化，因为记录应用各个部分的“为什么”以及其使用方法的“怎么做”，是一项会在应用生命周期内产生复利效应的投资。

## 总结

本章讨论了在部署应用之前应考虑的一般性问题。我们探讨了一些用于判断应用是否已准备好投入生产的指导原则，研究了部署指南，最后提供了一些关于创建长期存在、易于维护的应用的要点。
