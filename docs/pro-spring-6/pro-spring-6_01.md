# 1.  Spring 简介

每年都会有推文和博客文章宣称 Java 已经过时，并有闪亮的新技术取而代之。然而，这些每年都被证明只是行业谣言。Java 是一项不断发展的技术，自 1995 年首次发布以来，一直位列企业构建软件解决方案最常用的十大技术之列。许多库和框架都是基于 Java 构建的，其中许多以开源项目的形式提供给开发者，而另一些则因其管理的敏感信息（例如银行应用程序）而被安全地锁定在私有云中。

用 Java 编写的最流行的框架之一是 Spring 框架。Spring 的第一个版本于 2002 年 10 月发布，包含一个易于配置和使用、带有控制反转容器的小型核心。多年来，Spring 框架已成为 Java 企业版（JEE）服务器的主要替代品，并已发展成为一个成熟的技术体系，由许多不同的项目组成，每个项目都有其特定用途。无论您是想构建微服务应用程序、传统的 ERP（企业资源规划）系统，还是将您的应用编译为可在 GraalVM 上运行的原生镜像，Spring 都有相应的项目。

在本书中，您将看到许多不同开源技术的应用，所有这些技术都统一在 Spring 框架之下。使用 Spring 时，应用程序开发者可以利用大量开源工具，而无需编写大量代码，也无需将应用程序与任何特定工具过度耦合。

这是一个介绍性章节，涵盖了本书的重要细节，向您介绍 Spring 框架，解释为什么深入理解 Spring 对开发者如此有用，并描述正确使用 Spring 时它能发挥多么强大的作用。如果您已经熟悉 Spring，可以跳过本章，直接进入**第** **2** **章**。

## 什么是 Spring？

解释 Spring 最困难的部分之一可能就是准确地对它进行分类。**Spring** 在发布后不久，最初被描述为用于构建 Java 应用程序的*轻量级框架*，但这个说法引出了两个有趣的点：

*   您可以使用 Spring 构建任何 Java 应用程序（例如，独立应用、Web 应用、移动应用或 JEE 应用），这与许多其他框架（例如 Apache Struts，它仅限于 Web 应用程序）不同。
*   描述中的*轻量级*部分与类的数量或发行版的大小无关，而是定义了 Spring 整体哲学的原则——即*最小影响*。Spring 的轻量级体现在，您只需对应用程序代码进行很少的更改（如果有的话）就能获得 Spring Core 的好处，并且如果您在任何时候选择停止使用 Spring，您会发现这样做相当简单。

请注意，我们限定最后一句仅指 Spring Core——许多额外的 Spring 组件，例如数据访问，需要与 Spring 框架更紧密的耦合。然而，这种耦合的好处是显而易见的，并且我们在本书中介绍了尽量减少这种耦合对应用程序影响的技术。此外，如果将代码与框架耦合能带来开发速度和更好的依赖管理，从而减少升级期间的构建失败次数，那么这个代价难道不值得吗？而且，借助 Spring Native，Spring 应用程序现在可以在原生镜像（例如 GraalVM）上运行，这提供了各种优势，例如即时启动和减少内存消耗，因此 JavaScript 或 .NET 开发者通过说 Spring *重量级*来贬低它的日子已经一去不复返了。



### Spring 框架的演进

Spring 框架源于罗德·约翰逊（Rod Johnson）所著的《Expert One-on-One: J2EE Design and Development》一书（Wrox 出版社，2002 年）。在书中，约翰逊展示了他自己开发的、名为 Interface 21 框架的框架，该框架用于他自己的应用程序。这个框架被发布到开源世界后，成为了我们今天所熟知的 Spring 框架的基础。在过去的十年里，Spring 框架在核心功能、相关项目和社区支持方面都取得了巨大的发展。Spring 0.9 最初是一个社区项目，由几个核心模块组成，并且没有官方文档。VMware 接手后，在 **2.5** 版本中开始了从 XML 到 Java 配置（使用注解）的转型。

2012 年，Pivotal Software 从 VMware 分离出来，接管了 Spring，并将其从一个框架发展成为一个项目集合。在 Pivotal 旗下有两个主要版本：**4.x**，这是第一个完全支持 Java 8 的版本；以及 **5.x**，该版本完全放弃了对 XML 配置的支持。

与 Spring 4.x 一起，Spring Boot 发布了，并成为最常用的 Spring 项目之一。Spring Boot 对构建 Spring 应用程序持有一种“约定优于配置”的观点。Pivotal 的开发人员识别出了几种正在构建的应用程序类型，并使用一组默认的组件依赖关系和组件的自动配置为它们创建了模板。这些应用程序模板由一组 Spring Boot 启动器项目建模。例如，如果你想创建一个 Spring Web 应用程序，只需将 `spring-boot-starter-web` 依赖项添加到你的项目中，你就会得到一个具有默认配置的最小化 Spring Web 应用程序。这是可能的，因为 Spring Boot 有一些很酷的特性，比如嵌入式应用服务器（Jetty/Tomcat）、基于 Groovy 的命令行界面以及健康/指标监控。

Spring Boot 提供了基于一组稳定、精选的依赖关系快速开发应用程序的能力，所有必需的 Spring 框架组件都已正确配置。这很好，因为它减少了开发人员的设置工作量。然而，这也降低了对 Spring 框架深入理解的需求，这可能会误导开发人员，让他们对自己关于 Spring 框架的知识和理解产生虚假的信心。

当前的 Spring 项目列在 Spring 官方网站^(⁴)上，但为了方便起见，这里列出它们并附上每个项目的简短摘要。该网站可能列出不同的项目集，因为 Spring 生态系统相当动态。

*   **Spring Session** 提供了一个用于管理用户会话信息的 API 和实现。

*   **Spring Batch** 是一个框架，提供了构建轻量级且健壮的 Spring 应用程序的工具，这些应用程序能够处理海量数据。

*   **Spring AMQP** 提供了使用 Spring 构建基于 AMQP 的消息传递解决方案的工具。

*   **Spring Flo** 是一个 JavaScript 库，为管道和简单图形提供了一个基本的可嵌入 HTML5 可视化构建器。Spring Cloud Data Flow 是该项目的扩展。

*   **Spring for Apache Kafka** 提供了使用 Spring 构建基于 Kafka 的消息传递解决方案的工具。

*   **Spring Shell** 提供了通过依赖 Spring Shell jar 并添加自己的命令（这些命令作为 Spring Bean 上的方法出现）来构建功能齐全的 shell（即命令行）应用程序的工具。

## 控制反转还是依赖注入？

Spring 框架的核心基于**控制反转（IoC）**原则。IoC 是一种将组件依赖关系的创建和管理外部化的技术。任何程序（不仅仅是 Java）执行的操作，都是其相互依赖的组件（通常称为对象）之间交互的结果。解释控制反转还需要解释**依赖注入（DI）**，这是一个描述依赖对象如何在运行时由外部方连接起来的概念。请看图 1-1，它描绘了对象之间的两种关系类型以及这些对象如何“相遇”。

![](img/315511_6_En_1_Fig1_HTML.jpg)

三个框图。1. 组合。在一个框内，对象 A 通过创建指向 B。2. 聚合。对象 A 通过检索现有对象指向 B。3. 聚合。一个对象提供者框将现有对象注入 A，并从 B 检索现有对象。

图 1-1

对象关系及其“相遇”方式

对象 A 需要一个类型为 B 的对象来执行其功能，因此 *A 依赖于 B*。图 1-1 中显示的这两个对象相互作用的三种方式可以解释如下：

*   **组合**：对象 A 直接创建对象 B。这将它们绑定在一起，基本上对象 B 存在多久取决于对象 A 存在多久。这种情况在图 1-1 的第 (1) 部分中进行了描绘。

*   **聚合**：对象 A 自行检索已经存在的对象 B。这也将它们绑定在一起，并且只要对象 A 需要对象 B，对象 B 就必须存在。此外，对象 A 必须包含能够检索对象 B 的逻辑。这种情况在图 1-1 的第 (2) 部分中进行了描绘。

*   **依赖注入**允许切断这种绑定，如图 1-1 的第 (3) 部分所示，通过使用外部方将对象 B 提供给对象 A，这仍然是聚合，但没有直接绑定并且有所变化。

**控制反转**是一种设计原则，其中使用通用的可重用组件来控制特定问题代码的执行，例如检索依赖关系。因此，可以说 Spring 是一个用于执行依赖注入的**依赖处理器**，它是遵循 IoC 原则设计的，该原则也称为好莱坞原则：*别找我们，我们会找你。*

Spring 的 DI 实现基于两个核心 Java 概念：JavaBeans（也称为 POJO，普通 Java 对象）和接口。当你使用 Spring 作为 DI 提供者时，你可以灵活地以不同方式在应用程序中定义依赖配置。直到 Spring 2.5，XML 是唯一的方式。Spring 2.5 引入了一些注解来支持代码内的配置。Spring 3 引入了 Java 配置类，这成为了标准，尽管如果你确实需要，XML 仍然受支持。从版本 4 开始，Spring 提供了对基于 Groovy 的配置的支持。这意味着 Groovy 类可以成为合法的 Spring bean。

JavaBeans（POJO）提供了一种标准机制，用于创建可通过多种方式（例如构造函数和 setter 方法）进行配置的 Java 资源。在**第** **3** **章**中，你将看到 Spring 如何使用 JavaBean 规范来形成其 DI 配置模型的核心；事实上，任何由 Spring 管理的资源都被称为 *bean*。如果你不熟悉 JavaBeans，我们将在**第** **3** **章**开头提供一个快速入门介绍。

接口和 DI 是互惠互利的技术。显然，为应用程序设计接口并基于接口编码可以使应用程序更灵活，但将基于接口设计的应用程序连接起来的复杂性相当高，并且给开发者带来了额外的编码负担。通过使用 DI，你可以将应用程序中基于接口设计所需的代码量减少到几乎为零。同样，通过使用接口，你可以充分利用 DI，因为你的 bean 可以使用任何接口实现来满足其依赖关系。接口的使用还允许 Spring 利用 JDK 动态代理（代理模式）来提供强大的概念，例如用于横切关注点的面向切面编程（AOP）。

在 DI 的上下文中，Spring 更像是一个容器而非框架——它为你的应用程序类提供它们所需的所有依赖关系的实例——但它的方式侵入性要小得多。使用 Spring 进行 DI 仅仅依赖于在你的类中遵循 JavaBeans 命名约定——无需继承任何特殊类，也无需遵循专有命名方案。如果说有什么变化，那就是在使用 DI 的应用程序中，你唯一需要做的改变就是暴露 JavaBeans 上的更多属性，从而允许在运行时注入更多的依赖关系。



### 依赖注入的演进

过去几年，得益于 Spring 及其他 DI 框架的普及，依赖注入在 Java 开发者社区中获得了广泛认可。与此同时，开发者们确信使用 DI 是应用程序开发的最佳实践，其优势也已被充分理解。当 Java 社区进程（JCP）于 2009 年采纳 JSR-330（“Java 依赖注入”）时，DI 的普及性得到了正式确认。JSR-330 成为了一项正式的 Java 规范请求，正如你所料，该规范的负责人之一正是 Spring 框架的创始人 Rod Johnson。

在 JEE 6 中，JSR-330 成为整个技术栈所包含的规范之一。与此同时，EJB 架构（从 3.0 版本开始）也经历了重大革新；它采用了 DI 模型，以简化各种企业级 JavaBean 应用的开发。

虽然我们将对 DI 的全面讨论留到**第** **3** **章**，但仍有必要先看看使用 DI 相比传统方法的好处。

*   **减少胶水代码**：DI 极大地减少了为粘合应用程序各组件而必须编写的代码量。用于粘合组件的代码通常琐碎且重复，大量此类代码会增加解决方案的整体复杂性。当需要在 Java 命名和目录接口（JNDI）仓库中查找依赖项，或者当依赖项无法直接调用（如远程资源的情况）时，情况会变得更加棘手。在这些情况下，DI 通过提供自动的 JNDI 查找和远程资源的自动代理，能够真正简化胶水代码。

*   **简化应用程序配置**：有多种方式可用于配置可注入到其他类中的类。你可以使用相同的技术向“注入器”表达依赖需求，以便注入合适的 Bean 实例或属性。此外，DI 使得切换依赖项的不同实现变得简单得多。例如，使用 DI，一个针对 PostgreSQL 数据库执行数据操作的 DAO 组件，可以轻松地通过配置切换为在 Oracle 数据库上执行相同操作（正如你将在**第** **6** **章**中看到的）。

*   **能够在单一仓库中管理公共依赖项**：使用传统方法管理公共服务的依赖项（例如，数据源连接、事务和远程服务）时，你需要在需要它们的地方（在依赖类内部）创建依赖项的实例（或从某些工厂类中查找）。这会导致依赖项分散在应用程序的各个类中，修改它们可能会带来问题。当你使用 DI 时，所有关于这些公共依赖项的信息都包含在一个单一的仓库中，使得依赖项的管理更加简单且不易出错。

*   **提高可测试性**：为 DI 设计的类易于测试，因为依赖项可以轻松替换。例如，如果你想测试一个 DAO 组件，你可以通过配置用内存数据库替换具体的数据库。这样做的好处是测试执行速度更快，并且是在一个尽可能模拟生产环境的适当上下文中进行的。这种机制可以扩展到测试应用程序的任何层，对于测试 Web 组件尤其有用，你可以为 `HttpServletRequest` 和 `HttpServletResponse` 创建模拟实现。

*   **促进良好的应用程序设计**：典型的面向注入的应用程序设计方式是，所有主要组件都被定义为接口，然后创建这些接口的具体实现，并使用 DI 容器将它们连接起来。在 DI 和基于 DI 的容器（如 Spring）出现之前，这种设计在 Java 中也是可行的，但通过使用 Spring，你可以免费获得一整套 DI 特性，并且能够专注于构建应用程序逻辑，而不是构建一个支持它的框架。

从这份列表中可以看出，DI 为你的应用程序带来了许多好处，但它并非没有缺点。特别是，DI 可能会让不熟悉代码的人难以看清特定依赖项的哪个具体实现被注入到了哪个对象中。通常，这仅仅是开发者在 DI 方面经验不足时才会出现的问题。我们听过很多处于这种情况的同事提到 Spring 会“*自动魔法般地*”注入 Bean，或者抱怨他们不理解 Bean 来自何处，更糟的是，他们会问“为什么 Spring 找不到我的 Bean？”。随着经验增长并遵循良好的 DI 编码实践（例如，将每个应用层内的所有可注入类放在同一个包中），开发者将能够轻松地看清全局。在大多数情况下，DI 的巨大优势远远超过了这个小小的缺点，但在规划应用程序时，你应该考虑到这一点。



### 超越依赖注入

仅凭其先进的依赖注入（DI）能力，Spring Core 本身就是一个有价值的工具，但 Spring 真正卓越之处在于其众多附加功能，这些功能都基于 DI 原则精心设计构建。Spring 为应用程序的所有层级提供功能，从用于数据访问的辅助应用程序编程接口（API），到高级的 MVC（模型-视图-控制器）能力。Spring 中这些功能的伟大之处在于，尽管 Spring 通常提供自己的方法，但你可以轻松地将它们与 Spring 中的其他工具集成，使这些工具成为 Spring 家族的一等公民。

以下列表描述了最重要的 Spring 功能，并指出了本书中介绍这些功能的章节：

*   *面向切面编程*：AOP 提供了在单个位置实现横切逻辑（即应用于应用程序许多部分的逻辑）的能力，并让该逻辑自动应用于整个应用程序。Spring 的 AOP 方法（在**第** **5** 章中介绍）是为目标对象创建动态代理，并将对象与配置的通知编织在一起以执行横切逻辑。根据 JDK 动态代理的特性，目标对象必须实现一个接口，该接口声明了将应用 AOP 通知的方法。

*   *Spring 表达式语言*：表达式语言（EL）是一种允许应用程序在运行时操作 Java 对象的技术。然而，EL 的问题在于不同的技术提供了各自的 EL 实现和语法。例如，Java 服务器页面（JSP）和 Java 服务器面（JSF）都有各自的 EL，并且它们的语法不同。为了解决这个问题，统一表达式语言被创建出来。*Spring 表达式语言（SpEL）* 是在 Spring 3.0 中引入的。其语法类似于统一 EL，并提供了强大的功能，用于评估表达式以及在运行时访问 Java 对象和 Spring Bean。

*   *验证*：应用程序管理的数据必须遵守特定的验证规则。理想的情况是，无论数据操作请求是从前端、批处理作业还是远程（例如，通过 Web 服务、RESTful Web 服务或远程过程调用 [RPC]）发起的，包含业务数据的 JavaBean 中属性的验证规则都能以一致的方式应用。为了解决这些问题，Spring 通过 `Validator` 接口提供了一个内置的验证 API。该接口提供了一种简单而简洁的机制，允许你将验证逻辑封装到一个负责验证目标对象的类中。除了目标对象之外，`validate()` 方法还接受一个 `Errors` 对象，用于收集可能发生的任何验证错误。验证主题及其所有方面在**第** **11** 章中有详细讨论。

*   *访问数据*：数据访问和持久化似乎是 Java 世界中讨论最多的话题。Spring 提供了与精选数据访问工具的出色集成。此外，Spring 通过围绕标准 API 的简化包装器 API，使普通的 JDBC 成为许多项目的可行选择。Spring 的数据访问模块为 JDBC（**第** **6** 章）、Hibernate（**第** **7** 章）、JDO 和 JPA（**第** **8** 章）以及各种 NoSQL 数据库（**第** **10** 章）提供了开箱即用的支持。当使用 Spring API 通过任何工具访问数据时，你都可以利用 Spring 出色的事务支持。你将在**第** **9** 章中找到对此的全面讨论。

*   *管理事务*：Spring 为事务管理提供了一个出色的抽象层，允许进行编程式和声明式事务控制。通过使用 Spring 的事务抽象层，你可以轻松更改底层的事务协议和资源管理器。你可以从简单的、特定于资源的本地事务开始，然后迁移到全局的、多资源的事务，而无需更改代码。事务在**第** **9** 章中有详细讨论。

*   *对象映射*：大多数应用程序需要集成或向其他应用程序提供服务。一个常见的要求是与其他系统定期或实时交换数据。在数据格式方面，XML 曾经是最常用的，但如今 JSON 和 YAML 已经取而代之，这主要是因为 Jackson 项目提供了出色的支持（⁹）。从**第** **13** 章开始，我们将讨论如何以各种格式远程访问 Spring 应用程序以获取业务数据，你将看到如何在应用程序中使用 Spring 的对象映射支持。

*   *作业调度支持*：大多数重要的应用程序都需要某种调度能力。无论是向客户发送更新还是执行维护任务，在预定义时间调度代码运行的能力对开发者来说都是一个宝贵的工具。Spring 提供了可以满足大多数常见场景的调度支持。任务可以按固定间隔调度，也可以使用 Unix cron 表达式调度。另一方面，对于任务执行和调度，Spring 也与其他调度库集成。例如，在应用服务器环境中，Spring 可以将执行委托给许多应用服务器使用的 CommonJ 库。对于作业调度，Spring 还支持包括 JDK Timer API 和 Quartz（一个常用的开源调度库）在内的库。Spring 中的调度支持在**第** **12** 章中有全面介绍。

*   *Web 层的 MVC*：尽管 Spring 几乎可以用于任何环境（从桌面到 Web），但它提供了丰富的类来支持基于 Web 的应用程序的创建。使用 Spring，你在选择如何实现 Web 前端时拥有最大的灵活性。对于开发 Web 应用程序，MVC 模式是最流行的实践。在最近的版本中，Spring 已逐渐从一个简单的 Web 框架演变成一个成熟的 MVC 实现。首先，Spring MVC 中的视图支持非常广泛。除了对 JSP 和 Java 标准标签库（JSTL）的标准支持（这得益于 Spring 标签库的大力增强）之外，你还可以利用对 Apache Velocity、FreeMarker、Thymeleaf、XSLT、React 和 Mustache 模板的完全集成支持。此外，你还会发现一组基础视图类，可以轻松地向应用程序添加 Microsoft Excel、PDF 和 JasperReports 输出。从**第** **15** 章开始，我们将讨论如何使用 Spring MVC 开发 Web 应用程序。

*   *远程支持*：在 Java 中访问或暴露远程组件从来都不是一件简单的事情。使用 Spring，你可以利用对多种远程技术的广泛支持，快速暴露和访问远程服务：JMS、高级消息队列协议（AMQP）和 REST。如今，访问远程服务通常涉及实时流数据，应用程序必须适应数据流；这正是 Apache Kafka 的用武之地。Spring 如何与这些技术集成在**第** **13** 章中介绍，但 REST 除外，它在**第** **16** 章中介绍。



*   *简化异常处理*：Spring 真正能帮助减少大量重复样板代码的一个领域是异常处理。Spring 在此方面的核心理念是，Java 中过度使用了受检异常，框架不应强制你捕获任何你不太可能从中恢复的异常——我们完全赞同这一观点。实际上，许多框架的设计都是为了减少编写代码处理受检异常所带来的影响。然而，其中许多框架采取的方法是坚持使用受检异常，但人为地降低了异常类层次结构的粒度。使用 Spring 时你会注意到的一点是，由于使用非受检异常为开发者带来的便利，其异常层次结构异常精细。在本书中，你将看到许多示例，展示 Spring 的异常处理机制如何减少你必须编写的代码量，同时提高你在应用程序中识别、分类和诊断错误的能力。

*   *WebSocket 支持*：从 Spring Framework 4.0 开始，提供了对 JSR-356（“Java WebSocket API”）的支持。WebSocket 定义了一个用于在客户端和服务器之间创建持久连接的 API，通常实现在 Web 浏览器和服务器中。WebSocket 风格的开发为高效的全双工通信打开了大门，能够实现实时消息交换，从而构建高度响应的应用程序。WebSocket 支持的使用将在**第** **19** **章**中进一步详述。Spring Framework 提供了一个响应式 WebSocket API，你可以用它编写处理 WebSocket 消息的客户端和服务器端应用程序，这将在**第** **20** **章**中简要介绍。

*   *响应式编程*：响应式编程描述了一种设计范式，它依赖异步编程逻辑来处理原本静态内容的实时更新。它提供了一种高效的手段——使用自动化数据流——在用户发起查询时处理内容的数据更新。当 JDK 的响应式支持被推迟时，Spring 团队创建了自己的响应式流实现 Project Reactor，以便 Spring 能够抢先一步，为使用 Spring 构建响应式应用程序提供支持。如何开发响应式 Spring 应用程序将在**第** **20** **章**中介绍。

*   *保护应用程序安全*：在几乎所有商业交易都在互联网上完成的世界里，保护数据安全并确保对其的访问不再是可选项，而是必需项。Spring Security 项目^(¹⁰)，以前被称为 *Acegi Security System for Spring*，是 Spring 产品组合中的另一个重要项目。Spring Security 为 Web 应用程序和方法级安全提供了全面的支持。它与 Spring Framework 以及其他常用的身份验证机制（如 HTTP 基本认证、基于表单的登录、X.509 证书、OAuth2 和单点登录 (SSO) 产品（例如 CA SiteMinder））紧密集成。它为应用程序资源提供基于角色的访问控制 (RBAC)，并且在具有更复杂安全需求（例如数据隔离）的应用程序中，它支持使用访问控制列表 (ACL)。Spring Security 主要用于保护 Web 应用程序的安全，这将在**第** **17** **章**中介绍。响应式安全将在**第** **20** **章**中简要提及。

*   *监控应用程序*：一个由多个组件构成的应用程序，在投入生产后，应该有一套设置来确保构成应用程序的众多组件平稳运行。你需要监控其性能——资源使用情况、用户流量、请求速率、响应时间、瓶颈、内存问题等——以便能够克服这些限制并确保良好的最终用户体验。Spring Actuator 用于公开正在运行的应用程序的操作信息——健康、指标、信息、转储、环境等。它使用 HTTP 端点或 JMX Bean 使您能够与之交互。Actuator 使用 Micrometer，这是一个以维度为先的指标收集门面，其目标是允许你使用供应商中立的 API 对代码进行计时、计数和测量。尽管专注于维度指标，但 Micrometer 确实会映射到分层名称，以继续服务于像 Ganglia 这样的旧式监控解决方案或像 JMX 这样范围更窄的工具。转向 Micrometer 源于希望更好地服务于一波维度监控系统（想想 Prometheus、Datadog、Wavefront、SignalFx、Influx 等）的愿望。这种灵活性是 Spring 为依赖注入 (DI) 而设计的一个优势，它也允许轻松地切换监控系统。使用 Actuator 将在**第** **18** **章**中介绍。

*   *随处运行*：长期以来，技术演进让我们能够“做大”，我们找到了在提高 CPU 频率的同时保持其冷却的新方法，我们设法用更小的芯片提供更多的内存，但最近出现了一种新趋势。在私有云上部署应用程序让人们开始担心云成本，因此一个新兴趋势是构建需要更小 CPU 和更少内存来运行的紧凑型应用程序。用更少的资源消耗获得更好的性能意味着更低的云成本，这就是紧凑型原生镜像时代到来的原因。实验性的（目前）Spring Native 项目支持使用 GraalVM 原生镜像编译器将 Spring 应用程序编译为原生可执行文件。GraalVM 是一个高性能运行时，可显著提高应用程序的性能和效率，是微服务的理想选择。这意味着应用程序启动更快，运行时需要的资源更少。这使得它们适合以合理的成本部署在私有云（Amazon、GCP、Hetzner 等）中。如何使用 Spring Native 和其他好东西构建一个可使用 GraalVM 原生镜像运行的 Spring 项目，将在**第** **14** **章**中介绍。



## Spring 社区

Spring 社区是我们所遇到的所有开源项目中最好的社区之一。邮件列表和论坛始终活跃，新功能的开发进度通常也很快。开发团队真正致力于让 Spring 成为所有 Java 应用框架中最成功的一个，这一点从其所产出的代码质量中可见一斑。正如我们之前提到的，Spring 还得益于与其他开源项目的良好关系，考虑到完整的 Spring 发行版所依赖的大量组件，这一事实极为有利。从用户的角度来看，Spring 最好的特性之一或许是随发行版附带的优秀文档和测试套件。

Spring 几乎所有功能都提供了文档，这使得新用户能够轻松上手该框架。Spring 提供的测试套件非常全面——开发团队为所有内容编写测试。如果他们发现了一个 bug，会先编写一个能凸显该 bug 的测试，然后让测试通过，以此来修复 bug。修复 bug 和创建新功能并不仅限于开发团队！你可以通过官方 GitHub 仓库^(¹¹)，针对任何 Spring 项目组合提交拉取请求来贡献代码。或者，你也可以通过官方实验性 GitHub 仓库^(¹²)贡献代码，帮助实验性项目成长。

这一切对你意味着什么？简单来说，这意味着你可以对 Spring 框架的质量充满信心，并且可以确信，在可预见的未来，Spring 开发团队将继续改进这个本已卓越的框架。

## Spring 的替代方案

回到我们之前关于开源项目数量的评论，当你了解到 Spring 并非唯一提供依赖注入功能或用于构建应用程序的完整端到端解决方案的框架时，你不应感到惊讶。

一个流行的 DI 框架是 **Google Guice**^(¹³)。由搜索引擎巨头谷歌主导，Guice 是一个轻量级框架，专注于为应用配置管理提供 DI。它也是 JSR-330 的参考实现。

**Vaadin**^(¹⁴) 是一个用于 Java 的开源 Web 应用开发平台。Vaadin 包含一组 Web 组件、一个 Java Web 框架以及一套工具，使开发者能够仅使用 Java 编程语言、仅使用 TypeScript 或两者结合来实现现代 Web 图形用户界面。它的范围仅限于 Web 应用程序，但对于更复杂的应用程序，它可以轻松地与 Spring 集成，并利用 Spring IoC 容器的强大功能。

现已停用的 **JBoss Seam 框架**^(¹⁵) 曾经是 Spring 的一个不错替代品。当我在 2012 年第一次参加 Spring 培训时，我非常确信其用于配置 bean 的 Spring 注解模型是受其启发。

现已停用的 **PicoContainer**^(¹⁶) 是一个极其小巧的 DI 容器，它允许你在应用程序中使用 DI，而除了 PicoContainer 之外无需引入任何其他依赖。由于它仅仅是一个 DI 容器，编写更复杂的应用程序需要添加额外的框架，例如 Spring，在这种情况下，你从一开始就使用 Spring 会更好。然而，如果你只需要一个微小的 DI 容器，那么 PicoContainer 是一个不错的选择。
