# 1. 性能反模式

在深入探讨性能故障排查与调优的细节之前，首先从宏观层面理解性能缺陷是如何潜入代码中的，会大有裨益。本章介绍的四种反模式，将阐明你在 Java 服务端软件中会遇到的主要缺陷类型。在本书后续章节中，我们会反复引用本章所描述的性能反模式。

本章的目标如下：

*   学习如何从性能监控工具中那些看似陌生的原始数据入手，围绕它构建一个故事（即假设），用以描述代码为何变得如此缓慢。
*   识别大多数性能缺陷之间惊人的相似之处。
*   牢记四种主要性能反模式的名称，或者自行定义你的反模式。
*   学习如何评估一个给定的缺陷是处理上的大挑战还是小挑战。

当你初次着手调查一个性能问题时，你掌握的信息非常有限——也许只有本书后面四个 P.A.t.h. 章节中提到的某个监控工具提供的一两段可疑数据——但并没有确凿证据，也没有现成的修复方案。问题在于：你手头只有一张关于性能下降的粗略草图，而你需要一幅详细的画面来找出罪魁祸首。

为了填补这些细节，更好地理解缺陷背后的完整故事，本章提供了四种最常见性能缺陷的档案——我称之为“主要性能反模式”。

## 主要性能反模式

以下是主要性能反模式。在调查性能问题时，请问自己，你的缺陷最接近以下哪种反模式。

1.  **不必要的初始化**：在初始化某个较大进程时，系统频繁地重新处理一个已经处理过的小结果。多次执行的累积效应导致了额外的 CPU/内存消耗和/或更慢的执行速度，而这些本都可以避免。缓存这个小结果是主要的应对方法。
2.  **策略/算法低效**：配置不当或选择不佳的算法或编码策略导致了性能问题。策略是贯穿整个代码库使用的技术，而算法是用于实现单个组件的方案。
3.  **过度处理**：系统在执行不必要的工作。移除这些工作能带来可衡量的性能提升。一个例子是检索了过多的数据，而其中大部分数据都被丢弃了。另一个例子是在负载测试中包含了错误的、资源消耗大的用例——该用例在生产环境中很少使用。
4.  **大规模处理挑战**：试图处理并征服海量数据。很少有应用真正有这种需求，但它们确实存在。例如，查询 40 亿行数据，或通过慢速网络反复传输 10MB 数据等。

不幸的是，这些主要反模式之间存在重叠，一个性能缺陷可能同时符合两个或更多不同的主要反模式。为了解决这个分类模糊的问题，我根据（我个人的）经验，将这四种问题按从最常见到最不常见的顺序进行了排序。这样，你可以按顺序逐一排查，并选择第一个与当前问题最匹配的反模式。

## 主要反模式 #1：不必要的初始化

这种反模式经常出现在初始化阶段，即在真正开始执行某些操作之前。这是一种低效、CPU 和 I/O 密集型的编码方式，其中某些已经处理过的任务被不必要地反复处理（常常是令人厌烦地重复），导致了不必要的开销。例如，每次进行 Schema 验证时都重新读取和重新解析 XML Schema 的 `.xsd` 文件是不必要的，因为 `.xsd` 文件中的数据很少改变。与其重新读取/重新解析 `.xsd` 文件，不如存储并使用一个已经解析好的缓存版本。

嗯，`.xslt` 文件也是静态的，它们与 `.xsd` 文件存在同样的问题——在转换开始之前，必须从文件系统（或网络）读取并解析它们，但大多数系统在每次转换时都会承受重新读取和重新解析的开销。

事实证明，像 `.xsd` 和 `.xslt` 文件这样的静态数据无处不在，而重新读取或重新解析所有这些数据会带来性能损失。以下是存在同样问题的应用级数据示例：

*   **授权数据**：哪个用户被授予了哪些角色，哪些角色对哪些资源拥有权限？如果你决定缓存这些数据以获得良好的性能提升，那么可以按需轻松地使缓存失效，以适应对时间敏感的安全变更，比如撤销某个心怀不满的前雇员的权限。
*   **产品数据**：是的，对于沃尔玛和亚马逊来说，将其庞大的产品目录放入 RAM 是很困难的，但对于许多产品目录可以轻松放入 512 MB RAM 的组织来说，通过在系统启动时将产品配置加载到内存中，可以提升性能。
*   **组织数据**，例如物理工厂位置、世界各国列表、美国各州等。
*   **处理配置数据**，例如在银行系统中编排借贷流程的各种交易代码。

反复重新加载任何这类静态应用数据都会导致可衡量的、有问题的开销。然而，这些改进并不仅限于应用数据，因为还有一些更低层次的对象，反复重新加载它们的代价也很高昂。例如，许多 XML 处理 API 看起来似乎 100% 的处理都在内存中完成，直到你发现其中包含了大量文件系统读取、JAR 文件解压和同步操作。部分问题在于，互联网上大多数 XML 操作指南都提倡对性能问题不敏感的编码习惯。所有这些令人讨厌的 I/O 操作都可以通过使用另一种 API 消费习惯来避免。这里有很大的优化空间。所以，试试这些建议。你可以在我的 `github.com` 上的 `eostermueller` 仓库中找到相关代码——它被命名为 `crazyFastXml`：

[`https://github.com/eostermueller/crazyFastXml`](https://github.com/eostermueller/crazyFastXml)

*   对于 XSLT，缓存并重用 `javax.xml.transform.TransformerFactory.newTransformer()` 的输出
*   对于 SAX 解析，缓存并重用 `javax.xml.parsers.SAXParserFactory.newSAXParser()` 的输出
*   对于 StAX 解析，缓存并重用 `javax.xml.stream.XMLInputFactory.newInstance()` 和 `javax.xml.stream.XMLOutputFactory.newInstance()` 的输出
*   对于 XPath，不要使用 `javax.xml.xpath.XPathFactory.newXPath()`，而是使用 `org.apache.xpath.CachedXPathAPI`。

我们将在第 11 章中更详细地讨论如何检测和修复这些问题。

缓存正确的对象有助于避免第一种反模式（不必要的初始化）。同样的情况也适用于网络：缓存并重用网络连接（即 TCP 套接字）比每次请求都关闭并重新创建它要高效得多。例如：



*   考虑使用 Apache HTTP 客户端中的连接池，以避免重新创建 TCP Socket 的开销。在互联网上搜索“apache client connection management”即可找到相关资源。
*   某些服务端 JDBC 应用仍然会为每个 SQL 请求使用 `javax.sql.DriverManager.getConnection()` 来重新创建数据库连接的 TCP Socket，从而产生开销。相反，可以考虑使用 Hikari、C3P0 或由你的容器（无论是 JBoss、WebSphere、Jetty、Tomcat 还是其他容器）提供的 JDBC 连接池。

现在我们已经对几个罪魁祸首有了清晰的认识，接下来再谈谈一些常见的特征，这些特征将帮助你在查看性能数据时识别出这种“不必要的初始化”反模式：

*   **大量的小型 I/O 操作。** 这些例子并非指大型 I/O 操作（例如 50ms 或更长），而是指由少数子系统（如应用程序初始化、XML 处理、HTTP 请求等）频繁执行的许多小型 I/O 操作（可能每次仅需几毫秒），每次往返都会产生开销。
*   **性能不敏感的惯用法。** XML API 的四个问题表明，使用错误的惯用法来消费特定 API 可能会导致不必要的初始化。在四个 XML 案例中，互联网上一些最流行的惯用法恰好充满了性能问题。这是一个问题。

作为性能建议，“避免不必要的 I/O”听起来是老生常谈，但它恰恰是这类大量性能缺陷的根本原因。哎。让我们更加关注“避免不必要的 I/O”这条建议，并尽量减少对磁盘和网络的请求。

## 主要反模式 #2：策略/算法低效

这种主要反模式指的是选择了性能不佳的算法或策略。同时，它也包含了选择了正确的算法但使用了错误的算法配置/参数化的情况。比较算法的复杂度（使用“大 O”表示法）超出了这本小书的范围。

这里不是要你痴迷于你最喜欢的排序算法，或者使用“大 O”表示法来比较最坏情况下的复杂度。为什么？因为大多数企业软件的复杂度要低得多。它更多是关于将数据从 A 点移动到 B 点。这本书要讲的是，在我们甚至还没开始尝试处理更复杂的事情（比如算法）之前，我们就经常让自己陷入的各种性能麻烦。

这种反模式涉及算法（处理某些数据的结构化攻击计划），也涉及策略，比如数据访问策略，即应用程序用来构建 SQL 语句以检索/更新数据的方法。算法的实现通常位于单个位置。而另一方面，策略的调整和/或替换要困难得多，因为其代码分散在整个系统中。

你可以在开发周期的早期使用性能测试，以避免投资于一个性能不佳、替换成本高昂的代码策略。例如，你可以尽早发现执行大量（比如超过 25 个）数据库或其他后端请求是否能够表现良好（答案是不能）。

在关于持久化的第 9 章中，我将讨论服务端软件中的“有史以来最大的性能问题”。这个问题非常严重，以至于它延伸到了反模式 1 和 2。敬请期待——它真的很大，而且你很可能已经见过它了。你能猜到它是什么吗？

## 主要反模式 #3：过度处理

这种反模式指的是错误地尝试拥抱资源密集型用例，而这些用例的合理性往往只是勉强成立。一个常见的例子是，处理的数据量超过了单个最终用户在一屏或一个网页信息中可能使用的量。换句话说，期望在一个 HTML 表格中加载 50 页数据并在 100ms 内完成是不可能的。但是，加载 3 页数据并配上一个“下一页”按钮，性能则可以表现得相当不错。另一个例子是，期望一个特别复杂的用例在高吞吐量下表现良好，即使它在生产环境中只会经历较低的吞吐量。

虽然评估这种反模式需要一些技术专长，但它更多地依赖于业务专长。你特别需要知道哪些业务流程正在或将会在生产环境中使用，并且了解每个流程的大致百分比会很有帮助。

为了方便开发人员和 QA 人员，测试数据库中通常包含一些配置好的“超级客户”，用于展示大量令人眼花缭乱的业务功能，尽管客户在生产环境中从未变得如此庞大或复杂。在负载测试中出现这样的数据，是导致“过度处理”反模式的常见原因。

以下是不幸场景的一种可能表现：你创建了一个给系统施加压力的负载测试（第 4 章）。本书 P.A.t.h. 部分的四个章节将引导你连接监控工具，找到代码中最慢的部分和 CPU 消耗最高的部分。这一切都很好——这里没有反模式。问题在于，如果你过于急切，花费大量时间调优用户很少访问的某段特定代码，而这段代码恰好被无意中纳入了你的负载测试中。这就是浪费时间。

再举一个例子。考虑一个无意中过大的 RDBMS 结果集，它需要额外的时间来处理。SELECT 语句中过于宽泛的 WHERE 子句可能导致这种情况，比如一个放错位置的通配符，检索出了所有匹配任何条件的客户。但是，数据库中意外和无效的数据也可能是罪魁祸首。并且不要忘记前面提到的典型例子：你的系统浪费资源将大量数据塞进一个网页中，数据量远超单个用户所能消化。

所以请记住：性能调优既关乎于雕琢代码使其运行更快，也关乎于做出明智的选择，决定对哪些业务流程进行负载测试。它还关乎性能与用户界面设计之间的权衡——比如保持网页尺寸较小以兼顾性能问题。

这意味着，当你的头埋在你的系统引擎盖下，审视一个大型结果集带来的可疑性能问题时，你必须重新调整你的思路，考虑更抽象的事情，比如性能/可用性的权衡以及业务的吞吐量需求。这种人类思维的上下文切换并不容易，但请考虑一下它有多大的帮助：一些调优挑战完全可以避免，因为底层的用例是如此不切实际，以至于它一开始就不应该被包含在负载测试中。



## 主要反模式 #4：大型处理挑战

这种反模式描述了 I/O 和其他硬件限制所导致的重大性能挑战，例如在数十亿行数据中进行查找，或通过极慢的网络反复传输数兆字节的数据。企业级软件开发中很少遇到这类大型、棘手的挑战。但当你确实遇到时，我建议在设计和测试上投入更多精力，保持敬畏之心，并带有一丝谨慎。优化其他所有事情几乎都是小菜一碟。

“大型处理挑战”是四个主要反模式中的最后一个。正如我所提到的，反模式 1 出现得最频繁，而反模式 4 出现得最少。如果你身处技术前沿，不断尝试新事物，并且经常遇到这些大型挑战，那么你可以将自己视为例外。恭喜你。但对于我们这些编写企业级 Web 应用程序的其他人来说，很少有能引发这种恐惧感的大型处理挑战。以下是这些大型挑战的几个例子：

*   在一个包含 2 亿条记录的表中仅查找几条记录。
*   在慢速网络上（尤其是长距离）反复传输大量数据（例如超过 1 兆字节）。
*   多线程性能很难做到完美，但有许多现成的 API 和解决方案，其中别人已经实现了良好的性能。

你如何判断哪些处理挑战是“大型”的？这类挑战如此巨大，以至于它们会无视以下因素而出现：

*   语言（Java、C#、C++……）
*   操作系统
*   API 的选择（例如用于 XML 解析的 StAX 与 SAX）
*   代码访问 API 所使用的惯用方式，例如用于 XSLT 转换的 Xalan 与 Saxon

为这些大型挑战做准备需要付出巨大的努力，以至于准备工作似乎应该在创建设计文档时就开始了。如果你真的担心某个大型处理挑战，请尽早投入时间在项目中测试其可行性；也许硬件专家可以在此提供帮助。

开发人员常常会将偶然的数据激增误认为是大型处理挑战。例如，当查询过去 2 年的数据就能表现良好，并且业务方也乐于只保留一年的数据时，何必忍受查询过去 10 年历史数据所带来的缓慢响应呢？

本书中介绍的故障排除技术无疑能够识别出大型处理挑战（无论是真实的还是想象中的）所导致的性能问题。然而，我必须承认，为大型挑战创建解决方案超出了本书的范围，主要是因为这类真正的问题很少见。

### 评估处理挑战的规模

大型处理挑战（主要反模式 4）会出现在某种突出显示性能问题的性能雷达上，这完全合情合理，尤其是在对在线 Java 服务器端系统（本书的主要主题）进行故障排除时。

由于解决此类大型处理挑战需要付出艰苦努力，开发团队应预见到额外的工作量，并在排期中提高性能调优工作的优先级。然而，当较小的性能挑战出现在性能雷达上时，它们会干扰其他优先事项，例如调优大型性能挑战。它们还会从开发生命周期的其他部分（设计、开发、QA 等）窃取时间。

调优小型性能挑战本质上是一种麻烦；根据我的经验，小型性能挑战中的性能下降在所有性能缺陷中占据了出乎意料的大比例。

因此，在查看任何可能表明性能下降的数据时，请执行以下操作：

*   在心中粗略评估当前的处理是大型、中型还是小型挑战。

这里有一个重要提示，可以帮助你找到需要调优的小型挑战：通常，当任务需要编写类似 SOA 服务或 Web 请求的代码时，会有一个主要任务需要处理。例如，必须下单；必须返回查询结果；必须开始关店。几乎可以定义地说，在“主要任务”之前或之后发生的处理是一个小型挑战，任何相应的性能下降都是需要消除的麻烦。同样，所谓的“横切关注点”，如日志记录、审计、安全甚至监控本身，都可能是小型处理挑战的候选对象。

此外：

*   当一个看起来缓慢的小型处理挑战涉及一个公开可用的 API 时，该 API 的用户社区极有可能已经遇到过类似的性能下降问题，你可以从中学习。

花点时间去寻找这些性能经验——在文档页面、互联网搜索等任何地方。只需在你的搜索条件中将“performance”这个词附加到 API 名称后面，然后进行搜索。在用户论坛中提问。另外，当我说“公开可用的 API”时，当然，任何来自 github.com 和 mvnrepository.com 的内容都属于这一类。在我十年的调优经验中，我见过大量的性能缺陷，而互联网上的好人似乎总是能抢先一步。当处理挑战的规模很小且使用的 API 是公开可用时，其他人已经记录了这些问题并发现了合理的解决方法。

## 使用主要性能反模式

以下是关于如何应用这些反模式的逐步说明。

1.  首先，使用 P.A.t.h. 清单收集数据，该清单从第 8 章开始。在所有 P.A.t.h. 数据中，选择看起来问题最大的测量/结果，修复它将对性能产生最积极的影响。  
2.  询问这个“问题最大”的测量结果是否看起来像主要反模式 1。
    *   它是否涉及静态数据，或者至少不涉及动态数据？
    *   静态数据是否针对每个系统请求都重新处理？
    *   每次重新处理静态数据时，是否涉及少量的 I/O？
    *   ……等等
3.  如果数据似乎符合主要反模式 1，那么恭喜你，你已经为问题建立了一个不错的假设。要测试这个假设，请对你的系统应用一个修复，然后重新测试并重新评估性能。这就是为性能缺陷“构建案例”的方法。  
4.  如果“问题最大”的测量数据似乎不符合反模式 1，那么继续评估主要反模式 2。  
5.  继续评估所有四个主要反模式。它们一个都不符合的可能性极低。



## 别忘了

当今服务器端的大部分性能问题，通常都是由少数几个微不足道的小处理任务引起的。这些任务的性能本可以且应该表现良好，但实际上却并非如此。这些“微不足道的小处理任务”大多出现在主要性能反模式 1 和 2 中，即“不必要的初始化”和“低效的算法/策略”。实际上，反模式 1 是反模式 2 的一个特例。

请记住，与策略相比，低效的算法更容易重构或替换，因为策略的细节分散在整个代码库中。最典型的例子是一种非常“啰嗦”的数据库策略，它为每个请求执行数十条 SQL 语句：这种策略性能低下，且重构成本极高。

虽然不太常见，但另外两个主要性能反模式（3 和 4）是“过度处理”和“大规模处理挑战”。大规模处理挑战应在设计阶段就被识别出来（并引起警惕）。当你开始发现系统中可能变慢的部分时，别忘了问问是否是“过度处理”导致的。这种情况发生在你对一个不常用的业务流程施加了不切实际的大量负载时，或者当你的代码“广撒网”处理了大量数据，最终却丢弃了其中很大一部分时。

在本书的其余部分，我们将不断回顾这些主要性能反模式，以此作为一种快速沟通和理解各种严重拖累服务器端 Java 性能问题的根本原因的方式。
