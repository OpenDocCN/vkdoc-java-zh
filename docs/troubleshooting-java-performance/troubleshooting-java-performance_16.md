# 13. 结论

其他学科在工程实践和生产就绪性验证方面，更倾向于采取“先走后跑”的方法。例如，飞机空气动力学特性在首次试飞前会在风洞中进行测试。在建造全尺寸桥梁之前，会先制作一个比例模型来评估其承载能力。

然而，即使在今天，我们仍在编写无法应对生产负载的软件，仿佛一两个功能测试就近似于生产工作负载。事实并非如此。我们的交付成果必须包含处理生产负载这一部分。

为了弥补这一差距，你首先需要理解一个常见的误解，我鼓励你和朋友、同事讨论这个误解。不要羞于启齿。

这个误解是：让性能达标几乎是不可能的，因为你必须拥有庞大的计算环境和昂贵的监控工具，才能解决那些只有性能专家才能修复的问题。我们还认为许多环境监控不足——即所谓的“黑暗环境”。黑暗环境的存在是另一个误解，因为正如本书所示，每个 JDK 在每个黑暗环境中都有许多“即插即用”的工具可用。

但实际上，就像本书中的代码示例一样，大多数性能缺陷都可以在任何环境中，使用开源或免费可用的监控工具进行复现和检测。只需要一点有条理的方法（比如 P.A.t.h. 清单）的指导，就能帮助你定位最常见和最有害的缺陷。

最终，在投入生产之前，系统性能必须在全尺寸环境下，承受完整的生产工作负载进行验证。但是，试图在一个大型环境中一次性调整所有内容，通常会导致仓促且不完善的性能优化工作。这是一项昂贵且极其缓慢的工作，因为缺乏访问权限且维护要求高，比如需要半天或更长时间的代码重新部署。

相比之下，小型环境易于访问，你可以在一天内完成大约十个“修复-测试-调优”循环。最后，在小型环境中获得的早期反馈有助于避免因采用性能不可接受的方法而付出高昂的代价。

## 入门步骤

现在就开始调优你的系统非常简单。请遵循以下步骤：

1.  使用现有硬件创建一个**适度调优环境**（第 2 章）。不需要大量快速、昂贵的性能硬件。
2.  让我们解决开发环境中数据库规模过小的问题。无论是租用还是购买，为开发中的各个适度调优环境额外投资一些硬盘。十个单 TB 驱动器的成本很小。缺乏生产规模的数据集是定位性能缺陷的障碍。通常需要大量磁盘数据才能加剧 GC 和其他问题。你还需要给团队 40-80 小时的时间来开发一个可重复的解决方案，以生成生产规模的数据库。这是一个能带来巨大收益的适度要求。
3.  确保每个环境都有良好的性能可见性。本书中演示的“即插即用”工具在这方面大有裨益。当然，SQL 性能指标仍然是必需的，但你可以轻松使用开源的 Glowroot 或类似工具来实现。
4.  定期对你的代码施加负载，以识别明显的问题。请记住，“在负载下”是我们交付成果的一部分。本书中的大多数示例仅使用三个线程的负载和零思考时间就能展示性能问题。我称之为“3t0tt”。第 4、5、6 和 7 章重点介绍了负载生成。
5.  使用 **P.A.t.h. 清单**作为提醒，指导你从何处入手查找最常见的缺陷。请参阅第 9、10、11 和 12 章。第 1 章关于主要性能反模式的内容也会有所帮助。无论你是在生产中应对性能紧急情况，还是有其他开发优先级，用于排查性能问题的时间总是有限的。为了最大限度地减少排查时间，你可以轻松地在 30 分钟或更短时间内完成一次 P.A.t.h. 清单检查。

## 将本书与大型监控工具结合使用

尽管我一直在谈论小型、免费的监控工具，但我并不反对复杂的监控工具——甚至包括商业工具。例如 Java 分析器、APM 工具、网络追踪工具等。它们是我作为性能工程师日常工作的主要工具；我偶尔会使用一个许可证费用高达 7.5 万美元的工具。像 Glowroot 这样的新型开源 APM 工具正在成熟，也有一些不错的商业工具。我接受那些能正常工作的庞大工具。

但是，这些工具可能在一半甚至更多的环境中缺失，原因包括高昂的许可成本、对高开销的担忧以及其他因素。

我希望本书中介绍的“即插即用”的小型工具调优技术，能够为黑暗环境中的这些问题带来足够的光明，从而控制性能技术债务。

但是，本书中的负载生成技术、主要性能反模式、P.A.t.h. 清单以及其他调优方法，可以与任何环境（无论是黑暗环境还是非黑暗环境）中的任何监控工具集一起提供价值。

例如，如果在负载测试期间，Java 分析器或 APM 工具中的堆栈跟踪表明某个缓慢是由一个很少执行的业务流程引起的，那么你就知道应该从负载测试中移除该流程，因为它不会在生产中（以任何显著负载）遇到。这就是主要反模式 3：过度处理。

再举一个例子。高质量的 APM 工具能够提供我在第 3 章中概述的浩瀚指标海洋，我在那里谈到了指标的三个原因（检查我们是否满足了性能要求，评估我们的资源是否已用尽，以及识别导致缓慢的组件）。面对这片浩瀚的指标海洋，知道应该关注哪些可能会让人不知所措。嗯，这很简单。作为起点，确保你对 P.A.t.h. 清单中的每一项都有所关注。更具体一点，P.A.t.h. 清单应该指导你决定将哪些指标放在 APM “仪表盘”上。仪表盘是用户自行排列的一屏性能图表/指标。

“即插即用”工具的理念肯定缺少一样重要的东西：一个存储库。当你现场排查系统问题时，“即插即用”是一个很好的方法。但系统会在你不在场时出问题，拥有一个带有存储库的 APM 工具，能在你不在时捕获重要数据，这非常棒。缺乏存储库可能是“即插即用”工具集的致命弱点。话虽如此，“即插即用”工具始终是黑暗环境中的最后手段。

最后，请记住，第 4、5、6 和 7 章是关于负载生成的，无论你使用何种服务器端监控工具，这都是必需的。因此，当你足够关心可扩展性，并实际测试你的产品是否具备可扩展性时，可扩展性标尺就是为你准备的测试。



## 3t0tt（三线程负载，零思考时间）

我认为我们已非常接近这样一个节点：所有代码在提交前至少应接受一次轻量级的性能审查。以下是一种实现方式：向你的开发人员规定：

*   所有规模达到或超过 SOA 服务的组件变更，都应在恰好三个线程的负载且零思考时间的压力下进行审查。

我所说的“超过 SOA 服务”，是为了避免微基准测试的复杂性。为此，3t0tt 应仅包含至少有一次 I/O 操作的组件——例如一次或多次数据库访问，或一次或多次其他网络请求。将“极快”的代码排除在 3t0tt 测试之外也是合理的，同样是为了避免微基准测试的复杂性。或许可以划分出“必须 3t0tt”、“可选 3t0tt”和“从不 3t0tt”的响应时间基准。你可以自行设定阈值，但我会选择这样的数值：对于 25ms 及更快的代码，永远不做 3t0tt 测试；25ms 到 100ms 为可选；100ms 及以上则为必须。

为什么是三线程负载，而不是 42 或 95 线程负载？思路是，我们需要足够的活动来重现问题（包括多线程问题），但又不能有太多活动，以免给中等规模的调优环境（这是调优代码最高效的环境）带来过大的风险。

根据我的经验，三个线程足以暴露绝大多数多线程问题。将这种负载施加到存在多线程或可重入问题的代码上，你会看到 0.001% 到 10.000% 的请求以错误/异常告终。忽视这个“错误”指标，后果自负。JMeter 摘要报告和 JMeter 插件合成报告（一种更好的“摘要报告”）将此指标简称为“错误”。此外，如果你的代码习惯性地捕获并忽略异常，那么你就是在中断一个极佳的反馈流程——这个流程本可以帮助你利用 3t0tt 技术发现可修复的缺陷。

## 最后的话

你是否曾听同事说过这些话？

*   创建一个与生产环境规模匹配的性能环境成本太高。
*   网络部门不允许我们在那个网络上进行负载测试。
*   防火墙是否导致了性能问题？
*   重新部署那段代码需要一天时间。
*   为那条防火墙规则申请例外需要两天时间。

实际上，你永远不会听到最后那句话，因为你永远不可能这么快就修改防火墙规则。但除此之外，这些只是日常负载测试中所有障碍的一部分。这些都是合理的问题，但它们不能成为我们不让开发人员对其交付成果负责的借口——这些交付成果应包含在生产工作负载下可接受的性能。
