# 20. 采用 Scala



## 避免投入不足

在我开始认真研究 Scala 之前，我已经使用 Java 开发了十多年。我的第一次尝试只是浅尝辄止。团队自然持怀疑态度，因此我们决定将现有代码库的一部分划分出来尝试 Scala。我们将测试代码转换为 Scala，而主线开发仍保留在 Java 中。

事后看来，这是一个糟糕的主意。由于只处理测试代码，缺乏足够的规模或动力来推动改进。我们在测试代码中解决的问题类型已经相当成熟，现有的测试框架（如 JUnit）已经为我们解决了大部分问题。

似乎没有太多空间来真正使用语言特性。我们基本上只是用另一个恰好是 Scala 的测试框架重写了测试。我们几乎什么也没学到，最终将所有内容都回退到了 Java。这完全是浪费时间和精力。

我从中学到的一点是，要探索一门新语言，你需要解决真正的问题：设计问题、领域问题、业务问题。将代码库的一部分划分出来会限制你能探索的问题类型，从而限制你的学习。

## 避免用力过猛

下一次我有机会尝试 Scala 时，我走向了另一个极端。我直接跳入深水区，完全使用 Scala 进行开发，并极力尝试采用函数式编程、新模式、新架构设计……凡是与 Scala 相关的，我几乎都尝试了。

我之前没有做过太多函数式编程，所以很快就碰壁了。我面对的是完全陌生的概念和思想。我挣扎着，因为我试图在太短的时间内学习太多东西：新的框架、库和技术，以及一门新语言。我还没学会走就想跑。

我花了一段时间才意识到，尽管我是一名经验丰富的开发者，但在 Scala 和函数式编程方面，我实际上是个新手。我没有认识到这一点，事后看来，承诺以这种方式构建软件是不负责任的。我因专业知识而被聘用，却给自己设置了障碍，无法运用这些专业知识。

## 纯函数式编程是王道？

后来，我有幸与一些经验丰富的开发者共事，他们都精通函数式编程，且大多有 Haskell 背景。这很棒，因为我得以从他人的经验中受益，学习曲线也稍微平缓了一些。过了一段时间，团队开始采用非常高级的理念，很快我们就发现还有另一道坎要过。我们开始讨论更深层次的应用问题，以及是否能用函数式方法解决这些问题。

我在这里想说的是，对于不习惯函数式编程的团队来说，重量级的函数式编程理念可能相当陌生。即使在业界，我认为公平地说，完全拥抱这种风格的团队也寥寥无几。

我觉得，我们整体上还无法判断这种极端方式是否有助于解决现实世界的业务问题。它只是另一种做事方式。它确实代表了一种与主流截然不同的构建软件的方法，对我们来说，这可能有点过头了。它在团队中造成了一些分歧，有些人乐于尝试这种方法，而另一些人则不然。
