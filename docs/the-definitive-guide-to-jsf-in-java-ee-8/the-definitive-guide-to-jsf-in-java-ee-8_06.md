# 1. 历史

Bauke Scholtz^(1 ) 和 Arjan Tijms²

(1) 威廉斯塔德，库拉索

(2) 阿姆斯特丹，北荷兰省，荷兰

本章描述了 JSF 的历史，从其早期构想开始，一直讲到我们撰写本书时的现状。我们将讨论 JSF API（应用程序编程接口）本身是如何演变的，在此演变过程中发生了哪些重要事件，以及参与其中的一些关键人物。

这绝非完整的历史描述，读者应注意，实际发生的事件和参与的人员远多于我们在此所能提及的。

## 起初……

JSF 的历史相当悠久。其最初的 JSR，即 JSR 127，始于 2001 年。当时，Struts Web 框架极为流行，尽管它本身发布的时间也不算太久远（大约在 2000 年）。尽管 Struts 很受欢迎，但在 Java 领域仍有许多其他 Web 框架在使用，并且新的框架不断涌现。JavaServer Faces（JSF）的构想，是试图将标准化的 MVC（模型-视图-控制器）Web 框架基础引入整个 Java EE 平台。

在 Web 框架领域，争议屡见不鲜，JSF 也不例外。在其诞生之初就爆发了一场大争议：Apache 反对创建 JSF，理由是 Apache Struts 已经存在，而一个闭源的替代方案价值不大。因此，Apache 投票反对创建 JSF，并发表了以下评论：

> *此 JSR 与 Apache 开源项目 Struts 存在冲突。考虑到 Sun 当前不允许 JSR 在开源许可下独立实现的立场，我们认为在一个封闭环境中重新创建一项已在开放环境中可用的技术，价值甚微。*
>
> *就本 JSR 当前超越 Struts 的部分而言，我们鼓励提出此 JSR 的 Sun 开发者加入已经在领导 Struts 的 Sun 开发者行列，在 Apache 创建一个开放的解决方案，这样完成后才能确保能够以开源方式实现。*

最终，这场冲突得以解决。大约一年后，规范负责人艾米·福勒（Amy Fowler，曾因 Swing 闻名）被替换为克雷格·麦克拉纳汉（Craig McClanahan），他正是 JSF 据称要与之竞争的 Struts 项目的创始人。开源限制也被解除，名为 MyFaces 的开源 JSF 实现与（当时尚未命名的）参考实现及规范本身并行开发。MyFaces 最初于 2002 年 12 月在 sourceforge.net 上作为一个采用 LGPL 许可的项目启动，并于 2003 年 1 月发布了符合当时所谓“早期访问规范”的 0.1 版本。

开源实现是 Java EE 8 中最常见的实现方式，而在撰写本书时（2018 年），几乎没有任何 EE 规范仍以闭源方式实现。然而在 2001 年，这不仅是罕见的，对于新的 JSR 来说实际上是不被允许的。因此，允许开源实现是一个相当大的变革，而 JSF 有幸成为首个获此许可的规范。

尽管允许了开源实现，但规范的实际开发仍然秘密进行，不对外公开。没有公开的邮件列表，也没有供公众提交问题或表达愿望的跟踪系统（例如 JIRA 实例）。偶尔会有一些访谈，在 2002 年秋季，时任前规范负责人的艾米·福勒确实透露了关于 JSF 的不少细节，但总体而言，该项目对公众来说仍笼罩在神秘之中。

然而，JSF 背后的团队一直在努力工作。第一封发送到内部 JSR-127 列表的电子邮件是在 2001 年 8 月 17 日。与大多数项目一样，团队最初几个月用于收集需求并研究现有的竞争产品。同时，也选定了一个包名。最初的占位包名是 "javax.servlet.ui"，最终选择了 "javax.faces" 作为实际使用的包。首先考虑的技术架构是组件模型。对于基于组件的 MVC 框架来说，这显然是最重要的方面之一。在 2001 年的最后一个月和 2002 年的头两个月，团队研究了现在被称为托管 Bean（当时称为“对象管理器”）的概念。具有作用域、名称和依赖注入功能的托管 Bean 显然是 JSF 框架的另一个基石。在此期间，事件及其背后的模型也受到了关注。



2002 年第二季度，JSF 另外两大基石被提上议程：表达式语言（受 JSTL 启发），它对于实现所谓的从模板到后端代码的 bean 绑定至关重要；以及工厂查找器，它允许替换 JSF 的关键部分，尽管当时可能未被完全认识到，但这或许对 JSF 在约 16 年后仍具影响力做出了巨大贡献。

也正是在同一季度，Struts 之父、Tomcat Servlet 容器的架构师 Craig McClanahan 接任了规范负责人。关于使用 JSP 的讨论开始后不久，一场讨论——也许当时团队并未意识到——不幸地将在日后对 JSF 产生相当负面的影响。大约在 2002 年底，曾与 McClanahan 一样在 Tomcat 工作过的 Ed Burns 加入团队，担任联合规范负责人。Burns 最终成为了 JSF 的主要规范负责人，任期长达十多年。

当团队继续致力于诸如前述的受管 bean 和所谓的值绑定（即前述表达式语言绑定的 Java 表示形式）等工作时，第一片乌云在 2003 年春天出现：团队成员 Hans Bergsten 意识到，使用 JSP 作为 JSF 的模板语言存在非常现实且重大的问题。他向团队提出了这些担忧，但最终未被解决，反而在接下来的几个月里，团队将精力花在了值绑定的一个变体上；后来才清楚，方法绑定和状态保存机制是 JSF 另一个不太理想的实现。

JSF 1.0 及其当时尚未命名的参考实现最终于 2004 年 3 月 11 日发布——巧合的是，这仅比至今仍强大的另一个框架 Spring 1.0 的发布早了两周。MyFaces 在几天后的 3 月 19 日发布了其 1.0.0 alpha 版本。一个或许有趣的观察是，JSF 带着一个成熟的、基于 XML 的依赖注入框架正式发布，时间上恰好赶在以 DI 闻名的 Spring 正式发布之前。

JSF 1.0 普遍受到好评；尽管市场上竞争者众多，如 Tapestry、WebObjects、Velocity 和 Cocoon 等，但在发布后的几个月内，Horst Caymann 和 Hans Bergsten 等作者就出版了至少三本相关书籍，并且 eXo 平台（一个数字协作平台）立即开始使用 JSF。

然而，Hans Bergsten 早先的担忧几乎同样迅速地变得令人痛苦地清晰；JSP 技术基于从头到尾处理模板，遇到标签时立即写入响应。而 JSF 需要一种分阶段的方法，组件需要能够在向响应写入任何内容之前，检查并作用于从页面标签构建的组件树。这种不匹配导致了许多奇怪的问题，例如内容消失或渲染顺序错乱。

仅在 JSF 推出三个月后，Hans Bergsten 在他那篇传奇文章《通过抛弃 JSP 来改进 JSF》中，提出了放弃 JSP 的有力论据。Bergsten 在文中解释了 JSP 作为 JSF 模板语言是多么不合适，但他也带来了一线希望；由于 JSF 对可扩展性的出色支持，只需替换所谓的视图处理器（JSF 明确允许这样做），就可以相对容易地引入替代模板方案。然而，直到五年后，JSF 才真正附带了一个更合适的视图模板语言，尽管 JSP 在当时已基本被弃用，但在撰写本文时，它仍然存在于 JSF 中。

## 青春期

回到 2004 年，JSF 迎来了另一个第一；6 月 28 日，Ed Burns 宣布 Sun 发布了参考实现的源代码。这标志着一个重要的里程碑，因为在此之前，Sun 积极使用的大多数技术都是闭源的。最初，源代码是在有些奇特的 JRL 许可下发布的，但后来改为双重许可：GPL（附带类路径例外）和 CDDL。在发布这一公告的同时，也确立了一项传统：每个新功能或错误修复都应附带一个测试，并且在提交更改之前应执行所有现有测试。大约 14 年后，从事参考实现源代码工作的人员已大不相同，项目结构和代码约定也已改变，但测试驱动的传统仍以其原始形式得以保持。

那时，Ed Burns 决定将更多精力放在 JSF 的规范方面，因为 JSF 1.2 的规范工作已经立即启动，而早期团队成员之一 Jayashri Visvanathan 则承担了实现方面的领导角色，Ryan Lubke 担任 TCK（测试）工程师。

尽管 JSF 才问世几个月，各种 JSF 组件库已经开始涌现，尽管它们都是商业化的。其中包括 Oracle 的 ADF Faces。ADF Faces 在 JSF 1.0 正式发布之前就已列入 Oracle 的路线图，其首个早期访问版本于 2004 年 8 月 17 日发布。其负责人是 Adam Winer，他在创建 JSF 1.0 的团队中代表 Oracle。ADF Faces 主要包含一组丰富的组件，还有一个对话框框架，并且值得注意的是，它已经具备了部分页面渲染功能，这比后来出现的 AJAX 解决方案要早得多。ADF Faces 还包含一个实际可用的“for each”标签（af:forEach）。Adam Winer 在早期就解释说，这样的标签并非微不足道，但承诺 Oracle 会将相关知识回馈给 JSF 本身。

ADF Faces 组件主要源自早期的用户界面 XML 框架，Adam Winer 也是该框架的首席架构师。UIX 的早期版本曾使用过“Cabo”、“Baja”和“Marlin”等名称。UIX 是一个用于浏览器的富客户端框架。鉴于 JSF 与 UIX 有诸多相似之处，并且其负责人 Adam Winer 是原始 JSF 团队的成员，因此推测 UIX 影响了 JSF 或许并非不合理。这些相似之处包括：具有独立渲染器的组件概念、用于组合页面的 JSP 标签处理器和声明式选项，以及在 Java 中以编程方式实例化相同组件的能力。甚至存在概念上相似的数据绑定，尽管语法不够优雅。例如，UIX 不使用 value="#{user.age}"，而是使用 data:value="age@user"，但还需要在每个页面上定义一种生产者来声明“user”来自何处，然后将页面内容嵌套在该声明中。相比之下，JSF 和 EL 一直使用全局定义，并让用户自行避免名称冲突。

2004 年最早（如果不是最早的话）的开源组件库之一是 Matthias Unverzagt 的 OurFaces。由于当时 JSF 没有自己的资源 API 来提供图像等内容，OurFaces 需要在 web.xml 中添加一个 Servlet，即所谓的 SkinServlet（ourfaces.common.webapp.SkinServlet）。其重要性在于，这在当时的 JSF 库中变得相当普遍：要求用户在使用组件库之前，手动向 web.xml 添加一些内容。



2004 年末至 2005 年初的大部分时间，JSF 1.2 专家组（EG）都在处理各种 JSP 和 EL 问题，例如 JSTL 的`<c:forEach>`支持、JSP 中的 ID 生成，以及令人头疼的“内容交织”问题——即渲染时内容出现在响应中错误位置的现象。

尽管 OurFaces 可能是最早的组件库之一，但它并未持久，如今很少有人记得甚至听说过它。而另一个起源于 2005 年初的框架则截然不同，即 Alexander Smirnov 的 Telamon 框架，后更名为 Ajax4jsf。该框架是首批将 JSF 与当时新兴的 AJAX 技术相结合的框架之一。Ajax4Jsf 的巧妙之处在于，它可以通过将现有组件包裹在`<a4j:region>`标签中，为那些原本并未考虑 AJAX 支持的组件添加 AJAX 支持。这项技术被整合进了 Exadel 可视化组件平台（于 2006 年 3 月发布），该平台后来更名为 RichFaces，并成为最令人难忘的 JSF 组件库之一。

大约在同一时间，Alexander Smirnov 开始着手开发后来成为 RichFaces 的项目，而一家名为 ICEsoft 的公司也开始研发 JSF 组件库。ICEsoft 已运营数年，一直致力于开发名为 ICEbrowser 的基于 Java 的浏览器，以及名为 ICEbrowser beans 的产品——这些是“轻量级、可配置的 JavaBean 组件，可快速集成到 Java 客户端应用程序中”。在 2005 年 6 月 27 日的 JavaOne 大会上，ICEsoft 宣布了其 JSF 组件库——ICEfaces。该库同样基于 AJAX，但将 AJAX 直接集成到了组件中。ICEsoft 将其特定技术称为“专利申请中的 Direct-to-DOM™”，这基本上意味着来自服务器的更改会直接注入到网页的 DOM 树结构中。不过，最终版本并未立即发布，而是提供了早期访问版本。该版本是闭源的，但免费使用。

与此同时，受 Hans Bergsten 对 JSP 不适用性的担忧所启发，JSF EG 成员 Jacob Hookom 迎难而上，开始着手开发 Bergsten 所设想的替代模板语言。到 2005 年 8 月，这项工作已进展到可用的初始版本。这种模板语言的名字是什么？Facelets！它迅速席卷了整个 JSF 世界。Kito Mann 在第一个月就在 JSFCentral 上发表了关于它的系列文章的第一部分，几个月后，Richard Hightower 发表了著名的文章《Facelets fits JSF like a glove》。

Oracle 在 2005 年也没有停滞不前，在大约 16（！）个早期访问版本之后，于 2005 年底在安特卫普（现称 Devoxx）的 JavaPolis 会议上宣布，ADF Faces 将捐赠给 MyFaces，从而成为开源项目。

2006 年第一个月，Jacob Hookom 和 Adam Winer 思考了 JSF 状态保存机制的糟糕实现。该机制的工作原理是：首先从模板创建组件树，然后在请求接近尾声时，盲目地序列化整个树及其在请求期间可能放入的所有数据。在回传期间，树会从这种序列化形式中恢复（因此得名“恢复视图”阶段）。这是一种巨大的浪费，因为大部分信息已经在模板中可用。尤其是在使用客户端状态保存进行 AJAX 请求时，这会造成非常大的负担，而且在服务器上存储此状态时也是一个问题，因为它会极大地增加 JSF 的内存使用量。以这种糟糕方式进行状态保存的主要原因之一，再次与那个决定有关：支持 JSP。由于 JSF 1.2 即将定稿，不幸的是没有时间在 1.2 版本中修复这个问题。

尽管此时已经清楚 Facelets 是 JSF 的未来，但当 JSF 1.2 最终于 2006 年 5 月发布时，它仍然只包含 JSP。不过，情况也并非全然糟糕。得益于 JSF 和 JSP 专家组之间的合作，JSP 的修订版 JSP 2.1 发布了，它与 JSF 的需求更加契合。除此之外，JSP 的表达式语言和 JSF 的表达式语言被合并了。结果是 UEL（统一表达式语言）。UEL 的一个非常实际的优势是，JSF 组件不再需要手动将字符串转换为表达式，而是直接从模板语言接收`ValueExpression`。JSP 2.1 和 JSF 1.2 都成为了同时发布的 Java EE 5 的一部分。

2006 年 6 月 13 日，MyFaces 社区宣布捐赠的项目将更名为 Trinidad。不过，ADF Faces 在 Oracle 内部仍然存在，但它基于 Trinidad 并增加了一些额外功能（例如支持 Portals、JSR 227 等）。就在此两周前，即 2006 年 5 月 31 日，ICEsoft 宣布了其免费但仍是闭源的社区版。几个月后，即 2006 年 11 月 14 日，ICEsoft 在 MPL 许可下完全开源了 ICEfaces。当时仍为闭源并由 Exadel 销售的 RichFaces 也不会落后太久，大约四个月后，即 2007 年 3 月 29 日，Exadel 宣布与 Red Hat 合作，使 RichFaces 在开源许可下可用，并通过其 JBoss 集团提供和支持。



## 迈向成熟

2007 年 5 月 22 日，JSF 2.0 的规范制定工作正式启动。其范围雄心勃勃，不仅承诺修复人们长期抱怨的诸多问题，还将引入大量新特性。在 JSR 中提到的众多目标里，有一个从宏观角度来看尤为引人注目——将托管 Bean 设施从 JSF 中剥离出来，使其可供整个平台使用。

2007 年秋季，社区为 JSF 参考实现（RI）的名称进行了投票。四个名字脱颖而出，但如同常见情况，这些名字均未通过 Sun 公司法律部门的批准。最终，“Mojarra”被提出，或许令一些人意外的是，这个名字通过了法律审查。时任 JSF 主要提交者之一的 Ryan Lubke 于 2007 年 12 月 5 日正式宣布了这一消息。

大约一年后，2008 年 10 月 29 日，Çağatay Çivici 启动了一个新的库——PrimeFaces。其名称源于Çağatay 的昵称“擎天柱”（Optimus Prime），即虚构的《变形金刚》宇宙中英勇的汽车人领袖。Çağatay 长期参与 JSF 开发，此前曾致力于 YUI4JSF JSF 组件库的工作。PrimeFaces 最初基于 JSF 1.x，但随着 JSF 2.x 的临近以及该项目尚处早期阶段，它很快便转向了 JSF 2.x。

2009 年 7 月 1 日，期待已久的 JSF 2.0 终于问世。JSF 2.0 确实修复了业界对 JSF 几乎所有的抱怨；最终，Facelets 被纳入作为默认的视图模板语言。JSP 被有效弃用。Hookom 和 Winer 在三年前提出的状态保存问题也得到了解决；此后，JSF 仅保存增量状态（状态变更），并且在恢复视图时，组件树从模板重新加载，而非实际恢复。

多年来 JSF 社区提出的另一个重大关切——JSF 对回发的过度强调——也得到了解决；GET 请求在 JSF 2.0 中成为了一等公民。JSF 一个众所周知的可用性问题（有时被称为“陷阱”）是，对于某些操作，所涉及的数据在原始请求和回发期间必须保持一致。这在 JSF 1.x 中并非完全容易保证。JSF 2.0 为此引入了所谓的视图作用域，优雅地解决了该问题。自定义组件的创建——JSF 1.x 的另一个问题领域——也变得更加简单。JSF 2.0 还引入了对 AJAX 的核心支持（其模式借鉴了 Ajax4Jsf 的工作方式）、资源 API、系统事件以及其他诸多特性。

JSF 2.0 的目标之一——使其托管 Bean 设施可在 JSF 之外使用——通过 CDI 规范间接实现，该规范与 JSF 2.0 一同在 Java EE 6 中引入。CDI 规范本身也有悠久的历史，但其决定性特征之一是 CDI Bean 强烈基于 JSF 托管 Bean，并且本质上是后者的超集。

总而言之，所有这些修复和新特性的影响如此之大，以至于将社区基本一分为二：那些使用过 JSF 1.x 且再未回头的用户，以及那些转向 JSF 2.x 的用户，或者更具体地说，那些从 JSF 2.0 开始使用且从未接触过 1.x 的用户。这常常引发激烈争论，1.x 一方认为 JSF 糟糕透顶，而 2.x 一方则完全无法理解为何会如此。即使在撰写本文时（距 JSF 2.0 发布已近九年，比 JSF 1.x 存在的时间更长），这些情绪在某种程度上依然存在。

尽管 JSF 2.0 做了许多正确的事，但仍有一个错失的良机：即使 CDI 现已可用并取代了 JSF 的托管 Bean，JSF 却选择不立即弃用其托管 Bean 设施。更糟的是，它引入了一种基于注解的替代方案，以取代 JSF 1.x 用于定义托管 Bean 的基于 XML 的系统。既然 CDI 已经拥有像`javax.enterprise.context.RequestScoped`这样的注解，同时再引入一个功能完全相同的`javax.faces.bean.RequestScoped`注解，充其量只能说是值得商榷。专家组似乎意识到了这一冲突，因此设置了一个警告，指出这些新注解可能很快就会被平台功能所取代。

12 月 23 日，Cay Horstmann 在一篇题为“@javax.faces.bean.ManagedBean 是否已胎死腹中？”的文章中表达了他对这一非常不受欢迎情况的担忧。回应相当明确：包括 Java EE 书籍作者 Antonio Goncalves 在内的人们要求尽快纠正 JSF 2.0 犯下的这个巨大错误，并在即将到来的 JSF 2.1 维护版本中立即弃用`javax.faces.bean.ManagedBean`。该维护版本被要求（除其他事项外）纠正另一个错误（即 JSF 2.0 引入的问题：除了自定义`ResourceResolver`之外，还必须提供自定义`ExternalContext`，这一点非常不明确）。为何`javax.faces.bean.ManagedBean`在 JSF 2.1 MR 中未被弃用，至今仍是个谜。

虽然基于 JSF 1.x API 编写的应用程序大多可在 JSF 2.0 上无需更改或仅需少量修改即可运行，但组件库的处境则艰难得多。具体来说，平台提供的 AJAX 支持意味着现有组件库必须放弃自己的 AJAX 实现，并重新基于标准 API。显然，这绝非易事，组件库的迁移花费了很长时间，有些甚至从未真正完成切换。

在这方面，PrimeFaces 显然具有优势。作为一个相对较新且没有太多历史包袱的库，它进行切换相对容易。无论是否巧合，PrimeFaces 人气的上升似乎正是在 JSF 2.0 发布之后开始的，而这也正是 ICEfaces 和 RichFaces 似乎开始变得不那么流行的同一时间。尽管必须指出，确切的统计数据难以获取且包含多个方面（下载量、部署量、书籍、提问数量、可用职位、考虑不同行业等），但在 2012 年左右，PrimeFaces 似乎已成为更受欢迎的 JSF 组件库。

同年年初，2012 年 2 月 19 日，Arjan Tijms 和 Bauke Scholtz（巧合的是，他们也是本书的作者）启动了面向 JSF 的 OmniFaces 库。OmniFaces 的目标是成为 JSF 的实用工具库，本质上类似于 Apache Commons 和 Google Guava 之于 Java SE。Tijms 和 Scholtz 曾共同开发一个基于 JSF 的网站，发现两人都拥有一套在不同项目中重复使用的私有 JSF 实用工具，并且大量类似的实用工具在许多 JSF 项目中基本上被反复重写，部分散落在论坛消息和博客文章等地方。OmniFaces 的设立尤其不是为了与 PrimeFaces 等组件库竞争，而是为了与它们协同工作。因此，面向视觉的组件在很大程度上不在 OmniFaces 的范围内。



2012 年，JSF 2.2 的规范制定工作也正如火如荼地进行。JSF 2.2 最终于次年 5 月 21 日发布。JSF 2.2 特别提出了 Facelets 可运行的替代模式的正式版本：不再在视图上放置组件标签，而是放置纯 HTML，并通过一个特殊的 ID 将标签链接到组件。一般来说，这种模式对 JSF 开发者的吸引力稍弱，但特别受网页设计师的青睐，因为他们可以更轻松地使用纯 HTML 工具来处理此类视图。JSF 2.2 还引入了与 CDI 兼容的 `@ViewScoped` 注解，这消除了在 JSF 2.1 中仍需使用 JSF 托管 Bean 机制的最后一个理由，即在该版本中，`@ViewScoped` 仅适用于那些 Bean。JSF 2.2 还引入了两个重要的新特性：Faces Flow 和 Resource Contracts，但这些特性在实际中似乎很少被采用。

就在 JSF 2.3 启动之前，2014 年 7 月 20 日，RichFaces 负责人 Brian Leathem 在其博客上宣布，下一代版本的 RichFaces 5 将被取消。相反，RichFaces 将“追求稳定而非创新”，这意味着 JBoss 将使 RichFaces 4.x 与 JSF 2.2 兼容，并回溯移植一些原本为 RichFaces 5 开发的功能。虽然这篇博文有些乐观，但看起来 RichFaces 的结局已定。

2014 年 8 月 26 日，JSF 2.3 的规范制定工作启动。引入了一位新的联合规范负责人——Manfred Riem，他此前主要致力于 Mojarra 的实现方面，例如将数百个 JSF 著名的测试从古老且已退役的 Cactus 框架迁移到更现代的基于 Maven 的框架，并确保将大量未解决的 Mojarra 问题减少到可控数量。JSF 2.3 的开端或许有些引人注目，因为 Oracle 表示只有少量资源可用。在规范制定过程中，这些少量资源减少到了几乎无人能预料到的数字——绝对为零。基本上，在 JavaOne 2015 之后，几乎所有规范负责人都消失了，导致大多数规范工作突然停滞。Josh Juneau 在其著名的研究报告《Java EE 8，当前状态如何：自 2015 年底以来已完成工作的案例研究》中报道了此事，该报告通过展示电子邮件、提交和已解决问题的图表，无可辩驳地表明 Oracle 已经撒手不管了。

幸运的是，JSF 及其参考实现 Mojarra 的开放性使得规范制定工作及其在 Mojarra 中的实现很大程度上可以由其他 EG 成员继续进行，事实也确实如此。

与此同时，2016 年 2 月 12 日，Red Hat 宣布 RichFaces 将于同年晚些时候（即 2016 年 6 月）停止服务（EOL）。这个曾一度是最受欢迎的 JSF 组件库之一，常被称为“三大巨头之一”的项目，实际上已不复存在。2016 年 6 月 20 日，项目完成了最后一次实质性提交：“RF-14279: update JSDoc”。两天后，Red Hat 发布了 RichFaces 4.5.17，并将 GitHub 仓库设置为归档（只读）模式。仍是 JSF 2.3 EG 成员的 Brian Leathem 在几天后的 2 月 18 日宣布，他将不再从事任何与 JSF 相关的工作。

## 复兴

2016 年底，JSF 规范负责人短暂回归，但带来的消息是规范必须在短短几周内完成，以便启动（有些）冗长的最终确定流程。2017 年 3 月 28 日，JSF 2.3 最终发布，它带来了用 CDI 版本替换 JSF 原生工件的开端，并最终实现了本应多年前就完成的事情：弃用 JSF 托管 Bean 机制，转而使用 CDI Bean。其他特性包括：使用由 OmniFaces 捐赠的 Java EE WebSocket API 支持 WebSocket、系统中可用视图资源的自省，以及由 PrimeFaces 捐赠的搜索表达式框架。

继 JSF 2.3 规范有些动荡的开发过程之后，Oracle 在 2017 年宣布了更为动荡的消息：Java EE（因此也包括 JSF）将移交给 Eclipse 基金会。Oracle 将停止领导其之前拥有的规范，这同样包括 JSF。这意味着 Mojarra 将被重新授权，JSF 将通过一个新的流程来发展，并且很可能由不同的负责人领导。在撰写本文时，这一移交工作正在全面进行中。
