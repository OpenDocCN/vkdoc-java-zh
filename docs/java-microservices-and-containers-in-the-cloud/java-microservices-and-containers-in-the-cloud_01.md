# 1. 面向企业的微服务

自微服务的原始理念及其变体被讨论和考虑以来，已过去十多年。如今，这种架构风格已成为常态，而非例外。其工具和框架已足够成熟，架构原则也已在本地环境和云环境中得到验证和证明。对于这种软件架构如何区别于其前身，已有足够清晰的认识，因此本章仅讨论一些在其他地方未被提及的关键点。

本章首先概述计算范式，并解释当前微服务趋势的定位。你将了解到，微服务是分布式计算的另一种形式，但微服务的真正优势在于为微服务组件提供去中心化的业务事务处理能力。去中心化带来了对等微服务之间的自治性，从而为业务事务引入了一个显著特征，称为*最终一致性*。最终一致的系统具有更强的容错性，并为端到端业务事务提供可靠性，这些事务通常发生在相对不稳定的基于 HTTP 的传输协议上。接下来，本章将探讨微服务与“非微服务”之间的边界，这部分内容涵盖在服务粒度讨论中。在结束本章之前，我还会通过一个示例介绍*六边形架构*的概念。第 2 章中还有更多关于六边形架构的示例。

本章涵盖以下主题：

*   不同的计算架构范式
*   最终一致的软件系统
*   服务粒度
*   六边形架构隐喻
*   你的第一个 Java 微服务代码

## 计算架构范式

理解微服务架构风格中服务的分布式特性至关重要，因此本节将解释这种架构是什么，以及它如何区别于其他更熟悉的架构。



### 集中式计算

与如今常见且商品化的小型计算机或个人计算机相比，大型机拥有更强大的处理能力。早期版本的大型计算机有一个被称为*主机柜*的大型机箱，其中容纳了中央处理器和主存储器。大型计算机的特点是作为分时计算机运行的交互式用户终端，同时支持数百名用户和批处理。大型机用于商业事务处理，例如航空预订、银行业务等。一个事务包含一系列操作，包括磁盘 I/O、操作系统调用以及其他形式的数据从一个子系统到另一个子系统的传输，如图 1-1 所示。

事务处理或*批处理*是由中央大型机系统作为服务提供的，因此这种架构最好归类于*集中式计算范式*，其中中央计算机从头到尾做出关于业务事务成功完成的所有决策。

### 分布式计算

接下来是*分布式计算*范式，其中计算系统的不同组件位于不同的联网计算机上，它们通过相互传递消息来进行通信和协调行动。在此过程中，通常有一个节点或计算机充当业务事务的协调者——它协调（或编排）不同联网计算机上整个业务事务的基本计算组件，并决定事务的成功或失败。

分布式计算有多种变体。主要列举如下：

*   *客户端-服务器*：在客户端-服务器架构中，客户端从服务器请求和接收数据，并将其显示给用户。客户端机器托管了完成业务逻辑所需的大部分规则和验证。胖客户端是这种架构的特征，而客户端二进制文件的分发在此是一个麻烦事。

*   *三层架构*：三层架构将客户端智能迁移到中间层，以便可以使用无状态和瘦客户端。这简化了应用程序到单个中间件服务器的部署，并且客户端二进制文件的分发为零或最小化。大多数 Web 应用程序都是三层架构。

*   *N 层架构*：N 层架构，也称为多层架构，是一种软件工程设计模式，它将应用程序分离为逻辑层和物理层。每一层都有特定的职责和功能，并且各层通常由物理或逻辑边界（例如不同的服务器、进程或网络）分隔。

*   *点对点*：在这种架构中，没有特定的机器提供服务或管理网络资源。相反，所有职责均匀地分配给所有机器，称为对等点。每个对等点既可以充当客户端，也可以充当服务器。这种架构的例子包括 BitTorrent。比特币网络也是一种点对点网络，但比特币网络更符合另一种称为*去中心化架构*的计算范式，稍后将对此进行解释。

*   *微服务*：微服务是一种分布式架构，其中功能单元被识别为分布的基础。因此，在许多情况下，需要多个微服务才能完成单个业务事务。具有额外设计能力的微服务可以归类为另一种计算范式，称为*去中心化*，本章稍后将对此进行讨论。

### 去中心化计算

在去中心化计算中，应用服务由分布式网络中的各个计算设备或节点执行，不存在中心位置。这种软件开发（及分发）方式为开发者提供了极大的灵活性和成本节约，因为他们无需创建中央控制点。比特币、以太坊和 Uniswap 是几个采用去中心化计算的协议。

我不讨论图 1-3 为何描绘微服务架构的细节，因为我假设读者已熟悉这些基础知识。以下是该架构的几个特点：

*   *分布式*：业务应用根据功能（如机票搜索、预订和库存管理）进行划分和分布。
*   *独立性*：每个微服务可以独立于其他微服务进行开发和部署（前提是保持微服务间通信的 API 兼容性）。
*   *可用性*：如果某个微服务未运行，用户仍可通过其他正在运行的微服务获取应用功能（例如，若预订服务宕机，用户仍可进行搜索）。

因此，从某种意义上说，具备适当设计原语的微服务架构也可以作为去中心化应用运行（尽管“去中心化”这一术语更适用于涉及区块链的应用）。

## 最终一致性软件系统

理解微服务架构中服务的分布式特性至关重要。本节将解释其含义，以及它与其他更熟悉的架构有何不同。

### 原子一致性

图 1-4 展示了图 1-2 中的分布式架构，但此处用户正在预订航班。

![](img/619782_1_En_1_Fig4_HTML.jpg)

ACID 事务架构图。用户访问浏览器，浏览器包含用于可用性、预订、库存控制、服务和 ORM 的计算节点。每个进程的数据存储在独立的数据库中，并带有服务粒度和工件打包。

图 1-4

ACID 事务

参照图 1-4，典型的航班预订业务交易流程如下：

1.  浏览器将用户事务发送到业务层服务器。
2.  由于该事务是航班预订，因此由预订模块拦截。在执行预订之前，预订模块在启动一个 ACID^(²)事务后，向库存模块发送更新库存请求以减少座位库存。
3.  在先前启动的 ACID 事务内，库存模块更新库存数据库表。
4.  如果库存更新成功，预订模块将在同一事务上下文中，在其数据库表中创建预订记录。

如果预订表和库存表位于同一数据库，且预订和库存组件是同一中间层服务器中的两个模块，则该事务为本地事务。这意味着两个表的更新将以原子（ACID）方式要么全部成功，要么全部回滚。

### 最终一致性

本节考虑在微服务架构中相同的航班预订场景，此时处理过程是分布式的。见图 1-5。

![](img/619782_1_En_1_Fig5_HTML.jpg)

一种具有服务粒度和工件打包的分布式与去中心化架构。它包括多个用于可用性、预订或库存控制的计算节点。每个进程的数据存储在独立的数据库中。该架构连接到一个移动应用以供用户交互。

图 1-5

微服务间事务

相同的航班预订业务事务场景现在如下进行：

1.  预订应用将用户事务发送到业务层服务器。
2.  由于该事务是预订航班，因此由预订微服务拦截。在执行预订之前，预订微服务向库存微服务发送更新库存请求以减少座位库存。但是，你无法启动一个全局事务并在同一全局事务中发送更新库存请求。这是因为更新库存请求通常是同步的 REST 调用，并且不鼓励跨微服务的分布式事务。在没有分布式事务上下文的情况下，预订微服务会收到更新库存请求的成功或失败结果。
3.  库存模块更新库存数据库表，并返回成功或失败。
4.  如果库存更新成功，预订微服务将在其数据库表中创建预订记录，并向最终用户发送成功消息。

这里存在一个陷阱。如果更新库存数据库表的尝试意外成功，但预订微服务未能在其数据库表中创建预订记录，则这两个表之间的数据一致性就会出现问题。你需要手动干预来纠正此问题，并使这两个微服务恢复到最终一致状态。



### 最终一致性，去中心化系统

我已经发起了一场关于如何让微服务实现最终一致性的讨论。我还讨论了当更新库存数据库表成功，但预订微服务在预订数据库表中创建预订失败时的处理场景。现在考虑相反的场景——如果库存微服务没有运行，预订微服务无法完成预订，会发生什么？当从预订微服务到库存微服务的同步风格 REST 调用无法完成时，就会出现这种场景，因此预订创建也会失败。

图 1-6 展示了在图 1-5 所示架构基础上的改进，通过插入一个基于事件的通道来实现。

![](img/619782_1_En_1_Fig6_HTML.jpg)

一幅去中心化架构的示意图，展示了服务粒度、工件打包和事件通道。它包含多个用于可用性、预订或库存控制的计算节点。数据存储在独立的数据库中。该系统连接到一个用于用户交互的移动应用。

图 1-6

去中心化微服务

在这里，微服务之间并非紧密耦合。相反，你可以使用松散耦合的 JMS 或 AMQP 风格的消息传递协议。那么，航班预订业务事务的序列可能如下：

1.  浏览器将用户事务发送到业务层服务器。

2.  由于该事务是航班预订，因此被预订微服务拦截。在执行预订之前，预订模块向消息通道发送一条“航班已预订”消息。

3.  预订微服务将在其数据库表中创建一条预订记录，并向最终用户提供响应。

4.  “航班已预订”消息将被任何其他感兴趣的微服务（本例中为库存微服务）接收。

5.  库存微服务更新库存数据库表，并可选地向消息通道发送成功或失败消息。

通过将步骤 2 和步骤 3 保持在事务范围内，可以提高此业务事务的可靠性。尽管如此，设计中一个显著的变化是，业务事务跨越了多个微服务，并且事务的成功完成最终取决于这些不同微服务各自事务的整体完成情况。没有一个单一的微服务完全负责控制完整的端到端流程。相反，每个微服务执行自己的部分，从而使架构真正实现去中心化。

请注意，此讨论假设存在足够的库存。可能存在更复杂的场景，例如库存有限，但这超出了本文的讨论范围。

## 六边形架构隐喻

在学习松散耦合系统时，尤其是在微服务的背景下，讨论*六边形架构*是很有帮助的。我将在本节中进行讨论。

为了理解这个隐喻，我从典型的层次架构开始讨论。

### 端口与适配器架构

端口与适配器架构由 Alistair Cockburn 于 2005 年在其博客中提出。其描述如下：

“允许应用程序同等地由用户、程序、自动化测试或批处理脚本驱动，并且能够在与最终运行时设备和数据库隔离的环境中进行开发和测试。”

进一步解释，将应用程序视为系统的核心工件，所有输入和输出都通过一个或多个端口进出应用程序，这些端口将应用程序与各种外部工具、技术和交付机制隔离开来。应用程序不应知晓是谁/什么在发送输入或接收输出。这样做的目的是保护应用程序免受技术演进和业务需求变化的影响，因为这种演进可能会使应用程序在开发后不久就变得过时。

要理解端口与适配器架构，首先将分层架构从典型的南北方向转变为东西方向，如图 1-12 所示。

![](img/619782_1_En_1_Fig12_HTML.jpg)

图 1-12

分层架构，方向已改变

图 1-12 展示了典型的应用程序流程，该流程从用户界面中的代码开始，经过应用程序核心到达基础设施代码，再返回应用程序核心，最终向用户界面发送响应。

端口与适配器架构通过使用一个抽象层（以端口和适配器的形式实现）来解决保护和重用内层（业务层）的问题。

*   *端口*：端口是进入/离开应用程序的、与消费者无关的入口和出口点。在 Java 中，它是一个 Java 接口。或者用更抽象的术语来说，它可以是一个 HTTP REST 接口，或者是一个 JMS 或 AMQP 消息传递接口。

*   *适配器*：适配器类实现了一个其客户端已知的接口，并提供对客户端未知类实例的访问。适配器对象提供了接口所承诺的功能，而无需假设使用哪个类来实现该接口。

端口与适配器架构明确地标识了系统中三个基本的代码块：

1.  *接口*：使运行用户界面成为可能的代码，无论其用户界面类型如何。

2.  *业务*：业务逻辑代码，或称应用程序核心，用户界面通过它来驱动事务发生。

3.  *基础设施*：将应用程序核心连接到数据库、消息队列或第三方 API 等工具的基础设施代码。

在图 1-12 当前的方位中，左侧代表 UI 的适配器被称为*主*适配器或*驱动*适配器，因为它们发起对应用程序的操作。右侧代表与后端资源连接的适配器被称为*次*适配器或*被驱动*适配器，因为它们总是响应主适配器的操作。

主适配器或驱动适配器提供服务（给用户界面），而次适配器或被驱动适配器则消费服务（来自基础设施）。因此，在左侧，端口及其具体实现（用例）都属于应用程序内部。在右侧，端口属于应用程序内部，但其具体实现则属于外部。它包裹着来自某个提供者的外部基础设施。

### 六边形架构

图 1-12 展示了两侧——一侧是使用用户界面进入应用程序的入口，另一侧是离开应用程序进入基础设施的出口。因此，尽管应用程序有两个对称的侧（左侧和右侧），但每一侧都可以有多个入口/出口点。例如，一个 API（比如 REST API）和一个图形用户界面（GUI）可以是应用程序左侧的两个不同入口/出口点，而一个数据库和一个消息队列可以是应用程序右侧的两个不同入口/出口点。为了表示该应用程序可以有多个入口/出口点，你可以绘制一个具有多个边的应用程序图。这个图可以是任何多边形，但最终选择的是六边形，因此得名六边形架构。

return new ResponseEntity(product,

if (productFound == null) {
return new ResponseEntity(
HttpStatus.NOT_FOUND);
}
inMemoryDB.deleteProduct(productId);
return new ResponseEntity(
HttpStatus.NO_CONTENT);

method = RequestMethod.PUT,

return new ResponseEntity(
