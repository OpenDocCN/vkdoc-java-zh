# 同步对弈游戏

回到图 6-16。让我们在资源共享和资源利用的背景下进一步讨论。

你可能已经注意到，单个服务器套接字实例正在监听客户端连接。正是这个单一的服务器套接字负责处理所有传入的客户端连接。这是当今许多 Web 服务器的典型特征。一个典型的 Web 服务器，生成相对简短的内容（如包含文本或图像的网页），其响应大小约为 100KB。尽管生成或检索此页面只需不到一秒钟的时间，但将其传输到带宽仅为 80 kbps（@ 10 KB/s）的慢速客户端可能需要长达 10 秒的时间。向慢速客户端发送内容的问题，在一定程度上可以通过增大操作系统内核套接字缓冲区的大小来改善；然而，这并不是问题的通用解决方案。它可能会产生不良的副作用，需要巨大的资源，并且在达到某个最佳点后，资源将无法线性扩展。简而言之，这个单一的服务器套接字不能一直等待客户端连接上下文中发生的所有处理（如 I/O 操作）完成。如果它一直等待直到完成，那么它将需要很长时间才能接受下一个客户端连接。在生产级 Web 服务器中，会有数十、数百甚至更多的并发客户端连接访问 Web 服务器，因此服务器套接字必须更加智能，以便在合理的时间范围内做出响应。

图 6-16 中的架构利用线程来处理上述场景。线程是一组自包含的指令，操作系统可以调度这些指令在 CPU 核心上执行。应用程序并行运行多个线程有两个原因：

*   它们可以同时利用更多的计算核心。
*   它们使得并行执行操作变得容易（在此示例中，即执行在客户端连接上下文中编写的代码）。

然而，线程也会消耗系统资源。每个线程都使用内存和其他操作系统资源，并且需要通过称为上下文切换的过程在核心上换入换出。尽管现代服务器进程可以处理数百个活动线程，但一旦内存耗尽，或者高 I/O 操作迫使操作系统级别发生大量上下文切换，性能就会开始严重下降。因此，创建每个线程都必须谨慎。即使创建了线程，一种优化方法是重用线程。线程池可以实现这一点。此外，巧妙的设计还可以允许这些线程被重用以执行各种任务。在此示例中，来自同一线程池的线程被用于执行网络读取任务和网络写入任务。因此，总的来说，你的网络服务器通过使用线程池实现了智能实现，即使不是最智能^(⁹)的方式。有一个类比叫做“同步对弈或同步展示”。这指的是一种棋盘游戏展示（通常是国际象棋或围棋），其中一名玩家（通常是高段位玩家，如特级大师或段位棋手）同时与多名其他玩家进行多局游戏。这里实现的网络服务器类似于一位特级大师，他可以或多或少地并发接受连接，并将相关的计算分配给线程池中的线程，同样也是或多或少地并发执行。
