# 6. 调试与分析

调试与分析，除了编写代码之外，可能是使用 Apache NetBeans 最重要的两件事。调试是一门找出问题及其根本原因的艺术。Apache NetBeans 提供了一个功能非常丰富的调试环境，支持从多会话到多语言的各种场景。如果说调试能让代码正确，那么分析就能让代码运行得更快。Apache NetBeans 包含最先进的分析引擎之一，用于基于 Sun Microsystems 开发的 JFluid 技术进行内存和 CPU 分析。尽管 Apache NetBeans 支持多种不同编程语言和项目类型的调试，但本章重点介绍对 JVM 和 Java 语言的支持。

## 调试

### 入门

启动调试会话很简单。例如，我们使用 NetBeans 提供的 Anagram Game 示例。只需选择 *新建项目*，转到 *示例* 类别，展开树中的 *Java*，然后在 *项目* 列表中选择 *Anagram Game*。打开项目后，只需按下工具栏中的 *调试项目* 按钮，或使用项目节点上下文菜单中的菜单项即可。除了在调试器中运行项目外，Apache NetBeans 还支持将调试器附加到正在运行的进程。有不同的附加模式：*ProcessAttach*、*SocketAttach* 或 *SocketListen*。根据已安装的插件，还可以选择不同的调试器；例如，如果安装了 C++ 插件，则可以选择 `gdb`。

从左到右的操作如下：

*   **结束调试器会话 (⇧-F5)**。通过分离或终止被调试进程来结束调试会话，具体取决于调试器的使用模式。

*   **暂停调试器会话**。挂起被调试进程的所有正在运行的线程。

*   **继续调试器会话 (F5)**。恢复所有暂停的线程。

*   **单步跳过 (F8)**。将执行指针移动到下一行。

*   **单步跳过表达式 (⇧-F8)**。将执行指针向前移动一个表达式。

*   **单步进入 (F7)**。将执行指针向下移动一个堆栈层级，即进入此行中的方法调用。如果有多个方法调用，可以选择具体的方法调用。

*   **单步跳出 (**⌘**+F7)**。将执行指针向上移动一个堆栈层级，即跳出当前正在调试的方法。

*   **运行到光标处 (F4)**。将执行指针移动到光标所在的行。

*   **应用代码更改**。编译活动编辑器选项卡中的代码，并将更改后的代码推送到 JVM。此外，执行指针可能会在堆栈中向上移动，以适应新代码。

*   **切换在 GraalVM 脚本中暂停**。安排一个与 Truffle 多语言集成相关的挂起。要使用此操作，NetBeans 必须在 GraalVM 上运行。

*   **拍摄 GUI 快照**。为 Swing 组件图的当前状态拍摄快照，以便进行交互式检查和修改。

要进行真正的调试会话，最后需要的是*断点*。断点定义了调试器将自动挂起线程的位置，该线程恰好运行到定义的断点处。创建新断点最简单的方法是：在编辑器的装订线中单击正确的行，或者将光标放在调试器应暂停执行的行上，然后按 ⌘**-F8**（在 Mac 上）或 Ctrl+F8（在 Windows/Linux 上）。同样的操作可以用于移除断点，即触发的操作称为*切换断点*。一旦创建了断点，它会在装订线中用特殊符号标记出来。如图 6-4 所示。

调试视图

在视图底部有几个按钮，用于控制视图的行为（选项）。从左到右依次是：

*   **显示更少**。切换视图中显示的线程。默认只显示挂起的线程，但你可以通过此按钮在所有线程和仅挂起线程之间切换。

*   **显示线程组**。切换线程组的可见性，将列表样式视图切换为树形样式视图，并按线程组对可见线程进行分组。

*   **显示挂起/恢复表**。切换视图右侧列的可见性，显示线程的实际状态，即暂停或运行。

*   **显示系统线程**。切换系统线程的可见性。默认情况下，系统线程是隐藏的。

*   **显示监视器**。切换线程获取的监视器（锁）的显示。默认情况下，监视器不显示。

*   **显示限定名称**。切换完整限定类名的显示。这有助于识别显示在栈中的特定类实际来自何处。

*   **按挂起/恢复状态排序**。根据线程状态切换线程列表的排序顺序。

*   **按名称排序**。根据线程名称切换线程列表的排序顺序。

*   **按默认排序**。切换到默认排序顺序（按名称排序）。

除了可用于配置视图的选项外，该视图还允许与线程进行交互。首先要控制的是线程状态。通过单击视图右侧显示的按钮，可以挂起正在运行的线程，也可以恢复已挂起的线程。当应用程序在调试期间确实需要多个线程，并且需要按需挂起线程以确保正确的应用程序状态时，这可能会很有帮助。此外，多个线程经常会在当前不感兴趣（但仍需保持激活状态）的断点处停止。使用这些控件，可以恢复这些线程。

另一个有趣的技术是调查调用栈中不同帧的状态。活动帧以粗体突出显示（如图 6-5 所示），可以通过双击另一个栈帧条目或使用栈帧的上下文菜单项 *设为当前帧* 来更改。

栈帧的上下文菜单还有一个额外的有趣功能。除了在不同帧处检查栈之外，菜单项 *弹出至此* 允许将栈展开到选定的栈帧。这是一个很好的小帮手，因为它允许重试代码片段——尤其是在它们不修改状态的情况下。

如果上下文菜单不是直接在调用栈条目上调用的，你会得到一些额外的辅助操作。其中两个很直接，即 *恢复所有线程* 和 *挂起所有线程*，但第三个更有趣。*检查死锁* 工具对于处理棘手的多线程情况非常有用，因为死锁通常难以检测和调试。

*调试*视图中还隐藏着更多功能，但我们稍后会在多会话调试的上下文中进行探讨。

### 变量视图

*变量*视图用于检查（和更改）调试过程中所触及的所有对象的状态。它分为三个主要区域。

首先，显示来自*计算表达式*视图的代码片段求值结果（粉色图标）。

其次，显示一个监视列表（蓝色图标）。*监视*是一种对简单求值结果的持久化快捷方式，例如，在变量层级结构中位于较深层次的某个特殊字段，即使变量列表在调试器上下文变化（例如，调试过程中离开某个类）时会刷新，该字段也能保持静态可见。如果你暂时不需要更新监视，可以通过取消选中显示监视前的复选框来禁用每个监视。

第三，显示变量列表，例如，*this* 和*局部变量*（绿色图标）。

与其他视图一样，*变量*视图也可以自定义。自定义可以通过使用视图左侧的按钮之一来完成。从上到下依次为：

*   **在变量视图中显示求值结果**。切换是否在*变量*视图中显示来自*计算表达式*视图的代码片段求值结果。

*   **在变量视图中显示监视**。切换是否在*变量*视图中显示已定义的监视。

*   **创建新监视**。打开一个对话框，允许用户输入要监视的表达式。文本字段支持代码补全，以便更快地创建监视，减少输入量。

*   **打开格式化选项**。打开首选项窗口（选项窗口），允许用户定义特殊类型的对象在*变量*视图中的呈现方式。

*   **在可用时，将值显示为可视化属性编辑器**。如果 IDE 中为变量（字段）的类型注册了属性编辑器，则使用该属性编辑器来呈现/编辑该类型，而不是使用已定义的格式化逻辑。

另外，也可以通过双击标签 *“<输入新监视>”* 来创建监视。出现的文本字段也支持代码补全，如图 6-14 所示。

![../images/479166_1_En_6_Chapter/479166_1_En_6_Fig14_HTML.jpg](img/479166_1_En_6_Fig14_HTML.jpg)

*变量*视图中不同项目上的上下文菜单有助于进一步交互：

*   **删除**。删除一个监视：无论是固定监视、固定监视还是默认监视。

*   **全部删除**。删除所有监视。

*   **创建固定监视**。创建一个不基于当前上下文，而是基于全局上下文的监视。

*   **编辑**。打开一个对话框，允许更改要监视的表达式。文本字段支持代码补全，以便更快地创建监视，减少输入量。

*   **显示固定监视**。显示所有通过在源代码视图中固定表达式求值而创建的监视。

*   **新建监视**。打开一个对话框，允许用户输入要监视的表达式。文本字段支持代码补全，以便更快地创建监视，减少输入量。

*   **转到类型源代码**。在编辑器选项卡中打开附加到该类型的源代码。如果没有可用的源代码，则显示反编译的替代内容。

*   **查看方式**。切换视图组件，使用表格或树形表格来显示数据。

*   **显示引用**。打开*堆遍历器*视图，检查所选变量的实际引用网络。

*   **标记对象**。为对象附加一个标签，以帮助在后续调试过程中识别该对象。

*   **仅显示满足条件的行**。根据变量类型从视图中过滤行。提供不同的过滤类型。

有了所有这些功能，*变量*视图是一个非常强大的工具，可以帮助你了解应用程序的状态。在 Apache NetBeans 中，此视图还附带其他几个从不同角度支持调试过程的视图。我们首先深入了解的是*监视*视图。

### 断点

断点是调试的关键功能，因为它们允许在代码的任意位置暂停被调试的进程。在 Apache NetBeans 中，可以使用*断点*视图跨源文件管理和显示断点。除了将断点视为非结构化列表外，断点还可以分组到自定义或预定义的组中，例如，按编程语言分组的*断点组*。一个断点可以属于一个自定义组，并且如果需要，可以从一个组移动到另一个组。在我们深入探讨这些细节之前，让我们先创建一些新的断点。

在 Apache NetBeans 中，有多种创建断点的方法：

*   通过在装订线（显示行号的区域）中切换断点，可以创建或删除*类断点*、*字段断点*、*行断点*或*方法断点*。
*   通过使用主菜单中*调试*菜单下的菜单项*新建断点*（Mac 上为 ⇧-⌘-F8，Windows/Linux 上为 Ctrl+Shift+F8），可以创建所有类型的断点。

使用“新建断点”操作创建新断点可以在创建时对断点进行完整配置。第一步是选择调试器，这反过来会更改可用的断点类型列表。对于 *Java* 调试器，断点类型如下：

*   **类**。根据类状态暂停被调试的进程。
*   **异常**。根据抛出的异常暂停被调试的进程。
*   **字段**。如果访问了某个字段，则暂停被调试的进程。
*   **行**。如果到达文件中的特定行，则暂停被调试的进程。
*   **方法**。根据方法相关状态暂停被调试的进程。
*   **线程**。根据线程相关状态暂停被调试的进程。
*   **AWT/Swing 组件**。根据组件相关事件暂停被调试的进程。

所有断点类型都允许配置附加属性。它们分为两类：*条件*和*操作*，并且可能因断点类型而异。“新建断点”对话框如图 6-20 所示。

![../images/479166_1_En_6_Chapter/479166_1_En_6_Fig20_HTML.jpg](img/479166_1_En_6_Fig20_HTML.jpg)

图 6-20

*操作*属性描述了到达断点时会发生什么。每种断点类型都有四种不同的关联操作。

*   **挂起**。允许选择到达断点时将挂起哪些线程。可能的值有*断点线程*（到达断点的线程）、*所有线程*或*无线程（继续）*。选择*无线程*将仅打印*打印文本*操作下的文本，但不会停止任何线程。这在只需要不同点的值列表，但不需要与被调试进程进一步交互时很有用，例如，在多线程应用程序中，停止线程会改变运行时行为。
*   **启用组**。一旦到达此断点，则启用选定的断点组。结合下一个操作，这可以实现某种基于上下文的自动断点控制，以减少在代码特定区域手动启用或禁用断点的工作量。
*   **禁用组**。一旦到达此断点，则禁用选定的断点组。结合上一个操作，这可以实现某种基于上下文的自动断点控制，以减少在代码特定区域手动启用或禁用断点的工作量。
*   **打印文本**。一旦到达断点，将配置的文本打印到调试器控制台。

到达断点时打印文本使用一种特殊的语法来表示动态信息。动态信息由 `{}` 括起来，而外部的所有内容都是纯文本。支持以下动态语法：



*   `{=<expression>}`。根据到达断点时的上下文，打印表达式求值结果（见上文）；
*   `{className}`。打印定义断点的类名；
*   `{exceptionClass}`。打印与断点定义相关的异常类名；
*   `{exceptionMessage}`。打印与断点定义相关的异常消息；
*   `{fieldName}`。打印定义断点的字段的实际名称；
*   `{lineNumber}`。打印实际行号；
*   `{methodName}`。打印定义断点的方法的实际名称；
*   `{threadName}`。打印到达断点时的线程名称；
*   `{? threadStarted}{}{}`。根据`threadStarted`的状态，打印第一组或第二组中的文本。两组均可包含动态语法元素。

并非所有动态元素都适用于所有断点类型，例如`threadStarted`。*打印文本*操作文本字段中已预填了一个示例文本。

描述调试器在到达断点时停止所依据约束的*条件*，并非对所有断点类型都相同，但大多数断点类型遵循以下两个约束：

*   **条件**。描述一个表达式（见求值），该表达式必须求值为真，调试器才会停止。
*   **命中次数达到时中断**。描述断点必须被命中多少次，调试器才会停止。

命中次数有不同的比较器可用：

*   **等于**。命中次数必须等于定义的值，调试器才会停止。
*   **大于**。命中次数必须大于定义的值，调试器才会停止。
*   **倍数**。命中次数必须是定义值的倍数，调试器才会停止。

## 性能分析

### 入门

为在 Apache NetBeans 中打开的项目启动性能分析会话并不像调试那么简单，因为它需要更多的设置决策。因此，让我们逐步了解启动性能分析会话的过程（作为示例，我们将仅使用 Apache NetBeans 中的 Anagram Game 示例）。如果项目已打开，请在*项目*视图中选择项目节点，然后按工具栏中的**分析项目**按钮。现在需要执行的第一步是配置您的性能分析会话。必须执行此操作，因为 Apache NetBeans 分析器支持多种性能分析模式：

*   **遥测**。监控 CPU 和内存使用情况、线程数以及已加载的类。

*   **方法**。分析方法执行时间和调用次数，包括调用树。

*   **对象**。分析已分配对象的大小和数量，包括分配路径和代际计数。

*   **线程**。监控线程状态和时间。

*   **锁**。收集锁竞争统计信息。

*   **SQL 查询**。显示已执行的 SQL 查询、其持续时间和调用路径。

分析远程 VM 需要更复杂的设置，这需要使用特殊代理设置远程 JVM。

我们现在将更深入地了解所有性能分析模式，但首先了解一些基础知识。

### 基本性能分析操作

在分析应用程序期间，有一组通用操作，可以随时触发以支持性能分析过程。这些操作位于标有*进程*的性能分析工具栏的右端。

*   **从被分析进程获取线程转储**。从被分析进程获取线程转储，并在*线程转储*视图中打开它。

*   **从被分析进程获取堆转储**。从被分析进程获取堆转储，并在*堆遍历器*视图中打开它（请参阅本章后面的*堆遍历器*）。

*   **在被分析进程中请求垃圾回收**。请求在被分析进程中运行显式垃圾回收。这可用于确保堆转储不会被太多不再存活的对象污染。

### 遥测

*遥测*是一种非常基础的性能分析模式。它允许我们跟踪以下内容：

*   CPU 负载，区分实际 CPU 时间和 GC 消耗的时间。

*   堆大小和已使用的堆。

*   关键垃圾回收参数，例如，存活代（持续增加的数字表示缓存或内存泄漏）和 GC 间隔。

*   线程和已加载的类。

### SQL 查询

对 SQL 查询进行性能分析只是锦上添花。对应用程序 SQL 查询的 SQL 性能分析会收集语句的跟踪和执行时间。它还可以跟踪与数据库连接的建立，这可能有助于调整您的连接池。



### 堆遍历器

*堆遍历器*是一套用于分析通过 OpenJDK 内部堆转储机制获取的堆转储文件的工具集（在性能分析过程中也可以通过某些操作触发）。它是一个非常全面的分析工具，具备多种剖析堆转储文件的能力。

首先，有一个*摘要*视图，可按多个类别显示与堆转储相关的数据：

*   **基本信息**。显示基本信息，例如文件大小、转储日期。

*   **环境**。显示有关环境的信息；例如，快照是在哪个操作系统上获取的；JVM 的供应商和版本等详细信息。

*   **系统属性**。显示被转储 JVM 的所有系统属性。这些属性会内联显示在视图的实际区域中。

*   **堆转储时的线程**。显示与堆转储一起获取的线程转储。线程转储会内联显示在视图的实际区域中。

*   **应用程序窗口**。列出所有顶级 Swing 窗口，并可直接访问*实例*视图中的对应实例。

*   **最大对象**。根据对象树的保留大小（即属于该对象的所有内容），触发确定堆上最大对象的操作。由于需要计算转储中所有对象的保留大小，此操作可能需要一些时间。

在本章中，我们详细了解了 Apache NetBeans 的两大核心功能：*调试*和*性能分析*。

您已经学会了如何使用调试器来解决应用程序代码中的问题、调试多线程应用程序，以及从调试器与您的应用程序进行交互。

本章展示的这两个功能都易于使用，并且应该成为您开发者工具包的一部分。现在，我们将进入第二部分：“扩展 Apache NetBeans”。
