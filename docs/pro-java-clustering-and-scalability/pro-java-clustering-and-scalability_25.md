# 21. 懒部署 vs. 快速部署

也许你曾在一家生产环境部署流程非常“懒散”的公司工作过。也就是说，部署一次可能需要很多天，甚至好几个月。信不信由你，在实践中，这种情况并不难找到。

简而言之，这类问题往往是软件项目失败的根源。这是因为将新版本部署到生产环境需要如此长的时间，会带来诸多弊端。

*   客户不会乐意看到那些能为他们业务增值的功能，只停留在开发人员的机器上，而不是部署到生产环境。
*   新版本会包含大量新代码，因为开发人员长时间都在开发新功能，很可能一次性向应用程序引入许多错误。这使得修复这些（以及其他）错误变得更加困难。
*   由于部署间隔时间过长，团队将无法及时修复关键错误并交付软件。
*   开发人员辛勤工作，却看不到自己的代码上线运行。这真的很打击士气。

如你所见，不采用懒散部署流程的理由相当充分。

![A453568_1_En_21_Figa_HTML.jpg](img/A453568_1_En_21_Figa_HTML.jpg) 我推荐 Jez Humble 和 David Farley 合著的《持续交付》（Addison-Wesley, 2010）一书，它是对持续交付精彩世界的极好入门介绍。

相比之下，能够在几小时甚至几分钟内将代码交付到生产环境，则带来了许多好处，包括：

*   客户会很高兴，因为他们每周（甚至每天）都能使用新功能，从而为他们的业务增加价值。
*   你得到的是低风险的发布。每个新版本只包含一个小功能。如果出现问题，查找和修复起来要容易得多。这显然也意味着更低的成本。
*   错误修复可以非常迅速地交付。
*   开心的开发人员能看到他们的工作为他人创造价值。

如你所见，有充分的理由尽快将新版本发布到生产环境。
