# 6. 理解领域与架构之间的关系

如果不深入理解开发过程中将面临的挑战，就不可能创建出可扩展、高性能、安全、高可用且经济高效的架构。你首先必须深入理解的是应用程序将要构建的领域。这样，你才能为任务选择最佳的技术。

你必须特别关注非功能性需求，因为它们将告诉你架构应该有多健壮。此外，无论预算如何，你都必须始终平衡成本。你可能在为一家能向项目投入大量资金的公司工作，也可能在为一家更节俭的公司工作。作为专业人士，你应该能够为这些情况提供替代方案。这就是为什么你必须了解广泛的技术、供应商、语言等。

在处理新项目时，你应该基于应用领域问自己一些问题。显然，领域本身可以（也将会）随时间变化，你需要调整架构并在必要时做出更改。例如，假设你正在创建一个支付系统。你将如何对以下非功能性需求进行优先级排序：性能、可扩展性、安全性、可用性和易用性？

有些人可能会回答所有这些需求同等重要，虽然这听起来有道理，但并非正确答案。处理支付交易延迟两秒钟会毁掉你的公司吗？不会！另一方面，与支付相关的严重安全漏洞会毁掉你的公司吗？会的，尤其是当它公之于众时（如今这很容易），因为你的客户将不再信任你的公司。因此，在这个例子中，安全性可以被认为比性能具有更高的优先级。请注意，优先级是根据领域来定义的。

在确定了优先级列表后，你可以通过更多问题深入探讨每个需求。让我们考虑一下搜索引擎的性能需求。

*   搜索引擎是产生更多的写入还是读取？读取，对吧？所以，你已经可以排除许多技术了。
*   对于搜索引擎来说，搜索必须比简单的 SQL `LIKE` 智能得多。即使查询字符串存在词法错误，它也必须返回相关结果。这进一步缩小了你可以使用的技术范围。
*   当你进行水平扩展并提高读取性能时，一种常见做法是实现复制策略，以便相同的数据可以在集群的不同节点上用于读取。因此，你选择的技术应提供诸如集群和节点间复制等功能。

在考虑了诸如此类的问题后，假设你最终需要在 Solr 和 Elasticsearch 之间做出决定。这很好。你的问题已经缩小到仅在两种技术之间做选择。但即便如此，这也可能不是一件容易的事！

你团队的规模和技术水平也会影响你的选择。例如，如果你有一个在基础设施主题方面知识有限的小团队，也许你可能更愿意选择托管服务（软件即服务），而不是面对节点故障、可扩展性等基础设施挑战。

这只是一个例子，但软件架构的全部意义就在于做出这些类型的决策，以及更多其他决策，以满足应用领域的需求及其变化。

最大的问题在于，有数百万种技术可以解决相同的问题，尤其是在开源世界中。因此，要成为一名准备充分的专业人士，你必须每天努力学习，跟上时代的步伐。经常阅读工具的变更日志和路线图甚至也很重要。

回到聊天应用程序，我通过本章讨论的相同过程选择了一组技术。正如你可能想象的那样，你在本书中将使用的架构并非唯一可能，但它似乎是一个不错的选择！
