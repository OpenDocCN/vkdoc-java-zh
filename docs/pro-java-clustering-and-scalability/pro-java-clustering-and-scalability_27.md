# 23. 自动化测试的类型

可以编写的自动化测试有很多种类型。它们各有优缺点。以下是最常见的几种：

*   **单元测试**：提供快速反馈，但只测试代码中一个隔离的单元（例如，类中的一个方法）。例如，SQL 查询的问题不会被发现，因为单元测试不会访问数据库。单元测试通常只需几秒甚至几毫秒就能执行完毕。
*   **集成测试**：提供较慢的反馈，但会测试与数据库或 Web 服务等外部系统的集成。也就是说，它们比单元测试更全面。（但这并不意味着它们更好！这里要小心；记住，每种测试类型都有其自身的优缺点。）通常，集成测试需要几分钟才能执行完毕，但具体时间可能因运行的测试数量而有很大差异。
*   **验收测试**：提供的反馈比集成测试更慢，但它们实际上是在模拟系统中的用户，使用浏览器点击按钮和链接、填写并提交表单等操作。通常，它们需要几分钟才能执行完毕，但具体时间可能因运行的测试数量而有很大差异。
*   **压力测试**：包括向系统发送大量请求，以获取系统在这些条件下的行为反馈。

通常，单元测试和集成测试是最常用的。它们共同覆盖了许多场景，并避免了大量应用程序问题。此外，与其他类型的测试相比，它们也更容易编写（考虑到压力测试应在类似生产环境的环境中运行等）。

![A453568_1_En_23_Figa_HTML.jpg](img/A453568_1_En_23_Figa_HTML.jpg) 实际上，还有更多类型的自动化测试，例如安全测试，可以添加到应用程序的自动化测试套件中。开放 Web 应用程序安全项目¹（OWASP）是一个在线社区，它在 Web 应用程序安全领域创建了免费的文章、方法论、文档、工具和技术。例如，OWASP Zed Attack Proxy²（ZAP）是世界上最受欢迎的免费安全工具之一，由数百名国际志愿者积极维护。该工具可以帮助你自动发现 Web 应用程序中的安全漏洞，并且可以轻松集成到你的持续交付管道中（它甚至有一个可用的 Jenkins 插件³）。
