# 9. 命名空间

命名空间是一种特性，允许将函数、符号和变量分类到定义良好的组中。在本章后面，我们还会看到命名空间可以被转换并映射到实际的文件系统目录结构。

让我们先使用 REPL 来演示一个关于命名空间如何工作的简单示例。当我们运行 `lein repl` 时，会自动进入一个新的 `user` 命名空间，并且该命名空间预加载了 Clojure 核心库。你可以通过查看 REPL 控制台来判断当前所在的命名空间，控制台会在光标位置之前显示当前命名空间：`user=>`。

注意

如果你需要以编程方式引用命名空间，可以使用 `*ns*` 变量。

现在，让我们看看如何创建一个新的命名空间 `foo.bar`，然后在该新命名空间内定义一个分配给符号 `baz` 的变量。接着，我们回到 `user` 命名空间，为 `baz` 定义一个不同的值，并观察从不同命名空间引用该变量时会发生什么。

```
(ns foo.bar)
(def baz 1)
baz
;; 1
(ns user)
(def baz 2)
baz
;; 2
(ns foo.bar)
baz
;; 1
代码清单 9-1.
创建新命名空间的示例
```

在代码清单 9-1 中，你可以看到我们使用了 `ns` 宏来创建 `foo.bar` 命名空间。如果该命名空间已经存在（例如代码清单 9-1 中的 `user` 命名空间），那么调用 `ns` 宏并传入一个映射到已存在命名空间的符号，将会直接把你移入该指定命名空间，而不会先创建它。

因此，在前面的示例中，我们首先创建了 `foo.bar` 命名空间，然后将 `baz` 定义为值 `1`。接着，我们移回 `user` 命名空间并定义 `baz`，但这次赋予了一个不同的值——`2`。

从这个例子中，你可以看到我们如何根据定义和包含它们的命名空间，安全地定义具有不同行为和值的变量和函数。

## 什么是命名空间？

Clojure 的命名空间实际上只是一个从符号到变量/类的全局映射。每个命名空间类也可以包含自己的映射（主要是符号到变量的映射）。当前命名空间存储在一个 `Namespace` 对象中，并与动态变量 `*ns*` 关联。

注意

默认情况下，`user` 命名空间，以及随后使用 `ns` 宏创建的任何命名空间，都会预加载并映射 `clojure.core` 命名空间中的所有函数（以及 `java.lang` 命名空间中的类名）。

你可以在 REPL 中使用 `in-ns`（而不是 `ns`）创建一个新命名空间来验证这一点。`in-ns` 函数创建的命名空间不会预加载 `clojure.core` 命名空间。请参见代码清单 9-2 中的示例，该示例演示了新命名空间将缺少默认命名空间。

```
(in-ns 'beep)
;; beep 命名空间不存在
;; 所以这创建了它并将我们移入该命名空间
(loaded-libs)
;; 此函数位于 clojure.core 中
;; clojure.core 命名空间尚未预加载
;; 因此这将在 REPL 中显示错误
(clojure.core/refer 'clojure.core)
;; 我们显式加载 clojure.core 命名空间
(loaded-libs)
;; 我们现在成功执行了该函数
代码清单 9-2.
使用 in-ns 创建的新命名空间将缺少 clojure.core
```



## 加载命名空间文件

如果你正在从 REPL 测试现有应用程序，并希望`load`一个需要其他依赖项的代码文件，那么你需要使用`load`函数。`load`函数会尝试在你的“类路径”中找到给定路径，然后`load`该文件。类路径是 Java 用来识别代码库位置的机制，由于 Clojure 构建在 Java 之上，这意味着我们需要确保类路径中添加了相关路径。

项目管理工具 Leiningen 通过 `project.clj` 文件处理依赖项和库的添加。因此，只要你在其中列出了依赖项，就应该能够使用`load`函数将它们加载到 REPL 会话中。有关`load`函数的示例，请参见清单 9-3。

```
(load "/path/to/namespace")
(in-ns 'namespace)
清单 9-3.
从外部文件加载代码
```

现在，你在使用`in-ns`之前使用`load`的原因是，即使你有一个定义了指定命名空间的文件，除非你将其添加到 JVM 类路径，否则 REPL 无法定位它。`load`函数会将命名空间添加到类路径。

注意

确保路径以正斜杠开头，并且相对于 Leiningen 的 `src` 目录（参见下一条注释）；这是因为 Leiningen 工具会为每个 Leiningen 项目修改操作系统的 JVM 类路径。

虽然我们还没有介绍 Leiningen 项目自动化工具，但值得提及的是，Clojure 项目设置的一部分要求使用一个 `project.clj` 文件来引导应用程序。

如果你想跟着我接下来要解释的内容操作，最好先退出 REPL，然后在终端中运行以下命令：`lein new compojure-app foo`。这将为你生成一个新项目。进入该新项目目录（`cd foo`），你会发现有一个新的 `project.clj`。

花些时间查看这个新 Leiningen 项目中文件夹和文件的组织结构。现在不必担心这一切意味着什么或如何工作，我们将在后面的章节中介绍。但命名空间在设置一个真实的 Clojure 应用程序中扮演着重要角色，因此这将为你提供一个基本的起点，帮助你理解以下信息。

`project.clj` 引导文件中的设置之一是 `:handler` 键，它告诉 Leiningen 哪个 Clojure 文件是应用程序的入口点（还有一个 `:init` 函数，在处理程序之前执行）。

注意

对于大多数 Leiningen 项目模板，你会发现你的 Clojure 代码位于：`src/<项目名称>/<文件.clj>`。

例如，如果你的 `project.clj` 包含 `:handler foo.handler/app`，你实际上是在告诉 Leiningen 在 `src/foo/handler.clj` 定位入口点（并且在该文件中，你会找到引导应用程序的 `app` 函数）。

当从项目目录运行 `lein repl` 时，你可以执行 (`ns foo.handler`) 来跳转到该命名空间，并开始执行该命名空间暴露的函数。请参见清单 9-4 的简单示例。

```
(ns foo.handler)
(init) ;; foo 正在启动
清单 9-4.
将命名空间加载到 REPL 中
```

如果你对文件进行了更改——例如，假设你用新变量更新了 `handler.clj` 文件：`(def x 123)`——为了使该更改反映在已经运行的当前 REPL 实例中，你需要重新加载命名空间文件。请参见清单 9-5。

```
(load "/foo/handler")
x ;; 123
清单 9-5.
将命名空间重新加载到 REPL 中

如果你想显示所有已加载的命名空间，最简单的选择是使用 `(loaded-libs)` 函数。如果你在当前 REPL 实例中运行该命令，你会看到一个已为你加载的命名空间大列表（包括 REPL 启动后添加的任何新命名空间）。请参见清单 9-6。

```
#{clj-stacktrace.core clj-stacktrace.repl clj-stacktrace.utils clj-time.core clj-time.format clojure.core.protocols clojure.core.server clojure.edn clojure.instant clojure.java.browse clojure.java.io clojure.java.javadoc ...}

```

## 内部化

我们在第 2 章中已经介绍过变量，当时我指出：

> 变量在其他命名空间中不可用，除非它们被“内部化”到这些命名空间中。内部化是一种花哨的说法，意思是“在当前命名空间中查找变量 x；如果不存在则创建它”。但你也可以使用 `ns` 宏的 `:refer` 特性从另一个命名空间“内部化”变量。 ——我，第 2 章

那么，让我们来看看从另一个命名空间“内部化”数据的不同方式。有几种不同的选项：

*   `:use`
*   `:require`
    *   `:as`
    *   `:refer`
    *   `:all`

现在，我应该澄清一下，实际上只有两种方法可以内部化数据，那就是使用前面列表中的前两项（`:use` 和 `:require`）。列表中的其他项都是与 `:require` 相关的子集特性，在官方文档中被称为库规范。

注意

我将最后介绍 `:use`，因为自从 Clojure 早期版本以来，`:require` 的功能已大大改进，`:use` 被认为是不必要的。我介绍 `:use` 只是为了让你知道它曾经存在，但现在已经不再需要，因为 `:require` 的功能已经取代了它。

我将首先解释这些项各自的工作原理及其区别，但在那之前，我想先稍微绕个弯……

### 根绑定

在讨论变量时，理解它们还有一个称为根绑定的东西，并且被认为是“线程局部的”，这一点很重要。这在实践中的意思是，当你创建一个变量并为其赋值时——例如，`(def foo "hello")`——它的根绑定就是你定义的值（在本例中，是字符串 `"hello"`）。一旦你将一个值与变量关联起来，那就成为该变量的“根绑定”，这意味着从当前命名空间内创建的任何线程都将能够访问它，因为它们本质上是在同一个内存空间中创建的。这就是“线程局部”的含义，其他命名空间将无法看到该变量或其值。

如果你定义了一个变量但没有为其赋值，那么它被认为是“未绑定的”（参见清单 9-7 的示例）。

```
(def foo)
foo
;; #
清单 9-7.
未绑定变量的示例
```

### 动态变量

我们在第 2 章中已经介绍过动态变量（即 `^:dynamic` 属性和 `binding` 形式），但由于我们现在正在阅读关于命名空间的章节，因此有必要提一个如果在本书开头提及可能没有意义的问题：`ns` 宏之所以能够动态更改 Clojure 中的命名空间，是因为该宏修改了全局的 `*ns*` 变量，这意味着它可以被重新定义。

注意

定义动态变量的约定是将它们包裹在“耳罩”（即星号）中。例如，`*变量名*`。

注意

当直接使用这些函数时（而不是在 `ns` 宏中使用它们），你需要对任何符号/向量进行引用。

现在我们对命名空间如何与我们的 foo 应用程序协同工作有了更好的理解，让我们回过头来看看 `:require`、`:as`、`:refer`、`:all` 和 `:use` 特性，看看它们是如何工作的。

### :require

在 `ns` 宏中包含 `:require` 意味着它将把所有公共变量导入到当前命名空间中，然后可用的选项（`:as` 和 `:refer`）有助于确定如何访问导入的数据（我们将在接下来的几节中看到）。`:require` 的语法结构如清单 9-12 所示。

```
(ns 
(:require 
[ :refer [ ]]
[ :as ]
[ :as  :refer []])
(:gen-class))
清单 9-12.
:require 宏的语法结构
```

### :as

`:as` 库规范允许你重命名命名空间本身。

我们已经见过 `:require` 的简写形式，但如果你愿意，也可以使用长格式（参见清单 9-13）。

```
(ns foo.core
(:require [clojure.string :as str])
(:require [foo.bar :as fbr])
(:require [foo.baz :as fbz])
(:gen-class))
清单 9-13.
:require 的长格式版本
```

然而，实际上你更可能使用简写形式（参见清单 9-14）。这意味着你可以使用更短的符号来引用 `foo.bar` 和 `foo.baz` 命名空间内的所有公共变量。例如，不用写 `foo.bar/b`（来访问公共变量 `b`），你可以输入 `fbr/` `b`。

```
(ns foo.core
(:require [clojure.string :as str]
[foo.bar :as fbr]
[foo.baz :as fbz])
(:gen-class))
清单 9-14.
:require 的简写形式版本
```

注意

当使用 `require` 的函数版本时（即在 `ns` 宏之外），向量前的引号意味着你不必引用向量内的每个符号。例如，使用 `(require '[name.space :as nmsp])` 而不是 `(require ['name.space : as 'nmsp])`。

### :refer

`:refer` 库规范允许你只将需要的公共变量导入到当前作用域。所有其他公共变量仍然可用，但你必须提供完全限定的命名空间才能访问它们。参见清单 9-15。

```
(ns foo.core
(:require clojure.string
[foo.bar :refer [a beep]]
[foo.baz :as fbz :refer [c]])
(:gen-class))
清单 9-15.
:refer 库规范示例
```

在清单 9-15 中，我们可以看到我们按原样加载了 `clojure.string` 命名空间；我们没有以任何方式缩短它。所以，如果我们想使用该命名空间中的 `upper-case` 函数，我们将执行 `(clojure.script/upper-case "make me upper case!")`，随后它将返回 `"MAKE ME UPPER CASE!"`。

在第二个语句 `[foo.bar :refer [a beep]]` 中，我们说的是“让 `a` 和 `beep` 在当前作用域中可用，而该命名空间中所有其他公共变量，我们必须通过完整的命名空间来访问。”所以，输入 `a` 将返回 `2`；输入 `(beep)` 将返回 `"bop"`，而要访问 `b`，我们不能直接输入它，而必须输入 `foo.bar/b`，这将返回 `1`。

在第三个语句 `[foo.baz :as fbz :refer [c]]` 中，我们说的是与之前类似的事情，但现在我们还声明，当我们必须使用完整的命名空间来引用一个公共变量时，我们只需使用缩短后的 `fbz` 作为命名空间，而不是更长的 `foo.baz`。所以，如果你输入 `c`，你将得到 `3`，如果你想访问 `d` 或 `e`，那么你需要输入 `fbz/d` 和 `fbz/e`。

注意

对于第三种语句形式，你仍然可以使用长格式的命名空间。所以，`fbz/c` 和 `foo.baz/c` 都有效。



### :all

`:all` 这个库规范与 `:refer` 配合使用。因此，当使用 `:refer` 时，你需要提供一个符号集合，这些符号代表你想要导入当前命名空间的公共变量；而你可以选择提供 `:all` 作为该集合的替代选项。

```
(ns foo.core
(:require clojure.string
[foo.bar :refer :all])
(:gen-class))
清单 9-16.
:all 库规范示例
```

在清单 9-16 中，我们可以在当前命名空间中直接引用 `a`、`b` 和 `beep`，而无需提供完全限定的命名空间。例如，我们不必指定 `foo.bar/a`，直接使用 `a` 即可。

注意

你会在下一节中注意到，`:all` 实际上是 `:use` 的默认用法，因此它已被弃用。

### :use

在 `ns` 宏中包含 `:use` 意味着它会将所有公共变量导入当前命名空间，并用这些导入的变量覆盖任何同名的现有定义。请记住，函数也是通过 `defn` 宏赋值给变量的，因此公共函数也会在当前命名空间中被提供/覆盖。

如果你将 `foo.baz` 中的变量 `c` 改为另一个命名空间中已定义的变量（例如 `foo.bar` 中的 `var a`），你就会注意到使用 `:use` 时会出现问题。

尝试将 `(def c 3)` 改为 `(def a 3)`，然后从项目根目录运行 `lein repl`。你应该会在 REPL 输出的前几行看到一个异常（参见清单 9-17）。

```
a already refers to #'foo.bar/a in namespace foo.core.
清单 9-17.
使用 :use 混合变量时引发的异常
```

发生此异常是因为 `foo.baz` 试图重新定义变量 `a`，而该变量已被 `foo.bar` 定义。这就是使用标准 `use` 而不使用其库规范选项（如 `:exclude`、`:only` 或 `:rename`）时会出现的问题。

注意

当从 REPL 调用 `use` 时（例如 `(use 'foo.baz)`），它会重新定义任何定义而不发出警告。

要修复此异常（并随后让所有代码正常加载和运行），我们需要决定如何处理导入 `foo.baz` 命名空间的问题。

我们可以排除 `a` 和 `d` 变量，或者使用 `:only` 单独引入 `e` 变量（在这个例子中这样做更合理，因为输入更少）；我们甚至可以使用 `:rename` 来重命名 `c` 和 `d`。选择权在你手中。

但假设我们决定引入 `a` 变量。这意味着我们需要同时使用 `:only` 和 `:rename`。我们可以在清单 9-18 中看到一个示例，我们从 `foo.baz` 中引入 `a` 变量并将其重命名为 `new-a`。使用这种格式，我们可以引入所有需要的依赖项，而无需担心任何冲突。

```
(ns foo.core
(:use foo.bar)
(:use [foo.baz :only [d] :rename {a new-a}])
(:gen-class))
清单 9-18.
使用 :only 引入特定变量
```

### 其他内容？

虽然我并非追求面面俱到，但可能还是值得再提几件事。首先，你可以使用另一个宏来代替 `ns` 宏（我们之前实际上见过它，但我故意没有说明）。它叫做 `in-ns`。它们之间唯一的真正区别是 `in-ns` 不提供对任何库规范的访问（即 `:use`、`:require`、`:as`、`:refer` 或 `:all`）。

第二件值得一提的事情是使用库规范 `:verbose`、`:reload` 和 `:reload-all`。这些仅适用于 `require` 函数及其 `ns` 变体 `(:require ...)`。例如，如果添加了 `:verbose`，那么 Clojure 会打印出与 `load` 函数调用相关的任何信息。这可以通过显示底层正在进行的调用来帮助阐明 `:require` 调用在内部做了什么。

`:reload` 库规范确保即使一个库已被加载（例如，被另一个命名空间加载），如果它有任何更改，也应该重新加载。我自己没有理由使用这个行为，除了在 REPL 中反复试验并不断加载依赖项的时候。`:reload-all` 库规范的工作方式相同，但它会递归地检查所有已加载库的所有依赖项。

最后一项是 `import` 命令，它对于导入 Java 类和接口非常有用。清单 9-19 演示了其用法的一个基本示例。

```
(import java.util.Date)
(def now (Date.))
(str now) ;; "Thu Jun 29 09:24:04 BST 2017"

```

## 总结

在本章中，我们学习了各种命名空间符号和宏，如何在 REPL 中动态创建新的命名空间，以及如何在现有命名空间之间切换并在这些空间中执行代码。可能最重要的是，我们理解了如何在命名空间之间内联变量，以及如何通过可用的各种库规范选项来避免冲突。

在下一章中，我们将研究如何利用一种称为宏的特性来操作 Clojure 语言本身。这是一个强大的特性，我们必须小心使用，以免给我们的代码库增加复杂性和混乱。



# 10. 宏

可以说，Clojure 最有趣的特性之一，是其提供了一种功能，允许用户重新定义语言，并添加默认情况下不可用的自定义特性。我的意思是，你有多少次在编写代码（无论用什么语言，比如 JavaScript）时，会想“我真希望我有 Y 语言的 X 特性”？

对于像 JavaScript 这样的语言，我敢打赌这种情况经常发生。我相信会有一些 JavaScript 开发者反驳说，现在有可用的编译器，可以让你将有效的 ES7 代码编译成 ES5 代码（例如）。但这仍然无法解决将 ES5 代码编译成 ES3 代码的问题，在某些情况下你甚至无法做到这一点（因为 ES3 语言规范的限制）。更不用说现在需要两套编译步骤，才能确保你的代码能在尽可能广泛的设备上正常工作。

注意

渐进增强¹ 可以改善这种情况，但实现渐进增强所涉及的工作并不简单，而且与语言本身原生支持重新定义语言相比，这是一个极其强大的概念。

但即使你是一名 JS 开发者，并且能够利用现有的某个流行编译器，也可能找不到一个能提供你所需特性的编译器。在这种情况下，你最终可能会偶然发现像 SweetJS ² 这样的东西，它是一个允许你定义自己特性的编译器。但这又给完成工作带来了另一个障碍，因为你需要管理一个外部依赖。

利用 Clojure 宏特性的能力，使得 Clojure 核心库能够保持紧凑和专注，同时允许开发者向语言添加自己的特性，并且可以放心地知道宏系统是语言的原生组成部分，而不是某个可能最终失去支持的外部工具。

注意

Clojure 的作者们（过去曾多次）表现出一种习惯，即把使用宏系统构建的流行抽象吸收回官方的 Clojure 代码库中。这意味着，如果你使用宏创建了有用的东西，那么你的工作最终可能会被所有 Clojure 开发者使用。

让我们通过回顾两个我们在前几章中已经使用过、并且是 Clojure 核心库一部分的特定宏：`->` 和 `->>`，来开始理解宏。

```
(->  6 (/ 2) (+ 10))
;; 13
(->> 6 (/ 2) (+ 10))
;; 31/3
代码清单 10-1.
-> 和 ->> 宏的示例
```

在代码清单 10-1 中，我们有两行非常相似但返回截然不同值的代码。这两行代码的唯一区别是第二行多了一个尖括号字符 `>`。这个额外的字符改变了等式，但它是如何做到的呢？

在 Clojure 中，`->` 被称为“线程优先”宏，而 `->>` 被称为“线程最后”宏。对于线程优先宏，第一个参数（即我们示例中的 `6`）作为第一个参数传递给后续的每个函数调用。对于线程最后宏，第一个参数（同样是我们示例中的 `6`）作为最后一个参数传递给后续的每个函数调用。

为了更清晰地理解宏系统的工作原理，我们可以使用一个名为 `macroexpand` 的函数来展示宏的完整形式（基本上，它展示了如果宏系统不存在，我们自己会编写的内容）。

```
(macroexpand '(->  6 (/ 2) (+ 10)))
;; (+ (/ 6 2) 10)
(macroexpand '(->> 6 (/ 2) (+ 10)))
;; (+ 10 (/ 2 6))
代码清单 10-2.
使用 macroexpand 展开宏
```

要使用 `macroexpand` 函数，如代码清单 10-2 所示，我们需要提供我们之前运行的表达式。但是，我们如何防止这些表达式被实际执行，并且将表达式的结果传递给 `macroexpand` 呢？解决方案是引用该表达式，我们通过在表达式形式的第一个左括号前放置一个单引号来实现这一点。

注意

我们也可以将表达式包装在对 `quote` 的函数调用中；例如，`(quote (-> 6 (/ 2) (+ 10)))`，但 `'` 更简洁。

所以，我们可以看到，通过传入 `'(-> 6 (/ 2) (+ 10))`，我们得到了 `(+ (/ 6 2) 10)`。返回值是宏展开后的结果，类似地，在第二行，如果我们传入 `'(->> 6 (/ 2) (+ 10))`，我们会得到展开后的结果 `(+ 10 (/ 2 6))`。

注意“线程优先”（`->`）是如何将第一个参数 `6` 作为除法函数 `/` 的第一个参数，并且调用该函数的结果本身现在作为第一个表达式传递给加法函数 `+` 的调用。

对于“线程最后”（`->>`），我们将第一个参数 `6` 作为最后一个参数传递给除法函数，并且调用该函数的结果随后作为最后一个参数传递给 `+` 函数。

## 一路展开到底

`macroexpand` 函数对于确保引号表达式中的所有宏都被展开很有用，但这不包括宏内部的嵌套宏。让我通过一个例子来解释（参见代码清单 10-3）。

```
(cond (even? 2) "even" :else "odd") ;; "even"
(cond (even? 1) "even" :else "odd") ;; "odd"
(macroexpand '(cond (even? 2) "even" :else "odd"))
;; (if (even? 2) "even" (clojure.core/cond :else "odd"))
代码清单 10-3.
Macroexpand 不包含嵌套宏
```

在代码清单 10-3 中，我们有一个 `cond` 宏，当传入一个谓词时，它会根据谓词的结果返回 `"even"` 或 `"odd"`。在这种情况下，如果我们传入值 `2`，则返回结果是 `"even"`；如果我们传入 `1`，则该值会落入 `:else` 分支，因此返回值变为 `"odd"`。

从 `macroexpand` 函数的结果可以看出，我们得到了 `cond` 宏的一个展开版本。但我们遇到了一个问题：`cond` 宏内部再次调用了自身，因此，为了看到 `cond` 宏真正具有代表性的展开，我们应该改用 `macroexpand-all` 函数。

要访问 `macroexpand-all` 函数，你需要从 `clojure.walk` 命名空间将其加载到 REPL 中（因为它不是 `core` 命名空间的一部分）。`walk` 命名空间是标准 Clojure 语言的一部分，因此你不需要像对 `core.match` 和 `core.async` 那样更新项目依赖。相反，你可以在 REPL 内部或代码文件顶部使用 `require` 函数。

```
(require '[clojure.walk :as w])
代码清单 10-4.
引入 clojure.walk 命名空间
```

一旦你加载了 `clojure.walk` 命名空间（参见代码清单 10-4），你应该能够引用 `macroexpand-all` 函数来完全展开之前的 `cond` 宏示例。参见代码清单 10-5。

```
(w/macroexpand-all '(cond (even? 2) "even" :else "odd"))
;; (if (even? 2) "even" (if :else "odd" nil))
代码清单 10-5.
使用 macroexpand-all 展开 cond 示例
```

## 编写你自己的宏

创建自己的宏有五个方面，这取决于宏需要有多复杂：

*   定义宏 (`defmacro`)
*   引用 (`'`)
*   语法引用 (`` ` ``)
*   取消引用 (`∼`)
*   取消引用拼接 (`∼@`)
*   生成符号 (`gensym`, `#`)

有趣的是，第 2-5 项都可以在宏外部使用，因此我将首先演示它们。之后，我将剖析一个自定义宏，以理解它是如何工作的，以及我们可能面临的一些典型挑战。



## 引号（Quoting '）

在第二章 2 中，当我们探讨列表数据结构时，我曾提到有些读者可能对列表前的单引号感到好奇。正如当时所述，这被称为对数据结构的引号引用，目的是让 Clojure 编译器不尝试对其进行求值。

```
(def l '(1 2 3))
l
;; (1 2 3)
列表 10-6.
引号引用列表数据结构以防止求值
```

在列表 10-6 中，你应该注意到赋值给变量 `l` 的值实际上是一个列表数据结构 `(1 2 3)`，而不是 Clojure 编译器试图将 `1` 作为函数执行并将 `2`、`3` 作为函数参数传递的结果（这原本就行不通，因为 Clojure 中不存在名为 `1` 的函数）。

引号引用是一项实用的特性，适用于那些从算法角度出发，你确实需要使用真正的链表数据结构（而非向量或集合等）的场景。

注意

还有其他创建链表数据结构的方式：`(list 1 2 3)` 和 `(quote (1 2 3))` 是两种常见选择，不过 `quote` 函数可用于任何类型的 Clojure 形式。

## 语法引号（Syntax Quoting `）

语法引号的工作方式与标准引号功能相同，唯一的区别在于它会将每个表达式转换为完全限定的命名空间形式。

```
(def l '(1 2 3))
`(+ l)
;; (clojure.core/+ user/l)
列表 10-7.
语法引号生成完全限定的命名空间
```

在列表 10-7 中，你可以看到我们首先定义了变量 `l`，并将其赋值为一个字面列表数据结构。接着，我们在表达式 `(+ l)` 前使用了反引号（`` ` ``），结果得到了该表达式的限定命名空间版本。

这意味着 `+` 函数变成了 `clojure.core/+`，而变量 `l` 变成了 `user/l`（在此例中，我是在 REPL 环境中执行示例代码的）。

注意

如果你尝试对结果表达式 `(clojure.core/+ user/l)` 进行求值，会发现代码无法运行。这是因为我们需要将列表元素展平（即展开），使它们不再包含在括号内，而是作为 `+` 函数的内联参数。稍后我将演示如何使用反引号拼接语法来实现这一点。

## 反引号（Unquoting ∼）

当你引号引用一个表达式时，意味着我们希望编译器不对其或其任何子表达式进行求值。但有时，你希望表达式的大部分内容被引号引用（即不求值），而其中某些小的嵌套部分仍能被求值。

```
(def l '(1 2 3))
`(+ ∼l)
;; (clojure.core/+ (1 2 3))
列表 10-8.
对嵌套表达式求值，但不求值整个表达式
```

在列表 10-8 中，我们可以看到我们使用反引号引用了表达式，但随后使用了波浪号字符（`∼`）来“反引号”变量 `l`，使其最终被 Clojure 编译器求值。这意味着我们现在得到了一个未求值的命名空间 `+` 函数，后面跟着表达式 `(1 2 3)`。

注意

列表 10-8 的输出是不可执行的，因为表达式 `(1 2 3)` 是无效的。核心命名空间中不存在名为 `1` 的函数。

## 反引号拼接（Unquote Splicing ∼@）

在前面的示例中，我们演示了一些特定功能，但都产生了不可执行的结果。现在让我们花点时间来解决这个问题，并将下一个示例转变为可以成功执行的代码。

```
(def l '(1 2 3))
`(+ ∼@l)
;; (clojure.core/+ 1 2 3)
(clojure.core/+ 1 2 3)
;; 6
(eval `(+ ∼@l))
;; 6
列表 10-9.
使用反引号拼接展开求值序列
```

因此，在列表 10-9 中，我们可以看到我们使用了反引号引号语法来确保表达式不被求值。但在这个示例中，我们希望列表数据结构（赋值给变量 `l` 的那个）不是作为一个列表放在 `+` 函数旁边，而是作为单独的参数放在那里。

为了实现求值列表数据结构（使其不作为 `user/l` 放在 `+` 旁边）并随后展平该列表数据结构的目标，我们使用的不是反引号波浪号，而是反引号拼接 `∼@`。我们可以看到这返回了 `(clojure.core/+ 1 2 3)`，你可以将其粘贴到 REPL 中执行，以看到它产生正确的结果 `6`。

如果你想以编程方式求值形式 `` `(+ ∼@l) `` 的结果，你需要将其包裹在对 `eval` 函数的调用中（如列表 10-9 所示）。

注意

你不能在没有反引号语法的情况下使用反引号拼接，因为这会使得 Clojure 编译器很难理解哪些内容应该或不应该被求值。

## 生成符号（Generating Symbols gensym/#）

在使用宏时，你最终可能会遇到变量绑定（既在宏内部，也在包含它的局部作用域中）相互冲突的问题。为了解决这个问题，你可以利用 `gensym` 函数在宏内部生成一个唯一的符号。

注意

重要的是要认识到 `gensym` 总是会返回一个全新的唯一值，因此你还可以提供一个前缀字符串，以便在调试时帮助识别该符号。

```
(gensym)
;; G__791
(gensym "hello")
;; hello794
`hello#
;; hello__803__auto__
列表 10-10.
使用 gensym 创建唯一标识符
```

在列表 10-10 中，最后一个示例使用了 `#` 作为 `gensym` 的简写语法，这意味着我们必须使用反引号 `` ` `` 语法；否则编译器将不知道如何处理（这是因为 `#` 是符号的有效字符）。

同样值得注意的是，`#` 变体确保在整个引号表达式中使用相同的标识符，而每次对 `gensym` 的引用都会产生一个新值。



## 宏剖析

至此，我们已经基本了解了可用于创建动态且健壮的宏特性的各个独立部分。现在，让我们通过分析一个更复杂的宏示例代码片段并理解其工作原理，来将这些知识融会贯通。

Clojure 没有传统的 `for` 循环结构（这里所说的“传统”指的是像 C 语言这样的底层语言），在这种结构中，你需要指定一个起始值、一个条件和一个增量器，然后循环执行，直到指定的条件不再满足为止。

为了在 Clojure 语言中解决这个问题，让我们将这个缺失的结构实现为一个宏。我们即将定义的 `for-loop` 宏执行后，应该将数字零到九输出到 `stdout`（每行一个数字）。

注意

我建议你花足够的时间仔细研读清单 10-11，确保你能将目前学到的所有不同语法融会贯通。

```
(defmacro for-loop [[sym init check change] & steps]
`(loop [∼sym ∼init accumulated# nil]
(if ∼check
(let [new-value# (do ∼@steps)]
(recur ∼change new-value#))
accumulated#)))
(for-loop [i 0, (< i 10), (inc i)]
(prn i))
清单 10-11.
使用宏特性构建的自定义 for 循环结构
```

好的，让我们先看看我们是如何调用 `for-loop` 宏的。我们可以看到，我们传入了一个包含循环信息的向量：`[i 0, (< i 10), (inc i)]`。后面跟着一个“主体”表达式 `(prn i)`，我们期望它在每次循环迭代时执行。

注意

如果我们愿意，我们可以传入多个主体表达式，让当前的宏实现来执行它们。但为了简单起见并演示基本功能，我们每次迭代只提供了一个待求值的形式。

在宏内部，我们将传入的向量解构（即，从传入的向量数据结构中提取每个元素）为以下局部绑定：

*   `sym`：`i`
*   `init`：`0`
*   `check`：`(< i 10)`
*   `change`：`(inc 1)`

我们还获取了剩余的参数 `(prn i)`，我们假定这些参数是每次迭代要执行的主体，并使用 `& steps` 将它们赋值给一个 `steps` 集合。这意味着它被构造成一个列表，如下所示：`((prn i))`。

此时，我们开始定义宏的主体部分。概括来说，我们执行以下步骤：

*   创建一个 `loop`（作为后续 `recur` 的目标）。
    *   将符号 `i` 绑定到值 `0`。
    *   将动态生成的符号（即 `accumulated#`）绑定到值 `nil`。
*   使用 `if` 语句检查局部变量 `i` 的值是否小于 `10`。
    *   `check` 条件是在运行时传入宏的（即 `(< i 10)`）。
    *   初始时该值会小于 `10`，因为我们刚刚将其设置为 `0`（参见上一步）。
*   如果值大于或等于 `10`，那么 `else` 语句返回赋值给 `accumulated#` 的值。
    *   稍后我们将看到 `accumulated#` 如何在每次迭代中被更新为一个新值。
*   如果值**确实**小于 `10`，那么我们就执行提供的主体表达式。
    *   最后一个主体表达式的返回值被赋值给 `new-value#`。
    *   然后我们使用 `recur` 跳回到 `loop` 的开头，并传入新值。
    *   每次迭代传入的新值是执行 `change` 表达式（即 `(inc i)`）的结果。
    *   然后，`accumulated#` 被传入当前赋值给 `new-value#` 的值。

从这些步骤中可以看出，解决方案的核心在于使用 `loop` 和 `recur` 来实现递归执行代码，并且在每次递归时，我们传入更新后的值，使得循环内的逻辑能够决定是继续递归还是停止。

实际上，理解宏的基础知识就是这些了。你现在应该已经掌握了足够的信息，可以去大展身手了……我是说，去创造灵感了！

我还认为，我可能有必要说一句（尽管我已经在后悔了）：“能力越大，责任越大”；在 Clojure 中编写宏的语境下，这句话的意思是：使用宏会带来一些微妙的问题，尤其是在存在其他无需宏的解决方案时。如果你需要在编译时而不是运行时执行某些操作，那么，没问题，宏会有所帮助。但首先，请检查是否存在可以满足需求的标准函数。

注意

你不能像使用标准函数那样，在代码中传递宏（例如，作为函数参数）。这最终意味着你会在代码设计中失去一些可组合性，而可组合性是函数式编程的一个基本原则。

# 11. 面向对象

Clojure 令人困惑的一个方面是，尽管它旨在成为一种函数式编程语言，但它也确实提供了“面向对象”的特性（惊讶吧）。这与其他编程语言类似，例如 Scala，¹ 它也同时提供了 FP 和 OOP 语法。

何时在 Clojure 中使用 OOP 是一个棘手的讨论点，它取决于你所从事项目的需求，因此我们不会对此发表任何武断的言论。不过，就我个人而言，我喜欢我们即将看到的一些项目的结构方面，并且使用特定类型来帮助构建不同系统之间的边界也很有价值。

归根结底，Clojure 中的 OOP 特性并不像其他语言中那样突出，而且绝对远不及 Scala 中的那么多，Scala 似乎旨在实现一种均衡的划分。

在本章中，我将介绍一个（小）精选的特性组，具体如下：

*   `defprotocol`
*   `deftype`
*   `defrecord`
*   `reify`

## Java 互操作

## defprotocol

`defprotocol` 宏与以下两种面向对象类型（`defrecord` 和 `deftype`）配合使用，其主要目的是创建一个接口。

如果你对接口的概念不熟悉，它的作用是定义一个契约：所有声称支持该特定接口的对象，都必须保证提供该契约所规定的功能。

```
(defprotocol Foo
  (bar [this abc])
  (baz [this] [this abc]))
代码清单 11-1.
使用 defprotocol 的简单接口示例
```

在代码清单 11-1 中，我们首先使用 `defprotocol` 宏定义了一个名为 `Foo` 的新协议。在这个代码块中，我们定义了两个方法：`bar` 和 `baz`。这意味着任何声称支持此接口的对象，都必须为这些指定的函数及其特定的签名提供实现。

`bar` 方法接受两个参数。第一个参数是 `this`，它应该是我们调用 `bar` 方法的对象实例（因此，我们沿用了许多编程语言（如 Java）中传统的 `this` 命名约定），而第二个参数是 `abc`。

`baz` 方法的定义类似。唯一的区别在于该方法具有“多重参数数量”，这意味着它可以用一个或两个参数来调用，并且我们可以根据调用方法时提供的参数数量来定义两种不同的行为。

注意

`definterface` 是一个与 `defprotocol` 类似的功能。区别在于前者会生成一个实际的接口 `.class` 文件，Java 可以利用该文件将有效的对象传递给 Clojure 函数。

`defprotocol` 本身并不是非常有用，所以让我们进入下一节，看看它是如何被有效利用的。

## deftype

为了能够使用我们定义的协议（即接口），我们需要一个具体的对象（类实例）来分配给它。这就是宏 `deftype` 和 `defrecord` 发挥作用的地方。

`deftype` 和 `defrecord` 都会创建一个对象，但使用哪一个取决于微妙的实现细节。例如，`deftype` 宏会创建一个精简的 Java 类实例，该实例允许进一步的 Java 互操作以及内部数据字段的修改。然而，`defrecord`（详见下一节）的行为更像一个简单的不可变哈希映射。

```
(deftype Qux [data]
  Foo
  (bar [this args]
    (prn (str "hello " (subs (str (class this)) 6)
              " from bar! I've been given: " args)))
  (baz [this]
    (prn (class this) data))
  (baz [this args]
    (prn (str "hello " (subs (str (class this)) 6)
              " from baz! I've been given: " args))))
代码清单 11-2.
使用 deftype 的示例
```

好的，代码清单 11-2 中内容很多，让我们将其分解成几个部分。首先，我们可以看到我们定义了一个名为 `Qux` 的新类型，并指定它接受一个名为 `data` 的构造函数参数。

`deftype` 的目的是最终生成一个行为非常类似于典型 Java 类的类型。因此，在接下来的示例（代码清单 11-3）中，我们将看到，当我们初始化这个类型（类）时，我们能够将自己的特定数据传递给构造函数（随后在 `baz` 方法内部引用该数据）。

当我们定义行为（即方法 `bar` 和 `baz`）时，我们可以看到我们指定了第一个预期的参数是对象实例本身。这是因为该类型使用了一种动态分发形式（我们在第 7 章中学习过这个概念），因此它需要知道将行为分发给哪个类型。我们将在代码清单 11-3 中看到这一点。

注意

在 Clojure 中，你不必为协议定义的每个方法都定义行为。这似乎与其他基于面向对象的语言的工作方式相冲突（例如，通常在使用接口时，你同意实现其整个契约，因此你应该为所有指定的方法定义行为）。但 Clojure 并非如此。

```
(let [b (Qux. "bing bop")]
  (.baz b)
  (.bar b "x")
  (.baz b "y"))
;; user.Qux "bing bop"
;; "hello user.Qux from bar! I've been given: x"
;; "hello user.Qux from baz! I've been given: y"
代码清单 11-3.
调用我们的 Qux 类型（由 deftype 定义）
```

在代码清单 11-3 中，我们可以看到我们使用语法 `(ClassName. args)` 初始化了类型（构造函数），在我们的示例中是 `(Qux. "bing bop")`。然后我们将返回的对象实例赋值给 `b` 这个 let 变量，之后我们使用语法结构 `(.methodName objectInstance args)` 调用每个后续方法（包括不同的参数数量变体），例如 `(.bar b "x")`。



## defrecord

记录（record）是一种与映射（map）非常相似的数据结构，因为它是一个不可变的键/值存储，但它也类似于结构体（struct，其定义的对象具有特定的字段用于分配，因此方法可以应用于它）。

与 `deftype` 的一个关键区别在于，`defrecord` 是不可变的，而 `deftype` 可以改变其内部状态。但在讨论如何使用 `defrecord` 之前，让我们先考虑一下你可能想要使用它的一些原因。

注意

如果在记录上定义方法，则还必须定义一个协议（接口）。

如果你使用动态分发（例如， multimethods），那么你可能会遇到的一个问题是，当你实际上只改变了整体功能的一小部分时，你最终会多次定义相似的行为。使用 `defrecord`（以及 `defprotocol`）则可以将方法和行为分组到一个统一的类型之下。

从性能角度来看，与使用 `defrecord` 相比，multimethods 相当慢，如前所述，`defrecord` 只不过是一个不可变的持久化映射数据结构。

注意

在内部，记录被表示为 Java 类。

```
(defrecord Foo [bar baz qux])
(def f (Foo. "x" "y" "z"))
f ;; #user.Foo{:bar "x", :baz "y", :qux "z"}
(:bar f)
;; "x"
清单 11-4.
简单 defrecord 示例
```

因此，在清单 11-4 中，我们可以看到我们定义了一个名为 `Foo` 的新记录，并期望它拥有字段 `bar`、`baz` 和 `qux`。然后我们定义了一个 `Foo` 的新实例，并将其赋值给变量 `f`。

这个实例化过程的有趣之处在于记录名称后面使用了句点字符（点号）。这是 Clojure 与 Java “互操作”的方式之一（请记住，“在底层” `Foo` 记录实际上被表示为 Java 类）。

一旦我们创建了实例，我们就可以通过指定字段名称，后跟应分发行为以进行处理的对象实例 `(:bar f)` 来访问该字段。

注意

由于记录实际上是一个 Java 类，我们也可以使用点互操作符号来访问成员字段：`(.bar f)`。

现在，当我们定义记录时，会自动生成两个函数供我们使用。第一个是 `->` `Record`，第二个是 `map->Record`（其中 `Record` 是记录的名称）。使用前者，我们可以像这样创建实例：`(->Foo "x" "y" "z")`，这并没有太大区别。我个人更喜欢 Java 互操作语法。但使用后者，你可以使用映射数据结构而不是标准参数来实例化记录。这在你已经有一个想要封装到记录中的数据结构的情况下非常有用。例如，`(map->Foo {:qux "z" :bar "x" :baz "y"})`。

注意

使用 `map->Record` 时，映射数据结构中的键可以是任意顺序，这与 `Record.` 或 `->Record` 不同。

现在让我们考虑一个例子，我们希望我们的记录也包含一个名为 `message` 的方法（以及之前定义的字段 `bar`、`baz` 和 `qux`）。

```
(defprotocol IFoo
(message [obj]))
(defrecord Foo [bar baz qux]
IFoo
(message [obj] (str "hello " obj " from " bar ", " baz " and " qux)))
(def f (Foo. "x" "y" "z"))
f
;; #user.Foo{:bar "x", :baz "y", :qux "z"}
(.message f) ;; "hello user.Foo@b679dccc from x, y and z"
清单 11-5.
带有附加方法的 defrecord 示例
```

在清单 11-5 中，我们可以看到，因为我们想要在记录中包含一个方法（并且因为记录实际上只是 Java 类），我们首先需要定义一个协议（接口）。我使用了在接口名称前加上大写字母 `i` 的约定（但你可以使用任何对你最有意义的约定）。我们定义的方法接受一个参数 `obj`。

接下来，我们像之前一样定义我们的记录，但现在多了两行。第一行是我们希望支持的接口，第二行是协议的具体实现。你还可以看到该方法能够访问记录已初始化的其他字段。

最后，我们使用点 Java 互操作点符号来调用我们的方法。

## Reify

我们要看的最后一项是 `reify` 宏，它能够生成一个匿名类型。这与 `deftype` 和 `defrecord` 不同，后两者生成的是具名类型。`reify` 的唯一目的是提供一种方式来定义预先存在的协议/记录的**一次性**实现，同时它的主体是一个词法闭包，这意味着它可以访问周围的局部作用域。

这最好通过一个例子来演示（见清单 11-6），该例子突出了 `reify` 类型如何能够访问其周围的局部作用域以提供包装后的响应。

注意

请参阅 `clojuredocs.org` ³ 获取更实际的例子。

```
(defprotocol IFoo (message [obj]))
(defrecord Foo [bar baz qux] IFoo
(message [obj] (str "hello " obj " from " bar ", " baz " and " qux)))
(def f (Foo. "x" "y" "z"))
(defn wrapper [instance]
(reify IFoo
(message [_] (str "wrapped result: " (message instance)))))
(message (wrapper f))
;; "wrapped result: hello user.Foo@b679dccc from x,y and z"

```

## Compojure

Compojure² 是一个构建在 Ring³ 之上的路由库。如果你没有听说过 Ring，那么我将直接引用 Ring GitHub 页面上的话来解释它：

> Ring 是一个受 Python 的 WSGI 和 Ruby 的 Rack 启发的 Clojure Web 应用程序库。 —Ring. GitHub

现在，如果你同样不熟悉 WSGI 或 Rack，这不一定对你有帮助。本质上，这两个应用程序共享的是一个标准化的接口。因此，即使它们是独立的应用程序（为两种截然不同的语言构建），这也意味着如果你熟悉其中一个，那么另一个对你来说也将是可识别（且可理解）的。Ring 支持与 WSGI 和 Rack 类似的接口，这给开发者带来了信心，让他们觉得使用 Ring 应该很自然。

为了更好地理解这个接口，让我们从 Ruby 的 Rack 的角度来看一个典型的“Hello World”HTTP 请求/响应是什么样的（见清单 12-4）。

## 测试

Compojure 还在 `test/foo/test/handler.clj` 文件中为我们构建了一些基本测试（参见列表 12-9）。该文件利用内置的 Clojure 测试库 `clojure.test` ⁶ 来定义一些简单的测试，以验证 Web 服务器是否按预期运行。

注意

这些测试是集成测试，而非单元测试。简而言之，单元测试用于验证单个函数（或代码单元）的行为，而集成测试则验证整个系统是否按预期工作。有关两者区别的更多信息，我推荐阅读 Nathan Hughes 简短但精彩的解释。⁷

ring.mock.request
foo.handler))
(deftest test-app
(testing "main route"
(let [response (app (request :get "/"))]
(is (= (:status response) 200))
(is (.contains (:body response) "Hello World"))))
(testing "not-found route"
(let [response (app (request :get "/invalid"))]
(is (= (:status response) 404)))))

## 模板

在回顾 Leiningen 内置的各种模板（互联网上还能找到许多满足各类需求的其他模板）时，你会注意到一个现象：每个模板的树结构并没有太大差异。这既是好事（因为它提供了统一的工作基础），也是坏事（因为它让人难以区分不同模板的差异和适用场景）。

那么，总结一下我们目前所知的内容：Leiningen 使用模板的概念来动态构建新项目。它有一些内置模板，也有大量外部模板（比如我在十秒示例中使用的 Compojure 模板）。目前（撰写本文时）共有四个独立的内置模板，如下所示：

*   `template`：用于创建“lein new”模板的元模板

*   `app`：应用程序项目模板
*   `plugin`：Leiningen 插件项目模板

考虑到这种差异，如果你现在查看 `src` 目录下的 `core.clj` 文件内容，你会发现其内容与库模板的 `core.clj` 略有不同，因为它定义了一个 `main` 函数（以连字符为前缀：`defn –main`）。此外，在该文件的顶部，你会看到 `namespace` 宏包含了一个额外的配置设置 `(:gen-class)`。最后，`project.clj` 文件本身有一个 `:main` 属性，指向我们的 `core.clj` 文件，以表明这是执行 `lein run` 命令时应执行的文件。

现在让我们简要回顾一下这三个不同之处（`:main`、`-main` 和 `gen-class`），以确保我们理解它们的目的。`:main` 属性的作用是通知 Leiningen 应用程序的入口点，但你可能会注意到还定义了一个额外的属性：`^:skip-aot`。它的作用是跳过 Clojure 的“提前编译”（AOT）步骤，¹² 通过将代码编译成 JVM 字节码，有助于加快应用程序的启动速度。

考虑到 AOT 编译有助于提升性能，我们为什么可能想要跳过它呢？嗯，在某些情况下，你的代码可能依赖于只有在运行时才能知道的输入，因此如果你的代码引用了一个在编译时不可用的环境变量（例如，假设它只在部署应用程序的服务器上可用），那么这将意味着编译会将代码中引用该环境变量的所有地方都改为使用值 `nil`。

注意

有一些方法可以解决这类问题，但这超出了本书的范围/重点。

对 `(gen-class)` 和函数 `-main` 的引用是结合使用的，因为它们本质上是耦合的。`gen-class` 告诉 Clojure 编译器为你的命名空间创建一个新的 Java 类，而 `-main` 则定义了该 Java 类的行为。

最后，这一切都与 `project.clj` 中 `:main` 属性的使用相关联，因为它表明存在一个 Java 类应被用作应用程序的入口点。我建议阅读参考资料¹³ 以了解更多关于这些特性的信息。

## 项目文件

在所有不同类型的项目/模板中，有一个始终如一的元素就是项目文件本身：`project.clj`。这个文件的主要目的是引导你的应用程序启动。它通过定义项目的依赖关系，并在你首次运行项目应用程序代码时为你拉取这些依赖来实现这一点。

`project.clj` 还允许你关联元数据，例如项目描述和 URL，以及定义你想要使用的任何插件（以及更多其他内容）。

有关可用配置设置的完整列表，我建议你阅读 Leiningen 模板示例文件¹⁸，该文件非常详细地解释了每一个可用的设置（而且数量很多）。

让我们来看一个通用的 Compojure “Hello World” 应用程序（就像我们之前创建的那个）的 `project.clj` 文件，看看我们是否能更好地理解我们所获得的配置的不同方面。

```

在清单 12-16 中，我们可以看到一些 Clojure 代码，它使用 `defproject` 宏来构建项目的引导程序。它已经使用我们运行 `lein new...` 时提供的名称进行了设置，并将应用程序的版本默认为 `0.1.0-SNAPSHOT`。当我们准备发布应用程序时，可以将其更改为我们想要的任何值（例如，你可能希望将其更改为 1.0.0 或 0.0.1）。

注意

通常，这里使用的版本格式暗示了语义化版本控制¹⁹：`<主版本>.<次版本>.<补丁>`，而使用 `SNAPSHOT` 是 Java 版本命名约定。

接下来，我们有一个标准的 `:description` 字段和一个关联的 `:url` 字段，以便用户知道项目的源代码（或推广品牌）可以位于何处。

我们还有一个 `:dependencies` 字段（可以说是这个文件中最重要的项目之一）。对于 Compojure 应用程序，我们发现 Compojure 应用模板已经自动为我们预填了一些有用的依赖：

*   `org.clojure/clojure "1.8.0"` — Clojure 语言
*   `compojure "1.5.2"` — HTTP 请求路由库²⁰
*   `hiccup "1.0.5"` — 用于渲染 HTML 的库²¹
*   `ring-server "0.4.0"` — 用于 Ring 接口的 Web 服务器²²

注意

当你运行 Leiningen 时，它会自动拉取你所有的依赖，但你也可以随时使用子命令 `lein deps` 手动执行此操作。

继续往下看，我们有 `:plugins` 设置，它引入了插件 `lein-ring`²³，该插件负责自动化一些常见的基于 Ring 的任务。这个插件为 `project.clj` 文件添加了一个额外的设置，一旦指定，我们就需要包含以下属性：`:ring {…配置…}`。

正如我们从 `project.clj` 文件中看到的，当使用 Compojure 模板时，它会自动为我们预填 `:ring` 属性的配置，以便它知道主处理函数的位置，并识别用于设置和拆卸行为的函数（默认情况下这些是空操作函数，意味着它们是已定义的函数，但没有定义任何行为）。

接着往下看，我们看到 `:profiles` 设置。这允许我们根据环境定义某些行为。因此，我们可以看到有一个 `:production` 属性，它定义了应用程序在生产环境中应如何表现（例如，我们设置 `:auto-reload? false`，因为这种行为仅在本地开发应用程序代码时有用；我们还设置 `:stacktraces? false`，因为我们希望向用户隐藏这类信息）。类似地，我们有一个 `:dev` 属性，我们在其中指定仅在本地开发时有用的依赖。

如果你想了解更多关于这些配置选项的信息，那么我建议你查看 `lein-ring` 插件的 GitHub 页面²⁴，那里会提供最新的详细信息。

注意

你可以通过使用 `with-profile` 选项运行应用程序来在本地测试你的配置文件（例如，`lein with-profile production run some_app`）。

## compojure 与 compojure-app 对比

我们使用 `lein new compojure-app <应用名称>` 创建了一个“Hello World”示例，这为我们免费提供了很多东西，例如处理静态资源以及提供一些基本的 CSS 来配合我们的 HTTP 响应。

但我们也可以使用 `lein new compojure <应用名称>`（注意 `compojure-app` 现在只是 `compojure`），它仍然会像以前一样为我们提供一个简单的 Web 服务，但现在它被进一步精简了！这意味着没有花哨的 HTML 包裹着我们的“Hello World”消息，只有纯文本发送回浏览器。

我建议你探索这两种模板类型之间的其他差异，但你最终选择哪种模板取决于你对处理丰富 HTML 响应的许多繁琐方面的熟悉程度。对于大多数人来说，`compojure-app` 是快速高效启动和运行的最简单途径。



## 真实世界的库示例

有时最好看看真实项目是什么样子。例如，构建一个 Clojure Web 应用程序是一回事（大多数开发者都能应付——或许借助一些谷歌搜索），但我从个人经验中发现，构建一个被其他应用程序使用的库可能是一个相当令人困惑的过程。因此，我想向你展示一个我编写的小型 Clojure 库，它帮助我使用一个名为 Spurious²⁵ 的工具（这是几年前的一个例子，但原理仍然适用）。

我们将从理解 Spurious 是什么开始，然后查看我创建的库项目，并简要浏览一个使用该库的示例应用程序。但首先，让我们参考 Spurious 的 README 文件，看看它是如何描述自己的：

> Spurious 是一个工具集，允许在本地针对 AWS²⁶ 资源的子集进行开发。这些服务以 Docker 容器形式运行，Spurious 管理它们的生命周期和链接，因此你只需专注于使用这些服务。要使用 Spurious，你需要将每个 AWS 服务的端点和端口更改为 Spurious 提供的值。——Spurious. GitHub

简而言之，Spurious 试图在开发使用 AWS 服务的应用程序时为你节省资金（通过在本地运行这些服务）。这对我来说非常棒，因为我经常使用 AWS，它让我能够进行实验，而无需为可能永远不会成型的代码付费。

注意

为了让事情更复杂（对我自己而言），我需要我的库代码在标准 Clojure 应用程序中加载时能够工作，同时也要在应用程序从 Docker 容器内运行时能够工作（这不一定导致更多代码，但增加了整体过程的复杂性，以及对如何编写和测试我的库的理解难度）。

你可以在 GitHub²⁷ 上找到我的库代码，以及使用该库的示例应用程序²⁸。我不会详细讲解它们的细节，但我会提供链接，因为它们将作为有用的参考点。

### 本地测试

如果你不熟悉发布库的过程，我将快速绕道说明我需要做什么才能测试并将自己的库作为依赖项加载。

首先，我需要将我的库部署到 Clojure 的在线可用库仓库（称为 Clojars²⁹）。这需要你遵循 Leiningen³⁰ 提供的说明，但本质上涉及的步骤是注册一个账户，然后执行子命令 `lein deploy clojars`。

但是，在发布之前如何测试你的库代码呢（你怎么知道它能工作）？你不想推送版本 1.0.0，然后使用时发现有一个你未预料到的 bug。你想使用本地应用程序在本地测试库，这样你可以 100% 确定它能工作（这对我来说尤其重要，因为我的库需要在 Docker 容器内工作，而我不确定没有手动步骤就能验证这一点）。

解决这个问题的方法很简单：将库安装到本地缓存目录。要理解这意味着什么，你需要知道 Leiningen 会为你的依赖项创建一个本地缓存。它这样做是为了避免为你创建的每个项目反复下载相同的依赖项。

这意味着我们可以利用本地缓存来“欺骗”Leiningen，让它以为已经在线下载了依赖项（即从 Clojars 下载），即使该依赖项在线尚不存在。要做到这一点，你只需（在你的库项目目录内）运行 `lein install`，这会将你的库安装到本地的 `∼/.m2` 缓存目录中。

我有一个单独的 `src` 文件夹，其中又有一个名为 `spurious_aws_sdk_helper` 的文件夹，里面包含了库所需的所有 Clojure 文件。`core.clj` 负责加载其余文件：`dynamodb.clj`、`s3.clj` 和 `sqs.clj`。

`utils.clj` 文件（正如你所料）包含大量实用函数。该文件中只有两个函数是公开的，以便我们的消费应用程序可以调用它们来启动并运行服务；其余函数都是私有的，仅供库内部使用。

注意

请记住，你可以通过连字符 `defn-` 来判断函数何时被定义为私有（公开函数使用你已经熟悉的标准 `defn` 宏）。

一个值得关注的点是，`core` 命名空间中的 `configure` 函数如何使用多参函数，以便根据是否提供了 `opts` 映射来执行不同的行为。

```
(defn configure
([type]
(s3/setup  type)
(ddb/setup type))
([type opts]
(if-let [name   (:s3  opts)] (s3/setup  type name))
(if-let [name   (:sqs opts)] (sqs/setup type name))
(if-let [schema (:ddb opts)]
(ddb/setup type (yaml/parse-string schema)))))
清单 12-20.
简单多参函数示例
```

如清单 12-20 所示，如果没有提供 `opts`，我们只需为 S3 和 DynamoDB 调用 `setup` 方法；否则，如果提供了 `opts` 参数，我们使用 `if-let` 从提供的 `opts` 哈希中提取相关键（例如 `:s3`、`:sqs` 或 `:ddb`），然后将其与 `name` 符号关联。我们将再次为每个服务调用 `setup` 方法，这次会传入 `name`。

注意

我们之前没见过 `if-let`，但它是一个极其有用的宏，它允许我们仅在 `let` 绑定能够成功将数据解构到指定符号时，才执行一段代码（你还有类似的选项，例如 `when-let` 和 `letfn`，值得进一步探索）。

### 准备部署

现在，假设你有一个想要部署的项目（如果是库，那么如前所述，你需要将库部署到 Clojars，这样其他应用程序才能使用它）。

否则，如果这是一个标准应用程序而非库，那么 Leiningen 也提供了一个单一命令来帮助我们打包应用程序，使其能够轻松部署到生产服务器并几乎无需繁琐操作即可运行：`lein uberjar`。

当在 `project.clj` 中正确配置后（稍后会介绍），`uberjar` 命令将生成一个包含项目代码及其所有依赖项的单一 jar 文件。如果你之前没有处理过 `.jar` 格式，清单 12-21 将展示如何从命令行执行 jar 文件。

```
java -jar my_jar_file.jar   
清单 12-21.
通过终端执行 Jar 文件的示例
```

如果你使用 `lein-tar` 插件，³¹ 那么你可以使用 `lein tar` 命令；如果你使用 `lein-ring`，³² 那么你可以使用 `lein ring uberjar`（或者如果需要兼容旧服务器，可以使用 `lein ring war`）来构建 Web 应用程序。

现在，回到标准的 `lein uberjar` 命令，我之前提到你需要配置应用程序才能使该命令正常工作。你需要修改 `project.clj`，添加一个 `:main` 键，并指定一个指向应用程序入口点的命名空间。

如果你查看示例应用程序，³³ 你会看到我将命名空间设置为 `:main ^:skip-aot spurious-clojure-example.repl`。该命名空间指向 `src/spurious_clojure_example/repl.clj`，并且在该文件中我添加了一个名为 `main` 的新函数。`main` 函数负责启动 Web 服务器（参见清单 12-22）。

```
(defn -main []

```

注意

如前所述，我们的主入口点还需要 `(gen-class)`，所以你会注意到我也将其包含在了文件中。

实际上，这一切归结为（取决于你的应用程序功能）：修改 `project.clj` 中的 `:main` 键，使其指向一个包含 `main` 函数的文件，该函数以所需方式引导你的应用程序，并确保在文件顶部添加 `(gen-class)`（请参阅本章前面的内容以了解原因）。

# 13. 命令行应用程序

在本章中，我们将完成两件事：首先，我们将回顾 Clojure 的 `tools.cli` 库，以了解如何利用它来帮助我们设计和开发命令行应用程序。其次，我们将探讨如何打包此应用程序，以便我们能够分发它。

构建命令行应用程序是大多数开发者喜欢做的事情，因为它提供了一种丰富其工作环境（以及其团队或更广泛开发者社区的环境，如果他们设计的工具足够抽象，能够对除自己之外的更多人有用）的方式。在设计 CLI 工具时，可能性是无限的，因为你拥有很大的发挥空间。

一旦我们回顾了 `clojure/tools.cli` 和打包选项，你会发现我们已经涵盖了绝大部分你需要了解的内容。剩下的就靠你自己了；你如何设计和构建应用程序将取决于你想要构建的工具以及你试图解决的问题。

让我们首先使用我们在上一章中看到的 Leiningen `app` 模板创建一个新项目。命令是 `lein new app cli-foo`（你可以随意命名你的应用程序，但我选择了简单而有效的 `cli-foo`）。

在该项目中，打开你的 `project.clj` 并更新依赖项键，以包含 `clojure/tools.cli` 依赖项。参见清单 13-1 以查看上下文中的更改。

注意

你需要做的一件事（我们稍后会解释原因）是从 `project.clj` 中删除 `:target-path` 属性。

好的。现在让我们打开 `src/cli_foo/core.clj` 并进行清单 13-2 中所示的修改。这里有很多内容，所以我们将花一些时间来分解每个更改，以确保我们理解我们在做什么以及为什么这样做。

你首先会注意到我们更新了 `ns` 宏，以确保我们引入了来自 `cli` 命名空间的 `parse-opts` 函数。这允许我们解析传入的参数（用户在使用工具时提供的），并以对我们的应用程序有意义的方式智能地处理输入。

```
(ns cli-foo.core
(:require [clojure.tools.cli :refer [parse-opts]])
(:gen-class))
(defn exit [status msg]
(println msg)
(System/exit status))
(def cli-options
[["-p" "--port PORT" "端口号"
:default 80
:parse-fn #(Integer/parseInt %)
:validate [#(< 0 % 0x10000) "必须是一个介于 0 和 65536 之间的数字"]]
["-v" nil "详细程度级别"
:id :verbosity
:default 0
:assoc-fn (fn [opts k v] (update-in opts [k] inc))]
["-x" "--xxx WHATEVER", "此选项的描述"
:default 123]
["-h" "--help"
:id :custom-help]])
(defn -main [& args]
(let [{:keys [options arguments errors summary]} (parse-opts args cli-options)]
(cond
(:custom-help options) (exit 0 summary)
(not (nil? errors)) (exit 1 errors)
:else (exit 0 (parse-opts args cli-options)))))
清单 13-2.
我们命令行工具的主要应用程序逻辑
```

接下来你会注意到我们定义了一个 `exit` 函数。这个函数的目的是接受一个“退出码”（如果一切执行成功，通常为 `0`；如果出现错误，则为正数¹）。我们将退出码传递给 `System/exit` 函数，该函数将确保应用程序正确且恰当地关闭。该函数还接受一个 `msg` 参数，顾名思义，它将用于打印消息。消息被发送到 `stdout`（标准输出²），在这种情况下是终端屏幕。

之后，我们定义了一个 `cli-options` 变量，并为其分配了一个大型嵌套向量数据结构，其中包含一组非常具体的属性定义。向量中的每个项目定义了一个单独的标志。

注意

标志是我们提供给命令行应用程序以配置其行为的东西。例如，`ls -l -a` 是 `ls` 命令（列出所有文件和目录），并应用了两个标志 `-l` 和 `-a`，这会影响显示的输出。

我们可以看到我们定义了以下标志（这些都是任意编造的标志），我们很快就会看到，在应用程序本身中，除了将它们打印到屏幕上之外，我们实际上并没有对这些标志做太多处理，但它们被选用来演示 `tools.cli` 库的一些不同方面：

*   `-p`，`--port`：演示 `:parse-fn` 和 `:validate`
*   `-v`，`--verbosity`：演示 `:assoc-fn`
*   `-x`，`--xxx`：演示 `:default`
*   `-h`，`--help`：演示 `:id`

注意

要查看可用属性的完整列表，我建议你阅读源代码文件，其中记录了所有属性。³

我们将稍后回到这些标志的含义及其工作原理，但让我们先继续看代码的最后一部分，即 `-main` 函数。我们在这个函数中做的第一件事是调用 `parse-opts` 函数，并向其提供通过命令行工具用户传递给我们的参数，以及我们定义和配置的 `cli-options` 标志。

我们从 `parse-opts` 得到的是一个包含四个键的映射数据结构：`:options`、`:arguments`、`:summary` 和 `:errors`。参见清单 13-3 以了解分配给每个键的描述。我们将每个键分配给一个特定的符号以便于引用。

```
{:options     选项映射，以 :id 为键，映射到解析后的值
:arguments   未处理参数的向量
:summary     包含最小选项摘要的字符串
:errors      可能的错误消息字符串向量}
清单 13-3.
从 parse-opts 函数返回的映射
```

接下来，在 `-main` 函数中，我们定义了一个条件：如果用户包含了标志 `-h` 或 `--help`，那么我们应该调用 `exit` 函数，并传递由 `parse-opts` 返回的 `:summary` 值。

你还会注意到，我们通过查找我们在 `:options` 字段中为该标志定义的 `:id`（`:custom-help`）来识别是否指定了帮助标志，该字段由 `parse-opts` 返回。

接下来，在该条件之后，我们声明如果解析参数和/或标志时出现错误，那么我们应该调用 `exit` 函数，但提供退出码 `1`（表示错误），然后传递由 `parse-opts` 返回的错误消息本身（具体来说是 `:errors` 字段），以便用户可以看到他们可能做错了什么。

最后，我们有一个 `else` 语句，它表示调用 `exit` 函数并提供退出码 `0`（表示一切正常），然后简单地再次调用 `parse-opts` 并将其返回值用作我们将要显示给用户的内容。

至此，我们对这个应用程序的了解已经足够，可以开始研究如何编译和运行它了。我们接下来将这样做，然后在运行应用程序并探索如何与之交互之后，我们可以返回并回顾我们定义并分配给 `cli-options` 变量的标志配置设置，希望能更深入地理解它们在实际中是如何使用的。

## 重温标志

好的，现在让我们回顾一下我们定义的标志，以确保我们理解它们的配置。让我们从 `-p` 标志开始。

如果我执行 `integralist` 命令时没有定义任何标志，那么从输出中我会看到 `-p` (`--port`) 标志被赋予了默认值 `80`。回顾清单 13-2，我们可以看到这是因为我们将 `:default` 属性配置为该值。这意味着如果我执行 `integralist -p 123`，我们期望在命令的输出中看到 `123` 被分配给该标志（事实也确实如此）。这展示了我们如何通过定义默认值来使某些标志成为可选项。

如果我执行 `integralist -p abc`，那么命令的输出将是一个错误（参见清单 13-11）。从错误信息中我们可以看到，它期望的是一个数字，但输入却是 `abc`。如果我们回顾清单 13-2，可以看到这是因为我们为 `-p` 标志配置了一个 `:parse-fn` 属性，它允许我们将输入转换为另一种格式的值（在本例中，我们为 `:parse-fn` 分配了一个匿名函数，该函数尝试将输入转换为整数）。

```
[解析选项 "-p abc" 时出错：java.lang.NumberFormatException: 输入字符串: "abc"]
清单 13-11.
为 -p 标志分配了错误类型导致的错误
```

与 `:parse-fn` 属性类似的是 `:validate` 属性，它允许我们验证输入的值是否是我们期望/需要的。在这种情况下，我们希望确保数字在特定范围内。如果我传递 `65535` 作为值，那么一切都会正常工作。但如果我传递 `99999`，那么我们会看到一个错误（清单 13-12）。

```
[验证 "-p 99999" 失败：必须是介于 0 和 65536 之间的数字]
清单 13-12.
为 -p 标志分配了错误值导致的错误
```

现在，如果我们看一下 `-v` 标志，会发现它有两个我们尚未解释的属性：`:id` 和 `:assoc-fn`。`:id` 属性仅在您未定义长标志时才需要。这是因为 cli 工具有一个约定，即它们使用长标志的名称作为 `:options` 映射数据结构中的键。对于 `-v` 标志，我们没有提供长标志（因此使用了 `nil` 值）。

在某些命令行工具中，您会看到一种约定，即定义的标志实际上可以是动态的。例如，如果您的工具允许显示调试信息，您可能希望用户能够根据“级别”配置他们看到的调试信息量。这正是我们试图通过工具中的详细程度标志 (`-v`) 来表示的。因此，如果您执行 `integralist -v`，您会看到详细程度值被设置为 `1`，但如果您将标志改为 `-vv`，它将被设置为 `2`，同样，如果您将标志改为 `-vvv`，它将被设置为 `3`（依此类推）。

这就是 `:assoc-fn` 的用途。它接受整个 `:options` 映射数据结构以及标志的 `:id` 及其当前值（在我们的例子中，设置了 `:default` 值为 `0`）。有了这些信息，我们可以在清单 13-2 中看到，我们只是在 `:options` 映射中递增了标志的值。

注意

我们不会回顾 `-x` 或 `-h` 标志，因为它们实际上只使用了我们已经见过的 `:id` 和 `:default` 属性。

## 附录 A：约定

每种编程语言都有自己的约定，以确保对该语言编写的代码设计有共同的理解和一致性。由于存在语法相似的语系（例如 C 语言），一些约定也能够相当一致地跨不同编程语言迁移，因此这些约定在应用于多种语言时是有意义的。

例如，如果您曾经编写过 `for` 循环（任何类型的），那么您很可能会知道变量标识符 `i`、`j` 和 `k` 被用作每个嵌套循环级别的字符（参见清单 A-1，该清单使用 JavaScript 语言语法演示了一个 `for` 循环）。

```
for (i = 0; i < condition; i++) {
for (j = 0; j < condition; j++) {
for (k = 0; k < condition; k++) {
// 代码
}
}
}
清单 A-1.
for 循环中 i、j 和 k 变量的示例
```

注意

`i`、`j` 和 `k` 通常用于以下几个原因（取决于您与谁交谈）：`i` 和 `j` 在数学中已经使用了很长时间，因此从数学到编程的过渡得以延续。在实际层面上，`i` 有时被解释为整数（integer）的缩写，因此当需要嵌套级别时，从字母表开始向上移动到 `j`，然后是 `k` 是合理的。

Clojure 也有类似的命名约定。

### 函数

*   `f`、`g`、`h`：函数输入（注意与 `i`、`j`、`k` 的相似性，现在 `f` 代表函数）
*   `n`：整数输入（通常表示大小）
*   `i`：整数索引
*   `x`、`y`：数字
*   `s`：字符串输入
*   `xs`：被视为 `x` 的复数形式或序列
*   `coll`：一个集合（或序列）
*   `pred`：一个谓词闭包
*   `& more`：可变参数输入
*   `*var*`：动态变量应包裹在星号中（也称为“耳罩”）

### 宏

*   `expr`：一个表达式
*   `body`：一个宏体
*   `binding`：一个宏绑定向量

此外，每个插件的复杂度与其优先级和重要性相匹配。例如，优先级最低的插件 Rainbow Parentheses 也是最容易描述的。

### Rainbow Parentheses

这个插件使用起来非常简单。它会把所有单调的括号变成彩色编码（仅此而已）。其目的是让你更容易辨别特定 Clojure 形式的起始和结束位置。

你可能会想：何必这么麻烦？但请相信我，即使使用了 Clojure 文件格式主题，你的眼睛仍然难以在代码中找出特定的形式（尤其是当你的代码变得更复杂时，你会发现需要处理大量嵌套的括号），因此让它们变成彩虹色确实是一种福音。

### Sexp Mappings for Regular People

在基于 Lisp 的语言中，你会遇到所谓的 S 表达式。⁵ 用通俗的话说，它们只是符号格式的一部分，你在基于 Lisp 的语言中会看到它们以括号包裹表达式的方式出现（这又回到了我们在本书开头讨论的“代码即数据”的理念，即代码代表了相同的基础数据结构）。

因此，这个插件允许你操作 Clojure 代码（例如 Clojure 的“形式”）。但重要的是，不要将这个插件与另一个名为 Sexp ⁶ 的 Vim 插件混淆，后者也是一个用于操作 S 表达式的有效插件。但我们这里感兴趣的插件实际上是 Sexp 的扩展，只是为基于终端的 Vim 用户提供了更符合习惯的键位绑定。

这个插件有两个主要功能：

1.  自动括号匹配
2.  列表操作

第一项相当直接：如果你输入一个左括号 `(`，那么 Vim 会自动在其后插入一个右括号 `)`，然后将光标置于括号之间。这很棒，因为它能防止你忘记自己添加左括号或右括号，从而避免意外导致程序触发错误（这可能是一个相当烦人的 bug，因为在处理深度嵌套的形式时，很难识别出来）。

第二项，列表操作，是这个插件的核心。它提供了以下键位绑定：

*   `<f` 和 `>f`：向左或向右移动“形式”
*   `<e` 和 `>e`：向左或向右移动“元素”
*   `<(` 和 `>(`：向左或向右移动“左括号”
*   `<)` 和 `>)`：向左或向右移动“右括号”
*   `<I` 和 `>I`：向左或向右移动“插入点”

让我们快速演示一下这些键位绑定是如何工作的。假设我们处理的是以下代码（参见清单 B-1），每次操作后，我们都会与清单 B-1 进行比较，看看发生了什么变化。

```
(def myfn [f g h]
(* h (+ f g)))
清单 B-1.
一个处理计算的简单函数
```

#### 形式操作

键位绑定 `<f` 和 `>f` 会影响光标当前所在的最内层形式。因此，如果你的光标位于形式 `(+ f g)` 内部（或之上），并且你执行了 `<f`，那么你会发现该形式向左移动（新布局见清单 B-2）。

```
(def myfn [f g h]
(* (+ f g) h))
清单 B-2.
形式操作的结果
```

#### 元素操作

与形式操作类似，如果你的光标位于原始示例代码中的 `h` 符号（即“元素”）之上，并且你执行了 `>e`，那么你会发现该元素向右移动（新布局见清单 B-3）。

```
(def myfn [f g h]
(* (+ f g) h))
清单 B-3.
元素操作的结果
```

注意

你会看到清单 B-3 达到了与清单 B-2 相同的输出，但这是通过两种不同的机制实现的。

#### 括号操作

如果你的光标位于 `(+ f g)` 中的 `(` 字符之上（或内部），并且你在原始示例代码上执行了 `2<(`，那么你会发现该括号向左移动了两个位置（新布局见清单 B-4）。在这个例子中，我对 `<(` 使用了 Vim 的计数 `2`，因此它被执行了两次。

```
(def myfn [f g h]
((* h + f g)))
清单 B-4.
括号操作的结果
```

注意

这段代码在运行时现在会导致错误，但我这样做是为了演示该功能（由于自动括号插入功能，我很少需要操作单个括号）。

#### 插入点操作

如果你的光标位于某个形式之上（或内部），执行 `<I` 会将光标移动到该形式的开头，并自动进入 Vim 的插入模式。例如，将光标放在 `(+ f g)` 内部并执行 `<I`；你应该会看到光标被放置在 `+` 之前，准备开始输入。

类似地，如果你将光标放在 `(* h ...)` 内部，确保它位于嵌套在内的 `(+ f g)` 形式之外，然后执行 `>I`；你会看到光标被放置在该形式右括号之前。这个功能对于快速定位到长而复杂的形式的开头和结尾非常有用。

### Fireplace

Fireplace 插件是重头戏。使用这个插件，你可以将 Vim 连接到网络 REPL（在我们的例子中是 Leiningen，但如果你愿意，也可以使用其他 REPL），并直接在 Vim 内部动态评估你的 Clojure 代码。这将使你能够即时获得关于你编写的代码及其实际行为的反馈。这是一种极其强大且富有启发性的开发体验。

注意

很多人对这个插件的名字感到困惑：为什么叫 Fireplace？奥秘就在名字里：fi-REPL-ace（注意中间的 REPL）。



#### 所需步骤

我们需要执行的主要步骤如下：

1.  启动 REPL。
2.  在 Vim 中打开一个 Clojure 文件。
3.  运行 Fireplace 的 `:Connect` 命令。
4.  在 Vim 提示符中输入 REPL 的详细信息。

完成这些步骤后，我们就可以开始使用 Fireplace 特有的键位绑定来求值我们的 Clojure 代码了。

#### 首选工作流程？

你的工作流程可能与我不同（大多数 Vim 用户都有自己的做事方式）。对我来说，我使用 NeoVim，⁷ 因此我可以创建一个包含实际终端实例的分屏窗口（例如，使用其 `:terminal` 命令）。所以，我会有一个水平分割的窗口集。顶部窗口是我的 Clojure 代码，底部窗口是我的终端（也就是我启动 REPL 的地方）。

你可能不使用 NeoVim，但这没关系。如果你是 tmux ⁸ 用户，那么你可以有一个水平分割的窗格，效果基本相同。如果你不使用像 tmux 或 screen ⁹ 这样的终端复用器，那么你可以简单地启动两个 shell/终端实例；在我即将详述的初始设置步骤中，你需要在它们之间来回切换。

#### 连接到 REPL

好的，那么让我们从启动 REPL 开始：`lein repl`

在使用 Fireplace 插件从 Vim 连接到 REPL 之前，我们需要打开一个 Clojure 文件；这是因为 Fireplace 只有在找到 Clojure 格式的文件时才会加载其功能（如果你尝试从其他文件类型运行其任何命令，Vim 会提示无法识别该命令）。

在 Vim 中，你可以选择打开一个 Clojure 文件，或者简单地打开一个空缓冲区并将该缓冲区的格式设置为 Clojure 文件；例如，`setf clojure`。选择哪种方式并不重要。通常，如果我只是想摆弄一些 Clojure 代码，我会用空缓冲区启动 Vim 并使用 `setf clojure`（我相信有些人会主张直接使用 Leiningen 的 REPL，但我更喜欢在 Vim 内部工作）；否则，如果我在处理一个实际项目，我会打开我想要编辑的 Clojure 文件。

完成这些操作后，Vim 就可以使用 Fireplace 提供的 `:` `Connect` 命令了。使用此命令，你可以使用 Tab 补全来填写大部分必需的值；直接输入值，例如 `:Connect <VALUE>`；或者按 ENTER 让 Vim 自动提示你输入答案。我通常直接输入我们需要的 `:Connect nrepl://127.0.0.1:65103`。

如果你想知道我是从哪里得到 `127.0.0.1:65103`（即 `<LOCAL_IP>:<PORT_NUMBER>`）的，那么请看运行 `lein repl` 的那个屏幕。你应该会看到输出的第一行类似于清单 B-5 所示的内容，连接成功后你会看到一条类似于清单 B-6 的消息。

```
nREPL server started on port 
on host  -
nrepl://:
清单 B-5.
来自 lein repl 命令的抽象输出
```

注意

你可以使用 `localhost` 代替 `127.0.0.1`

```
Connected to nrepl://:
Scope connection to: 
清单 B-6.
连接到 REPL 后的 Vim 输出
```

要关闭此消息，请按 ENTER。

注意

`:` `Connect` 命令的语法结构是：`<PROTOCOL>://<LOCAL_IP>:<PORT_NUMBER> <PATH/TO/CODE>`。但请注意，项目代码的路径不应包含空格。例如，`∼/Some/Project` 没问题，但 `∼/Some Other/Project` 则不行。

#### Fireplace 命令

Fireplace 有相当多的键位绑定和命令，所以我强烈建议你阅读 Fireplace 的帮助文档（`:help fireplace`），因为我不会尝试涵盖所有内容。说真的，有些我甚至从未用过！但我最常用的那些，我在这里包含了详细信息（我鼓励你尝试给出的示例，以便熟悉它们）。

*   `:Doc <symbol>`（例如，`:Doc defn`）
*   `:FindDoc <symbol>`（例如，`:FindDoc de`）
*   `:Source <symbol>`（例如，`:Source defn`）

注意

当光标位于某个符号上时，执行 `K` 键位绑定是 `:Doc` 命令的快捷方式。如果需要退出 `:FindDoc` 结果屏幕，你可以按 Q 或一直滚动到底部然后按 ENTER。另外，当光标位于某个符号上时，执行 `[d` 或 `]d` 键位绑定是 `:Source` 命令的快捷方式。

#### Fireplace 键位绑定

*   `cpp`：求值最内层的表单
*   `cqp`：显示提示符以输入自定义表达式
*   `cqq`：显示提示符，并预填充当前表单
*   `cqc`：在命令行窗口内显示提示符
*   `cmm`：展开一个 Clojure 宏

在清单 B-7 中，我们可以看到名为 `calc` 的示例函数，我们将以此为基础来探索前面列表中显示的键位绑定。为了后续章节的演示，让我们将该代码添加到一个新的空缓冲区中，并使用 `setf clojure`（而不是从实际项目目录中运行它）。

```
(defn calc [f g]
(* (+ f g) (/ g f)))
清单 B-7.
用于验证键位绑定的示例函数
```

##### cpp

将光标放在清单 B-7 的第一行（即 `(defn calc [f g]`）上，然后执行键位绑定 `cpp`。你应该会看到结果显示在 Vim 实例的底部：`#'user/calc`。

你会注意到命名空间是 `user`，因为我们是在一个尚未写入文件的缓冲区中工作。如果我们是在一个真实的 Clojure 项目中工作，你可能会看到类似 `#'<path/to/your/project/namespace>/calc` 的内容。

如果你将光标放在一个函数调用上（例如 `(calc 100 50)`）并执行 `cpp` 键位绑定，那么它将求值该调用并返回实际结果。但请注意，如果（在此示例中）你还没有对 `calc` 函数定义本身执行过 `cpp`（即 REPL 尚未求值该函数定义），那么你会发现 REPL 无法求值对 `calc` 函数的调用，因为它不知道在哪里找到它。这将导致一个错误（参见清单 B-8）。

```
CompilerException java.lang.RuntimeException:
Unable to resolve symbol: calc in this context, compiling:(4:1)
清单 B-8.
对未求值的函数进行调用时出现的错误
```

当你忘记求值函数定义时，就会发生这种类型的错误；例如，你输入了大量代码，然后试图直接调用它，而没有先求值各个定义。有时，一次性重新求值整个缓冲区会更简单，你可以使用 `:%Eval` 来实现（你可以将其绑定到自定义键位上，使其执行起来更快更简单）。

##### cqp

在当前缓冲区的任何位置执行键位绑定 `cqp`，Vim 都会显示一个命令提示符，让你输入一个新的表达式进行求值。和之前一样，如果你尝试求值代码中的定义，请确保通过求值你想要使用的定义，或者先重新求值整个缓冲区，让 REPL 知道它们的存在。

##### cqq

`cqq` 键位绑定的工作原理是：获取当前缓冲区中光标所在位置的表单，并将其放入 Vim 命令提示符中，然后你可以在按 ENTER 求值表达式之前修改该命令。

注意

我发现 NeoVim 在使用 `:terminal` 窗格时，不会显示 `cqq` 键位绑定的求值结果。求值确实发生了（我们可以通过检查 `:Last` 来知道这一点，它会在预览窗口中显示最后一次 Clojure 求值结果）；只是没有显示出来。这被认为是一个 bug。



##### cqc

`cqc` 键绑定可在当前缓冲区的任意位置执行。它会打开 Vim 命令行窗口，允许你搜索之前的求值结果，并在重新求值前进行修改。

##### cmm

`cmm` 键绑定对于理解宏在抽象层之下的工作原理（即宏展开后的代码是什么样子）非常有用。以我们一直在使用的 `calc` 函数为例；如果在定义上执行 `cmm`，宏展开会将其求值为清单 B-9 所示的内容。

```
(def calc (fn* ([f g] (* (+ f g) (/ g f)))))
清单 B-9.
使用 cmm 键绑定展开的 Defn 宏
```

在清单 B-9 中，我们可以看到在 `defn` 宏上执行 `cmm` 会将其展开，从而让我们看到 `calc` 定义实际上只是一个 `def` 形式。它还展示了底层使用匿名函数来实现 `defn` 为我们抽象掉的功能。
