# 7. 模式匹配

模式匹配是基于预定义模式触发特定功能的能力。这些预定义模式应用于传入的消息（或用户输入）。

这种技术有一些相关的变体，被称为运行时多态和动态分发。本质上，这些概念指的是基于运行时已知的接收者来选择实现。模式匹配与之类似，但从实际角度来看，其范围更窄。

Clojure 有几种不同的机制来实现这种形式的运行时功能分发。第一种是使用一个名为 `core.match` 的 Clojure 库，¹ 另一种是使用一个名为 multimethods 的内置特性。

在本章的第一部分，我们将研究 `core.match`，在接下来的部分（“多态”）中，我们将研究 multimethods 的实现。



## core.match

要使用 `core.match`，你需要告知 Leiningen 加载它，因为它并非 Clojure 内置标准核心库的一部分。

目前最简单的方法——因为我们仍在 REPL 中运行所有示例——是停止当前打开的 REPL，并创建一个 `project.clj` 文件（参见代码清单 7-1）。

```
(defproject test "0.1.0-SNAPSHOT"
:dependencies [[org.clojure/clojure "1.8.0"]
[org.clojure/core.match "0.3.0-alpha4"]])
代码清单 7-1.
用于添加 core.match 依赖的新 project.clj 文件
```

创建此文件后，使用 `lein repl` 重新启动 REPL；你会发现 Leiningen 会使用 `project.clj` 的内容来设置其环境。REPL 启动后，你可以尝试执行代码清单 7-2 中的代码来引入 `match` 函数。

```
(require '[clojure.core.match :refer [match]])
代码清单 7-2.

```

不必担心理解项目文件或 `require` 函数调用的语法，我将在后续章节更详细地讨论 Leiningen 时进行解释。

### 示例：FizzBuzz

好了，现在你已经在 REPL 中拥有了 `match` 函数，我们可以开始尝试一些示例了。让我们从一个直接来自 `core.match` README 的简单示例开始（参见代码清单 7-3）。

```
(doseq [n (range 1 21)]
(println
(match [(mod n 3) (mod n 5)]
[0 0] "FizzBuzz"
[0 _] "Fizz"
[_ 0] "Buzz"
:else n)))
代码清单 7-3.
来自 core.match 的模式匹配示例
```

在代码清单 7-3 中，我们看到的是经典的编程面试题：FizzBuzz。其思路是打印从 1 到 20 的数字列表，但对于 3 的倍数，应打印消息“Fizz”而非数字；对于 5 的倍数，应打印消息“Buzz”而非数字。最后，对于既是 3 的倍数又是 5 的倍数，应打印消息“FizzBuzz”。程序的输出应如代码清单 7-4 所示。

```

Fizz

Buzz
Fizz

Fizz
Buzz

Fizz

FizzBuzz

Fizz

Buzz
代码清单 7-4.
上述模式匹配示例的输出
```

那么，让我们花点时间来理解代码清单 7-3 中代码示例的运行机制。首先，我们使用 `range` 函数创建一个从 1 到 20 的数字集合，然后打印对范围内每个元素执行 `match` 的结果。

`match` 语法允许我们定义一些想要匹配的模式类型；在此例中，它匹配的是 `(mod n 3)` 和 `(mod n 5)` 的结果，其中 `n` 是当前范围内的元素（例如，1，然后 2，然后 3，依此类推）。如果当前范围内的数字是 3 或 5 的倍数，那么 `mod` 应返回 `zero`。

接下来是各种预期的模式结果，以及当这些预期匹配时我们希望触发的功能。在代码清单 7-3 的示例中，我们首先检查，如果两个 `mod` 函数的结果都为零，则返回 `"FizzBuzz"`：`[0 0] "` `FizzBuzz` `"`。否则，如果匹配失败，则进入下一个条件模式检查。

下一个预期是检查我们是否能将范围内的数字被 3 整除：`[0 _] "Fizz"`。你会看到我们使用下划线字符 `_` 作为通配符，用于检查范围内的数字是否是 5 的倍数。我们这样做是因为我们不关心该检查返回了什么（因为如果该值是 5 的倍数，那么我们在第一步中就已经找到了完全匹配）。

如果该预期不匹配，那么我们将进入下一个预期：`[_ 0]`。这与前一个检查相反，我们现在在检查数字是否是 3 的倍数时使用 `_`（同样，我们不关心值是什么）。然后我们检查另一个值是否是 5 的倍数，结果应为 `zero`，因此此时应找到匹配并打印单词 `"Buzz"`。

你可以在代码末尾看到我们使用了 `:else` 语句来捕获并在未找到匹配时返回所提供的范围内的数字。此代码提供了成功通过 FizzBuzz 测试所需的所有条件。

### 匹配字面量

在前面的示例中，我们一直基于符号来匹配值。但我们也可以匹配字面量值（参见代码清单 7-6）。

```
(match ['foobar]
['foobar] :bar
['bazqux] :qux)
;; :bar
代码清单 7-6.
使用字面量值进行模式匹配
```

在代码清单 7-6 中，我们传入了一个字面量符号进行匹配。在这种情况下，该符号不指向任何其他值；它实际上求值结果就是它自身。然后我们有一个与之直接匹配的关联模式，因此我们成功匹配并返回相应的值。

现在，如果我们还需要提供一个回退机制（例如，当匹配失败时），那么我们可以使用 `:else` 关键字，如代码清单 7-7 所示。我们在前面实现 FizzBuzz 测试时看到过它的用法，如果未找到匹配，我们既不返回 `"Fizz"` 也不返回 `"Buzz"` 或 `"FizzBuzz"`，而只是返回传入的范围数字。

```
(let [a (+ 1 1)]
(match [99]
[a]   :success
:else :fail))
;; :fail
代码清单 7-7.
使用 :else 语句提供回退
```

### 匹配数据结构

到目前为止，我们已经看到了对表达式和直接字面量值的匹配。但也可以比较数据结构。

```
(match [[:a :b :c]]
[[:a :b _]] :success
:else :fail)
;; :success
(match [{:a 1 :b 1}]
[{:a _ :b 2}] :foo
[{:a 1 :b _}] :bar
:else :baz)
;; :bar
代码清单 7-8.
使用数据结构进行模式匹配
```

从代码清单 7-8 中，我们可以看到有两个示例。在第一个示例中，我们传入了一个向量数据结构；在第二个示例中，我们传入了一个映射。在这两种情况下，我们都能通过使用 `_` 通配符符号轻松找到匹配，以捕获那些我们不在乎或不确定其值的某些值。

注意

使用映射数据结构时，不能在键上使用通配符；它只能用于键的值。

使用向量时，你可能只需要部分匹配。例如，你可能只想匹配集合中的前几个元素。如果是这种情况，你可以使用 `&` rest 参数（你在第 6 章中会认识它）。参见代码清单 7-9 中的示例。

```
(match [[:a 1 :b 2 :c 3 :d 4]]
[[:a 1 :b 2]] "这需要完全匹配"
[[:a 1 :b _ & rest]] "rest 允许部分匹配"
:else :fail)
;; "rest 允许部分匹配"
代码清单 7-9.
使用 Rest 符号 & 进行部分模式匹配
```



### 安全防护

在继续探讨多态性之前，我想演示一下 `core.match` 的其他几个特性，例如，当使用映射数据结构时，你可能会无意中产生部分匹配，以及如何使用 `:or` 修饰符为被验证的数据实现安全防护。

```
(match [{:a 1 :b 2 :c 3 :d 4}]
[{:a _ :b 2}] "映射默认进行部分匹配"
:else :fail)
;; "映射默认进行部分匹配"
清单 7-10.
意外的部分模式匹配
```

在清单 7-10 中，我们可以看到，与向量不同（在向量中你可以使用 `&` 剩余符号来实现显式的部分匹配），映射数据结构默认会触发部分匹配。

你应该意识到这一点，以免意外匹配到你本应避免的内容。在这些情况下，最好定义一个更显式的匹配来防止意外匹配。

或者，你可以使用 `:only` 修饰符来定义你将接受的键，从而防止意外匹配的发生。请参见清单 7-11，其中演示了具体实现。

```
(match [{:a 1 :b 2 :c 3 :d 4}]
[({:a _ :b 2} :only [:a :b :c])] "未匹配，:only 期望三个键"
[({:a _ :b 2} :only [:a :b :c :d])] "匹配成功！"
[{:a _ :b 2}] "此断言永远不会被执行"
:else :fail)
;; "匹配成功！"
清单 7-11.
使用 :only 修饰符避免意外模式匹配
```

从清单 7-11 中我们可以看到，我们有一个包含四个键（`:a`、`:b`、`:c` 和 `:d`）的传入数据结构，并且在第一个匹配中，我们使用 `:only` 修饰符来声明，只有当匹配仅包含我们指定的键时，我们才接受。因此，没有匹配成功，我们进入下一个模式，这里再次存在部分匹配的可能性，但这次我们接受了它，因为我们的修饰符定义是：`[:a :b :c :d]`。

现在，在某些情况下，我们可能希望在一个模式内拥有可变数量的可能匹配。我们可以使用 `:` `or` 修饰符来实现这一点。请参见清单 7-12 中的示例。

```
(match [[1 2 3]]
[[1 (:or 3 4) 3]] :foo
[[1 (:or 2 3) 3]] :bar)
;; :bar
(match [{:a 3}]
[{:a (:or 1 2)}] :foo
[{:a (:or 3 4)}] :bar)
;; :bar
清单 7-12.
使用 :or 修饰符实现每个模式的变量匹配
```

在清单 7-12 中，我们有两个数据结构：一个向量和一个映射。对于向量示例，我们可以看到我们声明期望的数据结构由 `1` 后跟 `3` 或 `4`，再后跟 `3` 组成。如果这没有匹配，那么我们会尝试匹配一个由 `1` 后跟 `2` 或 `3`，再后跟 `3` 组成的数据结构（这实际上是匹配的）。

映射数据结构的示例原理基本相同，但我们使用 `:or` 修饰符来标识键的值。

最后，让我们看看 `:guard` 修饰符。它使我们能够指定一个函数来验证传入的值是否有效（而“有效”的含义取决于所使用的函数）。请参见清单 7-13，了解如何使用此修饰符的示例。

```
(defn div3? [n]
"一个函数，如果参数能被三整除则返回 true，否则返回 false"
(if (= (mod n 3) 0) true false))
(match [[2 3 4 5]]
[[_ (a :guard even?) _ _]] (format "我们匹配了第一个 %d" a)
[[_ (b :guard [odd? div3?]) _ _]] (format "我们匹配了第二个 %d" b))
;; "我们匹配了第二个 3"
清单 7-13.
使用 :guard 修饰符实现类型/值验证
```

在清单 7-13 中，我们可以看到有两个预期的匹配。在第一个匹配中，我们声明除了第二个元素（即值 `3`）之外，我们不关心向量数据结构中的任何其他传入值。但是，我们确实希望确保该值是奇数并且能被三整除。

因此，在第一个模式中，我们告诉 `:guard` 使用 `even?` 函数。我们知道值 `3` 不是偶数，所以该匹配会失败。我们继续到下一个模式，该模式指定了多个用于验证的函数（因此我们将它们放在一个向量内）。第一个用于验证的函数是 `odd?` 函数，它会通过，因为 `3` 是奇数。第二个用于验证的函数是我们自己定义的函数 `div3?`，它同样会通过，因为 `3` 确实能被自身整除。

注意

根据文档，目前不支持内联函数和简写函数。

总的来说，你可以看到模式匹配在各种不同场景下是多么强大的特性。现在，让我们来看看多态性，看看它能为我们提供什么。



## 多态

在大多数面向对象编程语言中，多态是指根据方法当前所属的对象来重新定义其行为的能力。这意味着你可以有一个包含 `speak` 方法的 `Human` 类，调用该方法时会返回人类会说的话（例如“你好”）；但你也可以有一个包含相同 `speak` 方法的 `Dog` 类，调用该方法时会返回更合适的内容，比如 `"Woof!"`。

在函数式语言中，由于类和对象不那么普遍，多态很难实现，因为缺乏类型；或者更具体地说，函数式语言倾向于避免为每个编程场景创建新类型，而在使用面向对象语言时，你通常会为这些场景创建新类型。

Clojure 通过支持运行时多态（也称为动态分发）来解决类型缺失的问题，其核心概念是：分发一个与可识别的多态函数匹配的值，该多态函数将处理给定值的行为。

Clojure 有几种处理方式，但我们感兴趣的是通过使用一种称为 multimethod 的特性。让我们看一个简单的 multimethod 示例（见清单 7-14），以了解它如何在 Clojure 中实现多态。

```
(defmulti foo :some-key)
(defmethod foo :a [this] (str "foo :a given " this))
(defmethod foo :b [this] (str "foo :b given " this))
(defmethod foo :c [this] (str "foo :c given " this))
(foo {:some-key :a}) ;; "foo :a given {:some-key :a}"
(foo {:some-key :b}) ;; "foo :b given {:some-key :b}"
(foo {:some-key :c}) ;; "foo :c given {:some-key :c}"
清单 7-14.
Multimethod 功能示例
```

我们首先调用宏 `defmulti`，并传入一个名称（在本例中，名称是符号 `foo`）。我们还提供了一个分发函数（在我们的示例中，参数是 `:some-key`）。

注意

你可能还记得，关键字在用于映射数据结构时会像函数一样工作。

至此，我们已经定义了一个名为 `foo` 的 multimethod，现在需要为它可能关联的每种不同类型定义其行为。这就是 `defmethod` 宏的用武之地。

你可以看到，在定义每个 `defmethod` 时，我们确保给它与 `defmulti` 相同的名称（`foo`），以及一个我们想要匹配的值（在本例中，我们为 `foo` 的每个变体定义了 `:a`、`:b` 和 `:c`）。

在每个 `defmethod` 定义的方括号中，你可以看到我们传入了一个符号（在本例中是 `this`），它引用了该方法关联的类型。因此，在我们的示例中，符号 `this` 的值将是 `{:some-key :a}`、`{:some-key :b}` 或 `{:some-key :c}`，具体取决于调用 `foo` 时匹配到的 `foo` 变体。

注意

我将类型命名为 `this`（秉承面向对象的精神，其自身值通常被称为 `this`），但你可以随意命名。

每个 `defmethod` 的主体是方括号后面的部分，它定义了每个 `defmethod` 版本应有的行为。你可以根据需要放置任何内容。在我们的示例中，我们只是根据匹配的类型返回相关的 `this` 值。

这意味着，正如你在清单 7-14 的示例中所见，我们现在可以调用带有映射数据结构的 `foo`，并触发相应的下游行为。作为参考，语法结构如清单 7-15 所示。

```
(defmulti
多态函数名称
分发函数)
(defmethod
多态函数名称
要匹配的模式值
[类型符号]
函数行为)
(多态函数名称
某些数据)
清单 7-15.
Multimethod 语法结构
```

在继续之前，让我们再看一个快速示例，以演示 `:` `default` 选项的工作原理。我们将沿用之前的示例，但稍作扩展（见清单 7-16）。

```
(defmulti foo :some-key)
(defmethod foo :a [this] (prn "A"))
(defmethod foo :default [this]
(prn (str "抱歉，不知道如何处理 '" (:some-key this) "' ?")))
(foo {:some-key :a}) ;; "A"
(foo {:some-key :d}) ;; "抱歉，不知道如何处理 ':d' ?"
清单 7-16.
演示 :default 选项
```

如你所见，我们定义了一个新的 `foo` 类型，如果找不到其他匹配项，它将由 `:default` 匹配。我们没有定义期望值为 `:d` 的 `defmethod`，因此 `:default` 将作为回退匹配。

## 总结

在本章中，我们学习了模式匹配概念的两个方面：第一个是严格意义上的模式匹配，使用 `core.match`；第二个是通过多态方法，能够使用动态分发来触发特定行为。对于这两种风格，我们研究了各种语法差异，以及如何保护我们的代码免受可能没有匹配项的输入的影响。

在下一章中，我们将探讨一个重大主题：并发。在那里，我们将介绍许多不同的选项来处理异步代码，这也有助于提高我们应用程序的性能和安全性。
