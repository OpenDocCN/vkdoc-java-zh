# 8. 并发

Clojure 作为函数式语言的主要卖点之一是其并发机制（它拥有相当多的并发机制）。为了理解并发的含义，我们首先需要了解它相关的问题空间，以及程序并发运行任务意味着什么。

一个计算机进程（例如，在你的计算机上运行的一个程序实例）能够生成多个“线程”。每个线程都有能力执行自己独特的一组任务，并且与其他线程中执行的任务是分开的。但是，从同一进程生成的线程共享相同的内存空间。

从同一父进程生成的多个线程能够访问相同的内存空间这一事实可能是一个问题，因为如果我们并发执行代码（即 CPU 在执行线程之间进行“上下文切换”），那么意味着对内存中某块数据的更改可能来自多个地方。

这就是为什么 Clojure 提供了不同的机制¹来处理并发，并允许以同步和协调的方式进行更改。但是，在开始使用 Clojure 时，很难理解为什么有这么多方法可以以并发方式处理不可变数据，更重要的是，何时应该使用它们。

我发现最简单的解决方案是使用一个矩阵（表 8-1）。

表 8-1.

在表 8-1 中，你会看到我们有四个特性：可重试、协调、异步和线程安全。让我们回顾一下每个特性的含义，以便更好地理解该矩阵。



## 可重试

对于所列类型之一，若要被视为“可重试”，必须发生了冲突（例如，已应用了多项更改），并且解决冲突必须需要重试该操作。

解决冲突的一种实现方式是 CAS（比较并交换）操作，这是一种底层操作，语言 API 不会直接暴露它。一个典型场景是，当尝试更新某个值因跨线程冲突（例如，两个线程同时更新同一个值）而失败时。

CAS 操作的“比较”部分首先检查是否有任何变化。例如，它会检查当前值是否已被另一个线程篡改。如果是这种情况，则会执行该操作的“交换”部分，并应用请求的更新。另一方面，如果 CAS 检测到已经发生了更改，那么它将获取更新后的值并重试该操作。

注意

STM（我们稍后会介绍）也被认为是可重试的，尽管它会重试操作，但这与 CAS 本身（一种底层原语）无关。

## 协调

对于所列类型之一，若要被视为“协调”，我们需要使用 Clojure 内置的 STM（软件事务内存）。STM 会验证跨线程数据的一致性。

STM 和 CAS 之间的区别在于，STM 会协调（即检查）多个引用，而 CAS 只关心它自己的引用。

“引用”这个概念目前可能无助于区分 CAS 和 STM，所以，别担心，稍后当我们研究如何创建和使用 `Ref` 类型时，我会更详细地讨论 STM，届时 STM 如何被视为“协调”而 CAS 被视为“非协调”应该会更清晰。

## 异步

对于所列类型之一，若要被视为“异步”，它不应阻塞其当前线程的处理。你可能会发现，某个特定类型可能不会阻塞，要么是因为它确实是“异步”的（即，它在另一个线程中运行），要么是因为它不是一个“阻塞”调用，而只是等待其值在稍后的某个时间点被解析。

## 线程安全

对于所列类型之一，若要被视为“线程安全”，它必须能在多个线程试图修改共享值时防止冲突。我们已经了解到，这可以通过使用 CAS 或 STM 来实现，但我们将在本章后面看到实际的代码示例。

现在，让我们来看一些每种类型的示例，以便我们更好地理解何时以及如何使用它们。

## Delay

`delay` 函数是异步的，因为当你定义其行为时，该行为不会立即执行，而是等到将来某个时刻你“解引用”它时才会执行。正因如此，`delay` 函数不像原子或 ref 那样是一种严格的并发机制，尽管人们通常不认为它属于同一家族，但它仍然有能力阻塞你的程序，所以我认为将其纳入本章是相关的。

澄清一下，解引用意味着获取值（对于 `delay` 来说，你是在强制某些行为发生）。你会看到所有并发机制——agent、atom、ref 等——都使用 `(deref …)` 函数来获取包含的值。

让人困惑的是，解引用一个值有时会导致当前线程阻塞，但这取决于你使用的机制以及代码运行的上下文。正如我们稍后将看到的，agent 即使在被解引用时也是非阻塞的。

另一方面，如果值已经可用（即，它已经被计算出来），那么调用 `deref` 会立即返回该值。但如果值尚未准备好，那么它可能仍在另一个线程上计算，因此你当前进行解引用调用的线程将被阻塞，直到值准备好。

注意

`(deref x)` 的一个便捷简写是 `@x`，其中 `x` 是一个 delay/promise/future/atom/agent/var/ref。

`delay` 函数阻止一个动作立即发生。它是异步的，直到被解引用时为止，此时它会阻塞，直到行为完成。一旦 `delay` 被执行，它会缓存其返回值（示例见清单 8-1）。

```
(def later (delay (prn "hello")))
;; 立即返回，不阻塞
;; ...在应用程序稍后的某个时刻...
@later ;; nil (打印 "hello")
@later ;; nil (注意这次没有打印 "hello")
清单 8-1.
使用 delay 的示例
```

在清单 8-1 中，我们可以看到，对 `later` 变量的第二次解引用调用不再打印消息 `"hello"`。这是因为分配给 `delay` 的动作已经被计算并缓存了。在这种情况下，没有返回值，因此返回了 `nil`。

使用 `delay` 时，你也可以使用 `force` 函数来代替 `deref` 或 `@`（示例见清单 8-2）。Clojure 开发者倾向于使用 `force`，因为它在语义上感觉更准确（因为你是在强制 `delay` 执行）。所以，选择你觉得更自然的方式。我个人使用 `@`，这样我的代码与其他也使用该符号的可用机制保持一致。

```
(def d (delay (prn "hello")))
;; 立即返回，不阻塞
(force d)
;; nil (打印 "hello")
清单 8-2.
使用 force 的示例
```

注意

你可以将超时机制与 `deref` 函数一起使用，但这仅适用于使用 `future` 或 `promise` 时（我将很快演示超时如何工作）。但请注意，超时功能不适用于 `@` 宏。

## Promise

`promise` 很像 `delay`，因为它在创建时不会阻塞当前线程，直到被解引用。`promise` 与 `delay` 的不同之处在于，它需要先有一个值被“交付”，然后才能成功解引用。

`promise` 的好处是，要交付的值在创建 `promise` 时不必存在。该值可以是另一个函数的结果。

这就是为什么 `promise` 作为“回调”功能很有用；你可以将 promise 传递给其他函数，意图是解引用它们，同时使用单独的机制来交付其值。参见清单 8-3。

```
(def foo (promise))
(future
  (prn "子线程正在处理...")
  (Thread/sleep 10000)
  (deliver foo :bar))
@foo
;; :bar (阻塞直到 promise 有值被交付)
清单 8-3.
promise 的示例
```

注意

promise 的值一旦交付就不能更改。如果你尝试交付一个新值，它将被忽略。

在清单 8-3 中，我们创建了一个 `promise` 并将其赋值给 `foo` 变量。然后我们使用一个 `future`（我将在下一节中更详细地讨论）来创建一个新线程，并从该独立线程向 `promise` 交付值（本例中为 `:bar`）。

最后，我们在父线程中解引用该 `promise`，并注意到该调用会阻塞父线程，直到 `future` 的线程完成处理并随后交付了 `promise` 的赋值。

`promise` 只是一个将保存值的容器。由你的应用程序决定 `promise` 何时实际接收一个值，它通过使用 `deliver` 调用传递一个值来实现这一点。

在你的代码中任何期望一个值的地方，你都可以使用 `promise`。如果你有一个接受参数的函数，那么你可以传入一个 `promise`，并在未来的某个时间点解析它。



## Future

`future` 允许在另一个线程中执行代码，这意味着它是非阻塞/异步的。如果你希望将某个特定的处理任务交给另一个线程，那么 `future` 就是你想要使用的工具。

注意

Clojure 会从线程池中创建线程，并在内部为你管理这些线程。

在 `future` 创建后的任何时刻，你都可以解引用它来获取其值。如果 `future` 的线程已经执行完毕（即，已完成值的计算），那么调用 `deref` 将返回计算出的值。

否则，如果在调用 `deref` 时 future 仍在运行，那么尝试解引用该 future 的线程（通常是主线程/父线程）将会被阻塞，直到该值可用为止。

在清单 8-3（见前文）中，我们使用 `future` 创建了一个子线程，并在该子线程中故意“休眠”了十秒钟才返回一个值，以模拟阻塞式 I/O 交互。

运行清单 8-3 中代码的结果是：当父线程解引用该 `future` 时，它被迫等待了十秒钟才返回一个值（因为当主线程解引用它时，该 `future` 尚未完成值的计算）。

正如之前在讨论 `delay` 函数时提到的，在你预期 `future` 可能会阻塞的场景下，更推荐使用 `deref` 而不是简写形式的 `@` 宏，原因是长格式的 `deref` 允许你指定一个超时时间，并在超时发生时提供一个回退值。参见清单 8-4。

```
(def f (future
(Thread/sleep 10000)
(println "done")
100))
(deref f 500 "fail")
;; "fail" (仅阻塞直到达到 500ms 超时)
清单 8-4.
设置超时和回退值的示例
```

在清单 8-4 中，我们可以看到，当我们使用 `deref` 时，我们传入了几个额外的参数。第一个是 `500`，这是我们打算等待响应的毫秒数。第二个是 `"fail"`，如果超过了 500ms 的阈值，这将是返回值。

重要的是要认识到，在独立线程中执行的任务会继续计算。超时是针对解引用值的操作，而不是针对执行线程本身。因此，在我们的示例中，如果我们继续等待，然后在稍后某个时间执行 `(deref f)`，我们会看到值 `100` 被成功计算出来。

## Atom

Atom 既是线程安全的，也是可重试的。这意味着 atom 使用一种称为 CAS（比较并交换）的技术来确保线程安全，从而避免了使用锁机制（我将在下一节讨论）的复杂性。

使用 CAS 时，如果两个线程试图修改一个共享变量，每个线程都可以进行自己的更改，而不会被另一个线程阻塞。这样做的原因是，在 CAS 操作将更改提交给 atom 之前，它会验证 atom 的值是否没有改变。如果值没有改变，那就很好——它会提交新值；但如果值已经改变（例如，可能另一个线程先修改了它），那么它将使用新的 atom 值重新启动操作。

更新 atom 的值需要使用 `swap!` 函数，该函数接受你想要修改的 atom 的名称以及一个将应用于 atom 值的函数作为参数（参见清单 8-5）。

```
(def counter (atom 0))
(swap! counter inc) ;; 1
清单 8-5.
使用 Atom 的示例
```

注意

`!` 后缀表示一个潜在的不安全方法；在函数式语言中，不安全通常意味着它会改变状态。

正如你在清单 8-5 中所见，我们创建了一个新的 `atom`，将其初始值设为 0，并将其赋值给 `counter` 变量。然后我们使用 `swap!` 函数来修改 atom 的当前值。

所以，在我们的例子中，我们使用了 `inc` 函数，但如果你需要以特定方式修改值或接受多个参数，你也可以使用匿名函数（或其简写形式）。

```
(def counter (atom 0))
(swap! counter inc) ;; 1
(swap! counter #(+ 2 %)) ;; 3
(swap! counter #(+ 2 %1 %2) 3) ;; 6
清单 8-6.

```

在清单 8-6 中，我们使用 `inc` 函数递增了 atom。接下来，我们使用一个匿名函数再次更新了 atom 的值。最后，我们使用相同的技术第三次更新了 atom，以演示 `swap!` 函数如何接受多个参数，然后将这些参数传递给指定的函数（即我们的匿名函数）。

注意

当使用用户自定义函数时，请确保它是幂等的（即，无副作用），因为该操作可能会被重试，因此副作用也会被重放。

在清单 8-7 中，我们利用 `set-validator!` 函数来确保 atom 新计算出的值符合我们的预期（如果不符合，该值将被丢弃）。在给定的示例中，我们希望确保更新后的值始终是偶数。这类似于我们在第 5 章中学到的函数的前置-后置条件行为。

```

在清单 8-7 中，你会看到我们的验证器函数确保 atom 只能被设置为偶数。如果我们单独使用 `inc`，它将产生一个奇数，因此会触发错误并阻止更改。

通过将指定 atom 的验证器函数重置为 `nil`，我们有效地从 atom 中移除了验证器。回顾清单 8-7，你会注意到在我们赋值 `nil` 之后，设置一个奇数（通过 `inc` 函数）会被成功接受。

`set-validator!` 也适用于 agent/var/ref。

有时你可能想重置原子的值，而不必关心当前值。为此，你需要使用 `reset!` 函数：`(reset! counter 0)`。

你可能还希望仅当原子的当前值与某个预设条件匹配时才更改其值。为此，你可以使用 `compare-and-set!` 函数。

`compare-and-set!` 函数的工作原理是：如果原子的当前值与指定的值匹配，则返回 `true`（随后会继续应用请求的更改）；否则，如果当前值不匹配，则返回 `false` 且不应用更改。清单 8-9 展示了其工作原理。

```
(def counter (atom 0))
(swap! counter #(+ 4 %))
@counter ;; 4
(compare-and-set! counter 4 0) ;; true
@counter ;; 0
(compare-and-set! counter 4 1) ;; false
@counter ;; 0
清单 8-9.

```

请注意，在清单 8-9 中，第二次调用 `compare-and-set!` 函数后，`counter` 原子的值如何保持为零。这是因为我们指定仅当当前值为 4 时才更改值，而当前值并非 4（在此之前它已被更改为零）。

## 锁

使用锁是解决多线程世界中并发问题的经典第一步。当有多个线程试图访问同一内存空间时，就会产生冲突。锁是一种处理并发的常见底层机制，但很难正确使用。这就是 Clojure 提供更高级抽象以实现“线程安全”的原因。

其工作方式是在要操作的数据周围应用一个锁，从而阻止其他线程在数据被“锁定”期间对其进行操作。例如，如果有两个线程同时运行（A 和 B），并且都想修改变量 `foo`，那么你就会面临一个不确定的事件（即，你不知道 A 和 B 谁会先访问到 `foo`）。

通过在代码中使用锁，如果结果是线程 B 先访问到 `foo`，那么由于它先到达，它就能在 `foo` 周围应用一个锁，因此当线程 A 试图访问 `foo` 时，它无法访问，必须等待直到线程 B 完成。当线程 B 完成时，锁被释放，这样线程 A 就可以获取锁，并在持有锁期间阻止任何其他线程访问 `foo`。

锁并不理想，因为在试图确定获取锁的正确顺序时，它们可能变得非常复杂。这之所以重要，是因为锁机制可能导致通常所说的死锁/活锁。这在利用多个锁的系统中更为明显（而不是仅使用单个锁的简单示例）。

### 死锁

死锁是指两个或多个线程陷入困境，因为它们都在等待另一个线程完成（但另一个线程也在等待其他被锁定的线程完成），这意味着无法取得任何进展，系统被锁定。

一个常见的例子是两个人互相鞠躬。这两个人必须遵守的规则是，你必须保持鞠躬姿势，直到对方有机会回礼。但如果两人同时鞠躬，那么他们将永远等待对方。

### 活锁

活锁在某些方面与死锁相似，但略有不同：两个或多个线程无法进展，不是因为它们被阻塞，而是因为它们彼此之间忙个不停。

通常给出的例子是走廊里的两个人不断向同一侧移动以避免对方；他们没有被阻塞，只是永远忙于试图让对方通过。

注意

原子与锁的不同之处在于，它使用 CAS（比较并交换）而非锁机制；这意味着锁不会像原子那样“重试”。

清单 8-12 演示了如何通过原子控制对共享变量的访问。在该示例中，我们想要控制如何将项目添加到向量中。我们希望结果是 `[1 2]`。

现在，在查看清单 8-12 之前，我们应该澄清一下，原子（正如我们在上一节中看到的）已经是线程安全的，因为它们使用 CAS 机制来防止多线程更改造成混乱，因此没有太多理由选择非常底层的锁机制而不是像 CAS 这样的抽象。

```
(def foo (atom []))
(future
(locking foo
(Thread/sleep 1000)
(swap! foo #(conj % 1))))
(locking foo
(swap! foo #(conj % 2)))
@foo
;; [1 2]
清单 8-12.
锁定代码块的示例
```

如果我们没有在 `swap!` 执行周围使用 `locking` 机制（你会注意到我们定义了两个锁定块），那么将会出现不一致的执行路径。

想象一下，如果在清单 8-12 中没有使用锁定块，会发生什么情况：在 `future` 线程内部调用的 `sleep` 意味着，在 `future` 线程内部的 `swap!` 完成计算之前，`future` 线程外部的 `swap!` 调用会先完成，因此结果将是 `[2 1]`，而不是我们期望的 `[1 2]`。

注意

在清单 8-12 中，你会注意到我们必须锁定 `foo` 两次。仅从子线程内部锁定 `foo` 不足以阻止该值被交换出去。这就是在实际应用中事情变得复杂的地方：在我们这个愚蠢的例子中，访问 `foo` 的代码彼此紧挨着，但在实践中很少出现这种情况，因此你很容易遗漏应该应用锁的地方，从而导致应用程序以意想不到且难以调试的方式崩溃。



## Agent（代理）

Agent 有点像异步版本的 Atom（原子）：它是一个共享的可变值，非阻塞（直到你尝试解引用该值），并且也是非协调的。

注意

我们尚未讨论协调的含义，但将在下一节中介绍。目前，只需知道非协调类型（如 agent 或 atom）仅表示一个独立的值即可。

要修改 agent 中包含的值，你有两种方法可用：

1.  `send`  
2.  `send-off`  

这两个函数具有相同的签名 `(a f & args)`，这意味着它接受一个对 Atom 的引用（`a`），后跟一个应改变其值的函数（`f`——这通常被称为动作），再后跟该动作所需的任意数量的参数（`& args`）。请参见清单 8-13。

```
(def a (agent 0))
(send a inc)
@a
;; 1
清单 8-13.
向 Agent 发送值的示例
```

注意

你可以向一个 agent 触发多个动作，它们会被存储在队列中并按顺序处理。

现在，`send` 和 `send-off` 之间有一个细微差别需要你注意：前者在从托管线程池中选取的独立线程中运行，而后者则在一个专门为该 agent 创建的新线程中运行。

大多数情况下你会使用 `send` 函数，但如果你需要应用一个 I/O 阻塞动作，那么你应该改用 `send-off`，因为这能让该线程从容处理，最终不会耗尽共享线程池的资源。

Agent 与其他类型（如 atom、future 和 promise）的一个重要区别在于：获取 agent 的值（解引用）不会导致当前线程阻塞。请参见清单 8-14。

```

(future
(Thread/sleep 5000)
(prn "increment the value")
(send a inc))
@a ;; 0
;; 5 seconds later...
@a ;; 1
清单 8-14.

```

这两个函数的区别在于：后者允许你指定一个超时时间：`(await-for timeout agent)`。让我们看一些示例来演示如何使用这些函数。

### Agent 错误

如果你导致一个 agent 出错，那么它会静默地出错（取决于错误类型）。过去，我曾导致一个 agent 失败，然后后来疑惑为什么我通过 `send` 调用发送的操作都没有被执行。让我们看一个导致 agent 失败的例子以及它的反应，如清单 8-18 所示。

```
(def a (agent 0))
(send a #(/ % 0))
;; 导致 agent 进入失败模式（未显示错误）
a
;; #object[clojure.lang.Agent 0xf1c5585 {:status :failed, :val 0}]
清单 8-18.
Agent 失败示例
```

注意

清单 8-18 的最后一行引用了该 agent，因此我们看到了其内部表示。在该表示中，你会注意到 `:status :failed`。

这种静默地变为失败状态的问题在于，所有未来的 `send` 命令也会失败，直到 agent 的状态被改回 `:ready`。

如果你需要验证 agent 收到了什么错误，可以调用 `agent-error` 函数并传入你想要检查的 agent；这将返回指定 agent 的最后一个已知错误。

此时，我们可以做几件事：第一是为你的 agent 定义一个错误处理器，这样当问题发生时就会变得清晰；第二是在你的代码意识到问题后重启 agent。参见清单 8-19。

```
(def a (agent 0))
(defn h [a e]
(prn "Agent value: " @a)
(prn "Agent error: " e))
(set-error-handler! a h)
(send a #(/ % 0)) ;; 打印以下内容...
;; "Agent value: " 0
;; "Agent error: " #error{:cause "Divide by zero"...
清单 8-19.
Agent 错误处理示例
```

注意

要移除错误处理器，你需要再次调用 `set-error-handler!` 函数，并传入 `nil` 作为处理器值。

此时，agent 处于失败状态，因此如果你尝试向它发送另一个操作，该操作将无法执行。要解决这个问题，我们需要重启 agent，可以使用 `(restart-agent a 0)` 来实现，其中我们向 `restart-agent` 函数提供要修改的 agent（`a`）以及我们希望 agent 重启时使用的新值（`0`）。

如果你希望用 agent 的最后一个已知值来重启它，可以将提供的值替换为 agent 的解引用值：`(restart-agent a @a)`。

现在，你可能希望 agent 自动重启。这是可能的，但并不像你想象的那么简单（例如，仅仅在错误处理器中调用 `restart-agent`）。清单 8-20 将演示一种可能的解决方案，该方案利用 `future` 将对 `restart-agent` 的调用与错误处理器解耦。

问题的真正原因在于，agent 实际上直到错误处理器执行完毕才会被标记为“失败”。这就是为什么使用 `future` 可以解决这个问题，因为它意味着错误处理器可以结束（因为 `future` 在单独的子线程上异步执行），并且 agent 的重启将会成功。

我通过将对 `restart-agent` 的调用包装在 try/catch 语句中发现了这一点，这揭示了消息“Agent does not need a restart.”

注意

重启 agent 时，任何已排队的操作将继续被处理，除非你在重启 agent 时提供了 `:clear-actions true` 选项：

```
(restart-agent a @a :clear-actions true)
```

```
(def a (agent 0))
(defn h [a e]
(prn "Agent value: " @a)
(prn "Agent error: " e)
(future (restart-agent a 0)))
(set-error-handler! a h)
(send a inc)
a ;; 1
(send a #(/ % 0))
;; 错误处理器 h 将被触发
(send a inc)

## 事务

Clojure 自带了自己的 STM（软件事务内存）。本质上，STM 的工作方式很像 CAS（比较并交换）操作，我们在之前讨论原子时已经见过。它们之间的区别在于，STM 不像原子（它是“非协调的”）那样只关心单个值，而是能够同时引用多个值（因此，STM 被认为是“协调的”）。

使用 STM 时有几个活动部件：

*   `dosync`
*   `ref`
*   `ref-set`
*   `alter`
*   `ensure`
*   `commute`

### dosync/ref/alter

第一个项（`dosync`）是事务容器，所有与 STM 执行相关的内容都应发生在这个形式内。第二个项（`ref`）是“引用”类型，它将持有要存储/验证/更新的值。

所有其他列出的项都是只能从事务内部使用的函数。这意味着任何尝试在 `dosync` 形式外部使用 `ref-set`、`alter`、`ensure` 或 `commute` 函数的操作都会导致错误。参见清单 8-21。

```
(def r (ref 0))
(dosync
(alter r inc))
;; 1
清单 8-21.
简单事务
```

在清单 8-21 中，我们创建了一个新的 `ref` 类型 `r`，并为其分配了初始值 `0`。设置完成后，我们使用 `dosync` 打开一个事务，并在其中将 `ref` 实例传递给 `alter` 函数，该函数将提供的函数（本例中为 `inc`）应用于 `ref` 当前持有的值。

与原子类型非常相似，我们需要确保 `dosync` 块内发生的任何操作都是幂等的，因为如果不同引用之间发生冲突，STM 可能会重新启动/重放整个事务。

### ref-set

在清单 8-22 中，我们将使用 `ref-set` 来更改 `ref` 实例的值。这允许我们使用另一个直接值来更改值，而不是使用函数来修改值。

```

(dosync
(ref-set r 0))
;; 0

```



### STM 重启策略

在清单 8-23 中，我们将看到如何使用 `alter` 函数来修改 `ref` 的值。这很可能是在处理 STM 时最常用的操作。STM 会尝试检测 `ref` 在其事务外部是否发生了更改，如果确实发生了更改，则会重启该事务。

你将在清单 8-23 中看到，我们尝试使用多个 `future` 同时修改该值三次（请记住，`future` 会在与父线程不同的独立线程中运行）。

请先通读代码，然后我将开始分解 STM 为解决其面临的冲突所采取的过程，这要归功于一个相当简单的多线程程序。

```
(def r (ref []))
(defn modify [r, f, a, s]
(dosync
(Thread/sleep s)
(alter r f a)))
(future (modify r conj :a 2000))
(future (modify r conj :b 1000))
(future (modify r conj :c 0))
@r
;; [:c :b :a]
清单 8-23.
STM 重启策略如何工作的示例
```

如果你立即尝试解引用 `r`，你会注意到它只包含一个值 `[:c]`；然后一秒钟后，它会包含 `[:c :b]`；最后三秒钟后，它会包含最终的集合 `[:c :b :a]`。

在清单 8-23 中，我们可以看到用户自定义的函数 `modify`，它首先启动一个事务（使用 `dosync`），然后休眠指定的秒数，再使用 `alter` 修改提供的引用变量内的值。

接着，我们启动三个线程（使用 `future`），并在每个线程中执行用户自定义函数 `modify`，同时向其传递相关的引用值、函数和休眠参数。

我们使用了非常具体的休眠值来演示，在应用程序设计过程中，非确定性代码并不一定会按照其定义的顺序完成。

注意

需要明确的是，我仅使用 `(Thread/sleep)` 来模拟非确定性行为。你的应用程序代码可能会使用某种形式的阻塞 I/O，其完成时间各不相同。因此，多次执行该代码可能会导致不同的结果顺序。

在清单 8-23 中，STM 采取的过程如下（这是一个过度简化的描述，但至少能让你了解其工作原理）：

*   `:a` 被传入（代码暂停两秒）。
*   `:b` 被传入（代码暂停一秒）。
*   `:c` 被传入。
    *   `alter` 函数使 STM 检查自事务开始以来 `ref` 的值是否已更改。
    *   此时值仍然是 `[]`（因为其他调用处于暂停状态）。
    *   更新后的值被应用。
*   `:b` 解除暂停并检查 `ref` 的值（因为它即将应用 `conj` 函数）。
    *   值不再是 `[]`，而是 `[:c]`，因此 STM 重启事务。
    *   此时 `:a` 仍处于暂停状态，因此事务完成。
*   `:a` 解除暂停并检查 `ref` 的值。
    *   值不再是 `[]`，而是 `[:c :b]`，因此 STM 重启事务。
    *   事务完成，因为没有其他事务导致重启。
    *   这意味着 `ref` 的值（最终）是 `[:c :b :a]`。

在现实场景中，事务完成所需的时间不会像刚才的例子那样清晰，但这至少让你了解了处理 STM 时所涉及的过程。

### 嵌套事务

### ensure

为了避免出现死锁或活锁情况（即竞争事务依赖于多个引用），我们可以利用 `ensure` 函数来保护引用免受其他事务的修改。

在清单 8-25 中，你可以看到我们有两个引用：`current-account` 和 `savings-account`，它们相加后的总值为 1100。我们唯一的条件是这两个账户的总和不能低于 1000。

在给定的示例中，我们尝试同时从两个账户中减去 100。这些请求中只有一个会成功，因为我们要求最低值为 1000 的约束意味着两个请求不能同时成功，因为那样我们会低于约束阈值。这个困境很好地说明了简单的并发需求如何导致争用。

```
(def current-account (ref 500))
(def savings-account (ref 600))
(defn withdraw [from available amount]
(dosync
(let [total (+ @from (ensure available))]
(Thread/sleep 1000) ; 允许更明显的上下文切换
(if (>= (- total amount) 1000)
(alter from - amount)
(println "抱歉，无法从 "
(:name (meta (var current-account)))
" 中提取 100，因为违反了约束")))))
(println "之前：活期账户余额为" @current-account)
(println "之前：储蓄账户余额为" @savings-account)
(println
"之前：总余额为"
(+ @current-account @savings-account))
(future (withdraw current-account savings-account 100))
(future (withdraw savings-account current-account 100))
(Thread/sleep 2000)
;; 休眠足够长的时间以允许两个事务完成
(println "之后：活期账户余额为" @current-account)
(println "之后：储蓄账户余额为" @savings-account)
(println
"之后：总余额为"
(+ @current-account @savings-account))
清单 8-25.
使用 ensure 协调多个条件的示例
```

### commute

`commute` 函数在修改指定引用时提供了一种“最后写入者获胜”的行为，当你希望获得最高并发性且操作顺序无关紧要时（事务旨在防止对 `ref` 的多次同时写入），该函数最为有用。

本质上，`commute` 会立即返回对引用应用所提供函数的结果，但在事务结束时，它会再次同步执行计算，并在此时实际更新引用的值。这种修改本身是可交换的，因此不会有任何事务被回滚。

这就是为什么在清单 8-27 中，执行该示例代码时你可能会多次看到相同的值被打印出来，但最终的实际结果始终是一致的。

```
(def foo (ref 0))
(defn inc-ref [r]
(dosync
(commute r inc)
(println @r)))
(let [threads (for [x (range 0 20)] (Thread. #(inc-ref foo)))]
(doall (map #(.start %) threads))
(doall (map #(.join %) threads))
(inc-ref foo))
清单 8-27.
使用 commute 的示例
```

注意

我们使用 `Thread.` 而不是 `future`，以便能够更精细地控制线程的执行时机，同时也表明我们希望等待所有线程完成后再继续执行。

如果你运行清单 8-27 中的代码，但将其修改为使用 `alter` 而不是 `commute`，那么你会注意到 `alter` 如何强制更新顺序，因此打印的值总是精确地从 1 计数到 21（而 `commute` 的最终结果仍然是 21，但达到该结果的过程是可交换的）。

此时，你将拥有一组可用的宏（例如 `<!!`、`>!!`、`chan`、`buffer` 等）。

`core.async` 的理念借鉴自 Go 语言 ²（谷歌自己广受欢迎的编程语言，其并发原语在开发者社区中备受推崇）。

尽管 Go 语言在现代使通道概念流行起来功不可没，但通道的设计可以追溯到 20 世纪 70 年代，始于 Tony Hoare 撰写的一篇名为《通信顺序进程》（CSP）的论文。

实际上，通道就是一个管道：你将数据从一端塞入，然后从另一端取出。从并发的角度来看，它之所以效果很好，是因为它保护了管道中的数据免受其他线程/进程的干扰。

使用通道最具挑战性的部分可能是，当你放入/取出数据时，它会阻塞你的应用程序。这是因为从根本上说，它们是在连接不同的进程。我们几乎总是将通道与另一个称为 go 块的概念一起使用。但我稍后会回到这个话题；首先，我认为一个简单的示例是必要的。参见清单 8-30。

注意

以下代码将无限期地阻塞正在运行的进程。你可以在 REPL 中按 `<Ctrl-c>` 来解除阻塞，但在设计应用程序时，明智的做法是注意通道如果设置不当可能会阻塞进程。

```
(def c (chan))
(>!! c :foo)
;; 将阻塞，直到有东西
;; 从通道的另一端取出 :foo
清单 8-30.
一个会阻塞的通道的简单示例
```

在清单 8-30 中，我们可以看到我们创建了一个变量 `c`，并为其分配了一个新的 `chan`（通道）。一旦有了这个通道，我们就可以使用 `>!!` 将值放入通道，使用 `<!!` 从通道中取出值。

前面示例的问题在于，一旦你调用 `<!!`，它就会无限期地阻塞。理想情况下，你希望做的是（至少对于一个可运行的示例而言）使用另一个线程将值放入通道，因为至少新生成的线程可以被阻塞，而不是阻塞你的主线程并导致 REPL 锁定！

清单 8-31 演示了一个稍微实用一些的示例，它不会阻塞你的 REPL。

```
(def c (chan))
(future (>!! c :foo))
;; 这个线程将被阻塞
(<!! c)
;; :foo
清单 8-31.
一个不会阻塞的通道的简单示例
```

在清单 8-31 中，我们将一条消息放入通道（在本例中，消息是关键字 `:foo`），并且由于这是在子线程中完成的（得益于使用了 `future`），我们能够继续在 REPL 的父线程中工作。从这里，你可以看到我们现在正在从通道中取出任何可用的值（使用 `<!!`），并且由于值可用，我们不会阻塞。

那么，举例来说，假设我现在想从我的通道 `c` 中再取出一个值。好吧，我已经取出了值 `:foo`，并且我知道通道中已经没有东西可取了，那么会发生什么呢？嗯，如果我再次取出（例如 `(<!! c)`），那将阻塞我的整个线程，直到我程序中的其他某处有代码将值推入通道（从而使我的取出操作解除阻塞）。

为了避免这个问题（同样，为了在 REPL 中进行实验），我们可以再次利用 `future`。参见清单 8-32 中的示例。

```
(future
(prn (str "嘿！一个新值 " (<!! c))))
;; 将阻塞，直到有值被放入通道
(>!! c "bar")
;; true (+ 副作用：打印出 "嘿！一个新值 bar")
清单 8-32.
通过 future 将值放入通道
```



### Go 块

在通道中使用 `future` 的问题在于，`future` 会从线程池中获取线程。因此，一开始可用的线程数量是有限的。

如果你通过通道的推/拉操作阻塞了一个线程，那么可用于多线程处理工作负载的线程就会减少一个。

注意

`Future` 与代理共享一个线程池。

Clojure 的 `core.async` 库还提供了一个 `go` 函数，它就像一个异步包装器：你提供给 `go` 块的主体将在单独的线程上异步执行，并且该线程来自专门用于 `go` 块的线程池。

好的，那么 `go` 块拥有一个独立的线程池，但它与 `future` 有何不同呢？嗯，使用 `go` 时，你可以“停放”一个线程，而不是阻塞它。

“停放”意味着进程会从线程中移除，从而允许该线程被另一个 `go` 块进程使用。当另一个进程完成后，它会被放回线程以完成执行（参见清单 8-33 的示例）。

```
(go
(Thread/sleep 1000)
(dotimes [x 5] (prn x)))
(prn "我没有被阻塞")
;; 这行会立即打印（随后打印数字 0-4）
清单 8-33.
go 块的简单示例
```

在 `go` 块内部使用通道时，需要注意，推入和拉取值所用的语法与在 `go` 块外部使用通道时略有不同（这是因为我们之前提到的关于“停放”`go` 块，而不是阻塞其运行所在的线程）。

通道推/拉操作的语法是 `>!!` 和 `<!!`，但在 `go` 块内部使用通道时，语法变为 `<!` 和 `>!`，这会导致进程被停放。这意味着，当现有的 `go` 进程被阻塞时，`go` 线程池中的线程可以被重用。参见清单 8-34。

```
(def c (chan))
(go
(dotimes [x 10]
(Thread/sleep 1000)
(>! c x))) ;; 每秒向通道放入一个新数字
(go
(while true
(prn (<! c)))) ;; 永远从通道拉取内容并打印
清单 8-34.
将 go 块与通道一起使用
```

在清单 8-34 中，我们最终每秒打印一个数字，打印出 0 到 9。`future` 和 `go` 块的另一个区别是，你可以访问 `future` 的返回值，而对于 `go` 块，你总是会返回一个通道类型，该通道会传递最后执行的表达式（参见清单 8-35）。

```
(def f (future (inc 1)))
@f
;; 2
(def g (go (inc 1)))
;; ManyToManyChannel
(<!! g)
;; 2
清单 8-35.
Go 块返回通道类型
```

### 线程函数

现在，你可能想知道 Clojure 还能提供多少种异步机制？嗯，`core.async` 还为我们提供了 `thread` 函数。实际上，它允许我们在另一个线程中执行代码（这没什么新意，很像 `go` 和 `future`），并且它也会返回一个通道，其中包含了最后放入通道的表达式（再次像 `go` 一样，所以也没什么额外的）。

唯一真正的区别在于，该线程是从一个“无界”线程池中获取的（这意味着它在理论上可以创建无限数量的线程）。

`(thread)` 的使用场景是，当你有一个长时间运行的进程，该进程可能会长时间阻塞，如果你改用 `go` 块，则会导致你的线程池被阻塞。但请记住，创建线程是一项昂贵的操作，而这正是线程池的优势所在。

注意

当我刚开始接触 Clojure 时，它提供的并发机制数量让我不知所措。直到我真正理解了问题领域，才明白所有这些机制确实都有其用途和存在的理由。

### 区别

通常不太明确的一点是，你究竟应该在何时使用通道。经验法则似乎是这样的：

*   如果你需要将一个同步过程变为异步，则创建一个线程（使用 `future`）。
*   如果你正在使用的 API 或代码已经是异步的，那么使用 `go` 块。

我还没有找到完全不符合这个规则的例子（至少在我使用 Clojure 的工作中是这样），但你的情况可能会有所不同。

在我们继续之前，`core.async` 还提供了一些其他有趣的功能：

*   `alt!`/`alt!!`
*   缓冲通道
    *   滑动/丢弃通道缓冲区
*   超时

### 交替

让我们从 `alt` 函数开始（我认为它代表 alternate？正如你将看到的，它会在提供的通道之间进行交替）。

`alt!`（停放）变体必须在 `go` 块内部调用，而 `alt!!` 变体可以在外部调用（类似于 `>!` 与 `>!!`）。你向 `alt!` 函数传递一组通道，哪个通道先获得值，就返回哪个通道（参见清单 8-36）。

```
(def a (chan))
(def b (chan))
(def c (chan))
(defn put-data [c n]
(go (Thread/sleep (rand 10))
(>! c (str "Hi " n))))
(put-data a "A")
(put-data b "B")
(put-data c "C")
(let [[result channel] (alts!! [a b c])]
(prn "结果: " result)
(prn "通道: " channel))
清单 8-36.
使用 alt!! 函数的示例
```

在清单 8-36 中，我们创建了三个独立的通道和一个名为 `put-data` 的函数，该函数在将值放入提供的通道之前会休眠一段不确定的时间。

然后我们有一个 `let` 块，它获取结果（以及产生该结果的通道）并打印出来。我们使用 `alt!!` 来确定哪个通道最先接收到值（其他结果将被丢弃）。

### 缓冲通道

接下来我们要介绍的是缓冲通道（及其变体：滑动和丢弃）。默认情况下，通道是无缓冲的，所以到目前为止，当我们创建一个通道时，如果你推入一个值，通道就会变满，你的线程将被阻塞，直到该值被从通道中拉出。

为了允许更大的并发性，你可以使用预定义的缓冲区空间来创建通道。假设缓冲区大小为 5，你可以向通道推入 5 个值而不会阻塞。参见清单 8-37。

```
(def c (chan 5))
(def v [:a :b :c :d])
(dotimes [i 4]
(>!! c (nth v i))
(prn "将 " (nth v i) " 放入通道。下一步..."))
清单 8-37.
缓冲通道示例
```

在清单 8-37 中，如果在创建通道时我们没有为其指定 `5` 的缓冲区大小，那么第一次调用 `>!!` 就会无限期地阻塞我们 REPL 的线程，直到我们要么终止正在运行的进程，要么在代码中已经定义了一个可以从通道中提取值的位置。



### 滑动/丢弃缓冲通道

我们可以借助 Clojure 将缓冲通道的概念进一步延伸，根据应用需求使用**滑动缓冲区**或**丢弃缓冲区**。

滑动缓冲区是指当另一个值被推入通道时，会丢弃第一个缓冲值（从结构上看，这类似于队列，但队列不会自动丢弃元素；这是 Clojure 的抽象为我们提供的功能）。丢弃缓冲区则是指当另一个值被推入通道时，会丢弃最后一个缓冲值。

在这两种情况下，由于通道处理缓冲区溢出的方式，都不会导致放入值的操作阻塞当前线程。请参见代码清单 8-38 中的示例。

```
(def s (chan (sliding-buffer 5)))
(def d (chan (dropping-buffer 5)))
(def v [:a :b :c :d :e :f :g :h :i :j])
(dotimes [i 10]
(>!! s (nth v i))
(prn "Put " (nth v i) " into the 'sliding buffer' channel"))
(dotimes [i 10]
(>!! d (nth v i))
(prn "Put " (nth v i) " into the 'dropping buffer' channel"))
(<!! s) ;; :f
(<!! d) ;; :a
代码清单 8-38.
滑动缓冲区和丢弃缓冲区示例
```

在代码清单 8-38 中，我们演示了滑动缓冲区和丢弃缓冲区。我们创建了不同的缓冲区类型，并在创建通道时将它们作为参数传入。然后，我们创建了一个循环，将十个值放入通道，而通道最多只能容纳五个值。因此，我们将看到每种缓冲区类型如何处理这个问题。

对于滑动缓冲区，如果我们从通道中取出一个值，首先会得到 `:f`。这表明前五个值（`:a`、`:b`、`:c`、`:d`、`:e`）在下一组值被推入时被挤出了。顾名思义，通道内容会先滑动出旧值。

对于丢弃缓冲区，如果我们从通道中取出一个值，首先会得到 `:a`。这表明最后五个值（`:f`、`:g`、`:h`、`:i`、`:j`）在缓冲区满的那一刻被丢弃了。这就像一个非常繁忙的夜总会入口：一旦俱乐部满员，门卫就会把你拒之门外。

注意

在社区中，大多数开发者似乎会创建无界通道，即一对一进出，并且仅在需要尽可能提高性能和吞吐量时，才使用滑动/丢弃缓冲区。

### 超时通道

这是我想与你分享的并发领域的最后一个特性，也是使用通道时的一种常见模式：使用超时通道来短路潜在的僵尸（或长时间阻塞）进程。请参见代码清单 8-39。

```
(def c (chan))
(let [[result channel]
(alts!! [c (timeout 5000)])]
(if result
(prn "Result: " result)
(prn "Timed out: " result))) ;; "Timed out: " nil
代码清单 8-39.
超时通道示例
```

在代码清单 8-39 中，我们创建了一个通道，然后从未向它发送任何内容。接着，我们使用 `alts!!` 尝试从通道中取出一个值，但不会无限期阻塞——因为我们可以传递一个通道集合，哪个通道先获得值，`alts!!` 就返回哪个——我们最初被阻塞了五秒钟，然后进程解锁，我们得到了 `nil` 的结果。

这是可行的，因为超时通道在设定的时间段内是一个活动通道，然后它会向自身发送一个 `nil` 值，这使得 `alt!!` 能够立即行动，并随后短路自身。
