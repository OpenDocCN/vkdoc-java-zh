*   什么是文本块
*   如何在代码中表示文本块

## 背景

你编写 Java 程序时很少有不处理字符串的情况。自 1.0 版本以来，Java 就支持将字符串作为 `String` 类的实例。它还支持 `String` 字面量，这些字面量是用双引号括起来的零个或多个字符的序列。`String` 字面量在源代码中表示，并且是 `String` 类的实例。使用 `String` 字面量最大的痛点在于：

*   如果一个字符串字面量无法放入一行，你需要将其拆分成更小的字符串，并使用字符串连接运算符（`+`）将它们重新组合成一个字符串。
*   如果你有一个包含行终止符的字符串，你需要在字符串中将换行符嵌入为转义序列（`\n`），并使用字符串连接运算符（`+`）。
*   如果你的字符串包含双引号，你需要在字符串中使用转义序列 `\"` 来表示它们。

使用 `String` 字面量的这些问题使得用字符串编写 Java 代码变得繁琐，并且存在巨大的可读性问题。假设你想表示一个包含三个城市的列表——每个城市占一行。以下是使用 `String` 字面量实现的方法：

```
String cities = "Panama\nParis\nPatna\n";
```

你可能同意这不太可读。你失去了它包含三行——每行包含一个城市名称——的视觉感。以下是实现相同目标的另一种尝试：

```
String cities = "Panama\n" +
"Paris\n" +
"Patna\n";
```

这一次，可读性稍好一些，但仍然包含转义序列（`\n`）形式的视觉杂乱。你能猜出以下 `String` 字面量表示的字符串吗？

```
String quote = "\"A fool thinks of himself to be wise. " +
"But a wise man knows himself to be a fool.\"\n" +
"–William Shakespeare\n";
```

这个 `String` 字面量表示以下威廉·莎士比亚的名言：

> *愚者自以为智，智者自知其愚。*

> ——威廉·莎士比亚

如果你能够像下面这样表示这句名言，岂不是更好？

```
String quote = """
"A fool thinks of himself to be wise. But a wise man knows himself to be a fool."
–William Shakespeare
""";
```

如果你能够像下面这样表示三个城市的列表——每个城市占一行——岂不是更好？

```
String cities = """
Panama
Paris
Patna
""";
```

在 Java 程序中像下面这样编写 HTML 代码，岂不是更简洁？

```
String html = """

Hello, HTML

""";
```

使用 `String` 字面量来表示复杂的多行文本并不仅限于像三个城市的列表和一句名言这样的字符串。通常，Java 程序包含非 Java 代码，例如 SQL、JSON、XML 和 HTML，这些代码以 `String` 字面量的形式存在。在 Java 中表示这些类型的代码一直很痛苦。一些开发者甚至将这些类型的代码放入文件中，打包到 JAR 中，并在运行时将文件读入 `String` 实例。这种方法让你摆脱了 `String` 字面量限制的束缚，但代价是增加了处理文件的额外麻烦！

解决 Java 开发者在用 `String` 字面量表示复杂的多行文本块时遇到的问题，一直是 Java 语言设计者相当长一段时间内的关注焦点。他们在 Java 12 中作为语言预览特性添加了对原始字符串字面量的支持，但后来撤回了，因此它从未进入 Java 12。你可以在 JEP 326 的 [`https://openjdk.java.net/jeps/326`](https://openjdk.java.net/jeps/326) 找到关于原始字符串字面量提案的详细信息。JEP 326 已被 JEP 355（[`https://openjdk.java.net/jeps/355`](https://openjdk.java.net/jeps/355)）取代，后者提议在 Java 13 中将文本块支持作为语言预览特性。在前面的示例中，当你使用 `"""` 来开始和结束字符串（例如将 HTML 代码表示为 `String`）时，你已经看到了文本块。文本块是本章讨论的主题。



## 什么是文本块？

文本块是由起始定界符和结束定界符括起来的零个或多个字符序列。文本块的内容可以跨越多行，并且你无需使用大多数转义序列。以下是使用文本块的语法：

该语法允许将结束定界符放在单独的一行，或与文本块内容结束的同一行：

起始定界符以三个双引号字符（`"""`）开头，后面可选地跟零个或多个空格、制表符或换页符，并以行终止符结束。

结束定界符由三个双引号字符（`"""`）组成。

文本块的内容是从起始定界符的行终止符之后立即开始，到结束定界符的第一个双引号之前立即结束的字符序列。

文本块是 `String` 类的一个实例。在任何你使用过 `String` 字面量的地方，都可以使用文本块。以下是几个文本块的示例。每个示例都会打印文本块，以向你展示生成的字符串：

```
String cities = """
Panama
Paris
Patna""";
System.out.println(cities);
Panama
Paris
Patna
```

此文本块与 `String` 字面量 `"Panama\nParis\nPatna"` 相同。

如果你将结束定界符放在单独的一行，则会在文本块内容的末尾附加一个换行符（\n）。以下文本块与 `String` 字面量 `"Panama\nParis\nPatna\n"` 相同：

```
String cities = """
Panama
Paris
Patna
""";
System.out.println(cities);
Panama
Paris
Patna
```

注意输出中的最后一行是一个空行，这是因为你将结束定界符放在了单独的一行。

考虑以下代码片段：

```
String cities = """
Panama
Paris
Patna
""";
System.out.println(cities);
Panama
Paris
Patna
```

你对输出感到惊讶吗？输出中的最后一行是空的，因为你将结束定界符放在了单独的一行。内容中的三行都向左缩进了 16 个空格。这是因为你将结束定界符移到了行首。比较最后两个示例中结束定界符的位置。结束定界符的位置会影响文本块的格式。你需要掌握其背后的规则。我将在后续章节中解释文本块的所有格式规则。

### 提示

文本块不是原始字符串——这意味着你输入的内容并不直接就是最终结果。文本块的内容会由编译器使用一组预定义的规则进行转换。

让我们看几个在文本块中使用起始定界符的无效示例。回想一下，在起始定界符的三个双引号字符（`"""`）之后，可以包含空格、制表符和换行符。你必须以行终止符结束它。`"""` 之后允许的字符是不可见字符。你不能为这些不可见字符使用转义序列。你只需在源代码中将其作为不可见字符输入即可。以下是无效的文本块，它们会产生编译时错误：

```
// 起始定界符中缺少行终止符。
String str = """Hello
""";
// 起始定界符中缺少行终止符。你可以在起始定界符中输入制表符，
// 但不能使用转义序列 \t。
String str = """\t
Hello
""";
// 起始定界符中缺少行终止符。
String str = """Hello""";
```

### 提示

文本块中的起始定界符必须使用行终止符放在单独的一行，并且文本块的内容必须从下一行开始。

## 文本块是一项预览特性

文本块是 Java 13 中的一项语言预览特性。有关预览特性的更多详细信息，请参阅第 5 章。第 5 章还解释了如何编译和运行使用 Java 预览特性的 Java 程序。

## 文本块会被转换

让我们从两个问题开始。文本块是原始字符串吗？文本块是多行 `String` 字面量吗？这两个问题的答案既是“是”也是“否”。在某种程度上，文本块既是原始字符串又是多行 `String` 字面量；在某种程度上，它两者都不是。为了澄清答案，让我们定义一下*原始*字符串和*多行* `String` *字面量*的含义。

原始字符串意味着字符串不经过任何处理。术语*原始*意味着“不以任何方式改变”。这不是你从文本块中得到的结果。文本块在最终生成字符串之前会经过转换。因此，文本块肯定不是 100% 的*原始*字符串。

那么文本块是多行 `String` 字面量呢？同样，术语*字面量*意味着它被原样表示。文本块并非如此，因为它会被转换。

从另一方面来看，文本块确实保留了原始字符串的大部分（如果不是全部）内容，并允许你将字符串跨越多行。从这个意义上说，它部分是原始的和多行的。

文本块有时被称为多行 `String` 字面量。即使是定义了文本块特性的 JEP 355，也将其称为多行 `String` 字面量。如果你想将其称为多行 `String` 字面量，我建议你称其为*带转换的多行* `String` *字面量*，这能让读者真正理解它是什么。

编译器使用一组预定义的规则来转换文本块的内容。转换后得到的字符串在类文件中被表示为 `String` 字面量。运行时对文本块一无所知。现在你可能明白为什么我称其为*带转换的多行 String 字面量*了。



### 提示

在文本块中，你可以选择退出某些转换，而默认情况下，其他转换是自动启用的。文本块的自动转换旨在满足大多数常见场景的需求。

```
......................
```

在这个文本块中，编译器假设开发者希望每行内容左对齐，第一行以两个空格开头，并使用四个空格来嵌套 HTML 标签。这是编译器在转换过程中所做的假设。

在 Java 程序中，通常会将文本块的内容与程序的其他部分（如变量名、赋值运算符和开始分隔符中的第一个双引号）垂直对齐。这些空白字符会作为每行内容的前导空白出现。一行中也可能出现非预期的尾随空白。前导空白可能用于缩进目的，例如在 HTML 代码中缩进嵌套标签。

开发者希望转换后保留在内容中的文本块空白字符称为**必要空白**。内容中作为源代码缩进一部分存在，但开发者不希望保留在最终字符串中的空白字符称为**附带空白**。

编译器会从内容中的所有非空行中移除附带空白。结束分隔符的起始位置可用于控制文本块中的附带空白。尾随空白始终被视为附带空白，编译器会将其全部移除。在接下来的几节中，我将解释编译器用于转换文本块的规则。

## 转换规则

编译器分三个阶段转换文本块。每个阶段处理不同类型的字符，并服务于不同的目的。按顺序对以下类型的字符进行转换：

*   行终止符
*   附带空白
*   转义序列

让我们深入探讨每种转换类型。这些规则起初可能有些令人不知所措。如果你不理解某些规则，不必担心。在介绍每条规则后，我会提供多个示例，详细解释规则中的每一步，并展示转换前后的字符串。



### 转换行终止符

不同平台的行终止符各不相同。Windows 使用回车符（CR）和回车换行符（CRLF）作为行终止符。类 UNIX 操作系统使用换行符（LF）作为行终止符。CR 由 ASCII 十进制数 13、Unicode 转义序列 `\u000D`、八进制转义序列 `\015` 以及转义序列 `\r` 表示。LF 由 ASCII 十进制数 10、Unicode 转义序列 `\u000A`、八进制转义序列 `\012` 以及转义序列 `\n` 表示。LF 也被称为换行字符。

考虑以下文本块：

```
"""
Hello
Terminator
"""
```

这个文本块中有多少个字符——15、17、19 还是其他？仅通过查看这个文本块很难确定确切的字符数。答案取决于编写此文本块所使用的平台。你可以看到 15 个可见字符——`Hello` 占 5 个，`Terminator` 占 10 个。内容中的两行各有一个行终止符，虽然不可见，但你能看到其效果。回想一下，行终止符可能是一个或两个字符，具体取决于平台。如果你在 Windows 上编辑此文本块，不可见的行终止符如下所示：

```
"""
HelloCRLF
TerminatorCRLF
"""
```

如果你在类 UNIX 操作系统上编辑此文本块，不可见的行终止符如下所示：

```
"""
HelloLF
TerminatorLF
"""
```

将 CR 和 LF 各计为一个字符，该文本块在 Windows 上包含 19 个字符，在类 UNIX 操作系统上包含 17 个字符。两个开发者在不同平台上编写相同的文本块，会得到不同的字符数。你可能会同意，这种差异既非本意，也不可接受。

文本块内容中的行终止符由编译器进行规范化处理。编译器将 CR 和 CRLF 替换为 LF。此转换确保具有相同视觉内容的文本块在不同平台上包含相同的最终字符串。这就是此阶段发生的全部操作。在此阶段转换后，无论平台或源代码编码如何，文本块的内容如下所示：

```
"""
HelloLF
TerminatorLF
"""
```

文本块可能包含转义序列，例如作为转义序列 `\r` 和 `\n` 的 CR 和 LF，这些在此阶段保持不变。所有转义序列将在最后阶段进行转换。

### 转换无关空白

在此阶段之前，所有行终止符都已规范化为 LF。在此阶段，我将行终止符简称为行终止符，它始终指代 LF。本节中的所有示例文本块均使用点号来显示其内容中的空格。

在此阶段，编译器会移除无关的前导和尾随空白。要从内容中所有非空行移除的前导空白数量计算如下：

*   计算时不考虑带有开始定界符的行。

*   仅由空白组成的行称为空行，计算时不考虑这些行。空行可能仅包含一个行终止符，或包含带有行终止符的空白。

*   如果结束定界符单独放在一行（无论是否带有空白），则计算时会考虑该行。请注意，根据空行的定义，当你将结束定界符单独放在一行时，该行是一个空行。然而，就本次计算而言，它是一行重要的行，不会被忽略。

*   统计每个非空行上前导空白的数量。每个空白计为 1。也就是说，如果一行包含两个空格，另一行包含一个空格和一个制表符，则此步骤将这两行的前导空白都计为 2。

*   计算所有非空行的前导空白计数的最小值。假设你有五个非空行，其前导空白计数分别为 15、19、23、19、15 和 13。所有计数的最小值为 13。

*   从所有非空行中移除上一步计算出的最小空白数量。

*   从所有行（不仅仅是非空行）中移除所有尾随空白。此步骤将移除非空行中的尾随空白。它还会将空行压缩为仅包含行终止符的空行。

#### 示例 1

考虑以下文本块：

```
String html = """
..............
......................
......................Hello, HTML.
.....................
................
..............""";
```

### 转义序列的转换

文本块中的转义序列在转换的第三阶段（即最后一个阶段）被解释。文本块支持与 `String` 字面量相同的转义序列集，例如 `\b`、`\t`、`\n`、`\f`、`\r`、`\"`、`\'`、`\\`、八进制转义（`\nnn`）和 Unicode 转义（`\uxxxx`）。它们被最后解释，以便开发者有机会使用转义序列来格式化文本块的内容。考虑以下文本块，它使用转义序列 `\t`（制表符）来格式化 HTML 代码中的嵌套标签：

```
String html = """
..............
..............\t
..............\t\tHello, HTML
..............\t
..............
..............""";
```

所有显示为点号的前导空格将在第二阶段转换中被移除。此阶段将解释嵌套行中的制表符，生成的字符串如下所示：

```

Hello, HTML

```

回顾一下，在第二阶段中，所有行终止符（CR、CRLF 和 LF）都被规范化为 LF。假设你想让所有行都以 CRLF（`\r\n`）结尾，而不是 LF（`\n`）。如何实现这一点？你可以通过在文本块中每行内容的末尾插入转义序列 `\r` 来实现。以下是该文本块：

```
String html = """
..............\r
..................\r
......................Hello, HTML\r
..................\r
..............\r
..............""";
```

生成的字符串如下所示。为了帮助你直观地理解转换后的字符串，我在字符串的每一行中显示了转换后的行终止符（LF）为 `\n`。空格照常显示为点号。

```
\r\n
....\r\n
........Hello, HTML\r\n
....\r\n
\r\n
```

如何向文本块中的任何行添加尾部空格？答案是使用转序列。空格的 ASCII 十进制值为 32，可以用八进制转义表示为 `\040`，用 Unicode 转义表示为 `\u0020`。让我们举一个简单的例子，将 `"Hello "` 表示为文本块。注意，`"Hello "` 中有一个尾部空格。该字符串的字符数是 6——`Hello` 中的 5 个字符和一个尾部空格。以下是我们的第一次尝试：

```
String str = """
Hello """;
System.out.printf("str: '%s'%n", str);
System.out.printf("str.length(): %d%n", str.length());
System.out.printf("str ends with a space: %b%n", str.endsWith(" "));
str: 'Hello'
str.length(): 5
str ends with a space: false
```

输出表明，从文本块生成的字符串不包含尾部空格。这是因为转换的第二阶段会移除所有尾部空白，从而删除了 `Hello` 后面的空格。

在这种情况下，你需要将尾部空格表示为八进制转义 `\040`。以下是第二次尝试：

```
String str = """
Hello\040""";
System.out.printf("str: '%s'%n", str);
System.out.printf("str.length(): %d%n", str.length());
System.out.printf("str ends with a space: %b%n", str.endsWith(" "));
str: 'Hello '
str.length(): 6
str ends with a space: true
```

恭喜！输出表明你成功地在文本块的一行中插入了尾部空格。但事情可能并不像你想象的那样就此结束！让我们尝试使用 Unicode 转义 `\u0020` 来添加尾部空格：

```
String str = """
Hello\u0020""";
System.out.printf("str: '%s'%n", str);
System.out.printf("str.length(): %d%n", str.length());
System.out.printf("str ends with a space: %b%n", str.endsWith(" "));
str: 'Hello'
str.length(): 5
str ends with a space: false
```

没有达到预期效果？感到惊讶吗？Unicode 转义在编译的非常早期阶段就被解释，早于任何其他处理。编译器在读取源代码时就会解释 Unicode 转义。编译器解释 Unicode 转义 `\u0020` 并将其转换为一个空格字符。当编译器开始文本块转换时，`\u0020` 已经被转换为空格字符 `' '`。文本块转换器根本看不到 Unicode 转义——它看到的是一个空格。转换器在处理所有尾部空白时会移除这个空格。这就是为什么在这个例子中你没有得到尾部空格的原因。为了证明这一点，让我们再看一个例子。

LF 的 Unicode 转义是 `\u000A`。考虑以下包含 `String` 字面量的代码片段，其中包含以下字符序列：`He`、`\u000A` 和 `llo`。

```
String str = "He\u000Allo";
```

这段代码能编译吗？不能。罪魁祸首就是 Unicode 转义的使用。在编译器处理 `String` 字面量之前，Unicode 转义 `\u000A` 已经被解释为 LF，使得源代码看起来像这样：

```
String str = "He
llo";
```

`String` 字面量不能跨越多行。这就是这段代码无法编译的原因。编译时错误是“未闭合的字符串字面量”。让我们用文本块来尝试这个例子，如下所示：

```
String str = """
He\u000Allo""";
System.out.printf("str: '%s'%n", str);
System.out.printf("str.length(): %d%n", str.length());
str: '             He
llo'
str.length(): 19
```

输出中有两个令人惊讶的地方。首先，你没有遇到任何编译时错误。其次，字符数是 19。你可能预期字符数是 6，而不是 19。暂停一下，尝试解释这个输出。

如果你无法解释这个输出，以下是解释。让我们用点号表示空格，来展示原始文本块：

```
String str = """
.............He\u000Allo""";
```

在文本块转换发生之前，编译器会将 `\u000A` 解释为 LF，这将导致以下文本块：

```
String str = """
.............He
llo""";
```

现在你可能能够看到最终的字符串，并解释之前的输出了。文本块的内容从一行变成了两行。第一行包含 13 个前导空格，第二行没有。编译器不会移除任何前导空格，因为所有非空白行的最小空白计数为零。最终的字符串如下：

```
.............He
llo
```

总字符数为 19——13 个前导空格，`He` 中的 2 个字符，`He` 后面的 LF 占 1 个字符，以及 `llo` 中的 3 个字符。

### 提示

你不需要使用转义序列 `\"` 和 `\n` 来在文本块中包含双引号和换行符。建议你改用 " 和行终止符。并且，要小心在文本块中使用 Unicode 转义，因为它们会在文本块转换之前被处理。

假设你想在一个文本块中嵌套另一个文本块。考虑以下第一次不成功的尝试：

```
String str = """
String test = """
Text blocks are fun if you understand them!
"""
""";
```

这个文本块无法编译。第二行中的三个双引号 `"... String test = """"` 被解释为第一行开始定界符的结束定界符。现在第三、第四和第五行不再是有效的 Java 代码。如果你想在文本块内部使用三个连续的双引号，你必须至少转义这三个双引号中的一个，以避免它们被解释为结束定界符。你可以像下面这样使之前的文本块有效。我选择转义第一个双引号。

```
String str = """
String test = \"""
Text blocks are fun if you understand them!
\"""
""";
```

## 连接文本块

文本块可以用于任何你使用 `String` 字面量的地方。你可以像连接 `String` 字面量一样连接它们。请记住保持代码的可读性，因为连接后的文本块会使代码难以阅读。

以下代码片段将一个文本块和一个 `String` 字面量连接，得到字符串 `"Hello, Text Block"`：

```
String str = """
Hello""" + ", Text Block";
```

这段代码的可读性不是很好。让我们将 `String` 字面量替换为文本块：

```
String str = """
Hello""" + """
, Text Block""";
```

可读性进一步降低了。你必须注意每个文本块的开始和结束定界符，才能理解发生了什么。

假设消息中的名称来自一个变量。要得到一个 `"Hello, <name>"` 字符串，其中 `<name>` 来自变量，你可以使用以下文本块。注意使用八进制转义序列 `\040` 来插入一个空格：

```
String name = "Text Block";
String str = """
Hello,\040""" + name;
```

这不是一个最优的解决方案。如果你的文本块中有占位符，其值来自变量，那么使用 `String` 类的 `formatted()` 方法配合字符串格式说明符是更好的选择。你可以将前面的代码片段重写如下：

```
String name = "Text Block";
String str = """
Hello, %s""".formatted(name);
```

`formatted()` 方法作为支持文本块的一部分，已在 Java 13 中添加到 `String` 类。它是一个实例方法。它接受一个可变参数，这些参数是文本块中格式说明符的值。如果你想基于模板生成文本，你可以使用带有格式说明符的文本块作为模板，并使用 `formatted()` 方法将所需文本填入模板。



## String API 的补充

以下实例方法已添加到 `String` 类中以支持文本块：

*   `String formatted`​`(Object... args)`

*   `String stripIndent()`

*   `String translateEscapes()`

**提示** 添加到 `String` 类中以支持文本块的这三个方法已被弃用并标记为待移除，因为文本块是一项预览特性，可能会在未来的版本中被移除或改进。你不应在生产环境中使用文本块及这些方法。

`formatted()` 实例方法使用指定的参数格式化字符串。其工作方式与 `String` 类的 `format(String format, Object...args)` 静态方法相同。使用 `formatted()` 方法而非 `format()` 方法可以使代码更简洁。除此之外，结果上没有区别。以下代码片段使用两种方法格式化同一个文本块：

```
String name = "Text Block";
// 使用 formatted() 实例方法
String str = """
Hello, %s""".formatted(name);
System.out.println(str);
// 使用 format() 静态方法
String str2 = String.format("""
Hello, %s""", name);
System.out.println(str2);
Hello, Text Block
Hello, Text Block
```

`stripIndent()` 方法会移除字符串中每一行开头和结尾的附带空白。当你从文件或其他来源接收可能包含附带空白的字符串时，可以使用此方法。在处理文本块时，你不需要使用此方法，因为文本块会自动移除附带空白。Java 12 添加了一个 `indent(int n)` 方法，该方法会在每行开头添加 `n` 个空格，并将每行的行终止符规范化为 LF。如果 n 小于零，则会从每行移除最多 n 个前导空白。清单 6-4 包含一个完整的程序，展示了这两个方法的使用。输出之后是对输出的讨论。

## 检测不一致的空白字符

在文本块中对行进行缩进时，可能会无意中混用空格和制表符等空白字符。当文本块移除无关空白字符时，这可能导致不理想的缩进效果。有时，你可能希望保留文本块中某行末尾的空白字符，但文本块会自动将其移除。考虑以下文本块，开发者希望用四个空格缩进行星名称：

```
String planets = """
Mercury
Venus
Earth
""";
```

此文本块表示的字符串应如下所示：

```
Mercury
Venus
Earth
```

当打印此文本块时，输出如下：

```
Mercury
Venus
Earth
```

问题在于三行中前导空白字符的使用不一致。我混用了空格和制表符来对齐行星名称，使它们看起来比上一行的行星名称向右缩进了四个空格。制表符在不同编辑器中可能显示不同。有时，一个制表符等于四个空格，有时则等于八个空格。

当文本块移除无关空白字符时，它会将每个空白字符计为一个，无论该空白字符使用的是哪种字符。例如，两个制表符和八个空格分别计为两个空白字符和八个空白字符，即使它们可能使两行内容在垂直方向上对齐到相同位置。

你可能需要花些时间来找出混用空白字符进行缩进的文本块中的缩进问题。JDK 13 提供了一个名为 `text-blocks` 的 `lint`，你可以用它来在这种情况下获取警告。编译源代码时，你需要使用 `-Xlint:text-blocks` 选项。该 lint 会发出两种警告：

*   针对用于缩进行的前导空白字符使用不一致的警告，这可能导致不理想的缩进。

*   针对尾部空白字符的警告，以便你知道尾部空白字符将被移除。如果需要保留尾部空白字符，请改用转义序列。

## 总结

自 Java 1.0 版本起，Java 就支持 `String` 字面量。`String` 字面量有一个很大的局限性：它必须在一行内表示。开发者经常需要在 `String` 字面量中表示多行文本，这迫使他们使用字符串拼接。将多行文本作为 `String` 字面量使用，一直是 Java 开发者约 24 年来的痛点。Java 语言设计者知道这个问题，并最终给予了关注。他们曾在 Java 12 中提出（后来撤回）对原始字符串的支持。在 Java 13 中，他们以语言预览特性的形式加入了对文本块的支持。

文本块是由开始定界符和结束定界符括起来的零个或多个字符的序列。文本块的内容可以跨越多行，并且你不需要使用大多数转义序列。开始定界符以三个双引号字符（`"""`）开头，后面可选地跟零个或多个空格、制表符或换页符，并以行终止符结束。结束定界符由三个双引号字符（`"""`）组成。

文本块是 `String` 类的一个实例。你可以在任何可以使用 `String` 字面量的地方使用文本块。

文本块的内容由编译器转换，最终生成一个 `String` 字面量。运行时永远不会看到（或不知道）文本块。运行时只看到转换文本块后生成的 `String` 字面量。

编译器分三个阶段执行文本块的转换。在第一阶段，它规范化行终止符。它将所有行终止符（CR、CRLF 和 LF）替换为 LF。

在第二阶段，编译器移除所有非空行中共同的前导空白字符。空行是仅包含 LF 或后跟 LF 的空白字符的行。文本块中结束定界符的位置在计算要移除的前导空白字符数量时起着重要作用。如果你不想移除任何前导空白字符，请将结束定界符放在单独一行的开头。此阶段不解释转义序列。此阶段会移除所有行中的尾部空白字符。

在转换的第三阶段也是最后阶段，编译器解释转义序列。

`String` 类新增了三个实例方法来支持文本块：`formatted()`、`stripIndent()` 和 `translateEscapes()`。这些方法已被弃用，因为它们的存在是为了支持作为预览特性的文本块。`formatted()` 方法用于使用格式说明符格式化文本块。`stripIndent()` 方法用于移除文本块中的无关空白字符。`translateEscapes()` 方法用于解释文本块中的转义序列。

在源代码中，可能会无意中混用不同类型的空白字符（如空格和制表符）来缩进文本块中的行，这可能导致不理想的缩进。有时，你可能希望保留行尾的空白字符，但这些字符会被编译器自动移除。使用编译器的 `-Xlint:text-blocks` 选项可以检测此类不一致的空白字符使用。当编译器检测到文本块中存在不一致的前导空白字符或尾部空白字符时，会打印警告。
