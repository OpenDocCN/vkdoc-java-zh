# 28. 安全密码

Michael Müller^(1 )

(1)德国，北莱茵-威斯特法伦州，布吕尔

Alumni 项目中特殊区域的访问权限将仅限于会员。这意味着我们需要将访问权限限制在授权人员范围内。在注册过程中，我们要求用户输入密码，并将其与账户实体一起存储。

由于 Alumni 项目暴露于网络环境中，存在有人可能篡改系统的潜在风险。密码需要特殊保护——它们绝不应以明文格式存储。虽然不存在 100% 抵御*破解者*（恶意黑客——也有善良的黑客）的方法，但通过遵循以下准则，可以以相当安全的方式存储密码：

*   对所有密码进行强加密

*   不授予任何访问解密后密码的权限

*   使用需要一定计算时间的算法

*   即使两个用户选择了相同的密码，也要计算出不同的哈希值

*   使用验证密码的算法，该算法所需时间几乎与密码长度无关



## 哈希

通过哈希函数可以实现无法解密的加密。*哈希*是哈希函数的输出，该函数将任意大小的文本（或其他数据）映射为固定大小的输出字符串。一个具有密码学实用价值的哈希函数，即使输入仅有微小变化，其输出也几乎在所有字符上都会不同。哈希函数是一种单向函数：你可以从文本计算出哈希值，但无法从哈希值反推出文本。不同的输入可能产生相同的哈希值，这种情况称为*碰撞*。一个好的哈希函数会尽可能减少碰撞。纯粹主义者可能会声称哈希并非加密，但它确实计算出了一个加密后的值。通过选择合适的算法，我们可以降低两个不同密码产生相同哈希值的风险。例如，如果你使用安全哈希算法（SHA），比如 SHA-512，发生此类碰撞的风险趋近于零。实现这样一个哈希函数相当简单，如清单 28-1 所示。

###### 清单 28-1 密码的简单加密（哈希化）

```
 1   public static byte[] getSHAHash(String secret) {
 2     try {
 3       MessageDigest md = MessageDigest.getInstance("SHA");
 4       byte[] digest = md.digest(secret.getBytes("utf-8"));
 5       return digest;
 6     } catch (UnsupportedEncodingException | NoSuchAlgorithmException ex) {
 7       Logger.getLogger("HashUtils").log(Level.SEVERE, null, ex);
 8       throw new RuntimeException(ex);
 9     }
10   }
```

策略是将哈希值存储到数据库中。这样，即使未经授权的人访问了数据库，他们也无法读取密码，只能看到哈希值。但使用如上清单所示的简单算法并不可取。我们需考虑一些缺点。如果哈希算法容易产生大量碰撞，那么在不掌握正确密码的情况下，生成与密码哈希表中存储的哈希值相同的哈希值就会变得更容易。例如，SHA-1 被认为并不真正安全，即使只有少数已知的大型文档产生了相同的哈希值。其他算法如 MD5 则更不安全。但这个明显的问题并非唯一的麻烦。

## 安全问题与缓解措施

破解者可能会尝试发起暴力攻击，发送大量不同的密码组合，直到命中正确的那个。简单的哈希计算速度非常快，攻击者可以在短时间内尝试大量密码。我们可以做的一件事是减慢用户认证和授权的处理过程。如果整个往返时间大约为 0.5–1.0 秒，这不会妨碍用户，但会显著拖慢攻击者的速度。

###### 提示

*认证*：这是正确的用户吗？*授权*：该用户拥有哪些权限？

假设破解者窃取了哈希值。如果他知道用于计算哈希值的算法，就可以对哈希值发起暴力攻击。同样，我们需要减慢处理过程，以防止破解者找到有效密码。我们可以在流程的任何环节延迟用户认证，但现在唯一的机会是使用一个耗时较长的哈希算法。纯 SHA 算法对此来说速度太快了。

如果破解者找到了一个有效密码，并且该哈希值被多个用户使用，那么破解者就能访问多个账户。为了防止这种情况，我们需要为不同用户生成不同的哈希值，即使他们选择了相同的密码。你可以通过“加盐”密码来实现这一点。假设你有一个包含随机值的字符串 salt。你可以通过将盐值添加到密码中来计算哈希值，如下所示：

```
byte[] hash = getSHAHash(salt + secret)
```

因为我们需要盐值来计算正确的哈希值，所以必须将盐值存储在某处。它可能存储在与密码哈希值相同的表中。

对于给定的密码，哈希函数总是计算出相同的哈希值。因此，可以预先存储短语及其哈希值。然后，你可以通过用哈希值查询数据库来查找密码。这样的表被称为*彩虹表*，并且可以从多个来源在线下载（搜索“rainbow tables download”会列出大量网址）。加盐密码也能提供针对此类彩虹表的保护。

用户通过浏览器输入密码。但存储哈希值的数据库位于服务器上的其他地方。这意味着密码需要从浏览器传输到服务器。为了保护传输过程中的密码，选择安全的传输层至关重要，例如传输层安全性（TLS），这是一种用于通信安全的加密协议。结合证书，应用服务器或上游 HTTP 服务器可以使用 HTTPS。配置服务器超出了本书的范围。

## 密码算法

综合以上所有因素，Alumni 使用了基于密码的密钥派生函数 2（PBKDF2）算法，如清单 28-2 所示。

###### 清单 28-2 密码的改进加密（哈希化）

```
 1   public static byte[] hashPassword(String password, byte[] salt,
 2                                     int iterations, int keyLength) {
 3     try {
 4       SecretKeyFactory skf = SecretKeyFactory.
 5           getInstance("PBKDF2WithHmacSHA512");
 6       PBEKeySpec spec = new PBEKeySpec(password.toCharArray(), salt,
 7                                        iterations, keyLength);
 8       SecretKey key = skf.generateSecret(spec);
 9       return key.getEncoded();
10     } catch (NoSuchAlgorithmException | InvalidKeySpecException ex) {
11       Logger.getLogger("HashUtils").log(Level.SEVERE, null, ex);
12       throw new RuntimeException(ex);
13     }
14   }
```

通过将迭代次数设置为一个较高的值，计算过程会因额外的计算轮次而变慢。在我的开发计算机上，大约 100,000 次迭代需要 0.5 秒。使用 100,047 或类似的非预期迭代次数不会影响感知时间，但会产生不同的哈希值。这类数字比大的偶数使用得更少，这进一步有助于提高安全性。

延迟取决于你的机器，因此不要直接复制这个精确值。你需要为你的机器找出最佳值。

KeyLength 决定了输出的长度。Alumni 使用的密钥长度为 1024，即 1024 位 / 8 = 128 字节。

这种粗略描述的方法可能无法满足银行应用的高要求，但对于像 Alumni 这样的社交应用来说已经足够。（如果你想了解更多，可以查阅 [`goo.gl/Spvzs`](http://goo.gl/Spvzs) 上那篇有趣的文章“安全密码存储”。）

除了安全存储密码外，在用户认证过程中还必须检查密码。通常，比较两个值会在第一个差异处失败。如果两个字符串在第一个字符或最后一个字符处不同，后一种情况的比较会花费稍长一点的时间。这个事实可能有助于破解者猜测密码。稍后，当我们执行用户认证和授权时，我们将逐字节比较哈希值直到末尾，而不会中途退出这个过程。



## 摘要

Alumni 的大部分功能仅限会员使用。常见的用户认证方式是采用用户名-密码组合。本章讨论了保护密码安全的若干方面。密码绝不能以明文形式存储，而应通过单向加密算法进行加密。哈希值的计算需要耗费一定时间，以抵御暴力破解攻击。本章讨论了 Alumni 中用于此目的的算法，但未深入探讨密码本身的强度或弱点，例如密码长度、特殊字符的使用等。
