# 16. 上下文与依赖注入

(1)德国北莱茵-威斯特法伦州布吕尔

有时，你需要一个类型可能在运行时更改或确定的对象实例，并且需要一个可靠的机制来执行该任务。幸运的是，Java EE 8 提供了上下文与依赖注入（CDI），可以解决这个问题。

## 从 new() 到 CDI

如果你编写一个需要访问底层面向对象服务的类，你需要获取该服务的一个实例并调用其方法。一种简单的方法是创建服务类的一个新对象：

```
DataService dataService = new DataService();
```

这种方法有几个缺点。如果你可以在不同的兼容服务之间进行选择，会发生什么？让我们将 DataService 声明为抽象类或接口，并实现一个具体的服务。你可以使用

```
DataService dataService = new MySQLDataService();
```

或

```
DataService dataService = new PostgreSQLDataService();
```

现在，你的应用程序仍然需要知道其中一个具体服务。如果你需要在运行时灵活选择服务，一种解决方案是使用 if 级联或 switch 语句，根据配置字符串选择具体的类。你的类仍然需要知道具体的实现。如果你想添加另一个实现，你的类也需要知道它。这是一种你通常不希望拥有的依赖关系。

诀窍在于将决策转移到服务的静态工厂方法中。这是 Erich Gamma 及其合著者在他们的书 *设计模式：可复用面向对象软件的基础*（Addison-Wesley，1994）中讨论的模式之一。

现在，我们的类变得独立于具体实现：

```
DataService dataService = DataService.createService("MySQL")
```

但除了那个接口（或抽象类）之外，我们的类仍然需要知道该服务，并负责创建对象，尽管我们将创建过程委托给了服务。在传统编程中，我们的代码负责控制流程。

现在，想象一下我们的类是一个包含所需服务的框架¹的扩展。那么框架可能负责控制流程。因此，我们的类是一个被框架调用的扩展点。并且框架可能会创建服务并将其注入到我们的类中。与传统编程相比，控制流程被反转了。这种方法被称为控制反转（IoC），由 Martin Fowler 在 2005 年描述（你可以在 [`martinfowler.com/bliki/InversionOfControl.html`](http://martinfowler.com/bliki/InversionOfControl.html) 阅读更多内容）。

CDI 是 Java EE 提供的一种 IoC。通过简单地用 @Inject 注解所需对象，注入变得非常容易。我们之前在项目中使用过它。

使用 CDI，容器将创建所有所需的对象并将其注入到我们的类中。但它不仅将属于框架的对象注入到我们的类中。CDI 还将我们应用程序（或其他应用程序）的特定对象注入到我们的类中。

但是 CDI 如何知道要注入哪个对象？它如何判断是要创建一个新对象，还是可以使用现有实例？

你可以用 CDI 能够识别和解释的特殊关键字来注解你的类。我们之前使用过这样的注解：javax.inject 包中的 @Named，它将 bean 声明为 CDI bean。这样的命名 bean 可以被 Java EE 注入到消费类中。除了 CDI bean，CDI 还可以注入企业 Java Bean（EJB）。

###### 注意

在不讨论 EJB 的情况下，我们之前在服务类中使用过它们，通过使用 @Stateless 注解，该注解将类声明为无状态 EJB。使用这种 bean，我们可以受益于容器的自动事务管理。另一方面，为此类服务类使用 CDI bean 过去迫使开发人员手动添加事务处理。一种通用的解决方案是创建一个拦截器，为每次 JPA 访问添加事务。如今，这已通过 @Transactional 注解包含在 CDI 中。

除了这种命名 bean 的静态声明之外，开发人员可以将任何返回对象的方法声明为生产者方法。这相当于 CDI 中的工厂方法。

一旦 CDI 知道要注入什么，它就需要了解对象的生命周期，以确定是使用现有对象还是需要注入新对象。CDI 也为此提供了一些注解。在构建 Books 时，我们仍然使用了一些。还记得那些 xxxScoped 的吗？表 16-1 简要列出了该组常见注解的概述。



###### 表 16-1 预定义的 CDI 作用域

| 注解 | 描述 |
| --- | --- |
| @ApplicationScoped | Bean 绑定到应用程序生命周期：它在应用程序启动时（至少是在首次使用该 Bean 时）被创建，并在应用程序关闭时被销毁。 |
| @SessionScoped | 带有该标记的 Bean 绑定到当前会话的生命周期。 |
| @RequestScoped | 带有 @RequestScoped 注解的 Bean 绑定到单个 HTTP 请求。 |
| @ConversationScoped | 这将生命周期绑定到所谓的*会话*期间。默认情况下，会话处于*瞬态*模式，这意味着它绑定到单个请求。但通过调用其 start 方法，会话会被设置为*长期运行*模式，并一直存在直到调用 end 方法。这允许开发者控制生命周期。 |
| @Dependent | 这是一个伪作用域：Bean 的生命周期依赖于其所有者的生命周期。这意味着它的生命周期依赖于它被注入到的 Bean。如果未定义作用域，这是默认值。 |

你也可以定义自己的作用域。JSF 基于 CDI 作用域定义了另外两个作用域：@ViewScoped 和 @FlowScoped。前者绑定到一个页面，无论对该页面发起多少次后续请求。后者可用于一系列页面（称为*流程*）。

不要将其与旧的 JSF 作用域混淆，后者在 Java EE 8 中已被弃用，本书将不会涉及。使用这些 JSF 作用域时，有一条规则：只有生命周期上下文更宽的 Bean 才能被用在另一个 Bean 中。使用 CDI，可以将生命周期较短的 Bean 注入到生命周期较长的 Bean 中。

乍一看，很难想象这如何工作。生命周期较短的 Bean 可能在我想要在另一个 Bean 中使用它时已经被销毁了。CDI 使用了一个简单的技巧：除了依赖模式外，CDI 不会注入对象的直接引用。相反，会创建一个代理对象，并注入对该代理的引用。因此，当访问生命周期较短的对象时，它可能在底层被一个新对象替换。

还记得我们用来打印组件树的 `FacesContext.getCurrentInstance()` 吗？尽管直到 JSF 2.2（是的，在 JSF 2.3 中是可能的），FacesContext 都不可被注入，但我们有类似的行为：每次你访问 FacesContext 的 `getExternalContext()` 时，你都会获得当前请求的信息，即使你可能是在几个请求之前获取的引用。

讨论 CDI 的所有方面远远超出了本书的范围。但我们将在接下来的几章中使用（并讨论）CDI 的某些方面。在开发图书编辑器时，我们将使用 `@ConversationScoped`。

## 总结

CDI 是控制反转模式的一种实现。它使框架能够创建类或接口的适当实例，并为我们的应用程序提供（注入）该实例。带有 `@Named` 注解的 Java 类自动符合注入条件，并将通过 `@Inject` 注解在适当的位置被注入。

CDI 本身提供了更多功能。例如，你可以通过为生产者定义接口来定义自己的注解。这超出了本书的范围。在本书中，我们将普遍使用预定义的注入。
