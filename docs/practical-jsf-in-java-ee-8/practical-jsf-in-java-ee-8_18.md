## 图书需求

Books 应用旨在向用户展示由某位作者撰写的书评概览。对于每本书，可以链接外部或在本应用中发布的评论。这些链接可能指向外部在线资源（例如其他网站），或者只是指向评论所发表的印刷媒介的相关信息。用户可以按类别列出图书并进行搜索。需求相当多，可以整理成小型用户故事：

*   作为评论作者，我希望输入图书信息
*   作为评论作者，我希望输入评论
*   作为评论作者，我希望维护类别等基础数据
*   作为用户，我希望阅读图书信息和评论
*   作为用户，我希望按标题、作者和 ISBN 搜索图书
*   作为用户，我希望按类别列出图书
*   作为用户，我希望检索并从所有图书和所有评论的列表中进行选择
*   作为用户，我希望选择我偏好的语言（如果可用）

所有面向用户的功能都使用相同的外观风格——也就是说，无论当前显示的是图书还是评论，导航栏都保持在相同位置。管理后台的外观风格则有所不同，以区分不同的领域。管理页面使用不同的背景色来标识管理区域。为此，使用了 JSF 模板化。稍后，在第四部分“校友”的章节中，我将讨论根据用户选择更改外观风格的方法。

每本书都将以表格形式显示，包含以下信息：

*   标题
*   副标题
*   作者
*   出版商
*   出版年份
*   语言
*   ISBN
*   简短描述文本
*   封面图片
*   外部和/或内部评论的引用
*   一个或多个类别的引用

尽管标题、作者、出版商等大部分信息在不同语言中保持不变，但其他信息（如简短描述文本和类别）可能会以不同语言显示。Books 的当前在线版本支持英语和德语。这并非技术限制——仅受限于翻译能力。

除了图书，Books 应用还显示评论。其结构很简单：

*   评论所涉及的图书
*   评论的语言
*   评论内容

该应用会自动适应不同的显示尺寸（得益于*响应式设计*）。

尽管需求相当简单，但这个应用展示了许多技术。Books 不仅仅是一个演示应用，它已上线运行在我的网站 [`it-rezension.de`](http://it-rezension.de) 上。

由于应用的数据结构良好且同质化，因此非常适合使用 SQL 数据库。Books 使用 JPA 来访问数据库。

外观风格以及对各种显示尺寸的适配是通过使用层叠样式表 (CSS) 实现的。

你可以在我的网站上看到 Books 的在线版本。顺便提一下，除了本书之外，你还可以将其用作参考书目。

## 开发顺序

那么——从哪里开始呢？是从一个可以提供给客户进一步评审的 GUI 原型开始？在许多项目中，这可能是首选方案。可以先草拟一些对话框，包括一些导航按钮，以展示软件的预期行为。这些原型可以使用演示软件构建，或者直接用目标语言（此处为 Java 与 JSF）实现。后一种方法能给客户带来更好的感受。

或者，我们应该通过绘制 UML 或其他图表来建模整体蓝图和活动？还是先设计数据模型（例如，使用实体图）？

在实践中，你会看到所有这些方法以及更多其他方法。哪种是启动项目的最佳方式取决于项目属性以及相关人员的个人偏好。

以下是我们称之为瀑布模型的过程：

1.  收集并记录所有需求
2.  设计并记录整体软件架构
3.  详细记录所有计划的功能
4.  编写所有代码
5.  测试整个系统
6.  编写用户文档并交付

这似乎是一种详尽的方法。但到系统交付时，需求可能已经发生变化，或者某些功能的表现可能与客户的预期不符（如果客户对预期有清晰的认识的话）。一个教训是：瀑布模型大多会失败。敏捷方法更可取。敏捷宣言（请阅读 [`agilemanifesto.org`](http://agilemanifesto.org) ）是这场运动中的一个里程碑。

将项目分解为可迭代、可增量实现的多个部分。对于每次迭代，我们都需要一个简短的规划会议。开发或完善一部分软件并进行测试，然后——非常重要的一点——交付给客户，或者至少交付给客户内部的替代者。这样，开发过程中就可以包含一个短反馈循环。客户可以快速了解软件的概况。需求变更可以迅速纳入。许多敏捷实践者不喜欢这种比较，但在敏捷中，每次迭代内的开发实际上遵循了类似于瀑布模型的过程，尽管是一个（非常）小的瀑布，并带有短反馈循环。

作为迭代过程的结果，开发顺序可能会因影响因素而改变。而详细的顺序变得不那么重要，因为在每次迭代之后，有时甚至在迭代之内，都可能进行修正。

对于像 Books 这样的数据驱动型应用（主要向用户展示存储在某个地方的数据），我倾向于尽快创建对话框。由于第一个对话框应该向用户展示一些未来的外观风格，我们需要一个整体布局，一种用于嵌入对话框的框架。

这个对话框应该使用什么数据？这取决于项目规模、参与人数、他们的经验等等。无论如何，数据模型是一个很好的起点。根据团队结构，对话框可以使用此模型并模拟任何持久化操作。数据库专家可能通过服务层将数据模型连接到数据库。

模拟数据可能对测试有用，或者当不同的人为一个功能的各个部分（例如，数据库部分、服务部分或表示部分）进行实现时有用。为了向用户展示正在进行的工作，通常使用一些真实的数据处理会更好。正如我们在下一章实现第一个对话框后所看到的，用于访问数据的服务层可以被泛化和重用。因此，从一开始就为每个对话框使用持久化数据是没有问题的。

就 Books 而言，数据结构并不难设计——我们从数据模型和数据库访问开始，然后创建使用这些数据的对话框。通过迭代工作，每个对话框都类似地重复此过程。

由于第一个对话框没有服务层，我们一次性填充了伪持久化数据。为了让你在学习过程中更清晰，实际操作有时可能与实际开发略有不同。



###### 创建图书项目

要开始编写图书程序，我们需要先创建项目。项目创建步骤与第 1 章所述相同，因此我假设你无需额外帮助即可自行创建项目（如有必要，可重读第一章）。

1.  使用 NetBeans 或你喜欢的 IDE，创建一个名为 Books 的新 Maven Web 项目。

2.  将 JSF 框架添加到项目中。

3.  由于 NetBeans 默认不支持 Java EE 8，你需要稍微更新一些生成的文件——特别是 pom.xml 和 web.xml（更多信息请参见第 6 章）。
