# 15. 选择合适的库

在本章中，我将分享一些关于库的看法，如何比较库与框架，以及一些选择合适库的工具。通常，对于完成同一项任务，会有不止一个库可供选择。因此，知道如何挑选合适的库是很有用的。

## 什么是库？

在我们的行业中，对于库和框架之间的区别并没有广泛认同的定义。我使用的区分标准是：

库是你调用的东西。框架是调用你的东西。

另一个实际的区别是，在库中，没有你未要求的事情会发生。代价是你需要自己连接各个部分，并编写一些引导代码来让一切启动并运行。

相比之下，在框架中，很多事情会自动发生，以节省你的工作量并让你的生活更轻松。代价是，当某些东西出问题，或者框架没有在应该自动调用你的代码时调用它，可能很难找出原因，甚至很难首先意识到出了问题。

当我使用框架时，遇到的一个主要现实问题是，当事情出错时，你会面对大量你之前并不了解的框架代码，并且突然必须理解它们才能继续前进。那些停止自动调用你代码的框架代码，可能需要大量工作才能理解发生了什么以及为什么。

不过，这里存在一个灰色地带。在第 9 章中，你配置了 Ktor 来处理会话 cookie 和授权。你在任何地方都看不到那段代码。你所做的只是将路由包裹在 `authenticate("auth-session")` 中并期望一切顺利。Ktor 底层有大量的映射和操作来使其工作，如果它停止工作，你不一定能知道原因。

Ktor 和库的好处在于，你确实调用了一个函数（`authenticate`），并且通过名称（`"auth-session"`）引用配置。所以它并没有那么隐晦和神奇。Ktor 不会自动对身份验证做任何事情；你必须告诉它去做。

当你使用库时，你并不一定能完全避免魔法失效的情况。但你可以大大降低这种情况发生的几率，从而从你的 Web 应用中消除一整类 bug。

## 流行度基本无关紧要

有些人将库的流行度视为一个重要指标。流行度并非*完全*无关紧要的指标。如果你发现两个库做同样的事情，你可能会选择更流行的那个。

然而，在选择库时将流行度作为主要指标可能会妨碍你独立思考。选择库时最重要的事情是运用你长期以来积累的所有良好品味和判断力，评估各种选项，并做出明智的选择。如果答案仅仅是“使用 Spring Security，因为每个人都在用 Spring”或“使用 React，因为每个人都在用 React”，那么你并没有真正思考。

将流行度作为指标所隐含的假设是，选择流行库（或框架）的人之所以这样做，是因为它总是最佳选择，并且每个选择该库的人都是在深思熟虑和评估之后才做出的选择。通常，决定哪些库流行更像是一种时代精神。而且，最流行的库往往是那些在*没有*深入思考什么对他们的项目合适的情况下选择的结果。

不过，请记住，有一种流行度很重要：库的可用性。如果你使用一种非常不流行的编程语言，你将不得不从头编写大量代码，而无法借助库的帮助。

在 Paul Graham 2001 年的文章《战胜平均数》（[*www.paulgraham.com/avg.xhtml*](http://www.paulgraham.com/avg.xhtml)）中，Graham 认为 Lisp 是一种非常出色的编程语言选择，仅仅因为 Lisp 语言本身的强大力量，他们就领先于竞争对手。这当然是事实（毕竟我骨子里是个 Lisp 爱好者），但仅限于一定程度。如果有人明天发明了宇宙中最好的语言，它超级高效，拥有健全且完备的类型系统，从不妨碍你，但解决了停机问题并且不可能写出 bug，你仍然可能无法领先于使用 Java 或 PHP 的竞争对手。这些平台上可用的大量第三方库几乎总能让你击败“完美语言 9000”的用户。



## 文档与稳定性至关重要

评估一个库的客观标准之一，是其文档质量和 API 的稳定性。

Ktor 就是一个在这两方面都表现卓越的典范。在 [*https://ktor.io/docs/welcome.xhtml*](https://ktor.io/docs/welcome.xhtml) 上提供的文档数量几乎无与伦比。库中的每一个细节都有文档说明，并附有大量示例和解释，涵盖了 Ktor 的所有模块。Ktor 也非常稳定。所有实验性功能都明确地用注解标记，你必须使用其他注解才能选择使用它们，这样就不会意外地自动补全到 Ktor 中未来可能发生变更的部分。一旦 Ktor 团队公开发布某个功能并移除不稳定标签，他们会尽一切努力避免破坏 API。

事实上，本书中选用的所有库都具备这些特性。Flyway、HikariCP、Kotliquery、Typesafe Config、Gson、bcrypt 和 Spring Security 都是文档出色且公共 API 几乎完全稳定的库。

如果非要取舍，我会优先考虑稳定性而非文档。Kotliquery 是一个小型库，没有成千上万的 GitHub 星标，文档也略显稀疏。但它的 API 面足够小，以至于详尽的文档并非那么重要；而且它是一个设计良好的库，一直稳定运行，我在新版本发布时安全升级，从未遇到任何问题。

## 避免遵循“某某方式”

当你阅读关于库和框架的资料时，会看到诸如“Ktor 方式”、“Spring 方式”或“Rails 方式”的说法。我倾向于避开这类模式，按自己的方式行事。（同时我也注重团队合作，所以这里的*我的方式*指的是*我们团队的方式*。）

这并不是说我完全忽视最佳实践。我会尽可能多地积累知识，了解业界人士如何组织代码和抽象，使其易于维护和阅读。

然而，没有哪种“某某方式”能涵盖所有情况，全盘照搬预设模式并不能很好地替代独立思考。就像你在选择项目要添加的库时，需要运用你所有的判断力和良好品味一样，编写大部分代码时也需要同样的技能。你应该始终主动决定要做什么，并质疑自己为什么这么做。没有永远有效或永远无效的模式，也没有一种编写软件的方法论是永远不需要用批判性思维去评估和深思的。

此外，如果项目中的所有程序员都习惯于编写函数式风格的 Clojure 应用并手写 SQL，那么凡事都“按 Rails 方式”并使用 ORM 来处理一切，未必是个好主意。尽管 ORM ActiveRecord 是 Rails 的主要卖点，但它可能会妨碍团队开发者的经验，并且不一定能增加任何价值。

## 不要回避平台库

当一门新语言诞生时，该语言社区往往倾向于从头开始编写一切，使其“语言原生”。虽然有很多库可以完成大多数事情，但新语言并不总能利用这些库。

然而，像 Clojure 和 Kotlin 这样的语言拥抱了平台。你不需要将所有东西都封装在 Kotlin 专属的库中才能使用。

许多库仅以 Java 库的形式存在。例如，用于与 Stripe 支付 API 交互的库只有 Java 版本，没有 Kotlin 版本。而且你实际上也不需要。从 Kotlin 调用 Java 代码非常容易。你在本书中已经做过很多次了。例如，在第 14 章中，你与所有 Java 代码中最具 Java 风格的代码——Spring Security API——进行了交互。Kotlin 的设计非常优美，能够充分利用 Java 提供的一切。

根据我的经验，有些程序员会拼命寻找语言原生的库。*也许 Java 平台上有一个通用的 bcrypt 库。但有一个 Kotlin 库可用，是用纯 Kotlin 从头编写的，那我们就用这个吧！也许它在 GitHub 上只有三颗星，而且并没有真正维护，但只要它是 100% 的 Kotlin，什么都可以。*

我建议不要这样做。在 Clojure 和 Kotlin 中，我都使用 Flyway 进行数据库迁移。它对现有的 Java 平台开发者来说很熟悉，拥有小巧灵活的 API 面，让我可以按照自己的意愿将迁移脚本写成纯 SQL，并且我可以依赖一个拥有庞大社区且积极维护的库，而不是某个 Clojure 或 Kotlin 专属的迁移库，后者未来被废弃的可能性要高得多。

此外，语言原生的库可能远没有更广泛使用的 Java 库那样功能丰富。例如，Flyway 能处理旧版 Oracle 的迁移。而你那个初出茅庐的花哨语言原生库，不太可能支持这类边缘情况。（是的，我刚把 Oracle 称为边缘情况。有本事来告我啊！）

## 边界与分层

我非常喜欢 Ktor 的一点是，它开箱即用地支持大量对 Web 应用有用的机制。例如，Ktor 提供了基于 Cookie 的身份验证、JWT 身份验证、用 Kotlin 编写 HTML 的 DSL、用于将 Jetty 和 Ktor 映射到 Spring Security 的 Servlet 等等。一个提供大量能良好协同工作的构建块的库，可以为你省去大量工作。这就是我选择 Ktor 作为本书主要工作引擎的原因。

然而，我强烈回避那些做得太多，却不给我逃生通道和可组合性的库。Ktor 让我控制它的运行方式，并让我从大量的功能菜单中挑选，而无需与它抗争。Ktor 默认不设置任何东西；我必须明确选择启用我想要的一切。

你用于数据库迁移的库 Flyway 也是如此。它有一些默认设置，例如可重复迁移与版本化迁移的命名前缀是什么，在类路径的哪个位置加载迁移文件等等。但你可以轻松地覆盖所有这些默认值。而且，最重要的是，在你调用创建的 Flyway 对象上的 `migrate()` 方法之前，Flyway 不会做任何事情。

Ktor 和 Flyway 也都提供了编程式 API。Ktor 有一种配置文件模式，它会从类路径加载配置文件，并根据该配置文件的内容进行自我设置。你可以直接调用 `io.ktor.server.netty.EngineMain.main()`，而不是使用它应该用于运行的属性来调用 `embeddedServer`。然后，你可以更新 Ktor 配置文件（类路径上的 *application.conf* 或 *application.yaml*），以指定你的 Ktor 初始化代码所在的类名。这种使用库的方式是我倾向于避免的。编程式 API 让我掌控一切，库不会规定我如何连接我的库集合。

在这方面，框架通常完全相反。Spring Boot 遵循约定优于配置的策略，仅仅在你的 Gradle 依赖中存在某个库，就足以让 Spring Boot 拾取它并假设你的设置。如果你对代码和显式性过敏，这很棒，但这几乎与我在编写 Web 应用时想要的完全相反。
