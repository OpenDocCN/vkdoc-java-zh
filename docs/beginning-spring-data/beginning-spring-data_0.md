第 1 章 应用程序架构

**类型 1：JDBC-ODBC 桥接器**

这种类型最初被用于第一批 JDBC 驱动程序，因为当时大多数数据库都支持 ODBC 访问。该驱动程序将 JDBC 转换为 ODBC，反之亦然，充当一个独立于数据库的适配器。你可以将其视为 JDK 的一部分，是一个通用驱动程序，因此项目中无需包含任何依赖项。

这种类型的驱动程序仅用于开发或测试；由于从 JDBC 转换为 ODBC 的性能问题以及安全方面的隐患，它不会在生产环境中使用。

**类型 2：基于客户端**

下一代驱动程序变得更加流行，因为它省去了将 JDBC 转换为 ODBC 的所有步骤，直接使用每个数据库厂商的原生库进行调用，这些库通常复用现有的 C/C++ 代码来创建。

其优势包括性能提升，因为无需在不同格式之间进行转换。但也存在一些缺点；例如，驱动程序必须安装在客户端机器上。

**类型 3：两层架构**

*两层架构*、*网络协议*和*带有中间件的纯 Java 驱动程序*都指向同一个概念。这种类型使用 JDBC，通过套接字与中间件服务器交互，进而与数据库通信。该中间件包含了所有不同数据库的驱动程序，因此无需在每台机器上都安装所有内容。

这种方法的主要问题在于需要一个专用服务器来与数据库交互，这意味着更多的信息传输，并引入了一个故障点。此外，厂商需要将 C/C++ 代码重写为纯 Java。

**类型 4：网络协议驱动程序**

*网络协议*或*原生协议驱动程序*是直接连接数据库最流行的方法之一。在这种类型中，驱动程序完全用 Java 编写，并使用每个厂商定义的协议与数据库通信。

这种类型提供了许多优势，包括与上一种类型相比性能卓越，并且无需在客户端或服务器上安装任何东西。但主要缺点是每个数据库都有一个使用不同协议的驱动程序。

**如何连接数据库**

根据性能要求，有几种方法，例如使用连接池。连接数据库的基本过程包括以下步骤。

1.  **导入类。** 你必须包含使用 JDBC 和连接数据库所需的所有类。大多数情况下，所有类都位于 `java.sql.*` 包中。

2.  **打开连接。** 使用 `DriverManager.getConnection()` 创建一个连接，该连接代表与数据库的物理连接。

3.  **执行查询。** 创建一个 `Statement` 对象，其中包含要在数据库中执行的查询。

4.  **获取数据。** 在数据库中执行查询后，你需要使用 `ResultSet.getXXX()` 方法来检索每一列的信息。

5.  **关闭连接。** 无论应用程序是否产生异常，你都需要显式关闭与数据库的连接。

**EJB**

企业级 JavaBean（EJB）是一种规范，用于构建可移植、可扩展且可重用的业务应用程序，而无需重新发明处理某些常见事务（如安全性、数据库访问、跨不同应用程序发送消息等）的方法。在 Java 的早期版本中，其中一些特性存在于其他库中，或者 JDK 内部提供了基本版本，因此当 EJB 出现时，它将这些特性统一到了同一个框架中。

**EJB 类型**

EJB 架构包含三个主要组件：EJB 容器、应用服务器和企业级 JavaBeans（EJB），后者又分为多种类型。

• **会话 Bean**：这些组件包含与用户任务或用例相关的所有逻辑。会话 Bean 在该任务或用例（如同一次对话）执行期间可用。但是，如果服务器崩溃或重启，会话 Bean 中保存的所有信息都将丢失。

会话 Bean 分为两种类型：有状态和无状态。主要区别在于，前者在调用后会保存状态，而后者（无状态）在调用后不保存任何信息。

• **消息驱动 Bean**：当需要应用程序之间的同步通信时，前面的 Bean 工作良好；但如果需要异步通信，消息驱动 Bean 则是正确的选择。最常见的实现使用 JMS（Java 消息服务）14。这种类型的 EJB 充当队列主题中的监听器，等待消息并在消息到达时执行相应操作。这种通信方式在微服务出现后变得流行。

• **实体**：此组件包含代表数据库表的类，并为数据库中的 CRUD（创建、检索/读取、更新、删除）操作提供接口。EJB 3.0 引入了对此类组件的更改，使用 JPA 替代了之前作为 EJB 规范一部分的实体 Bean。

**JPA**



Java 持久化 API（JPA）是一种直接与数据库持久化相关的规范 15。更具体地说，JPA 提供了一组所有提供商都必须遵循的接口，以确保它们遵循标准，这样你就可以在不费太多力气的情况下更换提供商。

使用 JPA，开发者可以通过一个代表表结构的 Java 类来访问数据库中的信息，并执行某些操作，如插入、更新、删除和检索。要执行这些操作，你需要使用代表表中常见内容的注解来标注类，例如表名、列大小以及表之间的关系。

JPA 提供了多种执行查询以检索信息的方式。一种是像 JDBC 那样直接使用 SQL 语句，并借助类来帮助构建语句；另一种则引入了一种抽象，这样你无需编写完整的语句，因为 JPA 的提供商会动态生成该语句。图 1-7 展示了 JPA 与应用程序的不同层级或结构。

**架构类型**

应用程序通常具有一种结构，有助于你理解在哪里可以找到逻辑，例如验证、业务逻辑、实体以及访问数据库的接口/类。所有这些对于理解层的层次结构以及创建新功能或应用程序时需要遵循的规则都很重要。

在 Java 中，像 Archunit22 这样的工具允许你定义并检查项目中的一切是否遵循你定义的结构。

**分层架构**

这种架构类型是最简单的，因为它将应用程序像蛋糕一样拆分成不同的层，每一层只能访问同一层或更低层的元素。这类架构从具有四层（如视图/表示层、业务逻辑层、持久化层和数据库层）的架构演变为三层或五层的架构。

对于正确的层数没有统一的标准。但是，如果你认为三层是较少的层数，那么 Spring Boot 由控制器、服务、仓库和数据库表示，而数据库在 Spring 中并不直接作为层出现。

**优点**

这种结构具有以下优点。

•   实现和维护简单，因为你只有一个层的层次结构。

•   每一层只有一个职责，并且不知道如何实现其他层的逻辑，因此你可以对某些层的逻辑进行修改，而不会直接影响其他层。

•   所有项目中的结构大致相同。你可以从一个项目切换到另一个项目，并知道在哪里找到所需内容。

**缺点**

这种架构类型具有以下缺点。

•   隐藏了用例，很难知道所有类或对象是如何连接起来表示一个用例的。

•   根据层的大小和数量，你可能会遇到可扩展性问题，因为你可能有多个类/



同一包或层中的接口。

**注意**：本书并未正式推荐您必须使用哪种架构类型，但大多数示例遵循分层架构。

**组件**

六边形架构在结构的每个部分内部都有多个组件，但我只提及理解基本思想最相关的部分。

• **基础设施** 包含负责使用不同协议与外部系统或数据库连接的端口和适配器。*端口*是与技术无关的接口，定义了访问应用程序或其他系统的方法。*适配器*是与特定技术相关的端口实现，例如对数据库的查询或 REST 控制器。

• **用例** 是系统的核心，因为它包含了与各种场景及其关联方式相关的所有逻辑。根据作者的不同，该组件可能被称为*应用程序层*或*领域层*，并将用例与领域组件相结合。

• **实体** 是应用程序的纯领域，包含应用程序组件中使用的所有值对象实体。

**优势**

这种结构具有以下优势。

• 您可以毫无问题地更改或替换适配器。
• 领域和用例是纯粹的，因此您可以理解业务逻辑。
• 您可以为了测试目的更改端口的实现。

这种架构类型具有以下缺点。

• 实现某些框架很困难，因为您可能会错误地判断什么是适配器/端口。
• 引入间接性来查找每个端口的实现。
• 没有指导方针解释有关项目组织的所有事项，因此任何人都可能引入错误。

**什么是 Spring Data？**

Spring Data 是 Spring 支持的一系列项目，旨在简化对 MongoDB、Redis、Cassandra、Neo4j、Memcached 以及所有关系型数据库的访问。在关系型数据库中，与 JDBC、Hibernate、JPA、JDO 或 MyBatis 等其他方案相比，Spring Data 提供了更简化的访问方式，减少了任何数据库操作所需的类、层和配置数量。

Spring Data1 首次出现在 2010 年的 Spring One 大会上。Rod Johnson（SpringSource 公司）和 Emil Eifrem（Neo Technologies 公司）尝试使用 Spring 将 Neo4j 数据库与应用程序连接起来。他们创建了用于在应用程序和数据库之间建立连接的类；这些类成为了 Spring Data 的基础。此后，该项目家族不断壮大，为不同的数据库提供支持，遵循着降低复杂性并在应用程序与数据库之间建立一定抽象层次的精神。

现在，让我们思考一下为什么需要使用 Spring Data 而不是数据库驱动程序。假设你有一个使用 Redis 的应用程序。维护这个数据库的成本很高，因此基础设施团队建议将信息迁移到经过高度优化的 Cassandra。这种情况下的问题是：如果你仅使用数据库驱动程序来实现所有持久化和检索信息的逻辑，那么你需要修改大量与 CRUD 操作、连接以及所有配置相关的代码行。但如果你使用 Spring Data，更换数据库只需修改实体中的注解和配置即可。

Spring Data 包含多个模块组；有些模块适用于所有类型的数据库，而另一些则针对特定类型。让我们来看看这些模块中最相关的一些。

• **Spring Data Commons** 提供了一个元数据模型，用于通过下一节中的仓库接口将 Java 类持久化到特定数据库中。

• **Spring Data REST**2 构建在 Spring Data 模块之上，帮助你使用与实体关联的仓库创建 RESTful API。

• **Spring Data JPA**3 提供了 JPA 的实现，它使用仓库接口，在大多数情况下支持所有 CRUD 操作。如果需要，你也可以创建具有特定逻辑的自定义仓库来访问数据库。

• **Spring Data JDBC**4 提供了使用 JDBC 通过仓库访问数据库的实现，但不具备 JPA 的更多功能，例如懒加载和缓存。

• **Spring Data for Apache Cassandra**5 支持 Cassandra 实体中的所有 CRUD 操作和 TTL 定义。它还支持配置数据库的多个实例。

• **Spring Data MongoDB**6 提供了使用仓库或 `MongoTemplate`（另一种执行 CRUD 操作的方式）在 MongoDB 数据库中执行所有操作的可能性。
