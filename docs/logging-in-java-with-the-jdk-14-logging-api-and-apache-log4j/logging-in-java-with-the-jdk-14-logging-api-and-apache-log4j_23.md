# 第 10 章：最佳实践



## 概述

在本书中，我们讨论了两种最流行的基于 Java 的日志记录 API：JDK 1.4 日志记录 API 和 Apache log4j API。这两种日志记录 API 在功能上相似，但 log4j API 提供了更大的灵活性和更强大的日志记录能力。这两种 API 都允许我们通过配置文件来配置和控制日志记录行为。这些 API 还提供了一系列我们可以用来在应用程序内部记录任何消息的方法。例如，我们可以通过编程方式覆盖日志记录框架中不同组件的状态。我们已经看到，在这两种日志记录 API 中，扩展现有的日志记录框架也是可能的，并且我们可以在框架内集成自己的自定义日志记录组件。这有助于我们在应用程序中实现特定的日志记录行为。尽管这非常有用，但在使用现有日志记录框架或通过添加自定义组件来扩展它时，仍有一些要点需要考虑。

请记住，日志记录并非大多数应用程序的主要目标。它仅仅是为了帮助我们跟踪应用程序的内部状态。它帮助我们更快地发现问题，发布消息以确定系统运行状况，并以更少的时间和成本维护和调试应用程序。另一方面，错误地使用日志记录可能会影响应用程序的速度和功能。因此，在为日志记录过程选择正确的组件以及使用这些组件的最佳方式时，需要格外小心。检查自定义日志记录组件是否以任何方式影响现有组件的速度和完整性也非常重要。

在本章中，我们将讨论一些对日志记录性能至关重要的要点。除非另有说明，这些要点对于 JDK 1.4 日志记录 API 和 Apache log4j API 均适用。

最后，本章将从内部工作机制和性能方面对 log4j 和 JDK 1.4 日志记录 API 进行比较，以便在您自己的应用程序中实现日志记录时，您能知道哪个 API 最能满足您的需求。

## 获取日志记录器

在 JDK 1.4 日志记录 API 和 log4j 中，日志记录框架的入口点是 `Logger` 对象。我们获取一个 `Logger` 对象的实例，并让它以不同的级别记录不同的消息。通常，在应用程序中，我们会获取命名的日志记录器。这些 `Logger` 对象应该使用什么命名约定呢？

对于日志记录器的名称应该是什么，并没有严格的规定。尽管如此，我们在为应用程序中的 `Logger` 对象选择命名约定时仍需格外小心。我们通过配置文件来配置一个 `Logger`。我们如何知道哪个应用程序组件正在使用哪个 `Logger` 对象？如果我们更改某个特定日志记录器的配置，它将如何影响整个应用程序？想象一下，我们有一个包含数百个不同组件的应用程序，它们都使用一个公共的日志记录配置文件。在这种情况下，我们肯定需要为日志记录器对象制定一个命名策略，以便我们知道每个组件正在使用哪些单独的日志记录器。通过采用设计良好的命名约定，我们可以轻松评估对特定日志记录器配置所做的任何更改所产生的影响。

|  | 最佳实践 | 对 `Logger` 对象使用基于包的命名约定。 |

一个很好的命名约定思路是基于包的命名。通常，应用程序组件被划分到多个包中。每个包又包含应用程序特定方面的相关类。使用基于包名的 `Logger` 对象是一个好习惯，因为这种约定易于维护。

|  | 最佳实践 | 当同一包中的组件可能需要不同的日志记录行为时，应根据组件执行的任务来区分日志记录器。 |

紧接着出现的问题是，一个包是否包含执行多种不同活动的类。例如，包 `com.apress.logging.log4j.util` 可能包含许多工具类。如果所有这些类都使用名为 `com.apress.logging.log4j.util` 的同一个日志记录器，那么更改此日志记录器的配置将影响该包内的所有工具类。这可能是不希望的。例如，`util` 包中的一个类可能正在将数据写入文件，而另一个类可能正在将数据发送到远程 TCP/IP 服务器。我们可能需要为执行 TCP/IP 操作的类启用详细的日志记录，但可能不需要为执行文件 I/O 的类提供相同级别的详细日志记录。在这种情况下，使用两个不同的日志记录器并根据它们执行的工作来命名可能会有所帮助。

要点在于为应用程序中使用的日志记录器制定一个合适的命名约定。这将提高日志记录器的可维护性。同时，它也有助于避免一个日志记录器的配置更改影响其他日志记录器这种不期望的情况发生。

## 使用日志记录器层次结构

JDK 1.4 日志记录 API 和 log4j 都支持日志记录器层次结构。日志记录器以父子关系组织。例如，名为 `com.foo` 的日志记录器是子日志记录器 `com.foo.bar` 的父日志记录器。在日志记录器层次结构中，子日志记录器从其直接父日志记录器继承属性和日志记录器组件。简而言之，这意味着子日志记录器捕获的所有日志记录事件将由子日志记录器本身及其父日志记录器共同处理。这也可能意味着，如果子日志记录器和父日志记录器使用相同的日志记录目标，日志消息将出现两次，这可能是不希望的。当然，在这两种 API 中，我们都可以关闭层次结构中对父日志记录器的使用。在 JDK 1.4 日志记录 API 中，使用以下代码：

```
setUseParentHandlers(false);
```

在 log4j 中，使用

```
setAdditivity(false);
```

或者通过配置文件添加以下行：

```
log4j.logger.[logger name].additivity=false
```

同样，在某些情况下，我们可能希望启用此日志记录器层次结构。

|  | 最佳实践 | 当一个应用程序组件试图通过使用属于同一包的不同组件将相同的信息记录到不同的目标时，尝试使用日志记录器层次结构。 |

想象一个订单处理应用程序，其中组件 `A` 接收订单。它处理订单，现在需要将特定于订单的日志记录信息存储到文件中，并且还希望将相同的信息发送回公司日志记录数据库。应用程序组件 `B` 将数据写入本地文件，组件 `C` 将日志记录信息写入公司数据库。现在需要考虑的重要点是，应用程序组件 `A`、`B` 和 `C` 是否属于一个包层次结构。例如，组件 `B` 和 `C` 可能属于包 `com.processor`，而组件 `A` 可能属于包 `com.processor.order`。另一个组件 `D` 可能处理基于电子邮件的订单，并且可能位于包 `com.processor.order.email` 中。在这种情况下，我们可以利用父子日志记录器关系。

要点在于，我们需要判断日志记录器是否能够基于日志记录活动（而不仅仅是包结构）来证明父子关系的合理性。如果它们在日志记录行为方面是父子关系，那么就使用这种关系。

|  | 最佳实践 | 如果启用了日志记录器层次结构，日志记录框架需要确定父日志记录器（在日志记录器树中递归向上查找）。这可能会降低性能。 |

使用 log4j 的基准测试结果表明，遍历日志记录器层次结构比正常的日志记录活动慢三倍。因此，在确定是否在应用程序中使用日志记录器层次结构时需要格外小心。本书提供的示例旨在保持父子关系关闭，除非有必要演示父子关系是如何工作的。



## 高效记录日志信息

记录什么以及如何记录？这些问题的答案决定了特定应用程序日志记录的有效性。请谨慎选择并制定策略，明确哪些信息用于调试，哪些信息用于日常维护。仔细决定在何处记录信息。还要决定日志消息的格式，以便将来其他应用程序能够处理和分析这些信息。

|  | 最佳实践 | 避免记录不必要的信息。这会使日志跟踪变得混乱，增加分析难度。 |

冗余且不必要的日志信息会在性能和实用性方面影响应用程序。即使我们在已部署的应用程序中关闭了某个日志级别，日志框架仍需为每条日志消息执行额外的级别检查。这种针对包含冗余和不必要信息的日志请求进行的级别检查，会影响整体日志记录的性能。因此，最好避免在应用程序中包含此类日志请求。

|  | 最佳实践 | 避免在日志请求中使用参数构造。 |

在开发自己的应用程序时，您可能认为已经避免了不必要的日志请求；然而，您可能仍然会遇到类似下面的日志请求：

```
logger.info("这是 "+var1+" 来自方法 "+var2+" 的日志");
```

这里涉及的成本是参数构造。请使用简单的 Java 技术，例如使用 `StringBuffer` 来拼接 `String` 对象，避免使用 "+" 连接两个 `String` 对象。这只是一个例子。一个编码糟糕的应用程序可能包含多个此类语句。

|  | 最佳实践 | 在尝试记录日志之前，先检查日志是否已启用。 |

假设我们有一个应用程序，它会记录客户订单列表中的所有条目（这几乎不可能！）。所有级别为 DEBUG 的信息都会被打印出来：

```
ArrayList list;
int length = list.size();

for(int k=0;k<size;k++)
{
    logger.debug("客户: "+ ( (Customer)list.get(k) ).toString() );
}
```

如果 DEBUG 级别被关闭了会发生什么？在 `for` 循环内部，参数构造仍然会发生，日志记录器会检查 DEBUG 级别是否启用，如果未启用，则会丢弃该请求。为了避免这种情况，我们可以这样写：

```
if(logger.isDebugEnabled())
{
    //在此处执行 for 循环
}
```

这将节省不必要的参数构造开销。另一方面，如果 DEBUG 级别已启用，则评估 DEBUG 级别是否启用可能会产生额外的开销。但与参数构造相比，这是一个微不足道的开销，因为它只占实际日志记录时间的 1% 左右。

## 本地化问题

通过 `java.util.ResourceBundle` 对象实现日志信息的本地化，是一个以不同语言发布日志信息的优秀特性。然而，决定在 log4j 中使用本地化功能可能比较棘手。

|  | 最佳实践 | 当您不使用本地化功能时，不要使用 `ResourceBundle` 对象。 |

构造和使用 `ResourceBundle` 对象的成本很高。如果您的应用程序目前只支持一种语言，请避免在代码中使用本地化。您将来随时可以启用此功能以支持不同的语言。

## 使用位置信息

JDK 1.4 日志 API 和 log4j 都支持日志请求的位置信息。JDK 1.4 日志 API 尝试从堆栈跟踪中确定日志请求的类和方法的名称。log4j API 则尝试检测发出日志请求的代码的确切行号。

|  | 最佳实践 | 确定位置信息是一项成本高昂的操作。如果可能，最好在应用程序日志记录中避免使用它。 |

动态生成位置信息并不可靠。它可能无法与许多使用优化技术的编译器正常工作。此外，日志框架需要遍历堆栈跟踪来确定位置信息，这会带来大量的额外开销。

## 格式化日志信息

日志信息的格式化很重要。一方面，信息越多越好。另一方面，每一点信息都会消耗 CPU。我们需要仔细决定日志信息的格式。有多种选择。XML 格式比简单的基于文本的输出成本更高，但 XML 格式的日志信息更具可重用性和可移植性。

log4j 中的 `PatternLayout` 对象在定义最终日志信息的元素和格式化指令方面提供了极大的灵活性。使用 `PatternLayout`，我们可以精确控制任何目标位置中日志信息的内容。对于基于 XML/HTML 的日志信息，我们需要使用单独的 `Layout` 对象。`PatternLayout` 负责格式化日志信息的内容，不参与日志信息的呈现。

日期是格式化时的主要关注点之一，因为它们很复杂且性能成本高昂。log4j API 提供了多种日期格式化对象，例如 `DateLayout` 和 `ISO8601DateFormat`。在所有可用的日期格式中，`ISO8601DateFormat` 是最便宜且最快的。

## 使用渲染器对象

在 log4j 中，`Renderer` 对象用于为传递给日志框架进行记录的任意 `Object` 构建一个 `String` 表示形式。

|  | 最佳实践 | `Renderer` 对象遵循类层次结构。任何超类 `Renderer` 都能够渲染其子类。 |

例如，如果我们有一个 `VehicleRenderer` 对象，只要 `Car` 是 `Vehicle` 的子类，它就能渲染 `Car` 对象。

|  | 最佳实践 | 对于子类的专门化渲染，请注册一个特定于子类的 `Renderer`。在这种情况下，不会调用超类 `Renderer`。 |

例如，我们可以注册一个 `CarRenderer` 对象来渲染所有 `Car` 对象，在这种情况下，`VehicleRenderer` 将不会被用来渲染 `Car` 对象。

## 使用异步日志记录

log4j API 提供了 `AsyncAppender` 对象来执行异步日志记录。该对象将日志事件存储在一个有界缓冲区中，并在达到最大缓冲区大小时释放这些事件。JDK 1.4 日志 API 也通过 `MemoryHandler` 对象提供了类似的功能。

|  | 最佳实践 | `AsyncAppender` 和 `MemoryHandler` 并不总能提高性能。 |

如果有界缓冲区很快被填满，那么管理有界缓冲区会产生额外的开销，从而降低性能。但是，在涉及长时间阻塞的网络或 I/O 访问，或者 CPU 密集型操作较少的情况下，使用 `AsyncAppender` 或 `MemoryHandler` 是非常有效的。



## 使用过滤器对象

在第 5 章关于 log4j 的讨论中，我们看到了 `Filter` 对象的示例，它们除了根据日志级别进行过滤外，还能根据某些特定于应用程序的标准来过滤日志请求。我们还了解了过滤器链如何使得根据多个标准检查日志请求成为可能。

但过滤器链涉及加载多个 `Filter` 对象，并调用每个对象的 `decide()` 方法来过滤日志请求。我们在第 8 章中演示的一种替代方案是编写一个自定义的 `Filter` 对象。

|  | 最佳实践  | 尽量在一个 `Filter` 对象中完成尽可能多的操作。 |

在需要创建多个自定义 `Filter` 对象的情况下，请尝试将它们设计成这样一个 `Filter` 对象可以服务于多种常见的过滤目的。毫无疑问，在一个 `Filter` 对象中嵌入各种应用程序逻辑会使其难以复用，并将其限制在特定的应用场景中。尽管如此，在某些情况下，这种做法仍然值得考虑。

假设我们正尝试将日志信息记录到文件中。我们想要检查日志事件是否由某个特定应用程序生成，以及它是否拥有执行文件 I/O 操作所需的必要安全权限。理论上，我们可以设计两个 `Filter` 对象：一个用于检查应用程序来源，另一个用于权限检查。这样，我们就有了两个可复用的 `Filter` 对象，分别用于事件来源检查和文件 I/O 权限检查，这很好。但是，如果我们并非在编写供其他应用程序使用的通用日志组件，那么这可能就没那么必要了。尝试将这两个检查过程合并到一个 `Filter` 对象中。这将节省加载和调用两个 `Filter` 对象所需的时间。

依作者之见，`Filter` 对象的设计是特定于应用程序的，并且通常需要一些耐心和时间来决定如何设计 `Filter` 对象。有时，你可能需要设计出能被许多其他应用程序组件复用的通用 `Filter` 对象；而有时，你则会编写特定于某个应用领域的 `Filter` 对象。在决定编写单个还是多个 `Filter` 对象之前，你需要进行权衡。

## 使用嵌套诊断上下文

log4j API 提供了嵌套诊断上下文（NDC）对象，用于用客户端特定的信息来填充日志信息。例如，处理多个客户端的基于服务器的应用程序（例如 servlet）通常需要区分由不同客户端活动产生的日志跟踪信息。

|  | 最佳实践  | 在处理多个客户端的服务器端组件中使用 `NDC` 来区分客户端。 |

`NDC` 的核心理念是包含唯一的客户端特定信息。在决定将哪些信息放入 `NDC` 以及其唯一性程度时需谨慎。

## 配置问题

log4j API 同时支持属性样式和 XML 样式的配置文件。还有一种通过编程方式配置每个 log4j 组件的方法。但是，如果不修改源代码，就无法更改编程式配置。为了充分利用 log4j 高度可配置的特性，我们必须通过配置文件来配置 log4j。

这一点同样适用于 JDK 1.4 日志 API。尽管 JDK 1.4 日志 API 不像 log4j 那样高度可配置，但我们仍然必须坚持通过配置文件来配置它。目前，JDK 1.4 日志 API 仅支持属性样式的配置。

并非 log4j 中的所有组件都可以通过属性样式的配置进行配置。因此，使用 XML 样式的配置文件通常是一个更好的实践。

如果你需要在系统运行期间更改配置，则必须为 log4j 配置实现 `configureAndWatch` 选项。这样，你就可以在不关闭系统的情况下更改配置文件。log4j 框架会在指定的时间间隔（默认为 60 秒）内检查配置是否发生更改，如果配置已更改，则会重新初始化框架。

可以通过在应用程序的起始点调用以下代码来使用 `ConfigureAndWatch` 属性：

```
PropertyConfigurator.configureAndWatch(props);
```

或

```
DOMConfigurator.configureAndWatch(configFileName);
```

log4j 框架将使用相同的配置文件来配置自身，只是额外增加了一个监视配置更改的功能。在基于服务器的应用程序中使用 log4j 时，如果每次日志配置更改都关闭服务器可能很困难，那么这一功能通常非常有用。

## 比较 log4j 和 JDK 1.4 日志 API

也许你已经注意到 log4j 和 JDK 1.4 日志实现之间惊人的相似之处。它们共享相同的基本组件，例如 `Logger`、`Level`、`Appender`/`Handler`、`Layout`/`Formatter`。这两个 API 在内部架构上也高度相似。然而，有一些基本的差异值得一提，其中一些差异在选择使用哪个日志 API 时可能至关重要。表 10-1 总结了这两个日志 API 之间的差异，随后将对每一点进行更详细的讨论。

表 10-1：JDK 1.4 日志 API 与 Apache log4j 的比较

| 特性 | JDK 1.4 日志 API | Apache log4j |
| --- | --- | --- |
| 配置 | JDK 1.4 日志 API 从 JDK 提供的默认配置文件开始。在 JDK 1.4 日志 API 中，配置文件中的配置顺序是顺序性的。父记录器必须在子记录器之前配置。JDK 1.4 日志 API 仅支持属性样式配置。 | Apache log4j 不假定任何默认配置。它会在类路径中查找 "log4j.properties" 或 "log4j.xml" 文件来初始化自身。在 log4j 中，配置顺序对初始化没有影响。Apache log4j 同时支持属性和 XML 样式的配置。 |
| 日志信息格式化 | JDK 1.4 日志 API 仅以文本格式或 XML 格式呈现日志信息。 | Apache log4j 支持多种输出格式，例如 HTML、XML、文本等。可以通过指定模式来控制日志信息的内容。 |
| 输出目标 | JDK 1.4 日志 API 仅支持套接字、控制台、文件和内存缓冲区作为输出目标。 | Apache log4j 支持更广泛的日志输出目标，包括 JMS 和数据库，以及 Telnet、SMTP 等协议。 |
| 过滤 | JDK 1.4 日志 API 允许将过滤器附加到 `Logger` 和 `Handler` 对象上。JDK 1.4 日志 API 无法实现针对多个标准的过滤器链。 | Apache log4j 仅将过滤器附加到 `Appender` 对象上。Apache log4j 支持过滤器链。 |
| 位置信息 | JDK 1.4 日志 API 会自动尝试确定日志请求的位置信息。无法控制此行为。 | 在 log4j 中，可以配置是否将位置信息包含在日志信息中。 |
| 错误处理 | JDK 1.4 日志 API 会将任何运行时异常抛回给调用应用程序。除非处理这些异常，否则可能导致应用程序崩溃。 | 在 log4j 中，错误在内部处理，不会抛回给调用应用程序。这使得应用程序和日志功能相互独立。 |



### 配置选项对比

与 log4j 不同，JDK 1.4 为日志记录 API 自带了一个默认配置文件。即使是应用程序开发者也无需定义任何自定义配置文件；JDK 1.4 日志记录 API 会使用默认配置文件自行配置。然而，log4j 不会对日志记录环境做任何假设。如果在执行环境的类路径中未找到配置文件，log4j 将无法正确初始化。

在配置选项方面，log4j 更加灵活，因为它同时支持属性样式和 XML 样式的配置。而 JDK 1.4 日志记录 API 仅支持属性样式的配置。

此外，JDK 1.4 的配置是顺序相关的——也就是说，配置的顺序很重要。例如，请看以下配置顺序：

```
a.b.level=SEVERE
a.level=INFO
```

你可能会惊讶地发现，日志记录器 `a.b` 将被分配其父日志记录器的级别 INFO。这是因为父日志记录器是在子日志记录器之后配置的，并且所有子日志记录器的配置都已根据父日志记录器的配置进行了更新。因此，在 JDK 1.4 日志记录 API 中，子日志记录器的设置应出现在父日志记录器的设置之后。父日志记录器会更新其子日志记录器的设置。

log4j 的配置与顺序无关。在 log4j 中，子日志记录器通过向上遍历层次结构，自动从其直接父日志记录器继承设置。

### 格式化器与布局对象对比

两种日志记录 API 都提供了格式化待生成日志信息的机制。在 log4j 中，这些对象被称为 `Layout` 对象，而在 JDK 1.4 日志记录 API 中，它们被称为 `Formatter` 对象。然而，JDK 1.4 日志记录 API 仅提供了两个 `Formatter` 对象：`SimpleFormatter` 和 `XMLFormatter`。log4j API 则提供了大量的 `Layout` 对象，能够以多种格式（如 HTML、XML、文本等）格式化信息。log4j API 还提供了 `PatternLayout`，这是一种强大的机制，用于在最终的日志输出中包含或排除日志事件的不同部分。

### 处理器与附加器对象对比

JDK 1.4 日志记录 API 提供了 `Handler` 对象，用于将日志信息发布到首选目标。类似地，log4j 提供了 `Appender` 对象来发布日志信息。目前，JDK 1.4 日志记录 API 只能发布到控制台、文件、套接字和内存缓冲区。然而，log4j 为日志记录目标提供了更多选择。使用 log4j，我们可以将日志信息发布到各种目标，例如控制台、文件、数据库、JMS、NT 事件日志、UNIX Syslog 等。

当然，两种 API 都可以扩展，以提供任意数量的自定义 `Handler` 和 `Formatter` 对象，从而支持记录到任何其他特定于应用程序的目标。但默认情况下，log4j 几乎涵盖了所有常用的日志记录目标。

### 过滤器对象对比

JDK 1.4 日志记录 API 提供了将 `Filter` 对象附加到 `Logger` 和 `Handler` 对象的功能。借助 `Filter` 对象，可以对日志记录决策进行更精细的控制。JDK 1.4 日志记录 API 只允许一个 `Filter` 对象与一个日志记录器关联。

然而，在 log4j 中，`Filter` 对象只能附加到 `Appender` 对象上。但 log4j 的 `Filter` 对象提供了更强大的功能。正如我们在第 8 章中讨论的，可以链接多个 `Filter` 对象来实现复杂的日志记录决策。我们还可以为特定的 `Appender` 对象配置多个 `Filter` 对象。JDK 1.4 日志记录 API 在这些 `Filter` 对象特性方面有所欠缺。JDK 1.4 日志记录 API 中的 `Filter` 对象要么接受要么拒绝一个日志记录请求，但它不能忽略一个日志记录请求并将其委托给其他 `Filter` 对象。因此，在 JDK 1.4 日志记录 API 中无法实现 `Filter` 链。

### 位置信息选项对比

JDK 1.4 日志记录 API 和 log4j 都能够生成关于日志记录请求产生位置的信息。然而，生成位置信息在性能方面代价非常高。

在 log4j 中，我们可以根据应用程序的需要启用或禁用位置信息。默认情况下，位置信息不是日志输出的一部分。我们可以通过在 `PatternLayout` 对象中指定位置信息属性来启用它，如第 6 章所述。

与 log4j 不同，JDK 1.4 日志记录 API 总是试图确定位置信息，并且这是无法避免的。此外，JDK 1.4 日志记录 API 用于动态生成位置信息的技术并非完全可靠。通常，这是通过分析应用程序的堆栈跟踪来完成的。但许多编译器采用不同的优化技术，这可能导致堆栈跟踪分析失败或产生错误信息。因此，应用程序开发者倾向于避免将位置信息作为日志输出的一部分。

### 错误处理对比

JDK 1.4 日志记录 API 的主要缺点在于它会将 `RunTimeException` 抛回给应用程序中的调用类。如果这些异常未在应用程序内部得到处理，应用程序将会崩溃。如果应用程序崩溃是由于日志记录本身的问题而非应用程序自身的问题，这种情况尤其不受欢迎。

然而，log4j API 在处理日志记录操作中产生的错误和异常方面更加优雅。log4j 中的所有 `Appender` 对象都有一个关联的 `ErrorHandler` 对象。诸如 `OnlyOnceErrorHandler` 之类的 `ErrorHandler` 对象会在错误首次发生时向控制台输出一条消息，并忽略后续的错误。这使得即使日志记录出现问题，应用程序也能继续正常运行。可以通过分析控制台消息来检测日志记录功能的问题，然后最终予以纠正。

此外，在 log4j 中，可以定义自定义的错误处理组件，这些组件可以执行各种操作来优雅地处理错误情况。这只需通过更改配置文件来使用任何 `ErrorHandler` 对象即可实现，完全不需要更改代码。

## 结论

至此，本书已接近尾声。我希望本书讨论的主题能为您提供关于两种最常用的基于 Java 的日志记录 API——JDK 1.4 日志记录 API 和 Apache log4j API——足够坚实的基础。两者都设计良好且易于使用。考虑到这两种 API 的现状，在我看来，log4j 略胜一筹。但同样，您并非总是需要如此复杂的日志记录机制。重要的是要理解，随着时间的推移，软件应用程序正变得越来越具有业务关键性。在这种背景下，具有更好可维护性和快速修复能力的应用程序将在市场中胜出。应用程序中精心规划的日志记录框架无疑将帮助其在竞争激烈的市场中占据优势。因此，现在是我们所有人重新考虑日志记录在应用程序中重要性的时候了。

随着大量建议不断通过 Sun 和 Apache 的邮件列表针对各自的日志记录 API 提出，我们很快将看到这两种 API 的改进和新特性。无论这些新特性是什么，本书中介绍的基础知识将保持不变，我希望本书现在和将来都能对您有所帮助。
