# 9. 连接性

> *Nos vies sont faites de tout un réseau de voies inextricables, parmi lesquelles un instinct fragile nous guide, équilibre toujours précaire entre le cœur et la raison.*

> *我们的生命由一张错综复杂的路径网络构成，其中一种脆弱的直觉引导着我们，那是在心灵与理智之间永远微妙的平衡。*
> 
> —乔治·多尔，《雪，爱》……

网络是任何物联网或边缘计算解决方案的核心。毕竟，连接性正是物联网区别于传统嵌入式计算的关键所在。为特定部署选择并实施正确的网络类型，其复杂性远超表面所见。L Peter Deutsch 及其在 Sun Microsystems 的同事早在 20 世纪 90 年代就指出了潜在陷阱，即如今众所周知的[分布式计算的谬误](https://en.wikipedia.org/wiki/Fallacies_of_distributed_computing)。列表如下：

1.  网络是可靠的。

2.  延迟为零。

3.  带宽是无限的。

4.  网络是安全的。

5.  拓扑结构不会改变。

6.  只有一个管理员。

7.  传输成本为零。

8.  网络是同构的。

由于物联网和边缘计算设备位于云或企业数据中心的边界之外，你可能已经预料到网络不可靠且带宽有限`–`除非你才刚刚入门。此外，你可能也没指望延迟为零，因为边缘计算基础设施的目标之一就是降低延迟。尽管如此，其余谬误仍然相关。安全性仍然是一个巨大挑战，网络拓扑会随时间变化，存在多个管理员，并且网络不会是同构的。

在本章中，我将探讨多种可用于实际部署的网络技术。之后，我将分享网络设计方面的考量。最后，我们将查看一些针对 Zephyr RTOS 的代码。

## 连接选项

将你的受限设备连接到网络有多种选择。其中一些直接支持 TCP/IP，这意味着设备理论上可以直接连接到互联网。蜂窝连接、Wi-Fi 和一些窄带选项就属于这种情况。其他技术不支持 TCP/IP，因此你需要部署一个网关来桥接本地设备网络与互联网。当然，我之前介绍的一些更高级别的协议可以在非 IP 传输上运行，并且仍然支持互联网连接。

在本节中，我将介绍目前市场上最主流的几种替代方案。但如何为你的项目选择合适的技术呢？以下是一些需要考虑的因素：

*   **硬件支持：** 是否有微控制器或网关在出厂时支持你考虑使用的技术？如果没有，市场上是否有射频模块可以与你计划使用的硬件集成？

*   **软件支持：** 你计划使用的 RTOS 或操作系统是否支持你选择的网络技术和特定射频模块？所需的库是专有的还是开源的？网络协议栈的成熟度如何？

*   **生态系统：** 你所评估的网络技术的供应商生态系统有多多样化？

*   **覆盖范围：** 你需要多大的有效范围来支持你的用例？是以米、数百米还是公里为单位？

*   **数据速率：** 考虑到你将从设备发送和接收的数据量，你需要多大的数据速率？你是否需要支持涉及行人、骑行者、汽车或其他车辆的移动性用例？

*   **电池寿命：** 你的设备是否由电池供电？如果是，你的目标电池寿命是多少？

*   **干扰：** 考虑到设备将部署的物理位置，你预计目标频段会有多少干扰？现场是否已经部署了使用相同技术或依赖相同频段的其他技术的设备？

*   **成本：** 与解决方案的其他方面相比，你可以在网络连接上投入多少资金？考虑到你的约束和要求，你计划使用的网络连接选项对开发时间、维护工作以及硬件和软件的生命周期有何影响？

最终，你可能需要做出妥协。现实世界可能很复杂，尤其是在涉及无线电技术时。



### 蓝牙

在本节列出的所有技术中，只有以太网和 Wi-Fi 像蓝牙一样普及。你的手机、汽车和笔记本电脑可能已经集成了这种无线技术。蓝牙由[蓝牙技术联盟](https://www.bluetooth.com/)管理。在蓝牙中，设备使用低功耗 UHF 无线电波在 2.402 GHz 至 2.4835 GHz 的频段上进行通信。该频段在全球范围内无需许可，但受限于工业、科学和医疗用途。^(⁴¹) 在其最广泛使用的模式下，发射功率上限为 2.5 毫瓦，因此有效距离可达 10 米（33 英尺）。

注意

*蓝牙*这个名字指的是 10 世纪的丹麦国王哈拉尔·蓝牙。它由英特尔公司的吉姆·卡达奇于 1997 年提出，他是蓝牙 SIG 的创始人之一。蓝牙原本是一个占位名称，计划在商业化之前替换掉，但商标问题阻碍了这一计划。无论如何，在它被更改之前，这个名字已经传遍了整个行业。这里的教训是：务必谨慎选择你项目的代码名称。

蓝牙技术有两种类型：低功耗蓝牙（LE）和基础速率/增强速率蓝牙（BR/EDR）。两种类型都允许两个设备之间进行点对点连接。两者也都在同一频段上运行，但低功耗蓝牙是更受欢迎的选择。它运行所需的能量少得多，并且能够创建网状网络。低功耗蓝牙的数据速率可达 1 Mbit/s（蓝牙 4 及以下）或 2 Mbit/s（蓝牙 5），而 BR/EDR 最高可达 3 Mbit/s。

蓝牙使用一种称为跳频扩频的无线电技术。它基于数据包，并在 79 个预定义信道之一上传输每个数据包。每个信道的带宽为 1 MHz。低功耗蓝牙信道使用 2 MHz 的间隔，容纳 40 个信道。此外，蓝牙依赖于主/从架构。在 BR/EDR 中，一个主设备最多可与七个从设备通信，形成一个微微网：用蓝牙术语来说就是一个*自组织*网络。BLE 理论上允许一个微微网中有数百万个设备，但超过几百个从设备是不切实际的，因为所有从设备共享相同的无线电时间与主设备通信。微微网中的主设备提供一个时钟，其他设备将其作为数据包交换的基础。一旦微微网建立，其成员会同时跳变无线电频率以维持网络，并避免干扰附近可能存在的其他微微网。

蓝牙 BR/EDR 要求设备进行配对。配对过程在两个设备之间建立信任关系，然后它们可以通过加密连接交换数据。低功耗蓝牙的工作方式不同。配对是可选的，并非强制。蓝牙 BLE 设备可以通过广播通告数据包使自己被其他设备发现。这种广播依赖于三个独立的信道（频率）以减少干扰。通告设备在至少一个信道上发送一个通告数据包。它会以称为*通告间隔*的周期重复发送。每个通告间隔会增加最多十毫秒的延迟，以降低多次连续冲突的概率。延迟的实际持续时间是随机确定的。

蓝牙是物联网解决方案中短距离连接的流行选择。它甚至在低成本 MCU 中也可用，并得到所有主流实时操作系统的支持。但是，请记住，蓝牙不使用 TCP/IP，这减少了你可以在其上使用的协议范围。然而，可以通过利用 6LoWPAN 在蓝牙上建立 IPv6 连接。6LoWPAN 由 IETF 维护，并在 [RFC 4944](https://datatracker.ietf.org/doc/html/rfc4944) 中记录。它由 [RFC 6282](https://datatracker.ietf.org/doc/html/rfc6282)（头部压缩）和 [RFC 6775](https://datatracker.ietf.org/doc/html/rfc6775)（邻居发现优化）更新。蓝牙 BLE 上的 6LoWPAN 在 [RFC 7668](https://datatracker.ietf.org/doc/html/rfc7668) 中涵盖。

不利的一面是，蓝牙以其有据可查的隐私和安全问题而闻名。蓝牙安全支持身份验证和加密。然而，糟糕的实现很常见，并且该协议容易受到拒绝服务攻击、垃圾信息和后门攻击。过去，安全研究人员主要将精力集中在识别配对过程中的蓝牙漏洞上。对协议其他部分的加强审查导致了更多弱点的发现。例如，2020 年，普渡大学的研究人员发现了蓝牙 BLE 重新连接过程中的问题，该过程发生在设备移出范围后又重新上线时。许多实现在设备重新连接期间使身份验证成为可选项，并且在特定情况下可以绕过身份验证过程。^(⁴²) 你应该彻底评估和测试你将部署的任何蓝牙设备的安全特性。



### 蜂窝网络（LTE、5G）

无线宽带通信在全球大部分人口稠密地区已广泛普及。因此，蜂窝连接成为许多类型的物联网和边缘计算部署的热门选择。在撰写本文时，LTE 和 5G 是仅有的两个合理选项，因为全球大多数无线运营商已开始逐步关闭其 3G 网络。3G、LTE 和 5G 是由[第三代合作伙伴计划 (3GPP)](https://www.3gpp.org/) 管理的技术。5G 是 LTE 的后续演进技术。

长期演进 (LTE) 是一种基于先前 GSM/EDGE 和 UMTS/HSPA 标准的无线宽带标准。LTE 的频率和频段因国家而异；不过，可以采购支持多个频段的设备和无线电模块。这些频段的范围在 450 MHz 至 5 GHz 之间。LTE 标准仅支持全 IP 网络上的分组交换。因此，运营商必须设计出承载语音通话的解决方案，例如 *长期演进语音承载* (VoLTE) 标准。其优势在于 LTE 和 5G 网络开箱即支持 TCP/IP。请注意，不要将 LTE 与 LTE-M 混淆，后者是一种窄带无线技术，我将在后面介绍。

5G 可以在低频段、中频段或高频段毫米波 (mmWave) 频率范围内实现。低频段 5G 使用 600–900 MHz 的范围，这与 LTE 重叠。这带来了 30 到 250 Mbit/s 的下载速度，略优于 LTE。低频段 5G 基站的有效范围与 4G 基站相同。另一方面，中频段 5G 依赖于 1.7 至 4.7 GHz 范围内的微波。这转化为 100 至 900 Mbit/s 之间的速度。中频段 5G 基站的范围半径可达数公里。最后，高频段 5G 目前在 24 至 47 GHz 的频率范围内运行，尽管未来可能会使用更高的频率。毫米波 5G 理论上可以提供超过每秒千兆比特 (Gbit/s) 的速度，与最佳有线互联网连接相当。然而，如此高频率的波很容易被墙壁或窗户的建筑材料阻挡。因此，基站的有效范围要小得多，从而推高了部署成本。因此，毫米波 5G 在人口密集的城市区域或大量人群聚集的场所（如体育场馆）之外非常稀缺。

LTE 和 5G 连接常见于网关和边缘节点中，其内部通常通过 USB 或 PCIe 连接集成适当的无线电模块。与智能手机一样，无线电模块需要一张 SIM（用户身份模块或用户识别模块）卡才能正常工作。此类卡是设计用于安全存储国际移动用户识别码 (IMSI) 号码及其对应密钥的集成电路。两者都用于在无线宽带网络上识别和验证用户。近年来，eSIM（嵌入式 SIM）越来越受欢迎。eSIM 是永久表面贴装在设备内部且可编程的；甚至可以远程配置它们。这提高了设备的可靠性和安全性。

许多网关支持两个或更多 SIM 卡，从而能够在主网络出现问题时使用备用网络作为回退。LTE 和 5G 连接也可用作带外连接解决方案来管理设备；在这种情况下，它可能具有成本效益，因为通常根据带宽消耗计费。

### 以太网

自 20 世纪 80 年代以来，以太网一直是有线局域网 (LAN)、城域网 (MAN) 和广域网 (WAN) 中使用的主导标准。到 80 年代末，它已经取代了 ARCNET 和令牌环等竞争替代方案。[IEEE 标准协会](https://standards.ieee.org/) 维护着以太网标准的各个版本。

最初的 10BASE5 以太网使用连接到网络上每台机器的同轴电缆（共享介质），而较新的变体则使用双绞线和光纤链路以及交换机。目前最常用的形式是 10BASE-T、100BASE-TX 和 1000BASE-T。它们提供 10 Mbit/s、100 Mbit/s 和 1 Gbit/s 的速度。该标准还有一个更高性能的版本，即万兆以太网（10GE、10GbE 或 10 GigE），但由于其每端口成本仍然较高，其采用率比以前的版本更有限。鉴于此，2.5GBASE-T（2.5 Gbit/s）和 5GBASE-T（5 Gbit/s）标准应运而生，以提供比 1000BASE-T 更高的吞吐量，而无需承担与 10GE 相关的成本。

以太网电缆允许的最大长度取决于所使用的布线技术。100BASE-TX 和 1000BASE-T 依赖于双绞线电缆。它们都规定了 100 米（330 英尺）的最大电缆长度，并且在推出时需要 5 类电缆。此后，IEEE 弃用了 5 类电缆；新安装使用超五类线。光纤变体通常使用小型可插拔 (SFP) 热插拔网络接口模块。根据所选电缆类型、连接器和标准版本，最大电缆长度从 550 米（约 1800 英尺）到 160 公里（约 100 英里）不等。

以太网不仅限于数据传输。以太网供电 (PoE) 描述了在双绞线以太网布线中传输电力和数据的几种标准或专用系统。有几种广泛使用的技术用于通过以太网传输电力。其中三种自 2003 年以来已标准化为 IEEE 802.3。最初的 IEEE 802.3af-2003 PoE 标准为每个端口提供高达 15.4 W 的直流电源（最小 44 V 直流电和 350 mA）。由于功率耗散，设备只能获得 12.9 W。更新的 IEEE 802.3at-2009 PoE 标准（通常称为 PoE+ 或 PoE plus）提供高达 25.5 W 的功率。该标准禁止使用电缆中的所有四对线来传输电力。IEEE 将 802.3af-2003 和 802.3at-2009 纳入 IEEE 802.3-2012 出版物。最后，IEEE 802.3bt-2018 标准（也称为 PoE++ 或 4PPoE）进一步提升了 802.3at 的能力。该标准引入了两种额外的功率类型：高达 51 W 的输送功率（类型 3）和高达 71.3 W 的输送功率（类型 4）。每对线需要处理 600 mA（类型 3）或 960 mA（类型 4）。所有 IEEE PoE 变体都支持千兆以太网 (1000BASE-T)，并且 802.3bt 可以在 2.5GBASE-T、5GBASE-T 和 10GBASE-T 上工作。

如今，大多数台式计算机都支持千兆以太网，几乎所有的物联网网关和边缘节点也是如此。许多 MCU 也可以配备以太网连接。例如，[Arduino Ethernet](https://www.arduino.cc/en/Main/arduinoBoardEthernet) 将 ATmega328 与 WIZnet 的 W5100 嵌入式以太网控制器相结合，提供对 10/100 兆位双绞线以太网连接的支持。可以添加符合 IEEE 802.3af 标准的 PoE 模块为电路板供电。Raspberry PI 还推出了[一个 802.​3af PoE 模块](https://www.raspberrypi.com/products/poe-hat/)，可用于为 Raspberry Pi 4 Model B 和 Raspberry Pi 3 Model B+ 电路板供电。当然，您将用于支持 PoE 的设备的任何网络交换机都需要支持正确版本的 IEEE 802.3。或者，也可以使用标准以太网交换机并部署 PoE 注入器，为来自非 PoE 设备的以太网信号增加电力。

以太网是为许多物联网和边缘计算设备提供有线连接的成熟解决方案。它特别适用于部署在室内固定位置的设备。PoE 在数字建筑和互联工厂等环境中非常常见。



### 窄带

低功耗广域网（LPWAN）是一种面向电池供电的受限设备和传感器的无线网络。其设计旨在优先考虑覆盖范围而非吞吐量，以优化设备的电池寿命。大多数 LPWAN 每个信道的数据速率在 0.3 kbit/s 到 170 kbit/s 之间。你可以自行搭建 LPWAN，也可以使用第三方部署的服务和基础设施。后一种方法的优势在于，你无需在现场部署和维护网关。

该市场存在多种相互竞争的标准。我重点关注了四种特定方案：DASH7、LoRaWAN、LTE-M 和 NB-IoT。目前市场上所有窄带方案都具备远距离、低功耗和高性价比的特点。LPWAN 在乡村环境下的工作范围可达 10 公里（6 英里）以上；在人口密集的城市环境中，仍能达到数公里。在功耗方面，电池寿命超过十年也并非罕见。此外，LPWAN 依赖于能够控制基础设施成本的拓扑结构。它们还使用轻量级协议来降低硬件复杂性，从而减少设备成本。一些流行的方案依赖于专有技术，这往往会导致收发器成本更高。

#### DASH7

DASH7 联盟协议（D7A）是一种开源的无线窄带协议，工作在 433 MHz、868 MHz 和 915 MHz 的免许可频段。该协议由[DASH7 联盟](https://www.dash7-alliance.org/)管理，这是一个为促进 DASH7 协议规范进一步发展而成立的非营利性互惠公司。你可以从 DASH7 网站免费下载该规范。^(⁴³) Eclipse 基金会很荣幸成为 DASH7 联盟的成员。

D7A 源自 ISO/IEC 18000-7 标准，该标准定义了在免许可的 433 MHz ISM 频段内用于有源射频识别（RFID）的接口。D7A 不再符合 ISO/IEC 18000-7 标准。该协议规范可免费使用，无需任何专利或许可要求。D7A 的覆盖范围较短。在工业应用中，其最大距离约为 500 米（1640 英尺），尽管设备制造商宣称在视距条件下读取范围可超过一公里（0.62 英里）。该协议提供典型的 1 秒延迟，同时平均功耗仅为 30 uA。根据所选无线电调制方式，有三种数据速率等级。DASH7 低速率提供 9.6 kbit/s 的等效比特率，DASH7 正常速率达到 56 kbit/s，而 DASH7 高速率则达到 167 kbit/s。

[GitHub 上的 Sub-IoT 项目](https://github.com/Sub-IoT/Sub-IoT-Stack)托管了 D7A 的一个开源实现。它采用 Apache 许可证 2.0 版发布。

#### LoRaWAN

LoRaWAN 涉及两种不同的技术：LoRa 和 LoRaWAN。

LoRa（源自“Long Range”）是一种专有的无线电调制技术。它由位于法国格勒诺布尔的 Cycleo 公司开发并获得专利，后来被 Semtech 收购。LoRa 芯片组由 Semtech 独家提供。LoRa 使用以下免许可的亚千兆赫兹无线电频段：

*   亚洲：AS923（915–928 MHz）
*   欧洲：EU868（863–870/873 MHz）
*   印度：IN865（865–867 MHz）
*   北美：US915（902–928 MHz）
*   南美：AU915/AS923-1（915–928 MHz）
*   全球：2.4 GHz

LoRa 可提供 0.3 kbit/s 至 27 kbit/s 的数据速率。在城市地区，有效范围可达 5 公里（3.1 英里），在更容易实现无障碍视距的乡村地区，有效范围可超过 15 公里（9.3 英里）。

另一方面，LoRaWAN 定义了软件通信协议和系统架构。它由成立于 2015 年的非营利组织 LoRa 联盟开发和维护。LoRaWAN 管理所有设备的频率、数据速率和功率。设备依赖于异常报告模式：它们在需要时异步传输数据。这些传输由现场部署的网关转发到中央服务器。LoRaWAN 连接是双向的，并且该协议支持多播寻址组，以便更高效地向设备批量分发消息。这对于分发空中软件更新尤其有用。

有多个开源的 LoRaWAN 软件栈可用。其中之一由[ChirpStack 项目](https://www.chirpstack.io/)提供，该项目包含网关桥接器、网络服务器和应用服务器等组件。所有组件均采用 MIT 许可证发布。

LoRaWAN 的远距离和广泛的行业支持使其成为大规模部署对数据速率要求不高的无线传感器的有吸引力的技术。智慧城市和智慧农业是两个很好的潜在用例。然而，无线电调制的专有性质是一个令人担忧的问题。

#### LTE-M 和 NB-IoT

LTE-M（或 LTE-MTC，代表“机器类型通信”）和 NB-IoT（窄带物联网）是 3GPP 为应对机器对机器和物联网用例而开发的两种 LPWAN 无线电技术标准。

LTE-M 依赖于现有的 LTE 网络，但并非所有有 LTE 连接的地方都必然可用。与 LTE 调制解调器相比，LTE-M 型号成本更低，电池寿命可达数年。与 LTE 网络相比，LTE-M 网络还提供更大的覆盖范围和更好的室内穿透能力。LTE-M 分为两类：Cat-M1 和 Cat-M2。Cat-M1 的上行和下行数据速率均为 1 Mbit/s；而 Cat-M2 的下行数据速率为 4 Mbit/s，上行数据速率为 7 Mbit/s。延迟平均在 50 到 100 毫秒之间。由于依赖于蜂窝基站，LTE-M 支持移动性用例，并支持 VoLTE 语音服务。

与 LTE-M 类似，NB-IoT 使用 LTE 标准的一个子集，但将带宽限制在单个 200 kHz 的窄带上。该技术专注于实现高密度连接可能性的室内空间的经济高效覆盖。对于广泛的用例，可以支持超过十年的电池寿命。NB-IoT 分为两类。Cat-NB1 是引入的第一个版本，其下行数据速率高达 26 kbit/s，上行数据速率高达 62 kbit/s。延迟范围为 1.6 到 10 秒。最新版本 Cat-NB2 将下行数据速率提升至 127 kbit/s，上行数据速率提升至 150 kbit/s。NB-IoT 还支持无线资源控制（RRC）连接重建，允许设备将其连接从一个小区转移到另一个小区。这提高了该技术对移动性用例（如行人、自行车和车辆监控）的适用性。

与 NB-IoT 相比，LTE-M 提供更好的数据速率和更低的延迟。因此，对于实时用例来说，它是更好的选择。其对语音通信的支持增强了其灵活性。此外，由于 LTE-M 是 LTE 的扩展，它支持漫游，而 NB-IoT 不支持。对于任何涉及连接位于仓库、地下位置和商业建筑中的设备的用例，NB-IoT 是更优越的技术。如果你只需要以低数据速率传输传感器数据，它也是一个很好的选择。

LTE-M 和 NB-IoT 都涉及 SIM 卡。然而，在特定位置有 LTE 或 5G 连接可用，并不能保证 LTE-M 和 NB-IoT 也可用。有些国家/地区只有其中一种方案可用，而另一种则没有。[GSM 协会](https://www.gsma.com)（GSMA）网站提供了一个有用的地图，展示了全球范围内 LTE-M 和 NB-IoT 服务的商业可用性。你可以在此处找到它：[`www.gsma.com/iot/deployment-map/`](http://www.gsma.com/iot/deployment-map/)



### Wi-Fi

Wi-Fi 是一组基于 IEEE 802.11 标准的无线网络协议，旨在与以太网无缝集成。Wi-Fi 的演进由非营利组织 [Wi-Fi 联盟](https://www.wi-fi.org/) 推动。截至撰写本文时，Wi-Fi 已取得显著的市场普及，全球使用中的设备超过 180 亿台，预计 2022 年出货量将达到 44 亿台。^(⁴⁴)

802.11 协议的第一个版本于 1997 年推出。此后，该标准陆续推出了多个后续版本。每个特定版本的标准都规定了所使用的无线电频段、最大覆盖范围以及可实现的最大速度。Wi-Fi 最常使用 2.4 GHz 和 5 GHz 无线电频段；这些频段被细分为多个信道。表 9-1 总结了每个 Wi-Fi 版本的主要特性。

Wi-Fi 部署的最大工作范围取决于多个因素，包括频段、无线电输出功率、所用接收器的灵敏度、调制技术以及所用天线的增益和类型。在距离较远以及墙壁等障碍物导致信号吸收更强的情况下，数据速率通常会较低。较低的无线电频率会带来更好的覆盖范围，但吞吐量较低。建筑材料对 5 GHz 信号的吸收程度高于 2.4 GHz 信号。然而，较高的频率提供了更多的信道，有助于减少多个网络竞争频谱的环境中的干扰。通过部署多个接入点或使用范围扩展器和信号放大器，可以增加 Wi-Fi 网络的覆盖范围。最近的 Wi-Fi 代际支持部署网状网络，例如，可改善建筑物内的覆盖范围。

许多单板计算机和 MCU 都支持 Wi-Fi。然而，虽然 Wi-Fi 的有效范围比蓝牙更大，但其功耗也高得多。为了解决这个问题，IEEE 引入了 802.11ah 标准，Wi-Fi 联盟将其称为 Wi-Fi HaLow。802.11ah 使用亚千兆赫兹 ISM 频谱。频段因地区而异，位于 755 至 928 MHz 之间。802.11ah 标准定义了多种调制方案和编码速率；可能的数据速率范围很广，从 150 kbit/s 到 86.7 Mbit/s。典型范围超过 1 公里（0.6 英里），并且该协议集成了多种功能以延长电池寿命。

对于物联网和边缘计算部署来说，Wi-Fi 无疑是一个可行的选择，尤其是当您的设备和传感器不依赖电池供电时。然而，其在覆盖范围上的局限性意味着，对于大面积户外部署，存在更好的选择。

### Z-Wave

Z-Wave 是一种主要用于智能家居部署的无线通信协议。它与蓝牙 BLE 类似，本质上是一个使用低能耗传输的网状网络。Z-Wave 联盟成立于 2005 年，负责管理该标准的演进。然而，Z-Wave 收发器芯片仅由该技术的当前所有者 Silicon Labs 及其被许可方独家提供。

Z-Wave 旨在以低延迟传输小型数据包，数据速率最高可达 100 kbit/s。网络上两个节点之间的最大距离为 30 到 40 米（98 到 131 英尺）。数据包可以在节点之间最多跳转四次，从而增加潜在的覆盖区域。Z-Wave 使用带有曼彻斯特编码的频移键控（FSK）调制。与 DASH7 和 802.11ah 一样，它使用 ISM 频段。Z-Wave 在欧洲以 868.42 MHz 传输，在北美以 908.42 MHz 传输；在其他地区，则根据当地法规使用其他频率。

一个 Z-Wave 网络至少需要包含一个主控制器和至少一个设备。控制器通常采用墙壁开关和手持控制器的形式，但软件应用程序也可以扮演这个角色。一个网络最多可包含 232 个设备，并且可以将多个网络桥接在一起。设备需要先与网络配对（或添加），然后才能通过协议进行控制。鉴于标准中定义的省电特性，大多数设备的预期电池寿命为数年。

Z-Wave 提供了一个安全层，可确保消息的完整性和机密性。它还实现了确保数据新鲜度的机制，这意味着基础设施将只处理最近发送的消息。Z-Wave 依赖于具有 128 位密钥长度的 AES 对称分组密码算法。过去，制造商可以生产不安全的设备。即使在今天，为了向后兼容，Z-Wave 网络也可以包含安全和不安全的节点。然而，安全性现在已是强制要求，因此您可以通过仅使用较新的设备来减轻担忧。

Z-Wave 是家庭自动化甚至智能建筑项目的一个有趣选择。鉴于其频段，它提供了比蓝牙更大的覆盖范围，并且比某些替代方案受到的干扰更少。然而，该技术在部分方面的专有控制令人担忧。



### Zigbee

Zigbee 是一种基于 IEEE 802.15.4 标准的规范，定义了一套高功率通信协议栈。Zigbee 由[连接标准联盟](https://csa-iot.org/)（前身为 Zigbee 联盟）管理。该组织发布应用配置文件，供应商可借此创建可互操作的产品。IEEE 802.15.4 标准与 Zigbee 之间的关系，类似于 IEEE 802.11 与 Wi-Fi 之间的关系。

Zigbee 在全球大多数地区使用 2.4 GHz 频段。特定设备在中国使用 784 MHz，在欧洲使用 868 MHz，在美国和澳大利亚使用 915 MHz。不过，大多数可用设备仍坚持使用 2.4 GHz 频段。数据传输速率从 20 kbit/s（868 MHz）到 250 kbit/s（2.4 GHz）不等。

注意

Zigbee 这个名称指的是蜜蜂返回蜂巢时，为告知同伴花丛、水源或新巢穴位置的方向和距离而跳的舞蹈。

Zigbee 专注于低功耗，因此在理想条件下，有效通信距离被限制在 10 到 100 米之间。与 Z-Wave 类似，Zigbee 设备可以形成网状网络，将数据传播到目的地。Zigbee 标准定义了三种设备类别：

*   **协调器 (ZC)**：网络的根节点。每个 Zigbee 网络恰好有一个协调器。协调器可以建立通往其他网络的桥接。

*   **路由器 (ZR)**：运行应用功能的节点。路由器可以将数据传播到其他设备。

*   **终端设备 (ZED)**：只能与路由器或协调器通信的叶节点。此类设备通常长时间处于休眠模式，从而实现长电池寿命。

只有连接标准联盟的成员才能以 Zigbee 名称销售产品。

与任何无线连接技术一样，Zigbee 也存在安全漏洞。该协议假设用于运行和保护网络的密钥被安全存储，并且不会以未加密方式传输。当然，安全实现存在缺陷或倾向于可用性和便利性而忽视安全性的设备，可能会泄露其密钥。此外，攻击者可能拦截数据包并将其重放给毫无戒心的设备，诱骗它们执行非预期的操作。一种可能的缓解措施是通过带外信道分发密钥并定期轮换。

与 Z-Wave 一样，Zigbee 是智能建筑项目和家庭自动化的不错选择。鉴于其通信距离和灵活性，组织也使用它来跟踪资产，或通过将智能电表集成到智能电网中来监控消费者能源消耗。

## 网络设计考量

我介绍的每种网络技术都具有影响您设计和部署网络方式的特性。然而，一些相关的考量因素独立于技术本身而普遍适用。我现在将对其进行回顾。

### 协议选择

我在本书第一部分介绍了几种协议。有些需要 TCP/IP，有些满足于 UDP，而另一些可以利用非 IP 传输。当然，您打算使用的协议将影响您为支持现场设备而部署的网络类型。所选网络的带宽成本和预期可靠性也会影响协议的选择，反之亦然。

许多现场网络不会直接连接到互联网，无论这种连接是否可能。在这种情况下，物联网网关将本地网络与其余基础设施桥接起来。这样的拓扑结构使得可以在本地使用（例如）蓝牙或 Z-Wave 收集传感器数据，并通过蜂窝 LTE 或 5G 链路将其发送到云 MQTT 代理。在这种情况下，网关可以预处理数据以保护机密性或节省带宽。

### 机密性

保护数据的机密性，无论是静态数据（存储在设备上）还是动态数据（传输过程中），都是任何物联网解决方案的关键关注点。自然，我介绍的几种网络技术都提供了一定程度的加密。但其他技术，如以太网，则没有。因此，规划机密性至关重要。

我能给出的最重要建议是：不要实现您自己的自定义安全层！即使您是一位经验丰富的开发人员，编写与安全相关的代码也是一项艰巨的任务。既定安全标准的价值在于，安全研究人员和大型组织会关注它们，确保新威胁被发现、讨论并最终得到解决。如果您构建自己的安全层，那么监视和应对新威胁的责任将落在您身上。

如果您采纳我的建议，那么您应该怎么做？首先，如果您使用依赖 TCP/IP 或 UDP 的协议，则强烈建议部署 TLS 或 DTLS。您应坚持使用协议和操作系统支持的最新版本标准。更具体地说，这意味着尽可能使用 TLS/DTLS 1.3，并且绝不使用任何早于 TLS/DTLS 1.2 的版本。另一种策略是执行有效载荷加密，尽管这需要您定期确定所使用的特定加密方法是否仍然安全。

注意

从版本 1.2 开始，TLS 和 DTLS 的版本号已调整为相同。

使用最新且受支持的 TLS/DTLS 版本只是第一步。您还需要在网络上的每个设备以及云或数据中心的后端资源上配置适当的密码套件。您选择的密码套件将影响电池寿命和带宽使用。通常，提供更高级别安全性的套件由于更复杂的加密计算而具有更高的计算要求。表 9-2 提供了当前使用的三种最合适认证方法的主要特性概览。

利用 TLS/DTLS 中的附加功能也是明智之举。例如，利用临时 Diffie-Hellman (DHE) 密钥交换模式，可以在密码套件支持时实现*前向保密*。当密钥交换使用临时 Diffie-Hellman 算法时，每次连接都会生成一个临时密钥。这些密钥不会被重复使用，这意味着如果后端服务的长期私钥被泄露，过去的交换仍然是保密的。另一个有用的功能是会话恢复。TLS 和 DTLS 握手需要大量时间和计算资源。鉴于此，会话恢复提供了一种机制，可以在多个连接之间重用最近协商的密钥。由此产生的简化握手更短且资源消耗更少。



大多数加密解决方案（不仅仅是 TLS 和 DTLS）所依赖的密钥和证书都有**定义的生命周期**。你需要在创建时指定它们的过期日期。这意味着任何依赖加密的系统都需要处理凭证（如密钥和证书）的轮换（即定期更换）。你可能会试图通过将过期日期设置得非常遥远来避免这个问题。**请不要这样做**，因为这会在你的基础设施中造成巨大的安全漏洞，因为随着时间的推移，密钥和证书更有可能被泄露。定期更换凭证将缩短攻击者的可利用窗口期。当然，执行轮换并不愉快，而且即使有自动化手段，也可能非常耗时。此外，如果你希望将密钥对设备本身保密，轮换可能更具挑战性，因为许多受限设备缺乏良好的随机源，而这是生成强加密密钥对的要求。然而，定期更换解决方案中使用的密钥和证书将使其更加安全。

### 弹性

物联网和边缘计算解决方案应假设网络不可靠，并且有时会不可用。这并不意味着你无法采取任何措施来提高其弹性。让我们来看一些最佳实践。

当设备部署在现实世界中时，它们更容易受到网络拓扑变化的影响。我们探讨过的一些连接选项并非基于 TCP/IP，而是通过它们能检测到的相邻节点来路由流量。因此，它们会自动适应附近节点的添加或移除。在这种情况下，构建弹性网络意味着要仔细定位节点，以确保少数节点的丢失不会影响其他节点的连接。对于基于 TCP/IP 的网络，关键在于依赖域名系统（DNS）而非 IP 地址。这确保了服务的可用性，不受实际托管它们的服务器的影响。当然，如果你只将 DNS 查询结果缓存很短的时间，这种策略效果会更好。这将确保设备能更快地适应网络拓扑或服务器可用性的变化。

除非你使用的是需要最大化电池寿命的受限设备，否则利用协议中的*保活*或*心跳*功能将带来更具弹性的网络连接。例如，如果没有这些功能，设备可能会在未检测到的情况下失去与运营商网络的连接，导致其 TCP/IP 会话失效。此外，尽管会话 TLS/DTLS 恢复可以缩短握手时间，但通过保持连接活跃可以避免这种恢复。如果你需要让设备休眠以降低功耗，仍然可以通过使用特定于协议的功能来减轻设备负担。例如，MQTT 持久会话使客户端能够重新连接到代理，而无需重新建立其先前的订阅；MQTT 5 中的会话过期机制确保未及时重新连接的设备的会话被清除。这些功能虽然不能消除重新建立底层 TCP/IP 连接的需求，但确实简化了流程。

我建议你对重新连接尝试采用**指数退避**策略。你将使用一种算法，逐步增加连接尝试之间的等待时间，而不是以固定间隔重试。这种技术将避免用大量请求淹没后端服务，确保服务恢复后能更平稳地恢复。

最后，另一个有用的策略是对连接和请求使用**激进的超时**设置。除非网络延迟高度不可预测，否则设备等待一次连接尝试超过几秒钟是没有意义的。这种“快速失败”策略也有助于延长电池寿命，尤其是在与指数退避结合使用时。

### 冗余连接

对于关键任务用例，网络中断可能导致重大物质损失或人员伤亡，依赖单一网络连接将构成单点故障。因此，许多网关和其他物联网设备提供冗余网络连接。对于依赖蜂窝网络（如 LTE、5G、LTE-M 或 NB-IoT）的技术，通常通过在设备中添加来自不同网络提供商的第二张备用 SIM 卡来实现。由于这些提供商通常采用订阅模式，费用由套餐、实际数据消耗或两者共同决定，因此这种方式通常具有成本效益。冗余连接可能流量很小，从而有助于控制成本。

如果你的连接要求非常严格，利用基于完全不同技术和频段的网络是一种可靠但成本更高的策略。当然，你需要考虑有效范围、数据传输速率、延迟和软件支持方面的差异。

### 带外管理

在许多企业场景中，需要最大化部署在偏远或难以到达位置的设备的正常运行时间。由于派遣现场支持人员既昂贵又耗时，通常更便宜的做法是依赖独立的带外网络连接来管理和排除设备故障。此类连接应使用与处理设备常规流量不同的技术。当然，为了控制成本，仅在物联网网关或边缘节点级别部署带外网络是合理的。

在某些情况下，部署专用设备通过其串行（UART）或以太网端口连接到本地管理的设备是合理的。此类设备可以在网络中断、设备关机、处于睡眠模式、休眠状态或无法访问时提供连接。

从安全角度来看，将管理流量隔离在专用网络上可以使解决方案在遭受攻击时更具弹性。由于带外网络的使用率低于主连接，因此更容易发现意外或可疑活动。

## 连接性与受限设备

大多数情况下，对网络连接技术的支持取决于你使用的操作系统或实时操作系统提供的网络协议栈以及可用的驱动程序。在 Linux 的情况下，事情很简单；目前大多数可用的无线电模块在内核树中都有直接支持，或者附带相关驱动程序。在实时操作系统的情况下，情况则不太稳定。现在让我们看看如何在 Zephyr RTOS 上使用 Wi-Fi 和蓝牙连接。



### Wi-Fi

在 Zephyr 代码仓库中，只有一个专注于 Wi-Fi 的通用示例应用程序。其描述位于：[`https://docs.zephyrproject.org/latest/samples/net/wifi/README.html`](https://docs.zephyrproject.org/latest/samples/net/wifi/README.html)。

该应用程序可在任何支持 Zephyr Wi-Fi 的开发板上运行。它不包含实际代码，而是展示了如何为 Wi-Fi 配置应用程序，并启用 `net_shell` 和 `net_wifi_shell` 模块。这些模块提供了一个命令行界面，可用于运行各种网络相关命令。例如，`net_shell` 实现了对 `ping` 和 `route` 等命令的支持，而 `net_wifi_shell` 则提供了扫描可用 Wi-Fi 网络、连接或断开连接的命令。

我复现了 `net_wifi_shell` 中扫描命令的输出：

```
shell> wifi scan
Scan requested
shell>
Num  | SSID                       (len) | Chan | RSSI | Sec
1    | YoRHa no.2, Type B         8     | 1    | -93  | WPA/WPA2
2    | YoRHa no.9, Type S         8     | 6    | -89  | WPA/WPA2
3    | Nausicaä                   13    | 11   | -73  | WPA/WPA2
4    | Forger Family              4     | 1    | -26  | WPA/WPA2

```

当然，考虑到 `net_shell` 和 `net_wifi_shell` 模块所消耗的资源，它们不应在生产应用程序中使用。

Zephyr 仓库中包含一个专注于 Wi-Fi 的优秀板级示例应用程序，名为 [Espressif ESP32 Wi-Fi Station](https://docs.zephyrproject.org/latest/samples/boards/esp32/wifi_station/README.html)。顾名思义，它支持乐鑫的 ESP32、ESP32-S2 和 ESP32-C3 系统级芯片（SOC）。该代码位于 Zephyr GitHub 代码仓库的以下位置：

[`https://github.com/zephyrproject-rtos/zephyr/blob/main/samples/boards/esp32/wifi_station`](https://github.com/zephyrproject-rtos/zephyr/blob/main/samples/boards/esp32/wifi_station)

该应用程序很简单；它将使用配置文件中静态指定的 SSID 和密码连接到网络。以下是 [来自主方法](https://github.com/zephyrproject-rtos/zephyr/blob/main/samples/boards/esp32/wifi_station/src/main.c) 的代码片段，展示了如何连接：

```
wifi_config_t wifi_config = {
.sta = {
.ssid = CONFIG_ESP32_WIFI_SSID,
.password = CONFIG_ESP32_WIFI_PASSWORD,
},
};
esp_err_t ret = esp_wifi_set_mode(WIFI_MODE_STA);
ret |= esp_wifi_set_config(ESP_IF_WIFI_STA, &wifi_config);
ret |= esp_wifi_connect();
```

由于受限设备扮演的是 Wi-Fi 客户端（而非接入点）的角色，因此传递给 `esp_wifi_set_mode` 的参数是 `WIFI_MODE_STA`。

该应用程序依赖于 ESP32 Wi-Fi 驱动程序，该驱动程序位于 Zephyr 仓库中的另一个位置：

[`https://github.com/zephyrproject-rtos/zephyr/blob/main/drivers/wifi/esp32/src/esp_wifi_drv.c`](https://github.com/zephyrproject-rtos/zephyr/blob/main/drivers/wifi/esp32/src/esp_wifi_drv.c)

该驱动程序又利用了 [Espressif IoT Development Framework](https://github.com/zephyrproject-rtos/hal_espressif)，该框架托管在 Zephyr 项目 GitHub 组织下的另一个仓库中。该框架支持相关 SOC 中的各种功能和外设。您可以在此处访问稳定版本的文档：[`https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/index.html`](https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/index.html)。

### 蓝牙

Zephyr 仓库中有大量蓝牙示例。它们的根目录位于：

[`https://github.com/zephyrproject-rtos/zephyr/blob/main/samples/bluetooth`](https://github.com/zephyrproject-rtos/zephyr/blob/main/samples/bluetooth)

Mesh 网络是蓝牙 BLE 技术的一项重要功能。有三个示例专注于它。在此，我将重点介绍那个简称为 `mesh` 的示例。大部分代码包含在 [`main.c`](https://github.com/zephyrproject-rtos/zephyr/blob/main/samples/bluetooth/mesh/src/main.c) 文件中，尽管针对 micro:bit 有一个有趣的定制，它会使用该开发板的 LED 显示屏。

要将蓝牙设备添加到 mesh 网络中，需要对其进行配置。配置是一个配置和安全过程。它由*配置器*管理，配置器可以是另一个设备或软件应用程序。配置器发现附近的蓝牙设备，并邀请它们加入网络。如果设备接受，则配置器和设备会交换公钥。然后每个设备生成一个会话密钥，该密钥将保护配置数据交换的安全。这些数据包括网络密钥、设备密钥、一个称为 IV 索引的安全参数（由网络上的所有设备共享）以及设备的单播地址。一旦交换结束，该设备就成为网络中的一个节点。

蓝牙 mesh 规范集成了*模型*的概念以促进互操作性。[蓝牙 mesh 术语表](https://www.bluetooth.com/bluetooth-technology/topology-options/le-mesh/mesh-glossary/%2523model)指出，一个模型

> *[…] 定义了一组状态、状态转换、状态绑定、消息和其他相关行为。节点内的元素必须支持一个或多个模型，并且正是这些模型定义了元素所具有的功能。蓝牙 SIG 定义了许多模型，其中许多被有意定位为“通用”模型，在广泛的设备类型中具有潜在用途。*

注意

有关蓝牙 mesh 模型的更多信息，您应该阅读 Martin Woolley 撰写的这份优秀白皮书：[Bluetooth Mesh Models – A Technical Overview](https://www.bluetooth.com/wp-content/uploads/2019/04/1903_Mesh-Models-Overview_FINAL.pdf)。

`mesh` Zephyr 示例应用程序定义了几个简单的模型。在带有 LED 的开发板上，一个通用的 `OnOff` 服务器模型通过 mesh 网络控制开发板第一个 LED 的状态。在配备按钮的开发板上，一个通用的 `OnOff` 客户端模型在按下按钮时向所有节点发送 `OnOff` 消息。您可以以标准方式配置运行该应用程序的设备，或者按下一个按钮来执行自我配置。标准配置器需要提供一个应用程序密钥并将其绑定到两个模型。另一方面，自我配置会将一个虚拟的应用程序密钥绑定到模型，并为设备使用一个随机的单播地址。

一旦您配置了设备，按下按钮将在 mesh 网络中传播 OnOff 消息，在接收到消息时打开或关闭 LED。

现在让我们深入代码。`main` 函数注册了按钮按下的处理程序并初始化了蓝牙子系统。初始化通过调用 `bt_enable` 执行。唯一的参数是对名为 `bt_ready` 的回调函数的引用，如下所示：

```
err = bt_enable(bt_ready);
if (err) {
printk("Bluetooth init failed (err %d)\n", err);
}
```

`bt_ready` 函数中的关键行是这一行：

```
err = bt_mesh_init(&prov, &comp);
```

这两个参数分别配置配置过程和模型。配置配置包含在下面复现的结构体中：

```
static const struct bt_mesh_prov prov = {
.uuid = dev_uuid,
.output_size = 4,
.output_actions = BT_MESH_DISPLAY_NUMBER,
.output_number = output_number,
.complete = prov_complete,
.reset = prov_reset,
};
```



对于模型来说，情况就没那么直接了。这里的 `comp` 结构体引用了另一个名为 `elements` 的结构体。

```
static const struct bt_mesh_comp comp = {
.cid = BT_COMP_ID_LF,
.elem = elements,
.elem_count = ARRAY_SIZE(elements),
};
```

而 `elements` 又整合了另一个名为 `models` 的结构体。这反映出，在此例中，应用程序声明了一个单一元素来容纳其所有模型。

```
static struct bt_mesh_elem elements[] = {
BT_MESH_ELEM(0, models, BT_MESH_MODEL_NONE),
};
```

除了应用程序所需的两个自定义模型外，`models` 结构体还使用了用于配置和健康状态的标准模型。声明如下：

```
static struct bt_mesh_model models[] = {
BT_MESH_MODEL_CFG_SRV,
BT_MESH_MODEL_HEALTH_SRV(&health_srv, &health_pub),
BT_MESH_MODEL(BT_MESH_MODEL_ID_GEN_ONOFF_SRV, gen_onoff_srv_op, NULL,
NULL),
BT_MESH_MODEL(BT_MESH_MODEL_ID_GEN_ONOFF_CLI, gen_onoff_cli_op, NULL,
NULL),
};
```

如你所见，两个自定义模型都附带了一个可能的操作列表。以下是其中之一：

```
static const struct bt_mesh_model_op gen_onoff_srv_op[] = {
{ OP_ONOFF_GET,       BT_MESH_LEN_EXACT(0), gen_onoff_get },
{ OP_ONOFF_SET,       BT_MESH_LEN_MIN(2),   gen_onoff_set },
{ OP_ONOFF_SET_UNACK, BT_MESH_LEN_MIN(2),   gen_onoff_set_unack },
BT_MESH_MODEL_OP_END,
};
```

每个操作都与特定的回调函数相关联。例如，`gen_onoff_get` 会调用 `onoff_status_send`。

```
static int onoff_status_send(struct bt_mesh_model *model,
struct bt_mesh_msg_ctx *ctx)
{
uint32_t remaining;
BT_MESH_MODEL_BUF_DEFINE(buf, OP_ONOFF_STATUS, 3);
bt_mesh_model_msg_init(&buf, OP_ONOFF_STATUS);
remaining = k_ticks_to_ms_floor32(
k_work_delayable_remaining_get(&onoff.work)) +
onoff.transition_time;
...
if (remaining) {
net_buf_simple_add_u8(&buf, !onoff.val);
net_buf_simple_add_u8(&buf, onoff.val);
net_buf_simple_add_u8(&buf, model_time_encode(remaining));
} else {
net_buf_simple_add_u8(&buf, onoff.val);
}
return bt_mesh_model_send(model, ctx, &buf, NULL, NULL);
}
```

如你所见，该函数中的大部分代码用于构建有效载荷，然后通过调用 `bt_mesh_model_send` 进行传输。

与 LwM2M 或 MQTT 等协议相比，直接使用蓝牙网络编程需要编写更多代码。通过利用 6LoWPAN，可以在蓝牙连接上使用此类协议，从而允许 IPv6 数据包在基于 IEEE 802.15.4 的网络上传输。然而，这可能会比蓝牙原生协议消耗更多电量。
