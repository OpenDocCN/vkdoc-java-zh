# 在多云部署中使用 MicroProfile OpenTracing 时面临的挑战

在多云环境中进行分布式追踪可能颇具挑战性。我们希望实现与单一云环境相同的目标，即可视化与请求关联的端到端追踪，该请求会穿越每个云内部及跨云的各项服务，但在处理不同的上下文传播格式以及各云以不同格式存储追踪数据时，可能会遇到复杂情况。

第一个挑战是确保追踪能够跨越不同的云环境持续进行。这之所以成为问题，是因为在撰写本书时，尚未有一种广泛采用或标准化的追踪上下文格式。通常，每个追踪系统使用不同的标头和格式来传播追踪上下文。例如，Zipkin 使用 B3 传播，Jaeger 使用 `ber-trace-id` 标头，Amazon X-Ray 使用 `X-Amzn-Trace-Id`，而 Google 使用 `X-Cloud-Trace-Context`。因此，如果请求需要在异构追踪系统间进行追踪，那么每次离开或进入不同环境时，都必须转换追踪上下文。这通常可以通过配置一个带有自定义注入器或提取器实现的追踪器来完成。然而，这目前超出了 MicroProfile OpenTracing 项目的范围。未来，追踪上下文格式可能会在 W3C 追踪上下文项目（[`www.w3.org/TR/trace-context/`](https://www.w3.org/TR/trace-context/)）下实现标准化。

第二个挑战，即使在同构的追踪环境中，也是可视化来自多云环境的追踪数据。这可能成为问题，因为每个云中的追踪数据可能存储在不同的数据库中，或采用不同的格式。这可以通过将数据复制到单一统一存储，或在系统之间按需发送缺失的追踪数据并进行适当的数据格式调整来克服。
