# MicroProfile OpenTracing

在现代微服务世界中，单个请求可能穿越运行在不同机器、数据中心甚至地理区域上的多个进程。

此类系统的可观测性是一项具有挑战性的任务，但如果做得好，它允许我们*讲述*每个单独请求的故事，而不是从指标和日志等信号中推导出的系统整体状态。在本章中，我们将向您介绍分布式追踪，并解释 OpenTracing 及其在 MicroProfile OpenTracing 1.3 中的集成。

在上一节中，我们了解了指标以及它们如何观测应用程序或每个单独的组件。这些信息无疑非常有价值，并提供了系统的宏观视图，但同时，它对于穿越多个组件的每个单独请求几乎没有任何说明。分布式追踪展示了请求端到端发生情况的微观视图，以便我们可以追溯性地理解应用程序中每个单独组件的行为。

分布式追踪是基于动作的；换句话说，它记录系统中与某个动作相关的所有内容。例如，它捕获请求的详细信息以及所有因果相关的活动。我们不会详细讨论这种追踪的工作原理，但简而言之，我们可以说明以下几点：

*   追踪基础设施将上下文元数据附加到每个请求，通常是一组唯一的 ID——`traceId`、`spanId` 和 `parentId`。
*   检测层记录性能分析数据，并在进程内部和进程之间传播元数据。
*   捕获的性能分析数据包含元数据和对先前事件的因果引用。

基于捕获的数据，分布式追踪系统通常提供以下功能：

*   根因分析
*   延迟优化——关键路径分析
*   分布式上下文传播——行李
*   上下文日志记录
*   服务依赖关系分析
