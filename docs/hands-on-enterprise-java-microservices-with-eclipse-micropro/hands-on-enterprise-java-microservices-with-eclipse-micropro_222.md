# 第 10 章

1.  Eclipse MicroProfile 提供了使用企业 Java 开发微服务的最佳方法之一。反过来，使用容器作为部署单元的微服务，提供了在云中和本地开发高度分布式系统（即云原生应用程序）的最佳方法之一。因此，基于 MicroProfile 的微服务促进了云原生应用程序的开发。
2.  云原生应用程序开发有两个互补的方面或组成部分：应用服务和基础设施服务。应用服务加速了云原生应用程序业务逻辑的开发，而基础设施服务则加速了其交付和部署。这两个方面是互补的，并且是云原生应用程序开发不可或缺的组成部分。
3.  云原生应用程序开发是一种构建和运行应用程序的方法，它充分利用了基于四个关键原则的云计算模型：a) 基于服务的架构（迷你服务、微服务、SOA 服务等）；b) 用于服务间通信的 API 驱动方法；c) 基于容器的底层基础设施；以及 d) DevOps 流程。架构和通信方面与云原生应用程序的开发关注点相关，而基础设施和流程方面则与其交付/部署相关。Eclipse MicroProfile 与这些原则相关，因为它支持实现微服务，这些微服务可以成为使用容器作为其底层基础设施的架构的一部分，在该架构中，微服务使用其 API 相互通信，并使用 DevOps 流程进行开发。
4.  这就是 Eclipse MicroProfile 如何为引导你走向云原生应用程序的八个步骤中的每一步做出贡献：
    1.  发展 DevOps 文化和实践：“通过拥抱 DevOps 的原则和文化价值观，并围绕这些价值观组织你的团队，来利用新技术、更快速的方法和更紧密的协作。”尽管这是一个组织和流程相关的步骤，但 Eclipse MicroProfile 作为微服务的规范，非常适合这种文化和流程的适应，因为微服务由于其特性，紧密支持 DevOps 流程。
    2.  使用快速单体加速现有应用程序：“通过迁移到现代的、基于容器的平台来加速现有应用程序——并将单体应用程序拆分为微服务或迷你服务以获得额外的效率提升。”在将单体应用拆分为微服务时，Eclipse MicroProfile 可以提供很大帮助。当你在单体应用中识别出有界上下文时，考虑使用 Eclipse MicroProfile 来实现每个实现该有界上下文逻辑的微服务。



*   3.  使用应用服务加速开发：“通过可重用性加速软件开发。云原生应用服务是开箱即用的开发者工具。然而，这些可重用组件必须经过优化并集成到底层云原生基础设施中，才能最大化收益。”内存数据网格（IMDG）和消息代理是能够加速业务逻辑开发的应用服务。使用 Eclipse MicroProfile 开发的微服务，可以通过在其方法体内调用这些应用服务来加以利用。Eclipse MicroProfile 在集成到应用服务（如 IMDG 或消息代理）时，不会施加任何限制。
    4.  为特定任务选择合适的工具：“使用一个基于容器的应用平台，该平台支持框架、语言和架构的恰当组合，并且能够根据你的特定业务应用需求进行定制。”Eclipse MicroProfile 是开发者在为特定任务选择合适的工具时可以使用的一种工具。例如，Red Hat Application Runtimes 是一系列运行时和工具的集合，其中包括 Eclipse MicroProfile、Node.js、Spring Boot 和 Vert.x。
    5.  为开发者提供按需自助的基础设施：“使用容器和容器编排技术来简化对底层基础设施的访问，让 IT 运维团队拥有控制权和可见性，并在数据中心、私有云和公有云等各种基础设施环境中提供强大的应用生命周期管理。”使用 Eclipse MicroProfile 开发的微服务可以部署到一个或多个容器中。通过轻松管理这些容器及其上运行的微服务架构，你可以加速开发周期，从而更快地为业务交付价值。
    6.  自动化 IT 以加速应用交付：“创建自动化沙箱来学习自动化语言和流程，建立跨组织的协作对话以定义服务需求，创建能够赋能用户并加速交付的自助服务目录，并使用计量、监控和计费策略与流程。”Eclipse MicroProfile 提供了指标、容错和健康检查等功能，所有这些都可以作为 IT 自动化流程的输入。

*   7.  实施持续交付和高级部署技术：“通过自动化交付、持续集成/持续交付（CI/CD）流水线、滚动蓝绿部署和金丝雀部署以及 A/B 测试，加速云原生应用的交付。”将微服务与 CI/CD 结合使用可以促进高级部署技术。例如，你可以将一个基于 MicroProfile 的、包含新功能的微服务作为蓝绿部署或金丝雀部署的一部分引入生产环境，一旦证明新功能按预期工作，就将所有流量切换到该服务上。
    8.  演进为更模块化的架构：“选择适合你特定需求的模块化设计，可以使用微服务、单体优先方法、迷你服务或它们的组合。”对于这一步，你可以使用 Eclipse MicroProfile 为新应用开发微服务，或者在你将单体应用的特定限界上下文拆分为微服务时使用它。

5.  十二要素应用是一种方法论，开发者在设计和实现微服务及应用时可以遵循它，而与其实现所用的编程语言或框架无关。开发者可以使用 Eclipse MicroProfile 作为框架来实现遵循十二要素应用的微服务。十二要素应用和 Eclipse MicroProfile 并非互斥，而是相辅相成的。
6.  市场上大多数（如果不是全部）函数即服务（FaaS）产品都支持 Java。因此，开发者可以使用 Eclipse MicroProfile 的多种实现之一（它们都是用 Java 编写的）来编写函数体。Eclipse MicroProfile 的易用性和丰富功能，结合 FaaS 平台的简洁性，可以极大地提升开发者更快地为业务交付价值的能力。此外，像 Quarkus 这样实现了 Eclipse MicroProfile 的技术，内存占用低且启动速度快，是 FaaS 的理想运行时。
7.  在跨云部署中使用 Eclipse MicroProfile 时，需要考虑以下几点：

*   需要使用云环境提供的任何 DNS 支持来配置云环境与本地环境之间的通信路由。
    *   配置 MicroProfile OpenTracing 以启用跨云环境的追踪捕获。

*   *   监控跨云环境拆分的 MicroProfile Metrics 信息。
    *   设置 CI 任务以定位到正确的云环境，从而维护正确的微服务。

8.  在多云环境中进行分布式追踪可能具有挑战性。我们希望实现与单一云环境相同的目标，即可视化与请求关联的单个端到端追踪，该请求会穿过每个云内部以及跨云的各个服务，但在处理不同的上下文传播格式以及每个云以不同格式存储追踪数据时，可能会遇到复杂情况。第一个挑战是确保追踪能够跨越不同的云环境。这是一个问题，因为在撰写本文时，还没有广泛采用或标准化的追踪上下文格式。通常，每个追踪系统使用不同的头部和格式来传播追踪上下文。第二个挑战，即使在同构的追踪环境中，也是可视化来自多云环境的追踪数据。这可能会有问题，因为每个云中的追踪数据可能存储在不同的数据库中，或者采用不同的格式。可以通过将数据复制到单个统一存储，或者按需在系统之间发送缺失的追踪数据并进行适当的数据格式调整来克服这一挑战。
9.  诸如 Istio 或 LinkerD 之类的服务网格在 Kubernetes 之上的平台层面提供服务，涉及服务发现、路由和容错等领域。其中一些服务也可以在 MicroProfile 中找到。当你将 MicroProfile 应用部署到这样的服务网格中时，需要考虑是使用 MicroProfile 版本还是网格版本。这里最可能受影响的 MicroProfile 特性是容错，尤其是重试逻辑。
