# 4. 什么是应用服务器？

上一章讨论了什么是 JSR 参考实现。Java EE 的定义也提到它是一组抽象规范的集合。Java EE 本身就是一个 Java 规范请求。

更准确地说，Java EE 被称为一个*伞式 JSR*，因为它封装了多个 JSR。例如，Java EE 7 就是 [JSR 342](https://www.jcp.org/en/jsr/detail%253Fid%253D342)。Java EE 8 是 [JSR 366](https://www.jcp.org/en/jsr/detail%253Fid%253D366)。因此，Java EE 本身就是一个需要经过 JCP JSR 流程，并遵守每个 JSR 要求的 JSR。

基于这个想法，并记住 JSR 流程要求每个 JSR 都必须有一个参考实现，这意味着作为伞式 JSR 的 Java EE 也必须有一个参考实现。

伞式 JSR 或 Java EE 的实现通常被称为*应用服务器*。应用服务器是 Java EE 规范的具体实现，你可以在其上实际运行代码。Java EE 的参考实现是 Glassfish 应用服务器。

应用服务器通常将开发者从许多原本需要自行管理的繁琐事务中抽象出来，例如数据源池化、缓存、集群以及其他开销。

应用服务器还必须通过 TCK 测试，才能被完全认证为符合给定的伞式 JSR。应用服务器也是 Java EE 可移植性的基础。作为开发者，通常建议你针对 `javax.*` 包（即标准的 Java EE 包）进行编码。

由于应用服务器遵循标准，使用 Java EE 包意味着你可以更换应用服务器，而你的代码通常只需很少甚至无需修改即可运行。仔细想想，这很有吸引力。

首先，没有任何一个应用供应商能够锁定你。因为你可以更换应用服务器，所以理论上，如果你对当前供应商不满意，可以随时更换。

市面上有许多应用供应商，有些免费，有些昂贵。在开源供应商中，比较流行的是 Payara Server（参见 [`http://payara.fish`](http://payara.fish)），它是一个源自 Glassfish、经过全面修补的应用服务器，可以免费下载使用。

Java EE 是一个抽象规范，其具体实现被称为*应用服务器*。作为一个 JSR，Java EE 所需的参考实现是 Glassfish 应用服务器。
