# 3. 什么是参考实现？

在上一章中，你了解了什么是 JSR。回顾一下，JSR 是向 JCP 提交的一份抽象请求，其中包含了对 Java 技术平台的添加建议。

由于它是抽象的，你无法直接运行它。JSR 需要某种形式的实现或具体化，才能对最终开发者有用。这就是*参考实现*概念的由来。

每个 JSR 都必须有一个参考实现，它是 JSR 文档中包含的规范的具体实现。这是 JCP 的一项要求。每一个 JSR 都有一个免费提供并与应用服务器捆绑在一起的参考实现。

JSR 还有一个技术兼容性工具包（TCK），它是“一套测试套件，至少名义上用于检查某个声称实现了特定 Java 规范请求（JSR）的实现是否符合规范。”^(³)

TCK 用于测试 JSR 实现是否符合规范。这也是 Java EE 被称为一套*标准化*规范集合的部分原因。

这个严谨的流程也确保了源自 JSR 文档的 API 的质量。一些流行的 JSR 参考实现（RI）包括：

*   JSR 380（Bean Validation 2.0）– Hibernate Validator 6
*   JSR 367（JSON-B 绑定）– Eclipse Yasson 1.0
*   JSR 370（JAX-RS 2.1）– Jersey
*   JSR 365（CDI 2.0）– WELD 3.0

这些是一些较新且更流行的 JSR 及其各自的参考实现。这些参考实现中的大部分都与应用服务器捆绑在一起，这将是下一章的主题。
